# TS-PLUGIN-PARITY-001: DTU Canonicalization Rules for VP-PLUGIN-003 Parity Evaluation

## Purpose

This document defines the canonicalization rules used when evaluating VP-PLUGIN-003 (DTU parity).
VP-PLUGIN-003 asserts that a plugin-produced OCSF record — generated by the TOML-driven
`PipelineExecutor` or a `.prx` WASM plugin — matches the reference record produced by the
corresponding DTU clone for the same input. This strategy document specifies exactly what
"matches" means for each field category.

## Scope

These rules apply to the post-migration comparison between:
- **Actual**: OCSF record produced by the spec-driven pipeline (TOML + optional WASM plugin)
- **Reference**: OCSF record produced by the DTU clone behavioral fixture for the same raw
  API response payload

## Rule A — Field-by-Field Equality

All named OCSF fields are compared by equality with the following qualifications:

- **Required fields**: Must be present in both actual and reference. Absence in either is a
  parity FAIL regardless of other fields.
- **Optional fields**: Absence in both actual and reference is a parity PASS. Absence in one
  but not the other is a parity WARN (not FAIL) unless the field is listed as required for
  the sensor/table in the SensorSpec.
- **Unknown/extension fields** in `raw_extensions`: Compared as JSON objects with
  unordered key equality. Extra keys in actual (not in reference) are parity WARN. Missing
  keys from reference are parity FAIL.

## Rule B — Null and Missing Field Handling

- JSON `null` and field absence are NOT equivalent. If reference has `field: null` and actual
  omits the field entirely, that is a parity WARN.
- If both sides have `null`, the field comparison passes.
- OCSF fields with a default value specified in the schema: if reference omits the field (using
  the default) and actual emits the default value explicitly, this is a parity PASS.

## Rule C — Timestamp Normalization

All timestamp fields are normalized to UTC ISO-8601 before comparison:

- Input format: any of ISO 8601, RFC 3339, Unix epoch seconds (i64), Cyberint custom format
- Output format: `YYYY-MM-DDTHH:MM:SS.sssZ` (millisecond precision, UTC)
- Tolerance: timestamps within 1 second of each other are a parity PASS (accounts for
  fetch-time insertion used as fallback). Differences > 1 second are a parity FAIL.
- If both sides use the fetch timestamp as fallback (field absent in source), the comparison
  is skipped for that field (PASS by convention — both took the same fallback path).

## Rule D — UUID vs Deterministic ID Handling

- **UUID v4/v7 fields** (e.g., `uid`, `finding_info.uid`): Compared as opaque strings.
  If the sensor API returns a stable identifier, actual and reference must match exactly.
  If the sensor API returns a non-deterministic ID, and the SensorSpec marks the field
  as `determinism: false`, comparison is skipped (PASS by convention).
- **Deterministic IDs** (constructed by the adapter, e.g., from `detection_id`):
  Must match exactly. No tolerance.

## Rule E — Array Field Ordering Invariance

- Arrays in OCSF fields (e.g., `attacks[]`, `evidences[]`) are compared as **multisets**
  (order-invariant). Two arrays with the same elements in different order are a parity PASS.
- Array element equality uses field-by-field equality (Rule A) applied recursively.
- Array length must match exactly. Length mismatch is a parity FAIL.

## Rule F — JSON Schema-Anchored Comparison

- Comparison is anchored to the OCSF JSON Schema for the event class (e.g., Detection
  Finding 2004). Fields not in the schema are treated as `raw_extensions` entries.
- Schema validation runs on reference before comparison. A reference that fails schema
  validation is flagged as a DTU gap (not a pipeline parity failure) and the test is
  skipped with a WARN annotation until the DTU is fixed.
- The compiled OCSF protobuf schema (from `ocsf-proto-gen`) is the canonical source.
  The JSON Schema is derived from it for comparison purposes.

## Rule G — Floating-Point Tolerance

- Floating-point fields (e.g., risk scores, confidence values): compared with absolute
  tolerance of `1e-6`. Differences within tolerance are parity PASS.
- If a sensor returns integer risk levels and the pipeline converts to float, and the
  reference uses the integer directly, a conversion of `5` → `5.0` is a parity PASS.

## Rule H — Parity Verdict Classification

| Outcome | Condition | Action |
|---------|-----------|--------|
| PASS | All required fields equal within rules A–G | VP-PLUGIN-003 satisfied for this test case |
| WARN | Optional field absent in one side; null vs missing; non-deterministic ID skipped | Logged; does not fail VP-PLUGIN-003; recorded in parity report |
| FAIL | Required field missing; value mismatch outside tolerance; array length mismatch; extra `raw_extensions` keys absent from reference | VP-PLUGIN-003 fails; pipeline must be corrected |
| SKIP | DTU gap identified (reference fails schema); test annotated | Tracked as DTU debt; pipeline not blamed |

A sensor table passes VP-PLUGIN-003 when all test cases for that `(sensor_id, table)` pair
produce PASS or WARN verdicts with zero FAILs.

## Rule I — Test Case Sourcing

Parity test cases are sourced from two pools:

1. **Real-sensor recordings**: Captured API response payloads from authenticated sessions
   against the live sensor APIs during the Wave 1 migration window. Stored as fixtures in
   `crates/prism-dtu-{sensor}/fixtures/parity/`. Each fixture includes the raw API response
   and the reference OCSF output produced by the Rust adapter before deletion.

2. **Synthesized cases**: Constructed to cover edge cases enumerated in the sensor BCs
   (e.g., null timestamps, unrecognized severity values, polymorphic IDs). Synthesized cases
   are generated deterministically from the DTU clone's generator infrastructure
   (`prism-dtu-common`). Synthesis covers at minimum: one happy-path case, one null-field
   case, one unrecognized-enum case, and one empty-result case per sensor table.

Minimum coverage requirement: at least 3 real-sensor recordings AND at least 3 synthesized
cases per `(sensor_id, table)` pair before VP-PLUGIN-003 is considered verified.

## Cyberint DTU Gap Note

The Cyberint DTU clone in `prism-dtu-cyberint` has known gaps in `incidents` endpoint
pagination behavior (per ADR-023 Rule 3). Wave 1/D must verify coverage of the `incidents`
endpoint before authoring parity tests for that table. Until then, parity tests for
`cyberint.incidents` are in SKIP status.
