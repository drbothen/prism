# BC-3.2.003: Per-org session token isolation via (OrgId, token) composite key

## Description

Bearer and session tokens issued by sensor adapters (e.g., CrowdStrike OAuth tokens, Claroty bearer tokens, Cyberint session UUIDs) are scoped per `OrgId`. The session token store is keyed by `(OrgId, token_string)` so that a token registered under `OrgId(A)` is not accepted as valid by an `is_valid_session` check performed in `OrgId(B)`'s context, even if the token string is identical. Refresh flows preserve the `OrgId` binding: when a token expires and is refreshed, the new token is stored under the same `OrgId` as the original.

## Preconditions

1. Session/token stores in client-mode DTU crates (`prism-dtu-cyberint::session_store`, and equivalent CachedToken stores in other crates) are keyed by `(OrgId, String)` post ADR-008 migration.
2. Token registration calls carry the `OrgId` of the org for which the token was issued.
3. Token validation calls carry the `OrgId` of the requesting query context.
4. The CrowdStrike `session_registry` (LruCache for pagination) is not re-keyed by this BC (ADR-008 §2.1 rationale: session IDs are UUID v7 generated by the org-scoped query planner and are probabilistically unique per org); this is confirmed or escalated per the open question below.

## Postconditions

1. A token `T_a` registered under `OrgId(A)` is returned as valid by `is_valid_session(org_id_A, T_a)`.
2. `is_valid_session(org_id_B, T_a)` returns invalid/not-found for any `org_id_B ≠ org_id_A`, even if the token string is identical.
3. When `T_a` expires and `OrgId(A)` refreshes to `T_a_prime`: `T_a_prime` is stored under `OrgId(A)`; `T_b` (OrgId(B)'s token) is unaffected by the refresh.
4. Token expiry and refresh for one org do not affect the token state of any other org.

## Invariants

1. Every session token store entry has the form `(OrgId, token_string)` — the OrgId component is never omitted or defaulted in production code paths.
2. Token validation (`is_valid_session`) always takes `(org_id, token)` as input, never just `token` alone.
3. The token refresh flow explicitly preserves the `OrgId` binding: `refresh(org_id, old_token) -> new_token` stores the new token under the same `org_id`.
4. Probabilistic uniqueness of token strings across orgs is not sufficient for formal isolation: structural keying by `(OrgId, token)` is required regardless of token generation strategy.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | orgA acquires token T_a; orgB also acquires T_a (same string, different contexts) | Stored as (org_id_A, T_a) and (org_id_B, T_a) independently; each is valid only in its own org context |
| EC-002 | orgA's token T_a expires; refresh produces T_a2; orgB's T_b is unaffected | (org_id_A, T_a2) replaces (org_id_A, T_a); (org_id_B, T_b) unchanged |
| EC-003 | is_valid_session(org_id_B, T_a) where T_a was only registered under org_id_A | Returns invalid (not-found); no error propagation beyond the false return |
| EC-004 | Token store reset_for(org_id_A) removes T_a; T_b for orgB unaffected | (org_id_A, T_a) removed; (org_id_B, T_b) remains |
| EC-005 | Cyberint session_store: adversarial test where both orgs use identical UUID v4 token string | Stored independently by (OrgId, token); cross-org is_valid_session returns false even with identical string |
