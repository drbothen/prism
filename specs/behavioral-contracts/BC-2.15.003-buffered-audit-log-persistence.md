# BC-2.15.003: Buffered Audit Log Persistence — Write to RocksDB Before stderr/Vector, Exponential Backoff on Forward Failure

## Preconditions
- An audit entry has been generated by any MCP tool invocation or internal operation (BC-2.05.001)
- The RocksDB `audit_buffer` column family is initialized (BC-2.15.001)

## Postconditions
- The audit entry is first persisted to the RocksDB `audit_buffer` column family with a time-ordered key: `audit:{timestamp_nanos}:{uuid}`
- After persistence to RocksDB, the entry is forwarded to two sinks in parallel:
  1. **stderr** (structured JSON via `tracing` crate) -- for local consumption and Vector pipeline pickup
  2. **Direct Vector sink** (if configured) -- TCP/HTTP forward to a Vector endpoint
- If stderr write fails: entry remains in RocksDB buffer; warning logged to an internal channel
- If Vector forward fails: exponential backoff retry with parameters:
  - Initial delay: 1 second
  - Max delay: 60 seconds
  - Backoff multiplier: 2x
  - Max retries: 10 (then entry is marked as "forward_failed" in buffer)
- Successfully forwarded entries are deleted from the RocksDB buffer
- On startup, any entries remaining in the `audit_buffer` (from previous crashes or forward failures) are re-forwarded using the same retry logic

## Invariants
- Persist-then-forward ordering: every audit entry is in RocksDB before any forwarding attempt
- No audit entry is lost: even if Vector is unreachable and stderr fails, entries survive in RocksDB
- Forwarding does not block the calling tool; the buffer forwarder runs as a background task

## Error Cases
| Error | Condition | Behavior |
|-------|-----------|----------|
| `E-AUDIT-001` | RocksDB write failure for audit entry | For write operations: tool invocation fails (fail-closed per BC-2.05.001). For read operations: entry logged to stderr only (best-effort) |
| `E-AUDIT-002` | Vector endpoint unreachable | Exponential backoff retry; entries accumulate in buffer |
| `E-AUDIT-003` | Buffer exceeds 100K entries | Overflow handling per BC-2.15.004 |

## Edge Cases
| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-15-011 | Vector endpoint goes down for 30 minutes | Entries accumulate in RocksDB; when Vector recovers, all buffered entries are forwarded in order |
| EC-15-012 | Server crash with 500 buffered entries | On restart, all 500 entries are re-forwarded |
| EC-15-013 | No Vector endpoint configured | Entries forwarded to stderr only; deleted from buffer after stderr write |
