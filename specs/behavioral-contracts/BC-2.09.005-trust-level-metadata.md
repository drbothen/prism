# BC-2.09.005: Trust-Level Metadata Per Response

## Preconditions
- An MCP tool has produced a response (success or error)
- The response is being wrapped in the standard response envelope

## Postconditions
- Every tool response includes `_meta.trust_level` in `structuredContent`
- Sensor data responses use `trust_level: "untrusted_external"` -- data originates from external sensor APIs monitoring adversarial environments
- Internal data responses (health checks, capability listings, credential management confirmations) use `trust_level: "internal"` -- data generated by Prism itself
- Error responses include `trust_level: "internal"` -- error messages are Prism-generated, not sensor-sourced
- The `trust_level` field is a string enum with exactly two values: `"untrusted_external"` and `"internal"`

## Invariants
- DI-006: Trust-level metadata reinforces the boundary between trusted and untrusted data

## Error Cases
| Error | Condition | Behavior |
|-------|-----------|----------|
| None | Trust level is always deterministic based on tool type | No runtime failure possible |

## Edge Cases
| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-09-011 | Cross-client query aggregating data from multiple sensors | `trust_level: "untrusted_external"` because the data still originates from external sensors |
| EC-09-012 | Error response that quotes a sensor API error message | `trust_level: "internal"` but the `upstream_message` field within the error is treated as untrusted; the error construction does not interpolate upstream messages into prose |
| EC-09-013 | Tool that returns a mix of internal and sensor data (e.g., capabilities plus sensor status) | `trust_level: "untrusted_external"` -- the more restrictive level applies when any part of the response contains external data |
