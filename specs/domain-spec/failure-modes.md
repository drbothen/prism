# Runtime Failure Modes

Each failure mode describes a specific way Prism can fail at runtime, its impact on the analyst experience, how it is detected, and the recovery path.

| ID | Subsystem | Failure Mode | Impact | Detection | Recovery |
|----|-----------|-------------|--------|-----------|----------|
| FM-001 | prism-sensors | **Sensor API unreachable**: HTTP connection to sensor API (CrowdStrike, Cyberint, Claroty, or Armis) is refused, times out, or returns HTTP 5xx. | Queries for the affected sensor return errors. Cross-client queries return partial results for other sensors. Analyst cannot investigate alerts from that sensor until connectivity is restored. | HTTP client error or timeout (configurable per-sensor, default 30s). Retry with exponential backoff (2s base, 30s max). Health check (`verify_connectivity()`) reports unreachable. | Structured error with `category: "transient"`, `retryable: true`, `suggestion: "Sensor API is currently unreachable. Retry in {backoff} seconds."`. After max retries exhausted, error escalates to `category: "upstream_error"` with suggestion to check sensor status independently. Cursor does not advance; next query retries from same position. |
| FM-002 | prism-sensors | **Auth token expired mid-session**: CrowdStrike OAuth2 access token expires during a query sequence. Cyberint cookie becomes invalid. Claroty or Armis bearer token is revoked. | Current query fails with HTTP 401/403. Subsequent queries also fail until credentials are refreshed or reconfigured. | HTTP 401 or 403 response from sensor API. CrowdStrike: OAuth2 middleware auto-detects and attempts refresh. Other sensors: no auto-refresh for static tokens. | CrowdStrike: automatic token refresh via OAuth2 client credentials flow, transparent to caller. Cyberint/Claroty/Armis: structured error with `category: "authentication"`, `retryable: false`, `suggestion: "Sensor credentials are expired or invalid. Update credentials via credential management tools and restart Prism."`. Audit entry records the auth failure with sensor and client context. |
| FM-003 | prism-state | **Disk full (state store)**: FileStore atomic write fails because the filesystem has insufficient space for the temp file. | Cursor state cannot be persisted. Per invariant DI-009, in-memory cursor does not advance. The current query cycle completes but the cursor position is not saved. Next session restart re-processes data from the last saved cursor. | `std::io::Error` with `ErrorKind::StorageSpace` (or equivalent OS error) during temp file write. | Error logged with `PRISM_STATE_DIR` path and current disk usage. Structured error with `category: "configuration"`, `retryable: false`, `suggestion: "Insufficient disk space for state persistence at {path}. Free disk space or change PRISM_STATE_DIR."`. Prism continues to serve read queries but warns that state is not being persisted. |
| FM-004 | prism-credentials | **OS keyring locked or unavailable**: macOS Keychain requires password entry, Linux libsecret is not running, or Windows Credential Vault is inaccessible. | Credentials cannot be retrieved. Sensor adapters fail to authenticate. All queries for affected sensors return credential errors. | `keyring-rs` returns platform-specific error on `get()` call. Startup probe (macOS permission pre-auth pattern) detects this at launch. | If detected at startup: fail-fast with actionable error listing the keyring error and suggesting encrypted file fallback. If detected mid-session: structured error with `category: "configuration"`, `suggestion: "OS keyring is unavailable. Restart Prism after unlocking keyring, or configure PRISM_CREDENTIAL_BACKEND=file with an encryption key."`. Fallback to encrypted file backend if configured. |
| FM-005 | prism-ocsf | **OCSF schema load failure**: The protobuf descriptor set compiled into the binary by `build.rs` is corrupted or incompatible with the prost-reflect runtime. | OCSF normalization is unavailable. Raw sensor data can still be returned but cross-sensor correlation via OCSF fields is not possible. | `prost-reflect` returns error when initializing `DescriptorPool` from compiled descriptor bytes. Detected at startup. | Fatal startup error: "OCSF protobuf descriptor failed to load: {error}. This is a build-time issue. Rebuild Prism with `cargo build`." Prism does not start. This is a build/release defect, not a runtime configuration issue. |
| FM-006 | prism-sensors | **Malformed sensor API response**: Sensor API returns HTTP 200 but the response body is not valid JSON, is truncated, or has an unexpected schema (missing required fields). | The specific query fails. Cursor does not advance for the affected page. Other sensors and clients are unaffected. | `serde_json` deserialization error. Lenient parsing (`#[serde(default)]`) absorbs missing optional fields, but completely invalid JSON or missing structural fields trigger errors. | Structured error with `category: "upstream_error"`, `retryable: true` (first attempt -- may be transient), `suggestion: "Sensor returned unexpected response format. Retrying once. If persistent, the sensor API may have changed its contract."`. Log the raw response body (truncated to 4KB) at debug level for diagnosis. If persistent across retries, escalate to `retryable: false` with suggestion to verify sensor API version compatibility. |
| FM-007 | prism-core | **Confirmation token expired**: Agent calls `confirm_action` after the 300-second token TTL has elapsed. | The irreversible write operation is not executed. The agent must restart the two-step confirmation flow. | `ConfirmationToken::validate()` compares `expires_at` against current time. | Structured error with `code: "TOKEN_EXPIRED"`, `category: "validation"`, `retryable: false`, `suggestion: "Confirmation token expired after 300 seconds. Call {original_tool_name} again to generate a new confirmation token."`. The `original_tool_name` is included in the token metadata so the error message is actionable. |
| FM-008 | prism-sensors | **Sensor API rate limited (HTTP 429)**: Sensor API returns HTTP 429 Too Many Requests, typically with a `Retry-After` header. | Current query is delayed. Subsequent queries for the same sensor may also be delayed until the rate limit window resets. Other sensors are unaffected. | HTTP 429 response status. `Retry-After` header parsed for backoff duration. | Structured error with `category: "transient"`, `retryable: true`, `retry_after_seconds: {value from Retry-After header or default 30}`, `suggestion: "Sensor API rate limit reached. Retry after {N} seconds."`. Internal rate tracker updates to prevent immediate re-request. If rate limiting persists, suggest cached results or reducing query frequency. |
| FM-009 | prism-state | **Cursor regression detected**: A new cursor derived from sensor records is less than the stored cursor for the same (client, sensor, source) tuple. This indicates a sensor API bug, clock skew, or data corruption. | Collection halts for the affected data source. Other sources and sensors continue normally. | `ensure_forward_progress()` comparison returns false. | Fatal error for the affected source: `PrismError::CursorRegression` with both cursor values. Structured error with `category: "upstream_error"`, `retryable: false`, `suggestion: "Cursor regression detected for {sensor}/{source}. The sensor API returned records older than the last processed position. This may indicate a sensor API issue. To reset, delete the state file at {path}."`. Audit entry records the regression with full cursor details. |
| FM-010 | prism-config | **Configuration validation failure at startup**: TOML config file has missing required fields, invalid values, or structural errors. | Prism does not start. No MCP session is established. | `--dry-run` mode validates and reports all errors. Normal startup validates config and fails fast on first error batch. | Multi-error aggregation: all validation errors reported in one pass with exact TOML paths: "Missing required field: clients.acme.sensors.crowdstrike.api_base", "Invalid value for clients.globex.capabilities.containment: expected boolean, got string". Exit code 1 with human-readable error output to stderr. |
| FM-011 | prism-mcp | **MCP client disconnection during long query**: Claude Code closes the stdio pipe (user quits, process killed) while Prism is mid-query against a sensor API. | In-flight HTTP requests continue to completion but MCP responses cannot be delivered. Cursor may or may not have been persisted depending on where in the pipeline the disconnection occurred. | Write to stdout returns broken pipe error. `rmcp` transport layer detects disconnection. | Prism initiates graceful shutdown: cancel in-flight tokio tasks, flush pending state writes, close HTTP clients. If cursor was persisted before disconnection, no data loss. If not, next session re-processes from last saved position. Shutdown completes within 5-second timeout, then force-exits. |
| FM-012 | prism-sensors | **Claroty polymorphic ID deserialization failure**: Claroty API returns an ID field that is neither a valid JSON string nor a valid JSON number (e.g., null, boolean, or nested object). | The specific record cannot be processed. Other records in the same batch are unaffected. | `serde` deserialization of `PolymorphicId` enum fails for the specific field. | Record is skipped with a warning log: "Claroty record skipped: unexpected ID type at field '{field}', value: {raw}". The record is included in a `skipped_records` array in the response metadata so the agent is aware. Cursor advances past the skipped record to maintain forward progress. |
