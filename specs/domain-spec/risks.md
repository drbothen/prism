# Risk Register

Risks are potential future events that could negatively impact Prism's delivery or operation. Each risk includes likelihood, impact, and specific mitigation actions.

| ID | Risk | Likelihood | Impact | Mitigation | Status | Category | Traced To |
|----|------|-----------|--------|------------|--------|----------|-----------|
| R-001 | **rmcp breaking changes**: rmcp is pre-1.0 (v0.8). A minor version bump could introduce breaking API changes to `#[tool_router]`, `Parameters<T>`, or `ServerHandler` trait, requiring significant rework of prism-mcp. | HIGH | HIGH | Pin exact rmcp version in `Cargo.lock` and `Cargo.toml`. Introduce a thin adapter layer (`McpAdapter` trait) between Prism's tool logic and rmcp types so tool handlers never import rmcp directly. Monitor rmcp GitHub repository for pre-release changelogs. Run weekly CI job against rmcp `main` branch. Budget 2 weeks contingency for MCP layer rework. | OPEN | Technical / Dependency | ASM-001, CAP-009 |
| R-002 | **CrowdStrike no Rust SDK**: CrowdStrike provides official SDKs for Python and Go (`gofalcon`) but not Rust. Prism must implement the CrowdStrike Falcon API client from scratch using `reqwest`, including OAuth2 client credentials flow, multi-region endpoint selection, two-step alert fetch, and FQL query construction. | HIGH | MEDIUM | Use poller-cobra's behavioral specification as the definitive contract for API endpoints, auth flow, and response schemas. Implement OAuth2 client credentials using the `oauth2` crate (well-maintained). Write integration tests against CrowdStrike sandbox API. Budget extra development time for the CrowdStrike adapter vs. sensors with simpler auth. | OPEN | Technical / Integration | ASM-008, CAP-001 |
| R-003 | **Sensor API contract changes**: Any of the 4 sensor vendors (CrowdStrike, Cyberint, Claroty, Armis) could change their API contracts (endpoints, field names, auth mechanisms, pagination behavior) without notice, breaking the corresponding sensor adapter. | MEDIUM | HIGH | Design sensor adapters with lenient parsing (`#[serde(default)]`, no `DisallowUnknownFields`). Use integration tests with recorded API responses (`wiremock` fixtures) to detect contract drift. Subscribe to vendor API changelogs. Implement per-sensor API version configuration so adapters can target a specific API version. Build the `DataSource<T>` trait so adapter fixes propagate without core changes. | OPEN | Technical / Integration | ASM-008, ASM-009, CAP-001 |
| R-004 | **OCSF field instability**: OCSF is a relatively young standard (v1.x). Proto field numbers generated by ocsf-proto-gen are NOT stable across OCSF versions. Field semantics could change between minor versions, invalidating cross-sensor correlation logic. | MEDIUM | MEDIUM | Pin OCSF schema version per Prism release. Commit generated `.proto` files to detect drift. Test OCSF mappings against multiple schema versions in CI. Use `raw_extensions` blob to preserve unmapped fields so no data is lost during schema transitions. Version the OCSF schema in the proto package path (`ocsf.v1_7_0`). | OPEN | Technical / Schema | ASM-002, ASM-005, CAP-003 |
| R-005 | **Prompt injection via sensor data**: Attackers who know they are being monitored embed prompt injection payloads in observable artifacts (hostnames, file paths, process names, alert descriptions). These flow through Prism into the LLM context window, potentially causing the AI agent to follow attacker instructions. | HIGH | HIGH | Four-layer defense: (1) structural separation -- untrusted data in `structuredContent` JSON fields, never interpolated into prose; (2) provenance framing -- tool descriptions warn LLM that field values are untrusted external data; (3) suspicious pattern flagging -- regex-based detection adds `_safety_flag` parallel fields; (4) trust-level metadata -- response envelope includes `trust_level: "untrusted_external"`. Accept that no sanitization is 100% effective; human-in-the-loop (analyst) is the ultimate safety boundary. | OPEN | Security | CAP-010, DI-006 |
| R-006 | **Credential exposure**: If Prism's credential storage is compromised, API keys for CrowdStrike, Cyberint, Claroty, and Armis across all MSSP clients are exposed simultaneously. Single point of compromise for all sensor credentials. | LOW | CRITICAL | AES-256-GCM encryption for file-backend credentials with external key material (env var or K8s secret). OS keyring as primary backend (hardware-backed encryption). Credential names sanitized against `[a-zA-Z0-9_\-\.]+` (prevents path traversal). Credential values never logged, never in MCP responses, never in error messages. File permissions enforced: 0600 for files, 0700 for directories. Per-client credential isolation via `TenantId` namespace. | OPEN | Security | DI-002, DI-014, CAP-004 |
| R-007 | **Multi-client data mixing**: Prism aggregates all MSSP clients in a single process. A bug in client scoping could cause Client A's data to be returned for a Client B query, or cursor state to cross-contaminate between clients. | MEDIUM | HIGH | `TenantId` newtype enforced at the type level across all subsystems. Per-client state directories. Per-client cache key prefixes. All log spans include `tenant = %tenant_id`. Integration tests explicitly verify cross-client isolation (HS-003 scenario group). Property tests with random client IDs verify no cross-contamination. | OPEN | Data Integrity | DI-008, CAP-001, CAP-002 |
| R-008 | **RocksDB corruption on crash**: If Prism crashes during a RocksDB write (schedules, detection state, cases, audit buffer), data could be lost or inconsistent. Cursor state is not at risk since cursors are ephemeral and in-memory only. | LOW | MEDIUM | RocksDB WAL ensures crash consistency -- incomplete writes are replayed or discarded on recovery. Dirty bit markers (BC-2.15.005) detect mid-query crashes and trigger recovery logic on restart. Epoch/counter tracking (DI-023) ensures exactly-once semantics after restart. MemoryStore banned from production builds (DI-011). Integration tests simulate crash scenarios (HS-006 scenario group). | OPEN | Data Integrity | DI-023, DI-013, CAP-011 |
| R-009 | **keyring-rs platform incompatibility**: keyring-rs may not work correctly on all target platforms (especially Linux distributions without libsecret/Secret Service, or headless environments). | MEDIUM | LOW | AES-256-GCM encrypted file backend as automatic fallback. Platform-specific integration tests in CI. Documentation of prerequisites per platform (libsecret on Linux, Keychain access on macOS). Startup probe pattern from serveMyAPI: test keyring access at launch and warn if unavailable. | OPEN | Technical / Platform | ASM-003, FM-004 |
| R-010 | **Rate limit exhaustion on sensor APIs**: Interactive query patterns (bursty, on-demand) may hit sensor API rate limits more aggressively than the periodic polling the Go pollers used. Multiple clients multiplied by interactive queries could exhaust per-API-key rate budgets. | MEDIUM | MEDIUM | LRU cache per client per sensor (bounded, from ADR-011). Per-sensor rate tracking with proactive budget warnings. Backoff with `Retry-After` header respect. Structured error responses with `retry_after_seconds` so the agent waits appropriately. Rate limit state exposed via sensor health monitoring (CAP-008). | OPEN | Operational | ASM-004, DEC-001, FM-001 |
| R-011 | **Cyberint auth mechanism change**: Cyberint's cookie-based authentication is unusual and may be deprecated in favor of standard bearer token or OAuth2 auth. The poller-express reference may reflect an outdated auth contract. | MEDIUM | LOW | Verify current Cyberint API auth mechanism before implementing the adapter. Design the `SensorAuth` trait to accommodate auth mechanism changes without modifying the adapter's data fetching logic. If Cyberint has moved to bearer token auth, this simplifies the implementation. | OPEN | Technical / Integration | ASM-009, CAP-001 |
| R-012 | **Confirmation token replay or forgery**: An attacker or malfunctioning agent could attempt to replay a consumed confirmation token or forge a token to execute irreversible write operations without proper confirmation. | LOW | HIGH | Tokens generated with cryptographic randomness (`rand::rngs::OsRng`). Single-use enforcement via in-memory consumed set. Action-specific content hash prevents using a token from one action to authorize a different action. 300-second expiry limits replay window. All token operations audit-logged. Tokens stored in-memory only (no persistence across restarts). | OPEN | Security | DI-007, CAP-006 |
