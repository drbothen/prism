# Domain Invariants

Domain invariants are rules that must hold true at all times during Prism's operation. Violation of any invariant indicates a bug or misconfiguration, not a recoverable runtime condition. Each invariant specifies the scope (which subsystem enforces it) and what happens on violation.

| ID | Rule | Scope | Violation Behavior |
|----|------|-------|--------------------|
| DI-001 | **Cursor Forward Progress**: A new cursor must be strictly greater than or equal to the stored cursor for the same (client_id, sensor_id, source_id) tuple. Cursor regression never occurs under correct operation. | prism-state: `ensure_forward_progress()` | Fatal `PrismError::CursorRegression` with both cursor values in the error message. The collection cycle halts for that source. Operator must investigate and potentially delete the state file. |
| DI-002 | **Credential Isolation Per Client**: Credentials are namespaced by `(client_id, sensor_id, credential_name)`. No code path allows reading Client A's credentials while operating in Client B's context. | prism-credentials: `CredentialStore` trait signature requires `TenantId` | Compile-time enforcement via type system. The `get()` method requires a `&TenantId` parameter. There is no "get all credentials" method that crosses client boundaries. Runtime audit logging confirms the client_id on every credential access. |
| DI-003 | **Feature Flag Deny-by-Default**: If no feature flag explicitly enables a write capability for a client, the capability is denied. The fallback at every level of the hierarchy is deny. | prism-core: `ClientCapabilities::is_enabled()` | Returns `false` when no matching path is found. Write tools gated by `is_enabled()` return a structured error explaining which capability is missing and how to enable it in TOML config. The tool is not registered in `tools/list` at all if compile-time cargo feature is also absent. |
| DI-004 | **Audit Completeness**: Every MCP tool invocation produces exactly one AuditEntry, regardless of success or failure. Write operations additionally log capability check results. | prism-mcp: tool dispatch middleware | If audit emission fails (e.g., tracing subscriber error), the tool invocation itself still proceeds but a warning is emitted to stderr. Audit completeness is verified by integration tests that assert log output for every tool call path. |
| DI-005 | **OCSF Schema Validity**: Every OcsfEvent produced by the normalization layer conforms to the OCSF protobuf schema compiled into the binary. Unknown fields are preserved in `raw_extensions`, not silently dropped. | prism-ocsf: `OcsfNormalizer` | If a required OCSF field cannot be mapped from the sensor record, the normalization produces a valid OCSF message with the field absent (OCSF fields are optional by design). A warning is logged with the unmapped field name and sensor record type. Invalid protobuf messages (encoding failure) are fatal errors logged and skipped, never delivered downstream. |
| DI-006 | **Prompt Injection Sanitization**: No MCP tool response interpolates untrusted sensor data into prose text. All sensor-originated string values are placed in `structuredContent` JSON fields with provenance metadata. | prism-mcp: response construction | Violation would be a code-level bug (string interpolation of sensor data into `content[].text`). Enforced by code review convention and integration tests that assert sensor field values appear only in `structuredContent`, not in prose summary text. Suspicious pattern detection flags are additive (parallel fields), never modifying original data. |
| DI-007 | **Confirmation Token Expiry**: A ConfirmationToken is valid for exactly 300 seconds from creation. Expired tokens are rejected with a structured error guiding re-request. Consumed tokens are rejected with a distinct error. | prism-core: `ConfirmationToken::validate()` | Expired token returns `PrismError::TokenExpired` with the original action summary so the agent can re-request. Consumed token returns `PrismError::TokenConsumed`. Both errors include `retryable: false` and `suggestion` fields guiding the agent to call the original write tool again. |
| DI-008 | **Client Data Separation**: State files, cache entries, log fields, and error messages are always scoped by `client_id`. No code path allows cross-client data leakage without explicit cross-client query (`client_id: null`). | All crates: `TenantId` newtype threading | State files are stored in per-client directories: `{state_dir}/{client_id}/{sensor_id}/{source_id}.json`. Cache keys include `client_id` as prefix. Log spans always include `tenant = %tenant_id`. Error messages never include data from a different client than the one being operated on. |
| DI-009 | **Persistence Before State Update**: In-memory cursor state is updated only AFTER successful persistence to the FileStore. If persistence fails, the cursor does not advance. | prism-state: collection loop ordering | This invariant is the fix for poller-cobra's state-ordering bug (HS-007-03). The code structure enforces: `file_store.save(new_cursor)?; self.current_cursor = new_cursor;`. If `save()` returns an error, the assignment never executes. |
| DI-010 | **Query Fingerprint Consistency**: The query fingerprint (SHA-256 of sorted config fields + limit) stored alongside cursor state must match the current configuration. A mismatch means the cursor is invalid for the current query parameters. | prism-state: startup validation | Fatal `PrismError::FingerprintMismatch` with both fingerprints and the actionable message "delete state file to reset". This prevents silently resuming a cursor that was built with different filter parameters, which would cause data gaps or duplicates. |
| DI-011 | **MemoryStore Production Ban**: The `MemoryStore` implementation of `CursorStore` is available only in `#[cfg(test)]` builds. Any attempt to use MemoryStore in a release binary is a compile error. | prism-state: conditional compilation | Compile-time enforcement via `#[cfg(test)]` on the `MemoryStore` type. This is the fix for poller-cobra's hardcoded MemoryStore bug (HS-007-01). No runtime check needed. |
| DI-012 | **Sealed Auth Trait**: The `SensorAuth` trait is sealed -- it cannot be implemented outside the `prism-sensors` crate. This prevents accidental composition that could route CrowdStrike's OAuth2 tokens through Cyberint's cookie middleware. | prism-sensors: sealed trait pattern | Compile-time enforcement. External crates attempting to implement `SensorAuth` get a compile error about the private `Sealed` supertrait. |
| DI-013 | **Atomic State Writes**: All state file writes use the temp-fsync-rename pattern. No state file is ever written in place. | prism-state: `FileStore::save()` | Write to temporary file in the same directory, fsync the temp file, rename over the target path. On rename failure, the temp file is cleaned up. This ensures crash-safety: the state file is either the old version or the new version, never a partial write. |
| DI-014 | **Credential Name Sanitization**: All credential names are validated against the pattern `[a-zA-Z0-9_\-\.]+` before any filesystem or keyring operation. | prism-credentials: input validation | `PrismError::InvalidInput` with the rejected name and the allowed pattern. This prevents path traversal attacks inherited from serveMyAPI (HS-007-06). Validation occurs at the `CredentialStore` trait boundary, before any backend is invoked. |
