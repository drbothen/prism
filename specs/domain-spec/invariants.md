# Domain Invariants

Domain invariants are rules that must hold true at all times during Prism's operation. Violation of any invariant indicates a bug or misconfiguration, not a recoverable runtime condition. Each invariant specifies the scope (which subsystem enforces it) and what happens on violation.

| ID | Rule | Scope | Violation Behavior |
|----|------|-------|--------------------|
| DI-001 | **Adapter Cursor Pagination Validity (Internal)**: An ephemeral pagination cursor used by the sensor adapter layer (CAP-001) is valid only within the adapter fetch session that created it. Cursors encode opaque adapter-specific state sufficient to fetch the next page from the sensor API. Cursors are never exposed to MCP clients -- they are internal to the adapter layer and consumed by the query engine (CAP-015) during fan-out. A cursor from one adapter fetch cannot be used with a different fetch or after the session ends. | prism-state: cursor validation on adapter page fetch | Invalid or expired cursor returns `PrismError::InvalidCursor` internally, causing the query engine to report a structured error to the agent. The agent never handles cursors directly -- it adjusts query scope (filters, `LIMIT`) to control result size. |
| DI-002 | **Credential Isolation Per Client**: Credentials are namespaced by `(client_id, sensor_id, credential_name)`. No code path allows reading Client A's credentials while operating in Client B's context. | prism-credentials: `CredentialStore` trait signature requires `TenantId` | Compile-time enforcement via type system. The `get()` method requires a `&TenantId` parameter. There is no "get all credentials" method that crosses client boundaries. Runtime audit logging confirms the client_id on every credential access. |
| DI-003 | **Feature Flag Deny-by-Default with Deny Override**: If no feature flag explicitly enables a write capability for a client, the capability is denied (implicit deny). Capabilities are stored as `BTreeMap<String, Effect>` where Effect is Allow or Deny. Resolution walks from the exact capability path upward; the most-specific matching rule wins. At the same specificity level, deny overrides allow. | prism-core: `ClientCapabilities::is_allowed()` | Returns `false` when no matching path is found (implicit deny) or when the most-specific matching rule is `Deny`. A deny at the same or more specific level always overrides an allow at a less specific level. Write tools gated by `is_allowed()` return a structured error explaining which rule denied the capability and how to adjust it in TOML config. The tool is not registered in `tools/list` at all if compile-time cargo feature is also absent. |
| DI-004 | **Audit Completeness (Split Behavior)**: Every MCP tool invocation AND every scheduled query execution AND every detection evaluation produces exactly one AuditEntry, regardless of success or failure. All three operation types use the same `AuditEntry` format and the same buffered audit pipeline (CAP-025). Write operations additionally log capability check results. Behavior on audit emission failure is split by operation type: **read operations** proceed with a warning; **write operations** fail-closed (the write is not executed if audit cannot be emitted); **scheduled/detection operations** proceed with a warning (they are internal, not agent-initiated mutations). | prism-mcp: tool dispatch middleware, prism-scheduler: execution loop, prism-detection: evaluation loop | For read operations: if audit emission fails, the tool invocation proceeds but a warning is emitted to stderr and `_meta.audit_warning` is included in the response. For write operations: if audit emission fails, the tool invocation is aborted with `PrismError::AuditRequired` — the write is not executed. For scheduled query executions and detection evaluations: if audit emission fails, execution proceeds with a warning logged to stderr. This ensures SOC 2 compliance for mutation operations while not blocking read-only queries or background operations on audit infrastructure failures. |
| DI-005 | **OCSF Schema Validity**: Every OcsfEvent produced by the normalization layer conforms to the OCSF protobuf schema compiled into the binary. Unknown fields are preserved in `raw_extensions`, not silently dropped. | prism-ocsf: `OcsfNormalizer` | If a required OCSF field cannot be mapped from the sensor record, the normalization produces a valid OCSF message with the field absent (OCSF fields are optional by design). A warning is logged with the unmapped field name and sensor record type. Invalid protobuf messages (encoding failure) are fatal errors logged and skipped, never delivered downstream. |
| DI-006 | **Prompt Injection Sanitization**: No MCP tool response interpolates untrusted sensor data into prose text. All sensor-originated string values are placed in `structuredContent` JSON fields with provenance metadata. | prism-mcp: response construction | Violation would be a code-level bug (string interpolation of sensor data into `content[].text`). Enforced by code review convention and integration tests that assert sensor field values appear only in `structuredContent`, not in prose summary text. Suspicious pattern detection flags are aggregated in the centralized `_meta.safety_flags` array on the response envelope — not stored as per-field parallel fields. Original data is never modified. |
| DI-007 | **Confirmation Token Expiry**: A ConfirmationToken is valid for exactly 300 seconds from creation. A token is expired when `now >= expires_at` (at exactly the expiry instant, the token is expired — boundary is inclusive for expiry). Expired tokens are rejected with a structured error guiding re-request. Consumed tokens are rejected with a distinct error. | prism-core: `ConfirmationToken::validate()` | Expired token returns `PrismError::TokenExpired` with the original action summary so the agent can re-request. Consumed token returns `PrismError::TokenConsumed`. Both errors include `retryable: false` and `suggestion` fields guiding the agent to call the original write tool again. |
| DI-008 | **Client Data Separation**: Cache entries, credential files, log fields, and error messages are always scoped by `client_id`. No code path allows cross-client data leakage without explicit cross-client query (`client_id: null`). | All crates: `TenantId` newtype threading | Credential files (encrypted file backend) are stored in per-client directories: `{state_dir}/{client_id}/`. Cache keys include `client_id` as prefix. Log spans always include `tenant = %tenant_id`. Error messages never include data from a different client than the one being operated on. |
| ~~DI-009~~ | ~~**Persistence Before State Update**~~ | **REMOVED** — No longer applicable. Cursors are ephemeral pagination tokens with no disk persistence. There is no stored cursor state to keep in sync with in-memory state. | N/A |
| ~~DI-010~~ | ~~**Query Fingerprint Consistency**~~ | **REMOVED** — No longer applicable. With ephemeral cursors, there is no persisted cursor state that could become inconsistent with changed query parameters. Query parameters are validated per-invocation. | N/A |
| ~~DI-011~~ | ~~**MemoryStore Production Ban**~~ | **REMOVED** — No longer applicable. Cursors are ephemeral and in-memory by design. There is no FileStore vs MemoryStore distinction — all cursor state is in-memory. | N/A |
| DI-012 | **Sealed Auth Trait**: The `SensorAuth` trait is sealed -- it cannot be implemented outside the `prism-sensors` crate. This prevents accidental composition that could route CrowdStrike's OAuth2 tokens through Cyberint's cookie middleware. | prism-sensors: sealed trait pattern | Compile-time enforcement. External crates attempting to implement `SensorAuth` get a compile error about the private `Sealed` supertrait. |
| ~~DI-013~~ | ~~**Atomic State Writes**~~ | **REMOVED** — No longer applicable for cursors. Cursor state is ephemeral and in-memory only. Atomic file writes remain applicable to other file operations (config, credentials) but are not a domain invariant for cursor state. | N/A |
| DI-014 | **Credential Name Sanitization**: All credential names are validated against the pattern `[a-zA-Z0-9_\-\.]+` before any filesystem or keyring operation. | prism-credentials: input validation | `PrismError::InvalidInput` with the rejected name and the allowed pattern. This prevents path traversal attacks inherited from serveMyAPI (HS-007-06). Validation occurs at the `CredentialStore` trait boundary, before any backend is invoked. |
| DI-015 | **Confirmation Token Cap**: A maximum of 100 active (unexpired, unconsumed) confirmation tokens may exist at any time. New token requests beyond this cap are rejected. | prism-core: `TokenStore::create()` | Returns `PrismError::TokenCapReached` with `suggestion: "Too many pending confirmation tokens. Wait for existing tokens to expire (300s TTL) or confirm/cancel pending operations."` This prevents memory exhaustion from unbounded token creation (ADV-2-002). Lazy cleanup of expired tokens runs before the cap check. |
| DI-016 | **Audit Fail-Closed for Writes**: Write operations (any mutation that changes sensor state) must not execute if audit emission fails. Read operations may proceed with a warning. **Note:** This invariant restates the write-specific fail-closed behavior from DI-004 for explicit cross-referencing. The canonical definition is DI-004. | prism-mcp: tool dispatch middleware | Write tool invocations are wrapped: audit is emitted first; if emission fails, the write is aborted with `PrismError::AuditRequired`. Read tool invocations proceed on audit failure with `_meta.audit_warning` in the response. This split ensures SOC 2 compliance for mutations while maintaining availability for queries. |
| DI-017 | **Single-Process Invariant**: Each Prism process serves a single analyst. In-memory state (cursors, tokens, cache) is not shared across processes. Concurrent access to the same state directory from multiple Prism processes is unsupported and may cause undefined behavior. | prism-core: process model | RocksDB's built-in LOCK file prevents the most common case (two processes using the same `state_dir`). The same-config-different-`state_dir` scenario is documented as unsupported but not enforced. The stdio transport model (one Prism process per analyst terminal session) naturally enforces single-process access. Documentation warns against running multiple Prism instances against the same config/state directory. |
| DI-018 | **Cache Bounds**: Cache entries are bounded by a configurable maximum per client per sensor (default 50). LRU eviction removes the least-recently-used entry when the bound is exceeded. The default of 50 scales to ~100MB for 50 clients x 4 sensors (within NFR-015 memory budget). | prism-cache: `ResponseCache` | Eviction is synchronous — inserting a new entry when at capacity evicts the LRU entry before insertion completes. Cache size is configurable per sensor in TOML. Cache metrics (entry count, hit rate, eviction count) are exposed via `check_sensor_health`. |
| DI-019 | **Query Security Limits**: All PrismQL queries are subject to hard security limits before and during execution. Max query length: 64KB (CWE-400). Max nesting depth: 64 (CWE-674). Max pipe stages: 32. Max materialized records per query: 10K. Query timeout: 30 seconds. These limits apply to the expanded query (after alias resolution). The 10K limit applies to pre-DataFusion-filter records (fetched from sensor APIs after push-down). The `explain_query` tool estimates record counts so the agent can narrow scope before executing. The limit is configurable via `PRISM_MAX_MATERIALIZED_RECORDS` environment variable for deployments with higher memory budgets. | prism-query: parser security checks + DataFusion execution bounds | Violations return structured errors with the specific limit exceeded, the actual value, and actionable suggestions to narrow the query. Query length is checked before parsing. Nesting depth is tracked via a counter in the recursive Chumsky parser. Pipe stage count is checked after parsing. Materialization limit is enforced during sensor fan-out collection (pre-Arrow conversion). Timeout is enforced via `tokio::time::timeout` wrapping the entire query execution. |
| DI-020 | **Alias Composition Depth and Cycle Detection**: Alias composition is capped at depth 3 (an alias may reference another alias, which may reference a third, but no deeper). Alias cycles are detected at config load time AND when `create_alias` is invoked at runtime, using a visited-set traversal of the alias dependency graph. | prism-config: alias validation at startup | Depth exceeding 3 is a fatal config error with the alias chain that exceeds the limit (e.g., "Alias 'a' -> 'b' -> 'c' -> 'd' exceeds max composition depth of 3"). Cyclic references are a fatal config error with the exact cycle reported (e.g., "Alias cycle detected: 'a' -> 'b' -> 'c' -> 'a'"). Both checks run during config validation, before any queries can execute. |
| DI-021 | **Required Column Constraint Enforcement**: Sensor adapter columns marked REQUIRED must be constrained in the WHERE clause. Queries without required constraints are rejected before any API calls with `E-QUERY-009` and an actionable message listing the required columns. This prevents full-scan of unbounded remote APIs where the sensor API itself requires certain parameters (e.g., a time range or entity ID). The check runs during query planning, after push-down classification and before fan-out. | prism-query: query planner push-down validation | Returns structured error `E-QUERY-009` with `category: "validation"`, `sensor` (which sensor requires the column), `required_columns` (list of column names marked REQUIRED for that sensor), and `suggestion` (example WHERE clause showing how to constrain the required columns). The query is never dispatched to the sensor API. |
| DI-022 | **Schedule Splay Distribution**: Concurrent API calls from scheduled queries are distributed by computing a percentage-based splay offset: `splay_offset = (interval * splay_percent / 100) * hash(client_id, schedule_name) / MAX_HASH` for each `(query_name, client_id)` pair. This ensures that scheduled queries targeting multiple clients do not all fire simultaneously (thundering herd). The splay offset is deterministic (same client/schedule pair always gets the same offset) and persisted to RocksDB so it is stable across restarts. The `splay_percent` defaults to 10% of the interval (matching osquery's `schedule_splay_percent`) and is configurable in TOML. The hash input includes both `client_id` and `schedule_name` to ensure different schedules for the same client get different offsets. | prism-scheduler: `Schedule::compute_splay()` (enforced by BC-2.12.001, BC-2.12.004) | If all scheduled queries for a given interval fire at the same instant (zero effective splay), the API rate limits of upstream sensors may be exceeded, causing cascading `429 Too Many Requests` errors. Splay is applied additively to `next_run`: `next_run = last_run + interval + splay_offset`. A schedule with `interval: 300s`, `splay_percent: 10`, and 50 clients produces splay offsets distributed across a 30-second window. |
| DI-023 | **Differential Exactly-Once Semantics**: The `(epoch, counter)` pair on each DiffState record ensures that each differential computation is performed exactly once per scheduled query execution. The epoch increments on process restart; the counter increments per execution within an epoch. Before computing a diff, the scheduler checks that the DiffState's `(epoch, counter)` matches the expected value. A mismatch (e.g., after a restart) triggers a full diff rather than a skip, preventing missed changes. Duplicate counter values (same epoch, same counter) are rejected — the diff is not recomputed. | prism-scheduler + prism-storage: DiffState epoch/counter validation | Epoch/counter mismatch on a DiffState read is not an error — it indicates a restart occurred and triggers a full diff (treating all current results as "added"). This ensures no changes are silently missed. Downstream consumers (detection rules, notifications) use the epoch/counter to deduplicate — they track the last-processed `(epoch, counter)` and skip already-seen diffs. |
| DI-024 | **Detection Rule Validation at Load Time**: All detection rules (from `.detect` files, TOML config, and runtime MCP tool creation) are validated before they are activated. Validation checks: (1) meta block contains required `name` and `severity` fields, (2) alert template contains required `title` and `description` fields, (3) sequence rules have at least one step, (4) regex patterns compile successfully with size limit enforcement via `RegexBuilder::size_limit(1MB)`, (5) CIDR literals parse as valid CIDR ranges, (6) rule size does not exceed 16KB, (7) condition nesting depth does not exceed 16 (intentionally lower than the query engine's 64-level limit in DI-019, since detection rules are simpler expressions that should not require deep nesting). Invalid rules are rejected with a multi-error report listing every validation failure (same pattern as config validation). | prism-detection: `DetectionRule::validate()` at load time and runtime creation | Invalid rules from config files cause a startup error (aggregated with other config errors). Invalid rules from runtime MCP tool creation return a structured error `E-RULE-001` with `category: "validation"`, `failures` (list of specific validation errors with field paths), and `suggestion` (corrective guidance per failure). The invalid rule is never activated or stored. |
| DI-025 | **Case State Transition Validity**: The Case entity's 5-state lifecycle (New → Acknowledged → Investigating → Resolved → Closed) permits exactly 12 valid transitions: forward linear (4), skip-ahead (6), and reopen (2: Resolved → Investigating, Closed → Investigating). All other transitions are invalid. Invalid transitions include: self-transitions (e.g., New → New), backward to New (from any state), and backward to Acknowledged (from Investigating, Resolved, or Closed). The `CaseStatus::can_transition_to(target)` method is an exhaustive match over all `(current, target)` pairs. | prism-cases: `CaseStatus::can_transition_to()` | Invalid transitions return `E-CASE-004` with `category: "validation"`, `current_status`, `requested_status`, `valid_transitions` (list of statuses reachable from the current status), and `suggestion` (e.g., "To modify a Resolved case, first reopen it by transitioning to Investigating"). The case is not modified. |
| DI-026 | **Audit Buffer Durability**: Audit entries destined for external systems are written to RocksDB (CAP-019, Audit domain) **before** any delivery attempt to the external system. This ensures that audit records survive process crashes and external system outages. The RocksDB write uses `WriteOptions { sync: true }` for the Audit domain (WAL flush to disk before acknowledging the write). Entries remain in the buffer until positive delivery confirmation is received. Buffer overflow (exceeding 100K entries) triggers purge of oldest entries with a warning — but the entries were already durably stored and are being purged only to prevent unbounded growth. | prism-audit + prism-storage: audit buffer write path | If RocksDB itself fails to write (disk full, I/O error), the audit entry is still emitted to the local `tracing` subscriber (stderr). The write-operation fail-closed invariant (DI-004/DI-016) applies to the local tracing emission, not the RocksDB buffer write. A RocksDB write failure for the audit buffer is logged as a warning but does not block query execution — it degrades the external delivery guarantee, not the local audit trail. |
| DI-028 | **Maximum Schedule and Rule Counts**: The system enforces configurable upper bounds on the total number of active schedules and detection rules. Default limits: max 500 schedules (`max_schedules`, configurable via `[defaults.limits]` in TOML), max 1000 rules (`max_rules`, configurable via `[defaults.limits]` in TOML). Attempts to create beyond these limits are rejected with structured errors (`E-SCHED-008`, `E-RULE-011`). These limits prevent unbounded growth of background work (each schedule produces periodic API calls; each rule produces per-record evaluation overhead). | prism-scheduler: `create_schedule` cap check, prism-detection: `create_rule` cap check (enforced by BC-2.12.001 create_schedule cap check, BC-2.13.006 create_rule cap check) | Schedule cap exceeded returns `E-SCHED-008` with `current_count`, `max_count`, and suggestion to delete unused schedules. Rule cap exceeded returns `E-RULE-011` with `current_count`, `max_count`, and suggestion to delete unused rules. Both limits are evaluated against active (non-deleted) entities only. |
| DI-029 | **Correlation Window >= Schedule Interval**: Correlation and sequence rule windows SHOULD be >= the schedule interval that feeds them. A warning is emitted at config load time if window < interval. The warning includes the recommended interval value (e.g., "Correlation rule 'brute_force' has a 60s window but is fed by schedule 'auth_monitor' with 300s interval — detections may be missed between runs. Recommended: set window >= 300s or interval <= 60s."). This is a SHOULD (warning), not a MUST (error), because some use cases intentionally use narrow windows with frequent schedules. | prism-config: schedule/rule cross-validation at startup (enforced by BC-2.06.005 config validation at startup) | Warning logged at startup; no runtime enforcement. The warning is informational — the schedule and rule are both activated regardless. |
| DI-030 | **Sensor Spec Validation at Load/Reload Time**: All sensor spec files (CAP-029) are validated at load time (startup) and reload time (BC-2.16.005). Invalid specs are rejected with actionable errors listing the exact TOML path that failed validation (e.g., `sensor.tables[0].steps[1].path_template`). A single invalid spec does NOT prevent other valid specs from loading — each spec file is validated independently. Validation covers: schema structure (required fields, valid types), variable reference resolution (no forward references, no undefined step names — DEC-038), OCSF field path validation (warnings for invalid paths, not errors), pagination config consistency, and rate limit hint validity. Multi-error reporting aggregates all validation failures per spec file in a single pass (same pattern as config validation BC-2.06.005). | prism-spec-engine: `SensorSpec::validate()` at load time, `reload_config` handler, `add_sensor_spec` handler | Invalid specs from files produce a startup warning (not a fatal error) listing each rejected spec with its validation errors. Invalid specs from `add_sensor_spec` MCP tool return `E-SPEC-001` with `category: "validation"`, `failures` (list of specific errors with TOML paths), and `suggestion` (corrective guidance per failure). Valid specs within the same reload are loaded normally regardless of invalid siblings. |
| DI-031 | **Hot Reload Atomicity (Three-Tier)**: Config reload via `reload_config` (BC-2.16.005) has three distinct atomicity tiers. **Tier 1 — prism.toml: all-or-nothing.** The entire config structure (clients, defaults, limits, feature flags) is validated as a unit. If any field fails validation, the entire reload is rejected and the previous `ConfigSnapshot` is retained in `arc_swap::ArcSwap`. **Tier 2 — aliases.toml: all-or-nothing.** Alias definitions (global + per-client) are validated as a unit including cycle detection (DI-020) and depth checks. If any alias fails, all alias changes are rejected. **Tier 3 — sensor spec files (*.sensor.toml): per-file independent.** Each spec file is validated independently (DI-030). Valid specs are loaded; invalid specs are rejected with warnings. A single invalid spec does NOT prevent other valid specs from loading. This allows operators to fix one broken spec without blocking the rest. The `reload_config` response reports per-tier results: which tiers succeeded, which failed, and per-file results for Tier 3. No partial state within a tier is ever applied. In-flight queries continue using their captured snapshot reference regardless of reload outcome (BC-2.16.006). | prism-config: `reload_config` handler, `ConfigSnapshot` construction | Failed Tier 1/2 reload returns `E-RELOAD-002` with the full validation error list. Failed Tier 3 specs return `E-RELOAD-003` per rejected file (other specs still load). No `notifications/tools/list_changed` is sent unless at least one tier changed. The previous `ConfigSnapshot` is retained for any tier that fails. |
| DI-032 | **Maximum Concurrent Schedule Executions**: At most 16 scheduled query executions may run concurrently (configurable via `PRISM_MAX_CONCURRENT_SCHEDULES` env var or `[defaults.limits]` TOML). Excess executions are skipped with E-SCHED-004 and retried at the next tick. This prevents API fan-out amplification (16 schedules × 10 concurrent sensor calls = 160 HTTP connections max). | prism-scheduler: semaphore-gated execution loop (enforced by BC-2.12.004 schedule execution loop) | Excess schedules are skipped (not queued) and logged at WARN level. The skipped execution's epoch/counter is NOT incremented. |
| DI-027 | **Resource Watchdog Enforcement**: Two distinct memory guards protect Prism. **(1) Per-query memory estimation:** Each query's memory usage is estimated from RecordBatch sizes (sum of batch byte sizes during materialization and DataFusion execution). Queries exceeding the active watchdog level's per-query memory budget are terminated via `tokio::task::abort()`. This is an estimation, not an RSS measurement -- it tracks the data the query is processing, not the full process footprint. **(2) Process-level RSS guard:** A separate monitor checks process RSS via platform-appropriate APIs (Linux, macOS, Windows) on a 3-second interval. If RSS exceeds 512MB (configurable via `watchdog_process_rss_limit_mb`), the process sends SIGTERM to self, triggering the graceful shutdown path (BC-2.10.010). If the 5-second grace period expires, forces exit with code 2. This is a last-resort safety net, not a per-query mechanism. Time limit violations are enforced via `tokio::time::timeout()`. Terminated queries return a structured error with partial result indication (DEC-033). Queries that cause process crashes (detected via CrashRecovery markers in RocksDB) or time out on 3 consecutive executions are denylisted for 86400 seconds. The denylist is persisted to RocksDB and survives restarts. Denylisted queries are rejected before execution begins. | prism-watchdog: `ResourceWatchdog::check()` during query execution | Per-query memory estimation checks run after each RecordBatch is materialized and after DataFusion execution completes. A single spike above the limit does not immediately kill — the check must observe the limit exceeded on two consecutive checks (grace period) to avoid killing queries that briefly spike during Arrow materialization. Process-level RSS checks run on a 3-second interval (matching osquery's watchdog interval). Timeout enforcement is immediate via `tokio::time::timeout()`. Denylist entries include the query hash, denial reason, and expiry timestamp. |

## Wave 3 Supplement — Multi-Tenant Identity Invariants (ADR-006, 2026-04-27)

> **Terminology note:** Wave 3 introduces `OrgId` (UUID v7 newtype) and `OrgSlug`
> (kebab-case string) via ADR-006. These supersede the pre-Wave-3 `TenantId`/`client_id`
> terminology in the invariants below:
>
> | Old term | Wave 3 replacement | Notes |
> |----------|--------------------|-------|
> | `TenantId` | `OrgId` (UUID v7) | Compile-time newtype; immutable after creation |
> | `client_id` | `OrgSlug` | Kebab-case, analyst-facing; mutable via rename operation |
> | `client` | `org` / organization | Terminology shift for multi-tenant clarity |
>
> DI-002 and DI-008 below retain their pre-Wave-3 wording for append-only ID
> stability. The Wave 3 semantic equivalents are: DI-002 "client_id" = `OrgId`;
> DI-008 "client_id" = `OrgSlug`. Enforcers: BC-3.1.001 (resolution), BC-3.1.002
> (audit fields), BC-3.1.003 (bijectivity), BC-3.1.004 (duplicate rejection).

| ID | Rule | Scope | Violation Behavior |
|----|------|-------|--------------------|
| DI-033 | **OrgRegistry Bijectivity**: The `OrgRegistry` maintains a strict bijection between `OrgId` (UUID v7) and `OrgSlug` (kebab-case string) at all times. For every registered `OrgId`, there is exactly one `OrgSlug` that resolves to it, and vice versa. Duplicate `OrgId` or duplicate `OrgSlug` registrations are rejected at `OrgRegistry::register` time. The bijection is preserved across slug rename operations: a rename atomically updates both the forward map (`OrgSlug → OrgId`) and the reverse map (`OrgId → OrgSlug`). No intermediate state is observable by concurrent readers during a rename. This invariant is enforced by BC-3.1.003 and BC-3.1.004. It supersedes the pre-Wave-3 uniqueness constraint implied by DI-002 for `TenantId` values. | prism-core: `OrgRegistry` (per ADR-006 D-047) | Duplicate `OrgId` at `register`: returns `Err(OrgRegistryError::IdConflict)` — registry unchanged. Duplicate `OrgSlug` at `register`: returns `Err(OrgRegistryError::SlugConflict)` — registry unchanged. Rename to an already-used slug: returns `Err(OrgRegistryError::SlugConflict)`. No partial state is applied on error. |
