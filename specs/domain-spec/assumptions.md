# Assumptions Requiring Validation

Each assumption is a belief about the environment, dependencies, or constraints that Prism's design relies on. If an assumption is proven wrong, the impact column describes what must change.

| ID | Assumption | Confidence | Validation Method | Impact if Wrong | Status | Traced To |
|----|-----------|------------|-------------------|-----------------|--------|-----------|
| ASM-001 | **rmcp 0.8 API stability**: The rmcp crate at version 0.8.x provides stable `#[tool_router]`, `Parameters<T>`, `ServerHandler`, and stdio transport APIs that will not undergo breaking changes before Prism reaches production. rmcp is pre-1.0, so SemVer does not guarantee stability. | MEDIUM | Pin exact rmcp version in `Cargo.lock`. Monitor rmcp GitHub releases and changelogs. Run CI against rmcp `main` branch weekly to detect upcoming breakage. Verify `notifications/tools/list_changed` support exists. | If rmcp breaks its API, Prism's entire MCP layer (`prism-mcp`) requires refactoring. Estimated impact: 2-4 weeks depending on scope of changes. Mitigation: thin adapter layer between Prism business logic and rmcp types. | UNVALIDATED | R-001, CAP-009 |
| ASM-002 | **OCSF schema backward compatibility**: OCSF v1.x maintains backward-compatible field semantics between minor versions. Adding new event classes or fields does not alter the meaning of existing fields. Proto field numbers generated by ocsf-proto-gen are NOT stable across OCSF versions (known; Prism pins OCSF version per release). | MEDIUM | Review OCSF changelog for each new version. Diff proto files generated by ocsf-proto-gen between OCSF versions. Test DynamicMessage field access against multiple OCSF schema versions. | If OCSF v1.x introduces breaking field semantic changes, all per-sensor OCSF mappers in `prism-ocsf` must be updated. Cross-sensor correlation logic in the agent layer may produce incorrect joins. Mitigation: pin OCSF version and upgrade deliberately. | UNVALIDATED | R-004, CAP-003 |
| ASM-003 | **keyring-rs cross-platform reliability**: The `keyring-rs` crate provides reliable access to OS-level credential storage on macOS (Keychain), Windows (Credential Vault), and Linux (libsecret/Secret Service) for the credential namespace pattern `(client_id, sensor_id, credential_name)`. | MEDIUM | Integration test on each target platform (macOS, Windows, Linux). Verify keyring-rs handles the credential key format used by Prism. Test behavior when keyring is locked, when libsecret is not installed, and when credential count exceeds typical limits. | If keyring-rs fails on a target platform, that platform falls back to the AES-256-GCM encrypted file backend exclusively. Analyst experience degrades (must manage encryption key externally) but functionality is preserved. If keyring-rs has namespace limits that conflict with Prism's `(client_id, sensor_id, name)` pattern, credential key encoding must be redesigned. | UNVALIDATED | CAP-004, FM-004 |
| ASM-004 | **Sensor API rate limits are sufficient for interactive use**: CrowdStrike, Cyberint, Claroty, and Armis APIs allow enough requests per minute for interactive query patterns (analyst asking questions via AI agent) without hitting rate limits frequently. Interactive use differs from polling: queries are bursty and on-demand rather than periodic. | LOW | Measure actual API rate limits for each sensor via documentation review and empirical testing. Profile a typical analyst session (queries per minute per sensor per client). Compare against documented rate limits. Implement per-sensor rate tracking from day one. | If rate limits are too restrictive for interactive use, Prism must implement aggressive caching (LRU per-client per-sensor), request coalescing (deduplicate identical concurrent queries), and proactive rate budget management. The agent experience degrades with "rate limited, retry in N seconds" errors. Worst case: some sensors may need a background polling mode to pre-fetch data. | UNVALIDATED | DEC-001, FM-001 |
| ASM-005 | **Protobuf field numbering stability within ocsf-proto-gen**: The ocsf-proto-gen library produces deterministic protobuf field numbers for a given OCSF schema version input. Running ocsf-proto-gen twice on the same schema version produces identical `.proto` files. | HIGH | Determinism is architecturally enforced by ocsf-proto-gen's use of `BTreeMap`/`BTreeSet` for iteration order. Verify by running generation twice and diffing output. Add CI check that generated proto files match committed versions. | If field numbers are non-deterministic, protobuf serialized data becomes version-dependent on the build environment, not just the OCSF schema. Proto-encoded data from different builds would be incompatible. Mitigation: commit generated `.proto` files and detect drift in CI. | UNVALIDATED | DI-005, CAP-003 |
| ASM-006 | **Analyst is a trusted MSSP employee**: The human operating Prism is a trusted employee of 1898 & Co with legitimate access to all MSSP client data. Client isolation is a data correctness concern (never accidentally mix client data), not an adversarial security boundary (client A's analyst cannot be prevented from seeing client B's data). | HIGH | Organizational policy confirmation with 1898 & Co management. Review deployment model: per-analyst local process with analyst's own credentials. | If the trust model changes (e.g., client-specific analysts who should only see their assigned clients), Prism needs an access control layer mapping analyst identity to allowed clients. The `TenantId` newtype and per-client config exist but no authorization check gates tool calls today. Estimated impact: add authorization middleware in prism-mcp. | VALIDATED | DI-008, CAP-002 |
| ASM-007 | **stdio transport is sufficient**: Claude Code's MCP integration uses stdio transport exclusively. SSE/Streamable HTTP transport is not needed for the initial deployment. One Prism process per analyst. | HIGH | Confirm with Claude Code documentation and 1898 & Co deployment plan. Test that Claude Code establishes stdio MCP sessions as expected. | If Claude Code or other MCP clients require HTTP transport, prism-mcp must add SSE/Streamable HTTP support (rmcp supports both). The tool logic is transport-agnostic, but session management and concurrent client handling become more complex. | VALIDATED | CAP-009 |
| ASM-008 | **CrowdStrike two-step fetch contract stability**: CrowdStrike's alert query API maintains the QueryV2-then-PostEntities two-step pattern. Alert IDs from QueryV2 can be used with PostEntities to retrieve full details. The 32 alert fields documented in poller-cobra remain valid. | MEDIUM | Verify against current CrowdStrike API documentation. Test with live CrowdStrike sandbox environment. Monitor CrowdStrike API changelog for breaking changes. | If CrowdStrike changes or deprecates the two-step pattern, the CrowdStrike sensor adapter must be rewritten. If alert fields change, OCSF mappings for CrowdStrike must be updated. Estimated impact: 1-2 weeks for the adapter, plus OCSF mapper updates. | UNVALIDATED | CAP-001, R-002 |
| ASM-009 | **Cyberint cookie-based auth remains supported**: Cyberint Argos API continues to accept `access_token` as an HTTP cookie for authentication, as documented in poller-express. | LOW | Verify with current Cyberint API documentation. Test with live Cyberint environment. Cyberint may have migrated to a standard bearer token or OAuth2 since the poller was written. | If Cyberint changes auth mechanism, the `CookieAuth` implementation in prism-sensors must be replaced. If they move to OAuth2 or bearer token, this simplifies the code. If they add a more complex auth flow (e.g., SAML), the adapter becomes more complex. | UNVALIDATED | CAP-001, CAP-004 |
| ASM-010 | **prost-reflect supports runtime DynamicMessage field access**: The `prost-reflect` crate provides the runtime field access capabilities needed for axiathon's DynamicMessage pattern: setting fields by name, getting fields by name, and serializing to protobuf wire format. | HIGH | Prototype the DynamicMessage wrapper against prost-reflect. Verify field access for all OCSF field types (string, int64, double, repeated, nested message, enum). Test with generated OCSF proto descriptors. | If prost-reflect does not support required patterns, an alternative runtime reflection library is needed, or the DynamicMessage pattern must be reimplemented. The most likely issue is performance (reflection vs. codegen) rather than functionality. | UNVALIDATED | CAP-003, ADR-003 |
