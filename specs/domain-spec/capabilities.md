# Domain Capabilities

This section enumerates all domain capabilities in Prism. Capabilities are either MCP-facing (exposed as tools to the agent) or internal (consumed by other subsystems, never exposed directly to MCP clients). Priority reflects implementation order: P0 is required for MVP, P1 for full launch, P2 for post-launch enhancement.

| ID | Capability | Description | Business Rule | Priority |
|----|-----------|-------------|---------------|----------|
| CAP-001 | Sensor Adapter Layer (Internal) | **Internal capability -- not exposed to MCP clients.** The sensor adapter layer provides authenticated, paginated, retried data fetch from each sensor's API. It is consumed exclusively by the query engine (CAP-015), not by MCP clients directly. Per-sensor adapters handle: auth (OAuth2, cookie, bearer), pagination (cursor-based, forward-only), retry with backoff, and raw data fetch. Each adapter call is scoped to a single `(client_id, sensor_id)` pair. The query engine orchestrates fan-out across adapters. Sensor data sources are dynamically registered based on client configuration. If no clients have CrowdStrike credentials configured, CrowdStrike-related sources are not available in the query engine's schema. The `explain_query` tool shows which sources are available per client. | Every adapter call must include a valid `client_id`. Responses are scoped exclusively to that client's sensor instance. Results are paginated internally with cursor-based forward-only traversal. The adapter layer is never invoked directly by MCP tool calls -- all data access flows through the query engine (CAP-015). Table availability is determined at startup from client configuration and updated on config reload. | P0 |
| CAP-002 | Cross-Client Fan-Out (Internal) | **Internal capability -- describes the query engine's cross-client fan-out behavior.** When the query engine (CAP-015) receives a query with `clients: null` or multiple client IDs, it fans out to all configured clients for the specified sensors via the sensor adapter layer (CAP-001). Each result item includes `client_id` provenance. Sensors unavailable for a client are reported as partial failures in the query response's `sensor_errors` array, not as query-level errors. | Cross-client fan-out is orchestrated by the query engine, not exposed as a separate MCP tool. Each result item includes `client_id` provenance. Partial failures are surfaced in `sensor_errors`, not as query-level errors. | P0 |
| CAP-003 | OCSF Normalization | Normalize all sensor records to OCSF v1.x via the DynamicMessage protobuf pattern. Enables cross-sensor correlation by mapping vendor-specific fields to a common schema (e.g., CrowdStrike `hostname` and Claroty `device_name` both map to OCSF `device.hostname`). | Every sensor record returned via MCP tools includes an `ocsf` field containing the normalized OCSF representation. Unmappable vendor-specific fields are preserved in a `raw_extensions` JSON blob. OCSF schema version is fixed per Prism release. | P0 |
| CAP-004 | Credential Management | Store, retrieve, rotate, and delete per-client per-sensor credentials via OS keyring (macOS Keychain, Windows Credential Vault, Linux libsecret) with AES-256-GCM encrypted file fallback. Namespace: `(client_id, sensor_id, credential_name)`. Credential mutation operations (`set_credential`, `delete_credential`) are gated by the `credential.write` capability path and follow the hidden-tools pattern (BC-2.04.005). `list_credentials` is read-only and always visible. | Credentials are isolated by client -- accessing Client A's CrowdStrike credentials while operating in Client B's context is a type error. Credential values are never included in logs, MCP responses, or error messages. All credential access is audit-logged. Credential mutations require `credential.write` capability to be allowed for the target client. | P0 |
| CAP-005 | Feature Flag Evaluation | Evaluate two-tier feature flags (compile-time cargo features + runtime per-client TOML config) to determine which write operations are available for each client. Runtime config uses `BTreeMap<String, Effect>` where Effect is Allow or Deny. Hierarchical resolution: most-specific-path wins (e.g., `sensor.crowdstrike.containment = deny` overrides `sensor.crowdstrike = allow`). Deny beats allow at the same specificity level. Implicit deny is the final fallback. | Deny-by-default: if no flag explicitly enables a capability, it is denied. Both tiers (compile-time and runtime) must permit the operation. Most-specific capability path wins during resolution; at equal specificity, deny overrides allow. Disabled tools are hidden from `tools/list`. A `list_capabilities` meta-tool exposes the full capability matrix per client with an `explain()` trace showing which rule determined each outcome. | P0 |
| CAP-006 | Write Operation Gating | Gate all write/mutation operations (host containment, alert acknowledgment, device actions) behind a three-tier risk classification: read (no gate), reversible write (dry-run default with preview), irreversible write (confirmation token with 300s expiry). | Reversible writes default to `dry_run: true`; the agent must explicitly set `dry_run: false`. Irreversible writes require a two-step flow: request returns a `ConfirmationToken`, then `confirm_action(token)` executes. Tokens are single-use, time-bounded, and include an action-specific content hash. | P1 |
| CAP-007 | Audit Logging | Log every MCP tool invocation with: client_id, sensor, tool name, parameters (secrets redacted), user identity, timestamp, feature flags evaluated, result summary, and safety flags triggered. Structured JSON via `tracing` crate, compatible with Vector pipeline. | Audit entries are append-only and include sufficient detail for SOC 2 Type II and ISO 27001 compliance. Write operations log both the capability check result and the execution outcome. Credential access events are logged with client/sensor context but never credential values. | P0 |
| CAP-008 | Sensor Health Monitoring | Expose per-sensor, per-client connectivity status via MCP resources and a dedicated health tool. Detect unreachable APIs, expired credentials, and rate limit exhaustion. | Health checks run on-demand (not polled). Each sensor adapter implements `verify_connectivity()`. Health status includes: reachable/unreachable, auth valid/expired, last successful query timestamp, and current rate limit state. | P1 |
| CAP-009 | Client Configuration | Load and validate per-client sensor mappings, credential references, and capability overrides from TOML configuration. Support `--dry-run` validation that prints redacted config and exits. Re-evaluate capabilities on config reload with `notifications/tools/list_changed` when the resolved tool set changes. | Client structure changes (add/remove clients) require server restart. Feature flags, credentials, sensor specs, and aliases are hot-reloadable via CAP-030 (`reload_config` MCP tool). Validation is multi-error: all problems reported in one pass. Missing required fields produce actionable error messages specifying the exact config path. | P0 |
| CAP-010 | Prompt Injection Defense | Sanitize all sensor data flowing through MCP responses to defend against attacker-controlled content (hostnames, file paths, process names) that could manipulate the LLM. Four-layer defense: structural separation, provenance framing, suspicious pattern flagging, trust-level metadata. | Untrusted sensor data is placed in `structuredContent` JSON fields, never interpolated into prose. Tool descriptions include security warnings about untrusted data. Suspicious patterns (e.g., strings containing "ignore previous instructions") are flagged in parallel `_safety_flag` fields but never stripped -- the analyst needs full forensic data. | P0 |
| CAP-011 | Adapter Pagination (Internal) | **Internal capability -- cursors are never exposed to MCP clients.** Provides ephemeral per-query pagination between Prism's sensor adapter layer and the upstream sensor APIs. Cursors are created when a paginated adapter fetch begins, used to retrieve subsequent pages from the sensor API, and discarded when the fetch completes. The MCP agent never sees cursor tokens -- instead, query results include `limit` and `total_available` fields so the agent knows how many results exist and can adjust query scope. No disk persistence of cursors. | Cursors are ephemeral pagination tokens internal to the sensor adapter layer, scoped to a single adapter fetch invocation. Each cursor encodes enough state to fetch the next page from the sensor API. Cursors are not persisted to disk and do not survive server restart. Cursor tokens are never included in MCP responses -- the agent sees `limit` (how many results were returned) and `total_available` (how many matched). The agent narrows scope or adjusts `LIMIT` in PrismQL to control result size. | P0 |
| CAP-012 | Cross-Sensor Correlation (via Query Engine) | A natural consequence of the query engine's (CAP-015) unified OCSF table model. When the query engine fans out to multiple sensors, all results are normalized to OCSF and materialized into a single Arrow table. The agent correlates across sensors simply by querying OCSF fields (`device.ip`, `device.hostname`, `time`) in PrismQL -- no explicit JOINs or separate correlation step required. This is not a separate subsystem; it emerges from the query engine's single-table design combined with OCSF normalization (CAP-003). | Correlation happens implicitly within the query engine's materialized table. The agent writes a single PrismQL query that spans sensors; the query engine handles fan-out, normalization, and materialization. No separate correlation API or tool exists. | P1 |
| ~~CAP-013~~ | ~~xMP Envelope Delivery~~ | **REMOVED** — backward compatibility with xMP envelope format is not required. Prism is a new system; it does not need to produce xMP-enriched payloads for the existing Vector pipeline. | N/A | N/A |
| CAP-014 | Response Caching | LRU cache per-client per-sensor with configurable TTL (60s for alerts, 300s for devices). Bounded entry count (configurable, default 50 per client per sensor). Cache invalidation is synchronous with write operations — any write that mutates sensor state invalidates related cache entries before the write confirmation is returned. Agents may bypass cache with `force_refresh: true` parameter. Cache metrics (hit rate, eviction count, entry count) are exposed via the `check_sensor_health` tool. | Cache keys are `(client_id, sensor_id, query_hash)`. TTL is per-data-source-category: alert-class sources default to 60s, device/asset-class sources default to 300s, configurable in TOML. LRU eviction when entry count exceeds the per-client-per-sensor bound. Write operations invalidate all cache entries for the affected `(client_id, sensor_id, source_id)` tuple synchronously before returning confirmation. `force_refresh: true` bypasses cache and populates a fresh entry. Cache is in-memory only — lost on server restart. | P1 |
| CAP-015 | Ephemeral OCSF Query Engine | The core of Prism's ephemeral federated query engine architecture. The primary query MCP tool (`query`) turns what looks like a database query into orchestrated live API calls: Query -> Fetch -> Normalize -> Compute -> Return. There is no database -- the data lake is conjured on demand and exists only for the duration of the query. Accepts `clients`/`sensors`/`sources` scoping parameters plus an PrismQL query string. Fan-out to configured sensor APIs (parallel, per-client per-sensor), normalize all responses to OCSF via DynamicMessage, materialize an ephemeral Arrow RecordBatch, execute via DataFusion over an in-memory MemTable, return results, tear down the table. Three query modes: filter (`field = value AND field >= value`), SQL (`SELECT ... FROM events WHERE ... GROUP BY ... ORDER BY ... LIMIT`), and pipe (`FROM source \| where ... \| sort ... \| head N`). Auto-detects mode from first token. Cross-sensor correlation via unified OCSF table — implicit single-table model, no explicit JOINs. Virtual fields `sensor`, `client_id`, `source` are filterable within queries. Tool parameters set scope; query predicates narrow within that scope (intersection semantics, never widening). Includes `explain_query` tool for dry-run inspection of alias expansion, field resolution, and planned API calls. Powered by Chumsky 0.10 parser + DataFusion execution (hybrid architecture). The query engine supports custom functions for the security domain, registered as DataFusion UDFs: `subnet_contains(cidr, ip)` (checks if an IP address falls within a CIDR range), `time_window(field, duration)` (creates a time-bounded filter relative to now), `ioc_match(field, pattern)` (matches indicators of compromise against configurable pattern sets). Additional security UDFs may be added in future releases. | Every query execution creates a fresh DataFusion `SessionContext`, registers materialized data as a `MemTable`, executes the query, and drops the context when complete. Security limits enforced: 64KB max query length, 64 max nesting depth, 32 max pipe stages, 10K max materialized records, 30s query timeout. Violations return structured errors with actionable suggestions. Push-down filters are translated to sensor API parameters where possible; remaining predicates applied post-normalization via DataFusion. Results include `query_context` echo-back (original query, expanded query after alias resolution, clients/sensors queried, execution time) and `sensor_errors` for partial failure transparency. | P0 |
| CAP-016 | Query Aliases | Reusable, parameterized query shortcuts persisted in a dedicated `aliases.toml` file (separate from the main `prism.toml`). `aliases.toml` is written atomically (temp + fsync + rename) by `create_alias` and `delete_alias`, and loaded at startup alongside main config. Global aliases defined in `[aliases]` section; per-client aliases in `[clients.{id}.aliases]` section. Per-client aliases override global aliases of the same name. Aliases are composable (one alias may reference another, max depth 3). Parameterized aliases use `template` + `defaults` syntax; all parameters must have defaults. Alias resolution occurs pre-parse (before Chumsky parser). Expanded query must still pass security limits. MCP tools: `create_alias` (create/update global or per-client alias with optional parameters), `list_aliases` (list all aliases visible to a client — global + per-client merged), `delete_alias` (delete an alias, requires confirmation if referenced by other aliases), `explain_alias` (show definition, parameters, expanded query, which clients define it). Alias mutation tools (`create_alias`, `delete_alias`) are gated by the `alias.write` capability path and follow the hidden-tools pattern (BC-2.04.005). `list_aliases` and `explain_alias` are read-only and always visible. | Alias composition depth capped at 3. Cycles detected at config load time — cyclic alias references are a fatal config error. Cross-client queries with per-client aliases fail with structured error listing which clients define and which are missing the alias. `create_alias` and `delete_alias` are write operations gated by `alias.write` capability; `create_alias` additionally follows write-operation gating (confirmation token) for updates to existing aliases. Alias expansion is always shown in query responses for AI transparency. | P1 |
| CAP-017 | Scheduled Queries | Run PrismQL queries on configurable intervals per client. The scheduler operates on a tick-based loop where each scheduled query has an `interval` (Duration) and a `splay_offset` computed from `(interval * splay_percent / 100) * hash(client_id, schedule_name) / MAX_HASH` to distribute API load across clients (inspired by osquery's `splayValue()` with persistent splay caching, using percentage-based splay for flexibility). Schedule state is persisted to RocksDB (CAP-019) under the Schedules domain: last_run timestamp, next_run timestamp, epoch (incremented on restart), and per-query counter (incremented per execution). Supports per-client and per-query intervals. Schedules can be defined globally or scoped to specific clients. Each scheduled execution flows through the standard query engine (CAP-015) pipeline. The scheduler skips execution if the previous run for the same (query, client) pair is still in-flight (DEC-028). Time drift is compensated by adjusting future pause durations, dropping accumulated drift beyond 60 seconds. | Every scheduled query execution produces a standard query result plus differential output (CAP-018). Splay offsets are deterministic per (query_name, client_id) pair and persist across restarts via RocksDB. The epoch/counter pair provides exactly-once semantics for downstream consumers (DI-023). Schedule state is restored on startup from RocksDB. Schedules are defined in TOML config or via pack definitions (CAP-023). | P0 |
| CAP-018 | Differential Results | Store the hash of last query results per `(query_name, client_id)` pair in RocksDB (CAP-019) under the DiffState domain. On each scheduled run, compute a hash of the current result set and compare against the stored hash. If changed, perform a full diff to identify added and removed records. Return only the delta (added/removed) to downstream consumers (detection rules, notifications, audit log). Hash-based change detection avoids expensive row-by-row comparison when results are unchanged. The diff state tracks: `previous_results_hash` (SHA-256 of serialized results), `previous_results` (serialized Arrow RecordBatch for row-level diff), `epoch` (matches scheduler epoch), and `counter` (matches per-query execution counter). Epoch/counter ensures each diff is computed exactly once per scheduled run (DI-023). Large diffs (10K+ new records) are truncated with a notification to the analyst (DEC-029). | Differential results are the primary output of scheduled queries. Unchanged results produce no output (silent skip). The diff algorithm operates on serialized row hashes for O(n) comparison. Previous results are bounded by the same 10K materialization limit (DI-019). Diff state is purged when a schedule is removed from config. | P0 |
| CAP-019 | Persistent Storage | RocksDB-backed persistence layer providing durable key-value storage organized by logical domains. Inspired by osquery's domain-based RocksDB organization where different subsystems (events, configuration, carves, etc.) use distinct key prefixes. Prism defines twelve storage domains via the `StorageDomain` enum: Default (general-purpose key-value storage), Schedules (scheduled query definitions, splay offsets, epoch counters, timing state), DiffResults (differential result history), DetectionRules (detection rule storage by scope), DetectionState (correlation windows, sequence trackers), Alerts (persisted alert records and indexes), Cases (case lifecycle, timeline, indexes), AuditBuffer (buffered audit log entries awaiting forwarding), DirtyBits (crash recovery markers), Watchdog (query denylist entries and watchdog state), Aliases (alias splay offsets and metadata), Decorators (periodic decorator cache values). Each domain uses a distinct key prefix for isolation. Keys are structured as `{domain}:{entity_id}:{sub_key}`. Values are serialized via bincode for compact binary encoding. RocksDB column families are used per domain for performance isolation. | RocksDB is opened once at startup with a configurable `state_dir` path (default `./state`, set via `--state-dir` CLI flag). Write-ahead log (WAL) is enabled for crash safety. Domain isolation ensures that a bulk purge of audit entries does not affect schedule state. Backup is supported via RocksDB's `BackupEngine`. The storage layer exposes a `StorageBackend` trait for testability — tests use an in-memory `BTreeMap` implementation. Single-process invariant (DI-017) applies; concurrent RocksDB access from multiple Prism processes is unsupported. | P0 |
| CAP-020 | Detection Rules | PrismQL-based detection rules with three match modes, inspired by axiathon's .axd DSL and adapted for Prism's scheduled (non-streaming) model. **Single-event mode:** stateless field comparisons evaluated per-record against differential results only (new records from CAP-018). **Correlation mode:** threshold-over-time-window with group-by; new records from the differential are added to the persisted sliding window state, and the threshold is evaluated over the full window. **Sequence mode:** ordered multi-event patterns with a shared key field and time window; new records from the differential advance the persisted sequence tracker state. Three rule scopes: **global** (MSSP baseline rules applied to all clients), **per-client** (overrides or additions for specific clients), **analyst-defined** (ad-hoc rules created at runtime via MCP tool). Rules are validated at load time (DI-024): meta block must contain name and severity, alert block must contain title and description, sequence must have at least one step, regex patterns are compiled and validated, field paths are optionally validated against the OCSF schema. Security limits enforced: max rule size 16KB, max nesting depth 16 (DI-024), regex size limit via `RegexBuilder::size_limit(1MB)`. | Detection rules are defined in `.axd` files in a rules directory, in TOML config per-client, or created at runtime via `create_detection_rule` MCP tool. Rules are loaded and validated at startup; invalid rules are rejected with actionable error messages listing every validation failure. Per-client rules override global rules of the same `rule_id`. The `list_detection_rules` tool shows all active rules per client with scope provenance. Detection evaluation runs after each scheduled query execution (CAP-017), processing the differential results (CAP-018) through all matching rules. Input source per match mode: single-event rules evaluate differential results only (new records); correlation and sequence rules use their persisted state augmented by new differential records. | P0 |
| CAP-021 | Alert Generation | When a detection rule (CAP-020) fires, an alert is generated with a structured payload: `alert_id` (UUID v7, time-sortable), `rule_id`, `client_id`, `severity` (inherited from rule), `message` (rendered from the rule's alert template with variable interpolation), and `matched_events` (event UIDs or row hashes that triggered the rule). Template interpolation supports four resolution levels: extra variables (correlation count, window), step-scoped variables (sequence step fields/counts), event field variables (resolved via OCSF field paths), and literal fallback for unresolved variables. Alerts are persisted to RocksDB (CAP-019) under the Alerts domain. Alerts are surfaced to the MCP client via `notifications/resources/updated` on the `alerts://` resource URI. MCP tools: `list_alerts` (paginated, filterable by client/severity/rule/time), `acknowledge_alert` (mark as seen), `get_alert` (full detail with matched events). | Every alert is persisted before notification. Alert deduplication key varies by match mode: **single-event** dedup key is `(rule_id, event_uid)` — the same event cannot trigger the same rule twice; **correlation** dedup key is `(rule_id, group_by_value_hash, window_bucket)` — duplicate alerts within the same correlation window are suppressed; **sequence** dedup key is `(rule_id, sequence_completion_hash)` — the same completed sequence cannot fire twice. Alert severity is immutable after creation (inherited from the rule at fire time). Alerts reference their source detection rule and matched events for full traceability. | P0 |
| CAP-022 | Case Management | Investigation tracking with a 5-state lifecycle state machine: **New** (alert auto-linked, case created) → **Acknowledged** (analyst has seen it) → **Investigating** (active work) → **Resolved** (root cause determined) → **Closed** (post-incident review complete). Forward transitions, skip-ahead (e.g., New → Resolved), and reopen (Resolved/Closed → Investigating) are supported — 12 valid transitions total (DI-025). Invalid transitions (self-transition, backward to New/Acknowledged) are rejected with structured errors. Disposition tags classify the outcome: `TruePositive { impact_level }`, `FalsePositive { reason }`, `Benign { explanation }`, `Inconclusive`. Disposition can be set at any status. Timeline annotations (note, status_change, alert_link, evidence_link, ot_impact) with author and timestamp provide investigation narrative. Auto-computed metrics: MTTD (case created_at minus earliest linked alert created_at) and MTTR (resolved_at minus created_at). Cases are persisted to RocksDB (CAP-019) under the Cases domain. MCP tools: `create_case`, `update_case_status`, `set_disposition`, `add_annotation`, `link_alert_to_case`, `list_cases`, `get_case`, `case_metrics`. | Cases are scoped by `client_id` — cross-client case access is prevented by the same `TenantId` typing as other entities. Every case mutation generates a `TimelineEntry` (auto-audit trail). Cases can be auto-created when high-severity rules fire (configurable per rule). **Note:** Auto-case-creation requires a dedicated BC (not yet written — tracked for story decomposition). Key design questions: severity threshold, deduplication strategy (new case per alert vs aggregate), capability gate bypass for system-generated cases, auto-title generation. The `case_metrics` tool computes MTTD, MTTR, counts by status/priority, and open/closed ratios per client or cross-client for MSSP dashboards. | P0 |
| CAP-023 | Query Packs | Named bundles of related scheduled queries (CAP-017), detection rules (CAP-020), and aliases (CAP-016) for specific MSSP workflows. Inspired by osquery's pack system with discovery queries and per-host sharding. Each pack has: `name`, `description`, `queries` (Vec of schedule references), `detections` (Vec of rule references), `discovery_query` (optional PrismQL query that must return >= 1 row for the pack to be active for a given client — DEC-034), `enabled` flag, and `per_client_overrides` (enable/disable specific queries or rules per client). Built-in packs: **incident-response** (recent detections, quarantined hosts, lateral movement indicators — runs every 5 minutes), **daily-triage** (overnight alert summary, new asset inventory, credential changes — runs every 24 hours), **compliance** (policy violations, configuration drift, audit trail gaps — runs every 12 hours). Packs are defined in TOML config as `[packs.{name}]` sections or as external TOML files referenced by path. Per-client pack assignment via `[clients.{id}.packs]`. MCP tools: `list_packs`, `explain_pack` (show contents, discovery status, client assignments). | Discovery queries are cached for 3600 seconds per (pack, client) pair to avoid redundant evaluation. Pack execution order: discovery check → query execution → detection evaluation. Queries within a pack are sorted by priority (if specified) then by name. Packs compose with but do not replace individual scheduled queries — a query can exist both standalone and within a pack. | P0 |
| CAP-024 | Resource Watchdog | Memory and CPU time limits per query execution, adapted from osquery's watcher-worker model for Prism's single-process architecture. Three graduated limit levels: **normal** (200MB per-query memory / 30s timeout — default), **restrictive** (100MB per-query memory / 15s timeout — for resource-constrained environments), **permissive** (512MB per-query memory / 120s timeout — for large MSSP deployments with higher memory budgets). The active level is configurable in TOML via `watchdog_level` (normal/restrictive/permissive) with per-limit overrides via `watchdog_memory_limit_mb` and `watchdog_query_timeout_s`. Per-query resource tracking: before and after each query execution, the watchdog snapshots process RSS and records the delta. Query denylisting: if a query causes a crash (detected via CrashRecovery storage domain in CAP-019 — the executing query marker persists if not cleaned up) or times out on 3 consecutive runs, it is denylisted for 86400 seconds (1 day). Denylisted queries are skipped with a warning in audit logs. The denylist is persisted to RocksDB. MCP tools: `watchdog_status` (current limits, denylisted queries, per-query resource history). | Watchdog limits apply to the entire query lifecycle: fan-out, normalization, materialization, and DataFusion execution. Memory checks use `procinfo` (Linux) or `proc_pidinfo` (macOS) for process RSS. Timeout is enforced via `tokio::time::timeout`. The 10K materialization limit (DI-019) is a separate, earlier check — the watchdog enforces broader resource bounds. Denylisted queries return `E-WATCHDOG-001` with the reason (crash/timeout), denylist expiry time, and the original query for debugging. | P0 |
| CAP-025 | Buffered Audit Logging | Extension of audit logging (CAP-007) with RocksDB-backed durability for audit entries destined for external systems (Vector pipeline, SIEM). Audit entries are written to RocksDB (CAP-019) under the Audit domain **before** being forwarded to external systems. A background forwarder thread wakes periodically, scans buffered entries, and attempts delivery. On delivery failure: exponential backoff with 2-second base and 60-second maximum delay. On delivery success: entries are deleted from the buffer. Configurable buffer maximum: 100K entries (default). When the buffer exceeds the maximum, the oldest entries are purged with a warning logged to stderr (DI-026 ensures entries reach RocksDB first — overflow purge is a last resort, not a data loss path). Inspired by osquery's `BufferedLogForwarder` pattern with RocksDB-backed buffering and configurable purge limits. | Audit entries are always written to the local `tracing` subscriber (stdout/stderr) synchronously AND to the RocksDB buffer. The buffer is specifically for reliable delivery to external systems that may be temporarily unavailable. The fail-closed behavior for writes (DI-004/DI-016) applies to the synchronous local emission, not the buffered external delivery. Buffer metrics (entry count, oldest entry age, delivery failure count, last successful delivery) are exposed via `watchdog_status`. | P0 |
| CAP-026 | Context Decorators | Auto-inject metadata into every query result and alert, inspired by osquery's decorator system adapted for Prism's deterministic MSSP context. Three injection phases: **config-time** (static values loaded from TOML at startup — `client_id`, `client_display_name`, `prism_version`), **query-time** (per-invocation values injected during result materialization — `analyst_id`, `query_name`, `pack_name`, `sensor_type`, `timestamp`), **periodic** (refreshed on a configurable interval — `sensor_connectivity_status`, refreshed every 300s by default). Unlike osquery's SQL-query-based decorators, Prism's decorations are deterministic metadata derived from config and session context — no SQL execution required. Decorations are injected as top-level fields in every query result row and in every alert payload. The decorator set is fixed per Prism release (not user-configurable) to ensure consistent MSSP audit trails. | Decorations are injected POST-DataFusion — after DataFusion execution completes, during response serialization. Decorators are NOT queryable within PrismQL. Pre-DataFusion queryability for `client_id`, `sensor`, and `source` is provided by virtual fields (CAP-015, BC-2.11.012), which are injected into the Arrow schema before DataFusion execution. Periodic decorators use a shared cache with configurable refresh interval; stale values are used if refresh fails (with a warning). Decorator field names are reserved — user-defined aliases or OCSF fields cannot shadow them. | P0 |
| CAP-027 | Security Domain UDFs | DataFusion custom scalar functions registered in the query engine's `SessionContext` for security-specific operations. Extends the existing UDF set (CAP-015 mentions `subnet_contains`, `time_window`, `ioc_match`) with additional security domain functions: **`subnet_contains(cidr, ip)`** — checks if an IP address falls within a CIDR range (uses `ipnet` crate), returns bool. **`ioc_match(field, pattern_list)`** — matches a field value against a list of IOC patterns (IP, domain, hash), returns bool; pattern lists are loaded from configurable IOC files and cached in memory. **`time_window(timestamp, duration)`** — creates a time-bounded filter relative to `now()`, returns bool. **`stix_pattern_match(field, stix_pattern)`** — evaluates a STIX 2.1 indicator pattern against a field value, returns bool. **`mitre_tactic(technique_id)`** — resolves a MITRE ATT&CK technique ID to its parent tactic name (uses embedded ATT&CK knowledge base), returns string. **`severity_label(numeric_severity)`** — maps a numeric severity (0-100) to a human-readable label (Info/Low/Medium/High/Critical) using configurable thresholds, returns string. | CAP-029 | Config-Driven Sensor Adapters | Sensor adapters defined in TOML spec files, not Rust code. Runtime-interpreted (not codegen). Approximately 80% of REST API sensors are fully config-driven; approximately 20% use Rust escape hatches for exotic auth or binary protocols. Adding a new sensor = drop a TOML spec file + add client config + set credentials. Column OCSF mappings are validated against the compiled protobuf schema (warnings for invalid paths, not errors).
