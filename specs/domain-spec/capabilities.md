# Domain Capabilities

This section enumerates all domain capabilities in Prism. Capabilities are either MCP-facing (exposed as tools to the agent) or internal (consumed by other subsystems, never exposed directly to MCP clients). Priority reflects implementation order: P0 is required for MVP, P1 for full launch, P2 for post-launch enhancement.

| ID | Capability | Description | Business Rule | Priority |
|----|-----------|-------------|---------------|----------|
| CAP-001 | Sensor Adapter Layer (Internal) | **Internal capability -- not exposed to MCP clients.** The sensor adapter layer provides authenticated, paginated, retried data fetch from each sensor's API. It is consumed exclusively by the query engine (CAP-015), not by MCP clients directly. Per-sensor adapters handle: auth (OAuth2, cookie, bearer), pagination (cursor-based, forward-only), retry with backoff, and raw data fetch. Each adapter call is scoped to a single `(client_id, sensor_id)` pair. The query engine orchestrates fan-out across adapters. | Every adapter call must include a valid `client_id`. Responses are scoped exclusively to that client's sensor instance. Results are paginated internally with cursor-based forward-only traversal. The adapter layer is never invoked directly by MCP tool calls -- all data access flows through the query engine (CAP-015). | P0 |
| CAP-002 | Cross-Client Fan-Out (Internal) | **Internal capability -- describes the query engine's cross-client fan-out behavior.** When the query engine (CAP-015) receives a query with `clients: null` or multiple client IDs, it fans out to all configured clients for the specified sensors via the sensor adapter layer (CAP-001). Each result item includes `client_id` provenance. Sensors unavailable for a client are reported as partial failures in the query response's `sensor_errors` array, not as query-level errors. | Cross-client fan-out is orchestrated by the query engine, not exposed as a separate MCP tool. Each result item includes `client_id` provenance. Partial failures are surfaced in `sensor_errors`, not as query-level errors. | P0 |
| CAP-003 | OCSF Normalization | Normalize all sensor records to OCSF v1.x via the DynamicMessage protobuf pattern. Enables cross-sensor correlation by mapping vendor-specific fields to a common schema (e.g., CrowdStrike `hostname` and Claroty `device_name` both map to OCSF `device.hostname`). | Every sensor record returned via MCP tools includes an `ocsf` field containing the normalized OCSF representation. Unmappable vendor-specific fields are preserved in a `raw_extensions` JSON blob. OCSF schema version is fixed per Prism release. | P0 |
| CAP-004 | Credential Management | Store, retrieve, rotate, and delete per-client per-sensor credentials via OS keyring (macOS Keychain, Windows Credential Vault, Linux libsecret) with AES-256-GCM encrypted file fallback. Namespace: `(client_id, sensor_id, credential_name)`. Credential mutation operations (`set_credential`, `delete_credential`) are gated by the `credential.write` capability path and follow the hidden-tools pattern (BC-2.04.005). `list_credentials` is read-only and always visible. | Credentials are isolated by client -- accessing Client A's CrowdStrike credentials while operating in Client B's context is a type error. Credential values are never included in logs, MCP responses, or error messages. All credential access is audit-logged. Credential mutations require `credential.write` capability to be allowed for the target client. | P0 |
| CAP-005 | Feature Flag Evaluation | Evaluate two-tier feature flags (compile-time cargo features + runtime per-client TOML config) to determine which write operations are available for each client. Runtime config uses `BTreeMap<String, Effect>` where Effect is Allow or Deny. Hierarchical resolution: most-specific-path wins (e.g., `sensor.crowdstrike.containment = deny` overrides `sensor.crowdstrike = allow`). Deny beats allow at the same specificity level. Implicit deny is the final fallback. | Deny-by-default: if no flag explicitly enables a capability, it is denied. Both tiers (compile-time and runtime) must permit the operation. Most-specific capability path wins during resolution; at equal specificity, deny overrides allow. Disabled tools are hidden from `tools/list`. A `list_capabilities` meta-tool exposes the full capability matrix per client with an `explain()` trace showing which rule determined each outcome. | P0 |
| CAP-006 | Write Operation Gating | Gate all write/mutation operations (host containment, alert acknowledgment, device actions) behind a three-tier risk classification: read (no gate), reversible write (dry-run default with preview), irreversible write (confirmation token with 300s expiry). | Reversible writes default to `dry_run: true`; the agent must explicitly set `dry_run: false`. Irreversible writes require a two-step flow: request returns a `ConfirmationToken`, then `confirm_action(token)` executes. Tokens are single-use, time-bounded, and include an action-specific content hash. | P1 |
| CAP-007 | Audit Logging | Log every MCP tool invocation with: client_id, sensor, tool name, parameters (secrets redacted), user identity, timestamp, feature flags evaluated, result summary, and safety flags triggered. Structured JSON via `tracing` crate, compatible with Vector pipeline. | Audit entries are append-only and include sufficient detail for SOC 2 Type II and ISO 27001 compliance. Write operations log both the capability check result and the execution outcome. Credential access events are logged with client/sensor context but never credential values. | P0 |
| CAP-008 | Sensor Health Monitoring | Expose per-sensor, per-client connectivity status via MCP resources and a dedicated health tool. Detect unreachable APIs, expired credentials, and rate limit exhaustion. | Health checks run on-demand (not polled). Each sensor adapter implements `verify_connectivity()`. Health status includes: reachable/unreachable, auth valid/expired, last successful query timestamp, and current rate limit state. | P1 |
| CAP-009 | Client Configuration | Load and validate per-client sensor mappings, credential references, and capability overrides from TOML configuration. Support `--dry-run` validation that prints redacted config and exits. Re-evaluate capabilities on client context switch with `notifications/tools/list_changed`. | Config changes require server restart (no hot-reload in stdio per-analyst model). Validation is multi-error: all problems reported in one pass. Missing required fields produce actionable error messages specifying the exact config path. | P0 |
| CAP-010 | Prompt Injection Defense | Sanitize all sensor data flowing through MCP responses to defend against attacker-controlled content (hostnames, file paths, process names) that could manipulate the LLM. Four-layer defense: structural separation, provenance framing, suspicious pattern flagging, trust-level metadata. | Untrusted sensor data is placed in `structuredContent` JSON fields, never interpolated into prose. Tool descriptions include security warnings about untrusted data. Suspicious patterns (e.g., strings containing "ignore previous instructions") are flagged in parallel `_safety_flag` fields but never stripped -- the analyst needs full forensic data. | P0 |
| CAP-011 | Adapter Pagination (Internal) | **Internal capability -- cursors are never exposed to MCP clients.** Provides ephemeral per-query pagination between Prism's sensor adapter layer and the upstream sensor APIs. Cursors are created when a paginated adapter fetch begins, used to retrieve subsequent pages from the sensor API, and discarded when the fetch completes. The MCP agent never sees cursor tokens -- instead, query results include `limit` and `total_available` fields so the agent knows how many results exist and can adjust query scope. No disk persistence of cursors. | Cursors are ephemeral pagination tokens internal to the sensor adapter layer, scoped to a single adapter fetch invocation. Each cursor encodes enough state to fetch the next page from the sensor API. Cursors are not persisted to disk and do not survive server restart. Cursor tokens are never included in MCP responses -- the agent sees `limit` (how many results were returned) and `total_available` (how many matched). The agent narrows scope or adjusts `LIMIT` in AxiQL to control result size. | P0 |
| CAP-012 | Cross-Sensor Correlation (via Query Engine) | A natural consequence of the query engine's (CAP-015) unified OCSF table model. When the query engine fans out to multiple sensors, all results are normalized to OCSF and materialized into a single Arrow table. The agent correlates across sensors simply by querying OCSF fields (`device.ip`, `device.hostname`, `time`) in AxiQL -- no explicit JOINs or separate correlation step required. This is not a separate subsystem; it emerges from the query engine's single-table design combined with OCSF normalization (CAP-003). | Correlation happens implicitly within the query engine's materialized table. The agent writes a single AxiQL query that spans sensors; the query engine handles fan-out, normalization, and materialization. No separate correlation API or tool exists. | P1 |
| ~~CAP-013~~ | ~~xMP Envelope Delivery~~ | **REMOVED** — backward compatibility with xMP envelope format is not required. Prism is a new system; it does not need to produce xMP-enriched payloads for the existing Vector pipeline. | N/A | N/A |
| CAP-014 | Response Caching | LRU cache per-client per-sensor with configurable TTL (60s for alerts, 300s for devices). Bounded entry count (configurable, default 50 per client per sensor). Cache invalidation is synchronous with write operations — any write that mutates sensor state invalidates related cache entries before the write confirmation is returned. Agents may bypass cache with `force_refresh: true` parameter. Cache metrics (hit rate, eviction count, entry count) are exposed via the `check_sensor_health` tool. | Cache keys are `(client_id, sensor_id, query_hash)`. TTL is per-data-source-category: alert-class sources default to 60s, device/asset-class sources default to 300s, configurable in TOML. LRU eviction when entry count exceeds the per-client-per-sensor bound. Write operations invalidate all cache entries for the affected `(client_id, sensor_id, source_id)` tuple synchronously before returning confirmation. `force_refresh: true` bypasses cache and populates a fresh entry. Cache is in-memory only — lost on server restart. | P1 |
| CAP-015 | Ephemeral OCSF Query Engine | The core of Prism's ephemeral federated query engine architecture. The primary query MCP tool (`query`) turns what looks like a database query into orchestrated live API calls: Query -> Fetch -> Normalize -> Compute -> Return. There is no database -- the data lake is conjured on demand and exists only for the duration of the query. Accepts `clients`/`sensors`/`sources` scoping parameters plus an AxiQL query string. Fan-out to configured sensor APIs (parallel, per-client per-sensor), normalize all responses to OCSF via DynamicMessage, materialize an ephemeral Arrow RecordBatch, execute via DataFusion over an in-memory MemTable, return results, tear down the table. Three query modes: filter (`field = value AND field >= value`), SQL (`SELECT ... FROM events WHERE ... GROUP BY ... ORDER BY ... LIMIT`), and pipe (`FROM source \| where ... \| sort ... \| head N`). Auto-detects mode from first token. Cross-sensor correlation via unified OCSF table — implicit single-table model, no explicit JOINs. Virtual fields `sensor`, `client_id`, `source` are filterable within queries. Tool parameters set scope; query predicates narrow within that scope (intersection semantics, never widening). Includes `explain_query` tool for dry-run inspection of alias expansion, field resolution, and planned API calls. Powered by Chumsky 0.10 parser + DataFusion execution (hybrid architecture). | Every query execution creates a fresh DataFusion `SessionContext`, registers materialized data as a `MemTable`, executes the query, and drops the context when complete. Security limits enforced: 64KB max query length, 64 max nesting depth, 32 max pipe stages, 10K max materialized records, 30s query timeout. Violations return structured errors with actionable suggestions. Push-down filters are translated to sensor API parameters where possible; remaining predicates applied post-normalization via DataFusion. Results include `query_context` echo-back (original query, expanded query after alias resolution, clients/sensors queried, execution time) and `sensor_errors` for partial failure transparency. | P0 |
| CAP-016 | Query Aliases | Reusable, parameterized query shortcuts persisted in a dedicated `aliases.toml` file (separate from the main `prism.toml`). `aliases.toml` is written atomically (temp + fsync + rename) by `create_alias` and `delete_alias`, and loaded at startup alongside main config. Global aliases defined in `[aliases]` section; per-client aliases in `[clients.{id}.aliases]` section. Per-client aliases override global aliases of the same name. Aliases are composable (one alias may reference another, max depth 3). Parameterized aliases use `template` + `defaults` syntax; all parameters must have defaults. Alias resolution occurs pre-parse (before Chumsky parser). Expanded query must still pass security limits. MCP tools: `create_alias` (create/update global or per-client alias with optional parameters), `list_aliases` (list all aliases visible to a client — global + per-client merged), `delete_alias` (delete an alias, requires confirmation if referenced by other aliases), `explain_alias` (show definition, parameters, expanded query, which clients define it). Alias mutation tools (`create_alias`, `delete_alias`) are gated by the `alias.write` capability path and follow the hidden-tools pattern (BC-2.04.005). `list_aliases` and `explain_alias` are read-only and always visible. | Alias composition depth capped at 3. Cycles detected at config load time — cyclic alias references are a fatal config error. Cross-client queries with per-client aliases fail with structured error listing which clients define and which are missing the alias. `create_alias` and `delete_alias` are write operations gated by `alias.write` capability; `create_alias` additionally follows write-operation gating (confirmation token) for updates to existing aliases. Alias expansion is always shown in query responses for AI transparency. | P1 |
