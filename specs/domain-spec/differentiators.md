# Competitive Differentiator Traceability

This section maps Prism's competitive differentiators to the specific capabilities that enable them. These differentiators define why Prism exists rather than using off-the-shelf SIEM integrations or per-sensor MCP wrappers.

| Differentiator | Why It Matters | Supporting Capabilities |
|---------------|----------------|----------------------|
| **AI-Native Interface (Not a Dashboard)** | Prism is consumed exclusively by AI agent harnesses (Claude Code), not by humans clicking through a web UI. Every design decision optimizes for LLM consumption: `structuredContent` with machine-parseable JSON, `outputSchema` for type-aware reasoning, provenance framing for untrusted data, and self-documenting error responses with retryability hints and actionable suggestions. Traditional security tools assume a human reading a screen; Prism assumes an LLM reasoning over structured context. This means analysts interact in natural language ("show me all critical alerts for Acme from the last 24 hours across all sensors") instead of navigating 4 different sensor dashboards. A single `query` tool replaces what would otherwise be dozens of per-sensor, per-data-source tools. | CAP-015 (Ephemeral OCSF Query Engine), CAP-010 (Prompt Injection Defense), CAP-007 (Audit Logging) |
| **Cross-Sensor Correlation via OCSF** | No existing tool lets an MSSP analyst correlate a CrowdStrike endpoint detection with a Claroty OT event and an Armis device vulnerability for the same IP address in a single query. Prism's query engine normalizes all sensor data to OCSF v1.x via the DynamicMessage protobuf pattern, materializes results into a unified Arrow table, and lets the agent query across sensors with a single PrismQL expression. The AI agent reasons about `device.ip`, `device.hostname`, and `time` fields consistently across CrowdStrike alerts, Claroty events, Armis vulnerabilities, and Cyberint threat intelligence -- no explicit JOINs or separate correlation step required. The current workflow (copy data between browser tabs, manually correlate in spreadsheets) is eliminated. | CAP-015 (Ephemeral OCSF Query Engine), CAP-003 (OCSF Normalization), CAP-012 (Cross-Sensor Correlation) |
| **Multi-Client Single Session** | MSSP analysts manage dozens of clients. Today, each client requires separate dashboard logins per sensor -- Client A's CrowdStrike, Client A's Claroty, Client B's CrowdStrike, etc. Prism's query engine accepts `clients` scoping parameters, including `clients: null` for all-client fan-out, letting an analyst work across all clients and sensors in a single AI session. "Show me which clients have unacknowledged critical alerts" is a single PrismQL query, not a manual survey of N client dashboards. Client data isolation is enforced by the `TenantId` newtype and per-client state/cache/credential namespacing, preventing accidental mixing while enabling intentional cross-client visibility. | CAP-015 (Ephemeral OCSF Query Engine), CAP-002 (Cross-Client Fan-Out), CAP-009 (Client Configuration), CAP-004 (Credential Management) |
| **Feature-Flagged Write Operations** | Prism supports the full sensor API including dangerous write operations (host containment, network isolation, alert acknowledgment), but gates them behind a defense-in-depth system that no other MCP security tool provides. The two-tier feature flag system (compile-time cargo features + runtime per-client TOML config) means an MSSP operator can distribute a read-only binary where write code physically does not exist, or enable writes per-client based on contractual agreements and risk appetite. The three-tier risk classification (read/reversible-write/irreversible-write) with dry-run defaults and confirmation tokens ensures the AI agent cannot accidentally contain a host without explicit human confirmation. This enables progressive trust: start read-only, enable reversible writes after confidence builds, enable irreversible writes for mature clients with strong change management. | CAP-005 (Feature Flag Evaluation), CAP-006 (Write Operation Gating), CAP-007 (Audit Logging) |
| **OCSF Normalization with Vendor Extension Preservation** | Unlike SIEM normalization that drops vendor-specific fields, Prism's DynamicMessage pattern preserves unmapped fields in a `raw_extensions` JSON blob. The AI agent gets both the normalized OCSF view (for cross-sensor reasoning) and the full vendor-specific data (for deep-dive investigation). A CrowdStrike detection's MITRE ATT&CK mapping and a Claroty device's Purdue level are both accessible, even though OCSF does not have standard fields for these. This avoids the "normalization tax" where correlation capability comes at the cost of losing vendor-specific detail. | CAP-003 (OCSF Normalization), CAP-012 (Cross-Sensor Correlation) |
| **Prompt Injection Defense for Security Data** | Prism operates in a unique threat environment: it ingests data from security sensors monitoring adversarial environments. Attackers who know they are being monitored can embed prompt injection payloads in hostnames (`ignore-previous-instructions.evil.com`), file paths, and process names. These flow through Prism into the LLM context window. No other security MCP tool addresses this attack surface. Prism's four-layer defense (structural separation, provenance framing, suspicious pattern flagging, trust-level metadata) is purpose-built for the security domain where tool response data is inherently attacker-influenced. | CAP-010 (Prompt Injection Defense) |
| **Unified Sensor Adapter Architecture** | The 4 Go pollers (poller-cobra, poller-express, poller-bear, poller-coaster) share 90%+ identical infrastructure code (cursor management, health probes, xMP enrichment, backoff, sink delivery) but each reimplements it independently, with different bugs. Prism's generic `DataSource<T>` trait and `SensorAdapter` trait eliminate this duplication. A bug fix in cursor management applies to all 4 sensors simultaneously. Adding a 5th sensor requires implementing the trait, not copying and modifying 2000 lines of boilerplate. The trait-based architecture also enables cross-sensor consistency: all sensors use the same backoff strategy, the same error categorization, and the same audit logging. The adapter layer is consumed by the query engine -- adding a new sensor automatically makes it queryable via PrismQL without any MCP tool changes. | CAP-001 (Sensor Adapter Layer), CAP-011 (Adapter Pagination), CAP-008 (Sensor Health Monitoring) |
| **SOC 2 / ISO 27001 Audit Trail** | Every MCP tool invocation is logged with structured JSON fields sufficient for SOC 2 Type II and ISO 27001 compliance: client_id, sensor, tool name, parameters (secrets redacted), user identity, timestamp, feature flags evaluated, result summary, and safety flags triggered. This is not an afterthought -- audit logging is a first-class subsystem with its own invariant (DI-004: every invocation produces exactly one AuditEntry). The audit trail integrates with the existing Vector pipeline for centralized log aggregation, fitting into 1898 & Co's existing compliance infrastructure. | CAP-007 (Audit Logging), CAP-005 (Feature Flag Evaluation), CAP-006 (Write Operation Gating) |
