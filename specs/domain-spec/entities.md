# Domain Entities

This section defines all domain entities in Prism, their key attributes, and the invariants that constrain them. Entity names use `PascalCase` per Rust convention.

| Entity | Description | Key Attributes | Invariants |
|--------|-------------|----------------|------------|
| **Client** | An MSSP client organization whose security sensors are managed through Prism. Maps to a `[clients.{id}]` section in TOML config. Also referred to as "tenant" in internal code (`TenantId` newtype). | `client_id: TenantId` (validated, non-empty, allowlist chars `[a-zA-Z0-9_-]`), `display_name: String`, `sensors: Vec<SensorConfig>`, `capabilities: ClientCapabilities` | `client_id` is unique across all config. Every MCP tool call must reference exactly one Client or null (cross-client). Client data is never mixed in state, cache, or log output. |
| **Sensor** | A security platform instance belonging to a Client. Each Client may have any number of sensors — both the four initial sensors (CrowdStrike, Cyberint, Claroty, Armis) shipped as TOML spec files and any additional sensors added via the config-driven spec system (CAP-029). All sensors use the same spec-driven adapter pipeline. | `sensor_id: String` (pattern-validated `^[a-z][a-z0-9_-]*$`, sourced from spec file `sensor_id`), `api_base: Url`, `credential_ref: String`, `enabled: bool`, `data_sources: Vec<SourceId>` | A Sensor belongs to exactly one Client. The `sensor_id` + `client_id` pair is unique. Disabled sensors are excluded from queries and health checks. |
| **SensorAdapter** | Runtime implementation of the sensor fetch pipeline, driven by a loaded `SensorSpec` (CAP-029). All sensors — including the four initial sensors (CrowdStrike, Cyberint, Claroty, Armis) — are instantiated from TOML spec files. The spec engine interprets the spec's multi-step fetch pipeline, auth type, pagination config, and column/OCSF mappings at runtime. Adapter instances support two fetch modes: **batch** (returns all records at once via `async fn fetch_batch() -> Vec<RecordBatch>`, suitable for small bounded result sets like single-entity lookups) and **streaming** (returns an `async Stream` yielding `RecordBatch` chunks, suitable for paginated API responses where results arrive across multiple pages). The query engine selects the mode based on the spec's declared preference per table and the presence of constraints that bound the result set. For the ~20% of sensors requiring exotic behavior (binary protocols, complex streaming), a `CustomAdapter` trait (BC-2.16.004) allows Rust code overrides of specific pipeline stages while the rest remains spec-driven. | `sensor_id: String`, `client_id: TenantId`, `spec: Arc<SensorSpec>`, `auth: Box<dyn SensorAuth>`, `http_client: reqwest::Client`, `backoff_config: BackoffConfig`, `fetch_mode: FetchMode` (per table: `Batch` or `Stream`), `custom_overrides: Option<Arc<dyn CustomAdapter>>` | Each adapter instance is bound to exactly one (Client, Sensor) pair. The `SensorAuth` trait is sealed — cross-sensor auth composition is prevented at the type level. Batch mode results are cacheable by the in-query cache; streaming mode results are materialized incrementally and cached after the stream completes. |
| **Credential** | A secret value (API key, OAuth2 client secret, bearer token, cookie) stored in the credential store, namespaced by `(client_id, sensor_id, credential_name)`. | `client_id: TenantId`, `sensor_id: String` (pattern-validated, matches spec file sensor_id), `credential_name: String` (e.g., "client_secret", "api_key"), `value: SecretString` (never logged) | Credential names validated against `[a-zA-Z0-9_\-\.]+` (prevents path traversal). Values encrypted at rest (AES-256-GCM in file backend, OS keyring otherwise). Cross-client credential access is a type error via `TenantId` scoping. |
| **DataSource** | A specific data feed within a sensor (e.g., CrowdStrike alerts, Claroty devices, Armis vulnerabilities). Defined by `TableSpec` in sensor spec files. | `source_id: SourceId`, `sensor_id: String`, `record_type: String` (e.g., `crowdstrike_alert`), `cursor_type: CursorType` | Each DataSource produces records of a single type. The `record_type` string follows `<sensor>_<entity>` naming. Cursor semantics are DataSource-specific (2-tuple vs 3-tuple). |
| **OcsfEvent** | A sensor record normalized to OCSF v1.x via the DynamicMessage protobuf wrapper. Provides runtime field access across all 83 OCSF event classes without per-class code generation. | `event_class: u16`, `message: DynamicMessage`, `raw_extensions: serde_json::Value`, `source_sensor: String`, `source_record_type: String` | OCSF schema version is fixed per Prism release. Field resolution follows four tiers: Prism metadata fields, proto descriptor fields, unmapped JSON blob, None. `raw_extensions` preserves vendor fields not mappable to OCSF. |
| **Cursor** | An ephemeral pagination token used **internally** by the sensor adapter layer (CAP-001) to navigate multi-page results from upstream sensor APIs. Cursors are never exposed to MCP clients -- the agent never sees cursor tokens. Instead, query results include `limit` and `total_available` so the agent can adjust query scope. Internal structure varies by sensor adapter. Not persisted to disk; discarded when the adapter fetch completes. | `token: String` (opaque, base64-encoded adapter-specific state), `client_id: TenantId`, `sensor_id: String`, `source_id: SourceId`, `created_at: DateTime<Utc>`, `expires_at: DateTime<Utc>` (created_at + 600s default TTL) | Cursors are ephemeral, in-memory only, and internal to the adapter layer -- no disk persistence, no survival across server restart, no exposure in MCP responses. Cursors expire after 600 seconds (configurable) to prevent stale pagination from consuming memory. The 600s TTL (vs 300s for confirmation tokens) accounts for the fact that multi-page sensor API fetches may take time during large fan-out operations. Maximum 200 active (non-expired) cursors enforced at any time as an internal resource management limit; the 200 cursor cap is a global per-process limit shared across all concurrent query engine executions. New cursor creation beyond this cap returns `E-STATE-002` internally, causing the query engine to report a structured error suggesting the agent narrow query scope. Expired cursors are lazily cleaned up on each new cursor creation request. Cursor tokens are valid only within the adapter fetch session that created them. |
| ~~**QueryFingerprint**~~ | **REMOVED** — No longer needed. With ephemeral pagination cursors (no disk persistence), there is no stored cursor state to invalidate on config drift. Query parameters are validated per-invocation. | N/A | N/A |
| **FeatureFlag** | A capability path in the hierarchical feature flag system (e.g., `sensor.crowdstrike.containment`). Resolved via two tiers: compile-time cargo features and runtime TOML per-client config. | `path: String` (dot-separated hierarchy), `compile_time_enabled: bool`, `runtime_enabled: bool` | Both tiers must permit the operation. Resolution walks the path hierarchy upward. Final fallback is always deny. Flag evaluation is audit-logged for write operations. |
| **ClientCapability** | The resolved capability set for a specific Client, built at config load time by merging profile defaults with `[clients.{id}.capabilities]`. Stored as a `BTreeMap<String, Effect>` where `Effect` is `Allow` or `Deny`. Most-specific capability path wins during resolution; deny overrides allow at the same specificity. | `client_id: TenantId`, `rules: BTreeMap<String, Effect>` (where `Effect` is `Allow \| Deny`) | Resolution walks from the exact capability path upward through parent segments; the first matching rule determines the effect. If no rule matches, implicit deny applies. Deny at the same or more-specific level overrides allow. Re-evaluated on config reload; triggers `notifications/tools/list_changed` when the resolved tool set changes. An `explain(capability) -> CapabilityExplanation` method returns the resolution trace for audit and debugging. |
| **ConfirmationToken** | A time-bounded, single-use token authorizing an irreversible write operation. Returned by the first call to a high-risk write tool; consumed by `confirm_action`. In-memory only — tokens are lost on server restart. | `token_id: String` (cryptographic random), `client_id: TenantId` (the client this token authorizes action for — prevents cross-client token replay; for global-scope operations (aliases, schedules, packs, global-scope rules), `client_id` is set to the sentinel value `"__global__"` since no specific client is targeted), `tool_name: String` (the originating write tool, e.g., `crowdstrike_contain_host`, `set_credential`), `action_params: serde_json::Value` (the original parameters for re-execution by `confirm_action`), `action_summary: String`, `action_hash: String` (SHA-256 of action params), `created_at: DateTime<Utc>`, `expires_at: DateTime<Utc>` (created_at + 300s), `consumed: bool` | Tokens expire after 300 seconds. A token is single-use: once consumed, it cannot be reused. The `action_hash` ensures the confirmed action matches the original request. The `client_id` must match the `client_id` provided to `confirm_action` — mismatches return a structured error. The `tool_name` and `action_params` fields enable `confirm_action` to re-dispatch the original write operation without the caller needing to re-supply parameters. Expired or consumed tokens return a structured error guiding the agent to re-request. Maximum 100 active (unexpired, unconsumed) tokens at any time; new token requests beyond this cap are rejected with a structured error. Tokens are in-memory only — server restart loses all tokens (agent receives `E-FLAG-008` and must re-request). |
| **CacheEntry** | An in-memory cached response for a sensor query, keyed by the 4-tuple `(client_id, sensor_id, source_id, query_hash)`. Serves the query engine's sensor fetch layer (CAP-015) via the sensor adapter layer (CAP-001) — cross-query cache sharing is the primary benefit, as multiple PrismQL queries that need the same underlying sensor data share cached adapter-level entries. Used by the response caching subsystem (CAP-014) to avoid redundant sensor API calls within the TTL window. | `client_id: TenantId`, `sensor_id: String`, `source_id: SourceId` (first-class key component — not hashed, enabling prefix-scan invalidation), `query_hash: String` (SHA-256 of canonicalized push-down filter parameters — includes sensor-native filter params translated from PrismQL predicates; excludes sort, limit, cursor, and force_refresh since cache stores the full API response before DataFusion post-processing. Relative time ranges resolved to absolute timestamps truncated to nearest minute before hashing.), `value: Vec<u8>` (serialized response), `ttl: Duration` (60s alerts, 300s devices — configurable), `created_at: DateTime<Utc>`, `hit_count: u64` | Cache entries are in-memory only — lost on server restart. LRU eviction when entry count exceeds the configurable per-client-per-sensor bound (default 50). Write operations invalidate all entries for the affected `(client_id, sensor_id, source_id)` tuple via prefix scan over the first three key components — no need to enumerate individual `query_hash` values. `force_refresh: true` bypasses and replaces the cached entry. `hit_count` is tracked for metrics exposed via `check_sensor_health`. |
| **AuditEntry** | A structured log record emitted for every MCP tool invocation. Written via `tracing::info!` with structured fields, consumed by the Vector pipeline. | `timestamp: DateTime<Utc>`, `trace_id: String`, `client_id: String` (for single-client tools, this is the TenantId; for query engine tools operating on multiple clients, this is `"multi_client"` and the full client list is in `clients_queried`; for cross-client queries with `clients: null`, this is `"all_clients"`), `clients_queried: Option<Vec<TenantId>>` (populated for subsystem 11 query engine tools that operate on multiple clients; null for single-client tools. This is a first-class compliance field for audit trail — not buried in `parameters`), `tool_name: String`, `parameters: serde_json::Value` (secrets redacted), `user_identity: String`, `result_summary: String`, `capability_checks: Vec<CapabilityCheckEvent>`, `safety_flags: Vec<String>` | Audit entries are append-only. Every tool invocation produces exactly one AuditEntry. Write operations include capability check details. Credential values are never present in any audit field. `user_identity` is resolved at startup via a three-tier chain: (1) TOML config `analyst_id` field, (2) `PRISM_ANALYST_ID` env var, (3) OS username detection. The resolved value is immutable for the session lifetime. |
| **ToolResponse** | The MCP tool response envelope wrapping all Prism query results. Uses `structuredContent` for machine-parseable data and `content[].text` for LLM-consumable summary. | `_meta: ResponseMeta` (tool, data_source, query_time, trust_level, safety_flags, pagination), `results: Vec<serde_json::Value>`, `content_summary: String` | All responses include `trust_level: "untrusted_external"` for sensor data. Safety flags are populated when suspicious patterns are detected in string fields. Result metadata (`limit`, `total_available`) is always present so the agent knows result size and can adjust query scope. |
| **ClientConfig** | The deserialized TOML configuration for a single Client, including sensor connections, credential references, and capability overrides. | `client_id: TenantId`, `display_name: String`, `sensors: HashMap<String, SensorConfig>`, `capabilities: Option<CapabilityOverrides>` | Config is validated at startup with multi-error aggregation. Missing required fields produce actionable errors with exact TOML paths. Capability overrides merge with `[defaults.capabilities]` using more-specific-wins semantics. |
| **QueryPlan** | The execution plan produced by parsing an PrismQL query string through the Chumsky parser and translating the resulting AST into a DataFusion logical plan. Encapsulates all information needed to execute a query: resolved client/sensor scope, push-down filters (translated to sensor API parameters), post-normalization filters (applied via DataFusion), OCSF field mappings, and the query mode (filter/SQL/pipe). | `original_query: String`, `expanded_query: String` (after alias expansion), `mode: QueryMode` (Filter, Sql, Pipe), `resolved_clients: Vec<TenantId>`, `resolved_sensors: Vec<String>`, `push_down_filters: HashMap<String, SensorNativeFilter>`, `post_filters: Vec<DataFusionExpr>`, `ocsf_field_mappings: Vec<FieldMapping>`, `projections: Option<Vec<String>>`, `aggregations: Option<Vec<AggExpr>>`, `sort_exprs: Option<Vec<SortExpr>>`, `limit: Option<usize>` | A QueryPlan is immutable after creation. The plan must pass all security limit checks (DI-019) before execution begins. Push-down filter classification happens during plan construction — each predicate is classified as push-down-capable or post-filter based on the target sensor's API capabilities. |
| **MaterializedTable** | An ephemeral Arrow RecordBatch table holding OCSF-normalized data for a single query execution. Created by fan-out to sensor APIs, OCSF normalization, and Arrow column construction. Registered as a DataFusion `MemTable` in a per-query `SessionContext`. Dropped when the query returns. | `schema: Arc<ArrowSchema>`, `batches: Vec<RecordBatch>`, `record_count: usize`, `approx_bytes: usize`, `sensors_contributed: Vec<String>`, `clients_contributed: Vec<TenantId>` | Bounded at 10K records (DI-019). If fan-out produces more than 10K records, materialization is aborted with a structured error suggesting scope narrowing. The table uses two-tier columnar storage: hot OCSF fields as flat Arrow columns + full event JSON in an `event_data` string column for `json_extract_string()` UDF access. Dropped when the owning `SessionContext` goes out of scope. |
| **Alias** | A reusable query shortcut persisted in `aliases.toml` (a dedicated file separate from the main `prism.toml`, written atomically by `create_alias` and `delete_alias`, loaded at startup). Can be global (applies to all clients) or per-client (overrides global alias of same name). Supports parameterized templates with default values. Composable — one alias may reference another alias, up to a maximum composition depth of 3. | `name: String` (validated identifier), `scope: AliasScope` (Global or Client(TenantId)), `query_template: String` (the PrismQL expression or template), `parameters: Option<HashMap<String, String>>` (parameter names to default values; None if not parameterized), `description: Option<String>`, `composed_from: Vec<String>` (names of other aliases referenced by this alias, populated at config load time) | Alias names must be valid identifiers (`[a-zA-Z_][a-zA-Z0-9_]*`). All parameters must have defaults so aliases are usable without arguments. Composition depth is validated at config load time — depth exceeding 3 is a fatal config error. Cyclic references detected at config load time via visited-set traversal — cycles are a fatal config error with the exact alias chain reported. Resolution order is inner-to-outer (resolve innermost alias first, substitute upward). |
| **Schedule** | A scheduled PrismQL query that runs on a configurable interval per client, managed by the scheduler (CAP-017). Schedules are defined in TOML config (global or per-client) or as part of a pack (CAP-023). State is persisted to RocksDB under the Schedules storage domain. | `query_name: String` (unique identifier), `prismql_query: String` (the PrismQL query to execute), `interval: Duration` (how often to run), `splay_offset: Duration` (computed from `(interval * splay_percent / 100) * hash(client_id, schedule_name) / MAX_HASH` for API load distribution), `clients: Vec<TenantId>` (target clients; empty means all configured clients), `last_run: Option<DateTime<Utc>>` (timestamp of last execution), `next_run: DateTime<Utc>` (computed as `last_run + interval + splay_offset`), `epoch: u64` (incremented on process restart, restored from RocksDB), `counter: u64` (incremented per execution, reset on epoch change), `enabled: bool`, `snapshot_mode: bool` (if true, store full results on every run instead of only the differential — useful for audit or baseline snapshots), `track_removed: bool` (if true, include records that disappeared between runs in the differential output — default true), `pack_id: Option<String>` (if this schedule belongs to a pack) | `query_name` is unique within its scope (global or per-client). The `interval` must be >= 60 seconds (minimum schedule granularity). `splay_offset` is deterministic per `(query_name, client_id)` pair and persisted to RocksDB for stability across restarts. The `epoch`/`counter` pair uniquely identifies each execution for exactly-once semantics (DI-023). `enabled: false` schedules are loaded but not executed. |
| **DiffState** | The stored state for differential result computation (CAP-018), keyed by `(query_name, client_id)`. Persisted in RocksDB under the DiffResults storage domain (StorageDomain::DiffResults, column family `diff_results`). | `query_hash: String` (SHA-256 of the PrismQL query string — detects query changes requiring full diff reset), `client_id: TenantId`, `previous_results_hash: String` (SHA-256 of serialized previous results — for fast change detection), `previous_results: Vec<u8>` (bincode-serialized Arrow RecordBatch of previous results — for row-level diff), `epoch: u64` (must match scheduler epoch), `counter: u64` (must match scheduler counter for the last diff), `last_diff_time: DateTime<Utc>` | Diff state is invalidated (reset to empty) when the `query_hash` changes (query was modified in config). Epoch/counter mismatch triggers a full diff (not an error — handles restarts gracefully). Previous results are bounded by the 10K materialization limit (DI-019). Stale diff state (last_diff_time older than 7 days with no matching schedule) is purged during periodic cleanup. |
| **DetectionRule** | A detection rule that evaluates query results to identify security-relevant conditions (CAP-020). Supports three match modes for different detection patterns. Can be defined in `.axd` files, TOML config, or created at runtime via MCP tool. | `rule_id: String` (unique identifier), `name: String` (human-readable display name), `description: Option<String>`, `prismql_predicate: String` (the PrismQL condition expression), `match_mode: MatchMode` (enum: `Single` — stateless per-record evaluation; `Correlation { window: Duration, threshold: u64, group_by: Vec<String> }` — threshold over time window with group-by fields; `Sequence { steps: Vec<SequenceStep>, window: Duration }` — ordered multi-event patterns with shared key field), `severity: Severity` (enum: Info, Low, Medium, High, Critical), `template: AlertTemplate { title: String, description: String }` (with `{variable}` interpolation placeholders), `scope: RuleScope` (enum: `Global` — MSSP baseline; `Client(TenantId)` — per-client override; `Analyst(String)` — ad-hoc, persisted to RocksDB not config files), `enabled: bool`, `correlation_config: Option<CorrelationConfig { window: Duration, threshold: u64, group_by: Vec<String> }>`, `sequence_config: Option<SequenceConfig { steps: Vec<SequenceStep { name: String, condition: String, step_type: StepType }>, window: Duration }>` | `rule_id` is unique across all scopes. Per-client rules override global rules of the same `rule_id`. All rules are validated at load time (DI-024); invalid rules are rejected with actionable errors listing every validation failure. `MatchMode` determines which config field is required: `Correlation` requires `correlation_config`, `Sequence` requires `sequence_config`, `Single` requires neither. Runtime-created rules (scope `Analyst`) are persisted to RocksDB under the Detections domain and survive restarts. |
| **Alert** | A generated alert from a detection rule match (CAP-021). Immutable after creation. Persisted to RocksDB under the Alerts storage domain. | `alert_id: String` (UUID v7, time-sortable), `rule_id: String` (the detection rule that fired), `client_id: TenantId` (the client context in which the rule matched), `severity: Severity` (inherited from rule at fire time, immutable), `message: String` (template-rendered from rule's alert template with variable interpolation), `matched_events: Vec<String>` (event UIDs or row hashes that triggered the rule), `created_at: DateTime<Utc>`, `acknowledged_at: Option<DateTime<Utc>>` (set by `acknowledge_alert` tool), `resolved_at: Option<DateTime<Utc>>` (set when linked case reaches Resolved status), `status: AlertStatus` (computed, not stored — derived from timestamps: `open` if `acknowledged_at` is null, `acknowledged` if `acknowledged_at` is set but `resolved_at` is null, `resolved` if `resolved_at` is set) | Alerts are append-only — they are created and may be acknowledged but never modified or deleted. `alert_id` is globally unique (UUID v7). `severity` is fixed at creation time and does not change if the rule's severity is later modified. The `status` field is computed at read time from `acknowledged_at` and `resolved_at` timestamps — it is never persisted directly. Alert deduplication key varies by match mode: single-event uses `(rule_id, event_uid)`, correlation uses `(rule_id, group_by_value_hash, window_bucket)`, sequence uses `(rule_id, sequence_completion_hash)` — duplicates within the same dedup window are suppressed (BC-2.13.013). Alerts are scoped by `client_id` via `TenantId` typing. |
| **Case** | An investigation case grouping one or more alerts with lifecycle tracking, disposition, and annotations (CAP-022). Persisted to RocksDB under the Cases storage domain. | `case_id: String` (UUID v7, time-sortable), `title: String` (human-readable case title; auto-generated from first linked alert's summary if not provided at creation time; updatable via `update_case`), `source_alert_ids: Vec<String>` (linked alert IDs, deduplicated), `client_id: TenantId`, `status: CaseStatus` (enum: New, Acknowledged, Investigating, Resolved, Closed — 12 valid transitions per DI-025), `disposition: Option<Disposition>` (enum: `TruePositive { impact_level: String }`, `FalsePositive { reason: String }`, `Benign { explanation: String }`, `Inconclusive`), `assignee: Option<String>` (analyst identifier), `timeline: Vec<Annotation>` (investigation entries with type note/status_change/alert_link/evidence_link/ot_impact, content, author, timestamp), `severity: Severity` (enum: Info, Low, Medium, High, Critical — inferred from highest-severity linked alert, or set explicitly at creation/update time), `updated_at: DateTime<Utc>` (set on every mutation — creation, status transition, annotation, alert linking, disposition change, assignee change), `created_at: DateTime<Utc>`, `resolved_at: Option<DateTime<Utc>>` (set on first transition to Resolved; not overwritten on subsequent Resolved transitions after reopen cycles), `closed_at: Option<DateTime<Utc>>` (set on transition to Closed), `mttd: Option<Duration>` (auto-computed: `created_at` minus earliest linked alert `created_at`), `mttr: Option<Duration>` (auto-computed: `resolved_at` minus `created_at`, populated when status reaches Resolved) | Cases are scoped by `client_id`. `alert_ids` are deduplicated — linking the same alert twice is a no-op. Every status transition and mutation generates a `TimelineEntry` in the timeline for full audit trail. `mttd` is recomputed when new alerts are linked. `mttr` is set when transitioning to Resolved (using `resolved_at`) and cleared when reopening (Resolved/Closed → Investigating). `resolved_at` is set on first transition to Resolved and not overwritten on subsequent transitions after reopen cycles. `closed_at` is set on transition to Closed and cleared on reopen. Disposition can be set at any status. Invalid status transitions are rejected per DI-025. |
| **Pack** | A named bundle of related scheduled queries, detection rules, and aliases for specific MSSP workflows (CAP-023). Defined in TOML config. | `pack_id: String` (unique identifier), `name: String` (human-readable display name), `description: Option<String>`, `queries: Vec<ScheduleRef>` (references to scheduled query definitions within the pack), `detections: Vec<RuleRef>` (references to detection rule definitions within the pack), `discovery_query: Option<String>` (PrismQL query that must return >= 1 row for the pack to be active for a client — DEC-034), `enabled: bool`, `per_client_overrides: HashMap<TenantId, PackOverride>` (enable/disable specific queries or rules per client) | `pack_id` is unique across all config. Discovery query results are cached for 3600 seconds per `(pack_id, client_id)`. Packs compose with individual schedules and rules — a pack does not "own" its queries; it references them. `per_client_overrides` can disable specific pack queries for a client without removing the client from the pack entirely. Built-in packs (incident-response, daily-triage, compliance) are hardcoded and cannot be deleted but can be disabled. |
| **StorageDomain** | An enum identifying the logical storage domains within the RocksDB persistence layer (CAP-019). Each domain maps to a distinct RocksDB column family (BC-2.15.001). | `variant: StorageDomainVariant` (enum with 12 variants matching BC-2.15.001 column families: `Default` — general-purpose key-value storage, column family `default`; `Schedules` — scheduled query definitions, splay offsets, epoch counters, timing state, column family `schedules`; `DiffResults` — differential result history, column family `diff_results`; `DetectionRules` — detection rule storage by scope, column family `detection_rules`; `DetectionState` — correlation windows, sequence trackers, column family `detection_state`; `Alerts` — persisted alert records and indexes, column family `alerts`; `Cases` — case lifecycle, timeline, indexes, column family `cases`; `AuditBuffer` — buffered audit log entries awaiting forwarding, column family `audit_buffer`; `DirtyBits` — crash recovery markers, column family `dirty_bits`; `Watchdog` — query denylist entries and watchdog state, column family `watchdog`; `Aliases` — alias definitions, splay offsets, and metadata, column family `aliases`; `Decorators` — periodic decorator cache values, column family `decorators`) | Each domain is an isolated key-value namespace. Domain variant names map directly to RocksDB column family names (lowercased, snake_case). Domain names are stable across Prism versions. The `DirtyBits` domain stores a marker when a query begins execution and removes it on completion — a marker present at startup indicates the previous process crashed mid-query (used by CAP-024 for query denylisting). |
| **InternalTable** | A DataFusion table backed by a RocksDB storage domain (CAP-028), registered in the query engine's catalog under the `prism` namespace. Implements `TableProvider` to expose persistent Prism data (alerts, cases, rules, schedules, diff results, audit entries, aliases) through the same PrismQL interface used for external sensor tables. | `table_name: String` (e.g., `"prism_alerts"`, `"prism_cases"`), `storage_domain: StorageDomainVariant` (the RocksDB domain this table reads from), `schema: Arc<ArrowSchema>` (Arrow schema derived from the corresponding entity definition), `access_mode: AccessMode` (enum: `ReadOnly` — for audit, diff_results; `ReadWrite` — for cases, alerts, rules, schedules, aliases; note: `ReadWrite` means the underlying data is mutable via dedicated MCP tools, but the table itself is always read-only in PrismQL) | `table_name` follows the pattern `prism_{domain_name}` where `domain_name` is the snake_case domain identifier. Underscore notation is required because dots are not valid in the PrismQL `source` production rule. The Arrow schema is fixed per Prism release and derived from entity definitions at compile time. Internal tables are always read-only via PrismQL — mutations go through dedicated MCP tools. Client data separation (DI-008) is enforced via `client_id` filtering on every scan. |
| **SensorSpec** | The TOML-defined sensor adapter specification (CAP-029). A declarative description of a sensor's API, loaded from a `.toml` file in the sensor specs directory. Runtime-interpreted by the `prism-spec-engine` crate to produce DataFusion table providers without Rust code changes. | `sensor_id: String` (unique identifier matching `^[a-z][a-z0-9_-]*$`), `name: String` (human-readable display name), `auth_type: AuthType` (enum: `OAuth2ClientCredentials`, `BearerStatic`, `CookieRoundtrip`, `ApiKey`), `base_url: Url`, `tables: Vec<TableSpec>` (one or more queryable tables), `rate_limit_hints: Option<RateLimitHints>` (requests_per_second, burst_size), `version: String` (semver) | `sensor_id` is unique across all loaded spec files — a duplicate `sensor_id` in a second file is a load-time error (E-SPEC-009). Each spec file maps to exactly one sensor. The `auth_type` declares which authentication flow to use; actual credentials come from the credential store (CAP-004), not the spec file. Validated at load time (DI-030). |
| **TableSpec** | A single queryable table within a sensor spec (CAP-029). Defines the output schema (columns), the fetch pipeline (steps), and pagination behavior. Registered as a DataFusion table at load time. | `table_name: String` (unique within the sensor, matching `[a-zA-Z0-9_]+`), `columns: Vec<ColumnSpec>` (output column definitions), `steps: Vec<FetchStep>` (ordered multi-endpoint fetch pipeline), `pagination: PaginationType` (enum: `CursorToken { cursor_response_path: String }`, `OffsetLimit { page_size: u32 }`, `None`), `cache_ttl_override: Option<Duration>` (overrides default cache TTL for this table), `ocsf_class: Option<String>` (OCSF event class for normalization, e.g., `"security_finding"`) | `table_name` is unique within its parent `SensorSpec`. Must have at least one column and at least one step. The fully qualified table name in PrismQL is `{sensor_id}.{table_name}`. |
| **ColumnSpec** | A column definition with OCSF mapping within a table spec (CAP-029). Defines the output column's name, type, behavioral options, and optional mapping to an OCSF field for cross-sensor correlation. | `name: String` (column name, unique within the table), `col_type: ColumnType` (enum: `String`, `Integer`, `Float`, `Boolean`, `Datetime`, `Json`), `options: ColumnOptions` (enum flags: `Required` — must appear in WHERE clause per DI-021, `Index` — suitable for push-down, `Additional` — triggers enrichment step, `Hidden` — excluded from schema introspection, `Optimized` — both Index and Additional), `ocsf_field: Option<String>` (dot-notation OCSF field path, e.g., `"device.hostname"`, `"severity_id"`), `description: Option<String>` | Column names must be unique within a table. Columns with `Required` option enforce WHERE clause constraints identically to built-in adapter required columns (DI-021). The `ocsf_field` is validated against the compiled OCSF schema at load time (warning for invalid paths, not error). |
| **FetchStep** | One step in a multi-endpoint fetch pipeline within a table spec (CAP-029). Steps execute sequentially, with later steps able to reference variables produced by earlier steps via `${step_name.field}` interpolation. | `name: String` (unique within the table's steps, used as variable scope name), `method: HttpMethod` (enum: `GET`, `POST`), `path_template: String` (URL path with variable interpolation, appended to sensor's `base_url`), `body_template: Option<String>` (request body with variable interpolation, for POST steps), `response_path: String` (JSONPath to the results array in the response), `pagination_cursor_path: Option<String>` (JSONPath to pagination cursor in response), `variables_produced: Vec<String>` (variable names extractable from this step's response for downstream steps), `fan_out_batch_size: Option<u32>` (batch size when a variable resolves to an array, default 100) | Step `name` must be unique within the table. Variable references (`${step_name.field}`) in templates are validated at load time — references to undefined or forward-declared steps are validation errors (DEC-038). Steps execute in array order; no parallel execution within a pipeline. |
| **ConfigSnapshot** | The immutable config state after a load or reload (CAP-030). Captured atomically and stored in an `arc_swap::ArcSwap` for lock-free query-time access. Queries hold a reference to the snapshot that was current when execution began; mid-query reloads do not affect in-flight queries. | `hash: String` (SHA-256 of concatenated config file contents — prism.toml + all sensor spec files + aliases.toml), `loaded_at: DateTime<Utc>`, `sensor_specs: Vec<SensorSpec>` (all loaded config-driven sensor specs), `client_configs: Vec<ClientConfig>` (all client configurations), `feature_flags: FeatureFlagConfig` (resolved feature flag state), `aliases: AliasConfig` (all alias definitions from aliases.toml) | `hash` enables change detection: if the hash matches between reload attempts, the reload is a no-op. The `ConfigSnapshot` is immutable after construction — no interior mutability. At most 2 snapshots exist simultaneously (current + previous held by in-flight queries). The previous snapshot is freed when the last reference is dropped (via Arc reference counting). |
