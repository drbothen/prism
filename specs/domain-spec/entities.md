# Domain Entities

This section defines all domain entities in Prism, their key attributes, and the invariants that constrain them. Entity names use `PascalCase` per Rust convention.

| Entity | Description | Key Attributes | Invariants |
|--------|-------------|----------------|------------|
| **Client** | An MSSP client organization whose security sensors are managed through Prism. Maps to a `[clients.{id}]` section in TOML config. Also referred to as "tenant" in internal code (`TenantId` newtype). | `client_id: TenantId` (validated, non-empty, allowlist chars `[a-zA-Z0-9_-]`), `display_name: String`, `sensors: Vec<SensorConfig>`, `capabilities: ClientCapabilities` | `client_id` is unique across all config. Every MCP tool call must reference exactly one Client or null (cross-client). Client data is never mixed in state, cache, or log output. |
| **Sensor** | A security platform instance belonging to a Client. Each Client may have 0-4 sensors (CrowdStrike, Cyberint, Claroty, Armis). | `sensor_id: SensorId` (enum: CrowdStrike, Cyberint, Claroty, Armis), `api_base: Url`, `credential_ref: String`, `enabled: bool`, `data_sources: Vec<SourceId>` | A Sensor belongs to exactly one Client. The `sensor_id` + `client_id` pair is unique. Disabled sensors are excluded from queries and health checks. |
| **SensorAdapter** | Runtime implementation of the `SensorAdapter` trait for a specific sensor type. Encapsulates authentication, API contract, pagination, and cursor semantics. One concrete type per sensor: `CrowdStrikeAdapter`, `CyberintAdapter`, `ClarotyAdapter`, `ArmisAdapter`. | `sensor_id: SensorId`, `client_id: TenantId`, `auth: Box<dyn SensorAuth>`, `http_client: reqwest::Client`, `backoff_config: BackoffConfig` | Each adapter instance is bound to exactly one (Client, Sensor) pair. The `SensorAuth` trait is sealed -- cross-sensor auth composition is prevented at the type level. |
| **Credential** | A secret value (API key, OAuth2 client secret, bearer token, cookie) stored in the credential store, namespaced by `(client_id, sensor_id, credential_name)`. | `client_id: TenantId`, `sensor_id: SensorId`, `credential_name: String` (e.g., "client_secret", "api_key"), `value: SecretString` (never logged) | Credential names validated against `[a-zA-Z0-9_\-\.]+` (prevents path traversal). Values encrypted at rest (AES-256-GCM in file backend, OS keyring otherwise). Cross-client credential access is a type error via `TenantId` scoping. |
| **DataSource** | A specific data feed within a sensor (e.g., CrowdStrike alerts, Claroty devices, Armis vulnerabilities). Implements the `DataSource<T>` trait for paginated fetching. | `source_id: SourceId`, `sensor_id: SensorId`, `record_type: String` (e.g., `crowdstrike_alert`), `cursor_type: CursorType` | Each DataSource produces records of a single type. The `record_type` string follows `<sensor>_<entity>` naming. Cursor semantics are DataSource-specific (2-tuple vs 3-tuple). |
| **OcsfEvent** | A sensor record normalized to OCSF v1.x via the DynamicMessage protobuf wrapper. Provides runtime field access across all 83 OCSF event classes without per-class code generation. | `event_class: u16`, `message: DynamicMessage`, `raw_extensions: serde_json::Value`, `source_sensor: SensorId`, `source_record_type: String` | OCSF schema version is fixed per Prism release. Field resolution follows four tiers: Prism metadata fields, proto descriptor fields, unmapped JSON blob, None. `raw_extensions` preserves vendor fields not mappable to OCSF. |
| **Cursor** | An ephemeral pagination token for navigating multi-page sensor query results within a single query session. Opaque to the agent ŌĆö internal structure varies by sensor adapter. Not persisted to disk; discarded when the query completes or the session ends. | `token: String` (opaque, base64-encoded adapter-specific state), `client_id: TenantId`, `sensor_id: SensorId`, `source_id: SourceId`, `created_at: DateTime<Utc>`, `expires_at: DateTime<Utc>` (created_at + 600s default TTL) | Cursors are ephemeral and in-memory only ŌĆö no disk persistence, no survival across server restart. Cursors expire after 600 seconds (configurable) to prevent stale pagination sessions from consuming memory. The 600s TTL (vs 300s for confirmation tokens) accounts for the fact that multi-page result navigation is interactive and may involve the analyst reviewing results between page fetches; 300s would be too aggressive for large result sets. Maximum 200 active (non-expired) cursors enforced at any time; new cursor creation beyond this cap returns `E-STATE-001` with suggestion to complete or abandon existing pagination sessions. Expired cursors are lazily cleaned up on each new cursor creation request. There is no forward-only progress invariant across separate queries; the same query may be re-issued for fresh results. Cursor tokens are valid only within the session that created them. |
| ~~**QueryFingerprint**~~ | **REMOVED** ŌĆö No longer needed. With ephemeral pagination cursors (no disk persistence), there is no stored cursor state to invalidate on config drift. Query parameters are validated per-invocation. | N/A | N/A |
| **FeatureFlag** | A capability path in the hierarchical feature flag system (e.g., `sensor.crowdstrike.containment`). Resolved via two tiers: compile-time cargo features and runtime TOML per-client config. | `path: String` (dot-separated hierarchy), `compile_time_enabled: bool`, `runtime_enabled: bool` | Both tiers must permit the operation. Resolution walks the path hierarchy upward. Final fallback is always deny. Flag evaluation is audit-logged for write operations. |
| **ClientCapability** | The resolved capability set for a specific Client, built at config load time by merging profile defaults with `[clients.{id}.capabilities]`. Stored as a `BTreeMap<String, Effect>` where `Effect` is `Allow` or `Deny`. Most-specific capability path wins during resolution; deny overrides allow at the same specificity. | `client_id: TenantId`, `rules: BTreeMap<String, Effect>` (where `Effect` is `Allow \| Deny`) | Resolution walks from the exact capability path upward through parent segments; the first matching rule determines the effect. If no rule matches, implicit deny applies. Deny at the same or more-specific level overrides allow. Re-evaluated on client context switch; triggers `notifications/tools/list_changed`. An `explain(capability) -> CapabilityExplanation` method returns the resolution trace for audit and debugging. |
| **ConfirmationToken** | A time-bounded, single-use token authorizing an irreversible write operation. Returned by the first call to a high-risk write tool; consumed by `confirm_action`. In-memory only ŌĆö tokens are lost on server restart. | `token_id: String` (cryptographic random), `client_id: TenantId` (the client this token authorizes action for ŌĆö prevents cross-client token replay), `tool_name: String` (the originating write tool, e.g., `crowdstrike_contain_host`, `set_credential`), `action_params: serde_json::Value` (the original parameters for re-execution by `confirm_action`), `action_summary: String`, `action_hash: String` (SHA-256 of action params), `created_at: DateTime<Utc>`, `expires_at: DateTime<Utc>` (created_at + 300s), `consumed: bool` | Tokens expire after 300 seconds. A token is single-use: once consumed, it cannot be reused. The `action_hash` ensures the confirmed action matches the original request. The `client_id` must match the `client_id` provided to `confirm_action` ŌĆö mismatches return a structured error. The `tool_name` and `action_params` fields enable `confirm_action` to re-dispatch the original write operation without the caller needing to re-supply parameters. Expired or consumed tokens return a structured error guiding the agent to re-request. Maximum 100 active (unexpired, unconsumed) tokens at any time; new token requests beyond this cap are rejected with a structured error. Tokens are in-memory only ŌĆö server restart loses all tokens (agent receives `E-FLAG-008` and must re-request). |
| **CacheEntry** | An in-memory cached response for a sensor query, keyed by `(client_id, sensor_id, query_hash)`. Used by the response caching subsystem (CAP-014) to avoid redundant sensor API calls within the TTL window. | `client_id: TenantId`, `sensor_id: SensorId`, `query_hash: String` (SHA-256 of canonicalized query parameters ŌĆö includes: filter params (severity, status, time_range, source_id), sort params, and page_size (cache stores paginated responses, so different page sizes produce different payloads); excludes: cursor (pagination state changes per page but the underlying query is the same), force_refresh (bypass flag, not a query parameter)), `value: Vec<u8>` (serialized response), `ttl: Duration` (60s alerts, 300s devices ŌĆö configurable), `created_at: DateTime<Utc>`, `hit_count: u64` | Cache entries are in-memory only ŌĆö lost on server restart. LRU eviction when entry count exceeds the configurable per-client-per-sensor bound (default 50). Write operations invalidate all entries for the affected `(client_id, sensor_id, source_id)` tuple synchronously. `force_refresh: true` bypasses and replaces the cached entry. `hit_count` is tracked for metrics exposed via `check_sensor_health`. |
| **AuditEntry** | A structured log record emitted for every MCP tool invocation. Written via `tracing::info!` with structured fields, consumed by the Vector pipeline. | `timestamp: DateTime<Utc>`, `trace_id: String`, `client_id: TenantId`, `tool_name: String`, `parameters: serde_json::Value` (secrets redacted), `user_identity: String`, `result_summary: String`, `capability_checks: Vec<CapabilityCheckEvent>`, `safety_flags: Vec<String>` | Audit entries are append-only. Every tool invocation produces exactly one AuditEntry. Write operations include capability check details. Credential values are never present in any audit field. `user_identity` is resolved at startup via a three-tier chain: (1) TOML config `analyst_id` field, (2) `PRISM_ANALYST_ID` env var, (3) OS username detection. The resolved value is immutable for the session lifetime. |
| **ToolResponse** | The MCP tool response envelope wrapping all Prism query results. Uses `structuredContent` for machine-parseable data and `content[].text` for LLM-consumable summary. | `_meta: ResponseMeta` (tool, data_source, query_time, trust_level, safety_flags, pagination), `results: Vec<serde_json::Value>`, `content_summary: String` | All responses include `trust_level: "untrusted_external"` for sensor data. Safety flags are populated when suspicious patterns are detected in string fields. Pagination metadata (cursor, has_more, total_count) is always present. |
| **ClientConfig** | The deserialized TOML configuration for a single Client, including sensor connections, credential references, and capability overrides. | `client_id: TenantId`, `display_name: String`, `sensors: HashMap<SensorId, SensorConfig>`, `capabilities: Option<CapabilityOverrides>` | Config is validated at startup with multi-error aggregation. Missing required fields produce actionable errors with exact TOML paths. Capability overrides merge with `[defaults.capabilities]` using more-specific-wins semantics. |
