# Domain Events and Processing Stages

This section documents the key domain events in Prism's runtime lifecycle. Each event represents a significant state transition or processing stage. Events are listed in the order they typically occur during a query or write operation.

| Event/Stage | Trigger | Preconditions | Outcomes |
|-------------|---------|---------------|----------|
| **MCP Tool Invocation** | AI agent sends a JSON-RPC 2.0 `tools/call` request via stdio transport. The request includes `tool_name` and `arguments` containing at minimum a `client_id` parameter. | MCP session is established. The tool is registered in `tools/list` (passes both compile-time and runtime feature flag checks). Arguments conform to the tool's `Parameters<T>` JSON Schema. | 1. AuditEntry is initialized with timestamp, trace_id, tool_name, and parameters (secrets redacted). 2. Input validation via schemars-derived JSON Schema. 3. `client_id` is resolved to a `TenantId`. 4. Dispatch proceeds to the appropriate tool handler. On validation failure, structured error returned with `category: "validation"`, `failed_parameter`, and `allowed_values`. |
| **Client Context Resolution** | A `client_id` parameter is extracted from tool arguments. If `client_id` is null, this triggers cross-client fan-out (CAP-002). | The `client_id` must exist in TOML config or be null. | 1. `ClientConfig` is loaded for the specified client. 2. `ClientCapabilities` are resolved (defaults merged with client overrides). 3. For write operations, capability check is performed and audit-logged. 4. For null `client_id`, all configured clients are enumerated for fan-out. If `client_id` is unknown, `PrismError::Config` with "client '{id}' not found in configuration" message. |
| **Feature Flag Evaluation** | A write tool handler checks whether the requested capability is enabled for the resolved client. This stage is skipped for read-only tools. | Tool handler has resolved `client_id` and the requested capability path (e.g., `sensor.crowdstrike.containment`). | 1. Compile-time gate: `#[cfg(feature = "crowdstrike-write")]` -- if absent, the tool was never registered. 2. Runtime gate: `ClientCapabilities::is_enabled("sensor.crowdstrike.containment")`. 3. `CapabilityCheckEvent` emitted via `tracing::info!` with client_id, capability path, and result (Allowed/Denied/DryRun). 4. If denied, structured error returned explaining which capability to enable and the TOML config path. |
| **Credential Resolution** | A sensor adapter needs authentication credentials to call a sensor API. Resolution follows the chain: `_FILE` env var, direct env var, keyring, then error. | `(client_id, sensor_id, credential_name)` tuple is known. At least one credential source must be configured. | 1. Check `PRISM_{SENSOR}_{CREDENTIAL}_FILE` env var; if present, read file contents. 2. Fallback to `PRISM_{SENSOR}_{CREDENTIAL}` env var. 3. Fallback to `CredentialStore::get(client_id, sensor_id, credential_name)`. 4. Credential access is audit-logged (client_id, sensor_id, credential_name -- never the value). 5. On failure, `PrismError::Credential` with actionable message listing the resolution chain attempted. |
| **Sensor API Query** | A sensor adapter calls the external sensor API (CrowdStrike, Cyberint, Claroty, or Armis) to fetch data. Each sensor has specific API patterns: CrowdStrike uses two-step fetch (QueryV2 then PostEntities), Cyberint uses cookie auth, Claroty uses POST-for-read, Armis uses AQL. | Credentials are resolved. HTTP client is configured with appropriate auth middleware. Cursor state (if resuming) is loaded from FileStore. | 1. HTTP request(s) sent to sensor API with backoff (2s base, 30s max). 2. Response parsed with lenient JSON decoding (`#[serde(default)]`, no `DisallowUnknownFields`). 3. Polymorphic IDs handled (Claroty returns string OR number). 4. Records extracted and cursor derived from last record. 5. On HTTP 429, backoff with `Retry-After` header respected. On HTTP 401/403, `PrismError::Sensor` with `category: "authentication"`. On timeout, retry with exponential backoff. |
| **OCSF Normalization** | Raw sensor records are transformed to OCSF v1.x protobuf messages via per-sensor mapper implementations. Occurs after successful sensor API query. | Raw sensor records are deserialized. OCSF protobuf schema is compiled into the binary (via build.rs). Per-sensor field mappings are defined. | 1. Sensor-specific mapper extracts fields from the raw record. 2. Fields are set on a `DynamicMessage` instance using four-tier resolution. 3. Vendor-specific fields not in OCSF schema are preserved in `raw_extensions` JSON blob. 4. `enum-value-map.json` provides human-readable captions for OCSF enum values. 5. Unmappable required fields logged as warnings. 6. The resulting `OcsfEvent` includes both the DynamicMessage and the raw extensions. |
| **Forward Progress Validation** | After deriving a new cursor from fetched records, the system validates that the new cursor is greater than or equal to the stored cursor. | A new cursor has been derived. A stored cursor exists (or this is the first fetch, where any cursor is valid). | 1. `ensure_forward_progress(stored, new)` compares cursor components. 2. If valid, proceed to persistence. 3. If regression detected, `PrismError::CursorRegression` halts the source. This catches sensor API bugs that could cause backward pagination. Invariant DI-001 is enforced here. |
| **State Persistence** | After successful data processing (query response formatted or batch delivered), cursor state and batch receipt are written to FileStore. This MUST occur before in-memory state is updated (DI-009). | Forward progress validated. FileStore directory exists for `(client_id, sensor_id, source_id)`. | 1. New cursor + query fingerprint serialized to JSON. 2. Written to temp file in same directory as target. 3. Temp file fsynced. 4. Renamed over target file (atomic). 5. Batch receipt appended (bounded to most recent N entries). 6. Only after successful rename: in-memory cursor updated. On failure: cursor does not advance, next cycle retries from same position. |
| **Write Confirmation Flow** | For irreversible write operations (e.g., host containment), the first tool call returns a `ConfirmationToken` instead of executing the action. The agent must call `confirm_action(token)` to proceed. | Feature flag evaluation passed (capability allowed). Tool is classified as irreversible write (risk tier: high). | 1. First call: action parameters validated, dry-run preview generated (what would happen), `ConfirmationToken` created with 300s expiry and action hash. Token stored in-memory. 2. Token returned to agent with action summary. 3. Second call (`confirm_action`): token validated (not expired, not consumed, hash matches). 4. If valid: action executed against sensor API, token marked consumed, full audit entry emitted. 5. If expired/consumed: structured error with `suggestion: "call the original tool again to get a new token"`. |
| **Audit Entry Emission** | At the conclusion of every tool invocation (success or failure), the complete AuditEntry is emitted as a structured JSON log event via `tracing::info!`. | Tool invocation has completed (or failed). All capability checks, sensor calls, and safety flag evaluations are recorded. | 1. AuditEntry fields populated: timestamp, trace_id, client_id, tool_name, redacted parameters, result summary, capability checks, safety flags. 2. Emitted via `tracing::info!` with structured fields. 3. Consumed by tracing-subscriber JSON formatter. 4. Written to stderr (stdout reserved for MCP). 5. Compatible with Vector pipeline for centralized audit aggregation. |
| **Tool List Refresh** | When the analyst switches client context (calls a tool with a different `client_id` than the previous call), the available tools may change due to per-client capability differences. | Previous tool call used a different `client_id`. The new client has different `ClientCapabilities` than the previous one. | 1. New `ClientCapabilities` resolved for the target client. 2. Tool registry re-evaluated: write tools enabled/disabled per new capabilities. 3. `notifications/tools/list_changed` sent via MCP protocol. 4. MCP client (Claude Code) re-fetches `tools/list`. 5. Agent sees updated tool list reflecting the new client's permissions. |
