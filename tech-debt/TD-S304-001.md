# TD-S304-001: ConfirmationTokenStore full integration for alias delete

**Story:** S-3.04
**Status:** resolved (implemented in pass-1: CR-008 — real two-step token lifecycle via token_store.generate/consume)
**Severity:** tech_debt

## Description

`alias_tools::delete_alias` uses a synthetic `ConfirmationToken` instead of calling
`ConfirmationTokenStore::generate()` + `ConfirmationTokenStore::consume()`. The full
token lifecycle (generate → return token_id to caller → caller passes token_id back →
consume + validate action_hash) is deferred.

## Current Behavior

`delete_alias` accepts any non-None `token_id` as confirmation. It builds a synthetic
`ConfirmationToken` to satisfy `AliasStore::delete`'s type signature. No actual
cryptographic token validation occurs.

## Required Fix

1. When `token_id` is `None`: call `token_store.generate(...)` to issue a real
   `ConfirmationToken` with the correct `action_hash` (SHA-256 of alias name + scope).
2. When `token_id` is `Some(id)`: call `token_store.consume(id, client_id, tool, params)`
   to validate and consume the token (BC-2.04.010).
3. Update the response to include the `token_id` in the first-call JSON.

## Blocked By

`ConfirmationTokenStore::generate` requires `action_params: serde_json::Value` and
returns a `ConfirmationToken` with a crypto-random `token_id`. The delete tool needs
to store the `token_id` and return it to the MCP caller, who re-submits it on
confirmation. The MCP routing layer (not yet implemented for aliases) handles this.
