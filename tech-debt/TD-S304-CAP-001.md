# TD-S304-CAP-001: Cap total alias count to prevent unbounded aliases.toml growth

**Story:** S-3.04
**Status:** open
**Severity:** tech_debt
**Filing:** S-3.04 local adversary pass-1 OBS-004

## Description

`AliasStore::create_or_update` does not enforce a maximum alias count. A tenant could
create an unbounded number of aliases, causing `aliases.toml` to grow without limit and
potentially exhausting disk space or slowing atomic rename operations.

## Current Behavior

No cap is enforced. `create_or_update` adds entries up to system limits.

## Required Fix

Add a configurable `max_aliases` field to `AliasStore` (defaulting to 1,000 per the
domain-spec guidance) and check it in `create_or_update` before new entries:

```rust
if existing_idx.is_none() && self.entries.len() >= self.max_aliases {
    return Err(PrismError::AliasLimitExceeded {
        count: self.entries.len(),
        limit: self.max_aliases,
    });
}
```

Add `E-ALIAS-007` (alias store at capacity) to the error taxonomy in `prism_core::error`.
