# TD-S302-006: Fuzz-smoke timeout vs heavy deps growth

| Field | Value |
|-------|-------|
| **ID** | TD-S302-006 |
| **Severity** | P3 |
| **Status** | OPEN |
| **Filed** | 2026-05-06 |
| **Source** | PR-129 fuzz-smoke CI timeout; sibling of TD-VSDD-058 |
| **Story** | S-3.02 |

## Summary

Each story that adds heavy transitive dependencies to `prism-query` increases the
`fuzz-smoke-vp021` CI job's cold-build time under libfuzzer + AddressSanitizer
instrumentation. This is a recurring pattern: S-3.01 required a timeout bump from
12 → 20 min (TD-VSDD-058); S-3.02 required a further bump from 20 → 35 min.

## Root Cause

The `fuzz-smoke-vp021` CI job builds the entire `prism-query` transitive closure
from scratch under the sanitizer profile (separate cache namespace from the main
build). AddressSanitizer instrumentation compiles every dependency individually,
so each new heavy crate added to `prism-query` linearly extends the cold-build time.

S-3.02 added 9 new transitive dependencies:
- `datafusion 53.1`
- `arrow 58`
- `tokio`
- `tracing`
- `async-trait`
- `proptest`
- `prism-sensors`
- `prism-ocsf`
- `prism-credentials`

Observed: CI runs 25471243470 and 25471244604 both terminated at ~20 min with
`cargo-fuzz`, `cargo`, `build-script-build`, `c++`, and `cc1plus` orphan processes
still in the sanitizer build phase — the 60s fuzz run was never reached.

## Remediation Applied

PR-129 bumped `timeout-minutes: 20 → 35` in `.github/workflows/ci.yml` for the
`fuzz-smoke-vp021` job. This provides headroom for S-3.07 and DTU integrations
while still failing fast on genuine hangs.

## Future Risk

The 35-min budget will eventually be exhausted as the dep graph grows. Current
trajectory (5 min added per major story wave) suggests the ceiling will be hit
again around S-3.07 or the first DTU integration story.

## Proposed Mitigations (for future consideration)

1. **Feature-gate fuzz target deps**: Gate `datafusion`/`arrow`/heavy analytics
   deps behind a Cargo feature in `prism-query`. The fuzz target could build with
   `--no-default-features` to avoid compiling the full analytics stack under
   AddressSanitizer — reducing cold-build time significantly.

2. **Separate sanitizer cache key**: Add `datafusion`/`arrow` to the
   `rust-cache` `cache-on-failure: true` path for the `fuzz-smoke` key to improve
   warm-build hit rates across PR runs.

3. **Dynamic timeout based on dep count**: Not directly supported by GitHub Actions,
   but could be approximated by a pre-step that reads `Cargo.lock` dep count and
   sets `ACTIONS_STEP_DEBUG` or adjusts a matrix variable.

4. **Move fuzz target to a lightweight crate**: Extract the fuzz target into a
   thin wrapper that depends only on `prism-query`'s parser core (not the full
   sensor + analytics dep graph).
