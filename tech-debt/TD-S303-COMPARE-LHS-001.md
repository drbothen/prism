# TD-S303-COMPARE-LHS-001: Compare arm lhs silently ignored in predicates_from_ast

**Story:** S-3.03
**Status:** open
**Severity:** tech_debt

## Description

In `crates/prism-query/src/explain.rs`, the `Predicate::Compare { lhs, op, rhs }` arm
of `predicate_to_exprs` uses `lhs.clone()` when constructing `Expr::Compare`, but the
`column_name` passed to `predicate_as_string` is extracted independently by
`classify_predicates` (via `column_name_from_expr`). The result is that a complex lhs
expression (e.g. `LOWER(field) = 'value'`) is passed through to the `Expr::Compare`
node but the rendered string uses the column_name derived by the pushdown classifier,
which may not match the lhs if the lhs is not a simple field path.

O-LOCAL-NEW-2: `predicate_to_exprs` passes `lhs: lhs.clone()` into the `Expr::Compare`
node, but `predicate_as_string` uses the `column_name` field from the `Predicate` struct
(set by `classify_predicates`). In practice, this only matters for non-field-path lhs
expressions that the current parser does not produce, so the impact is latent.

## Current Behavior

For all predicate forms currently generated by the PrismQL parser, lhs is always a
simple field path, so `column_name_from_expr(lhs)` and `column_name` agree. The
divergence would surface only with complex lhs expressions (function calls, casts)
that are not yet supported by the parser.

## Required Fix (if user-required)

Either:
1. Remove the `lhs` field from the `Expr::Compare` node in `predicates_from_ast` output
   and rely solely on `column_name` from the classifier (simpler, less information).
2. Make `predicate_as_string` use the `lhs` field from the `Expr::Compare` node directly
   rather than the separately-derived `column_name`, ensuring full round-trip fidelity
   for complex lhs expressions.

Option 2 is preferred once complex lhs parsing lands.

## Deferred In

S-3.03 adversarial local pass-2 (O-LOCAL-NEW-2). No user-visible impact at current
parser scope. Filed for triage when complex lhs expression parsing is added.
