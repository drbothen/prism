## Summary

AC-9 cold-start integration test (`test_ac9b_cold_start_descriptor_tags_rows_as_live`) only
verifies that `inject_source_type` sets `_source_type = "live"` on cold-start descriptor rows.
It does NOT cover the full execution path because `run_materialization_pipeline` body is
currently `todo!()`.

## Deferred Coverage

When the `run_materialization_pipeline` body is implemented (post-stub implementation cycle),
add the following full AC-9 test:

1. **Mock SensorAdapter call** — assert the sensor adapter is invoked on cold-start (no buffer
   hit), verifying the live-fetch code path executes.
2. **EventBufferStore write assertion** — assert that fetched rows are written to the buffer
   store after a successful cold-start live fetch.
3. **INFO log capture** — use a `tracing` test subscriber to assert that the INFO log
   `"cold-start: fetching live data from sensor"` (or equivalent) fires exactly once.

## Context

The current stub implementation intentionally limits AC-9 coverage to structural/tag-injection
tests that do not require a live pipeline body. The test was renamed from
`test_ac9b_cold_start_triggers_live_fetch_and_writes_to_buffer` to
`test_ac9b_cold_start_descriptor_tags_rows_as_live` to accurately reflect what is actually
verified.

## Target

Post-stub `run_materialization_pipeline` implementation cycle (story TBD in Wave 4 or Wave 5).
