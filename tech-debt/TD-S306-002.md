# TD-S306-002: reject_write_verbs_in_filter reuses E-QUERY-010 error code

**Filed:** 2026-05-06
**Story:** S-3.06 (PR #130 review, F-PR130-CR-002)
**Severity:** tech_debt
**Status:** open / deferred

## Finding

`reject_write_verbs_in_filter` (in `filter_parser.rs`) returns `E-QUERY-010`
("Internal Prism table is write-protected") when a write verb appears in filter
mode. This is the same error code used by `is_internal_prism_table` in
`sql_parser.rs` for a completely different condition.

The two conditions are:
1. `E-QUERY-010` — SQL DML targeting a `prism_*` internal table (write-protected)
2. Filter mode write-verb rejection — grammatically invalid in filter mode

Using the same code for both conditions makes programmatic error handling
ambiguous: callers cannot distinguish "you tried to write to a protected table"
from "write verbs are not allowed in filter mode."

## Location

- `crates/prism-query/src/filter_parser.rs` — `reject_write_verbs_in_filter`
- `crates/prism-query/src/sql_parser.rs` — `is_internal_prism_table`
- `crates/prism-query/src/error.rs` — `E-QUERY-010` constructor

## Why Deferred

Introducing a new error code (`E-QUERY-025` or similar) requires:
1. Adding a new error variant + constructor in `error.rs`.
2. Updating `reject_write_verbs_in_filter` to emit the new code.
3. Updating `BC-2.11.004` (parse behavioral contract) to enumerate the new code.
4. Updating all tests that assert `E-QUERY-010` for filter-mode rejection.

This is not security-blocking — the rejection itself is correct (filter mode
correctly rejects write verbs). The error code collision is a minor
programmatic-API inconsistency.

## Fix Scope

1. Add `E-QUERY-025: write verb not valid in filter mode` to `error.rs`.
2. Update `reject_write_verbs_in_filter` to emit `E-QUERY-025`.
3. Update BC-2.11.004 §error-codes to enumerate `E-QUERY-025`.
4. Update test assertions that check for `E-QUERY-010` in filter-mode contexts.

**Target story:** S-3.07 or a dedicated error-code cleanup maintenance story.
