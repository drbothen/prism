# TD-S305-007: hit_count verification requires cache metrics inspection API

**Filed:** 2026-05-06
**Story:** S-3.05
**Severity:** tech_debt
**Status:** resolved

## Finding

`test_BC_2_07_003_hit_count_incremented_on_cache_hit` verifies that `hit_count` on a
cache entry is incremented on each `get()` call. The test body had a `todo!()` at line 366.

## Resolution

Per-entry `hit_count` design was rejected in S-3.05 pass-2 (CR-005). An aggregate
`QueryCache::total_hits()` backed by `AtomicU64` was implemented instead. BC-2.07.003 v4.4
ratifies this design. The test `test_BC_2_07_003_total_hits_incremented_on_cache_hit`
is complete with no `todo!()`. TD-S305-007 is closed.

## Why Deferred (original)

`QueryCache::get()` incremented `hit_count` in the `CacheEntry` on each hit, but there was
no public API to inspect the `hit_count` of a cached entry directly. The test would have needed:
1. A `QueryCache::get_entry_with_meta(key)` method returning `CacheEntry`, OR
2. A `QueryCache::hit_count_for(key)` method.

These metrics inspection APIs were considered out of scope for S-3.05 but needed for the
`check_sensor_health` observability story.

## Resolution Path (completed)

Design pivot: per-entry `hit_count` replaced by aggregate `QueryCache::total_hits(): u64`
using `AtomicU64`. This is simpler, sufficient for observability, and avoids the need for
a `CacheEntry`-exposing API. BC-2.07.003 v4.4 documents this as the ratified design.
