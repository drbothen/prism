# TD-S306-003: DmlNode.filter carries sentinel Literal(Bool(true)), not the parsed predicate

**Filed:** 2026-05-06
**Story:** S-3.06 (PR #130 review, security review observation)
**Severity:** tech_debt
**Status:** open / deferred

## Finding

The `filter` field in `DmlNode` for `DELETE` and `UPDATE` statements carries a
sentinel value `Literal(Bool(true))` (presence indicator) rather than the actual
parsed predicate expression. The `check_unbounded_write` function parses and
validates the WHERE clause at parse time to enforce E-QUERY-022 (unbounded write
rejection), but the parsed predicate is discarded after the check. Only the
sentinel is stored in `DmlNode.filter`.

As a result, S-3.07 (write execution) must re-parse the original query string to
recover the actual WHERE predicate for execution dispatch. This defeats the goal
of `DmlNode` carrying "sufficient context for S-3.07 to dispatch without
re-parsing" (BC-2.11.004 architecture note in `write_ast.rs`).

**Note:** As of PR #130 pass-4 (F-PR130-SEC-003), `DmlNode.filter` now carries
the ACTUAL parsed predicate — this tech-debt item was closed by the SEC-003 fix.
The pr-description security review note (line ~156) referenced this as an open
concern, but SEC-003 resolved it. This file is preserved for historical
traceability.

## Status

**CLOSED** — resolved by F-PR130-SEC-003 (pass-3 fix: `DmlNode.filter` now
stores the actual predicate, not a sentinel).
