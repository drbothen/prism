# TD-S302-001: Scoping performance — Vec<OrgSlug> to HashSet + &[OrgSlug] borrows

**Filed:** 2026-05-06
**Story:** S-3.02 (PR #129 review, F-PR129-CR-012/013)
**Severity:** tech_debt
**Status:** open / deferred

## Finding

Two functions in `crates/prism-query/src/scoping.rs` use `Vec<OrgSlug>` where a `HashSet`
or a borrowed `&[OrgSlug]` would be more appropriate:

1. `intersect_query_client_predicates`: accepts `Vec<OrgSlug>` by value and performs
   O(n×m) intersection. Should accept `&[OrgSlug]` and use `HashSet` for the inner lookup.
2. `resolve_clients`: builds a `Vec` and performs linear scans. Could use `HashSet` for
   O(1) existence checks.

## Location

- `crates/prism-query/src/scoping.rs` — `resolve_clients`, `intersect_query_client_predicates`

## Why Deferred

Current implementation is correct and the client scope is bounded (MSSPs typically
manage 10–100 clients, not millions). Performance impact is negligible in practice.
The fix is straightforward but crosses a public API surface (`ClientRegistry`) that
future stories may need to stabilize first.

## Fix Scope

1. Change `intersect_query_client_predicates` signature to `&[OrgSlug]` borrows.
2. Build `HashSet<&OrgSlug>` from the tool-scope slice for O(1) membership checks.
3. Update `resolve_clients` to use `HashSet` for existence checks.
4. Update all call sites (no external callers since both are `pub(crate)`).

**Target story:** Maintenance sweep or dedicated S-3.X performance pass.
