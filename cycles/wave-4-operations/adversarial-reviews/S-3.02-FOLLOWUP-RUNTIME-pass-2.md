## Novel Pass-2 Findings

### F-LP2-CRIT-1: Capability gate bypassed via subquery references in WHERE/HAVING/ORDER BY

**Severity:** CRITICAL **Confidence:** HIGH (Standing Rule 3 #3: documented-but-unenforced via incomplete AST coverage)

`extract_source_names_for_capability_check` at materialization.rs:808-828 walks only top-level `from.source.raw` and `joins`. Does NOT recurse into:
- `InSubquery { subquery: Box<SqlQuery> }` in WHERE clauses
- HAVING clause subqueries
- Compound expressions referencing `prism_audit` indirectly

**Concrete attack vector at HEAD `99d49b20`:**
```sql
SELECT alert_id 
FROM crowdstrike_detections 
WHERE alert_id IN (SELECT trace_id FROM prism_audit)
```
- `check_internal_table_capabilities` (engine.rs:412) sees only `crowdstrike_detections` → passes WITHOUT requiring `Capability::AuditRead`
- `register_internal_tables` (engine.rs:418) unconditionally registers `prism_audit` for DataFusion
- DataFusion plans the subquery → calls `RocksDbTableProvider::scan` for `prism_audit`
- `scan()` does NOT consult capabilities (closure moved check to `execute_inner` per "approach b")
- **Audit data leaks to caller without `AuditRead` capability**

This is a NOVEL pass-2 finding — fix-pass-1 introduced this regression by choosing "approach b" (pre-execution gate) over "approach a" (gate inside scan()). Approach b only works if source-name extraction is exhaustive across all AST positions.

**Trace:** BC-2.15.011, Standing Rule 3 #3.
**Recommendation:** Production-grade fix requires DEFENSE-IN-DEPTH:
- (a) Make `extract_source_names_for_capability_check` recursively walk all subqueries (In/InSubquery, HAVING subqueries, etc.)
- (b) ADDITIONALLY add capability check inside `RocksDbTableProvider::scan()` using DataFusion `Session` extensions to thread caller's capability set
Both layers needed. Approach b alone is insufficient (this finding); approach a alone has plumbing complexity. The right production-grade fix is BOTH.

### F-LP2-MED-2: in_query_cache key omits push_down_params per BC-2.11.005

**Severity:** MEDIUM **Confidence:** HIGH

materialization.rs:363-368:
```rust
let cache_key = format!(
    "{}:{:?}:{}",
    target.client_id.as_str(),
    target.sensor_type,
    &target.source_table
);
```

But materialization.rs:161-163 doc states cache "is keyed on `(client_id, sensor_id, source_id, push_down_params)`". BC-2.11.005 mandates `push_down_params` in the key.

Without it, two AST stages referencing same `(client, sensor, source_table)` with different WHERE filters share cached results — incorrect data leakage between query stages. Self-joins or multi-stage Pipe queries hit this.

**Trace:** BC-2.11.005, F-LP1-MED-2 partial-closure.
**Recommendation:** Include `where_filters` in cache key.
