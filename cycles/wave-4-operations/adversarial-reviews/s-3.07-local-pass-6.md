### F-PASS6-MED-002: `write_dispatch.rs:359` hardcodes `error_code = "E-SENSOR-070"` regardless of which `SensorError` variant the adapter returned

**Severity**: MEDIUM
**Confidence**: HIGH (latent — currently inert; activates on first real adapter override)

**Locations**:

| File | Lines | Content |
|------|-------|---------|
| `crates/prism-query/src/write_dispatch.rs` | 354-362 | `error_code: "E-SENSOR-070".to_string()` unconditional in fan_out error arm |
| `crates/prism-sensors/src/adapter.rs` | 194-208 | `SensorError` enum: `WriteNotImplemented`, `SensorAuthFailed`, `SensorRateLimited`, `SensorResponseParse` — each has different semantic code (E-SENSOR-070/001/020/003 respectively) |

**Problem**: Currently inert because all adapters use the default `WriteNotImplemented` body and no real override exists. However, the first real adapter override (W3-FIX-S307-001 per the story plan) will activate this code path. When activated, every auth failure, rate-limit, or parse error in the write fan_out path will be stamped `error_code = "E-SENSOR-070"` in the structured audit field while the human-readable `detail` string carries the truth (e.g., `"E-SENSOR-001: auth failed: ..."`).

Net effect: audit consumers and SIEM systems correlating on the structured `error_code` field will mis-bucket all write errors as "write not implemented" when they are actually auth failures, rate limits, or parse failures. The detail field has truth; the typed field lies.

This is a NEW defect class not previously surfaced in passes 1-5. It is a fan_out variant↔code mapping defect.

**Remediation**:
1. Extract `SensorError::error_code(&self) -> &'static str` helper method on the `SensorError` enum in `crates/prism-sensors/src/adapter.rs`:
   ```rust
   pub fn error_code(&self) -> &'static str {
       match self {
           SensorError::WriteNotImplemented => "E-SENSOR-070",
           SensorError::SensorAuthFailed => "E-SENSOR-001",
           SensorError::SensorRateLimited => "E-SENSOR-020",
           SensorError::SensorResponseParse => "E-SENSOR-003",
           // ... other variants
       }
   }
   ```
2. Replace hardcoded string in `write_dispatch.rs:359`: change `error_code: "E-SENSOR-070".to_string()` to `error_code: e.error_code().to_string()`.
3. Concentrates variant↔code mapping in one source-of-truth (the enum definition), eliminating the scatter.
4. Add unit test asserting each `SensorError` variant maps to its canonical code string.

### F-PASS6-LOW-001: `WritePlan::from_write_node` and `from_dml_node` use `format!("{:?}", literal_value)` for params — leaks Debug repr into audit trail and token hash

**Severity**: LOW
**Confidence**: MEDIUM

**Locations**:

| File | Line | Content |
|------|------|---------|
| `crates/prism-query/src/write_pipeline.rs` | 85 | `format!("{:?}", arg.value)` in `from_write_node` |
| `crates/prism-query/src/write_pipeline.rs` | 141 | `format!("{:?}", a.value)` in `from_dml_node` |

**Problem**: Stringifies `Literal` via `Debug` formatter. A `Literal::String("acknowledged")` becomes `"String(\"acknowledged\")"` in the audit params map rather than the clean value `"acknowledged"`. Similarly `Literal::Integer(42)` becomes `"Integer(42)"` rather than `"42"`.

The token hash is internally consistent — both generation and consumption use the same formatter, so no hash break. The issue is audit trail observability: operators querying "what value did the analyst write?" receive Debug-formatted internal Rust type names rather than human-readable values. SIEM rules and audit log queries expecting `params["value"] == "acknowledged"` will silently fail.

This is a NEW defect class not previously surfaced in passes 1-5. It is a Debug-format observability leak.

**Remediation**:
1. Replace `format!("{:?}", arg.value)` with a purpose-built `Literal::display_value()` method or `impl Display for Literal`:
   - `Literal::String(s)` → `s.clone()` (raw string, no quotes)
   - `Literal::Integer(n)` → `n.to_string()`
   - `Literal::Float(f)` → `f.to_string()`
   - `Literal::Boolean(b)` → `b.to_string()`
   - `Literal::Null` → `"null".to_string()`
2. **CAREFUL**: Before changing, grep for all tests asserting on the current Debug-format string representation. If any tests pin `"String(\"...\""` patterns, coordinate the change to update both the formatter and the test assertions in one burst.
3. Consider whether the token hash should be regenerated for existing plans (likely a non-issue since plans are ephemeral and not persisted between sessions).
