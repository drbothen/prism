### F-PASS2-HIGH-001: Three independent `*-write` Cargo features can drift silently

**Severity**: HIGH
**Location**:
- `crates/prism-query/Cargo.toml:105-109`
- `crates/prism-sensors/Cargo.toml:46-50`
- `crates/prism-security/Cargo.toml:19-23`
- `crates/prism-query/src/write_pipeline.rs:298-329` (consumer site)
- `crates/prism-security/src/feature_flag.rs:254-299` (alternate consumer)

**Policy**: POL-4 (semantic anchoring integrity) — BC-2.04.001 declares ONE compile-time feature per sensor.

**Problem**: BC-2.04.001 postcondition states: "all CrowdStrike write operation code... is excluded from the binary via `#[cfg(feature = "crowdstrike-write")]`" — i.e., a single canonical feature. The implementation defines `crowdstrike-write` (and the three sibling features) **independently in three Cargo.toml files**:
1. `prism-query/Cargo.toml:105` — consumed by inline `cfg!(feature = "crowdstrike-write")` in `write_pipeline.rs:300`.
2. `prism-security/Cargo.toml:19` — consumed by `crowdstrike_write_gate()` in `feature_flag.rs:254`.
3. `prism-sensors/Cargo.toml:46` — declared but ZERO source files reference it (orphan flag).

`prism-query`'s `Cargo.toml:26` declares `prism-sensors = { path = "../prism-sensors" }` and `prism-security = { path = "../prism-security" }` with no feature pass-through. Building with `cargo build -p prism-query --features crowdstrike-write` enables prism-query's feature only; prism-security's and prism-sensors's features remain off. The result: the inline `cfg!` gate in `write_pipeline.rs` says "Present" while `prism_security::feature_flag::crowdstrike_write_gate()` (if called) returns "Absent". The two gates can disagree on the same sensor in the same binary.

Additionally, BC-2.04.001's stronger physical-exclusion postcondition ("the code does not exist", "physically cannot execute") is **not** met by `cfg!()` runtime checks — both `cfg!` branches are always compiled. The pass-1 CRIT-3 fix declared the features but did not gate any actual `WriteExecutor` / `WriteDispatcher` / `safety_check` code under `#[cfg(feature)]`, so the entire write code path remains in default-feature binaries.

**Impact**: BC-2.04.001 is the foundational compile-time security gate. With three independent flags, a future contributor enabling `crowdstrike-write` in one Cargo.toml may believe they have enabled the full feature, but the gate state across crates can diverge undetectably. The runtime fall-back (deny-by-default in default builds) masks this drift. A minor refactor that introduced `cfg`-gated write code in any one crate would cause silent drift in the others.

**Remediation**:
1. Make `prism-query`'s `crowdstrike-write` feature transitively enable both `prism-security/crowdstrike-write` AND `prism-sensors/crowdstrike-write`. Update `[features]` in `prism-query/Cargo.toml`:
   ```toml
   crowdstrike-write = ["prism-security/crowdstrike-write", "prism-sensors/crowdstrike-write"]
   ```
2. Replace the inline `cfg!(feature = "crowdstrike-write")` dispatch in `write_pipeline.rs:298-329` with calls to `prism_security::feature_flag::crowdstrike_write_gate()` etc. — single source of truth.
3. Either gate `WriteExecutor`/`WriteDispatcher`/`safety_check` modules under `#[cfg(any(feature = "crowdstrike-write", ...))]` to satisfy BC-2.04.001's physical-exclusion postcondition, OR file a spec amendment narrowing BC-2.04.001 to the runtime-cfg!() interpretation. The current state is neither.

### F-PASS2-HIGH-002: Pass-1 HIGH-6 reverses story-specified Phase 5a/5b ordering without spec amendment

**Severity**: HIGH
**Location**: `crates/prism-query/src/write_dispatch.rs:166-189`

**Policy**: POL-4 (semantic anchoring integrity)

**Problem**: Story Task 6 (S-3.07-write-execution.md:222-246) explicitly orders Phase 5 as:
- **6a**: Audit INTENT record (fail-closed) — "no sensor API is contacted **until the intent is durable**"
- **6b**: Write semaphore acquisition
- **6c**: Write fan-out

The pass-1 HIGH-6 fix reversed this to **5b → 5a → 5c** (semaphore-first), as documented in `write_dispatch.rs:169-181`:
```
HIGH-6: Acquire write semaphore permit BEFORE audit intent write.
This prevents orphaned audit intent records when the semaphore is exhausted.
Phase 5b (semaphore) → Phase 5a (audit) → Phase 5c (sensor) order.
```

The HIGH-6 reasoning is plausible — but the swap reverses an explicit story decision that ordered intent-first specifically so every write **attempt** produces an audit trace, even attempts that fail at the throttle. Under the new order, throttled attempts produce **no audit record at all**, eroding BC-2.05.009's audit-coverage guarantee.

The semaphore in this code path can never realistically `Err` (the comment at 174-176 confirms this: "the semaphore is only closed when the Arc is dropped — which cannot happen while this future is executing"). So the HIGH-6 "orphaned audit record" risk it cites is largely theoretical, while the audit-coverage regression is real.

**Impact**: BC-2.05.009 audit-coverage drift; possible regulatory/compliance gap (analyst attempts that hit the semaphore wait briefly leave no audit trace; in pathological deadlock scenarios with `Semaphore::close()` ever invoked, no intent record exists). Also: this change deviates from the story spec without an ADR or spec amendment — POL-4 violation.

**Remediation**: Either:
1. Restore the story-specified order (intent-first, then semaphore) — accepting the (theoretical) orphaned-intent risk on semaphore close, OR
2. File a spec amendment / ADR documenting the rationale for swapping 5a/5b (with explicit acknowledgment that throttled attempts produce no audit) and update the story Task 6 ordering. The current state — silent reversal in code — violates POL-4.
