### F-PASS3-MED-001: Error-code register skew — E-QUERY-028/029 used with semantics that conflict with architecture catalog [process-gap candidate]

**Severity**: MEDIUM
**Confidence**: HIGH
**Policy**: POL-4 (semantic_anchoring_integrity); POL-1 (append_only_numbering — codes reused with different semantics)
**Locations**:
- `crates/prism-core/src/error.rs:466-478` (`WriteEndpointNotRegistered` claims E-QUERY-028)
- `crates/prism-core/src/error.rs:480-491` (`UnregisteredWriteTarget` claims E-QUERY-029)
- `.factory/specs/architecture/write-operations.md:639-640` (catalog: E-QUERY-028=throttle, E-QUERY-029=AdapterRegistry-not-init)
- `crates/prism-query/src/write_pipeline.rs:108-127` (fix-pass-1 from_dml_node site emitting E-QUERY-029)

**Problem**: Architecture-level error catalog declares the canonical write-error register E-QUERY-020..029. Two code positions are mis-assigned:

| Code | Architecture-spec semantics | Implementation semantics |
|------|----------------------------|--------------------------|
| E-QUERY-028 | "Write fan-out rate limit: sensor write API is throttled" (degraded, retryable) | "write endpoint not registered: sensor 'X' has no verb 'Y'" (broken, no retry) |
| E-QUERY-029 | "Write endpoint declared in spec but not found in AdapterRegistry. Sensor may not be configured for client" (broken, structural) | "unregistered write target: table 'X' is not registered" (broken, structural — but missing client_id) |

Pre-existing E-QUERY-023 ("Write verb '{verb}' is not available for source '{source}'") is the architecturally-correct code for `WriteEndpointNotRegistered` but no E-QUERY-023 variant exists in error.rs. Similarly E-QUERY-024 + E-QUERY-026 are missing entirely from impl.

**Why is this NEW**: Both error variants are introduced ENTIRELY by the 15-commit S-3.07 chain. Develop's error.rs contains neither. Pass-2 missed them; fresh context in pass-3 surfaces the catalog/impl drift.

**Impact**:
1. Operations runbooks built from `write-operations.md:639-640` would advise "investigate adapter initialization" on E-QUERY-029, but operators would actually need to fix PrismQL spec table registration — wrong escalation path.
2. SLO dashboards keying on E-QUERY-028 retry-after telemetry will see zero events, masking real 429s.
3. POL-4 violation: implementation rebrands two codes from the catalog.
4. E-QUERY-023, -024, -026 remain unimplemented despite catalog declaration — coverage gap.

**Remediation Options**:
(a) **Code follows catalog** (preferred): Rename `WriteEndpointNotRegistered` → use E-QUERY-023; keep `UnregisteredWriteTarget` as E-QUERY-029 but rewrite message to include `{client_id}`. Reserve E-QUERY-028 for future 429 retry path. Add E-QUERY-024 + E-QUERY-026 stubs OR file a TD for their deferral.
(b) **Catalog follows code**: File spec amendment rewriting E-QUERY-028/029 catalog rows + move rate-limit semantics to E-QUERY-030.

The standing project rule "Mis-anchoring always blocks convergence" requires resolving this before declaring CLEAN.
