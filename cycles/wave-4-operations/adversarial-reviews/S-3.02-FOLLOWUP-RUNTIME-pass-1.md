## Findings

### F-LP1-CRIT-1: `register_internal_tables` never invoked from `QueryEngine::execute`

`engine.rs:309` creates a fresh `SessionContext` per query but never calls `internal_tables::register_internal_tables` on it. `materialization.rs:444-449` says "Skip internal tables (prism_*) — already registered via register_internal_tables", flips `any_external_table_registered = true`, then jumps to `execute_against_session` (line 467) which calls `session_ctx.sql(query_str)` — DataFusion will return "table not found" for `prism_audit`. AC-5 only passes because the test calls `register_internal_tables` *directly* on a SessionContext outside `QueryEngine::execute`.

**Severity:** CRITICAL. **Trace:** BC-2.15.011, AC-5, AD-022 §B step 7.
**Recommendation:** invoke `register_internal_tables(&session_ctx, storage)` from `execute_inner`/`execute_scheduled` (requires threading `Arc<dyn RocksStorageBackend>` into `QueryEngine`), OR explicitly state that internal-table queries are out-of-scope for this story and add a real test that exercises `prism_audit` *through* `QueryEngine::execute`.

### F-LP1-CRIT-2: Cross-sensor auth downcast bug — placeholder `CrowdStrikeAuth` passed to ALL adapter types

`materialization.rs:400-404` constructs a `CrowdStrikeAuth` placeholder and passes it as `auth` to every `adapter.fetch()` call regardless of sensor type. Production `CyberintAdapter::fetch` (`crates/prism-sensors/src/auth/cyberint.rs:284-289`) downcasts via `auth.as_any().downcast_ref::<CyberintAuth>()` and returns `SensorError::Internal { detail: "auth downcast to CyberintAuth failed" }` on type mismatch. Same applies to Claroty and Armis adapters.

**Severity:** CRITICAL. **Trace:** BC-2.11.005, BC-2.01.013.
**Recommendation:** either (a) construct the correct `*Auth` variant per `target.sensor_type` (requires real credential resolution — outside this story's scope per implementer note), OR (b) add a per-sensor-type auth dispatch helper, OR (c) raise an explicit `PrismError::CredentialResolutionGap` so the failure is observable rather than masked as a silent partial failure. The current state means production calls die with confusing "downcast failed" errors. The implementer's surfaced spec drift item ("CredentialResolver sealed barrier") is **insufficiently characterized**: the placeholder is reachable from production paths, not test-only.

### F-LP1-CRIT-3: Production CrowdStrikeAdapter `org_id` mismatch will reject every fan-out call

`materialization.rs:384` builds `SensorSpec { org_id: prism_core::OrgId::new(), ... }` (a freshly-randomized OrgId per fetch). `crates/prism-sensors/src/auth/crowdstrike.rs:396-401` enforces `if spec.org_id != self.org_id { return Err(SensorError::OrgIdMismatch { ... }) }`. The randomized spec.org_id will essentially never match the adapter's bound org_id → every CrowdStrike call returns `OrgIdMismatch` and the error is logged and pushed into `_sensor_errors` (which is also discarded — see F-LP1-CRIT-5). The whole CrowdStrike fan-out path is dead in production.

**Severity:** CRITICAL. **Trace:** BC-3.2.001, AC-1.
**Recommendation:** the OrgSlug → OrgId mapping deferral the implementer surfaced is more severe than characterized. Either thread the real `OrgId` from the adapter registry's key, or use `get(org_id, sensor_type)` not `get_all_for_sensor_type`. The latter intentionally bypasses per-org adapter selection and breaks BC-3.2.001 invariant 1.

### F-LP1-CRIT-4: AC-1 and AC-7 pass vacuously with empty `AdapterRegistry`

`tests/execute_integration_tests.rs:268` (AC-1) and `:645` (AC-7) call `make_engine(AdapterRegistry::new(), ...)` — empty registry. With no adapters, `get_all_for_sensor_type` returns empty Vec, no batches are produced, `result.batches.is_empty()`, and the `for batch in &result.batches` loops never execute. The assertions `result.returned_results <= 5` (AC-1) and the virtual-field downcast checks (AC-7) pass without exercising any virtual-field injection or `_sensor` value.

**Severity:** CRITICAL (test rigor regression). **Trace:** AC-1, AC-7, BC-2.11.012, S-7.01 Partial-Fix Regression Discipline.
**Recommendation:** AC-1 and AC-7 must register a `StubAdapter` (the implementer already added the helper for AC-2/AC-3/AC-6) so the tests actually verify what they claim. Per S-7.01 sub-clause (b) "fix applied to primary, sibling not updated" — blast radius = 2.

### F-LP1-CRIT-5: `sensor_errors` accumulated then silently discarded

`materialization.rs:362` declares `let mut _sensor_errors: Vec<String> = Vec::new();` and pushes errors at line 434, but the function signature returns only `Vec<RecordBatch>` — the errors are dropped at function return. `engine.rs:355-361` and `:442-449` build `QueryResult` with `sensor_errors: Vec::new()` always. BC-2.11.005 mandates partial-failure surfacing; BC-2.11.011 postcondition: "Partial failures (some clients succeed, some fail) return results from successful clients **plus `sensor_errors`** for failures." This is a SOUL.md #4 violation (silent failure swallowing).

**Severity:** CRITICAL. **Trace:** BC-2.11.005, BC-2.11.011, EC-002, SOUL.md #4.
**Recommendation:** change `run_materialization_pipeline` return type to `(Vec<RecordBatch>, Vec<String>)` or wrap in a struct, and propagate to the `QueryResult.sensor_errors` field.

### F-LP1-HIGH-1: Filter and Pipe modes silently discard all materialized data

`materialization.rs:483-503` matches only `Ast::Sql(SqlStatement::Select(_))`. For `Ast::Filter(_)` and `Ast::Pipe(_)`, `execute_against_session` returns `Ok(Vec::new())` with the comment "callers get results from the table_batches in the outer scope" — but the outer scope (`run_materialization_pipeline`) returns `collected` (line 467), which is the empty Vec. The `table_batches` accumulator at line 358 is dropped at function end, never returned. **Filter and Pipe queries return zero rows even when the sensor produced data.**

**Severity:** HIGH. **Trace:** BC-2.11.002, BC-2.11.005.
**Recommendation:** for Filter/Pipe modes, either (a) collect-and-return `table_batches.into_values().flatten().collect()` directly, or (b) execute via DataFusion using the registered MemTables and a synthesized `SELECT *` query.

### F-LP1-HIGH-2: `RocksDbTableProvider::scan` does NOT deserialize bincode 2.x payloads

Story §Tasks step 5: "Deserialize values (bincode 2.x encoding per AD-012)." Implementation at `internal_tables.rs:218-255`: `build_batch_from_kv` puts raw value bytes into the FIRST schema field as best-effort UTF-8, and empty strings into all other fields. Audit-buffer rows materialize as `{timestamp: <raw-bytes-as-utf8>, event_type: "", org_id: "", payload: ""}` — semantically wrong. Cargo.toml does NOT include `bincode = "2"` for prism-query.

**Severity:** HIGH. **Trace:** AD-012, BC-2.15.011, story Task step 5.
**Recommendation:** add `bincode = "2"` to prism-query and decode value bytes per the canonical schema for each domain, OR amend the story to declare this as deferred and add a follow-up TD ticket.

### F-LP1-HIGH-3: `requires_audit_read` capability gate documented but NOT IMPLEMENTED

`internal_tables.rs:73-74`, `:92-94`, `:352-355`, `:381` all reference `requires_audit_read = true` for `prism_audit`. Module doc claim at `:14-16` says "Accessing `prism_audit` requires the `audit.read` capability. If denied, `E-QUERY-011` is returned. (BC-2.15.011)" — a **lie**, no capability check happens.

**Severity:** HIGH (security gap + doc-vs-code drift). **Trace:** BC-2.15.011, AD-022.
**Recommendation:** either implement the capability gate in `scan()` (consult `Session` state for capabilities) or remove the misleading doc and the unused `requires_audit_read` field.

### F-LP1-HIGH-4: Internal tables (`prism_*`) do NOT receive virtual field injection

BC-2.11.012 postcondition: "[`_sensor`] values: `\"crowdstrike\"`, `\"cyberint\"`, `\"claroty\"`, `\"armis\"` for external tables; **`\"prism\"` for internal RocksDB-backed tables**". `internal_tables.rs::scan` returns batches built only from `descriptor.schema`, with no `_sensor`, `_client`, or `_source_table` columns. AC-7 would fail if the test query was `SELECT * FROM prism_audit` instead of `crowdstrike_detections` (it doesn't catch this because empty registry → empty results — see F-LP1-CRIT-4).

**Severity:** HIGH. **Trace:** BC-2.11.012 postcondition.
**Recommendation:** inject `_sensor = "prism"`, `_client = "<system>"`, `_source_table = "<bare-domain-name>"` into `RocksDbTableProvider::scan` output.

### F-LP1-HIGH-5: AC-4 push-down bypasses existing `pushdown::classify_predicates`

The `pushdown::classify_predicates` function described in engine.rs:3 module-doc step 3 is not invoked.

**Severity:** HIGH. **Trace:** BC-2.11.007, ADR-022 §C wiring-not-redesign, story Architecture Compliance Rule.
**Recommendation:** route push-down through the existing `pushdown::classify_predicates` (already merged via S-3.02). Currently it's dead code in this path.

### F-LP1-HIGH-6: AC-6 fan-out duplicates work — every (client × adapter) pair fires

`materialization.rs:366-368` calls `get_all_for_sensor_type(target.sensor_type)` returning ALL CrowdStrike adapters regardless of OrgId. With 2 clients × 2 stub adapters, 4 fetch calls happen instead of 2. In production with multiple-org-keyed real adapters, this would cause duplicate API requests against the SAME sensor for unrelated orgs.

**Severity:** HIGH. **Trace:** BC-3.2.001 (per-org isolation), AC-6.
**Recommendation:** restore per-org adapter selection via `registry.get(org_id, sensor_type)`. Until then, document explicitly and emit `tracing::warn!` when more than one adapter matches.
