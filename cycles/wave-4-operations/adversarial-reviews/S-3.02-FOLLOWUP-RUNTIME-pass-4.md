`walk_expr` at `materialization.rs:965-990` matches `Expr::InSubquery` / `Compare` / `Logical` / `Not`. The wildcard arm at line 987-988 reads `// Other Expr variants (Literal, Field, VirtualField, FuncCall, Star) have no subqueries.` — **factually incorrect**.

Per `ast.rs:608-622`, `FuncCall::Aggregate { args: Vec<Expr> }` and `FuncCall::Scalar { args: Vec<Expr> }` carry `Vec<Expr>`. A query like `SELECT severity_label(id IN (SELECT trace_id FROM prism_audit)) FROM crowdstrike_detections` would slip past Layer 1; Layer 2 catches it at scan time.

**Why MEDIUM not CRITICAL:** defense-in-depth holds (Layer 2 enforces). **Why MEDIUM not LOW:** same Standing Rule 3 #3 pattern as F-LP3-CRIT-1 (rated CRIT in pass-3) applied to a different AST position; the doc claim "Walks ALL AST positions where a subquery can appear" at `materialization.rs:885` is false.

**Recommendation:** Either (a) extend `walk_expr` to walk `FuncCall::Aggregate { args }` and `FuncCall::Scalar { args }` recursively + add 4th RED→GREEN test; OR (b) demote doc claim to honest enumeration. Path (a) is the production-grade fix per user mandate.
