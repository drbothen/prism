### SEC-001: Internal Table Guard Bypassed via Mixed-Case Table Names
- **Severity:** HIGH
- **CWE:** CWE-178 (Improper Handling of Case Sensitivity)
- **OWASP:** A03:2021 – Injection (SQL injection class via table name manipulation)
- **File:** `crates/prism-query/src/sql_parser.rs:1149–1150`

**Attack Vector:**
An attacker submits `DELETE FROM PRISM_audit WHERE id=1` or `UPDATE Prism_cases SET x=1 WHERE id=1`. The ident parser extracts the table name preserving original case (e.g., `"PRISM_audit"`). `is_internal_prism_table` checks `table_name.starts_with("prism_")` — case-sensitive, lowercase only. `"PRISM_audit"` does NOT start with `"prism_"` in a case-sensitive comparison, so the guard returns `false` and the DML node is constructed without triggering `E-QUERY-010`.

**Impact:** The `E-QUERY-010` write-protection guard for internal Prism tables is bypassed. An attacker who can issue DML queries could write to `prism_audit`, `prism_cases`, `prism_rules`, `prism_schedules`, or `prism_aliases` by using any non-lowercase form of the `prism_` prefix.

**Evidence:**
```rust
// sql_parser.rs:1149
pub(crate) fn is_internal_prism_table(table_name: &str) -> bool {
    table_name.starts_with("prism_")  // case-sensitive — PRISM_audit passes through
}

// build_delete_parser: ident parser preserves original case
let ident = ident_char
    .repeated()
    .at_least(1)
    .to_slice()
    .map(|s: &str| s.to_string());  // no .to_ascii_lowercase()
```

The existing tests (`test_BC_2_11_004_is_internal_prism_table_*`) only test lowercase input — there is no test for `"PRISM_audit"`, `"Prism_cases"`, etc.

**Proposed Mitigation:**
Normalize the table name to lowercase before the guard check, or make `is_internal_prism_table` case-insensitive:

```rust
pub(crate) fn is_internal_prism_table(table_name: &str) -> bool {
    table_name.to_ascii_lowercase().starts_with("prism_")
}
```

Add tests: `assert!(is_internal_prism_table("PRISM_audit"))`, `assert!(is_internal_prism_table("Prism_cases"))`.

This must also be applied consistently in `build_delete_parser`, `build_update_parser`, and `build_insert_parser` — all three call `is_internal_prism_table` with the raw `table` string.

### SEC-002: DML WHERE Predicate Depth Not Post-Checked (Inconsistent Security Layer)
- **Severity:** MEDIUM
- **CWE:** CWE-400 (Uncontrolled Resource Consumption)
- **OWASP:** A05:2021 – Security Misconfiguration (depth limits applied inconsistently across code paths)
- **File:** `crates/prism-query/src/filter_parser.rs:350–355`, `crates/prism-query/src/sql_parser.rs:826–850`

**Attack Vector:**
A SELECT query's WHERE clause has its predicate nesting depth checked post-parse via `check_sql_query_nesting_depth_with` (called in `parse_sql_with_limits`). DML statements (DELETE/UPDATE/INSERT INTO) take a different path: `parse_dml_internal` calls `parse_sql_dml` which dispatches directly to `build_delete_parser`, `build_update_parser`, or `build_insert_parser` — none of which call the post-parse nesting depth checks.

Additionally, `parse_dml_internal` receives `_limits: &security::ParseLimits` but the parameter is unused (note the leading underscore). The snapshotted limits are not threaded into `parse_sql_dml`.

**Impact:** An attacker crafting a DML query with a deeply nested AND/OR expression in the WHERE clause would not be caught by post-parse depth enforcement, only by the pre-parse paren depth check (which limits paren nesting to 64 but does not limit AND/OR chain length). This creates a security layer inconsistency that could be exploited for Chumsky backtracking DoS on paren-free deep operator chains. The severity is mitigated by the pre-parse paren depth guard and the sentinel discard of the predicate, but the inconsistency is a security concern.

The INSERT INTO path is additionally affected: `build_insert_parser` calls `build_sql_parser()` directly rather than `parse_sql_with_limits`, so the nested SELECT's depth/list-size are not checked.

**Evidence:**
```rust
// filter_parser.rs:350-355
fn parse_dml_internal(
    input: &str,
    _limits: &security::ParseLimits,  // UNUSED — underscore prefix confirms this
) -> Result<Ast, Vec<ParseError>> {
    crate::sql_parser::parse_sql_dml(input)  // limits not forwarded
}
```

**Proposed Mitigation:**
1. Remove the leading underscore from `_limits` and forward limits to `parse_sql_dml`.
2. Add post-parse depth checks in `parse_sql_dml` for DELETE/UPDATE WHERE predicates.
3. For INSERT INTO...SELECT, call `parse_sql_with_limits` (or add depth+list-size checks inline) rather than `build_sql_parser()` directly.

### SEC-003: Sentinel Pattern in DmlNode.filter Creates Implicit Type Contract for S-3.07
- **Severity:** MEDIUM
- **CWE:** CWE-707 (Improper Neutralization — Type Confusion class)
- **OWASP:** A04:2021 – Insecure Design
- **File:** `crates/prism-query/src/sql_parser.rs:941–945`, `crates/prism-query/src/write_ast.rs:112–116`

**Attack Vector:**
`DmlNode.filter` for DELETE/UPDATE uses a sentinel value `Expr::Literal(Bool(true))` to indicate "a WHERE clause was present" — the actual parsed predicate is discarded. This is documented in the PR description but is a latent type-confusion hazard.

S-3.07 (write executor, not yet implemented) will receive a `DmlNode` where `filter = Some(Expr::Literal(Bool(true)))`. If S-3.07 executes the filter as-is (treating the sentinel as the actual WHERE predicate), every row matches (`Bool(true)`) — converting a bounded write into an unbounded one. This would be the opposite of the intended E-QUERY-022 protection.

**Impact:** If S-3.07 evaluates `DmlNode.filter` directly without re-parsing, it will apply `WHERE true` (all rows) to DELETE/UPDATE operations. This is an unbounded write — the guard that was meant to prevent it silently converts to a sentinel that enables it.

**Evidence:**
```rust
// build_delete_parser and build_update_parser:
let where_present = kw_ci("WHERE")
    .padded()
    .ignore_then(predicate.padded())
    .or_not()
    .map(|p| p.map(|_| crate::ast::Expr::Literal(crate::ast::Literal::Bool(true))));
    //                   ^ actual predicate silently discarded; sentinel stored
```

The `DmlNode` doc comment says "WHERE clause predicate for UPDATE / DELETE" — misleading, since it's actually a sentinel, not the predicate.

**Proposed Mitigation:**
1. Replace the sentinel approach with a dedicated type: `filter: Option<RawWhereClause>` where `RawWhereClause` stores the original string span or a typed sentinel enum (`FilterSentinel::WasPresent`). This makes the sentinel semantics unambiguous.
2. Update the `DmlNode.filter` doc comment immediately to warn S-3.07 implementers: "SENTINEL VALUE — not the actual predicate. Re-parse from the original query string."
3. Add a TODO/FIXME comment at the sentinel construction site.
4. Before S-3.07 merges, a BC amendment must specify that S-3.07 MUST re-parse from the original query string, not evaluate `DmlNode.filter` directly.

### SEC-004: WriteVerbSource Trait Unnecessarily Fully Public — External Impl Possible
- **Severity:** LOW
- **CWE:** CWE-668 (Exposure of Resource to Wrong Sphere)
- **OWASP:** A05:2021 – Security Misconfiguration
- **File:** `crates/prism-query/src/write_verb_registry.rs:34`, `crates/prism-query/src/lib.rs:54`

**Attack Vector:**
`WriteVerbSource` is `pub trait` and `WriteVerbRegistry` is `pub struct` in a `pub mod write_verb_registry`. An external crate can implement `WriteVerbSource` and construct a `WriteVerbRegistry` with arbitrary verb lists via `WriteVerbRegistry::from_source`. While `parse_pipe_with_write` is `pub(crate)` (preventing external use of the registry with the parser), this is defense-in-depth exposure: if any public function is added in a future story that accepts `&WriteVerbRegistry`, the attacker's custom implementation could flow in.

**Impact:** Currently unexploitable because the parser entry points that consume `WriteVerbRegistry` are `pub(crate)`. LOW severity because the exploit chain is blocked at the parser entry point. However, the exposure widens the attack surface unnecessarily.

**Evidence:**
```
lib.rs:54: pub mod write_verb_registry;
write_verb_registry.rs:34: pub trait WriteVerbSource {
write_verb_registry.rs:128: pub struct WriteVerbRegistry {
```

None of `WriteVerbSource`, `WriteVerbRegistry`, or `WriteVerbRegistry::from_source` are in BC-2.11.006's `restricted_symbols` list, suggesting this exposure was considered acceptable. It should be explicitly documented as a design decision.

**Proposed Mitigation:**
Either:
1. Add a comment in BC-2.11.006 acknowledging that `WriteVerbSource`/`WriteVerbRegistry` are intentionally `pub` for S-3.07 integration, OR
2. Make them `pub(crate)` (same as parser entry points) if S-3.07 will live in the same crate.

### SEC-005: extract_sensor_prefix Returns Some("") on Leading-Underscore Input
- **Severity:** LOW
- **CWE:** CWE-1284 (Improper Validation of Specified Quantity in Input)
- **OWASP:** A03:2021 – Injection (information disclosure via error messages)
- **File:** `crates/prism-query/src/pipe_parser.rs:893–905`

**Attack Vector:**
`extract_sensor_prefix("_internal")` returns `Some("")` — an empty string prefix. Callers that use the `source_sensor` value in error messages (E-QUERY-023 suggestions) or sensor-specific verb lookups will receive an empty string as the sensor name. `WriteVerbRegistry::verbs_for_sensor("")` falls through to the "all verbs" fallback path, which may expose the full verb list regardless of sensor context.

**Impact:** Information disclosure — error messages for E-QUERY-023 may list ALL write verbs for ALL sensors when the source starts with `_`. This is a minor information leakage (verb names are not secrets), but could aid attackers in understanding the available write surface. The `Some("")` case is explicitly tested and documented, so this is an accepted design choice, but the security implication is not documented.

**Evidence:**
```rust
// pipe_parser.rs:903-905
sep_pos.map(|pos| source_raw[..pos].to_string())
// For "_internal": pos=0, returns source_raw[..0] = "" → Some("")
```

**Proposed Mitigation:**
Add a check: if `pos == 0` (leading separator), return `None` rather than `Some("")`. Update the doc comment to clarify why `Some("")` is the current behavior and document the information disclosure implication.

### SEC-006 (OBSERVATION): Visitor Does Not Walk WriteNode
- **Severity:** OBSERVATION
- **CWE:** N/A
- **File:** `crates/prism-query/src/visit.rs:169–174`

`walk_pipe_query` iterates over `q.stages` but does not visit `q.write` (the `Option<WriteNode>`). Any analysis pass implemented via the `Visitor` trait (security analysis, field enumeration, etc.) will silently skip the write node's `verb` and `args` fields. This is not a security vulnerability in S-3.06's scope but is a forward-looking concern: if S-3.07 adds AST visitors for sanitization or audit, they would silently miss write-node content.

**Proposed Mitigation:** Add `walk_write_node` to the visitor and call it from `walk_pipe_query` when `q.write.is_some()`. File as tech debt before S-3.07.

## Verdict

**NEEDS-CHANGES**

One HIGH finding (SEC-001) blocks merge: `is_internal_prism_table` must normalize case before the prefix check. Fix is a one-line change with two additional test cases.

SEC-002 (MEDIUM) should be addressed before S-3.07 to ensure consistent security limit enforcement across all parse paths.
