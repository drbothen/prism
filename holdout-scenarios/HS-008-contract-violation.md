## HS-008-01: OCSF Schema Version Mismatch

**Title:** Prism compiled against OCSF v1.7.0 receives events from a sensor using newer schema

**Preconditions:**
- Prism built with ocsf-proto-gen targeting OCSF v1.7.0
- Proto field numbers assigned via sequential alphabetical ordering (ocsf-proto-gen's ADR-05)
- Sensor API returns data with fields from OCSF v1.8.0 (hypothetical)

**Steps:**
1. Prism normalizes CrowdStrike alerts to OCSF v1.7.0 Detection Finding
2. An upstream system sends OCSF v1.8.0 events (new fields added)
3. Prism attempts to deserialize v1.8.0 event using v1.7.0 proto definitions

**Expected Outcome:**
- New fields unknown to v1.7.0 proto are preserved as unknown fields (proto3 behavior)
- Existing fields remain accessible
- No deserialization crash
- WARNING logged: `"OCSF version mismatch: event version v1.8.0, compiled schema v1.7.0"`
- The proto field number instability is understood: ocsf-proto-gen assigns field numbers alphabetically (ADR-05), so adding a field in v1.8.0 shifts subsequent field numbers -- this breaks wire compatibility
- Mitigation: version-aware normalization that maps to the compiled proto version
- Three-tier field alias resolution (axiathon) handles field name changes across versions

**Repos Tested:** ocsf-proto-gen (sequential alphabetical field numbering -- most significant architectural limitation), axiathon (OcsfVersionAliasMap for cross-version field resolution)

## HS-008-02: Proto Field Number Instability Across OCSF Versions

**Title:** Proto field numbers shift when OCSF schema adds or removes attributes

**Preconditions:**
- Prism built with OCSF v1.7.0 protos: field `severity_id` has field number 15
- OCSF v1.8.0 adds a field alphabetically before `severity_id`: `security_state` (field number 15, pushing severity_id to 16)

**Steps:**
1. Prism serializes an OCSF event using v1.7.0 field numbers
2. A downstream consumer compiled with v1.8.0 field numbers attempts to deserialize
3. Field number 15 is now `security_state` in v1.8.0, but contains `severity_id` data from v1.7.0

**Expected Outcome:**
- Wire-incompatible: v1.7.0 serialized data CANNOT be correctly deserialized by v1.8.0 consumer (field number mismatch)
- This is a KNOWN LIMITATION of ocsf-proto-gen's ADR-05 (sequential alphabetical field numbering)
- Prism's mitigation strategy documented: pin to a specific OCSF version, coordinate upgrades across all components
- Proto definitions generated at build time (not runtime) -- version pinned in Cargo.toml / build.rs
- Enum value map (enum-value-map.json) also version-specific

**Repos Tested:** ocsf-proto-gen (ADR-05: field number instability, BTreeMap-ordered determinism)

## HS-008-03: Protobuf Version Conflict Between Dependencies

**Title:** Prism's dependency tree contains incompatible prost versions

**Preconditions:**
- Prism depends on ocsf-proto-gen output (compiled via prost-build)
- Prism also uses axiathon patterns that depend on prost-reflect
- axiathon's production workspace uses prost 0.14, spike uses prost 0.13

**Steps:**
1. Cargo build resolves dependencies
2. Two crates require different prost versions (0.13 vs 0.14)
3. Check for compilation errors or runtime type mismatches

**Expected Outcome:**
- Prism pins a single prost version (0.14, the latest)
- No duplicate prost versions in dependency tree (`cargo tree -d` shows no duplicates for prost)
- prost-reflect version compatible with chosen prost version
- DynamicMessage from prost-reflect works with protos generated by prost-build (same version)
- This resolves axiathon's documented incompatibility: production workspace (prost 0.14, prost-reflect 0.15) vs spike workspace (prost 0.13, prost-reflect 0.14)

**Repos Tested:** axiathon (prost 0.14 vs 0.13 incompatibility), ocsf-proto-gen (proto generation consumed by prost-build)

## HS-008-04: xMP Envelope Format Drift

**Title:** Prism's xMP envelope diverges from the format expected by downstream Vector pipeline

**Preconditions:**
- Existing Vector pipeline configured to parse xMP envelope format
- Prism generating enriched payloads

**Steps:**
1. Prism generates xMP envelope for a CrowdStrike alert
2. Vector receives the payload
3. Vector attempts to parse the envelope using existing configuration

**Expected Outcome:**
- Envelope format EXACTLY matches: `{"data": <record>, "record_type": "<sensor>_<entity>", "xmp": {"site": "...", "cluster_name": "...", "node_name": "..."}}`
- `record_type` standardized to `<sensor>_<entity>` (resolving poller inconsistency: cobra has no prefix, coaster uses `armis_` prefix)
- `ocsf` field added as optional (from poller-bear's envelope, standardized across all sensors)
- `xmp` object has exactly 3 fields: `site`, `cluster_name`, `node_name` (no extras that would break Vector parsing)
- No field name changes, no nesting changes, no type changes from current production envelope
- Backward compatibility test: current Vector config parses Prism output without modification

**Repos Tested:** poller-cobra (xMP envelope), poller-express (xMP envelope), poller-bear (xMP + ocsf field), poller-coaster (xMP with armis_ prefix)

## HS-008-05: Sensor API Contract Change (Breaking)

**Title:** Claroty xDome API changes endpoint behavior between versions

**Preconditions:**
- Prism's Claroty adapter coded against xDome API v1
- Claroty releases API v2 with changed response structure

**Steps:**
1. Prism calls POST `/api/v1/alerts/` (v1 endpoint)
2. Claroty returns 301 redirect to `/api/v2/alerts/` (or v1 returns deprecated response)
3. Response structure has changed: field names renamed, nested objects restructured

**Expected Outcome:**
- If v1 endpoint still works: Prism continues using v1 (no automatic upgrade)
- If v1 returns redirect: Prism does NOT blindly follow redirect (security: redirect could be to a different domain)
- If v1 returns deprecation warning header: Prism logs warning, continues operating
- Deserialization handles missing expected fields gracefully (serde defaults, not crash)
- Unknown new fields in response preserved via lenient parsing
- Prism's generic DataSource trait makes API version upgrades per-sensor changes, not system-wide refactors

**Repos Tested:** poller-bear (Claroty API contract: 9 POST-for-read endpoints), mcp-claroty-xdome (same Claroty API, 5 query endpoints)

## HS-008-06: CrowdStrike Two-Step Fetch Contract Violation

**Title:** CrowdStrike QueryV2 returns IDs that PostEntitiesAlertsV1 cannot resolve

**Preconditions:**
- Prism using CrowdStrike two-step fetch: QueryV2 for IDs, PostEntitiesAlertsV1 for details
- Race condition: alert deleted between step 1 and step 2

**Steps:**
1. Prism calls QueryV2, receives 100 alert IDs
2. Between step 1 and step 2, 5 alerts are resolved/deleted in CrowdStrike
3. Prism calls PostEntitiesAlertsV1 with all 100 IDs
4. API returns 95 alerts (5 missing)

**Expected Outcome:**
- Prism handles partial response: 95 of 100 IDs resolved
- Missing 5 IDs logged as warnings, not errors
- Cursor advances based on the IDs that WERE returned (not the requested set)
- No crash, no infinite retry loop on missing IDs
- xMP envelope generated for 95 alerts, 5 gaps acknowledged
- This is an inherent race condition in the two-step fetch pattern documented in poller-cobra

**Repos Tested:** poller-cobra (two-step fetch: QueryV2 -> PostEntitiesAlertsV1)

## HS-008-07: Armis AQL Query Syntax Rejected

**Title:** Armis API rejects AQL query due to syntax change or deprecation

**Preconditions:**
- Prism configured with AQL query for Armis alerts
- Armis API changes AQL syntax rules

**Steps:**
1. Prism sends configured AQL query to Armis GetSearch endpoint
2. Armis returns HTTP 400 with error: "Invalid AQL syntax"
3. Prism handles the error

**Expected Outcome:**
- HTTP 400 treated as configuration error, not transient failure
- NO exponential backoff retry (400 is not retryable -- config must be fixed)
- Clear error message: `"Armis AQL query rejected: invalid syntax. Review PRISM_ARMIS_ALERTS_AQL configuration."`
- Sensor marked as UNHEALTHY until config corrected
- Other sensors and tenants unaffected
- Query fingerprint mechanism would detect if the AQL was changed between restarts

**Repos Tested:** poller-coaster (AQL query forwarding, query fingerprint validation)

## HS-008-08: MCP Protocol Version Mismatch

**Title:** MCP client speaks a different protocol version than Prism server

**Preconditions:**
- Prism implements MCP via rmcp 0.8 (JSON-RPC 2.0 over stdio)
- Client uses a newer MCP SDK with additional protocol features

**Steps:**
1. Client sends `initialize` request with capabilities not supported by Prism
2. Prism responds with its supported capabilities
3. Client sends a tool call using a parameter format Prism does not expect
4. Prism handles the unexpected input

**Expected Outcome:**
- `initialize` handshake succeeds -- Prism advertises only supported capabilities
- Unknown capabilities in client request ignored (forward-compatible)
- Unknown tool parameters validated via JsonSchema (Parameters<T> pattern from tally)
- Invalid parameters rejected with proper JSON-RPC error code (not crash)
- ServerHandler trait implementation handles protocol negotiation correctly
- Error response follows JSON-RPC 2.0 specification exactly

**Repos Tested:** tally (rmcp 0.8, ServerHandler, Parameters<T> + JsonSchema), mcp-claroty-xdome (MCP protocol handling, Zod schema validation equivalent)
