# S-3.2.03 — prism-dtu-crowdstrike: Multi-tenant state segregation

## Narrative

As a Prism MSSP operator, I want the CrowdStrike DTU clone to key its containment and
detection-status stores by `(OrgId, resource_id)` so that containment actions and
detection status updates for Client A cannot bleed into Client B's view, even when both
clients have devices or detections with the same CrowdStrike-assigned IDs.

## Objective

Migrate `prism-dtu-crowdstrike/src/state.rs` per ADR-008 §2.1 Step 6c:
- Re-key `containment_store` (line 86) and `detection_status_store` (line 88) to
  `HashMap<(OrgId, String), V>`.
- Leave `session_registry` (line 90, `LruCache<String, SessionData>`) unchanged —
  per D-048, pagination session IDs are org-scoped at the query-engine layer, not the
  clone state layer.
- Add `reset_for(org_id)`, rename `reset()` → `reset_all()`.
- Add per-org proptest coverage (TD-DTU-MUTATE-COVERAGE-001).
- Document the `session_registry` non-re-keying rationale in a code comment.

## Tasks

1. **Migrate `containment_store`** (`src/state.rs` line 86):
   ```rust
   // Before
   pub containment_store: Mutex<HashMap<String, ContainmentStatus>>,
   // After
   pub containment_store: Mutex<HashMap<(OrgId, String), ContainmentStatus>>,
   ```

2. **Migrate `detection_status_store`** (`src/state.rs` line 88):
   ```rust
   // Before
   pub detection_status_store: Mutex<HashMap<String, String>>,
   // After
   pub detection_status_store: Mutex<HashMap<(OrgId, String), String>>,
   ```

3. **Leave `session_registry` unchanged** (`src/state.rs` line 90):
   ```rust
   // INTENTIONALLY NOT re-keyed — D-048 (ADR-008 §Decision Refinements):
   // Pagination session IDs are org-scoped at the query-engine layer.
   // The query engine generates session IDs with OrgId embedded in the
   // UUID v7 time field (org-temporal uniqueness). The clone never sees a
   // session ID that could collide across orgs. Re-keying would require
   // passing OrgId at session-lookup time from the HTTP header — incorrect
   // layer. See ADR-008 §2.1 D-048.
   pub session_registry: Mutex<LruCache<String, SessionData>>,
   ```

4. **Update all call sites** in containment and detection route handlers:
   - `state.containment_store.lock()?.insert(device_id, status)` →
     `state.containment_store.lock()?.insert((org_id, device_id), status)`
   - Same pattern for `detection_status_store`.
   - Thread `OrgId` through route handler state/extensions.

5. **Add `DEFAULT_ORG_ID` test constant** (`#[cfg(test)]` only) and update existing tests.

6. **Add `reset_for(org_id: OrgId)`** and rename `reset()` → `reset_all()` with shim.

7. **Add `tests/multi_tenant.rs`**:
   - `prop_containment_cross_org_isolation` — write containment for org_A, assert org_B lookup is None.
   - `prop_detection_cross_org_isolation` — same for detection_status_store.
   - `prop_reset_for_selectivity` — reset org_A, verify org_B intact.
   - `test_session_registry_not_rekeyed` — unit test confirming session_registry is
     bare String keyed with a comment cross-referencing D-048.

8. **Gate**: `cargo test -p prism-dtu-crowdstrike` green.

## Acceptance Criteria

**AC-001** — Containment store cross-org isolation (traces to BC-3.2.001 postcondition 1):
Given device `"host-001"` is set to `ContainmentStatus::Contained` for `org_id_A`,
When `state.containment_store` is read with key `(org_id_B, "host-001")`,
Then the result is `None`.

**AC-002** — Detection status cross-org isolation (traces to BC-3.2.001 postcondition 1):
Given detection `"det-999"` has status `"closed"` for `org_id_A`,
When `state.detection_status_store` is read with key `(org_id_B, "det-999")`,
Then the result is `None`.

**AC-003** — Write does not affect other org (traces to BC-3.2.001 postcondition 2):
Given org_A sets containment for `"host-001"`,
When org_B reads containment for `"host-001"`,
Then org_B's result is unchanged from its prior state.

**AC-004** — `session_registry` not re-keyed (traces to BC-3.2.003 precondition 4, D-048):
Given the migration is applied,
When the `session_registry` field is inspected,
Then its type remains `Mutex<LruCache<String, SessionData>>` (bare String key) and a
code comment references ADR-008 D-048 rationale.

**AC-005** — `reset_for` is selective (traces to BC-3.2.001 edge case EC-004):
Given containment entries for both org_A and org_B,
When `state.reset_for(org_id_A)` is called,
Then org_A containment and detection stores are empty; org_B entries are intact.

**AC-006** — OrgId-flipping proptest kills mutation (traces to BC-3.2.001 VP-079):
Given adversarial org_A ≠ org_B and any shared device_id,
When containment is stored under org_A and read under org_B,
Then the result is always `None`.

**AC-007** — All route handler call sites updated (traces to BC-3.2.001 invariant 1):
Given `cargo test -p prism-dtu-crowdstrike`,
When the test suite runs,
Then all 0 compile errors (every call site passes `OrgId` as required by the new type signature).

## Architecture Compliance Rules

- **Forbidden dependencies:** `prism-dtu-crowdstrike` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`.
- `session_registry` MUST remain bare-String keyed. Any PR that re-keys it must
  document the full D-048 rationale revision before merging.
- `DEFAULT_ORG_ID` MUST be `#[cfg(test)]` only.
- Both `containment_store` AND `detection_status_store` must be migrated atomically in
  the same PR — partial migration is not acceptable (it would leave the type invariant violated).
- The D-048 code comment in `session_registry` is MANDATORY — it documents the
  intentional exception and prevents future contributors from "fixing" it incorrectly.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | CrowdStrike device IDs are CID-scoped but not globally unique; two clients share ID `"abc-123"` | Stored under `(org_id_A, "abc-123")` and `(org_id_B, "abc-123")` independently; no collision |
| EC-002 | Pagination session ID arrives in `X-DTU-Session-Id` header with no org context | Looked up in `session_registry` by raw string per D-048; org-scoping was applied at session ID generation in query engine |
| EC-003 | `reset_for(org_A)` clears containment AND detection stores atomically for org_A | Both stores are reset_for'd in the same call; no partial reset |
| EC-004 | Detection status written for org_A then org_A's store reset; org_B's detection unaffected | Confirmed by `prop_reset_for_selectivity` |

## Previous Story Intelligence

- S-6.07 (prism-dtu-crowdstrike Wave 1): containment, detection, and session route
  handlers are built. This story is a mechanical type migration.
- Pattern matches S-3.2.01 (Claroty) and S-3.2.02 (Armis) — use same proptest
  strategy. Key difference: TWO stores to migrate (not one), and session_registry
  explicitly NOT migrated with mandatory comment.
- D-048 rationale: verified that CrowdStrike pagination session IDs embed OrgId via
  UUID v7 time field at query-engine layer. Do not add OrgId parameter to session
  lookup — it is wrong-layer enforcement.

## Risks

| Risk ID | Description | Mitigation |
|---------|-------------|-----------|
| R-3.2.03-001 | Partial migration (only one of the two stores re-keyed) | Migrate both in the same commit; CI will catch compile errors |
| R-3.2.03-002 | session_registry accidentally re-keyed by a well-intentioned future contributor | Mandatory D-048 code comment prevents this; ADR-008 is the authoritative source |
| R-3.2.03-003 | LruCache type constraint — `(OrgId, String)` must implement Hash + Eq | OrgId is UUID (derives Hash, Eq); String derives both — no issue |
