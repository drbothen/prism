# S-5.07 — prism-mcp: Multi-Repo Git Config Subscriptions

## Narrative

As a Prism MSSP operator, I want to declare multiple Git repositories as config sources
with priority-based merge precedence, so that shared policy configs (high priority) can be
maintained centrally and per-client overrides (lower priority) layered on top — with
automatic background sync, SSH key isolation, and CLI commands to inspect effective config
values and trace their origin.

## Objective

Implement multi-repo git subscription sync per `config-schema.md` §Multi-Repo Git
Subscriptions and AD-018: background git fetch via `git2`, per-source SSH key handling,
per-source `sync_interval`, priority-based merge precedence (higher priority wins;
lexicographic source name breaks ties), and the `prism config diff`, `prism config diff
--pending`, `prism config show --effective`, and `prism config show --trace <key>` CLI
subcommands. Hot reload integration: S-1.12's `notify` watcher fires on local config
changes written by git sync; S-5.07 provides the git fetch layer that writes those
changes.

**Scope boundary:** This story builds on top of S-5.05 (single-file TOML config loading).
S-5.05 must be complete before this story begins. The credential reference model (AD-017
opaque) is established in S-1.06 and S-1.07; this story consumes it but does not redefine
it.

**BC note:** No new BCs are created for this story. The multi-repo sync behavior is an
integration and operational extension of the existing config domain. The BCs listed in
frontmatter are the existing BC-2.06.* contracts that this story's implementation
continues to satisfy (e.g., the effective merged config must still validate per BC-2.06.005,
env overrides still win per BC-2.06.008). The config diff/trace tooling provides operator
transparency over the merge behavior established in BC-2.06.004.

## Tasks

1. Define `[[config_sources]]` schema extension in `crates/prism-mcp/src/config/mod.rs`:
   - Add `config_sources: Vec<ConfigSource>` to `PrismConfig`
   - `ConfigSource { name: String, url: String, branch: String, priority: u32,
     ssh_key_ref: Option<CredentialRef>, sync_interval_secs: u64,
     enabled: bool }`
   - `priority`: higher value wins on key conflicts (e.g., 100 > 50 > 10)
   - `ssh_key_ref`: opaque credential reference per AD-017 — resolved at sync time via
     `CredentialStore::get()`, never stored in memory longer than the fetch call
   - `sync_interval_secs`: per-source polling interval (minimum 60 seconds enforced;
     values below 60 are silently clamped to 60)
   - Validate: `name` is unique across all `config_sources` entries; return `E-CFG-010`
     on duplicate

2. Implement `GitConfigSource` in `crates/prism-mcp/src/config/git_source.rs`:
   - `GitConfigSource::new(source: ConfigSource, cred_store: Arc<dyn CredentialStore>) -> Self`
   - `GitConfigSource::fetch_and_write(&self, local_cache_dir: &Path) -> Result<(), ConfigError>`:
     - Clone or fetch the repo to `{state_dir}/config_cache/{source.name}/`
     - Use `git2::Repository::clone` or `git2::Remote::fetch` with SSH credentials
     - SSH key handling: call `CredentialStore::get(...)` at fetch time; pass key bytes to
       `git2::Callbacks::credentials()`. Never store the key on disk or in memory beyond
       the fetch call.
     - On failure: log `WARN` via `tracing`, return `Err` — do NOT crash the server or
       block other sources from syncing
     - After successful fetch: write merged snapshot to `{local_cache_dir}/{source.name}.toml`
     - Trigger hot reload via the S-1.12 notify watcher (the written file change fires the
       watcher automatically — no explicit signal needed)
   - Error codes:
     - `E-CFG-011`: SSH authentication failure (bad key or key not found)
     - `E-CFG-012`: Network error during fetch
     - `E-CFG-013`: TOML parse error in fetched config

3. Implement background sync scheduler in `crates/prism-mcp/src/config/sync.rs`:
   - `ConfigSyncScheduler::run(sources: Vec<GitConfigSource>, state_dir: PathBuf)`:
     - Spawn one `tokio::task` per source
     - Each task: `tokio::time::interval(Duration::from_secs(source.sync_interval_secs))`
     - Each tick: call `source.fetch_and_write(&state_dir)`, log outcome
     - Source failures do not propagate — each source task is independent
   - `ConfigSyncScheduler::trigger_immediate(source_name: &str)`: call from `prism config sync`
     CLI to force an immediate fetch outside the interval

4. Implement priority-based config merge in `crates/prism-mcp/src/config/merge.rs`:
   - `PriorityMerge::merge(sources: &[(priority: u32, name: &str, config: PrismConfig)]) -> MergedConfig`:
     - Sort sources descending by `priority`
     - For each config key: first non-absent value wins (higher priority source wins)
     - Tie-breaking: if two sources have equal `priority`, lexicographically smaller `name`
       takes precedence (deterministic, documented)
     - Result includes `resolution_trace: HashMap<ConfigKey, SourceAttribution>` for
       the `--trace` CLI command
   - `SourceAttribution { source_name: String, priority: u32, value: ConfigValue }`
   - Merge is applied BEFORE S-5.05's env var override layer — env vars always win
     (BC-2.06.008 invariant preserved)

5. Implement `prism config diff` subcommand in `crates/prism-mcp/src/cli/config_diff.rs`:
   - `prism config diff`: compare current effective config (post-merge, post-env-override)
     with the on-disk TOML files. Shows keys that differ.
   - `prism config diff --pending`: compare the last-synced remote state with what would
     become effective if git sources were re-fetched right now (dry-run sync, no write)
   - Output format: `git diff`-style colored output to stderr; or JSON with `--format json`
   - Keys containing credential refs are masked as `"***"` in diff output

6. Implement `prism config show` subcommand in `crates/prism-mcp/src/cli/config_show.rs`:
   - `prism config show --effective`: print the fully-merged, env-overridden effective
     config as TOML (same masking rules as `--dry-run` from S-5.05)
   - `prism config show --trace <key>`: for a given dot-separated config key
     (e.g., `clients.acme.sensors.crowdstrike.enabled`), print:
     ```
     Key: clients.acme.sensors.crowdstrike.enabled
     Effective value: true
     Source: git_source_name (priority 100)
     Overridden by: [none] | env var PRISM_... | source X (priority N)
     ```
   - Return `E-CFG-014` if the key does not exist in any source

7. Implement merge conflict logging in `crates/prism-mcp/src/config/merge.rs`:
   - When two sources with equal priority conflict on the same key:
     - Log `INFO`: `"Config key '{key}': conflict between '{source_a}' and '{source_b}'
       at priority {p}. Resolved to '{source_a}' (lexicographic precedence)."`
     - This is NOT an error — it is expected behavior and documented

8. Write tests in `crates/prism-mcp/tests/git_config_tests.rs`:
   - Test `PriorityMerge` with 3 sources at priorities 100/50/10 — verify higher priority wins
   - Test priority tie: two sources at priority 50, verify lexicographic name wins
   - Test `SourceAttribution` trace: verify `--trace` returns correct source for each key
   - Test sync failure (mocked network error) does NOT affect other sources' sync tasks
   - Test SSH key rejection (E-CFG-011) does not crash scheduler

## Acceptance Criteria

**AC-1:** Given 3 `[[config_sources]]` entries at priorities 100, 50, and 10, all with
overlapping keys, When the config is merged, Then the source with priority 100 provides
the effective value for each overlapping key.
(traces to BC-2.06.004 postcondition 1)

**AC-2:** Given two sources at equal priority 50, one named `"alpha"` and one named
`"beta"`, When they conflict on a key, Then `"alpha"` wins (lexicographic), and an INFO
log is emitted noting the tie resolution.
(traces to BC-2.06.004 postcondition 1)

**AC-3:** Given `prism config show --trace clients.acme.sensors.crowdstrike.enabled`,
When executed, Then the output names which source provided the value, its priority, and
whether any higher-priority source also set the key.
(traces to BC-2.06.004 postcondition 1)

**AC-4:** Given a git source with `ssh_key_ref = { source = "env", key = "CS_SSH_KEY" }`
and that env var is set, When a sync occurs, Then the key bytes are retrieved from the env
at fetch time and used for git2 authentication. The key is not written to disk.
(traces to BC-2.06.003 postcondition 1; follows AD-017 opaque model)

**AC-5:** Given a network failure during a git source sync, When the sync task runs,
Then a `WARN` log is emitted, the previously-synced config for that source is retained
(not cleared), and other sources continue syncing on their own intervals.
(traces to BC-2.06.005 postcondition 1 — server remains valid)

**AC-6:** Given `prism config diff --pending`, When a git source has a newer commit
not yet fetched, Then the output shows the keys that would change if the fetch were applied.
(traces to BC-2.06.001 postcondition 1 — operator visibility into pending changes)

**AC-7:** Given `PRISM_STORAGE_STATE_DIR=/tmp/prism` in the environment and all
`[[config_sources]]` specify `state_dir = "/var/prism"`, When config is loaded, Then
`config.server.state_dir` is `/tmp/prism` (env overrides all sources).
(traces to BC-2.06.008 postcondition 1)

**AC-8:** Given two `[[config_sources]]` entries with the same `name`, When config
is loaded, Then `E-CFG-010` is returned listing the duplicate name.
(traces to BC-2.06.005 postcondition 1)

## Architecture Compliance Rules

- SSH key bytes MUST NOT be written to disk at any point. They are loaded from the
  credential store at fetch time and discarded after the git2 fetch call completes.
  This is a hard requirement per AD-017.
- git2 fetch calls MUST be executed in `tokio::task::spawn_blocking` — git2 is
  synchronous and will block the async runtime for the duration of the network fetch.
- Each source's sync task MUST be independent — a failure in one source's task MUST
  NOT cancel or delay other sources' tasks. Use separate `tokio::task::spawn` per source.
- The effective config MUST be the result of: git merge → env override (in that order).
  Env overrides MUST always be applied last (BC-2.06.008).
- Sync interval MUST be clamped to a minimum of 60 seconds. Configs specifying a lower
  interval are silently clamped; this prevents accidental API rate-limit hammering on
  git hosts.
- `prism config show --effective` MUST redact all credential reference values. The masking
  logic from S-5.05's `redact.rs` must be reused, not reimplemented.

## Previous Story Intelligence

S-5.05 (Config Loading and Validation) established `PrismConfig`, `ConfigError`, `redact.rs`,
and `validate()`. This story extends `PrismConfig` with `config_sources: Vec<ConfigSource>`
and wraps the existing loader: the merged config from all git sources is assembled first,
then passed through S-5.05's existing `validate()` and `env_override()` pipeline.

S-1.12 (Hot Reload and Runtime Management) established the `notify` watcher. The git sync
writes new TOML files to the local cache directory, which the existing watcher picks up
automatically — no new watcher registration is needed. The hot reload atomicity invariant
(CI-002: failed validation retains old config) applies to merged configs too.

S-1.06 (Credential Store) provides `CredentialStore::get()`. The `GitConfigSource` holds
an `Arc<dyn CredentialStore>` injected at construction time, following the same dependency
injection pattern used by `BackendSelector` in S-1.06.

## Notes

- The local cache directory `{state_dir}/config_cache/` stores the last-successfully-fetched
  snapshot from each git source. On startup, Prism reads these cached snapshots (if they
  exist) before the first sync completes — this ensures the server can start without waiting
  for network fetches.
- Merge conflict resolution (priority, then lexicographic name) is deterministic and
  documented. MSSP operators should use distinct priority values to avoid ties; the
  lexicographic fallback is a safety net, not the recommended workflow.
- `prism config diff --pending` performs a dry-run fetch (clone to a temp dir) without
  modifying the live cache. This lets operators preview changes before the next sync
  interval fires. For large repos, this may take a few seconds — document in the CLI help.
- The sync interval minimum of 60 seconds is intentional. GitHub, GitLab, and Gitea all
  apply rate limits to unauthenticated and authenticated fetches. 60s gives ample headroom
  for 10 sources within GitHub's 5,000 requests/hour limit.
