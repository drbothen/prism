# S-6.02 — prism-bin: End-to-End Integration Smoke Tests

## Narrative

As a Prism platform maintainer, I want a suite of end-to-end smoke tests that exercise
the full pipeline — from binary startup through MCP tool calls to RocksDB persistence
and graceful shutdown — so that regressions in any subsystem are caught before release
and I can ship with confidence that the integrated system works as a whole.

## Objective

Build a mock sensor HTTP server and a test harness that spawns `prism-bin` as a child
process, communicates via the stdio MCP protocol, and verifies the full query, detection,
scheduling, case management, credential, and shutdown pipelines. All smoke tests must
pass in CI within 60 seconds total.

## Tasks

1. Implement mock sensor HTTP server in `crates/prism-bin/tests/smoke/mock_sensor.rs`:
   - Serve canned responses for CrowdStrike, Claroty, Cyberint, and Armis API endpoints
   - Support all endpoints exercised by the smoke tests (device list, vulnerability
     list, alert list, auth endpoints)
   - Configurable response delays to test timeout behavior
   - Configurable error injection: HTTP 429, 401, 500 on demand
   - Bind to a random available port; return the bound address for test config generation
   - Use `tokio::net::TcpListener` + minimal `hyper` or `axum` handler

2. Implement test harness in `crates/prism-bin/tests/smoke/harness.rs`:
   - `PrismTestProcess::spawn(config: TestConfig) -> PrismTestProcess`
   - Generates a temporary `prism.toml` pointing sensor URLs to the mock server
   - Generates a temporary `state_dir` for RocksDB
   - Spawns `prism serve` as a child process (using `std::process::Command` with
     `stdin(Stdio::piped())`, `stdout(Stdio::piped())`, `stderr(Stdio::piped())`)
   - `send_mcp(&mut self, request: Value) -> Value`: write JSON-RPC to stdin, read
     response from stdout
   - `shutdown(&mut self)`: send SIGTERM, wait for process exit, verify exit code
   - Auto-cleanup: `Drop` impl sends SIGTERM if test exits without explicit shutdown

3. Implement query pipeline smoke test in `tests/smoke/query_pipeline.rs`:
   - Spawn Prism with mock sensors
   - Send MCP `initialize` → verify tool list includes `query`
   - Send `query` tool call with `SELECT * FROM crowdstrike.devices LIMIT 5`
   - Verify response: OCSF-normalized JSON array, 5 or fewer results, no error
   - Verify `class_uid` field present in each result (OCSF normalization applied)

4. Implement scheduled query pipeline smoke test in `tests/smoke/schedule_pipeline.rs`:
   - Create a schedule with 1-second interval (test only — not production config)
   - Wait for first tick (poll `get_diff_results` until non-empty, timeout 10s)
   - Verify differential results contain the expected columns
   - Delete the schedule; verify it no longer appears in `list_schedules`

5. Implement detection pipeline smoke test in `tests/smoke/detection_pipeline.rs`:
   - Create a detection rule matching a field present in mock sensor data
   - Run a query that triggers the rule (the mock sensor serves data that matches)
   - Verify an alert appears in a subsequent `list_alerts` or `get_diagnostics` call
   (if `list_alerts` is not yet implemented, verify via internal audit log output)

6. Implement case management pipeline smoke test in `tests/smoke/case_pipeline.rs`:
   - Call `create_case` with a test case
   - Call `update_case` to transition to "investigating"
   - Call `get_case` and verify status is "investigating"
   - Call `case_metrics` and verify the case appears in counts
   - Call `update_case` to transition to "resolved" with disposition
   - Verify `case_metrics` reflects the resolved state

7. Implement credential pipeline smoke test in `tests/smoke/credential_pipeline.rs`:
   - Call `configure_credential_source` to register a test credential source reference (e.g., env var pointing to a test API key)
   - Run a query that uses the credential (verify the mock sensor received the key in
     the Authorization header)
   - Call `delete_credential` and verify subsequent queries fail with `E-AUTH-*`
   - Verify the audit log records both `configure_credential_source` and `delete_credential` events

8. Implement shutdown smoke test in `tests/smoke/shutdown.rs`:
   - Send SIGTERM to running Prism process
   - Verify exit code is 0 and process exits within 5 seconds
   - Verify RocksDB LOCK file is absent after shutdown
   - Spawn a second Prism process against the same state_dir
   - Verify it starts successfully (LOCK was released)

9. Implement concurrent-open prevention test in `tests/smoke/shutdown.rs`:
   - Spawn first Prism instance (holds RocksDB LOCK)
   - Attempt to spawn second Prism instance against the same state_dir
   - Verify second instance exits with a non-zero code and an error message mentioning
     the LOCK file (DI-017)

10. Configure CI timeout: add `#[tokio::test(timeout = Duration::from_secs(60))]` or
    equivalent to the combined smoke test runner so the full suite fails if it exceeds
    60 seconds.

## Acceptance Criteria

**AC-1:** Given a fresh Prism instance with mock sensors returning canned CrowdStrike
device data, When `SELECT * FROM crowdstrike.devices LIMIT 5` is executed via the `query`
tool, Then the response is a JSON array of up to 5 OCSF-normalized records, each
containing `class_uid`.

**AC-2:** Given a schedule with a 1-second interval, When the tick fires, Then
`get_diff_results` returns non-empty differential output within 10 seconds of schedule
creation.

**AC-3:** Given a detection rule matching mock alert data, When a query returns matching
events, Then an alert record is created and visible via diagnostics or alert listing.

**AC-4:** Given a case created and transitioned through "investigating" → "resolved",
When `case_metrics` is called, Then the resolved case appears in the appropriate metric
bucket.

**AC-5:** Given SIGTERM sent to a running Prism instance, When received, Then the process
exits with code 0 within 5 seconds and the RocksDB LOCK file is absent afterwards.

**AC-6:** Given a running Prism instance holding the RocksDB LOCK, When a second Prism
instance starts against the same state directory, Then the second instance exits with a
non-zero code and a message explaining the LOCK conflict.

**AC-7:** Given all smoke tests are run sequentially in CI, When all pass, Then the
total wall-clock time is under 60 seconds.

## Architecture Compliance Rules

- The test harness MUST communicate via stdio MCP protocol (not internal Rust function
  calls) — the tests must exercise the actual transport layer
- Mock sensor server MUST bind to `127.0.0.1` only — never `0.0.0.0` in test code
- Each smoke test MUST use an isolated temporary directory for RocksDB state — tests
  MUST NOT share state directories
- The harness `Drop` impl MUST send SIGTERM on cleanup — never leave orphan processes
- Smoke tests MUST NOT use `sleep` with fixed durations — poll with timeout instead
  (prevents flakiness on slow CI machines)

## Dev Notes

- The 1-second schedule interval is only safe in tests because the scheduler uses a
  configurable tick frequency. Production configs should enforce a minimum interval
  (e.g., 60 seconds). The test config bypasses this minimum for smoke test speed.
- The detection pipeline test may need a short poll loop (check every 100ms, timeout
  5s) after the triggering query to wait for the detection engine to process the results
  and create the alert asynchronously.
- CI time budget: 60 seconds total for all smoke tests. Roughly: startup 3s, query 2s,
  schedule 12s (wait for tick), detection 5s, case 3s, credential 5s, shutdown 5s,
  LOCK test 5s. Total ~40s with margin. If the schedule tick test becomes the bottleneck,
  consider using a mock scheduler clock in the test harness.
- The credential pipeline test's verification that the mock sensor "received the key in
  the Authorization header" requires the mock server to log received headers. Build this
  into `mock_sensor.rs` from the start.
