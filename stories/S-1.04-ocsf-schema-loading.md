# S-1.04 — prism-ocsf: Schema Loading and DynamicMessage

## Narrative

As a Prism platform implementer, I want the OCSF normalization foundation established
in prism-ocsf, so that raw sensor records from CrowdStrike, Cyberint, Claroty, and
Armis can be converted to DynamicMessage instances conforming to the pinned OCSF
protobuf schema — enabling DataFusion materialization and cross-sensor querying in
Wave 3.

## Objective

Build the OCSF normalization infrastructure: compile-time schema loading via
ocsf-proto-gen, runtime DescriptorPool initialization, DynamicMessage creation,
OCSF version pinning, enum display map, and event class selection. Field mapping
for each sensor is deferred to S-1.05. Two verification properties (VP-016 proptest,
VP-022 fuzz target) validate output correctness and panic safety.

## Tasks

1. Create `crates/prism-ocsf/Cargo.toml`:
   - Dependencies: `prism-core`, `prost`, `prost-reflect` (for DynamicMessage),
     `serde`, `serde_json`, `tracing`
   - Build dependencies: `ocsf-proto-gen`, `prost-build`
   - Dev dependencies: `proptest`, `cargo-fuzz` target config

2. Create `crates/prism-ocsf/build.rs`:
   - Invoke `ocsf_proto_gen::compile_ocsf_schema()` to generate Rust protobuf
     descriptors from the bundled OCSF JSON schema files (BC-2.02.001)
   - Emit `cargo:rerun-if-changed=ocsf-schema/` so the build reruns when schema
     changes
   - Write the OCSF version string to `OUT_DIR/ocsf_version.txt` for inclusion via
     `include_str!()` at compile time (BC-2.02.009)

3. Implement `DescriptorPool` initialization in `crates/prism-ocsf/src/pool.rs`:
   - Use `OnceLock<DescriptorPool>` — initialized exactly once at process startup
   - `OcsfDescriptors::get() -> &'static DescriptorPool` is the only public access
   - The pool is built from the protobuf file descriptor set emitted by `ocsf-proto-gen`
   - Panic on initialization failure (boot-time invariant, not recoverable)

4. Implement `ocsf_version() -> &'static str` in `crates/prism-ocsf/src/version.rs`:
   - Returns the compile-time OCSF schema version via `include_str!()` from
     `OUT_DIR/ocsf_version.txt`
   - Version format: semver string, e.g., `"1.3.0"` (BC-2.02.009)

5. Implement `EventClassSelector` in `crates/prism-ocsf/src/class_selector.rs`:
   - Enum or struct mapping sensor + record_type → OCSF class_uid (BC-2.02.012)
   - Known mappings (MUST be validated against the pinned OCSF schema version at
     implementation time using ocsf-proto-gen output — these are initial assignments
     based on OCSF v1.3.0+ class definitions, subject to verification):
     - CrowdStrike detection → Detection Finding (class_uid 2004)
     - CrowdStrike incident → Incident Finding (class_uid 2005)
     - Cyberint alert → Detection Finding (class_uid 2004) + OSINT profile
     - Claroty alert → Detection Finding (class_uid 2004)
     - Claroty asset → Device Inventory Info (class_uid 5001)
     - Claroty vulnerability → Vulnerability Finding (class_uid 2002)
     - Armis device → Device Inventory Info (class_uid 5001)
     - Armis alert → Detection Finding (class_uid 2004)
   - IMPORTANT: The implementing agent MUST run ocsf-proto-gen against the pinned OCSF
     schema version (BC-2.02.009) and verify that each class_uid exists in the compiled
     descriptors. If a class_uid is not present, check the latest OCSF schema release
     (currently v1.7.0+) and update the mapping. Security Finding (2001) is DEPRECATED
     since OCSF v1.1.0 — do not use it.
   - `EventClassSelector::select(sensor: &str, record_type: &str) -> Result<u32, PrismError>`
   - Unknown combinations return `Err(PrismError::OcsfUnknownEventClass)`

6. Implement OCSF enum display map in `crates/prism-ocsf/src/enum_map.rs`:
   - `OcsfEnumMap` struct backed by `HashMap<(String, u32), &'static str>`
   - Populated at startup from the protobuf descriptor's enum value options
   - `OcsfEnumMap::display_name(field: &str, value: u32) -> Option<&'static str>`
   - Examples: `severity_id` → {1: "Informational", 2: "Low", 3: "Medium", 4: "High",
     5: "Critical"}, `activity_id` → {1: "Create", 2: "Read", 3: "Update", 4: "Delete"}
   - (BC-2.02.010)

7. Implement `OcsfNormalizer` in `crates/prism-ocsf/src/normalizer.rs`:
   - `OcsfNormalizer::new() -> Self` — holds a reference to the `OcsfDescriptors`
   - `OcsfNormalizer::normalize(sensor: &str, record_type: &str, raw: serde_json::Value) -> Result<DynamicMessage, PrismError>`:
     1. Call `EventClassSelector::select(sensor, record_type)` to get `class_uid`
     2. Look up the message descriptor from `DescriptorPool` for the class
     3. Create an empty `DynamicMessage` for that descriptor
     4. Delegate field population to the sensor-specific mapper (S-1.05 provides these)
     5. Return the populated `DynamicMessage` (BC-2.02.002)
   - Never panic on malformed input — all errors return `Err(PrismError::OcsfNormalizationFailed)`

8. Write VP-016 proptest in `crates/prism-ocsf/src/tests/proptest_normalizer.rs`:
   - Strategy: generate arbitrary valid-looking JSON objects
   - For each, call `normalize("crowdstrike", "detection", json_val)`
   - If `Ok(msg)`, assert the message can be encoded to bytes and decoded back
     (valid protobuf round-trip)
   - Property name: `prop_normalize_output_is_valid_protobuf`

9. Write VP-022 fuzz target in `fuzz/fuzz_targets/normalize_fuzz.rs`:
   - Input: arbitrary `&[u8]`
   - Attempt `serde_json::from_slice()` then call `normalize()`
   - Assert: no panic (all errors must be returned via `Result`, never unwrap/panic)
   - Run requirement: 30 minutes minimum with no panics found

10. Write unit tests in `crates/prism-ocsf/src/tests/`:
    - Test `ocsf_version()` returns a non-empty semver string
    - Test `EventClassSelector::select("crowdstrike", "detection")` returns 2001
    - Test `EventClassSelector::select("unknown", "unknown")` returns Err
    - Test `OcsfEnumMap::display_name("severity_id", 4)` returns Some("High")
    - Test `OcsfEnumMap::display_name("severity_id", 99)` returns None

## Acceptance Criteria

**AC-1:** Given the compiled binary, When `ocsf_version()` is called, Then it returns
a non-empty semver string matching the pinned OCSF schema version baked in at compile
time. (BC-2.02.009)

**AC-2:** Given the `DescriptorPool` singleton, When queried for the Detection Finding
event class descriptor (class_uid 2004), Then it returns a descriptor with `class_uid`
field present. (BC-2.02.001)

**AC-3:** Given a raw CrowdStrike detection JSON `{"detection_id": "abc123",
"severity": "High"}`, When `OcsfNormalizer::normalize("crowdstrike", "detection", json)`
is called, Then a `DynamicMessage` is returned with `class_uid = 2004` (Detection Finding,
per OCSF v1.3.0). (BC-2.02.002)

**AC-4:** Given `OcsfEnumMap::display_name("severity_id", 4)`, Then it returns
`Some("High")`. (BC-2.02.010)

**AC-5:** Given `OcsfEnumMap::display_name("severity_id", 4)`, When the result is
used, Then `display_name("severity_id", 99)` returns `None` (unknown enum value
handled gracefully, not panicked). (BC-2.02.010)

**AC-6:** Given `EventClassSelector::select("crowdstrike", "detection")`, Then it
returns `Ok(2001)`. (BC-2.02.012)

**AC-7:** Given `EventClassSelector::select("claroty", "alert")`, Then it returns
`Ok(2004)`. (BC-2.02.012)

**AC-8:** Given `EventClassSelector::select("vendor_x", "unknown_type")`, Then it
returns `Err(PrismError::OcsfUnknownEventClass)`. (BC-2.02.012)

**AC-9:** VP-016 proptest passes: all `Ok` normalize outputs are valid protobuf
(encode → decode round-trip succeeds).

**AC-10:** VP-022 fuzz target runs 30 minutes with no panics on arbitrary byte input.

## Architecture Compliance Rules

- `DescriptorPool` initialization MUST use `OnceLock` — never `Mutex<Option<...>>`
  (the lock is held for the entire process lifetime; `OnceLock` avoids poisoning)
- `normalize()` MUST NOT panic — all errors via `Result`; fuzz target (VP-022) enforces this
- Build script MUST use `ocsf-proto-gen` — do not hand-write protobuf `.proto` files
- `OcsfNormalizer` MUST be `Send + Sync` (used from async tokio runtime)
- The `EventClassSelector` MUST be a compile-time constant structure — no runtime
  configuration of class mappings

## Notes

- `ocsf-proto-gen` is an internal build tool that reads OCSF JSON schema files and
  emits a protobuf file descriptor set. Confirm the tool is available in the workspace
  before beginning the build script.
- The `DescriptorPool` singleton panics on failure because it is initialized at process
  boot — a missing descriptor pool means the binary cannot function at all. This is
  intentional and correct; it is not a recoverable error.
- S-1.05 (field mapping) depends on this story. S-1.04 provides the `DynamicMessage`
  infrastructure; S-1.05 fills in the field values for each sensor.
- The fuzz target (VP-022) must be placed under `fuzz/fuzz_targets/` as per
  cargo-fuzz conventions. Configure `Cargo.toml` in the `fuzz/` workspace member.
