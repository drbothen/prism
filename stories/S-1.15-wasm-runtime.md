# S-1.15 — prism-spec-engine: WASM Plugin Runtime

## Narrative

As a Prism platform operator, I want to extend sensor adapters, infusions, and actions
with custom logic written in any language (Rust, Go, Python, JavaScript, C#), so that
MSSP engineers can ship `.prx` WASM Component Model plugins without modifying Prism's
core Rust codebase, and those plugins run in a sandboxed environment with no ability to
escape their defined host interface.

## Objective

Implement the WASM Component Model plugin runtime per AD-019: load `.prx` files using
`wasmtime` with component model support, define and implement WIT host interfaces for
sensor, infusion, and action plugins, enforce sandbox constraints (memory limits, CPU
time limits, no filesystem/network direct access), implement hot reload via `notify`
crate, implement plugin discovery from the plugins directory, expose host functions
(HTTP proxy, key-value store, config getter), and isolate plugin panics so they do not
crash the host process.

## Tasks

1. Define `PluginRuntime` and core types in `crates/prism-spec-engine/src/plugin/mod.rs`:
   - `PluginType`: `Sensor | Infusion | Action`
   - `PluginMetadata { plugin_id: String, name: String, version: String, plugin_type: PluginType, path: PathBuf }`
   - `PluginRuntime`: holds `wasmtime::Engine`, `wasmtime::component::Linker<HostState>`, and the plugin registry `HashMap<String, Arc<LoadedPlugin>>` behind `arc_swap::ArcSwap`
   - `LoadedPlugin { metadata: PluginMetadata, component: wasmtime::component::Component, pre_instance: wasmtime::component::InstancePre<HostState> }`
   - `HostState { http_client: reqwest::Client, config: Arc<PluginConfigMap>, kv_store: Arc<PluginKvStore> }` — thread-safe state passed to every plugin instance

2. Set up wasmtime engine and linker in `crates/prism-spec-engine/src/plugin/loader.rs`:
   - Initialize `wasmtime::Engine` with `Config::new().wasm_component_model(true).epoch_interruption(true)`
   - Enable epoch interruption for CPU time limiting
   - Configure `wasmtime::component::Linker<HostState>` with all host function bindings (step 4)
   - `load_plugin(path: &Path) -> Result<LoadedPlugin>`:
     - Read `.prx` bytes (WASM Component binary)
     - Compile with `Component::from_binary(&engine, bytes)?`
     - Pre-instantiate with `linker.instantiate_pre(&component)?`
     - Validate WIT interface by attempting to lift exported functions (step 6)
   - Cache compiled modules — `Component` compilation is expensive; do not recompile on every call

3. Implement WIT interface definitions in `crates/prism-spec-engine/wit/`:
   - `prism-sensor-plugin.wit`: sensor plugin interface — `fetch-page`, `name`, `version`
     (defined in AD-019; copy exact interface from architecture spec)
   - `prism-infusion-plugin.wit`: infusion plugin interface — `enrich-single`, `enrich-batch`,
     `name`, `version` (from `specs/architecture/infusions.md` WIT block)
   - `prism-action-plugin.wit`: action plugin interface — `fire-alert`, `fire-case`,
     `fire-report`, `name`, `version` (from `specs/architecture/actions.md` WIT block)
   - All three WIT files include the shared `host` interface for HTTP, logging, config, and
     the shared `log-level` enum

4. Implement host functions in `crates/prism-spec-engine/src/plugin/host_functions.rs`:
   - `host::http_request(method, url, headers, body) -> HttpResponse`:
     - Plugins CANNOT make direct network calls — all HTTP goes through this host proxy
     - Execute via `HostState.http_client` (reqwest)
     - Apply URL allowlist validation: if the plugin's `[source.allowed_urls]` is configured,
       reject requests to unlisted domains with HTTP 403 equivalent
     - Timeout: 10 seconds per request (separate from per-call CPU limit)
     - Audit-log all outbound HTTP calls (plugin_id, method, url, status, latency_ms)
   - `host::log(level, message)`: forward to tracing at the corresponding level, prefixed with
     `[plugin:{plugin_id}]`
   - `host::get_config(key) -> Option<String>`: look up key in `HostState.config` (populated
     from `[infusion.plugin_config]` / `[action.plugin_config]` at load time). Returns `None`
     for unknown keys — never errors.
   - `host::kv_get(key) -> Option<String>` / `host::kv_set(key, value)`: simple per-plugin
     key-value store backed by `HostState.kv_store`. Scoped to the plugin — plugins cannot
     access other plugins' KV state. Persisted via `CacheBackend` trait (defined in
     prism-core, implemented by prism-storage) injected at startup — prism-spec-engine
     does NOT depend on prism-storage directly. Uses `plugin_state` CF in production,
     key format: `"{plugin_id}:{key}"`, values encoded as bincode 2.x.

5. Implement sandbox enforcement in `crates/prism-spec-engine/src/plugin/sandbox.rs`:
   - Memory limit: configure wasmtime `StoreLimits` with `memory_size: 64 * 1024 * 1024`
     (64MB default, configurable per plugin via `[plugin_config].memory_limit_mb`)
   - CPU time limit: use epoch interruption. Set `Store::epoch_deadline` to
     `(timeout_seconds * epoch_ticks_per_second)`. Start a background tokio task that
     increments the engine epoch every 1ms. On deadline: trap the plugin, return
     `Err(PluginError::Timeout { plugin_id, duration_ms })`
   - Panic/trap isolation: wrap all `instance.call_*` invocations in a catch that converts
     `wasmtime::Trap` into `Err(PluginError::Trapped { plugin_id, message })`. The host
     process MUST NOT unwind past this boundary.
   - Plugin instance is created fresh per call (not reused across calls) to avoid state
     leakage between invocations. Instantiation cost is amortized by `InstancePre`.

6. Implement plugin discovery in `crates/prism-spec-engine/src/plugin/discovery.rs`:
   - Scan `{config_dir}/plugins/*.prx` on startup
   - For each `.prx` file, attempt `load_plugin(path)` and WIT interface validation
   - WIT validation: attempt to look up the expected exported function names on the
     component. If `name()`, `version()`, and the primary dispatch function are present,
     accept. Otherwise, reject with `E-PLUGIN-001: "Plugin '{path}' does not implement a
     recognized Prism WIT interface. Expected one of: prism:sensor-plugin, prism:infusion-plugin,
     prism:action-plugin."`
   - Incompatible plugins are logged and skipped — they do not block other plugins from loading
   - On startup: scan all `.prx` files, load valid ones, populate `PluginRuntime.registry`

7. Implement hot reload in `crates/prism-spec-engine/src/plugin/hot_reload.rs`:
   - Watch `{config_dir}/plugins/*.prx` via `notify` crate (same watcher as sensor/infusion)
   - On file create/modify: load the new `.prx`, validate WIT, compile (may take ~100ms for
     large WASM modules), swap the entry in the plugin registry via arc-swap
   - In-flight calls using the old `InstancePre` will complete normally (Arc keeps it alive)
   - On file delete: remove the plugin from the registry. Callers that had a reference to the
     old plugin via Arc will complete their call; new calls get `E-PLUGIN-011: "Plugin
     '{plugin_id}' is not loaded. Run list_plugins to see available plugins."`
   - Failed recompilation (bad WASM): log error, retain old plugin. Do not unload a working
     plugin just because a new version fails to compile.

8. Implement public API for callers (S-1.14 infusion bridge, S-4.08 action engine):
   - `PluginRuntime::enrich_single(plugin_id: &str, input_value: &str, input_type: &str, config: &PluginConfigMap) -> Result<Option<serde_json::Value>, PluginError>`
   - `PluginRuntime::enrich_batch(plugin_id: &str, inputs: &[String], input_type: &str, config: &PluginConfigMap) -> Result<Vec<Option<serde_json::Value>>, PluginError>`
   - `PluginRuntime::fire_alert(plugin_id: &str, ctx: AlertContext, config: &PluginConfigMap) -> Result<ActionResult, PluginError>`
   - `PluginRuntime::fire_case(plugin_id: &str, ctx: CaseContext, config: &PluginConfigMap) -> Result<ActionResult, PluginError>`
   - `PluginRuntime::fire_report(plugin_id: &str, ctx: ReportContext, config: &PluginConfigMap) -> Result<ActionResult, PluginError>`
   - All methods: instantiate from `InstancePre`, set epoch deadline, call WIT export, return
     result or `PluginError`.

9. Write tests in `crates/prism-spec-engine/tests/plugin_tests.rs`:
   - Load a minimal valid infusion `.prx` fixture → verify registered in runtime
   - Call `enrich_single` → verify returned value matches expected fixture output
   - Simulate plugin trap (compile a WAT module that `unreachable`s) → verify `Err(Trapped)`
     returned, host process continues
   - Simulate plugin timeout (WAT module with infinite loop) → verify `Err(Timeout)` within
     configured deadline, host process continues
   - Verify memory limit: WAT module that allocates > 64MB → verify `Err(MemoryExceeded)`
   - Verify hot reload: drop `.prx` file, reload with modified fixture → verify updated behavior
   - Verify `host::http_request` proxy: mock HTTP server, plugin calls `http_request` → verify
     request goes through host proxy, not direct from plugin WASM

## Acceptance Criteria

**AC-1:** Given a valid `.prx` file implementing `prism:infusion-plugin@0.1.0`, When
`PluginRuntime::load_plugin` is called, Then the plugin is compiled, validated, and
added to the registry without error.
(traces to BC-2.17.006 / INV-PLUGIN-006)

**AC-2:** Given a `.prx` plugin that panics (executes `unreachable` instruction), When
any call is made to the plugin, Then `Err(PluginError::Trapped)` is returned and the
Prism host process continues executing normally.
(traces to BC-2.17.001 / INV-PLUGIN-001)

**AC-3:** Given a plugin configured with the default 5-second CPU time limit, When the
plugin executes an infinite loop, Then the call returns `Err(PluginError::Timeout)` within
6 seconds (5s limit + 1s tolerance) and the host process is unaffected.
(traces to BC-2.17.004 / INV-PLUGIN-004)

**AC-4:** Given a plugin that attempts to open a file via a WASI filesystem call, When
the plugin executes, Then the call fails with a WASM trap because no filesystem WASI
interface is linked — the plugin cannot read host files.
(traces to BC-2.17.002 / INV-PLUGIN-002)

**AC-5:** Given a plugin that calls `host::http_request` with a URL, When the call
executes, Then the request is issued via the host's `reqwest::Client` (not directly from
WASM), and the call is audit-logged with plugin_id, URL, method, and response status.
(traces to BC-2.17.002 / INV-PLUGIN-002)

**AC-6:** Given a loaded plugin `.prx`, When the file is replaced with a new version and
the file watcher fires, Then the plugin registry is updated via arc-swap, in-flight calls
using the old module complete without error, and new calls use the new module.
(traces to BC-2.17.005 / INV-PLUGIN-005)

**AC-7:** Given a `.prx` file that does not export the `name`, `version`, or expected
dispatch functions, When `PluginRuntime::load_plugin` is called, Then it returns
`Err` with error code `E-PLUGIN-001` and the plugin is not added to the registry.
(traces to BC-2.17.006 / INV-PLUGIN-006)

**AC-8:** Given two different plugins loaded in the same `PluginRuntime`, When plugin A
calls `host::kv_set("mykey", "myval")` and plugin B calls `host::kv_get("mykey")`,
Then plugin B receives `None` (KV stores are scoped per plugin, not shared).

**AC-9:** Given a plugin registered with the default memory limit (64 MiB) via
`wasmtime::StoreLimits`, When the plugin's WASM execution attempts to allocate memory
beyond 64 MiB (e.g., a WAT module calling `memory.grow` past the limit), Then the
`StoreLimits` memory guard fires a WASM trap, the trap is caught at the
`instance.call_*` boundary, and the calling method returns
`Err(PluginError::MemoryExceeded { plugin_id, limit_mb: 64 })`. A `WARN`-level log
entry is emitted: `"Plugin '{plugin_id}' exceeded memory limit of 64MB"`. The plugin
instance and its `wasmtime::Store` are dropped immediately. The host process is
unaffected. The plugin registry entry is retained (the plugin is not unregistered).
(traces to BC-2.17.003 / INV-PLUGIN-003 — memory limit enforced per plugin instance, default 64MB)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| WASM plugin runtime | `prism-spec-engine` (SS-16, SS-17) | Mixed |

`prism-spec-engine` hosts the `PluginRuntime` powered by `wasmtime` (AD-019). WASM module
loading, sandbox initialization, and WIT interface binding are effectful; the plugin
execution dispatch logic (selecting the right plugin, calling the right export) is pure.
Per `architecture/module-decomposition.md`, SS-17 (WASM Plugin Runtime) is owned by
`prism-spec-engine`. `prism-operations` consumes `PluginRuntime` for action delivery
plugins; `prism-spec-engine` uses it for sensor and infusion plugins.

## Architecture Compliance Rules

- `wasmtime::Store` MUST be created fresh per plugin call — stores are not thread-safe
  and must not be reused across async tasks. The `InstancePre` is thread-safe and
  cloneable; instantiation from `InstancePre` is fast (nanoseconds).
- Epoch interruption background task MUST be a long-running tokio task started once at
  `PluginRuntime` construction time — not spawned per plugin call. Use
  `tokio::time::interval(Duration::from_millis(1))` to tick the engine epoch.
- Plugin KV store writes to RocksDB MUST be scoped under the `plugin_state` CF, not the
  `decorators` or `infusion_cache` CFs — key format: `"{plugin_id}:{key}"`.
- Host HTTP proxy MUST enforce URL allowlist if configured. No plugin may make arbitrary
  outbound network calls — all calls must be audited and allowlisted.
- Do NOT enable WASI for plugin instances. The only host interface available to plugins
  is the explicitly-linked `host` interface defined in the WIT files. Filesystem, network,
  process, and environment WASI interfaces must not be linked.
- Plugin compilation (`Component::from_binary`) is CPU-intensive and should NOT block
  the tokio runtime thread. Spawn in `tokio::task::spawn_blocking` during hot reload.

## Dev Notes

- The wasmtime Component Model's `InstancePre::instantiate` call is very fast (allocates
  a new `Store` and links imports against a pre-validated component) — benchmarks show
  ~1-10 microseconds. This makes per-call instantiation practical without performance
  concerns.
- WAT fixtures for tests must be compiled to WASM (`.wasm`) during the build. Use
  `build.rs` with `wat::parse_file` to compile `.wat` files to `.wasm`, then wrap
  as WASM Component Model components using `wasm-tools component new`. Alternatively,
  check in pre-compiled `.wasm` test fixtures.
- The WIT files in `crates/prism-spec-engine/wit/` are the source of truth for the
  plugin contract. `wit-bindgen` generates Rust host bindings in `build.rs`. Third-party
  plugin authors will compile their code against these WIT files to produce `.prx` files.
- Plugin KV store is for transient plugin state (e.g., OAuth token caching by a threat
  intel plugin) — not for long-term data. Size limit: 1MB per plugin total in the KV
  store. Exceeding the limit returns `E-PLUGIN-003` from `kv_set` (with `{resource}`=`"kv_store"`, `{limit}`=`"1MB"`).
- `E-RULE-012` rejection of API-backed infusion UDFs in detection rules (from S-1.14)
  relies on knowing which registered UDF names come from `type = "plugin"` infusions.
  The `InfusionRegistry` must expose a `is_plugin_backed(udf_name: &str) -> bool` query
  for the detection rule validator to call.
