## Objective

Implement credential CRUD operations, query-time resolution, secret redaction, and audit event emission.

## Behavioral Contracts

| BC ID | Title |
|-------|-------|
| BC-2.03.005 | Credential CRUD Operations via MCP Tools (Mutations Require Confirmation Token) |
| BC-2.03.006 | Credential Resolution at Sensor Query Time |
| BC-2.03.007 | Secret Redaction in Logs, Errors, and MCP Responses |
| BC-2.03.009 | resolve_secret() for _FILE Env Var and K8s Secret Mount Compatibility |
| BC-2.03.010 | Credential Access Audit Logging |

## Tasks

1. Implement `configure_credential_source` (accepts source type refs only: env/file/vault/keyring — NEVER raw values), `credential_status` (returns source-type metadata, never raw values), `delete_credential`, `list_credentials` operations — mutations require confirmation token (BC-2.03.005)
2. Implement credential resolution chain for sensor query time: check credential store by `(tenant, sensor, name)` → return opaque reference, never raw value (BC-2.03.006)
3. Implement `Secret<T>` wrapper type that redacts value in `Display`, `Debug`, `serde::Serialize` — value only accessible via `.expose()` (BC-2.03.007)
4. Implement `resolve_secret()` function: check `{name}_FILE` env var (read file contents), then `{name}` env var, then credential store lookup (BC-2.03.009)
5. Implement `AuditEvent` emission for all credential access: who accessed, which credential (name only, not value), outcome (BC-2.03.010)

## Acceptance Criteria

- **AC-1:** Given a `configure_credential_source` request (with source type ref: env/file/vault/keyring) without confirmation token, When the mutation is attempted, Then it returns E-CRED-003 requiring confirmation (BC-2.03.005)
- **AC-2:** Given a configured sensor with credential ref `"api_key"`, When a query resolves the credential, Then the raw value is passed to the HTTP client internally but never returned in MCP response (BC-2.03.006)
- **AC-3:** Given a `Secret<String>` containing `"sk-12345"`, When `format!("{:?}", secret)` is called, Then output is `"Secret(***)"` (BC-2.03.007)
- **AC-4:** Given env var `CROWDSTRIKE_API_KEY_FILE=/tmp/key.txt` containing `"secret"`, When `resolve_secret("CROWDSTRIKE_API_KEY")` is called, Then it returns `"secret"` (BC-2.03.009)
- **AC-5:** Given a credential access, When the audit event is emitted, Then it contains `operator_id`, `credential_name` (not value), `access_type`, and `timestamp` (BC-2.03.010)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Credential CRUD operations | `prism-credentials` (SS-03) | Effectful |
| MCP credential tool surface | `prism-mcp` (SS-10) | Effectful |

`prism-credentials` implements CRUD against `KeyringBackend` and `EncryptedFileBackend`.
`prism-mcp` exposes these as MCP tools. Audit trail writes flow through `prism-audit`.
Per `architecture/module-decomposition.md`, `prism-mcp` depends on `prism-credentials` via
the `CredentialStore` trait and on `prism-audit` via `AuditEmitter`.

|----|----------|-------------------|
| EC-001 | `configure_credential_source` called without confirmation token | Returns E-CRED-003 requiring confirmation (BC-2.03.005) |
| EC-002 | `resolve_secret()` finds `{name}_FILE` env var pointing to a non-existent file | Returns `Err(CredentialStoreError)` with file path (BC-2.03.009) |
| EC-003 | `Secret<T>` value printed via `{:?}` or `{}` | Output is `"Secret(***)"` — value never appears (BC-2.03.007) |

| EC-005 | `credential_status` called for non-existent credential | Returns `None` / not-found response — never errors with secret context |

## Dev Notes

- `Secret<T>` must implement `Zeroize` on drop to clear memory.
- `credential_status` returns source-type metadata (source type, name, tenant, sensor) — the raw secret value is never returned via MCP.
- Confirmation token requirement for mutations is enforced by `prism-security` (S-1.08/S-1.09); this story integrates with that system.
- `resolve_secret()` priority order: `{name}_FILE` env var > `{name}` env var > credential store. This matches Kubernetes secret mount conventions.
- Audit events must be emitted even on resolution failures (log failure outcome, not the attempted secret value).
