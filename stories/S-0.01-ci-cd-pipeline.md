# S-0.01 — devops: CI/CD Pipeline and Release Workflow

## Narrative

As a Prism maintainer, I want a fully automated CI/CD pipeline that gates every PR on
formatting, clippy, tests, dependency audits, and semver compatibility across 5 target
platforms, runs Kani proofs and fuzz corpus on merge to main, and publishes signed
release binaries to GitHub Releases and crates.io on git tag, so that the project
enforces quality automatically and releases are reproducible and trustworthy.

## Objective

Implement the GitHub Actions workflow files for the PR gate, post-merge verification,
and release pipeline. Configure Dependabot for dependency update automation. All five
build targets run in parallel on each PR. Kani proofs (300 s / 8192 MB each) and fuzz
corpus jobs (30 min) run post-merge to avoid blocking the PR cycle. The release workflow
is triggered by a `v*` tag and produces signed binaries, SHA-256 checksums, a Homebrew
formula update PR, and a Chocolatey publish stub.

## Tasks

1. Create `.github/workflows/ci.yml` — PR Gate:
   - Trigger: `on: [push, pull_request]` targeting all branches
   - Matrix strategy: 5 runners in parallel:
     - `macos-latest` (ARM64) with `aarch64-apple-darwin` target
     - `macos-13` (x86_64) with `x86_64-apple-darwin` target
     - `ubuntu-latest` with `x86_64-unknown-linux-gnu` target
     - `ubuntu-latest` with `x86_64-unknown-linux-musl` target (install musl-tools)
     - `windows-latest` with `x86_64-pc-windows-msvc` target
   - Step sequence per runner (strict order, matching `tooling-selection.md`):
     1. `cargo fmt --check` — formatting gate
     2. `cargo clippy -- -D warnings` — lint gate (clippy.toml applied automatically)
     3. `cargo test --workspace` — unit + proptest suite
     4. `cargo deny check` — license allowlist + advisory denylist (deny.toml)
     5. `cargo audit` — RustSec vulnerability scan
     6. `cargo semver-checks` — API compatibility check (lib crates only)
   - Cache: use `actions/cache` with `~/.cargo/registry` and `target/` keyed on
     `Cargo.lock` hash
   - No secrets required for the PR gate — all checks are local

2. Create `.github/workflows/post-merge.yml` — Post-Merge Verification:
   - Trigger: `on: push` to `main` branch only
   - Job 1: `kani-proofs`
     - Runner: `ubuntu-latest`
     - Install Kani via `cargo install --locked kani-verifier`
     - Run: `cargo kani --workspace --timeout 300 --mem-limit 8192`
     - `timeout-minutes: 120` (19 proofs × 300 s max = ~95 min worst-case)
     - Upload proof report as artifact (artifact name: `kani-report-${{ github.sha }}`)
   - Job 2: `fuzz-corpus`
     - Runner: `ubuntu-latest`
     - Install: `cargo install cargo-fuzz`
     - Run all 6 fuzz targets for 30 minutes each:
       - `prism-query/fuzz_prismql_parser`
       - `prism-query/fuzz_alias_expansion`
       - `prism-ocsf/fuzz_normalize`
       - `prism-spec-engine/fuzz_spec_parser`
       - `prism-operations/fuzz_template_interpolation`
       - `prism-security/fuzz_injection_scanner`
     - Use `--jobs 6 --time 1800` (30 min per target, parallelized)
     - Upload corpus artifacts per target
   - No blocking of main — failures notify via GitHub Actions summary only (these are
     discovery jobs, not hard gates)

3. Create `.github/workflows/release.yml` — Release Workflow:
   - Trigger: `on: push` with `tags: ['v*']`
   - Matrix: same 5 targets as CI (runs in parallel)
   - Build step per target:
     - `cargo build --release --locked --target ${{ matrix.target }}`
     - Strip binary (non-Windows): `strip target/${{ matrix.target }}/release/prism`
     - Archive: `.tar.gz` for Unix targets, `.zip` for Windows
   - Checksum step: `sha256sum` each archive → `checksums.txt`
   - Sign: `gh release create ${{ github.ref_name }}` with `--verify-tag` and upload
     all archives + `checksums.txt`
   - Secret: `GITHUB_TOKEN` (automatically provided, no additional secrets needed)
   - Homebrew formula update job (runs after all matrix jobs complete):
     - Checkout the tap repo (`1898co/homebrew-tap`) using `HOMEBREW_TAP_TOKEN` secret
     - Update `Formula/prism.rb`: replace `url` with the release archive URL and
       `sha256` with the macOS ARM64 tarball checksum
     - Open PR against the tap repo via `gh pr create`
   - Chocolatey publish stub:
     - Use the Windows matrix artifact
     - Run `choco pack` on `packaging/chocolatey/prism.nuspec`
     - Run `choco push` with `CHOCOLATEY_API_KEY` secret
   - crates.io publish:
     - Run `cargo publish -p prism-core`, then `prism-credentials`, etc. in dependency order
     - Use `CRATES_IO_TOKEN` secret
     - Gate: only publish if all matrix builds passed

4. Create `.github/dependabot.yml`:
   ```yaml
   version: 2
   updates:
     - package-ecosystem: cargo
       directory: "/"
       schedule:
         interval: weekly
         day: monday
       open-pull-requests-limit: 5
       labels:
         - dependencies
         - automated
     - package-ecosystem: github-actions
       directory: "/"
       schedule:
         interval: weekly
         day: monday
       open-pull-requests-limit: 5
   ```

5. Create `CODEOWNERS`:
   - `*` ownership assigned to repository maintainers (placeholder: `@1898co/prism-core`)
   - `.github/workflows/` requires additional review from `@1898co/platform`
   - `.github/dependabot.yml` requires `@1898co/platform` review

## Acceptance Criteria

**AC-1:** Given a PR that fails `cargo fmt --check`, When the CI workflow runs, Then the
PR gate fails on the format step and the PR cannot be merged.

**AC-2:** Given a PR that introduces a `clippy::await_holding_lock` violation, When the
CI workflow runs, Then `cargo clippy -D warnings` fails and the PR gate blocks merge.

**AC-3:** Given a PR, When the CI workflow runs, Then all 5 platform matrix jobs
(macOS ARM64, macOS x86_64, Linux glibc, Linux musl, Windows) run in parallel and all
must pass before the PR status check is green.

**AC-4:** Given a PR that introduces a dependency listed in the RustSec advisory
database, When `cargo audit` runs, Then the PR gate fails and merge is blocked.

**AC-5:** Given a merge to `main`, When the post-merge workflow runs, Then the
`kani-proofs` job executes all 19 Kani proofs with a 300 s timeout and 8192 MB memory
limit, and uploads the report as a GitHub Actions artifact.

**AC-6:** Given a git tag `v1.0.0` pushed to the repository, When the release workflow
runs, Then 5 platform binaries are built, archived, SHA-256 checksums are computed,
and a GitHub Release is created with all archives and `checksums.txt` attached.

**AC-7:** Given a successful release build, When the release workflow completes, Then
a PR is opened against the Homebrew tap repository with updated `url` and `sha256` in
`Formula/prism.rb`.

**AC-8:** Given a successful release build, When the release workflow completes, Then
`cargo publish` is invoked for all published crates in dependency-safe order using
`CRATES_IO_TOKEN`.

**AC-9:** No secrets (tokens, API keys) appear in GitHub Actions logs — all sensitive
values are referenced via `secrets.VARNAME` and GitHub Actions automatically masks them.

## Architecture Compliance Rules

- The PR gate step order MUST match `tooling-selection.md` § CI Pipeline Structure:
  fmt → clippy → test → deny → audit → semver-checks. No reordering permitted.
- `cargo clippy` MUST use `-D warnings` — soft warnings that don't block CI violate
  AD-008.
- Kani proofs MUST run in a separate post-merge job (not the PR gate) — proof runtime
  (up to 95 min) would make PRs unusable.
- The release workflow MUST use `--locked` for `cargo build` and `cargo publish` to
  ensure reproducible builds from `Cargo.lock`.
- Secrets MUST be stored in GitHub Actions Secrets — never hardcoded in workflow files.
- The release workflow MUST NOT run on PR branches — tag trigger only (`tags: ['v*']`).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Matrix job passes on 4/5 platforms, fails on one | PR gate fails; all 5 must be green |
| EC-002 | Kani proof timeout exceeded (> 300 s) | Proof counted as failing; post-merge job reports failure but does not block main |
| EC-003 | crates.io publish fails mid-sequence | Release job fails; already-published crates are NOT re-published (idempotency via `--no-verify` is NOT used; use `--allow-dirty` only if git is clean) |
| EC-004 | Homebrew tap PR already exists from a previous failed release | `gh pr create` exits non-zero; workflow reports warning but does not block release |
| EC-005 | Dependabot opens a PR that breaks `cargo deny check` | The PR gate fails; Dependabot cannot merge its own PRs automatically |
