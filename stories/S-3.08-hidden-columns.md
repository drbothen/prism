# S-3.08 — prism-query: Hidden Columns

## Narrative

As a Prism query engine, I want certain columns to be queryable in WHERE/GROUP BY
predicates but excluded from SELECT * results so that internal API metadata (raw
response JSON, request IDs, cost tracking fields) does not clutter analyst result
sets unless explicitly requested — matching the osquery hidden column pattern.

## Objective

Add a `hidden: bool` flag to column metadata in sensor and internal table descriptors.
Hidden columns are included in the DataFusion schema (queryable in predicates, grouping,
and explicit SELECT), but filtered out of SELECT * projections. Expose four default
hidden columns (`_raw_api_response`, `_query_cost_ms`, `_rate_limit_remaining`,
`_api_request_id`) and support custom hidden columns declared in sensor spec files.

## Tasks

1. Extend `ColumnDescriptor` (or equivalent column metadata type) with `hidden: bool`:
   - Default: `false` (columns are visible by default)
   - Location: wherever `SensorTableDescriptor` and `InternalTableDescriptor` store
     per-column metadata — likely `prism-spec-engine` or `prism-query/src/schema.rs`

2. Extend sensor spec `[[sensor.tables.columns]]` schema:
   ```toml
   [[sensor.tables.columns]]
   name = "_raw_api_response"
   col_type = "Utf8"
   hidden = true                  # new field; default false
   ```
   Validate in `SpecParser`: `hidden` is optional boolean, default `false`.

3. Register hidden columns in DataFusion schema normally:
   - They appear in the Arrow schema for the MemTable
   - DataFusion can reference them in WHERE, GROUP BY, ORDER BY, JOIN ON clauses
   - This is NOT about omitting them from the schema — it is about omitting them
     from SELECT * expansion only

4. Implement SELECT * filtering in the query materialization/result path:
   - After DataFusion `collect()`, before returning results to the MCP layer:
     inspect the schema for columns with `hidden = true`
   - If the query used `SELECT *` (or filter/pipe mode which implicitly selects all):
     strip hidden columns from the returned `RecordBatch` using Arrow
     `RecordBatch::project()` with a column index list that excludes hidden columns
   - If the query explicitly named a hidden column (e.g., `SELECT _raw_api_response
     FROM crowdstrike_alerts`): return it — explicit beats implicit

5. Detecting "explicit vs implicit" SELECT:
   - The parsed `QueryPlan` already tracks whether the projection is `SelectAll` or
     `SelectColumns(Vec<String>)`. Use this flag in the result layer.
   - Filter mode and pipe mode without a `fields` stage → `SelectAll` → strip hidden
   - SQL `SELECT *` → `SelectAll` → strip hidden
   - SQL `SELECT col1, col2` → `SelectColumns` → include hidden columns if named
   - Pipe `| fields + _raw_api_response` → the `fields` stage produces an explicit
     column list → include hidden if named

6. Define default hidden columns injected automatically into every sensor table
   (in `default_hidden_columns.rs`):
   - `_raw_api_response: Utf8` — full API response JSON for the record (before
     OCSF normalization); injected during materialization alongside `_sensor` etc.
   - `_query_cost_ms: Float64` — per-record fetch time in ms (proportional share
     of the sensor API call latency; populated from `QueryMetrics` in S-3.09)
   - `_rate_limit_remaining: Int64` — rate limit remaining from the API response
     header at fetch time (e.g., `X-RateLimit-Remaining`); `NULL` if not provided
   - `_api_request_id: Utf8` — trace/request ID from API response headers
     (e.g., `X-Request-Id`, `X-Trace-Id`); `NULL` if not provided

7. Expose hidden column information in `explain_query` output:
   - Add `hidden_columns: Vec<String>` to the explain response
   - Example: `"hidden_columns": ["_raw_api_response", "_query_cost_ms", ...]`
   - Helps analysts discover that hidden columns exist without needing docs

## Acceptance Criteria

**AC-1:** Given a sensor table with column `_raw_api_response` marked `hidden = true`,
When `SELECT * FROM crowdstrike_alerts` executes, Then `_raw_api_response` is NOT
present in the result schema or any returned row.

**AC-2:** Given the same table, When `SELECT _raw_api_response, severity FROM
crowdstrike_alerts WHERE severity_id >= 4` executes, Then `_raw_api_response` IS
present in results — explicit SELECT includes hidden columns.

**AC-3:** Given a filter mode query `severity >= high`, When results are returned,
Then none of the four default hidden columns (`_raw_api_response`, `_query_cost_ms`,
`_rate_limit_remaining`, `_api_request_id`) appear in the result set.

**AC-4:** Given a WHERE clause referencing a hidden column (`WHERE _api_request_id =
'abc-123'`), When the query executes, Then the WHERE predicate evaluates correctly
against the hidden column, even though that column is not in SELECT * output.

**AC-5:** Given a pipe mode query `FROM crowdstrike_alerts | where severity_id >= 3`,
When results are returned, Then hidden columns are absent. When followed by
`| fields + _raw_api_response`, Then `_raw_api_response` is present.

**AC-6:** Given `explain_query(query: "SELECT * FROM crowdstrike_alerts")`, When the
response is returned, Then it includes `"hidden_columns": ["_raw_api_response",
"_query_cost_ms", "_rate_limit_remaining", "_api_request_id"]` (or the table-specific
hidden set).

## Architecture Compliance Rules

- Hidden columns MUST be included in the DataFusion Arrow schema — omitting them from
  the schema would break WHERE/GROUP BY references. The filtering is applied AFTER
  DataFusion execution, in the result serialization layer.
- The four default hidden columns (`_raw_api_response`, `_query_cost_ms`,
  `_rate_limit_remaining`, `_api_request_id`) MUST be injected at the same point
  as `_sensor`, `_client`, `_source_table` virtual fields — in `materialization.rs`
  (or equivalent), not in each sensor adapter.
- `_raw_api_response` content MUST be treated as untrusted external data and passed
  through `InjectionScanner::scan()` if it appears in any MCP response field that
  the AI agent reads — consistent with the 4-layer injection defense.
- Do NOT use DataFusion column metadata or Arrow field metadata to store the hidden
  flag — DataFusion may not preserve arbitrary field metadata through projections.
  Maintain the hidden flag in Prism's own `ColumnDescriptor` and apply the filter
  in Prism's result layer.

## Notes

- The osquery hidden column pattern (columns queryable in WHERE but absent from
  SELECT *) is the direct model. osquery uses this for columns like `pid_with_namespace`
  that are expensive to compute and only needed in specific queries.
- `_raw_api_response` is the highest-value hidden column for power users: it lets
  analysts inspect the raw sensor API JSON without extracting every field into OCSF.
  Keeping it hidden prevents it from bloating every result set.
- `_query_cost_ms` depends on `QueryMetrics` from S-3.09. If S-3.09 is not yet
  implemented, this column should be `NULL` rather than blocking this story.
- Hidden columns in sensor spec files are opt-in per column. There is no mechanism
  to hide a column that another column definition made visible — the spec is the
  source of truth.
