# S-5.01 — prism-mcp: Server Bootstrap and Tool Registration

## Narrative

As a Claude Code user integrating Prism into an MSSP analyst workflow, I want the MCP
server to start cleanly, register all tools, and respond to MCP protocol messages over
stdio, so that analysts can immediately discover and invoke Prism capabilities without
any additional configuration.

## Objective

Implement the MCP server using rmcp 1.4 with full tool registration, stdio transport,
and graceful shutdown. This story establishes the MCP protocol surface that all
downstream stories (tool implementations) attach to.

## Tasks

1. Create `crates/prism-mcp/Cargo.toml` with dependencies: `rmcp = "1.4"`, `tokio`,
   `tracing`, `prism-core`, `prism-query`, `prism-operations`, `prism-spec-engine`.

2. Implement `PrismServer` struct in `crates/prism-mcp/src/server.rs`:
   - Implement `rmcp::ServerHandler` trait
   - Hold `Arc<PrismContext>` (shared state: config, RocksDB handle, scheduler, etc.)
   - Respond to `initialize` with server capabilities, protocol version, and tool list
   (BC-2.10.001)

3. Register all tools via `#[tool_router]` macro in `crates/prism-mcp/src/tools/mod.rs`:
   `query`, `explain_query`, `create_schedule`, `list_schedules`, `delete_schedule`,
   `get_diff_results`, `create_rule`, `list_rules`, `delete_rule`, `create_case`,
   `update_case`, `list_cases`, `get_case`, `case_metrics`, `create_alias`,
   `list_aliases`, `delete_alias`, `explain_alias`, `set_credential`,
   `delete_credential`, `list_credentials`, `credential_status`, `reload_config`,
   `add_sensor_spec`, `list_sensor_specs`, `create_pack`, `list_packs`, `delete_pack`,
   `watchdog_status`, `list_capabilities`, `get_help`, `get_diagnostics`,
   `acknowledge_alert`, `health_check` (BC-2.10.002)

4. Implement conditional tool registration: at startup, check compile-time feature flags
   (e.g., `crowdstrike-write`, `cyberint-write`) and runtime `FeatureFlag` state; omit
   write tools from the registered set when their flag is disabled (BC-2.10.003)

5. Implement stdio transport in `crates/prism-mcp/src/main.rs`:
   - Wire `rmcp::transport::stdio()` to `PrismServer`
   - Ensure all `tracing` output goes to stderr (configure subscriber with
     `with_writer(std::io::stderr)`)
   - stdout is reserved exclusively for MCP JSON-RPC framing (BC-2.10.006)

6. Implement graceful shutdown in `crates/prism-mcp/src/main.rs`:
   - Install `tokio::signal::ctrl_c()` and `tokio::signal::unix::signal(SIGTERM)` handlers
   - On signal: set shutdown flag, flush audit buffer, close RocksDB cleanly
   - Hard timeout: if shutdown takes > 5s, force exit with code 1 (BC-2.10.010)

7. Write integration tests in `crates/prism-mcp/tests/bootstrap.rs`:
   - Spawn `PrismServer` in-process with a mock `PrismContext`
   - Assert `tools/list` returns 35+ entries
   - Assert feature-flag-gated tools are absent when flag is off

## Acceptance Criteria

**AC-1:** Given `PrismServer` startup with a valid `PrismContext`, When an MCP
`initialize` message is received, Then the server responds with a capabilities object
that includes the protocol version and a non-empty tool list.
(traces to BC-2.10.001 postcondition 1)

**AC-2:** Given all compile-time and runtime feature flags enabled, When `tools/list`
is requested, Then the response contains at least 35 tool entries including `query`,
`health_check`, and `list_capabilities`.
(traces to BC-2.10.002 postcondition 1)

**AC-3:** Given the binary compiled without the `crowdstrike-write` feature flag, When
`tools/list` is requested, Then CrowdStrike write tools (e.g., `acknowledge_alert` write
variants) are absent from the list.
(traces to BC-2.10.003 postcondition 1)

**AC-4:** Given stdio transport is active, When a `query` tool call is processed, Then
the JSON-RPC response is written to stdout and all log lines are written to stderr, with
no log content appearing on stdout.
(traces to BC-2.10.006 invariant 1)

**AC-5:** Given a running `PrismServer`, When `SIGTERM` is delivered, Then the server
completes in-flight requests, flushes the audit buffer, closes RocksDB, and exits within
5 seconds.
(traces to BC-2.10.010 postcondition 1)

## Architecture Compliance Rules

- `prism-mcp` MUST NOT directly import `rocksdb` — all storage access goes via
  `prism-storage` through `PrismContext`
- `PrismServer` MUST be `Send + Sync` (rmcp requires it for async handler dispatch)
- stdout MUST be reserved exclusively for MCP JSON-RPC; any diagnostic write to stdout
  is a correctness bug (Claude Code reads stdout as protocol data)
- Feature flag checks at tool registration MUST use the same `FeatureFlag` resolver
  defined in S-1.08 — do not duplicate flag logic
- Tool handler functions MUST be individually testable in isolation from the server
  (accept `Arc<PrismContext>` directly, not `&self` on a server struct)

## Notes

- rmcp 1.4's `#[tool_router]` macro generates the dispatch table at compile time. All
  tools must be registered in a single `impl` block on `PrismServer` decorated with
  `#[tool_router]`.
- The tool stub bodies in this story should return `Err(PrismError::NotImplemented)`
  where the full logic lives in subsequent wave-5 stories. This keeps the bootstrap story
  focused on registration, not implementation.
- RocksDB's LOCK file prevents concurrent open. If a second `prism serve` is started
  while one is running, it will fail at RocksDB open (before MCP transport starts). This
  is intentional per DI-017.
