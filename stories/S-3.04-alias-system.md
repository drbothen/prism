# S-3.04 — prism-query: Alias System (P1)

## Narrative

As an analyst or LLM agent, I want to create, resolve, compose, and manage named query
fragments (aliases) so that frequently used predicates like "high severity detections"
can be saved once and referenced with `@high_sev` in any query, reducing repetition and
enabling consistent query patterns across an MSSP's client base.

## Objective

Implement the query alias system: `create_alias`, `list_aliases`, `delete_alias`, and
`explain_alias` MCP tools backed by `aliases.toml` file-based persistence, with scope
support (global and per-client), optional parameterization, pre-parse `@alias` expansion,
recursive composition up to depth 3, cycle detection, keyword/OCSF collision validation,
capability gating, and Kani/proptest/fuzz verification properties.

## Tasks

### 1. Define core types in `crates/prism-query/src/alias_types.rs`

Define the shared data types used across all alias modules:

```rust
/// Alias scope — mirrors the BC-2.11.008 `scope` parameter.
pub enum AliasScope {
    Global,
    Client(ClientId),
}

impl AliasScope {
    /// Parse "global" | "client:<id>" strings from MCP tool input.
    pub fn parse(s: &str) -> Result<Self, PrismError>;

    /// Derive the ConfirmationToken client_id sentinel per BC-2.11.008:
    ///   Global  -> "__global__"
    ///   Client(id) -> id.as_str()
    pub fn token_client_id(&self) -> &str;
}

/// A single stored alias entry.
pub struct AliasEntry {
    pub name:        String,
    pub scope:       AliasScope,
    pub query:       String,                               // raw PrismQL template
    pub parameters:  Option<HashMap<String, ParamDefault>>, // param name -> default value
    pub description: Option<String>,
}

/// A single parameter default value (must be a PrismQL atomic literal).
pub struct ParamDefault {
    pub value: String,   // stored as string; re-validated at substitution time
}
```

### 2. Implement `AliasStore` in `crates/prism-query/src/alias_store.rs`

Persistence target: `aliases.toml` (NOT RocksDB). Atomic write pattern (temp file +
fsync + rename) mirrors the credential state files pattern established in S-1.08/S-1.09.

**File-first ordering** per BC-2.11.008:
1. Validate proposed alias against in-memory state (cycle/depth checks, keyword/OCSF
   collision, parse validation).
2. Write `aliases.toml` atomically. If the write fails, return `E-IO-001`; in-memory
   registry is left unchanged.
3. Only after a successful file write, update the in-memory alias registry.

Key methods:

- `AliasStore::load(path: &Path) -> Result<Self, PrismError>`: deserialize `aliases.toml`
  at startup; validate all entries (no cycles, all referenced aliases exist, depth <= 3).

- `AliasStore::create_or_update(entry: AliasEntry, token: Option<ConfirmationToken>) -> Result<CreateResult, PrismError>`:
  - Validate `alias.write` capability (see Task 4).
  - Validate alias name: `[a-zA-Z_][a-zA-Z0-9_]*`, 1–64 chars.
  - Validate no keyword/OCSF collision (see Task 5); return `E-ALIAS-006` on conflict.
  - Validate `parameters` defaults: each default value must be a PrismQL atomic literal
    (StringLiteral, IntegerLiteral, FloatLiteral, BooleanLiteral, DurationLiteral, or
    Identifier); return `E-ALIAS-004` on violation.
  - Run cycle detection and depth check via `AliasResolver`; return `E-ALIAS-002` or
    `E-ALIAS-003` on violation.
  - If alias already exists at the specified scope (same name + scope): return a
    `ConfirmationToken` (client_id = `scope.token_client_id()`). Do NOT persist until
    `confirm_action` is called (write-operation gating per S-1.09/CAP-006 pattern).
  - If alias does not exist: apply file-first write sequence, then update in-memory registry.

- `AliasStore::get(name: &str, scope: &AliasScope) -> Result<Option<&AliasEntry>, PrismError>`:
  retrieve by (name, scope).

- `AliasStore::list(scope_filter: Option<&AliasScope>) -> Vec<&AliasEntry>`:
  - `None` -> return all aliases (global + all per-client), sorted alphabetically by name
    within each scope group (BC-2.11.013).
  - `Some(Global)` -> return only global aliases.
  - `Some(Client(id))` -> return only aliases for that client; does NOT include globals.

- `AliasStore::delete(name: &str, scope: &AliasScope, force: bool, token: ConfirmationToken) -> Result<DeleteResult, PrismError>`:
  - Deletion ALWAYS requires a `ConfirmationToken` (BC-2.11.014 — every deletion is a
    write-operation-gated action, not just forced ones).
  - Before deletion: check dependents via `AliasStore::dependents(name, scope)`.
  - If dependents exist and `force` is false: return `E-ALIAS-005` listing dependents.
  - If dependents exist and `force` is true: cascade-delete all dependents; re-resolve
    dependent list at confirmation time (per BC-2.11.014 cascade semantics).
  - Apply file-first write sequence; return `E-IO-001` on write failure without touching
    in-memory registry.
  - Emit audit entry (DI-004).

- `AliasStore::dependents(name: &str, scope: &AliasScope) -> Vec<String>`:
  scan all alias definitions (both global and per-client) for references to `@name`;
  return list of dependent alias names. O(n) over all aliases; acceptable at expected
  cardinality (< 1,000 per deployment).

### 3. Implement `AliasResolver` in `crates/prism-query/src/alias_resolver.rs`

BC-2.11.009 defines a five-step expansion sequence:

```
1. Detection      -- find @identifier tokens (regex `@([a-zA-Z_][a-zA-Z0-9_]*)`)
2. Scope resolution -- per-client alias overrides global alias of same name
3. Parameter substitution -- replace `{{param}}` placeholders with caller args or defaults
4. Recursive expansion -- expand nested aliases (depth counter; max 3)
5. Security check -- reject expanded query exceeding 64KB
```

Key methods:

- `AliasResolver::expand(query: &str, store: &AliasStore, scope: &AliasScope, args: &HashMap<String, String>, depth: u32) -> Result<String, PrismError>`:
  - **Step 1 Detection**: scan for `@identifier` tokens using regex
    `@([a-zA-Z_][a-zA-Z0-9_]{0,63})`.
  - **Step 2 Scope resolution**: for each `@alias_name`:
    - First look up `(alias_name, scope)` (per-client if scope is `Client(id)`).
    - Fall back to `(alias_name, Global)` if per-client is absent.
    - If neither exists: return `E-ALIAS-001` with available aliases for the current scope.
  - **Step 3 Parameter substitution**: substitute caller-supplied `args` into the alias
    template using `{{param_name}}` placeholders. Missing params use defaults from
    `AliasEntry::parameters`. Unknown param name in `args`: return `E-ALIAS-004`.
    Each substituted value MUST be validated as a PrismQL atomic literal token (same
    rules as `ParamDefault` above); compound expressions (containing `|`, `(`, `)`, `=`,
    `!=`, `>`, `<`, `>=`, `<=`, `AND`, `OR`, `NOT`) are rejected with `E-ALIAS-004`
    to prevent query injection.
  - **Step 4 Recursive expansion**: call `expand()` on the substituted definition with
    `depth + 1`. If `depth >= 3`: return `E-ALIAS-003`.
  - **Step 5 Security check**: if expanded query length exceeds 64KB, return `E-QUERY-003`.
  - Resolution order is inner-to-outer: innermost aliases are expanded first.

- `AliasResolver::detect_cycle(name: &str, definition: &str, store: &AliasStore) -> Result<(), PrismError>`:
  - DFS over the alias reference graph starting from `name`.
  - If `name` is encountered again during traversal: return `E-ALIAS-002` with the exact
    cycle chain.
  - MUST run at creation time — not only at expansion time — so the store never contains
    a cycle (BC-2.11.009 invariant DI-020).

### 4. Implement capability gate in `crates/prism-query/src/alias_capability.rs`

Per BC-2.11.008 preconditions:

- `alias.write` is a cargo feature + runtime TOML capability flag.
- For `client:<client_id>` scope: check `alias.write` against the target client's
  capability set. Depends on the feature flag system from S-1.08.
- For `global` scope: check that `alias.write = Allow` for at least one configured
  client (visibility check). The operation is authorized if any single client's
  capability set includes `alias.write = Allow`.
- Return a structured capability error (using the pattern from S-1.08) if the check fails.

### 5. Implement keyword/OCSF collision validation

Per BC-2.11.008 invariants and `E-ALIAS-006`:

- **PrismQL keyword list**: `SELECT`, `FROM`, `WHERE`, `AND`, `OR`, `NOT`, `LIMIT`,
  `ORDER`, `BY`, `AS`, `IN`, `LIKE`, `IS`, `NULL`, `TRUE`, `FALSE`, `BETWEEN`,
  `GROUP`, `HAVING`, `JOIN`, `ON`, `DISTINCT`, `UNION`, `ALL`, `WITH`, `CASE`, `WHEN`,
  `THEN`, `ELSE`, `END` (case-insensitive comparison).
- **OCSF field names**: derived from the OCSF protobuf descriptor loaded at startup
  (e.g., `severity`, `activity_name`, `src_endpoint`, `dst_endpoint`, `device`, `actor`,
  `metadata`, `observables`, `time`, `category_uid`, `class_uid`). The reserved name list
  is dynamic — loaded once at startup and stored as a `HashSet<String>` (normalized to
  lowercase).
- If the alias name matches any keyword or OCSF field name (case-insensitive): return
  `E-ALIAS-006` with a structured error indicating whether the conflict is a keyword or
  an OCSF field name.

### 6. Implement MCP tool handlers in `crates/prism-query/src/alias_tools.rs`

**`create_alias` tool** (BC-2.11.008):
- Input: `name`, `scope` (required), `query` (required), `parameters` (optional),
  `description` (optional).
- Parse `scope` into `AliasScope`.
- Check `alias.write` capability (Task 4).
- Validate name, detect keyword/OCSF collision (E-ALIAS-006), detect cycle (E-ALIAS-002),
  check depth (E-ALIAS-003).
- Validate parser accepts the template (with parameter placeholders treated as valid tokens).
- Call `AliasStore::create_or_update(entry, None)`.
- If alias already existed: return a `ConfirmationToken` response (client_id per
  `scope.token_client_id()`). Agent must call `confirm_action` to complete.
- If alias was new: return structured success with the created alias entry and its
  expanded form.

**`list_aliases` tool** (BC-2.11.013):
- Input: `scope` (optional; `"global"`, `"client:<id>"`, or null).
- Parse scope filter. Call `AliasStore::list(scope_filter)`.
- Return JSON array of `{ name, scope, query, parameters, description }` objects.
- Sort alphabetically by name within each scope group.
- Emit audit entry (DI-004).

**`delete_alias` tool** (BC-2.11.014):
- Input: `name` (required), `scope` (required), `force` (optional, default false).
- Check `alias.write` capability.
- Check alias exists at scope; return `E-ALIAS-001` if not.
- Check `scope` references a valid client (if `client:<id>`); return `E-CFG-001` if not.
- Deletion ALWAYS requires a confirmation token (not only when force is true). Issue a
  `ConfirmationToken` with `action_summary` describing the alias to be deleted and
  `dependent_aliases` listing any dependents as a warning.
- Upon `confirm_action`: re-resolve dependents at confirmation time, call
  `AliasStore::delete(name, scope, force, token)`.
- If dependents exist and `force` is false: `E-ALIAS-005`.
- Return success with name(s) of deleted alias(es); include cascade-deleted dependents
  in the response when `force` was true.
- Emit audit entry (DI-004).

**`explain_alias` tool** (BC-2.11.015):
- Input: `name` (required), `scope` (optional).
- Resolve scope using default precedence if null: per-client overrides global when a
  client context is available.
- Return: `{ name, scope, query, expanded, parameters, description, composition_chain,
  composition_depth }`.
  - `composition_chain`: ordered list of alias names expanded during resolution
    (e.g., `["my_alias", "inner_alias"]`).
  - `composition_depth`: integer depth of the chain.
- Run parse validation on the expanded query; return parse errors as structured errors.
- Emit audit entry (DI-004).

### 10. Write VP-037 fuzz target in `crates/prism-query/fuzz/vp037_alias_fuzz.rs`


## Acceptance Criteria

**AC-1 (traces to BC-2.11.009 postcondition step 1, step 4):** Given
`create_alias("high_sev", scope="global", query="severity_id >= 3")`, when `@high_sev`
is used in a query string, then `AliasResolver::expand()` replaces `@high_sev` with
`severity_id >= 3` before the query reaches the Chumsky parser.

**AC-2 (traces to BC-2.11.009 postcondition step 4, invariant DI-020):** Given alias A
= `"@B AND active = true"`, alias B = `"@C OR category = 'malware'"`, alias C =
`"severity_id >= 3"` (depth 3), when A is expanded, then the fully expanded string is
`"severity_id >= 3 OR category = 'malware' AND active = true"`.

**AC-3 (traces to BC-2.11.009 invariant DI-020, BC-2.11.008 postcondition — cycle):**
Given alias A = `"@B AND foo"`, alias B = `"@A OR bar"` (cycle A → B → A), when
`create_alias("A", ...)` is called, then `E-ALIAS-002` is returned and alias A is NOT
persisted (cycle detection runs at creation time before any file write).

**AC-4 (traces to BC-2.11.009 invariant DI-020, error E-ALIAS-003):** Given alias A =
`"@B"`, B = `"@C"`, C = `"@D"`, D = `"literal"` (depth 4), when a query containing
`@A` is expanded, then `E-ALIAS-003` is returned — never `Ok(_)`.

**AC-5 (traces to BC-2.11.008 preconditions, invariant — keyword/OCSF):** Given a
`create_alias` call with `name = "severity"` (an OCSF field name) or `name = "WHERE"`
(a PrismQL keyword), then `E-ALIAS-006` is returned with a structured error indicating
whether the conflict is a PrismQL keyword or an OCSF field name, and no alias is created.

**AC-6 (traces to BC-2.11.009 postcondition step 2):** Given a global alias `high_sev`
and a per-client alias `high_sev` scoped to `client:acme` with a different definition,
when `AliasResolver::expand()` runs for a query in the `client:acme` scope, then the
per-client definition is used; when the same expansion runs with `scope=Global`, then
the global definition is used.

**AC-7 (traces to BC-2.11.009 postcondition step 3, error E-ALIAS-004):** Given a
parameterized alias `recent_alerts` with template `"severity_id > {{min_sev}} AND
active = true"` and default `min_sev = "3"`, when `recent_alerts(min_sev="critical OR
1=1")` is called, then `E-ALIAS-004` is returned (compound expression rejected as
parameter value). When called with `min_sev="5"` (valid IntegerLiteral), expansion
succeeds and produces `"severity_id > 5 AND active = true"`.

**AC-8 (traces to BC-2.11.013 postconditions):** Given three aliases exist (one global,
two per-client for `client:acme`), when `list_aliases(scope=null)` is called, then all
three are returned sorted alphabetically by name within each scope group, each entry
including `name`, `scope`, `query`, `parameters`, and `description`.

**AC-9 (traces to BC-2.11.014 postconditions, invariant — confirmation required):** Given
alias `high_sev` exists, when `delete_alias("high_sev", scope="global")` is called
WITHOUT a confirmation token, then a `ConfirmationToken` is returned (not a deletion
success). The alias is deleted only after `confirm_action` is called with the matching
token.

**AC-10 (traces to BC-2.11.014 error E-ALIAS-005):** Given alias `high_sev` exists and
alias `composite` references `@high_sev`, when `delete_alias("high_sev", scope="global",
force=false)` is confirmed, then `E-ALIAS-005` is returned listing `composite` as a
dependent and deletion does NOT proceed.

**AC-11 (traces to BC-2.11.015 postconditions):** Given alias `composite` = `"@high_sev
AND active = true"` and `high_sev` = `"severity_id >= 3"`, when `explain_alias("composite")`
is called, then response contains `expanded = "severity_id >= 3 AND active = true"`,
`composition_chain = ["composite", "high_sev"]`, and `composition_depth = 2`.

**AC-12 (traces to BC-2.11.008 preconditions — alias.write capability):** Given
`alias.write` is NOT enabled in the runtime TOML configuration, when `create_alias` or
`delete_alias` is called, then a structured capability error is returned and no alias is
created or deleted.

**AC-13 (traces to BC-2.11.008 postcondition — update confirmation):** Given alias
`high_sev` already exists at scope `global`, when `create_alias("high_sev",
scope="global", ...)` is called, then a `ConfirmationToken` is returned with
`client_id = "__global__"`. The alias is updated only after `confirm_action` is called
with the matching token.

**AC-14:** VP-012, VP-013, VP-025, and VP-037 all pass.

## Architecture Compliance Rules

- Alias expansion is a PRE-PARSE step — `expand()` runs on the raw query string BEFORE
  it is passed to the S-3.01 Chumsky parser; never pass un-expanded queries with `@`
  tokens to the parser.
- Persistence is `aliases.toml` (file-based). Do NOT use RocksDB or any RocksDB column
  family for alias storage. The atomic write pattern (temp file + fsync + rename) is the
  same pattern used by credential state files from S-1.08/S-1.09.
- File-first ordering is mandatory: validate → write file → update in-memory. In-memory
  state is NEVER updated before the file write succeeds.
- Cycle detection MUST run at alias creation time, not only at expansion time — the store
  must remain cycle-free as an invariant (DI-020).
- Scope is mandatory on all write operations (`create_alias`, `delete_alias`) and is the
  primary namespace dimension — alias names alone are not globally unique.
- Scope resolution in `expand()` follows per-client-overrides-global precedence.
- The `@alias_name` sigil is intentional — it avoids collision with SQL identifiers. Do
  NOT change the sigil without a BC update.
- Alias definitions are stored as raw query fragment strings — NOT pre-parsed AST nodes.
  Text substitution at expansion time is intentional (simpler, auditable).
- Do NOT use "per analyst session" scoping — aliases are scoped by `AliasScope`
  (Global or Client(ClientId)) as defined in BC-2.11.008, not by session.
- All write operations (`create_alias`, `delete_alias`) require `alias.write` capability
  (S-1.08 feature flag system). Read operations (`list_aliases`, `explain_alias`) do not.
- All deletions require a confirmation token (S-1.09 pattern) — not just forced cascade
  deletions.

## Error Code Reference

All alias errors use the `E-ALIAS-*` taxonomy. Do NOT use `E-QUERY-*` for alias-specific
errors.

| Code | Condition |
|------|-----------|
| `E-ALIAS-001` | Alias does not exist at the specified scope |
| `E-ALIAS-002` | Alias creation would create a cycle |
| `E-ALIAS-003` | Alias composition depth exceeds 3 |
| `E-ALIAS-004` | Parameter value fails type validation (not a PrismQL atomic literal) / unknown parameter name |
| `E-ALIAS-005` | Alias has dependent aliases and `force` is not `true` |
| `E-ALIAS-006` | Alias name conflicts with a PrismQL keyword or OCSF field name |
| `E-IO-001` | `aliases.toml` atomic write failed |
| `E-CFG-001` | `scope` references a client ID that does not exist in configuration |

## Previous Story Intelligence

This story depends on three predecessors:

**S-3.02 (QueryEngine):** `AliasResolver::expand()` is called inside
`QueryEngine::execute()` before the Chumsky parser. The integration point is in
`engine.rs` from S-3.02. The alias integration point in `engine.rs` should be a no-op
when the alias store is not initialized, falling through to raw query string parsing.

**S-1.08 (Feature Flags):** The `alias.write` capability check is implemented using the
feature flag system from S-1.08. Do not re-implement capability gating — call the
existing gate mechanism.

**S-1.09 (Confirmation Tokens):** Both `create_alias` (when updating an existing alias)
and ALL `delete_alias` calls issue and validate confirmation tokens using the pattern
from S-1.09. Do not re-implement token issuance — use the existing `ConfirmationToken`
infrastructure.

The regex-based `@identifier` scanning is intentionally simple — it does not require
the Chumsky parser from S-3.01. This keeps the alias expansion step independent of parse
mode (filter, SQL, or pipe all work the same way).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-11-021 | Creating a per-client alias with the same name as a global alias | Valid; per-client alias overrides global for that client during `expand()` |
| EC-11-022 | Deleting an alias that is referenced by another alias | Deletion BLOCKED with `E-ALIAS-005` listing dependents; use `force: true` for cascade or delete dependents first |
| EC-11-023 | Alias name is a substring of a field name (e.g., alias `ip` and field `device.ip`) | Only standalone `@identifier` tokens matched; dotted field names are NOT alias candidates |

| EC-11-033 | No aliases defined anywhere | `list_aliases` returns empty array; not an error |

## Notes

- The depth limit of 3 is intentional and conservative. LLM agents typically compose
  aliases 1–2 levels deep. Depth 3 allows `@A → @B → @C → literal` which covers all
  realistic use cases. Raising the limit requires a BC amendment.
- `AliasStore::dependents()` is O(n) over all aliases — acceptable at expected cardinality
  (< 1,000 aliases per deployment). The scan uses the same regex as `AliasResolver`.
- The alias system is P1 priority — it MUST be skippable if wave scheduling requires
  accelerating P0 stories. The integration point in `engine.rs` (S-3.02) should be a
  no-op when the alias store is not initialized.
- VP-025 now targets TOML key determinism (not RocksDB key determinism) because
  `aliases.toml` is the persistence layer. The proof property is identical in spirit:
  the TOML table key for `(alias_name, scope)` must be deterministic across calls.
- VP-037 is a broad fuzz target. Provide all four edge case categories: cycles (A → A),
  deep nesting (A → B → C → D → E), self-reference (`@A` inside A's definition), and
  non-UTF-8 bytes in alias names. Each MUST produce `Err(_)`, not a panic.
- Parameter injection defense (AC-7) is security-critical. The restriction to PrismQL
  atomic literals is enforced both at alias creation time (for defaults) and at
  substitution time (for caller-supplied args). This prevents query injection through
  parameterized alias values.
