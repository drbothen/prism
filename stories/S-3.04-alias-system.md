# S-3.04 — prism-query: Alias System (P1)

## Narrative

As an analyst or LLM agent, I want to create, resolve, compose, and manage named query
fragments (aliases) so that frequently used predicates like "high severity detections"
can be saved once and referenced with `@high_sev` in any query, reducing repetition and
enabling consistent query patterns across an MSSP's client base.

## Objective

Implement the query alias system: `create_alias`, `list_aliases`, `delete_alias`, and
`explain_alias` MCP tools backed by RocksDB persistence, with pre-parse `@alias` expansion,
recursive composition up to depth 3, cycle detection, and Kani/proptest/fuzz verification
properties.

## Tasks

1. Implement alias persistence in `crates/prism-query/src/alias_store.rs`:
   - Use file-based `aliases.toml` persistence per BC-2.11.008: write atomically
     (temp file + fsync + rename). Load at startup, hot-reload on file change.
   - Key: alias name; Value: alias definition (query fragment)
   - `AliasStore::create_or_update(name: &str, definition: &str) -> Result<(), PrismError>`:
     validate alias name (alphanumeric + underscore, 1–64 chars); if alias already exists
     at same scope, treat as update with confirmation token (per BC-2.11.008 — NOT a
     rejection); persist to aliases.toml (BC-2.11.008)
   - `AliasStore::get(name: &str) -> Result<Option<String>, PrismError>`: retrieve by name
   - `AliasStore::list() -> Result<Vec<AliasEntry>, PrismError>`: iterate CF, return all
     (`AliasEntry { name: String, definition: String }`) (BC-2.11.013)
   - `AliasStore::delete(name: &str) -> Result<(), PrismError>`: remove from CF; check for
     dependents before deletion (BC-2.11.014)
   - `AliasStore::dependents(name: &str) -> Result<Vec<String>, PrismError>`: scan all alias
     definitions for `@name` references — used by delete to warn of cascade impact

2. Implement `AliasResolver` in `crates/prism-query/src/alias_resolver.rs` (BC-2.11.009):
   - `AliasResolver::expand(query: &str, store: &AliasStore, depth: u32)
     -> Result<String, PrismError>`
   - Find all `@identifier` tokens in `query` using regex `@([a-zA-Z_][a-zA-Z0-9_]{0,63})`
   - For each `@alias_name` found:
     - Fetch definition from `AliasStore`
     - If not found: return E-ALIAS-001 (alias not found)
     - If `depth >= 3`: return E-ALIAS-003 (max expansion depth exceeded)
     - Recursively call `expand()` on the definition with `depth + 1`
     - Replace `@alias_name` in query with expanded definition (text substitution)
   - Return fully expanded query string

3. Implement cycle detection in `alias_resolver.rs` (BC-2.11.009):
   - On `create_alias(name, definition)`: call `AliasResolver::detect_cycle(name, definition,
     store)` before persisting
   - `detect_cycle`: perform DFS over the alias reference graph starting from `name`;
     if `name` is encountered again during traversal, return E-ALIAS-002 (cycle detected)
   - Cycle detection MUST run at creation time — not only at expansion time — so the
     store never contains a cycle

4. Implement `create_alias` tool logic in `crates/prism-query/src/alias_tools.rs`
   (BC-2.11.008):
   - Validate name and definition
   - Run cycle detection
   - Persist to RocksDB via `AliasStore::create()`
   - Return structured success response with alias name

5. Implement `list_aliases` tool logic in `alias_tools.rs` (BC-2.11.013):
   - Call `AliasStore::list()`
   - Return JSON array of `{ "name": "...", "definition": "..." }` objects
   - Sort by name for deterministic output

6. Implement `delete_alias` tool logic in `alias_tools.rs` (BC-2.11.014):
   - Call `AliasStore::dependents(name)` — if non-empty, return warning listing dependent
     aliases; require `force: true` param to proceed with deletion
   - Call `AliasStore::delete(name)`
   - Return success with name of deleted alias

7. Implement `explain_alias` tool logic in `alias_tools.rs` (BC-2.11.015):
   - Accept alias name as input
   - Expand the alias fully using `AliasResolver::expand()`
   - Return: `{ "name": "...", "definition": "...", "expanded": "...", "expansion_chain": [...] }`
   - `expansion_chain`: list of intermediate expansions showing each substitution step

8. Write VP-012 Kani proof in `crates/prism-query/proofs/vp012_depth_limit.rs`:
   - Property: for any alias expansion chain where depth would exceed 3,
     `AliasResolver::expand()` returns `Err(E-ALIAS-003)` — never `Ok(_)`

9. Write VP-013 proptest in `crates/prism-query/proofs/vp013_cycle_detection.rs`:
   - Property: `detect_cycle()` correctly identifies all circular references in
     arbitrarily generated alias graphs (up to 10 nodes)
   - Use proptest's graph generation strategy

10. Write VP-025 Kani proof in `crates/prism-query/proofs/vp025_cache_key.rs`:
    - Property: for any alias name, the RocksDB key derivation is deterministic —
      `key(name) == key(name)` for all valid name strings

11. Write VP-037 fuzz target in `crates/prism-query/fuzz/vp037_alias_fuzz.rs`:
    - `fuzz_target!(|data: &[u8]| { /* build arbitrary alias graph from data, call expand() */ })`
    - Must never panic on cycles, deep nesting, self-reference, or non-UTF-8 content

## Acceptance Criteria

**AC-1:** Given `create_alias("high_sev", "severity_id >= 3")`, When `@high_sev` is used
in a query string, Then `AliasResolver::expand()` replaces `@high_sev` with `severity_id >= 3`
before the query reaches the parser. (BC-2.11.008, BC-2.11.009)

**AC-2:** Given alias A = `"@B AND active = true"`, alias B = `"@C OR category = 'malware'"`,
alias C = `"severity_id >= 3"` (depth 3), When A is expanded, Then the fully expanded string
is `"severity_id >= 3 OR category = 'malware' AND active = true"`. (BC-2.11.009)

**AC-3:** Given alias A = `"@B AND foo"`, alias B = `"@A OR bar"` (cycle A → B → A),
When `create_alias("A", "@B AND foo")` is called, Then E-ALIAS-002 is returned and alias
A is NOT persisted. (BC-2.11.009)

**AC-4:** Given alias A = `"@B"`, B = `"@C"`, C = `"@D"`, D = `"literal"` (depth 4),
When a query containing `@A` is expanded, Then E-ALIAS-003 is returned. (BC-2.11.009)

**AC-5:** Given three aliases exist, When `list_aliases` is called, Then all three are
returned sorted alphabetically with their definitions. (BC-2.11.013)

**AC-6:** Given alias `"high_sev"` exists and alias `"composite"` references `@high_sev`,
When `delete_alias("high_sev", force: false)` is called, Then a warning listing
`"composite"` as a dependent is returned and deletion does not proceed. (BC-2.11.014)

**AC-7:** Given alias `"high_sev"` = `"severity_id >= 3"` and `"composite"` = `"@high_sev
AND active = true"`, When `explain_alias("composite")` is called, Then response contains
`expanded = "severity_id >= 3 AND active = true"` and the expansion chain showing both
substitution steps. (BC-2.11.015)

**AC-8:** VP-012, VP-013, VP-025, and VP-037 all pass.

## Architecture Compliance Rules

- Alias expansion is a PRE-PARSE step — `expand()` runs on the raw query string BEFORE
  it is passed to the S-3.01 parser; never pass un-expanded queries with `@` tokens to
  the parser
- Cycle detection MUST run at alias creation time, not only at expansion time — the store
  must remain cycle-free as an invariant
- The `@alias_name` syntax uses `@` as the sigil specifically to avoid collision with
  SQL identifiers — do NOT change the sigil without a BC update
- Alias definitions are stored as raw query fragment strings — they are NOT pre-parsed
  into AST nodes; text substitution at expansion time is intentional (simpler, auditable)
- Cross-tenant alias isolation: aliases are namespaced per analyst session, not per
  client — a single MSSP operator's aliases apply across all their clients

## Previous Story Intelligence

This story depends on S-3.02 being complete because:
- `AliasResolver::expand()` is called inside `QueryEngine::execute()` before the parser;
  the integration point is in `engine.rs` from S-3.02
- The RocksDB column family for aliases was initialized in S-2.01; S-3.02's engine
  has access to the storage layer

The regex-based `@identifier` scanning is intentionally simple — it does not require
the parser from S-3.01. This keeps the alias expansion step independent of parse mode
(filter, SQL, or pipe all work the same way).

## Notes

- The depth limit of 3 is intentional and conservative. LLM agents typically compose
  aliases 1–2 levels deep. Depth 3 allows `@A → @B → @C → literal` which covers all
  realistic use cases. Raising the limit requires a BC amendment.
- `AliasStore::dependents()` scans all alias definitions for `@name` references using the
  same regex as `AliasResolver`. This is O(n) over all aliases — acceptable for the
  expected cardinality (< 1,000 aliases per deployment).
- The alias system is P1 priority — it MUST be skippable if wave scheduling requires
  accelerating P0 stories. The alias integration point in `engine.rs` (S-3.02) should
  be a no-op when the alias store is not initialized, falling through to raw query
  string parsing.
- VP-037 is a broad fuzz target: provide it with all four edge case categories:
  cycles (A → A), deep nesting (A → B → C → D → E), self-reference (`@A` inside A's
  definition), and non-UTF-8 bytes in alias names. Each should produce an Err, not a panic.
