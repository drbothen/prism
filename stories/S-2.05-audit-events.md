# S-2.05 — prism-audit: Specialized Audit Events

## Narrative

As a Prism compliance operator, I want specialized audit events for credential access,
feature flag evaluations, and confirmation token lifecycle, and I want all audit entries
to be compatible with the Vector log pipeline, so that every security-sensitive action
has a dedicated, structured record that downstream SIEM and compliance tooling can
consume without transformation.

## Objective

Extend the `prism-audit` crate with four specialized audit event types: credential access
events, Vector pipeline-compatible serialization, feature flag evaluation events for write
operations, and confirmation token lifecycle events. All event types share the base
`AuditEntry` structure from S-2.04 and add domain-specific detail fields.

## Tasks

1. Implement `CredentialAccessEvent` in `crates/prism-audit/src/credential_events.rs`:
   - Embeds in `AuditEntry.parameters` as a `CredentialAccessDetail` struct:
     - `credential_name: String` — the name/reference of the credential (never the value)
     - `access_type: CredentialAccessType` — `Read | Write | Delete | Rotate`
     - `requesting_context: RequestingContext` — contains `tool_name`, `client_id`,
       `trace_id` of the parent invocation (BC-2.05.005)
   - `emit_credential_event(name: &str, access_type: CredentialAccessType, ctx: &QueryContext)`
     constructs and emits the entry via `AuditEmitter::emit()`

2. Enforce credential-name-only invariant in `CredentialAccessEvent`:
   - A compile-time doc comment on `CredentialAccessDetail` MUST state: "credential_value
     MUST NOT appear in this struct — only the name/reference"
   - Add a property test (proptest) asserting that serialized `CredentialAccessDetail`
     JSON does not contain any field named `value`, `secret`, `password`, or `token`
     when populated via `emit_credential_event()` (BC-2.05.005)

3. Implement Vector pipeline compatibility in `crates/prism-audit/src/vector_compat.rs`:
   - `VectorAuditEntry` is a newtype wrapper around `AuditEntry` with a custom
     `Serialize` implementation that adds Vector-required fields:
     - `"@timestamp"`: RFC 3339 copy of `timestamp`
     - `"host"`: hostname from `std::env::var("PRISM_HOST_ID")` or `gethostname()`
     - `"service"`: fixed string `"prism"`
     - `"log.level"`: `"info"` for success, `"error"` for failure (BC-2.05.007)
   - `to_vector_json(entry: &AuditEntry) -> serde_json::Value` produces the
     Vector-compatible JSON without modifying the stored entry

4. Implement feature flag evaluation audit events in `crates/prism-audit/src/flag_events.rs`:
   - `FlagEvalDetail` struct:
     - `client_id: String`
     - `capability_path: String` — dot-separated path e.g. `"sensors.crowdstrike.write"`
     - `evaluation_result: bool` — final grant/deny decision
     - `resolution_trace: Vec<FlagResolutionStep>` — ordered list of rules evaluated
     - `FlagResolutionStep { rule_id: String, matched: bool, reason: String }` (BC-2.05.009)
   - `emit_flag_eval(detail: FlagEvalDetail, ctx: &QueryContext)` constructs and emits
     the entry; called from the feature flag evaluation path in `prism-flags`

5. Implement confirmation token lifecycle audit events in
   `crates/prism-audit/src/token_events.rs`:
   - `TokenLifecycleDetail` struct:
     - `token_id: String` — opaque token identifier (not the token value)
     - `event_type: TokenEvent` — `Generated | Consumed | Expired`
     - `action_summary: String` — human-readable description of the gated action
     - `expiry_time: DateTime<Utc>` — when the token expires/expired (BC-2.05.010)
   - `emit_token_generated(token_id: &str, action_summary: &str, expiry: DateTime<Utc>)`
   - `emit_token_consumed(token_id: &str)` — sets `expiry_time` to `now`
   - `emit_token_expired(token_id: &str, expiry: DateTime<Utc>)` — called by the
     token expiry sweep task

## Acceptance Criteria

**AC-1:** Given a credential read operation for `"crowdstrike_api_key"`, When
`emit_credential_event()` is called, Then the audit entry contains
`credential_name: "crowdstrike_api_key"`, `access_type: "Read"`, and the requesting
context fields — and the serialized JSON does not contain the credential value. (BC-2.05.005)

**AC-2:** Given any `AuditEntry`, When `to_vector_json()` is called, Then the resulting
JSON contains `"@timestamp"`, `"host"`, `"service": "prism"`, and `"log.level"` fields
matching the Vector schema. (BC-2.05.007)

**AC-3:** Given a write operation that requires a feature flag check for
`"sensors.crowdstrike.write"`, When the flag evaluation audit is emitted, Then the entry
contains the full `resolution_trace` with each `FlagResolutionStep`. (BC-2.05.009)

**AC-4:** Given a confirmation token generated for action `"isolate host acme-ws-01"`,
When `emit_token_generated()` is called, Then the persisted audit entry contains
`action_summary: "isolate host acme-ws-01"` and `expiry_time` — and does NOT contain
`token_id` in the persisted parameters JSON (minimum-disclosure / correlation-prevention
per BC-2.05.010 postcondition). (BC-2.05.010)

## Architecture Compliance Rules

- `CredentialAccessDetail` MUST NOT contain a `value` field of any type — enforce via
  a property test and a deny clippy annotation on any field named `value` in that module
- `VectorAuditEntry` MUST NOT modify the stored `AuditEntry` — `to_vector_json()` is a
  read-only view; the stored entry in RocksDB always uses the canonical `AuditEntry` format
- `FlagEvalDetail` and `TokenLifecycleDetail` are embedded in `AuditEntry.parameters` as
  JSON — they MUST NOT be separate RocksDB entries; they share the `audit_buffer` CF
  with all other audit entries

## Previous Story Intelligence

S-2.04 established `AuditEntry`, `AuditEmitter::emit()`, and `redaction.rs`. All
specialized event types in this story embed their detail structs in `AuditEntry.parameters`
and call `emit()` via the same path — they are not parallel mechanisms. The credential
redaction from S-2.04 applies to the `parameters` field of all entries, including these
specialized ones.

## Dev Notes

- `to_vector_json()` produces a flat JSON object, not a nested structure — Vector's
  default parser expects flat fields. Nested structs in `parameters` are serialized as
  a JSON string value under the `"parameters"` key, not as nested JSON objects.
- The `"log.level"` field should use Vector's canonical values (`"info"`, `"error"`,
  `"warn"`) rather than Rust tracing level names.
- `emit_token_consumed()` and `emit_token_expired()` are distinct events — a consumed
  token is intentionally used, while an expired token was not used in time. Both are
  security-relevant for audit purposes.
- `FlagResolutionStep.reason` should be human-readable (e.g., `"client_id matched
  allowlist rule R-001"`) to make audit entries useful without reference to internal rule
  IDs.
