# S-6.17 — prism-dtu-splunk-hec: DTU for Splunk HTTP Event Collector — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Splunk HTTP Event
Collector (HEC) at fidelity level L2 (stateful), so that integration tests for the
Splunk HEC log-forwarding destination can validate HEC token auth, index routing,
`sourcetype` capture, batch ingestion (newline-delimited JSON), HEC response codes
(code 0 success, code 6 invalid format, code 7 token mismatch), and 429 simulation
without hitting live Splunk infrastructure.

## Objective

Implement `prism-dtu-splunk-hec` — an L2 stateful behavioral clone of the Splunk HEC.
The DTU implements the `/services/collector/event` endpoint for structured event ingestion
(single or batch NDJSON), validates `Authorization: Splunk {token}` header, captures
received events with their `index` and `sourcetype` metadata for test assertion, and
returns HEC-format response codes. See `dtu-assessment.md §3.7.2` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-splunk-hec/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `SplunkHecClone` struct implementing `BehavioralClone`:
   - `received_events(&self) -> Vec<CapturedHecEvent>`: test API for assertions

3. Create `src/state.rs` — `SplunkHecState`:
   - `received_events: Mutex<Vec<CapturedHecEvent>>`
   - `CapturedHecEvent { event: Value, index: Option<String>, sourcetype: Option<String>, time: Option<f64> }`
   - `request_count: AtomicU32`
   - `reset()` clears both

4. Create `src/routes/collector.rs`:
   - `POST /services/collector/event`:
     - Validate `Authorization: Splunk {token}` header:
       - If absent: return 403 `{"text": "No data", "code": 2}` (HEC code 2 = missing token)
       - If present but `auth_mode == "reject"`: return 403 `{"text": "Invalid token", "code": 7}`
       - Any non-empty token accepted as valid when not in reject mode
     - Parse body as NDJSON (newline-delimited JSON objects):
       - Each line must be a valid JSON object with `event` field required
       - If malformed (not valid JSON): return 400 `{"text": "Invalid data format", "code": 6}`
       - If `event` field missing: return 400 `{"text": "Event field required", "code": 6}`
     - For each parsed event: capture `event`, `index`, `sourcetype`, `time` fields into store
     - On success: return 200 `{"text": "Success", "code": 0}`
     - Rate-limit: if threshold exceeded, return 429 with `Retry-After: 60`
   - `POST /services/collector/raw`:
     - Raw text ingestion; stores raw body as `event: {raw}` with no field validation
     - Returns 200 `{"text": "Success", "code": 0}` on success

5. Create `GET /dtu/received-events` test API:
   - Returns `{"events": [{event, index, sourcetype, time}, ...], "count": N}`
   - Integration tests assert `index` routing correctness (events land in correct Splunk index)

6. Create fixture files:
   - `fixtures/success-response.json`: `{"text": "Success", "code": 0}`
   - `fixtures/invalid-format-response.json`: `{"text": "Invalid data format", "code": 6}`
   - `fixtures/token-mismatch-response.json`: `{"text": "Invalid token", "code": 7}`

7. Create `tests/fidelity.rs`:
   - Tests valid NDJSON batch, token auth failure, invalid JSON format, index routing assertion
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /services/collector/event` with `Authorization: Splunk my-token`
and a NDJSON body `{"event": "test message", "index": "security", "sourcetype": "prism"}`,
Then the response is HTTP 200 `{"text": "Success", "code": 0}` AND `GET /dtu/received-events`
shows the event with `index: "security"` and `sourcetype: "prism"`.

**AC-2:** Given `POST /services/collector/event` without `Authorization` header, Then
the response is HTTP 403 `{"text": "No data", "code": 2}`.

**AC-3:** Given `auth_mode=reject` configured, When any request with an `Authorization` header
arrives, Then the response is HTTP 403 `{"text": "Invalid token", "code": 7}`.

**AC-4:** Given `POST /services/collector/event` with a body that is not valid NDJSON,
Then the response is HTTP 400 `{"text": "Invalid data format", "code": 6}`.

**AC-5:** Given a NDJSON body with 3 events on separate lines, Then `GET /dtu/received-events`
returns 3 captured events (batch NDJSON parsing works).

**AC-6:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then HTTP 429.

## Architecture Compliance Rules

- DTU clone MUST match the Splunk HEC API contract exactly (same endpoints, same
  request/response shapes, same HEC response codes) as documented in `dtu-assessment.md §3.7.2`.
- DTU clone MUST NOT introduce new behaviors not present in the real Splunk HEC.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution — all state is in-process.
- HEC response format MUST use integer `code` fields (not strings) to match the real HEC
  wire format: `{"text": "...", "code": N}`. Using string codes is a breaking deviation.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — never compiled
  into production binaries.
- See `.factory/specs/architecture/dtu-assessment.md` (or `dtu-assessment.md §4`) for
  cross-cutting DTU rules including ephemeral port binding and `BehavioralClone` contract.
- **Forbidden dependencies:** `prism-dtu-splunk-hec` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | NDJSON body with empty lines between events | Empty lines skipped; valid events parsed |
| EC-002 | `index` field absent from event | Event captured with `index: None`; DTU does not enforce index existence |
| EC-003 | Single event posted (not NDJSON batch) | Valid JSON object treated as a single-event batch |
| EC-004 | `/services/collector/raw` POST with arbitrary text | Stored as `{"event": "{raw body}"}` with `index: None` |
| EC-005 | Network timeout via FailureLayer::Latency | Prism log forwarder must handle; DTU does not enforce client-side timeout |
| EC-006 | `reset()` called during active event ingestion | In-flight capture completes; post-reset `received_events` is empty |
| EC-007 | Auth token contains spaces (e.g., `Authorization: Splunk my token`) | DTU accepts any non-empty value after `Splunk ` prefix; spaces in token permitted |

## Dev Notes

L2 (stateful) fidelity: the DTU captures received events with their full HEC metadata
(`index`, `sourcetype`, `time`) for test assertion. The HEC response code format
(`{"text": "...", "code": N}`) is the primary integration contract — Prism's log
forwarder must parse these response codes to distinguish success (0) from format error
(6) vs token error (7). The `GET /dtu/received-events` test API enables index routing
assertions without Splunk access.

Fidelity L2 is appropriate for the Splunk HEC destination because the primary integration
risk is Prism correctly formatting NDJSON event batches, setting the HEC token, and routing
events to the correct Splunk index. L4 fault injection is not required for this destination.
