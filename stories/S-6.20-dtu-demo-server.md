# S-6.20 — prism-dtu-demo-server: Unified Multi-Clone Demo Harness

## Narrative

As a Prism analyst, demo presenter, or CI system, I want a single command that starts
multiple DTU clones with configurable ports, fixture sets, and failure modes, so that
live demos and recorded walkthroughs work identically, and so production Prism can be
pointed at the harness via a matching config file.

## Objective

Implement `prism-dtu-demo-server` — a binary crate that boots multiple DTU clones in a
single process for both live demos (analyst-facing walkthroughs, stakeholder presentations,
sales demos) and recorded demos (VHS tapes, shell transcripts, CI regression artifacts).

Closes the following gaps identified in DTU design review:

- Only `S-6.09` ships a `demo_server` binary; no multi-clone launcher exists.
- Per-clone demo servers each hardcode a fixed port — no coordinated port table.
- Static fixture sets cannot be overridden at runtime without recompiling.
- All existing demo servers are plain HTTP only; TLS is required for stakeholder demos.

This crate orchestrates the six merged DTU clones (S-6.07, S-6.08, S-6.09, S-6.10,
S-6.14, S-6.15) via their `BehavioralClone` trait interfaces. It does NOT duplicate
any clone logic.

**Actual workspace crates (verified against Cargo.toml @ develop HEAD 94033a69):**
- `prism-dtu-crowdstrike` (S-6.07, L4)
- `prism-dtu-claroty` (S-6.08, L4)

5. Implement `src/harness.rs` — `DemoHarness`:

   **Harness ownership model:** (H1)
   The harness owns clone objects by-value in a `Vec<ClonePair>`. It does NOT hold
   `Box<dyn BehavioralClone>` directly in long-lived fields across await points — instead,
   each `ClonePair` is created, `start_on` is called immediately (while the `&mut` borrow
   is active for that specific pair only), and then the `ClonePair` is moved into the Vec.
   The harness also holds `Vec<tokio::task::JoinHandle<()>>` (supervisor task handles,
   parallel to the `Vec<ClonePair>`) and a `tokio::sync::broadcast::Sender<()>` for the
   shutdown signal.

   ```
   DemoHarness
   ├── pairs: Vec<ClonePair>          (owns clone objects by-value; indexed by clone position)
   ├── tasks: Vec<JoinHandle<()>>     (supervisor task handles; parallel index to pairs)
   └── shutdown_tx: broadcast::Sender<()>  (one channel, all clones subscribe)
   ```

   `ClonePair` holds a vendor clone and its associated DTU metadata clone (for future
   dual-endpoint testing; for the 6 crates in scope this version, DTU metadata is
   carried alongside the vendor clone in a single struct):

   ```rust
   pub struct ClonePair {
       pub name: String,
       pub clone: Box<dyn BehavioralClone>,
       pub bound_addr: Option<SocketAddr>,
   }

   impl ClonePair {
       /// Factory method. Construct a pair from a named clone.
       pub fn new(name: impl Into<String>, clone: Box<dyn BehavioralClone>) -> Self {
           Self { name: name.into(), clone, bound_addr: None }
       }
   }
   ```

   `ClonePair::start_on` is a single call taking both the bind address and a shutdown
   receiver: `pair.clone.start_on(bind_addr, Some(shutdown_rx)).await`. The pair stores
   the returned `SocketAddr` in `pair.bound_addr`. (M1, M3)

   **`await_holding_lock` compliance:** The harness holds `Box<dyn BehavioralClone>` inside
   `ClonePair` which is stored in a `Vec`. There is no `Mutex` wrapping — each clone is
   accessed sequentially during startup (no concurrent `&mut` borrows). During normal
   operation, clone objects are touched only by the supervisor tasks (which own their
   `JoinHandle`, not the clone object) and by `stop_all()` (which iterates `pairs`
   sequentially, no lock held across await). This design satisfies `await_holding_lock =
   "deny"` without mutex suppression. (H1)

   **`start_all()` implementation:**
   ```
   for each enabled ClonePair:
     1. Create broadcast::Receiver<()> from shutdown_tx
     2. Construct SocketAddr from (clone_config.bind, clone_config.port)
     3. Call pair.clone.start_on(bind_addr, Some(shutdown_rx)).await
        - On success: store returned SocketAddr in pair.bound_addr
        - On EADDRINUSE or other bind failure:
          * Call stop_all_started(already_started_count).await
          * Return Err(...) propagating the bind error
     4. Spawn supervisor tokio::task: awaits clone's server future (already running
        inside start_on's spawn). Store JoinHandle in self.tasks.
   ```

   **Partial-startup cleanup:** If clone N fails to bind, the harness calls
   `pair.clone.stop().await` on the N-1 already-started pairs before returning
   `Err`. The harness is the cleanup owner — not `BehavioralClone::stop`. Rationale:
   the harness already owns the clone list; a harness-side cleanup loop is the only
   component that knows which clones successfully started. `BehavioralClone::stop` is
   a per-clone abort, not a fleet rollback. (H4, H5)

   **`stop_all()` implementation:** (H2)
   ```
   1. Send shutdown signal: self.shutdown_tx.send(()).ok()
      (ignores "no receivers" error if all tasks already exited)
   2. tokio::time::timeout(Duration::from_secs(5), join_all(&mut self.tasks)).await
      - If all tasks complete within 5s: graceful drain succeeded
      - If timeout elapses: for each task NOT yet complete, call pair.clone.stop().await
        (hard-abort path via JoinHandle::abort() inside stop())
   3. For every pair regardless of path taken: call pair.clone.reset().await.ok()
   ```

   **Stop ordering:** DTU clone stops BEFORE the harness releases its own shutdown
   signal reference, but within a clone pair both vendor and DTU metadata aspects
   stop together (they share one JoinHandle per pair in the current single-server-per-pair
   design). No cross-clone ordering dependency exists — all pairs stop in parallel via the
   broadcast signal, then the hard-abort pass iterates pairs sequentially. (H2)

   **Shutdown timeout values:** 5s graceful drain; 10s is the outer process-level
   watchdog (SIGKILL from the OS after SIGTERM goes unanswered — not managed by the
   harness). The harness only enforces the 5s graceful window; after that it calls
   `stop()` (abort) immediately without an additional delay. (H2)

   **`bound_addr()` precondition:** All 6 clones use `.expect("called before start()")`
   in their current `bound_addr()` implementation. The harness MUST call `start_on`
   for each clone BEFORE reading `bound_addr` for the URL table. The harness startup
   sequence in `start_all()` (above) guarantees this: `pair.bound_addr` is populated
   from the `start_on` return value before the URL table is printed. The harness does
   NOT call `clone.bound_addr()` directly after Task 14 — it reads `pair.bound_addr`
   instead. This avoids the pre-start panic path. (M1)

   **Demo harness error reporting:** The harness aborts on clone start failure by
   default. Each `ClonePair` in `DemoConfig` may carry `continue_on_error: bool`
   (default `false`). If `continue_on_error: true` for a pair, a bind failure logs
   `[WARN] <name> failed to start: <err> — continuing per continue_on_error=true` and
   skips that pair; partial-startup cleanup still runs for any previously-started pairs
   that must be stopped due to the failure. (M5)

   Additional methods:
   - `print_url_table()`: formats and prints the URL table to stdout

   **Implementation note on teardown (ADR-002 amendment):** Pass 3 of adversarial
   review (F-6.20-P03-H-001 + F-6.20-P03-H-002) established that the no-trait-extension
   position from F-6.20-04 was incorrect — the demo harness cannot provide configurable
   bind address OR graceful shutdown without extending `BehavioralClone`. The trait is

   `clone.start_on(bind, Some(shutdown_rx))` and `clone.stop()` for lifecycle management.
   `BehavioralClone::reset()` still only clears captured state — it does not stop the
   HTTP server.

14. **Update 6 existing clone crates to implement `start_on` and `stop`** (cross-story
    impact of ADR-002 amendment):

    Each clone must implement the two new `BehavioralClone` methods: `start_on(bind,
    shutdown)` and `stop()`. The per-crate delta varies significantly because 4 of the 6
    crates do NOT have a `server_handle` field and do NOT use `with_graceful_shutdown`.
    The "one-line update" characterization in v1.3 was wrong; this task provides the

    **Stop semantics:** `stop()` calls `JoinHandle::abort()` on the server task handle.
    This is an immediate abort — no graceful drain in `stop()` itself. Graceful drain
    happens via the `shutdown_rx` passed to `start_on`; `stop()` is the hard-abort fallback
    called only after the 5s graceful timeout elapses. All axum `start_on` implementations
    MUST use `axum::serve(...).with_graceful_shutdown(...)` wired to the shutdown receiver
    so that graceful drain works when the broadcast fires. (H2)

    **Harness startup cleanup (partial-startup failure):** If clone N fails during
    `start_all()`, the harness (not `BehavioralClone`) calls `stop()` on the N-1
    already-started clones. `BehavioralClone::stop()` is the per-clone abort primitive;
    the harness is the fleet coordinator. (H4, H5)

    **Constructor fallibility summary (M3 — factory pattern awareness):**

    | Crate | Constructor | Fallibility | Reason |
    |-------|-------------|-------------|--------|
    | crowdstrike | `new() -> Self` | Infallible | No fixture I/O at construction |
    | claroty | `new() -> Self` | Infallible | No fixture I/O at construction |
    | threatintel | `new() -> Self` | Infallible | Fixture registry seeded with defaults, no file I/O |
    | cyberint | `new() -> anyhow::Result<Self>` | Fallible | Loads `alerts.json`, `alerts-page2.json`, `threats.json` at construction |
    | armis | `new() -> anyhow::Result<Self>` | Fallible | Loads `devices.json`, `device-activity.json`, `alerts.json` |
    | nvd | `new() -> anyhow::Result<Self>` | Fallible | Loads `fixtures/cves.json` |

    The harness factory in `src/harness.rs` must handle both patterns. Use a helper:
    ```rust
    fn build_clone_pairs(config: &DemoConfig) -> anyhow::Result<Vec<ClonePair>> {
        let mut pairs = Vec::new();
        if config.clones.crowdstrike.enabled {
            pairs.push(ClonePair::new("crowdstrike", Box::new(CrowdstrikeClone::new())));
        }
        if config.clones.cyberint.enabled {
            // fallible — propagate with ?
            pairs.push(ClonePair::new("cyberint", Box::new(CyberintClone::new()?)));
        }
        // ... repeat for each enabled clone
        Ok(pairs)
    }
    ```

    The spec does NOT require unifying constructor fallibility — the harness wraps the
    asymmetry in `build_clone_pairs`. A `impl_new_fallible!` macro is not introduced.

    **`StubConfig.bind` vs `start_on` addr precedence (M4):**
    `start_on(addr, shutdown)` parameter always wins. `StubConfig.bind` is only
    consulted by the `start()` default-impl shim (which the harness never calls directly).
    The harness constructs the `SocketAddr` from `(demo_config.clones.<name>.bind,
    demo_config.clones.<name>.port)` and passes it to `start_on`. This eliminates
    ambiguity: there is exactly one source of truth for the bind address in harness context.
    Document this in the `start()` default-impl body comment:
    `// NOTE: bind addr comes from start_on param; StubConfig.bind only used by this shim`.

    **Scope note:** This task is owned by the S-6.20 implementer in the same PR.

response end.

**AC-8:** Given `cargo build -p prism-dtu-demo-server` is run WITHOUT `--features dtu`,

## ADR-002 Amendment — BehavioralClone Trait Extension (Pass 3)

**Context:** ADR-002 established the L2 DTU clone template and the `BehavioralClone` trait.
Pass 1 adversarial review (F-6.20-04) enforced a no-trait-extension constraint: the demo
harness must not add methods to `BehavioralClone`. That constraint was accepted in v1.0–v1.2.

**Finding:** Pass 3 adversarial review surfaced that the no-trait-extension position was
incorrect. Two independent implementation requirements cannot be satisfied without trait
extension:

1. **F-6.20-P03-H-001 (Configurable bind address):** The demo harness needs each clone to
   bind to an operator-specified address (e.g., `127.0.0.1:17080` from `demo.toml`). All 6
   merged clone crates hardcode `TcpListener::bind("127.0.0.1:0")` in `start()` with no
   override path. Adding `bind` to `StubConfig` alone is insufficient — the trait's `start()`
   signature gives the harness no way to pass the bind address into the clone's server loop.

2. **F-6.20-P03-H-002 (Graceful shutdown signal):** The harness needs to deliver a shutdown
   signal to each clone's axum server (to wire `with_graceful_shutdown`). The existing trait
   has no mechanism for this. The previous v1.2 `stop_all()` design (broadcast then
   runtime-drop) silently aborts in-flight requests — not graceful drain.

**Revised position (accepted):** Extend `BehavioralClone` with two new methods. This is the
least-bad option — the alternatives (shared global state, process signals, or building
routers outside the clone) are worse coupling.

**Revised `BehavioralClone` trait** (to be implemented in `crates/prism-dtu-common/src/clone.rs`
as part of S-6.20):

```rust
#[async_trait]
pub trait BehavioralClone: Send + Sync + 'static {
    /// Start the clone with the default bind address (127.0.0.1:0) and no shutdown signal.
    /// Existing clone implementations and integration tests call this method — no changes
    /// required in any of the 6 merged clone crates for backward compat.
    async fn start(&mut self) -> anyhow::Result<()> {
        self.start_on("127.0.0.1:0".parse().unwrap(), None).await.map(|_| ())
    }

    /// NEW: Start with an explicit bind address and optional shutdown receiver.
    /// Returns the bound SocketAddr. The demo harness calls this method.
    /// Implementations MUST wire the shutdown receiver into
    /// axum::serve(...).with_graceful_shutdown(...) for graceful drain.
    async fn start_on(
        &mut self,
        bind: SocketAddr,
        shutdown: Option<tokio::sync::broadcast::Receiver<()>>,
    ) -> anyhow::Result<SocketAddr>;

    /// NEW: Forcibly abort the server task via JoinHandle::abort().
    /// Called by the harness when the 5s graceful drain timeout elapses.
    /// Also called by the harness during partial-startup cleanup (N-1 clones).
    /// No default impl — each clone must implement it.
    async fn stop(&mut self) -> anyhow::Result<()>;

    /// Reset all captured state (requests, counters, injected errors) to initial values.
    /// Does NOT stop the HTTP server.
    async fn reset(&self) -> anyhow::Result<()>;

    /// Reconfigure the stub at runtime (e.g. change failure mode, latency).
    async fn configure(&self, config: serde_json::Value) -> anyhow::Result<()>;

    /// Return the SocketAddr the stub is actually bound to.
    /// Panics if called before start_on(). Harness MUST only call this after
    /// start_on() returns Ok — it reads pair.bound_addr (the returned SocketAddr)
    /// rather than calling bound_addr() directly.
    fn bound_addr(&self) -> SocketAddr;

    /// Convenience: HTTP base URL derived from bound_addr.
    fn base_url(&self) -> String {
        format!("http://{}", self.bound_addr())
    }
}
```

**Pass 4 architect check:** Pass 4 findings (H1, H2, H5) required clarifying the harness
ownership model and stop semantics but do NOT change the shape of the trait itself. The
trait as defined above is sufficient — the implementation decisions (ClonePair structure,
stop ordering, timeout values) live in the harness, not in the trait. No ADR amendment is
required for Pass 4 resolutions. The trait shape is stable.

**Backward compatibility:** The default impl for `start()` delegates to `start_on()`, so
all existing call sites (`CyberintClone::start()`, integration test harnesses) continue to
work without modification. The 6 merged clone crates implement `start_on` and `stop` per
Task 14 above.

**`lib.rs` doc comment fix (L2):** `prism-dtu-common/src/lib.rs:6` currently claims
"All 13 per-surface DTU crates (S-6.07–S-6.19)". The workspace on develop HEAD has 6 DTU
clone crates. Update the doc comment to:
`All per-surface DTU clone crates build on this foundation (currently 6; target: 13 when
S-6.11–S-6.13, S-6.16–S-6.19 land).`
This is a doc-only change; no behavior impact.

- `await_holding_lock = "deny"` workspace lint MUST pass without suppression: the harness

| EC-004 | Configure payload too large (>1 MiB sent to `/dtu/configure` on a clone port) | Each clone's own endpoint rejects with HTTP 413 `{"error": "payload too large — max 1 MiB"}` — no harness proxy layer |
| EC-005 | Simultaneous `POST /dtu/configure` calls on different clones | Each clone holds its own `Arc<State>` with independent locks — no cross-clone contention; both succeed concurrently |
| EC-006 | `prism-dtu-demo-server start` called without `--features dtu` (binary absent) | Cargo does not produce the binary; user sees `error: target 'prism-dtu-demo-server' requires the features: dtu`. No main.rs cfg attr needed. |
| EC-007 | `--tls` flag without `tls` feature compiled in | Print `[ERROR] TLS support not compiled. Rebuild with --features dtu,tls.` and exit 1 |
| EC-008 | SIGTERM received while a clone is mid-request | Graceful shutdown signal sent via broadcast channel; `axum::serve(...).with_graceful_shutdown(...)` receives it and stops accepting new connections. In-flight requests complete within 5s timeout. Requests still in-flight after 5s are force-aborted via `clone.stop()` (JoinHandle::abort). No partial-response corruption: the harness relies on HTTP/1.1 connection close after response end. |
| EC-009 | `--bind-any` passed but `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND` not set (or set to wrong value) | Print `[ERROR] Non-loopback binding requires both --bind-any AND PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK. See R-DEMO-001.` and exit 1 |
| EC-010 | Non-loopback binding active; `/dtu/configure` or `/dtu/reset` called on a clone port | Admin endpoints are served normally — no runtime 403 enforcement at the clone level. The two-factor gate (F-6.20-P03-M-001) is the sole protection. A `[SECURITY WARNING]` log lists admin URLs at startup. |
| EC-011 | TD-WV0-05 pre-check fails (missing threatintel or nvd route files) | Print `[ERROR] TD-WV0-05 cleanup not found. Route to TD-WV0-05 remediation before implementing S-6.20.` and stop. |
| EC-012 | Partial-startup failure: clone N fails to bind after N-1 have started | Harness calls `stop()` on each of the N-1 already-started clones, then exits 1. No listener leaks. (AC-11) |
| EC-013 | `bound_addr()` called before `start_on()` completes | Panics with `"called before start()"`. Harness MUST NOT call `bound_addr()` directly — reads `pair.bound_addr` (populated by `start_on` return value) instead. |

## Risks

| Risk ID | Description | Mitigation |
|---------|-------------|-----------|
| R-DEMO-001 | Demo server accidentally bound to non-loopback in a stakeholder environment — exposes internal fixture data on a network interface | Default bind to `127.0.0.1`/`::1`; non-loopback bind requires BOTH the `--bind-any` CLI flag AND `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK` env var. Two-factor gate prevents accidental non-loopback binding; no additional runtime enforcement at admin endpoints (F-6.20-P03-M-001 resolution). Log a prominent `[SECURITY WARNING]` when both factors are present, listing all admin URLs. This is new policy established by S-6.20 — no prior ADR defines it. |
| R-DEMO-002 | Dynamic fixture seeding via `POST /dtu/configure` on a clone port breaks VHS tape determinism | Log every configure call with timestamp and payload hash to stderr; provide `POST /dtu/reset` on each clone port to restore fixture state; document configure + reset in README |
| R-DEMO-003 (hidden dep) | TD-WV0-05 retroactive cleanup may not be complete at implementation time — if `prism-dtu-threatintel` or `prism-dtu-nvd` is missing expected route files, S-6.20 will fail to compile or behave unexpectedly | Task 3 pre-implementation check gates S-6.20 work on TD-WV0-05 completion. If check fails, route to TD-WV0-05 remediation story before proceeding. |
