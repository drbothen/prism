# S-6.20 — prism-dtu-demo-server: Unified Multi-Clone Demo Harness

## Narrative

As a Prism analyst, demo presenter, or CI system, I want a single command that starts
multiple DTU clones with configurable ports, fixture sets, and failure modes, so that
live demos and recorded walkthroughs work identically, and so production Prism can be
pointed at the harness via a matching config file.

## Objective

Implement `prism-dtu-demo-server` — a binary crate that boots multiple DTU clones in a
single process for both live demos (analyst-facing walkthroughs, stakeholder presentations,
sales demos) and recorded demos (VHS tapes, shell transcripts, CI regression artifacts).

Closes the following gaps identified in DTU design review:

- Only `S-6.09` ships a `demo_server` binary; no multi-clone launcher exists.
- Per-clone demo servers each hardcode a fixed port — no coordinated port table.
- Static fixture sets cannot be overridden at runtime without recompiling.
- All existing demo servers are plain HTTP only; TLS is required for stakeholder demos.

This crate orchestrates the six merged DTU clones (S-6.07, S-6.08, S-6.09, S-6.10,
S-6.14, S-6.15) via their `BehavioralClone` trait interfaces. It does NOT duplicate
any clone logic.

## Tasks

1. Create `crates/prism-dtu-demo-server/Cargo.toml`:
   - `publish = false`, `description = "Unified multi-clone demo harness for Prism DTU clones"`
   - `[features] dtu = ["prism-dtu-crowdstrike/dtu", "prism-dtu-claroty/dtu", "prism-dtu-cyberint/dtu", "prism-dtu-armis/dtu", "prism-dtu-threatintel/dtu", "prism-dtu-nvd/dtu"]`
   - Optional feature `tls = ["rcgen", "rustls"]`
   - Binary target with `required-features` (see Task 4):
     ```toml
     [[bin]]
     name = "prism-dtu-demo-server"
     path = "src/main.rs"
     required-features = ["dtu"]
     ```
   - `[lints] workspace = true`
   - Dependencies: `prism-dtu-common` (path), `prism-dtu-crowdstrike` (path),
     `prism-dtu-claroty` (path), `prism-dtu-cyberint` (path), `prism-dtu-armis` (path),
     `prism-dtu-threatintel` (path), `prism-dtu-nvd` (path), `tokio` (1.x full),
     `axum` (0.7.x), `serde` (1.x derive), `toml` (0.8.x), `clap` (4.x derive),
     `tracing` (0.1.x), `tracing-subscriber` (0.3.x), `anyhow` (1.x)
   - Optional deps: `rcgen` (0.13.x), `rustls` (0.23.x)

2. Define `DemoConfig` TOML schema in `src/config.rs`:
   ```toml
   [clones.crowdstrike]
   enabled = true
   port = 17080          # 0 = ephemeral
   fixture_set = "default"
   initial_failure_mode = "none"
   seed = 42
   tls = false

   [clones.claroty]
   enabled = true
   port = 17081
   # ... same fields
   ```
   All fields have defaults; minimal config is `[clones.<name>] enabled = true`.

3. Pre-implementation check — verify TD-WV0-05 retroactive cleanup has landed:
   - `test -f crates/prism-dtu-threatintel/src/routes/dtu.rs` (must exist)
   - `grep -l 'get_health' crates/prism-dtu-nvd/src/` (must return a hit)

   If either check fails, STOP and route to TD-WV0-05 remediation story before
   proceeding. Both crates must expose the expected route structure before S-6.20
   can import them.

4. Implement `src/main.rs` with Clap subcommands. Do NOT add `#![cfg(feature = "dtu")]`
   to `main.rs` — feature gating is handled entirely by `required-features = ["dtu"]` in
   `Cargo.toml` (Task 1). The binary simply does not exist in non-dtu builds because Cargo
   skips it at the target level.

   Subcommands:
   - `prism-dtu-demo-server start --config <path> [--tls] [--bind-any]`
     - Validates config; prints field-level errors on invalid TOML
     - Starts each enabled clone on its configured port (or ephemeral if `port = 0`)
     - Prints URL table to stdout:
       ```
       | Clone         | URL                       | Fixture | Failure |
       |---------------|---------------------------|---------|---------|
       | crowdstrike   | http://127.0.0.1:17080    | default | none    |
       | claroty       | http://127.0.0.1:17081    | default | none    |
       ```
     - Blocks on SIGINT/SIGTERM; logs graceful shutdown and stops all clones
   - `prism-dtu-demo-server stop` — sends SIGTERM to a backgrounded harness PID (reads
     `.prism-dtu-demo-server.pid`)
   - `prism-dtu-demo-server configure <clone> <json>` — convenience wrapper over
     `POST http://127.0.0.1:{port}/dtu/configure`

5. Implement `src/harness.rs` — `DemoHarness`:
   - Holds `Vec<Box<dyn BehavioralClone>>` with clone metadata (name, configured port)
   - `start_all()`: iterates; binds each clone; collects `(name, url, fixture, failure)` rows
   - `stop_all()`: calls `reset()` on each clone (signals shutdown via tokio task abort)
   - `print_url_table()`: formats and prints the URL table to stdout

6. Implement `POST /dtu/<clone>/seed` admin endpoint in `src/seed_endpoint.rs`:
   - This is a thin harness wrapper — it forwards the JSON payload verbatim to the named
     clone's existing `POST /dtu/configure` endpoint and returns that endpoint's HTTP
     status and body back to the caller unchanged.
   - No `BehavioralClone` trait extension. No new methods on any trait.
   - Returns 404 if the clone name is not found.
   - Returns 413 if the payload exceeds 1 MiB (before forwarding).
   - Bind address for admin routes (`/dtu/configure`, `/dtu/reset`, `/dtu/<clone>/seed`)
     is `127.0.0.1` by default. Non-loopback binding requires BOTH the `--bind-any` CLI
     flag AND the environment variable `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK`.
     This is a new policy established by this story (S-6.20). See R-DEMO-001 and the Risks
     section. No prior ADR defines this rule.
   - When non-loopback binding IS active (both factors satisfied), admin endpoints
     (`/dtu/configure`, `/dtu/reset`, `/dtu/<clone>/seed`) are DISABLED and return HTTP 403
     regardless of the interface the request arrives on.

7. Implement optional TLS in `src/tls.rs` (only compiled with `features = ["tls"]`):
   - Uses `rcgen` to generate a self-signed certificate at startup (ECDSA-P256, 30-day validity)
   - Serves clone HTTPS via `rustls`
   - Prints cert fingerprint to stdout so analysts can add to trust store for demos
   - `main.rs` emits a clearly visible warning: `[DEMO-GRADE TLS] Self-signed cert. Not for production.`

8. Provide canonical configs:
   - `configs/demo.toml`: all 6 merged clones enabled on sequential ports
     (crowdstrike=17080, claroty=17081, cyberint=17082, armis=17083,
     threatintel=17084, nvd=17085; one port per clone, stable across recordings)
   - `configs/prism-demo.toml`: Prism production config preset pointing sensors at
     the demo server URLs (so `prism start --config configs/prism-demo.toml` routes
     all queries through the harness). Each sensor config uses
     `credential_ref = 'env:DEMO_FAKE_<SENSOR>_TOKEN'` where `DEMO_FAKE_*` env vars
     are set by the demo launch script to fixture-matching fake tokens (e.g.,
     `DEMO_FAKE_CROWDSTRIKE_TOKEN=dtu-fake-cs-token`).

9. Implement `--deterministic-logging` flag in `src/main.rs`:
   - When passed, configures `tracing-subscriber` to suppress timestamps, PIDs, and
     request-ids from log output.
   - Combined with `seed = 42` in `demo.toml`, this makes response bodies and log
     output reproducible across runs for the same request sequence.
   - VHS rendering determinism is out of scope for this flag.

10. Provide `crates/prism-dtu-demo-server/README.md`:
    - Live demo example: `cargo run -p prism-dtu-demo-server --features dtu -- start --config configs/demo.toml`
    - Recorded demo example with VHS: how to use the URL table to parameterise VHS tape env vars
    - Seeding example: `curl -XPOST http://127.0.0.1:17082/dtu/cyberint/seed -d @fixture-override.json`
    - TLS example: `cargo run -p prism-dtu-demo-server --features dtu,tls -- start --config configs/demo.toml --tls`
    - `DEMO_FAKE_*` env var documentation: explain that `configs/prism-demo.toml` uses
      `credential_ref = 'env:DEMO_FAKE_<SENSOR>_TOKEN'` and that `scripts/start-demo.sh`
      exports these vars automatically.

11. Create `scripts/start-demo.sh`:
    - Exports all `DEMO_FAKE_*` env vars with fixture-matching fake tokens:
      ```bash
      export DEMO_FAKE_CROWDSTRIKE_TOKEN=dtu-fake-cs-token
      export DEMO_FAKE_CLAROTY_TOKEN=dtu-fake-claroty-token
      export DEMO_FAKE_CYBERINT_TOKEN=dtu-fake-cyberint-token
      export DEMO_FAKE_ARMIS_TOKEN=dtu-fake-armis-token
      export DEMO_FAKE_THREATINTEL_TOKEN=dtu-fake-ti-token
      export DEMO_FAKE_NVD_TOKEN=dtu-fake-nvd-token
      ```
    - Then invokes `prism-dtu-demo-server start --config configs/demo.toml`.
    - Script is committed to the repository under `scripts/`.

## Acceptance Criteria

**AC-1:** Given a valid `demo.toml` enabling all 6 merged clones (`crowdstrike`, `claroty`,
`cyberint`, `armis`, `threatintel`, `nvd`), when `prism-dtu-demo-server start --config
demo.toml` runs, then all 6 clones bind their configured ports and a URL table is printed
to stdout with one row per enabled clone.

**AC-2:** Given the demo server is running, when an analyst runs
`curl http://127.0.0.1:17080/devices/queries/devices/v1 -H "Authorization: Bearer x"`,
then the response HTTP status and body structure match the S-6.07 CrowdStrike fixture
contract (cursor pagination, `resources` array).

**AC-3:** Given the demo server is running, when `POST /dtu/<clone>/seed` is called
with a JSON body, then the harness forwards the body to that clone's `POST /dtu/configure`
and returns the same HTTP status and body back to the caller. If the named clone does not
exist, the seed endpoint returns HTTP 404.

**AC-4:** Given the demo server is started with `--tls` and the `tls` feature enabled,
when a client connects over HTTPS using the auto-generated self-signed certificate, then
the TLS handshake succeeds and the request reaches the clone (verified via `curl --cacert
<cert> https://127.0.0.1:<port>/dtu/health`).

**AC-5:** Given the demo server is running, when SIGTERM is received (or Ctrl-C), then
all clone servers stop gracefully within 5 seconds and the process exits with code 0.

**AC-6:** Given the canonical `configs/prism-demo.toml` AND `DEMO_FAKE_*` env vars are
exported with fixture-matching fake tokens (as set by `scripts/start-demo.sh`), when
production Prism is started with `prism start --config configs/prism-demo.toml`, then
Prism sensor adapters resolve to the demo server URLs and all sensor queries are routed
through the harness (verified by querying `prism://sensors/health` and seeing 6 clones
reported healthy).

**AC-7:** Given `seed = 42` is configured AND `--deterministic-logging` is passed, when
the harness is run twice with the same request sequence, then the JSON response bodies
for each request are byte-identical across runs. VHS rendering determinism is out of scope.

**AC-8:** Given `cargo build -p prism-dtu-demo-server` is run WITHOUT `--features dtu`,
when the build completes, then no `prism-dtu-demo-server` binary is produced (cargo skips
the target because `required-features = ["dtu"]`).

**AC-9 (bind security):** Given `demo.toml` sets a `bind` field with a non-loopback IP
AND `--bind-any` is NOT passed (or `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND` is not set to
`I-UNDERSTAND-THE-RISK`), when the harness starts, then it refuses to bind any clone,
prints an error citing R-DEMO-001, and exits with code 1.

**AC-10:** Given the harness is running, when `curl http://127.0.0.1:<port>/dtu/health`
is issued for each of the 6 clone ports (17080–17085), then all 6 return HTTP 200 with
`{"status":"ok"}`.

## Architecture Compliance Rules

- MUST NOT compile into production binaries: handled by `required-features = ["dtu"]`
  in `Cargo.toml` `[[bin]]` declaration (same outcome as S-6.09 demo_server precedent;
  cargo skips the target entirely when `dtu` feature is absent).
- MUST respect ADR-002 L2 clone template: does NOT duplicate clone logic; only
  orchestrates existing clones through the `BehavioralClone` trait.
- Demo-only admin endpoints (`POST /dtu/<clone>/seed`, `/dtu/configure`, `/dtu/reset`,
  and TLS cert generation) MUST be loopback-only by default. Non-loopback binding
  requires BOTH `--bind-any` CLI flag AND
  `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK` env var. This two-factor
  requirement is new policy established by S-6.20 (see R-DEMO-001); no prior ADR
  defines it. When non-loopback binding is active, admin endpoints return HTTP 403.
- MUST NOT depend on production Prism crates (`prism-sensors`, `prism-query`,
  `prism-operations`, `prism-mcp`, `prism-spec-engine`, `prism-audit`). Adding any of
  these to `Cargo.toml` MUST fail via cargo-deny rule.
- Canonical port assignment in `demo.toml` (17080–17085) MUST be stable across
  recordings to ensure VHS tape byte-identical replay (AC-7).
- `configs/demo.toml` and `configs/prism-demo.toml` MUST be committed to the repository
  under `configs/`; they are not generated at runtime.

**Forbidden dependencies:** `prism-dtu-demo-server` MUST NOT depend on `prism-sensors`,
`prism-query`, `prism-operations`, `prism-mcp`, `prism-spec-engine`, or `prism-audit`.

- S-6.07..S-6.10, S-6.14, S-6.15: All merged. Their `BehavioralClone` implementations
  are the direct dependencies. S-6.20 does not re-implement any of their logic.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Port conflict: configured port already bound by another process | Print `[ERROR] Port 17080 already in use (EADDRINUSE). Choose a different port in demo.toml.` and exit 1 |
| EC-002 | Invalid TOML config | Validate before starting any clones; print field-level errors (e.g. `[clones.crowdstrike.port]: invalid type — expected integer, got string`); exit 1 |
| EC-003 | Clone crate not compiled in (cfg gate miss) | Emit a `[WARN] prism-dtu-<name> not compiled — skipping. Enable with --features dtu` and skip; do not abort harness boot |
| EC-004 | Seed payload too large (>1 MiB) | Reject with HTTP 413 `{"error": "payload too large — max 1 MiB"}` before forwarding |
| EC-005 | Simultaneous `POST /dtu/configure` calls on different clones | Each clone holds its own `Arc<State>` with independent locks — no cross-clone contention; both succeed concurrently |
| EC-006 | `prism-dtu-demo-server start` called without `--features dtu` (binary absent) | Cargo does not produce the binary; user sees `error: target `prism-dtu-demo-server` requires the features: dtu`. No main.rs cfg attr needed. |
| EC-007 | `--tls` flag without `tls` feature compiled in | Print `[ERROR] TLS support not compiled. Rebuild with --features dtu,tls.` and exit 1 |
| EC-008 | SIGTERM received while a clone is mid-request | Tokio graceful shutdown: in-flight requests complete within 5s timeout, then force-abort |
| EC-009 | `--bind-any` passed but `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND` not set (or set to wrong value) | Print `[ERROR] Non-loopback binding requires both --bind-any AND PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK. See R-DEMO-001.` and exit 1 |
| EC-010 | Non-loopback binding active; admin endpoint called | Return HTTP 403 `{"error": "admin endpoints disabled when non-loopback binding is active"}` |
| EC-011 | TD-WV0-05 pre-check fails (missing threatintel or nvd route files) | Print `[ERROR] TD-WV0-05 cleanup not found. Route to TD-WV0-05 remediation before implementing S-6.20.` and stop. |

## Risks

| Risk ID | Description | Mitigation |
|---------|-------------|-----------|
| R-DEMO-001 | Demo server accidentally bound to non-loopback in a stakeholder environment — exposes internal fixture data on a network interface | Default bind to `127.0.0.1`/`::1`; non-loopback bind requires BOTH the `--bind-any` CLI flag AND `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK` env var. Log a prominent `[SECURITY WARNING]` when both factors are present. When non-loopback binding is active, admin endpoints (`/dtu/configure`, `/dtu/reset`, `/dtu/<clone>/seed`) are disabled and return 403. This is new policy established by S-6.20 — no prior ADR defines it. |
| R-DEMO-002 | Dynamic fixture seeding via `POST /dtu/<clone>/seed` breaks VHS tape determinism | Log every seed call with timestamp and payload hash to stderr; provide `POST /dtu/<clone>/reset` endpoint to restore fixture state; document seed + reset in README |
| R-DEMO-003 (hidden dep) | TD-WV0-05 retroactive cleanup may not be complete at implementation time — if `prism-dtu-threatintel` or `prism-dtu-nvd` is missing expected route files, S-6.20 will fail to compile or behave unexpectedly | Task 3 pre-implementation check gates S-6.20 work on TD-WV0-05 completion. If check fails, route to TD-WV0-05 remediation story before proceeding. |
