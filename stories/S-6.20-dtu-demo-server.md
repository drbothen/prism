# S-6.20 — prism-dtu-demo-server: Unified Multi-Clone Demo Harness

## Narrative

As a Prism analyst, demo presenter, or CI system, I want a single command that starts
multiple DTU clones with configurable ports, fixture sets, and failure modes, so that
live demos and recorded walkthroughs work identically, and so production Prism can be
pointed at the harness via a matching config file.

## Objective

Implement `prism-dtu-demo-server` — a binary crate that boots multiple DTU clones in a
single process for both live demos (analyst-facing walkthroughs, stakeholder presentations,
sales demos) and recorded demos (VHS tapes, shell transcripts, CI regression artifacts).

Closes the following gaps identified in DTU design review:

- Only `S-6.09` ships a `demo_server` binary; no multi-clone launcher exists.
- Per-clone demo servers each hardcode a fixed port — no coordinated port table.
- Static fixture sets cannot be overridden at runtime without recompiling.
- All existing demo servers are plain HTTP only; TLS is required for stakeholder demos.

This crate orchestrates the six merged DTU clones (S-6.07, S-6.08, S-6.09, S-6.10,
S-6.14, S-6.15) via their `BehavioralClone` trait interfaces. It does NOT duplicate
any clone logic.

5. Implement `src/harness.rs` — `DemoHarness`:
   - Holds `Vec<Box<dyn BehavioralClone>>` with clone metadata (name, configured port)
   - Holds `Vec<tokio::task::JoinHandle<()>>` as `DemoHarness.tasks` — one supervisor
     task handle per enabled clone
   - Holds a `tokio::sync::broadcast::Sender<()>` as `DemoHarness.shutdown_tx`
   - `start_all()`: for each enabled clone, create a `broadcast::Receiver<()>` from
     `shutdown_tx`. Construct a `SocketAddr` from `(clone_config.bind, clone_config.port)`.
     Call `clone.start_on(bind_addr, Some(shutdown_rx)).await` — this binds the server
     with graceful shutdown wired in. Store the clone (for later `reset()`) and spawn a
     supervisor `tokio::spawn` task that awaits the clone's server future. Store the
     resulting `JoinHandle` in `self.tasks`. Collect `(name, url, fixture, failure)` rows
     for the URL table.
   - `stop_all()`:
     1. Send shutdown signal: `self.shutdown_tx.send(())` — this unblocks each clone's
        `graceful_shutdown` future (the `axum::serve(...).with_graceful_shutdown(...)` path).
     2. Wait for all supervisor `JoinHandle`s with a 5-second timeout via
        `tokio::time::timeout`. In-flight requests complete gracefully within this window.
     3. For any clone whose supervisor task has NOT completed within the 5s timeout,
        call `clone.stop()` — this aborts the server task forcibly.
     4. Call `clone.reset()` on every clone regardless of whether graceful or forced.
     The graceful drain up to 5s, then force-abort, is the standard
     `axum::serve(...).with_graceful_shutdown(...)` pattern (see EC-008 and AC-5).
   - `print_url_table()`: formats and prints the URL table to stdout

   **Implementation note on teardown (ADR-002 amendment):** Pass 3 of adversarial
   review (F-6.20-P03-H-001 + F-6.20-P03-H-002) established that the no-trait-extension
   position from F-6.20-04 was incorrect — the demo harness cannot provide configurable
   bind address OR graceful shutdown without extending `BehavioralClone`. The trait is
   extended per the ADR-002 Amendment section below. The harness uses
   `clone.start_on(bind, Some(shutdown_rx))` and `clone.stop()` for lifecycle management.
   `BehavioralClone::reset()` still only clears captured state — it does not stop the
   HTTP server.

**AC-4:** Given the demo server is started with `--tls` and the `tls` feature enabled,

<cert> https://127.0.0.1:<port>/dtu/health`).

unblocks each clone's graceful_shutdown future so in-flight requests can complete. Requests

relies on HTTP/1.1 connection close after response end.

**AC-6:** Given the canonical `configs/prism-demo.toml` AND `DEMO_FAKE_*` env vars are
exported with fixture-matching fake tokens (as set by `scripts/start-demo.sh`), when
production Prism is started with `prism start --config configs/prism-demo.toml`, then

**AC-7:** Given `seed = 42` is configured AND `--deterministic-logging` is passed, when

**AC-8:** Given `cargo build -p prism-dtu-demo-server` is run WITHOUT `--features dtu`,

## ADR-002 Amendment — BehavioralClone Trait Extension (Pass 3)

**Context:** ADR-002 established the L2 DTU clone template and the `BehavioralClone` trait.
Pass 1 adversarial review (F-6.20-04) enforced a no-trait-extension constraint: the demo
harness must not add methods to `BehavioralClone`. That constraint was accepted in v1.0–v1.2.

**Finding:** Pass 3 adversarial review surfaced that the no-trait-extension position was
incorrect. Two independent implementation requirements cannot be satisfied without trait
extension:

1. **F-6.20-P03-H-001 (Configurable bind address):** The demo harness needs each clone to
   bind to an operator-specified address (e.g., `127.0.0.1:17080` from `demo.toml`). All 6
   merged clone crates hardcode `TcpListener::bind("127.0.0.1:0")` in `start()` with no
   override path. Adding `bind` to `StubConfig` alone is insufficient — the trait's `start()`
   signature gives the harness no way to pass the bind address into the clone's server loop.

2. **F-6.20-P03-H-002 (Graceful shutdown signal):** The harness needs to deliver a shutdown
   signal to each clone's axum server (to wire `with_graceful_shutdown`). The existing trait
   has no mechanism for this. The previous v1.2 `stop_all()` design (broadcast then
   runtime-drop) silently aborts in-flight requests — not graceful drain.

**Revised position (accepted):** Extend `BehavioralClone` with two new methods. This is the
least-bad option — the alternatives (shared global state, process signals, or building
routers outside the clone) are worse coupling.

**Revised `BehavioralClone` trait** (to be implemented in `crates/prism-dtu-common/src/clone.rs`
as part of S-6.20):

```rust
#[async_trait]
pub trait BehavioralClone: Send + Sync + 'static {
    /// Start the clone with the default bind address (127.0.0.1:0) and no shutdown signal.
    /// Existing clone implementations and integration tests call this method — no changes
    /// required in any of the 6 merged clone crates for backward compat.
    async fn start(&mut self) -> anyhow::Result<()> {
        self.start_on("127.0.0.1:0".parse().unwrap(), None).await.map(|_| ())
    }

    /// NEW: Start with an explicit bind address and optional shutdown receiver.
    /// Returns the bound SocketAddr. The demo harness calls this method.
    /// Implementations wire the shutdown receiver into axum::serve(...).with_graceful_shutdown(...).
    async fn start_on(
        &mut self,
        bind: SocketAddr,
        shutdown: Option<tokio::sync::broadcast::Receiver<()>>,
    ) -> anyhow::Result<SocketAddr>;

    /// NEW: Forcibly abort the server task. Called by the harness when the 5s graceful
    /// drain timeout elapses. No default impl — each clone must implement it.
    async fn stop(&mut self) -> anyhow::Result<()>;

    /// Reset all captured state (requests, counters, injected errors) to initial values.
    /// Does NOT stop the HTTP server.
    async fn reset(&self) -> anyhow::Result<()>;

    /// Reconfigure the stub at runtime (e.g. change failure mode, latency).
    async fn configure(&self, config: serde_json::Value) -> anyhow::Result<()>;

    /// Return the SocketAddr the stub is actually bound to.
    fn bound_addr(&self) -> SocketAddr;

    /// Convenience: HTTP base URL derived from bound_addr.
    fn base_url(&self) -> String {
        format!("http://{}", self.bound_addr())
    }
}
```

**Backward compatibility:** The default impl for `start()` delegates to `start_on()`, so
all existing call sites (`CyberintClone::start()`, integration test harnesses) continue to
work without modification. The 6 merged clone crates need only implement `start_on` and
`stop` — the one-line `start()` body in each clone is replaced by the trait default.

**StubConfig.bind:** `SocketAddr` field added with default `127.0.0.1:0`. The demo harness
constructs `StubConfig.bind` from `(demo_config.clones.<name>.bind, demo_config.clones.<name>.port)`
before calling `clone.start_on(config.bind, Some(shutdown_rx))`.

**Cross-story impact:** Task 14 documents the one-line update to each of the 6 merged clone
crates required by this amendment. The implementer owns this in the S-6.20 PR.

## Architecture Compliance Rules

- MUST NOT compile into production binaries: handled by `required-features = ["dtu"]`
  in `Cargo.toml` `[[bin]]` declaration (same outcome as S-6.09 demo_server precedent;
  cargo skips the target entirely when `dtu` feature is absent).
- MUST respect ADR-002 L2 clone template: does NOT duplicate clone logic; only
  orchestrates existing clones through the `BehavioralClone` trait.
- Per-clone admin endpoints (`POST /dtu/configure`, `POST /dtu/reset`, and TLS cert
  generation) MUST be loopback-only by default. Non-loopback binding requires BOTH
  `--bind-any` CLI flag AND `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK`
  env var. This two-factor requirement is new policy established by S-6.20 (see
  R-DEMO-001); no prior ADR defines it. The two-factor gate is the sole enforcement
  mechanism — no runtime HTTP 403 is returned at clone admin endpoints (F-6.20-P03-M-001).
  There is NO harness-level admin proxy — the `/dtu/<clone>/seed` wrapper has been
  removed (F-6.20-P02-M-002).
- MUST NOT depend on production Prism crates (`prism-sensors`, `prism-query`,
  `prism-operations`, `prism-mcp`, `prism-spec-engine`, `prism-audit`). Adding any of
  these to `Cargo.toml` MUST fail via cargo-deny rule.
- Canonical port assignment in `demo.toml` (17080–17085) MUST be stable across
  recordings to ensure VHS tape byte-identical replay (AC-7).
- `configs/demo.toml` and `configs/prism-demo.toml` MUST be committed to the repository
  under `configs/`; they are not generated at runtime.

**Forbidden dependencies:** `prism-dtu-demo-server` MUST NOT depend on `prism-sensors`,
`prism-query`, `prism-operations`, `prism-mcp`, `prism-spec-engine`, or `prism-audit`.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Port conflict: configured port already bound by another process | Print `[ERROR] Port 17080 already in use (EADDRINUSE). Choose a different port in demo.toml.` and exit 1 |
| EC-002 | Invalid TOML config | Validate before starting any clones; print field-level errors (e.g. `[clones.crowdstrike.port]: invalid type — expected integer, got string`); exit 1 |
| EC-003 | Clone crate not compiled in (cfg gate miss) | Emit a `[WARN] prism-dtu-<name> not compiled — skipping. Enable with --features dtu` and skip; do not abort harness boot |
| EC-004 | Configure payload too large (>1 MiB sent to `/dtu/configure` on a clone port) | Each clone's own endpoint rejects with HTTP 413 `{"error": "payload too large — max 1 MiB"}` — no harness proxy layer |
| EC-005 | Simultaneous `POST /dtu/configure` calls on different clones | Each clone holds its own `Arc<State>` with independent locks — no cross-clone contention; both succeed concurrently |
| EC-006 | `prism-dtu-demo-server start` called without `--features dtu` (binary absent) | Cargo does not produce the binary; user sees `error: target `prism-dtu-demo-server` requires the features: dtu`. No main.rs cfg attr needed. |
| EC-007 | `--tls` flag without `tls` feature compiled in | Print `[ERROR] TLS support not compiled. Rebuild with --features dtu,tls.` and exit 1 |
| EC-008 | SIGTERM received while a clone is mid-request | Graceful shutdown signal sent via broadcast channel; `axum::serve(...).with_graceful_shutdown(...)` receives it and stops accepting new connections. In-flight requests complete within 5s timeout. Requests still in-flight after 5s are force-aborted via `clone.stop()`. No partial-response corruption: the harness relies on HTTP/1.1 connection close after response end. |
| EC-009 | `--bind-any` passed but `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND` not set (or set to wrong value) | Print `[ERROR] Non-loopback binding requires both --bind-any AND PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK. See R-DEMO-001.` and exit 1 |
| EC-010 | Non-loopback binding active; `/dtu/configure` or `/dtu/reset` called on a clone port | Admin endpoints are served normally — no runtime 403 enforcement at the clone level. The two-factor gate (F-6.20-P03-M-001) is the sole protection: harness refuses to start in non-loopback mode without both `--bind-any` AND `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK`. A `[SECURITY WARNING]` log lists admin URLs at startup. Operators enabling non-loopback binding accept responsibility for network exposure. |
| EC-011 | TD-WV0-05 pre-check fails (missing threatintel or nvd route files) | Print `[ERROR] TD-WV0-05 cleanup not found. Route to TD-WV0-05 remediation before implementing S-6.20.` and stop. |

## Risks

| Risk ID | Description | Mitigation |
|---------|-------------|-----------|
| R-DEMO-001 | Demo server accidentally bound to non-loopback in a stakeholder environment — exposes internal fixture data on a network interface | Default bind to `127.0.0.1`/`::1`; non-loopback bind requires BOTH the `--bind-any` CLI flag AND `PRISM_DTU_DEMO_ALLOW_NETWORK_BIND=I-UNDERSTAND-THE-RISK` env var.
