## Objective

Implement the confirmation token system for irreversible write operations.

## Behavioral Contracts

| BC-2.04.010 | Confirmation Token Consumption via confirm_action |
| BC-2.04.011 | Token Expiry at 300 Seconds with Structured Error Recovery |
| BC-2.04.012 | Token Content Hash Verification Prevents Action Tampering |

## Tasks

1. Define `RiskTier` enum (`Read`, `Reversible`, `Irreversible`) and classify all operations (BC-2.04.007)
2. Implement dry-run default for `Reversible` operations: `dry_run` param defaults to `true` (BC-2.04.008)
3. Implement `ConfirmationTokenStore`: generate token with UUID v7, SHA-256 content hash of action params, 300s expiry, 100-token active cap (BC-2.04.009)
4. Implement `confirm_action` flow: accept token, verify not expired, verify content hash matches, consume (single-use) (BC-2.04.010)
5. Implement 300s expiry with structured error `E-FLAG-003`: `"Token expired for action '{action_summary}'"` (BC-2.04.011)
6. Implement content hash verification: SHA-256 of serialized action params must match token's stored hash (BC-2.04.012)
7. Write VP-007 Kani proof: token at exactly 300s is expired (boundary condition)
8. Write VP-008 Kani proof: consumed token cannot be consumed again (single-use)
9. Write VP-009 Kani proof: modified action params produce different hash → rejection
10. Write VP-010 Kani proof: 101st token generation when 100 active → error

## Acceptance Criteria

- **AC-1:** Given an irreversible operation (`contain_host`), When requested without token, Then a confirmation token is returned with action summary (BC-2.04.009)
- **AC-2:** Given a valid token at 299s, When `confirm_action` is called, Then the operation executes (BC-2.04.010)
- **AC-3:** Given a token at 301s, When `confirm_action` is called, Then `E-FLAG-003` is returned (BC-2.04.011)
- **AC-4:** Given a token for `contain_host(device_id: "A")`, When `confirm_action` is called with `device_id: "B"`, Then content hash mismatch is rejected (BC-2.04.012)
- **AC-5:** Given a reversible operation (`acknowledge_alert`), When called without `dry_run: false`, Then it returns a dry-run preview (BC-2.04.008)
- **AC-6:** VP-007, VP-008, VP-009, VP-010 Kani proofs all pass

## Verification Properties

| VP ID | Description |
|-------|-------------|
| VP-007 | Kani proof: token at exactly 300s is expired (boundary condition) |
| VP-008 | Kani proof: consumed token cannot be consumed again (single-use invariant) |
| VP-009 | Kani proof: modified action params produce different hash → rejection |
| VP-010 | Kani proof: 101st token generation when 100 active → error |

## Notes

- UUID v7 is used for tokens because it embeds a timestamp, enabling efficient expiry scanning without a separate timestamp field.
- The 100-token active cap prevents unbounded memory growth from unconfirmed operations. When the cap is hit, the system MUST return E-FLAG-003 (token cap exceeded). Eviction is NOT allowed — per VP-010 Kani proof which asserts 101st token is rejected.
- Content hash is computed over the canonically serialized action params (deterministic JSON or bincode). The serialization format must be documented in the BC so implementer and test-writer use the same approach.
- Dry-run preview must describe what would happen in enough detail for the operator to make a confident confirmation decision. It must not actually execute any sensor write.
- `RiskTier::Read` operations never require confirmation tokens and are not dry-run gated.
