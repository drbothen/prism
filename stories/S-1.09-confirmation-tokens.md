# S-1.09 — prism-security: Confirmation Tokens (P1)

## Narrative

As a Prism platform implementer, I want a confirmation token system for
irreversible write operations in prism-security, so that AI agents cannot
accidentally execute high-risk actions without explicit operator confirmation —
with single-use tokens, 300-second expiry, content-hash verification, and a
100-token active cap preventing unbounded memory growth.

## Objective

Implement the confirmation token system for irreversible write operations.

## Behavioral Contracts

| BC ID | Title |
|-------|-------|
| BC-2.04.007 | Three-Tier Risk Classification for Operations |
| BC-2.04.008 | Dry-Run Default for Reversible Write Operations |
| BC-2.04.009 | Confirmation Token Generation for Irreversible Write Operations (100-Token Active Cap) |
| BC-2.04.010 | Confirmation Token Consumption via confirm_action |
| BC-2.04.011 | Token Expiry at 300 Seconds with Structured Error Recovery |
| BC-2.04.012 | Token Content Hash Verification Prevents Action Tampering |

## Tasks

1. Define `RiskTier` enum (`Read`, `Reversible`, `Irreversible`) and classify all operations (BC-2.04.007)
2. Implement dry-run default for `Reversible` operations: `dry_run` param defaults to `true` (BC-2.04.008)
3. Implement `ConfirmationTokenStore`: generate token with cryptographic random string (per BC-2.04.009 postcondition 1), SHA-256 content hash of action params, 300s expiry, 100-token active cap (BC-2.04.009)
4. Implement `confirm_action` flow: accept token, verify not expired, verify content hash matches, consume (single-use) (BC-2.04.010)
5. Implement 300s expiry with structured error `E-FLAG-003`: `"Token expired for action '{action_summary}'"` (BC-2.04.011)
6. Implement content hash verification: SHA-256 of serialized action params must match token's stored hash (BC-2.04.012)
7. Write VP-007 Kani proof: token at exactly 300s is expired (boundary condition)
8. Write VP-008 Kani proof: consumed token cannot be consumed again (single-use)
9. Write VP-009 Kani proof: modified action params produce different hash → rejection
10. Write VP-010 Kani proof: 101st token generation when 100 active → error

## Acceptance Criteria

- **AC-1:** Given an irreversible operation (`contain_host`), When requested without token, Then a confirmation token is returned with action summary (BC-2.04.009)
- **AC-2:** Given a valid token at 299s, When `confirm_action` is called, Then the operation executes (BC-2.04.010)
- **AC-3:** Given a token at 301s, When `confirm_action` is called, Then `E-FLAG-003` is returned (BC-2.04.011)
- **AC-4:** Given a token for `contain_host(device_id: "A")`, When `confirm_action` is called with `device_id: "B"`, Then content hash mismatch is rejected (BC-2.04.012)
- **AC-5:** Given a reversible operation (`acknowledge_alert`), When called without `dry_run: false`, Then it returns a dry-run preview (BC-2.04.008)
- **AC-6:** VP-007, VP-008, VP-009, VP-010 Kani proofs all pass

- **AC-7:** Given all MCP tools are classified into exactly one of three risk tiers at
  registration time, When each tool is invoked: (a) a `Read` tool (e.g., `list_alerts`,
  `get_device_info`) executes immediately with no gate; (b) a `Reversible Write` tool
  (e.g., `acknowledge_alert`, `add_tag`) returns a dry-run preview by default when
  `dry_run` is not explicitly set to `false`; (c) an `Irreversible Write` tool (e.g.,
  `crowdstrike_contain_host`, `delete_credential`) returns a `ConfirmationToken` and
  requires a subsequent `confirm_action` call before the operation executes. Risk
  classification is defined at registration time and cannot change at runtime. Destructive
  operations (delete sensor, wipe endpoint) are not exposed via MCP at all.
  (traces to BC-2.04.007 postcondition — three-tier risk classification and gate routing)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Confirmation token store | `prism-security` (SS-04) | Mixed |

`prism-security` implements `ConfirmationTokenStore` — the one-time-use token mechanism
that gates high-risk write operations. Token generation and validation logic is pure;
`SystemTime` access for expiry is the only effectful boundary. Per
`architecture/module-decomposition.md`, `prism-mcp` and `prism-query` consume
`ConfirmationTokenStore` for write confirmation workflows (AD-022).

| EC-006 | Read operation (`RiskTier::Read`) goes through confirmation gate check | Returns Allow immediately with no token and no dry-run step (BC-2.04.007) |

## Dev Notes

- Token IDs are cryptographic random strings per BC-2.04.009 (not UUIDs). Implementation may use any CSPRNG-backed scheme (e.g., 256-bit random hex).
- The 100-token active cap prevents unbounded memory growth from unconfirmed operations. When the cap is hit, the system MUST return E-FLAG-007 (token cap exceeded per BC-2.04.009 postcondition). E-FLAG-003 is reserved for expired tokens per BC-2.04.011. Eviction is NOT allowed — per VP-010 Kani proof which asserts 101st token is rejected.
- Content hash is computed over the canonically serialized action params (deterministic JSON or bincode). The serialization format must be documented in the BC so implementer and test-writer use the same approach.
- Dry-run preview must describe what would happen in enough detail for the operator to make a confident confirmation decision. It must not actually execute any sensor write.
- `RiskTier::Read` operations never require confirmation tokens and are not dry-run gated.
