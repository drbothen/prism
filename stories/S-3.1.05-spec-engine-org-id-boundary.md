# S-3.1.05 — prism-spec-engine: scope sensor specs per OrgId

## Narrative
- **As a** spec engine (the component that loads and serves sensor TOML specs)
- **I want** to store and retrieve sensor specs keyed by `OrgId` internally, while
  accepting `OrgSlug` at the user-facing API boundary and resolving it via `OrgRegistry`
- **So that** spec lookups are rename-stable and the registry/store layer enforces
  org identity at the type level

## Description

ADR-006 §4 Step 2 migrates the spec engine's internal spec store from slug-keyed to
`OrgId`-keyed. The user-facing API (e.g., `get_spec_for_client(slug, sensor)`) continues
to accept `OrgSlug` for ergonomics, but the pipeline immediately resolves it to `OrgId`
via `OrgRegistry::resolve` and passes `OrgId` to the internal store. The internal store
`HashMap<(OrgId, String), SensorSpec>` replaces the former `HashMap<(OrgSlug, String), SensorSpec>`.
This makes spec lookups rename-stable and cross-tenant spec isolation structurally
enforced.

## Acceptance Criteria

**AC-1 (traces to BC-3.1.001 postcondition 1):** The internal spec store is keyed by
`(OrgId, sensor_name)`. The user-facing method `get_spec(slug: &OrgSlug, sensor: &str)`
calls `OrgRegistry::resolve(slug)` to obtain `OrgId` before indexing the store. If
`resolve` returns `None`, the method returns `Err(SpecError::UnknownOrg(slug))`.

**AC-2 (traces to BC-3.1.001 invariant 1):** `OrgRegistry::resolve` is called exactly
once per `get_spec` invocation and not at all during internal store operations — the store
layer accepts `OrgId` only.

**AC-3 (traces to BC-3.1.001 invariant 3):** If `OrgRegistry` is not yet initialized
when `get_spec` is called, the method returns `Err(SpecError::RegistryNotInitialized)`
rather than panicking. (In practice this is unreachable post-startup, per BC-3.1.001
invariant 3, but the API must not panic.)

**AC-4 (traces to BC-3.1.001 postcondition 3):** Specs stored under `(OrgId(A), "crowdstrike")`
are not returned by `get_spec(slug_B, "crowdstrike")` even if slug_B accidentally collides
with slug_A's string representation — the UUID-level keying prevents this.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `get_spec(unknown_slug, sensor)` | `Err(SpecError::UnknownOrg)` — slug not in OrgRegistry |
| EC-002 | `get_spec(slug_A, "armis")` when orgA has no armis spec | `Err(SpecError::SensorNotFound)` — org found but spec absent |
| EC-003 | Org renames slug-A → slug-A2; `get_spec(slug_A2, sensor)` | `Ok(spec)` — OrgId unchanged; spec stored under OrgId, not slug |
| EC-004 | `OrgRegistry` has two orgs with same sensor name | Each spec stored under its respective OrgId; no collision |
