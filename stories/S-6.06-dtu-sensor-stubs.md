# S-6.06 — prism-dtu: DTU Sensor Stub Infrastructure

## Narrative

As a Prism test infrastructure consumer, I want behaviorally-complete in-process sensor
stubs for all four primary sensors (CrowdStrike Falcon, Claroty xDome, Cyberint, and
Armis Centrix), each with deterministic responses, configurable latency and failure
modes, and pagination simulation, so that integration tests (VP-033, VP-036) and future
holdout evaluations can run against realistic sensor behavior without live API access.

## Objective

Implement a `prism-dtu` crate containing four sensor stubs. Each stub is an in-process
HTTP server (using `axum`) that serves deterministic, seeded responses matching the real
sensor API schemas. Stubs support configurable latency, failure injection (429, 500,
network timeout), and pagination. All four stubs launch via `just integration-test`.
VP-033 (audit buffer integration test) and VP-036 (SessionContext drop integration test)
are wired to execute against these stubs.

## Stub Delivery Model — Design Decision

**Options evaluated:**

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| In-process Axum server (chosen) | No docker dependency; test-time startup in ~50ms; deterministic port allocation via `TcpListener::bind("127.0.0.1:0")`; runs on all CI platforms | Slightly more test code per stub | **Selected** |
| `mockito` per-test mocking | Zero setup; single-request mocking | Cannot simulate stateful pagination, auth token lifecycle, or rate limit state across requests | Rejected for integration tests |
| `docker-compose.integration.yml` | Realistic networking | Requires Docker in CI; slow startup; non-deterministic between runs without fixed seeds | Rejected (CI complexity) |

**Decision:** In-process Axum HTTP server per stub, launched in a `tokio::task::spawn`
and bound to a random ephemeral port. Tests receive the base URL after binding. The
stub is stopped by dropping the `JoinHandle`. This is the standard pattern for
integration tests in the Rust async ecosystem.

## Tasks

1. Create `crates/prism-dtu/Cargo.toml`:
   - `[lib]` crate type, published as `test-utils` (not a production dependency)
   - Gated via `#[cfg(any(test, feature = "dtu"))]` at the crate level
   - Dependencies: `axum` (0.7.x), `tokio`, `serde_json`, `rand`, `rand_chacha`
     (for seeded RNG), `http` (1.x), `tower` (for middleware), `async-trait`
   - Dev dependency of `prism-sensors`, `prism-query`, `prism-audit`

2. Create shared infrastructure in `crates/prism-dtu/src/`:
   - `lib.rs`: re-export all four stub constructors + `StubConfig` type
   - `config.rs`: `StubConfig { seed: u64, latency_ms: u64, failure_mode: FailureMode }`
     where `FailureMode` is an enum: `None`, `RateLimit { after_n_requests: u32 }`,
     `InternalError { at_request_n: u32 }`, `NetworkTimeout { after_ms: u64 }`
   - `seed.rs`: `SeededRng::from_seed(seed: u64) -> ChaCha20Rng` — deterministic RNG
     for generating response body content
   - `router.rs`: shared middleware:
     - `LatencyLayer`: `tower::Layer` that sleeps `config.latency_ms` before forwarding
     - `FailureLayer`: `tower::Layer` that returns the configured failure response after
       N requests (tracked in `Arc<AtomicU32>`)
   - `pagination.rs`: shared pagination helper — given a `Vec<T>` and a page cursor
     (string-encoded offset), return `(page_items, next_cursor)` where cursor is
     `None` when exhausted

3. Implement the **CrowdStrike Falcon stub** in `crates/prism-dtu/src/crowdstrike.rs`:
   - Endpoints (matching CrowdStrike Falcon API v2 schema):
     - `GET /detects/queries/detects/v1` — paginated detection IDs list
     - `POST /detects/entities/detects/GET/v1` — fetch detection details by IDs
     - `POST /devices/queries/devices/v1` — paginated host IDs query
     - `POST /devices/entities/devices/GET/v2` — fetch host details by IDs
     - `POST /devices/entities/devices-actions/v2?action_name=contain` — host containment
     - `POST /devices/entities/devices-actions/v2?action_name=lift_containment` — uncontain
     - `PATCH /detects/entities/detects/v2` — update detection status
   - Auth: Bearer token in `Authorization` header; respond 401 if absent or malformed
   - Rate limit: when `FailureMode::RateLimit` is configured, return 429 with
     `Retry-After: 60` header after N requests
   - Pagination: `LIMIT`/`OFFSET`-style using `limit` + `offset` query params for
     device/detection lists; `next_cursor` is included in responses when more pages exist
   - Response bodies: generated deterministically from `SeededRng` — same seed +
     same request → same response content
   - Write response: `contain` returns `{"resources": [{"device_id": "$id",
     "containment_status": "contained"}]}` with status 202

4. Implement the **Claroty xDome stub** in `crates/prism-dtu/src/claroty.rs`:
   - Endpoints:
     - `GET /api/v1/assets` — paginated asset inventory (page + page_size params)
     - `GET /api/v1/alerts` — alert list (with status filter support)
     - `GET /api/v1/vulnerabilities` — vulnerability list (CVE-linked)
     - `GET /api/v1/assets/{id}/communication-graph` — device communication graph
     - `POST /api/v1/assets/{id}/tags` — tag a device (write endpoint)
     - `DELETE /api/v1/assets/{id}/tags/{tag_id}` — remove a tag
   - Auth: API key in `X-Claroty-API-Key` header; respond 401 if absent
   - Pagination: `page` + `page_size` query params; response includes `total` and `page`
   - Response shapes match xDome REST API documentation (field names, types)

5. Implement the **Cyberint stub** in `crates/prism-dtu/src/cyberint.rs`:
   - Endpoints:
     - `GET /api/v1/alerts` — threat intel alerts (with severity filter)
     - `GET /api/v1/assets/monitored` — monitored asset list
     - `GET /api/v1/digital-risk/findings` — digital risk findings
     - `POST /api/v1/alerts/{id}/status` — update alert status (acknowledge/close)
   - Auth: Bearer token; respond 401 if absent
   - Pagination: cursor-based (`cursor` param, `next_cursor` in response)
   - Write: `status` endpoint responds 200 with updated alert object

6. Implement the **Armis Centrix stub** in `crates/prism-dtu/src/armis.rs`:
   - Endpoints:
     - `GET /api/v1/devices` — device inventory (paginated)
     - `GET /api/v1/devices/{id}/activity` — device activity log
     - `GET /api/v1/devices/risk-scores` — risk score list
     - `GET /api/v1/policies/violations` — policy violation list
     - `POST /api/v1/devices/{id}/tags` — tag a device
     - `DELETE /api/v1/devices/{id}/tags/{tag_name}` — remove a tag
   - Auth: Bearer token; respond 401 if absent
   - Pagination: page-based

7. Wire VP-033 (audit buffer integration test) in `crates/prism-audit/tests/`:
   - Start the CrowdStrike stub with `StubConfig { seed: 42, latency_ms: 0,
     failure_mode: FailureMode::None }`
   - Configure `SensorAdapter` to point to the stub's base URL
   - Execute a query that triggers an audit write: fire a query against
     `crowdstrike_detections`, verify the audit buffer write completes before the
     sensor response is processed
   - Assert: `AuditEntry` exists in `InMemoryBackend` for the query before the query
     tool returns its result

8. Wire VP-036 (SessionContext drop integration test) in `crates/prism-query/tests/`:
   - Start the CrowdStrike stub with `StubConfig { seed: 42, latency_ms: 100,
     failure_mode: FailureMode::None }`
   - Execute a query and deliberately drop the `SessionContext` mid-execution (via
     a `CancellationToken`)
   - Assert: `SessionContext` is dropped before the error is propagated to the caller,
     and a panic in the executor does not leave the context alive

9. Create fidelity check harness in `crates/prism-dtu/src/fidelity.rs`:
   - `FidelityCheck::run(stub_url, sensor_name) -> FidelityReport`:
     - Issue one request to each stub endpoint
     - Validate response schema: field names, types, required fields
     - Return `FidelityReport { sensor, endpoint, passed: bool, failures: Vec<String> }`
   - Intended for use in a `#[test]` that runs as part of `just integration-test`

10. Confirm the `integration-test` target exists in `Justfile` (added in S-0.02):
    ```just
    integration-test:
        cargo test --workspace --features dtu -- --test-threads=1 integration
    ```

## Acceptance Criteria

**AC-1:** Given `just integration-test` is run, When all four stubs start, Then each
stub binds to an ephemeral port, serves at least one request from the fidelity check,
and all fidelity checks pass (response schema matches documented API shape).

**AC-2:** Given a CrowdStrike stub configured with `seed: 42`, When the same
`GET /devices/queries/devices/v1?limit=10` request is made twice, Then both responses
are byte-identical (deterministic seeding).

**AC-3:** Given a stub configured with `FailureMode::RateLimit { after_n_requests: 5 }`,
When the 6th request arrives, Then the stub returns HTTP 429 with a `Retry-After: 60`
header.

**AC-4:** Given VP-033 integration test runs against the CrowdStrike stub, When a query
executes, Then the audit buffer write is confirmed durable in `InMemoryBackend` before
the query result is returned to the caller. Test passes.

**AC-5:** Given VP-036 integration test runs against the CrowdStrike stub, When
`SessionContext` is dropped mid-query, Then the context is confirmed dropped before
the error propagates. Test passes.

**AC-6:** Given a paginated query against the CrowdStrike stub (requesting 50 detections
with `limit=20`), When the stub handles 3 paginated requests, Then each response contains
the correct page cursor and the total response count is 50.

**AC-7:** Given an auth request to any stub without a valid `Authorization`/API key
header, When the stub processes the request, Then it returns HTTP 401 matching the real
sensor's error format.

## Architecture Compliance Rules

- Stubs MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — they must not be
  compiled into production binaries
- Stubs MUST bind to `127.0.0.1:0` (ephemeral port) — never a hardcoded port, which
  would conflict in parallel test execution
- Stubs MUST be deterministic: same seed + same request sequence → same responses. Any
  use of `rand::thread_rng()` (non-seeded) is forbidden
- Stubs MUST NOT make outbound network calls — they are fully self-contained
- `prism-dtu` MUST NOT depend on production prism crates beyond `prism-core` type
  definitions — it is test infrastructure and must not create circular dependencies
- Fidelity checks MUST be run as part of `just integration-test` — they are not optional
- VP-033 and VP-036 integration tests MUST use the DTU stubs, not live sensor APIs

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Stub receives request for undeclared endpoint | Return 404 with JSON body `{"error": "endpoint not found"}` |
| EC-002 | Pagination request for page beyond last page | Return empty `resources` array with `next_cursor: null` |

| EC-005 | Two integration tests start stubs simultaneously | Each stub gets a different ephemeral port; no port conflicts |
| EC-006 | Fidelity check field mismatch (stub response missing required field) | `FidelityReport.failures` lists the field name; fidelity test fails with clear output |
