# S-4.02 — prism-operations: Differential Results and Packs

## Narrative

As a Prism operations engine, I want differential result computation and query pack
management, so that recurring schedules can surface only new or removed records across
executions and analysts can group related schedules into reusable packs with conditional
client-capability gating.

## Objective

Implement differential result computation (added/removed sets between schedule
executions), epoch/counter tracking for exactly-once semantics, the `get_diff_results`
MCP tool, pack loading from `.pack.toml` files with conditional execution based on client
capabilities, and pack CRUD tools.

## Tasks

1. Implement `DiffEngine` in `crates/prism-operations/src/diff/diff_engine.rs`
   (BC-2.12.005):
   - `DiffEngine::compute_diff(prev: &HashSet<RowHash>, curr: &[Row]) -> DiffResult`
   - `DiffResult { added: Vec<Row>, removed: Vec<RowHash>, epoch: u64 }`
   - Row hashing: `RowHash = blake3::hash(serde_jcs::to_vec(&row)?)` — JCS RFC 8785
     canonical JSON normalizes field ordering before hashing, ensuring order-independence
     and version-stability across struct refactors. Use `blake3 = "1.8"` (R-4, no CVEs).
     Use `serde_jcs = "0.1"` for canonical JSON serialization (ADR-018 §2.1).
     Do NOT use raw bytes or bincode for hashing — serialization-order instability would
     produce spurious diff entries across refactors.
   - `compute_diff` is a pure function: same inputs always produce same output regardless
     of row order in `curr`

2. Implement diff result storage in
   `crates/prism-operations/src/diff/diff_storage.rs` (BC-2.12.005):
   - Store previous epoch's `HashSet<RowHash>` in RocksDB `diff_results` CF under
     org-scoped keys per ADR-018 §2.6 + ADR-008:
     - `diff:{org_id}:{schedule_id}:prev` — row hash set from the N-1 execution;
       value encoding: `[0x01][bincode-encoded Vec<[u8; 32]>]` (1-byte version prefix
       per ADR-018 v0.2 §2.6 for future hash-size migration)
     - `diff:{org_id}:{schedule_id}:cur`  — row hash set from the N execution;
       same version-prefixed encoding: `[0x01][bincode-encoded Vec<[u8; 32]>]`
     - `diff:{org_id}:{schedule_id}:added` — rows in cur not in prev (bincode, no version prefix)
     - `diff:{org_id}:{schedule_id}:removed` — rows in prev not in cur (bincode, no version prefix)
   - Apply zstd compression at CF level (`set_compression_type(DBCompressionType::Zstd)`)
   - The 200MB figure applies to the RocksDB **block cache** for this CF
     (`BlockBasedOptions::set_block_cache`), NOT total disk usage — disk is managed via
     rolling two-snapshot retention (old `:prev` deleted on each fire) and age-based
     eviction via compaction filter (default 30 days, configurable via
     `PRISM_DIFF_EVICTION_DAYS`). (ADR-018 §2.6)

3. Implement `EpochTracker` in `crates/prism-operations/src/diff/epoch.rs` (BC-2.12.006):
   - Monotonically incrementing epoch counter per schedule, persisted to `diff_results` CF
   - Epoch key format: `epoch:{org_id}:{schedule_id}` (ADR-018 §2.6)
   - `EpochTracker::advance(org_id, schedule_id) -> u64`: use RocksDB
     `set_merge_operator_associative` to atomically increment — call
     `db.merge(epoch_key, &(1u64).to_le_bytes())` then `db.get(epoch_key)` to read result.
     Merge function: deserialize existing `u64::from_le_bytes`, add operand, re-serialize
     with `saturating_add`. This guarantees no lost-update races under concurrent workers.
     (ADR-018 §2.2 / R-9)
   - CRITICAL: Only `merge()` and `get()` are permitted on epoch keys — NEVER `put()`,
     which would overwrite without invoking the merge operator
   - The `diff_results` CF MUST have `set_merge_operator_associative` registered at CF
     open time; opening without it causes RocksDB error on encountering merge operands
   - On executor completion: call `advance()` exactly once per successful execution
   - On executor error: do NOT advance epoch — idempotent retry semantics

4. Implement `get_diff_results` MCP tool handler in
   `crates/prism-operations/src/diff/get_diff_results.rs` (BC-2.12.007):
   - Parameters: `schedule_id`, `epoch_start: u64`, `epoch_end: u64`
   - Return: list of `DiffResult` records for the requested epoch range
   - Validate: `epoch_start <= epoch_end` — if violated, return `E-SCHED-004` (invalid
     epoch range; NEW code distinct from E-SCHED-002 per P1-S-4.02-A-L-003); range <= 100
     epochs per call
   - Return E-SCHED-002 if schedule not found; E-SCHED-003 if epoch range not available
     (already evicted)

5. Implement pack types and `.pack.toml` parser in
   `crates/prism-operations/src/pack/pack.rs` (BC-2.12.008):
   - `Pack { id: PackId, name: String, version: String, schedules: Vec<PackSchedule>,
     required_capability: Option<CapabilityId> }` (ADR-018 §2.3 schema)
   - `PackSchedule { name: String, sql_query: String, interval_seconds: u64 }`
   - Pack-derived `ScheduleEntry` adds: `pack_origin: Option<PackId>`,
     `pack_version: Option<String>` — both `None` for individually-defined schedules
     (ADR-018 §2.3)
   - TOML format per ADR-018 §2.3: `[pack]` header with `name`, `version`,
     `required_capability` (optional); `[[schedules]]` entries with `name`, `sql_query`,
     `interval_seconds`
   - Pack loading scans configured pack directories for `*.pack.toml` files on startup
     and on hot reload signal; each pack name must match `^[a-z0-9-]+$`

6. Implement capability-gated pack expansion (BC-2.12.008, ADR-018 §2.4 + §2.5):
   - Pack expansion happens at **load time** (registration), NOT at runtime/per-tick
   - At registration: call `prism-security::feature_flags::is_enabled(required_capability,
     org_id)`. If disabled, derived `ScheduleEntry` records are persisted with
     `enabled: false`; pack is still stored for admin visibility
   - Schedule executor (S-4.01 Task 5) skips schedules where `enabled == false`
   - When capability flag toggled: pack manager receives the event via
     `prism_security::feature_flags::flag_change_watcher() -> tokio::sync::watch::Receiver<FeatureFlagSnapshot>`
     (subscribed at startup, per ADR-018 v0.2 §2.5.1). On receipt of a `FeatureFlagSnapshot`,
     re-evaluate all packs for the affected org whose `required_capability` matches any
     changed flag; update derived `ScheduleEntry.enabled` accordingly; emit
     `ScheduleChangeNotification::Updated` for each affected schedule via the ADR-013 §2.7
     watch channel

7. Implement pack CRUD tools in `crates/prism-operations/src/pack/pack_crud.rs`
   (BC-2.12.009):
   - `create_pack`/`register_pack`: parse `.pack.toml`, apply capability gate (Task 6),
     expand each `[[schedules]]` entry to a `ScheduleEntry` with `pack_origin` set,
     persist all to RocksDB `schedules` CF; packs are **org-scoped** (not global
     cross-org) — "global" means available to all clients within a single org's context
     (DRIFT-402-002). Key prefix `{org_id}:pack:{pack_id}` per ADR-008.
   - Collision semantics (ADR-018 v0.2 §2.7):
     - Same `pack_id` + same `version` + same `org_id` → no-op (idempotent success)
     - Same `pack_id` + DIFFERENT version (higher) + same `org_id` → update (replace derived schedules)
     - Same `pack_id` + DIFFERENT version (lower, regression) + same `org_id` → `Err(PackVersionRegression)`
     - Different `pack_id` with same `pack.name` (name collision) + same `org_id` → `Err(PackNameCollision)`
       (pack names must be unique per org)
   - Name collision with individual schedules: if an individual schedule name matches
     `{pack_name}:{schedule_name}`, return `Err(ScheduleNamePackCollision)` with error code
     `E-SCHEDULE-NAME-PACK-COLLISION` — pack-derived schedules win (ADR-018 §2.7)
   - `list_packs`: return all packs for the calling org with schedule count and required
     capabilities
   - `delete_pack`: remove pack metadata and all derived `ScheduleEntry` records for
     the org; no confirmation required (packs are configuration, not operational data);
     if `pack_id` not found, return `E-PACK-NOT-FOUND` (NEW — add to error taxonomy;
     P1-S-4.02-A-M-001)

8. Write VP-019 proptest in `crates/prism-operations/props/vp019_diff_determinism.rs`:
   - Property: for any set of rows, `compute_diff` output is identical regardless of
     the order in which rows appear in `curr`
   - Use `proptest` with randomly ordered `Vec<Row>` inputs

## Acceptance Criteria

**AC-1:** Given previous results `[A, B, C]` and current results `[B, C, D]`, When
`DiffEngine::compute_diff` runs, Then `added = [D]` and `removed = [A]`. (BC-2.12.005)

**AC-2:** Given previous results `[A, B, C]` and current results `[C, B, D]` (same rows,
different order), When `compute_diff` runs, Then output is identical to AC-1 —
order-independence is guaranteed. (BC-2.12.005)

**AC-3:** Given epoch 5 execution completes successfully, When `EpochTracker::advance` is
called, Then epoch advances to 6 and is durably persisted to RocksDB. (BC-2.12.006)

**AC-4:** Given epoch 5 execution fails with an error, When the executor error path runs,
Then the epoch remains at 5 (no advance) and a retry will re-execute at epoch 5.
(BC-2.12.006)

**AC-5:** Given `get_diff_results(schedule_id: "s1", epoch_start: 3, epoch_end: 5)`,
When called, Then diffs for epochs 3, 4, and 5 are returned. (BC-2.12.007)

**AC-6:** Given a pack requiring capability `"crowdstrike"`, When a client without that
capability is evaluated, Then the pack's schedules are skipped for that client with a
skip log entry. (BC-2.12.008)

**AC-7:** Given `create_pack` with two schedules, When `list_packs` is called, Then the
pack appears with `schedule_count: 2` and its required capabilities listed.
(BC-2.12.009)

**AC-8:** VP-019 proptest passes — `compute_diff` is deterministic and order-independent
for arbitrary row sets.

## Architecture Compliance Rules

- `DiffEngine::compute_diff` MUST be a pure function — no I/O, no side effects. Storage
  operations (reading/writing prev hash sets) are the caller's responsibility, not the
  engine's.
- Row hashing MUST use canonical JSON (`serde_jcs = "0.1"`, JCS RFC 8785) before applying
  blake3. Hashing raw bytes or bincode output is FORBIDDEN — serialization-order instability
  across struct refactors would produce spurious diff entries. (ADR-018 §2.1)
- Epoch advance MUST use `db.merge(epoch_key, &(1u64).to_le_bytes())` — NEVER `put()` on
  epoch keys. `put()` overwrites without invoking the merge operator, causing lost updates
  under concurrent workers. The `diff_results` CF MUST have `set_merge_operator_associative`
  registered at CF open time. (ADR-018 §2.2 / R-9)
- All `diff_results` CF keys MUST include `{org_id}:` prefix per ADR-008 universal
  re-keying rule. Key format: `diff:{org_id}:{schedule_id}:{suffix}` and
  `epoch:{org_id}:{schedule_id}`. (ADR-018 §2.6)
- Pack schedules MUST be expanded into `ScheduleEntry` records at load/create time (not
  per-tick). Runtime expansion is FORBIDDEN — it adds O(N×M) TOML parsing per tick and
  creates TOCTOU hazards on file modification. (ADR-018 §2.4)
- Capability checks for pack execution MUST occur at load time (pack registration), not
  per-fire. Per-fire capability checks add steady-state overhead for a condition that
  changes rarely. (ADR-018 §2.5)
- The 200MB cap on the `diff_results` CF applies to the RocksDB **block cache**
  (`BlockBasedOptions::set_block_cache`), NOT to total disk usage. Disk is bounded by
  rolling two-snapshot retention (only `:prev` and `:cur` kept per schedule) and
  age-based compaction-filter eviction. Do NOT attempt to enforce the 200MB cap via
  application-level byte counting. (ADR-018 §2.6)

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `compute_diff` called with empty `curr` (all rows removed) | `added = []`, `removed` = all hashes from prev; epoch advances on success |
| EC-002 | `compute_diff` called with empty `prev` (first execution) | `added` = all rows in `curr`, `removed = []`; epoch = 1 |
| EC-003 | `EpochTracker::advance` called concurrently by two executors | RocksDB merge operator serializes; exactly one increment per call; no duplicate epoch values |
| EC-004 | `get_diff_results` requested with `epoch_start > epoch_end` | Returns `E-SCHED-004` (NEW — invalid epoch range: start > end). Distinct from E-SCHED-002 (schedule not found) to avoid error-code overload (P1-S-4.02-A-L-003). Add E-SCHED-004 to error taxonomy. |
| EC-005 | `get_diff_results` requested for evicted epoch range | Returns E-SCHED-003 (epoch range not available) |
| EC-006 | Pack `.pack.toml` references an unknown capability | Pack is rejected at load time with structured validation error; other packs load normally |
| EC-007 | `delete_pack` on a pack_id that does not exist | Returns `E-PACK-NOT-FOUND` (NEW error code — add to error taxonomy); no-op on storage (P1-S-4.02-A-M-001) |
| EC-008 | Row with duplicate hash in `curr` (identical records) | Hash appears once in current set; no duplicate in `added`; determinism preserved |

## Dev Notes

- The 200MB cap for `diff_results` is informed by the osquery memory budget (200MB
  per-query) from the project memory budget. Differential state is bounded storage, not
  unbounded append.
- "Exactly-once" epoch semantics means: each successful execution increments the epoch
  exactly once, and a retry of a failed execution uses the same epoch number. This is
  "at-least-once execution with exactly-once epoch advancement on success."
- Packs are global (not client-scoped) because they represent query templates, not
  client data. The capability gating is what makes them client-aware at execution time.
- `delete_pack` does not require confirmation because packs are configuration artifacts,
  not operational data. A deleted pack can be recreated from its `.pack.toml` source.
  Contrast with `delete_schedule` in S-4.01 which requires confirmation for active
  (data-producing) schedules.
