# S-4.02 — prism-operations: Differential Results and Packs

## Narrative

As a Prism operations engine, I want differential result computation and query pack
management, so that recurring schedules can surface only new or removed records across
executions and analysts can group related schedules into reusable packs with conditional
client-capability gating.

## Objective

Implement differential result computation (added/removed sets between schedule
executions), epoch/counter tracking for exactly-once semantics, the `get_diff_results`
MCP tool, pack loading from `.pack.toml` files with conditional execution based on client
capabilities, and pack CRUD tools.

## Tasks

1. Implement `DiffEngine` in `crates/prism-operations/src/diff/diff_engine.rs`
   (BC-2.12.005):
   - `DiffEngine::compute_diff(prev: &HashSet<RowHash>, curr: &[Row]) -> DiffResult`
   - `DiffResult { added: Vec<Row>, removed: Vec<RowHash>, epoch: u64 }`
   - Row hashing: `RowHash = blake3(canonical_json(row))` — field ordering normalized
     before hashing to ensure order-independence
   - `compute_diff` is a pure function: same inputs always produce same output regardless
     of row order in `curr`

2. Implement diff result storage in
   `crates/prism-operations/src/diff/diff_storage.rs` (BC-2.12.005):
   - Store previous epoch's `HashSet<RowHash>` in RocksDB `diff_results` CF under key
     `diff:{schedule_id}:prev`
   - Apply zstd compression at CF level; enforce 200MB cap via RocksDB block cache config
     for this CF
   - Diff snapshots older than the retention window (configurable, default 7 days) are
     evicted by a background compaction filter

3. Implement `EpochTracker` in `crates/prism-operations/src/diff/epoch.rs` (BC-2.12.006):
   - Monotonically incrementing epoch counter per schedule, persisted to RocksDB
   - `EpochTracker::advance(id) -> u64`: atomically read-increment-write under RocksDB
     `merge_operator` (counter merge) to guarantee no duplicate epoch values
   - On executor completion: call `advance()` exactly once per successful execution
   - On executor error: do NOT advance epoch — idempotent retry semantics

4. Implement `get_diff_results` MCP tool handler in
   `crates/prism-operations/src/diff/get_diff_results.rs` (BC-2.12.007):
   - Parameters: `schedule_id`, `epoch_start: u64`, `epoch_end: u64`
   - Return: list of `DiffResult` records for the requested epoch range
   - Validate: `epoch_start <= epoch_end`, range <= 100 epochs per call
   - Return E-SCHED-002 if schedule not found; E-SCHED-003 if epoch range not available
     (already evicted)

5. Implement pack types and `.pack.toml` parser in
   `crates/prism-operations/src/pack/pack.rs` (BC-2.12.008):
   - `Pack { id: PackId, name: String, schedules: Vec<PackSchedule>,
     requires_capabilities: Vec<CapabilityId> }`
   - `PackSchedule { query: String, interval: Duration, label: String }`
   - TOML format: `[pack]` header, `requires = ["crowdstrike"]`, `[[schedules]]` entries
   - Pack loading scans configured pack directories for `*.pack.toml` files on startup
     and on hot reload signal

6. Implement conditional execution gating in pack executor (BC-2.12.008):
   - When a pack fires for a client, check `client.has_all_capabilities(pack.requires)`
   - If client lacks any required capability: skip pack execution for that client, log
     skip with reason
   - Pack schedules that pass capability check are submitted to the scheduler as regular
     `ScheduleEntry` records (pack is expanded at load time, not at runtime)

7. Implement pack CRUD tools in `crates/prism-operations/src/pack/pack_crud.rs`
   (BC-2.12.009):
   - `create_pack`: accept pack definition (name, schedules, required capabilities),
     persist to RocksDB `schedules` CF with `pack:{id}` key prefix; packs are global (not client-scoped)
   - `list_packs`: return all packs with schedule count and required capabilities
   - `delete_pack`: remove pack and its expanded schedule entries; no confirmation
     required (packs are configuration, not data)

8. Write VP-019 proptest in `crates/prism-operations/props/vp019_diff_determinism.rs`:
   - Property: for any set of rows, `compute_diff` output is identical regardless of
     the order in which rows appear in `curr`
   - Use `proptest` with randomly ordered `Vec<Row>` inputs

## Acceptance Criteria

**AC-1:** Given previous results `[A, B, C]` and current results `[B, C, D]`, When
`DiffEngine::compute_diff` runs, Then `added = [D]` and `removed = [A]`. (BC-2.12.005)

**AC-2:** Given previous results `[A, B, C]` and current results `[C, B, D]` (same rows,
different order), When `compute_diff` runs, Then output is identical to AC-1 —
order-independence is guaranteed. (BC-2.12.005)

**AC-3:** Given epoch 5 execution completes successfully, When `EpochTracker::advance` is
called, Then epoch advances to 6 and is durably persisted to RocksDB. (BC-2.12.006)

**AC-4:** Given epoch 5 execution fails with an error, When the executor error path runs,
Then the epoch remains at 5 (no advance) and a retry will re-execute at epoch 5.
(BC-2.12.006)

**AC-5:** Given `get_diff_results(schedule_id: "s1", epoch_start: 3, epoch_end: 5)`,
When called, Then diffs for epochs 3, 4, and 5 are returned. (BC-2.12.007)

**AC-6:** Given a pack requiring capability `"crowdstrike"`, When a client without that
capability is evaluated, Then the pack's schedules are skipped for that client with a
skip log entry. (BC-2.12.008)

**AC-7:** Given `create_pack` with two schedules, When `list_packs` is called, Then the
pack appears with `schedule_count: 2` and its required capabilities listed.
(BC-2.12.009)

**AC-8:** VP-019 proptest passes — `compute_diff` is deterministic and order-independent
for arbitrary row sets.

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Differential results engine | `prism-operations` (SS-12) | Mixed |
| Diff result persistence | `prism-storage` (SS-15) | Effectful |

`prism-operations` implements `DiffEngine` — compares current scheduled query results
against the previous snapshot stored in the `diff_results` RocksDB CF (AD-004). Set
difference computation is pure; RocksDB reads/writes and Arrow serialization are
effectful. Per `architecture/module-decomposition.md`, `DiffEngine` is called by
`Scheduler` after each query execution.

## Architecture Compliance Rules

- `DiffEngine::compute_diff` MUST be a pure function — no I/O, no side effects. Storage
  operations (reading/writing prev hash sets) are the caller's responsibility, not the
  engine's.
- Row hashing MUST normalize field order before hashing (canonical JSON). Hashing raw
  serialized bytes would produce different hashes for semantically equal rows with
  different field orderings.
- Epoch advance MUST use an atomic RocksDB merge operation, not a read-then-write
  sequence. A TOCTOU race on epoch increment would violate exactly-once semantics.
- Pack schedules are expanded into `ScheduleEntry` records at load/create time. The
  pack concept is a composition convenience, not a runtime dispatch mechanism.
- The 200MB cap on the `diff_results` CF is enforced at the RocksDB block cache layer,
  not via application-level byte counting. Use `set_block_cache` with the appropriate
  `Cache` size when opening the CF.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `compute_diff` called with empty `curr` (all rows removed) | `added = []`, `removed` = all hashes from prev; epoch advances on success |
| EC-002 | `compute_diff` called with empty `prev` (first execution) | `added` = all rows in `curr`, `removed = []`; epoch = 1 |
| EC-003 | `EpochTracker::advance` called concurrently by two executors | RocksDB merge operator serializes; exactly one increment per call; no duplicate epoch values |
| EC-004 | `get_diff_results` requested with `epoch_start > epoch_end` | Returns E-SCHED-002 (validation failure: invalid epoch range) |
| EC-005 | `get_diff_results` requested for evicted epoch range | Returns E-SCHED-003 (epoch range not available) |
| EC-006 | Pack `.pack.toml` references an unknown capability | Pack is rejected at load time with structured validation error; other packs load normally |
| EC-007 | `delete_pack` on a pack_id that does not exist | Returns structured not-found error; no-op on storage |
| EC-008 | Row with duplicate hash in `curr` (identical records) | Hash appears once in current set; no duplicate in `added`; determinism preserved |

## Dev Notes

- The 200MB cap for `diff_results` is informed by the osquery memory budget (200MB
  per-query) from the project memory budget. Differential state is bounded storage, not
  unbounded append.
- "Exactly-once" epoch semantics means: each successful execution increments the epoch
  exactly once, and a retry of a failed execution uses the same epoch number. This is
  "at-least-once execution with exactly-once epoch advancement on success."
- Packs are global (not client-scoped) because they represent query templates, not
  client data. The capability gating is what makes them client-aware at execution time.
- `delete_pack` does not require confirmation because packs are configuration artifacts,
  not operational data. A deleted pack can be recreated from its `.pack.toml` source.
  Contrast with `delete_schedule` in S-4.01 which requires confirmation for active
  (data-producing) schedules.
