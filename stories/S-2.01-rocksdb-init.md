# S-2.01 — prism-storage: RocksDB Initialization and Domain Operations

## Narrative

As a Prism platform implementer, I want a RocksDB-backed storage layer with 15 column
families, domain-scoped key-value operations, and crash-recovery dirty bits, so that all
platform subsystems have durable, domain-isolated persistence with safe restart semantics.

## Objective

Implement the prism-storage crate as the persistence foundation for the entire platform.
This story establishes the StorageBackend trait, the RocksDbBackend implementation with
all 16 column families, domain-scoped KV operations, and the dirty bit crash-recovery
protocol. An InMemoryBackend (BTreeMap-based) is also produced for unit tests.

## Tasks

1. Define `StorageBackend` trait in `crates/prism-storage/src/backend.rs`:
   - `get(domain: StorageDomain, key: &[u8]) -> Result<Option<Vec<u8>>, PrismError>`
   - `put(domain: StorageDomain, key: &[u8], value: &[u8]) -> Result<(), PrismError>`
   - `put_batch(domain: StorageDomain, entries: &[(&[u8], &[u8])]) -> Result<(), PrismError>`
   - `remove(domain: StorageDomain, key: &[u8]) -> Result<(), PrismError>`
   - `scan(domain: StorageDomain, prefix: &[u8]) -> Result<Vec<(Vec<u8>, Vec<u8>)>, PrismError>`
   - `scan_range(domain: StorageDomain, start: &[u8], end: &[u8]) -> Result<Vec<(Vec<u8>, Vec<u8>)>, PrismError>`
   - Trait must be `Send + Sync + 'static`

2. Implement `RocksDbBackend` in `crates/prism-storage/src/rocksdb_backend.rs`:
   - Open or create the database at `{state_dir}/prism.db`
   - Initialize exactly 16 column families on first open: `default`, `schedules`,
     `diff_results`, `detection_rules`, `detection_state`, `alerts`, `cases`,
     `audit_buffer`, `dirty_bits`, `watchdog`, `aliases`, `decorators`,
     `infusion_cache`, `action_state`, `plugin_state`, `event_buffer` (BC-2.15.001)
   - Store CF handles in a `HashMap<StorageDomain, Arc<ColumnFamily>>`
   - Map `StorageDomain` enum variants to CF handles via
     `StorageDomain::column_family_name()` (defined in S-1.01)

3. Configure RocksDB options in `RocksDbBackend::open()`:
   - Write buffer size: 64 MB per CF
   - Max open files: 256
   - Compression: LZ4 for all CFs
   - WAL enabled (default)
   - Bloom filters enabled for point lookups (10 bits per key)
   - `create_if_missing: true`, `create_missing_column_families: true`

4. Implement startup health check in `RocksDbBackend::health_check()`:
   - Verify DB opens without error
   - Verify all 16 CF handles are accessible
   - Perform a test write/read/delete cycle on the `default` CF
   - Return `Err(PrismError::StorageHealthCheckFailed)` on any failure

5. Implement corruption recovery in `RocksDbBackend::recover_or_exit()`:
   - On `DB::open()` failure indicating corruption, attempt `DB::repair()`
   - If repair fails, log the error and exit with code 3
   - If repair succeeds, retry `DB::open()` once

6. Implement schema version check in `RocksDbBackend::check_schema_version()`:
   - Sample up to 10 entries from the `default` CF
   - Validate presence of a version tag key `_schema_version`
   - If version tag is absent (fresh DB), write the current version
   - If version tag mismatches, return `Err(PrismError::SchemaMismatch)` (BC-2.15.001)

7. Implement domain-based KV operations in `RocksDbBackend` implementing `StorageBackend`:
   - Each operation resolves `StorageDomain` → CF handle via the stored map
   - Return `Err(PrismError::StorageDomainNotFound)` if the domain handle is missing
   - `scan` uses `prefix_iterator` with the given prefix bytes
   - `scan_range` uses `iterator_cf` with `IteratorMode::From(start, Direction::Forward)`
     filtered to `< end` (BC-2.15.002)

8. Implement `InMemoryBackend` in `crates/prism-storage/src/memory_backend.rs`:
   - `BTreeMap<(StorageDomain, Vec<u8>), Vec<u8>>` protected by `RwLock`
   - Implements the full `StorageBackend` trait
   - `scan` returns all entries whose key starts with the given prefix for the domain
   - Gated behind `#[cfg(any(test, feature = "test-utils"))]` so downstream crates
     (prism-audit, prism-operations, etc.) can use it in their tests via
     `prism-storage = { ..., features = ["test-utils"] }` as a dev-dependency

9. Implement dirty bit protocol in `crates/prism-storage/src/dirty_bits.rs`:
   - `set_dirty(query_id: &str)`: write `dirty_bits` CF entry with key=query_id,
     value=timestamp, using `WriteOptions { sync: true }` (survives OOM kill) (BC-2.15.005)
   - `clear_dirty(query_id: &str)`: delete the entry from `dirty_bits` CF
   - `check_dirty_on_startup() -> Vec<String>`: scan all entries in `dirty_bits` CF,
     return list of uncleared query IDs; caller logs warning and initiates recovery
   - Called at startup before accepting queries (BC-2.15.005)

10. Handle exclusive lock: if another process holds the RocksDB LOCK file,
    `DB::open()` returns an OS-level lock error; map this to
    `Err(PrismError::StorageLockHeld { path: state_dir.to_path_buf() })` (E-STORE-005)

## Acceptance Criteria

**AC-1:** Given a fresh state directory, When `RocksDbBackend::open()` is called, Then
all 16 column families exist and are accessible via their domain handles. (BC-2.15.001)

**AC-2:** Given an existing database with all 16 CFs, When `RocksDbBackend::open()` is
called again, Then the open succeeds and no duplicate CFs are created (idempotent).

**AC-3:** Given `StorageDomain::Alerts`, When `put(b"key1", value)` is called and then
`get(b"key1")` is called, Then the value round-trips correctly. (BC-2.15.002)

**AC-4:** Given `StorageDomain::Schedules` with keys `[b"tenant:acme:a", b"tenant:acme:b",
b"tenant:other:c"]`, When `scan(b"tenant:acme:")` is called, Then only the two `acme`
keys are returned. (BC-2.15.002)

**AC-5:** Given a query starting, When `set_dirty(query_id)` is called, Then the entry
is written to the `dirty_bits` CF with `sync: true`. (BC-2.15.005)

**AC-6:** Given a crash during a query (dirty bit was set, never cleared), When Prism
restarts and `check_dirty_on_startup()` runs, Then the uncleaned query_id is returned in
the list and a warning is logged. (BC-2.15.005)

**AC-7:** Given another Prism process holds the LOCK file, When a second process calls
`RocksDbBackend::open()`, Then `Err(PrismError::StorageLockHeld { .. })` is returned
(E-STORE-005).

## Architecture Compliance Rules

- `prism-storage` depends on `prism-core` and no other internal crates
- All backend types MUST be `Send + Sync` (required for async runtime use)
- `RocksDbBackend` MUST NOT be constructed outside of `prism-storage` — use the
  `StorageBackend` trait for all external references
- Dirty bit writes MUST use `WriteOptions { sync: true }` — this is the durability
  guarantee; omitting sync defeats the crash recovery protocol
- `InMemoryBackend` MUST be gated with `#[cfg(any(test, feature = "test-utils"))]` —
  downstream crates add `prism-storage/test-utils` as a dev-dependency feature
- CF initialization MUST be idempotent — `create_missing_column_families: true` handles
  subsequent opens safely

## Notes

- RocksDB CF handles cannot be safely cloned across threads; use `Arc<DB>` with the
  CF handle stored inside a `HashMap` under `Arc<RwLock<...>>` or use `DB::cf_handle()`
  at operation time.
- The `dirty_bits` CF intentionally uses `sync: true` writes only — this is expensive
  but correctness requires it. All other CFs use the default async write path.
- The `bincode` encoding version must be pinned to v2 (breaking change from v1 API);
  use `bincode::serde::encode_to_vec` and `decode_from_slice`.
- The 16 column families MUST exactly match `StorageDomain::all()` from prism-core;
  any mismatch will produce a schema mismatch error on startup.
