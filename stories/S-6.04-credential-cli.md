# S-6.04 — prism-bin: prism credential CLI Subcommand Group

## Narrative

As an MSSP operator, I want a `prism credential` CLI subcommand group that lets me
securely set, import, inspect, list, and rotate sensor API credentials without ever
exposing the values in terminal output, logs, or process arguments, so that credentials
remain opaque to any observer — including Claude and any AI model with access to the
terminal session.

## Objective

Implement the `prism credential` subcommand group in `prism-bin`, wiring it to the
`CredentialStore` backends established in S-1.06 and S-1.07. The subcommands are:
`set`, `import`, `status`, `list`, and `rotate`. All commands enforce the AD-017
AI-opaque credential invariant: credential values never appear in stdout, stderr, logs,
or error messages. Exit codes follow a structured convention: 0 success, 1 unexpected
error, 2 invalid credential reference, 3 backend unavailable.

1. Add `credential` to the clap `Commands` enum in `crates/prism-bin/src/main.rs`:
   ```rust

   enum Commands {
       Serve(ServeArgs),
       Init(InitArgs),
       Health(HealthArgs),
       Register(RegisterArgs),
       InstallSecopsfactory(InstallPackArgs),
       Credential(CredentialArgs),    // replaces placeholder from S-6.01 scope expansion
       MigrateStorage(MigrateArgs),   // replaced by S-6.05
       Logs(LogsArgs),                // Placeholder — dispatches to "not yet implemented"
   }
   ```

2. Define `CredentialArgs` in `crates/prism-bin/src/commands/credential/mod.rs`:
   ```rust
   #[derive(Args)]
   pub struct CredentialArgs {
       #[command(subcommand)]
       command: CredentialCommand,
   }

   #[derive(Subcommand)]
   enum CredentialCommand {
       Set(SetArgs),
       Import(ImportArgs),
       Status(StatusArgs),
       List(ListArgs),
       Rotate(RotateArgs),
   }
   ```

3. Implement `prism credential set` in `commands/credential/set.rs`:
   - Args:
     - `--client <ID>` (required)
     - `--sensor <NAME>` (required)
     - `--name <CREDENTIAL_NAME>` (required)
     - `--from-file <PATH>` (mutually exclusive with `--from-stdin` and interactive)
     - `--from-stdin` (mutually exclusive with `--from-file` and interactive)
     - `--from-env <VARNAME>` (read value from a named env var; mutually exclusive)
   - Interactive path (default when no source flag provided):
     - Use `rpassword::prompt_password("Enter value: ")` — does NOT echo to terminal
     - Prompt twice and verify match: "Confirm value: "
   - From-file path: read bytes from path, validate UTF-8, wrap in `SecretString`
   - From-stdin path: read from `stdin` without echo (check `atty::is(atty::Stream::Stdin)`;
     if stdin is a TTY, disable echo; if piped, read raw bytes) — never print the value
   - From-env path: call `std::env::var(varname)`, wrap in `SecretString`,
     unset the env var after reading (`std::env::remove_var`) to prevent child process
     leakage
   - Call `CredentialStore::set(client_id, sensor, name, secret_value)`
   - On success: print "Credential '{name}' set for {client}/{sensor}." — never print
     the value
   - On `PrismError::InvalidCredentialName`: exit code 2
   - On `PrismError::CredentialStoreError`: exit code 3

4. Implement `prism credential import --from-env` in `commands/credential/import.rs`:
   - Args:
     - `--from-env` (required flag)
     - `--client <ID>` (optional; if omitted, import for all clients in config)
     - `--pattern <REGEX>` (optional; default `PRISM_{CLIENT}_{SENSOR}_{NAME}`)
   - Scan environment variables matching the pattern (case-insensitive)
   - For each match: parse `CLIENT`, `SENSOR`, `NAME` from the var name, call
     `CredentialStore::set` with the value wrapped in `SecretString`
   - Remove each env var after reading: `std::env::remove_var(var_name)`
   - On completion: print a summary table — columns: sensor, name, result — where
     result is "set" or "error: <reason>". Values NEVER appear in the table.
   - On partial failure: exit code 1

5. Implement `prism credential status` in `commands/credential/status.rs`:
   - Args:
     - `--client <ID>` (required)
     - `--sensor <NAME>` (optional; if omitted, show status for all sensors of the client)
   - For each credential reference in config (resolved from prism.toml):
     - Call `CredentialStore::exists(client_id, sensor, name)` → bool
     - Determine status: `resolved` (exists=true), `missing` (exists=false)
     - If the credential is a `SecretString` typed field, also check if it's non-empty
   - Print a table: credential name | status | backend
   - Status values: `resolved`, `missing`, `expired` (if backend supports TTL — keyring
     does not, so `expired` only applies to Vault backend stubs)
   - NEVER print credential values, lengths, partial values, or hashes
   - Exit 0 if all resolved, exit 2 if any missing, exit 3 if backend unavailable

6. Implement `prism credential list` in `commands/credential/list.rs`:
   - Args:
     - `--client <ID>` (optional; if omitted, list all clients)
     - `--sensor <NAME>` (optional filter)
   - Call `CredentialStore::list(client_id)` → `Vec<(sensor, name)>` pairs
   - Print as a table: client | sensor | credential name
   - NEVER print values — only names (namespace keys)
   - Exit 0 always (empty list is a valid result, not an error)

7. Implement `prism credential rotate` in `commands/credential/rotate.rs`:
   - Args:
     - `--client <ID>` (required)
     - `--key <CREDENTIAL_NAME>` (required — the credential name to rotate)
     - `--sensor <NAME>` (required)
   - Interactive flow:
     1. Call `CredentialStore::exists(...)` — print warning if not found
     2. Prompt for new value using `rpassword::prompt_password` (no echo)
     3. Call `CredentialStore::set(...)` with new value (overwrites old)
     4. Print "Credential '{name}' rotated for {client}/{sensor}."
   - Note: keyring and encrypted-file backends do not support atomic swap; the
     implementation is: set new value → done. If set fails, old value may still be
     intact (no partial corruption).
   - Exit 0 on success, exit 2 on invalid name, exit 3 on backend error

8. Write unit tests in `crates/prism-bin/src/commands/credential/tests.rs`:
   - Test `set --from-env VARNAME`: verify the env var is removed after reading
   - Test `status` with a mock CredentialStore that returns `exists=false`: verify
     exit code is 2 and output contains "missing" not any value substring
   - Test `list` with a mock store returning 3 credentials: verify output contains 3
     rows with no value data
   - Test `set` with `--name ../../etc/passwd`: verify exit code is 2 (BC-2.03.008)
   - Test `import --from-env` with a matching env var: verify the var is unset after
     import and the summary prints "set" not the value

## Acceptance Criteria

**AC-1:** Given `prism credential set --client acme --sensor crowdstrike --name api_key`
run interactively, When the user enters a value at the hidden prompt, Then the value does
not appear in stdout, stderr, terminal history (no echo), or Prism logs. (BC-2.03.001,
AD-017)

**AC-2:** Given `prism credential set ... --from-stdin` with a piped value, When the
command runs, Then it reads from stdin without a TTY echo, sets the credential, and exits
0. The value never appears in output. (BC-2.03.006)

**AC-3:** Given `prism credential set ... --name "../../etc/passwd"`, When the command
runs, Then it exits 2 with an error message about invalid credential name — no store
operation is attempted. (BC-2.03.008)

**AC-4:** Given `prism credential status --client acme`, When credentials exist for 3
sensors and 1 is missing, Then the output table shows "resolved", "resolved", "resolved",
"missing" with no values shown, and the exit code is 2. (BC-2.03.001)

**AC-5:** Given `prism credential list --client acme`, When 5 credentials exist, Then
the output lists 5 rows with client, sensor, and name columns — no values column exists
or is shown. (BC-2.03.001)

**AC-6:** Given `prism credential import --from-env --client acme` with env var
`PRISM_ACME_CROWDSTRIKE_API_KEY=secret`, When the command runs, Then the credential is
stored, the env var is removed from the process environment, the summary prints "set"
for that entry, and the value does not appear in output. (BC-2.03.007)

**AC-7:** Given the credential backend is unavailable (e.g., encrypted-file path
unwritable), When any `prism credential` subcommand runs, Then the exit code is 3 and
an error message explains the backend is unavailable. (BC-2.03.011, BC-2.03.012)

**AC-8:** Given `prism credential rotate --client acme --sensor crowdstrike --key
api_key`, When the user enters a new value at the hidden prompt, Then the credential
is updated in the store and exit code is 0. (BC-2.03.009)

**AC-9:** Given the OS keyring service is available (macOS Keychain, Linux Secret
Service, or Windows Credential Vault) and `keyring` backend is configured, When
`prism credential set --client acme --sensor crowdstrike-prod --key api_token <value>`
runs, Then the value is stored in the platform keyring under namespace
`prism/acme/crowdstrike-prod/api_token` (as the service/account key), the OS keyring
encrypts the value (hardware-backed where available), and a subsequent `prism credential
status` confirms the credential is present without disk fallback.
(traces to BC-2.03.002 postcondition — OS keyring namespace and storage)

**AC-10:** Given the `encrypted_file` backend is configured and `PRISM_CREDENTIAL_KEY`
is set, When `prism credential set --client acme --sensor cyberint --key api_key
<value>` runs, Then the credential is stored at
`{credentials_dir}/acme/cyberint/api_key.enc` with file mode `0600` and parent
directories with mode `0700`. The file format contains a 32-byte salt, 12-byte nonce,
and ciphertext+tag. The derived encryption key is not stored on disk alongside the
encrypted file.
(traces to BC-2.03.003 postcondition — AES-256-GCM encrypted file layout and permissions)

**AC-11:** Given two clients `acme` and `globex` each with a CrowdStrike `api_key`
credential, When `prism credential list --client acme` is run, Then only `acme`'s
credentials appear in the output; `globex`'s credentials are absent. When `prism
credential list --client globex` is run, only `globex`'s credentials appear. The
3-tuple `(client_id, sensor_id, credential_name)` uniquely identifies each credential
and isolation is enforced at the storage layer.
(traces to BC-2.03.004 postcondition — namespace isolation by (client_id, sensor_id, credential_name))

**AC-12:** Given a new credential `(acme, crowdstrike, api_key)` does not yet exist,
When `prism credential set --client acme --sensor crowdstrike --key api_key <value>`
runs, Then the credential is created immediately with `status: "created"` and exit
code 0 (no confirmation token required for initial creation). Given the credential
already exists, When the same command runs to overwrite it, Then the CLI presents a
confirmation prompt or returns a `ConfirmationToken` flow before overwriting (per
BC-2.04.009 irreversible-write gating), and the overwrite only proceeds after confirmation.
Credential values are never included in MCP responses, logs, or error messages.
(traces to BC-2.03.005 postcondition — set creates immediately; overwrite requires confirmation)

**AC-13:** Given any `prism credential` subcommand runs (get, set, delete, list), When
the operation completes (success, not_found, or error), Then a `tracing::info!`
structured log entry is emitted with `event_type: "credential_access"`, `operation`
("get"|"set"|"delete"|"list"), `client_id`, `sensor_id`, `credential_name`, `backend`
("keyring"|"encrypted_file"), `result` ("success"|"not_found"|"error"), and `timestamp`
(UTC). The credential value is NEVER included in the log entry.
(traces to BC-2.03.010 postcondition — credential access audit logging)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Credential CLI subcommands | `prism-bin` | Effectful |
| Credential store operations | `prism-credentials` (SS-03) | Effectful |

`prism-bin` implements `prism credential` CLI subcommands (`add`, `remove`, `list`,
`rotate`) as an out-of-band management interface independent of the MCP server. These
subcommands directly call `prism-credentials` backends (keyring or encrypted file).
All operations are effectful (OS keychain, filesystem, stdin password prompts). Per
`architecture/module-decomposition.md`, `prism-bin` depends on `prism-credentials`.

## Architecture Compliance Rules

- Credential values MUST use `secrecy::SecretString` at ALL points in this crate —
  never unwrap to `String` for any output operation
- `rpassword` is the ONLY permitted method for interactive hidden input — do NOT use
  `std::io::stdin().read_line()` which echoes to terminal
- `std::env::remove_var` MUST be called after reading `--from-env` and `import
  --from-env` values — leaving env vars set allows child processes to inherit secrets
- Exit code convention is MANDATORY:
  - 0 = success
  - 1 = unexpected internal error
  - 2 = invalid credential reference (bad name, missing required arg)
  - 3 = backend unavailable
- `prism-bin` MUST NOT import `secrecy::ExposeSecret` directly — values are passed
  into the CredentialStore as `SecretString` and the store handles exposure internally
- Output tables MUST NOT include a "value" column, even showing partial or hashed values
