# S-5.03 — prism-mcp: Resources and Prompts

## Narrative

As a Claude Code AI agent orchestrating MSSP workflows, I want to access structured
resources (client list, sensor inventory, OCSF schema, health status) and pre-built
prompt templates for common security operations workflows, so that I can quickly orient
to the environment and guide analysts through repeatable investigation patterns.

## Objective

Implement MCP resources for client and sensor inventory, MCP prompts for common security
workflows, the `health_check` tool, and the `prism://health` resource. These give AI
agents a structured way to explore the Prism environment before issuing queries.

## Tasks

1. Implement MCP resource handlers in `crates/prism-mcp/src/resources.rs`:
   - `prism://clients`: return JSON array of all configured client IDs, each with
     `{ client_id, sensor_count, enabled_sensors: [String] }` (BC-2.10.008)
   - `prism://sensors/{client_id}`: validate `client_id` via `TenantId::new()`, then
     return JSON array of sensor configs for that client: `{ sensor_type, status,
     credential_ref, sources: [String] }` (BC-2.10.008)
   - `prism://schema/{sensor}/{source}`: return the OCSF schema definition for a given
     sensor + source combination, sourced from the spec engine (BC-2.10.008)
   - Register all three resource patterns with rmcp's resource router

2. Implement MCP prompts in `crates/prism-mcp/src/prompts.rs`:
   - `investigate_alert`: structured workflow for alert investigation — query affected
     assets, correlate with vulnerability data, check lateral movement indicators
   - `triage_case`: step-by-step case triage including severity assessment, asset
     enumeration, and initial containment checklist
   - `vulnerability_assessment`: workflow for querying CVE exposure across sensor data,
     grouping by asset criticality
   - `coverage_gap_analysis`: workflow to identify clients/sensors with missing data
     or degraded coverage using the health resource
   - `incident_response`: full IR workflow from alert triage through case creation and
     escalation
   - Each prompt accepts template variables (e.g., `{{ client_id }}`, `{{ alert_id }}`)
     and returns a rendered string (BC-2.10.009)

3. Implement `health_check` tool in `crates/prism-mcp/src/tools/health.rs`:
   - Input: `{ client_id: String, sensors: Option<Vec<String>> }` — if `sensors` is
     omitted, check all sensors for the client
   - For each sensor: exercise the actual API endpoint (a lightweight list or version
     call), not a synthetic ping
   - Output: `{ client_id, checked_at: DateTime<Utc>, sensors: [SensorHealthResult] }`
   - `SensorHealthResult`: `{ sensor_type, status: up|down|degraded|auth_invalid,
     latency_ms: u64, error: Option<String> }` (BC-2.08.005)

4. Implement `prism://health` resource in `crates/prism-mcp/src/resources.rs`:
   - Return cached health data from the last `health_check` invocation per client
   - If no health check has been run: return `{ status: "unknown", message: "Run
     health_check to populate this resource" }`
   - Cache is per-client, held in `PrismContext` (BC-2.08.006)

5. Write tests in `crates/prism-mcp/tests/resources.rs`:
   - Test `prism://clients` returns all configured clients with correct sensor counts
   - Test `prism://sensors/{client_id}` with invalid client_id returns 404-equivalent
   - Test `prism://health` returns "unknown" before any health_check is run
   - Test `health_check` with a mock sensor returns structured per-sensor result
   - Test each prompt can be rendered with template variables substituted

## Acceptance Criteria

**AC-1:** Given a Prism instance configured with two clients ("acme", "globex"), When
`prism://clients` is read, Then the response is a JSON array containing both clients,
each with `sensor_count` and `enabled_sensors` populated.
(traces to BC-2.10.008 postcondition 1)

**AC-2:** Given `prism://sensors/acme` for a client with CrowdStrike and Claroty sensors,
When read, Then the response lists both sensors with their `sensor_type`, `status`, and
`credential_ref`.
(traces to BC-2.10.008 postcondition 2)

**AC-3:** Given the `investigate_alert` prompt with `client_id: "acme"` and
`alert_id: "ALT-001"`, When rendered, Then the output is a structured multi-step
investigation template with both variables substituted.
(traces to BC-2.10.009 postcondition 1)

**AC-4:** Given `health_check(client_id: "acme")` with a reachable mock CrowdStrike
sensor, When called, Then the response includes a `SensorHealthResult` for CrowdStrike
with `status: "up"` and a `latency_ms` value.
(traces to BC-2.08.005 postcondition 1)

**AC-5:** Given a successful `health_check` run at 10:30, When `prism://health` is
subsequently read, Then the response contains the cached results from 10:30 including
per-sensor statuses.
(traces to BC-2.08.006 postcondition 1)

**AC-6:** Given `prism://health` is read before any `health_check` has been run, When
read, Then the response contains `status: "unknown"` and an instructional message, not
an error.
(traces to BC-2.08.006 postcondition 2)

## Architecture Compliance Rules

- Resource URIs MUST follow the `prism://` scheme — do not use HTTP URLs or bare paths
- The `prism://schema/{sensor}/{source}` resource MUST source its data from the spec
  engine (S-1.11), not from hardcoded schema definitions in prism-mcp
- The `health_check` tool MUST perform actual API calls — it MUST NOT return cached
  data from `prism://health` (the resource caches the tool's output, not vice versa)
- Prompt templates MUST be stored as static strings in `prompts.rs`, not as files on
  disk — this avoids filesystem path dependencies in the MCP server
- `SensorHealthResult.error` MUST be sanitized (prompt injection defense) before
  inclusion in the response

## Notes

- Prompts are distinct from tools in MCP: they are retrieved via `prompts/get` and are
  intended as starting templates for user workflows, not executable operations. Keep
  prompt text concise (under 500 tokens each) to avoid overwhelming the AI context.
- The `coverage_gap_analysis` prompt is particularly valuable for MSSP use cases — it
  should guide the AI to cross-reference sensor health data with expected coverage
  requirements.
- Health cache should have a TTL (suggest 5 minutes) to avoid staleness. After TTL
  expiry, `prism://health` should return the cached data with a `stale: true` flag
  rather than clearing it.
