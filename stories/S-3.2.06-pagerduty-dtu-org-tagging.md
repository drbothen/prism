# S-3.2.06 — prism-dtu-pagerduty: Shared-mode OrgId ingress tagging

## Narrative

As a Prism MSSP operator, I want the PagerDuty DTU clone to embed the `OrgId` UUID
string in the `custom_details` field of every incident payload it captures so that
incidents from different client organizations can be attributed to their source without
routing org identity through PagerDuty dedup keys, URLs, or headers.

## Objective

Apply the same shared-mode OrgId tagging pattern as S-3.2.05 to `prism-dtu-pagerduty`.
The `incident_registry` state store is NOT re-keyed — it remains keyed by PagerDuty
`dedup_key` (MSSP-scoped per ADR-008 §1.2). The `dedup_key` remains MSSP-generated
(not org-scoped). Add `DtuMode::Shared` metadata. Validate mode immutability per
BC-3.2.005.

## Tasks

1. **Add OrgId to incident capture method** (`src/state.rs`):
   Update `incident_registry` insert to embed `org_id` in the stored `IncidentRecord`
   JSON or as a dedicated field:
   ```rust
   pub fn capture_incident(&self, org_id: OrgId, dedup_key: String, payload: Value) {
       let record = IncidentRecord {
           dedup_key: dedup_key.clone(),
           org_id: org_id.to_string(),  // UUID form
           payload,
       };
       let mut registry = self.incident_registry.lock().expect("poisoned");
       registry.insert(dedup_key, record);
   }
   ```

2. **Verify `dedup_key` is MSSP-scoped, NOT org-scoped**:
   The `dedup_key` for PagerDuty deduplication is MSSP-generated (e.g., a UUID based
   on the MSSP case ID). It must NOT embed the org_id. The `org_id` appears only in
   `custom_details` or the `IncidentRecord` metadata field. Add a unit test asserting
   this separation.

3. **Thread OrgId through route handlers** (`src/routes/`):
   Route handlers that call `capture_incident` must extract `OrgId` from request
   state/extensions and pass it to the state method.

4. **Add DtuMode::Shared metadata** — same approach as S-3.2.05:
   Confirm `DtuMode` is available from `prism-dtu-common` (done in S-3.2.05).
   Set `DTU_MODE: DtuMode = DtuMode::Shared` in the crate.

5. **Add `tests/org_tagging.rs`**:
   - `test_org_id_in_custom_details` — capture incident for org_A; inspect stored
     `IncidentRecord`; assert `org_id == uuid-A`.
   - `test_dedup_key_not_org_scoped` — assert dedup_key contains no org_id UUID.
   - `test_concurrent_incidents_distinguished` — org_A and org_B capture incidents;
     each record contains its sender's OrgId.
   - `test_mode_immutable_after_startup` — DtuMode::Shared cannot be changed at runtime.

6. **Gate**: `cargo test -p prism-dtu-pagerduty` green.

## Acceptance Criteria

**AC-001** — OrgId in `custom_details` field (traces to BC-3.2.004 postcondition 1):
Given org_A sends an incident to the shared PagerDuty DTU,
When the captured `IncidentRecord` is inspected,
Then the record contains `org_id: "<uuid-A>"` in its metadata or `custom_details` field.

**AC-002** — OrgId absent from dedup_key and HTTP routing (traces to BC-3.2.004 postcondition 2):
Given any PagerDuty incident payload,
When the `dedup_key`, URL, and headers are inspected,
Then none of them contain the `org_id` UUID or org slug.

**AC-003** — Concurrent incidents from different orgs (traces to BC-3.2.004 postcondition 4):
Given org_A and org_B both capture incidents concurrently,
When both `IncidentRecord`s are inspected,
Then each contains its sender's OrgId UUID independently.

**AC-004** — Mode metadata absent from query results (traces to BC-3.2.004 postcondition 5):
Given an analyst queries incident data from this shared DTU,
When result rows are inspected,
Then no `"mode"`, `"shared"`, or `"dtu_mode"` fields appear in result rows.

**AC-005** — DtuMode::Shared set at startup and immutable (traces to BC-3.2.005 postcondition 1, invariant 1):
Given config TOML has `mode = "shared"` for the PagerDuty DTU,
When the process starts and runs,
Then `DtuMode::Shared` is the mode for the process lifetime; no in-process API changes it.

**AC-006** — Invalid mode string rejected (traces to BC-3.2.005 postcondition 3):
Given config TOML has `mode = "SHared"` (wrong case),
When the process attempts to start,
Then serde deserialization fails with a human-readable error; the process does not start.

## Architecture Compliance Rules

- `OrgId` UUID string MUST appear in `custom_details` or `IncidentRecord.org_id` field — NOT in `dedup_key`, URL, or headers.
- `dedup_key` MUST remain MSSP-scoped (not org-scoped). This is confirmed by `test_dedup_key_not_org_scoped`.
- `incident_registry` (`HashMap<String, IncidentRecord>`) is NOT re-keyed by OrgId — it remains bare-String keyed per ADR-008 §1.2.
- **Forbidden dependencies:** `prism-dtu-pagerduty` MUST NOT depend on `prism-sensors`, `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`.
