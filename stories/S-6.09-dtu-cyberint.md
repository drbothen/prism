# S-6.09 — prism-dtu-cyberint: DTU for Cyberint API — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Cyberint API at
fidelity level L2 (stateful), so that integration tests for `cyberint_alerts`,
`cyberint_threats`, cookie-based auth (`CookieRoundtrip`), and alert status write paths
can run deterministically without real Cyberint credentials.

## Objective

Implement `prism-dtu-cyberint` — an L2 stateful behavioral clone of the Cyberint API.
The clone implements all 6 in-scope endpoints plus the cookie auth flow, maintains a
stateful alert status store and session cookie registry, and exercises Prism's unique
`CookieRoundtrip` auth handler in integration tests. See `dtu-assessment.md §3.3` for
full scope matrix.

Note on irreversible writes: the `close_alert` write (Irreversible) is included in
scope. The DTU models this by transitioning alert status to `"closed"` with no undo
within a session — matching the real API behavior. `reset()` is the only way to restore
the fixture state for test isolation.

## Tasks

1. Create `crates/prism-dtu-cyberint/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `uuid` (1.x with v4 feature)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `CyberintClone` struct implementing `BehavioralClone`:
   - Fields: `config: StubConfig`, `state: Arc<CyberintState>`,
     `server_handle: Option<JoinHandle<()>>`, `bound_addr: Option<SocketAddr>`
   - `start()`: binds ephemeral port, spawns axum router
   - `reset()`: calls `state.reset()`

3. Create `src/state.rs` — `CyberintState`:
   - `alert_store: Mutex<HashMap<String, AlertStatus>>` —
     `alert_id → AlertStatus { status: String, closed: bool }`
   - `session_store: Mutex<HashSet<String>>` — valid session tokens (UUID strings)
   - `reset()` clears both stores; pre-populates `alert_store` from fixture IDs with
     status `"open"` and `closed: false`

4. Create `src/routes/auth.rs`:
   - `POST /login`: accepts any body; generates UUID session token; inserts into
     `session_store`; returns `Set-Cookie: cyberint_session={uuid}; Path=/; HttpOnly`
   - Response: 200 `{"message": "Login successful"}`
   - Session token is any UUID — no cryptographic validation in DTU

5. Cookie auth middleware:
   - All routes except `POST /login` and `POST /dtu/*` must validate the `Cookie` header
   - If `Cookie` header is absent or empty: return 401 `{"error": "unauthorized", "code": 401}`
   - Implementation: axum middleware extractor or route-level guard

6. Create `src/routes/alerts.rs`:
   - `GET /api/v1/alerts` or `POST /api/v1/alerts`: alert list with cursor pagination
     - Query/body param: `cursor` (string); response includes `next_cursor` or null
     - Loads fixture from `fixtures/alerts.json`
     - Merges alert status from `alert_store`
   - `GET /api/v1/alerts/{alert_id}`: alert detail
     - Returns single alert object with current status from `alert_store`
     - 404 if `alert_id` not in store
   - `PATCH /api/v1/alerts/{alert_id}/status`: acknowledge alert
     - Body: `{"status": "acknowledged"}`
     - Updates `alert_store[alert_id].status = "acknowledged"`
     - Returns 200 `{"alert_id": "...", "status": "acknowledged"}`
   - `POST /api/v1/alerts/{alert_id}/close`: close alert (Irreversible in session)
     - Updates `alert_store[alert_id] = AlertStatus { status: "closed", closed: true }`
     - Returns 200 `{"alert_id": "...", "status": "closed"}`
     - Subsequent PATCH acknowledge on a closed alert returns 400
       `{"error": "alert already closed"}`

7. Create `src/routes/threats.rs`:
   - `GET /api/v1/threat-intel`: threat intelligence feed
     - Loads fixture from `fixtures/threats.json`
     - Cursor pagination: same pattern as alerts
     - Response: `{"data": [...], "next_cursor": "..." or null}`

8. Create fixture files:
   - `fixtures/alerts.json`: 20 alert objects with fields: `alert_id`, `title`,
     `severity` (`critical/high/medium/low`), `status` (`"open"`), `created_at`
     (using multiple timestamp formats — some ISO 8601, some Unix epoch — to exercise
     Prism's timestamp normalization), `source`, `type`, `affected_assets`
   - `fixtures/threats.json`: 15 threat intelligence items: `id`, `title`, `type`,
     `severity`, `confidence`, `iocs` (list of `{type, value}`)
   - Note: the multi-format timestamp fixtures are the key fidelity element for Cyberint
     (documented in `dtu-assessment.md §3.3` error simulation — Cyberint uses
     inconsistent timestamp formats)
   - `fixtures/alerts-page2.json`: 5 additional alerts for pagination testing (cursor
     simulation)

9. Create `POST /dtu/configure` and `POST /dtu/reset`:
   - `configure`: accepts `{"auth_mode": "reject"}` or `{"rate_limit_after": N}`
   - `reset`: calls `state.reset()` and resets FailureLayer counters

10. Create `tests/fidelity.rs`:
    - `#[tokio::test] fn cyberint_dtu_fidelity()`: starts `CyberintClone`, runs
      `FidelityValidator` against all 6 endpoints + auth flow
    - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /login` with any body, Then the response is HTTP 200 with a
`Set-Cookie` header containing a session token; subsequent requests that include this
cookie receive HTTP 200 (cookie auth round-trip works).

**AC-2:** Given any request to `/api/v1/alerts` without a `Cookie` header, Then the
response is HTTP 401 `{"error": "unauthorized", "code": 401}` — exercising Prism's
`E-AUTH-004` path.

**AC-3:** Given `PATCH /api/v1/alerts/{alert_id}/status` with `{"status": "acknowledged"}`,
Then the response is HTTP 200 AND subsequent `GET /api/v1/alerts/{alert_id}` returns the
alert with `status: "acknowledged"` (stateful transition persists).

**AC-4:** Given `POST /api/v1/alerts/{alert_id}/close`, Then the response is HTTP 200
with `status: "closed"` AND subsequent `PATCH` for acknowledge returns HTTP 400
`{"error": "alert already closed"}` (irreversible close enforced).

**AC-5:** Given `GET /api/v1/alerts` returns fixtures with mixed timestamp formats
(ISO 8601 and Unix epoch), Then these formats are present in `fixtures/alerts.json`
and the DTU returns them verbatim — exercising Prism's timestamp normalization.

**AC-6:** Given cursor pagination: `GET /api/v1/alerts` without cursor returns first
page with `next_cursor` set, Then `GET /api/v1/alerts?cursor={next_cursor}` returns
the second page with `next_cursor: null`.

**AC-7:** Given `FailureMode::RateLimit` configured via `POST /dtu/configure`, When
the threshold is exceeded, Then the response is HTTP 429 — maps to `E-SENSOR-003`.

**AC-8:** Given `reset()` is called, Then all alert statuses revert to `"open"`,
session store is cleared, and a new login is required for subsequent authenticated
requests.

## Architecture Compliance Rules

- DTU clone MUST match the Cyberint API contract exactly (same endpoints, same
  request/response shapes, same status codes) as documented in `dtu-assessment.md §3.3`.
- DTU clone MUST NOT introduce new behaviors not present in the real Cyberint API.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution — all state is in-process.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — never compiled
  into production binaries.
- The cookie auth middleware (`CookieRoundtrip`) MUST validate the session token from the
  `Cookie` header before any authenticated route executes; missing cookie returns 401.
- See `.factory/specs/architecture/dtu-assessment.md` for cross-cutting DTU rules including
  ephemeral port binding and `BehavioralClone` contract.
- **Forbidden dependencies:** `prism-dtu-cyberint` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `close_alert` on an already-closed alert | HTTP 400 `{"error": "alert already closed"}` — idempotent check |
| EC-002 | `GET /api/v1/alerts/{alert_id}` for unknown alert_id | HTTP 404 `{"error": "alert not found"}` |
| EC-003 | `POST /login` called twice | Both succeed; each returns a new session token; both tokens valid until `reset()` |
| EC-004 | Cursor pagination with invalid cursor value | DTU returns first page (cursor not found → start from beginning) |
| EC-005 | Out-of-scope endpoint `/api/v1/digital-risk/findings` requested | HTTP 404 `{"error": "endpoint not found"}` |
| EC-006 | Auth failure simulation via `POST /dtu/configure {"auth_mode": "reject"}` | All subsequent authenticated requests return 401 regardless of cookie |
