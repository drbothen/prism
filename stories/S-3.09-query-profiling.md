# S-3.09 — prism-query: Query Performance Profiling

## Narrative

As a Prism analyst or operator, I want per-query performance metrics (API latency,
rows fetched, rate limit state, memory usage, push-down effectiveness) included in
every query response and surfaced in `get_diagnostics` so that slow queries can be
diagnosed without enabling trace-level logging or guessing which sensor is the
bottleneck.

## Objective

Define a `QueryMetrics` struct, instrument `QueryEngine::execute()` to populate it,
include it in the query response `_meta` envelope, feed it into `get_diagnostics
(subsystem: "fanout")`, and maintain a ring buffer of the last 100 query metric
snapshots for aggregate diagnostics. Also feed `QueryMetrics` into `explain_query`
as estimated vs. actual comparison.

## Tasks

1. Define `QueryMetrics` in `crates/prism-query/src/metrics.rs`:
   ```rust
   pub struct QueryMetrics {
       pub trace_id: String,
       pub wall_time_ms: u64,
       pub parse_time_ms: u64,
       pub plan_time_ms: u64,
       pub fan_out_time_ms: u64,
       pub normalize_time_ms: u64,
       pub datafusion_time_ms: u64,
       pub per_source_latency_ms: HashMap<String, u64>,
                                  // key: "{sensor}/{client}"
       pub total_rows_fetched: u64,       // before DataFusion filtering
       pub total_rows_returned: u64,      // after DataFusion filtering
       pub memory_peak_bytes: u64,        // from GreedyMemoryPool high-water mark
       pub cache_hits: u32,               // in-query dedup cache hits (S-3.11)
       pub cache_misses: u32,
       pub push_down_predicates_count: u32,
       pub post_filter_predicates_count: u32,
       pub sensors_queried: Vec<String>,
       pub sensors_errored: Vec<String>,
       pub rate_limit_remaining: HashMap<String, Option<i64>>,
                                  // key: "{sensor}/{client}"; None = header absent
   }
   ```
   Derive `serde::Serialize` for JSON inclusion in `_meta`.

2. Instrument `QueryEngine::execute()` in `engine.rs`:
   - Use `std::time::Instant` checkpoints at each phase boundary (parse, plan,
     fan-out, normalize, DataFusion execute)
   - Per-source latency: record `Instant::now()` before each `SensorAdapter::fetch()`
     task and compute elapsed on completion; key as `"{sensor_id}/{client_id}"`
   - `total_rows_fetched`: sum of `RecordBatch.num_rows()` across all fan-out results
     before DataFusion filtering
   - `total_rows_returned`: `RecordBatch.num_rows()` of final DataFusion output
   - `memory_peak_bytes`: read from `GreedyMemoryPool` via DataFusion's
     `MemoryPool::reserved()` at the end of execution (after `collect()`)
   - `cache_hits`/`cache_misses`: populated by in-query dedup cache (S-3.11);
     default 0 until S-3.11 is implemented
   - `push_down_predicates_count` / `post_filter_predicates_count`: populated by
     the query planner from `QueryPlan.push_down` and `QueryPlan.post_filter` lengths

3. Include `QueryMetrics` in query response `_meta` envelope:
   ```json
   {
     "_meta": {
       "trace_id": "01905a7b...",
       "total_rows": 47,
       "sensor_errors": [],
       "metrics": {
         "wall_time_ms": 312,
         "per_source_latency_ms": { "crowdstrike/acme": 234, "armis/acme": 156 },
         "total_rows_fetched": 200,
         "total_rows_returned": 47,
         "memory_peak_bytes": 8388608,
         "push_down_predicates_count": 2,
         "post_filter_predicates_count": 1
       }
     }
   }
   ```

4. Maintain a ring buffer of recent query metrics in `ring_buffer.rs`:
   - `QueryMetricsBuffer`: fixed-size circular buffer, capacity 100
   - Thread-safe: wrap in `Arc<Mutex<QueryMetricsBuffer>>`; stored in `QueryEngine`
   - After each `execute()` or `execute_scheduled()`, push the completed
     `QueryMetrics` into the buffer (do NOT block on lock; skip if lock is contended —
     diagnostics are best-effort, not a correctness requirement)
   - Expose `snapshot() -> Vec<QueryMetrics>` for `get_diagnostics` consumers

5. Feed `QueryMetrics` into `get_diagnostics(subsystem: "fanout")`:
   - Add `"recent_query_metrics"` field to the fanout diagnostic response
   - Include: last 10 queries with their `trace_id`, `wall_time_ms`,
     `per_source_latency_ms`, `total_rows_fetched`, `total_rows_returned`
   - Aggregate stats: `avg_wall_time_ms`, `p50_wall_time_ms`, `p95_wall_time_ms`
     computed from the ring buffer

6. Feed `QueryMetrics` into `explain_query` output (S-3.03):
   - When `explain_query` is called with `include_actuals: true` and a `trace_id`:
     look up the trace_id in the ring buffer and include the actual metrics alongside
     the estimated plan
   - If trace_id not found in buffer: include only estimated plan (existing behavior)

## Acceptance Criteria

**AC-1:** Given a PrismQL query that fans out to 2 sensors × 1 client, When the query
completes, Then `_meta.metrics` in the response contains `per_source_latency_ms` with
2 entries, each a non-negative integer.

**AC-2:** Given a query that fetches 200 rows from the sensor API but DataFusion
filters down to 47 rows, When metrics are inspected, Then `total_rows_fetched == 200`
and `total_rows_returned == 47`.

**AC-3:** Given a query with 2 push-down predicates and 1 post-filter predicate, When
metrics are inspected, Then `push_down_predicates_count == 2` and
`post_filter_predicates_count == 1`.

**AC-4:** Given `get_diagnostics(subsystem: "fanout")` is called after 5 queries,
When the response is returned, Then `recent_query_metrics` contains up to 5 entries
with `trace_id`, `wall_time_ms`, and `per_source_latency_ms` for each.

**AC-5:** Given 101 queries have completed, When the ring buffer is inspected, Then
it contains exactly 100 entries (oldest dropped — not 101).

**AC-6:** Given `explain_query(query: "...", include_actuals: true, trace_id:
"01905a7b...")` where that trace_id exists in the ring buffer, When the response is
returned, Then actual metrics are included alongside the estimated plan.

**AC-7:** Given all three PrismQL modes (filter, SQL, pipe), When each executes a
query, Then `QueryMetrics` is populated identically — the profiling is mode-agnostic.

## Architecture Compliance Rules

- `QueryMetrics` collection MUST NOT add blocking I/O to the query execution hot path.
  Use `Instant::now()` (nanosecond resolution, no syscall beyond OS timer) for all
  timing. The ring buffer push is behind a non-blocking mutex (`try_lock`).
- `memory_peak_bytes` is read from DataFusion's `MemoryPool::reserved()` — this is
  the DataFusion-tracked allocation, not process RSS. It complements (but does not
  replace) the watchdog's RSS monitoring (S-2.02).
- `per_source_latency_ms` keys MUST use the format `"{sensor_id}/{client_id}"` —
  consistent with the format used in `get_diagnostics(subsystem: "fanout")` so
  that consumers can correlate individual queries with aggregate sensor stats.
- `QueryMetrics` MUST be serialized via `serde::Serialize` — do NOT hand-craft JSON.
  This ensures the schema stays consistent as fields are added.
- Ring buffer lock contention: if `try_lock()` fails, skip the push and increment a
  `metrics_dropped_count` atomic counter. Expose this in `get_diagnostics` so
  operators know if metrics are being lost under high query concurrency.

## Previous Story Intelligence

S-3.02 implements `QueryEngine::execute()` and establishes the `_meta` envelope
structure. The `metrics` field is a new top-level key added to the existing `_meta`
object — do not restructure `_meta`; only add the new key.

S-3.03 implements `explain_query`. The `include_actuals` enhancement in this story
requires adding a `trace_id` parameter to `explain_query` — coordinate with S-3.03
to ensure the parameter is accepted (and ignored gracefully if the ring buffer has
no matching entry).

S-2.02 implements the watchdog and RSS monitoring. `memory_peak_bytes` in
`QueryMetrics` is a complement — DataFusion's pool tracks internal allocations that
the RSS watchdog may not fully attribute to a single query.

## Notes

- The 100-query ring buffer is sized to cover a typical 5-minute window at 1 query
  per 3 seconds (the high end for ad-hoc usage). Scheduled queries run less frequently
  and are also captured. This is not a persistent metrics store — it is a rolling
  debug window.
- `_query_cost_ms` (the hidden column from S-3.08) should be populated from
  `per_source_latency_ms` proportionally per-row: `per_source_latency_ms[source] /
  rows_from_source`. This requires S-3.08 and S-3.09 to be wired together; if S-3.08
  runs first, populate `_query_cost_ms` as NULL until S-3.09 is implemented.
- `cache_hits` and `cache_misses` are 0 until S-3.11 (in-query dedup caching)
  is implemented. Leave the fields in the struct from day one so that S-3.11 only
  needs to wire up the counters, not change the struct.
