# S-3.09 — prism-query: Query Performance Profiling

## Narrative

As a Prism analyst or operator, I want per-query performance metrics (API latency,
rows fetched, rate limit state, memory usage, push-down effectiveness) included in
every query response and surfaced in `get_diagnostics` so that slow queries can be
diagnosed without enabling trace-level logging or guessing which sensor is the
bottleneck.

## Objective

Define a `QueryMetrics` struct, instrument `QueryEngine::execute()` to populate it,
include it in the query response `_meta` envelope, feed it into `get_diagnostics
(subsystem: "fanout")`, and maintain a ring buffer of the last 100 query metric
snapshots for aggregate diagnostics. Also feed `QueryMetrics` into `explain_query`
as estimated vs. actual comparison.

## Tasks

### Phase A — Materialization Pipeline Build-Out (S-3.02 deferred body)

A1. Parse phase: invoke `prism_parser::parse(query_str)` (BC-2.11.001) -> returns `Ast`. On parse error, return `PrismError::QueryParseError` (per existing taxonomy).

A2. Plan phase: build `QueryPlan` from AST (BC-2.11.011 push-down planner / BC-2.11.012 post-filter classification) — separate predicates into `push_down: Vec<Predicate>` (sensor-API-evaluable) and `post_filter: Vec<Predicate>` (DataFusion-evaluable). Use existing `crate::planner` module (or create one).

A3. Fan-out phase: for each `(sensor_id, client_id)` pair in scoping, dispatch `SensorAdapter::fetch(scoping, push_down)` concurrently via `tokio::spawn` + `JoinSet`. Apply 30s timeout per BC-2.11.006 via `tokio::time::timeout`. Collect `Vec<RecordBatch>` per source.

A4. Normalize phase: apply OCSF normalization + virtual field injection (`_client`, `_sensor`, `_query_cost_ms` placeholder for S-3.08) to each RecordBatch.

A5. DataFusion phase:
   - Construct ephemeral `SessionContext` with `GreedyMemoryPool` (200MB cap per BC-2.11.006)
   - Register normalized RecordBatches as DataFusion tables (table name = sensor_id)
   - Apply post-filter predicates via SQL or DataFrame API
   - Apply 10K record cap per BC-2.11.006 (LIMIT clause or stream truncation)
   - Call `.collect()` to materialize results
   - Drop SessionContext after collection (BC-2.11.005 ephemerality)

A6. Result envelope: assemble `QueryResult { batches, total_available, is_truncated, returned_results, context, sensor_errors, metrics }` — note `metrics` field is NEW; add to struct in this story.

A7. `execute_scheduled()`: same materialization pipeline as `execute()` but DOES NOT drop the SessionContext — returns `Arc<SessionContext>` to caller for detection-engine reuse (S-4.03). All other phases identical.

## Acceptance Criteria

**AC-1:** Given a PrismQL query that fans out to 2 sensors × 1 client, When the query
completes, Then `_meta.metrics` in the response contains `per_source_latency_ms` with
2 entries, each a non-negative integer.

**AC-2:** Given a query that fetches 200 rows from the sensor API but DataFusion
filters down to 47 rows, When metrics are inspected, Then `total_rows_fetched == 200`
and `total_rows_returned == 47`.

**AC-3:** Given a query with 2 push-down predicates and 1 post-filter predicate, When
metrics are inspected, Then `push_down_predicates_count == 2` and
`post_filter_predicates_count == 1`.

**AC-4:** Given `get_diagnostics(subsystem: "fanout")` is called after 5 queries,
When the response is returned, Then `recent_query_metrics` contains up to 5 entries
with `trace_id`, `wall_time_ms`, and `per_source_latency_ms` for each.

**AC-5:** Given 101 queries have completed, When the ring buffer is inspected, Then
it contains exactly 100 entries (oldest dropped — not 101).

**AC-6:** Given `explain_query(query: "...", include_actuals: true, trace_id:
"01905a7b...")` where that trace_id exists in the ring buffer, When the response is
returned, Then actual metrics are included alongside the estimated plan.

**AC-7:** Given all three PrismQL modes (filter, SQL, pipe), When each executes a
query, Then `QueryMetrics` is populated identically — the profiling is mode-agnostic.

**AC-8 (BC-2.11.001):** Given a valid PrismQL query string and scoping, When `engine.execute(query, opts)` is called, Then it returns `Ok(QueryResult)` with non-empty `batches`. Reference test: `tests/end_to_end_tests.rs::test_S_3_09_AC_8_execute_returns_ok_for_valid_query`.

**AC-9 (BC-2.11.005):** Given `engine.execute()` completes, When the function returns, Then no `Arc<SessionContext>` is held by `QueryEngine` — the SessionContext was dropped within the call. Reference test: probe `Arc::strong_count` on a session captured via test harness.

**AC-10 (BC-2.11.006 — timeout):** Given a sensor that hangs longer than 30s, When `engine.execute()` is called, Then it returns `Err(PrismError::QueryTimeout)` after ~30s. Reference test using `tokio::time::pause()` + mock adapter.

**AC-11 (BC-2.11.006 — record cap):** Given a query whose results would exceed 10K rows, When `engine.execute()` is called, Then `QueryResult.is_truncated == true` and `batches` total <= 10K rows.

**AC-12 (BC-2.11.011/012):** Given a query with predicates that are partially push-downable, When `engine.execute()` is called, Then `QueryMetrics.push_down_predicates_count + post_filter_predicates_count == total_predicates_in_query`, and `push_down_predicates_count` matches the per-sensor evaluable subset.

**AC-13 (BC-2.11.005 — execute_scheduled):** Given `engine.execute_scheduled(query, clients)` is called, When it returns, Then the returned tuple contains `(QueryResult, Arc<SessionContext>)` with `Arc::strong_count(&context) >= 1` (caller owns lifetime).

## Architecture Compliance Rules

- `QueryMetrics` collection MUST NOT add blocking I/O to the query execution hot path.
  Use `Instant::now()` (nanosecond resolution, no syscall beyond OS timer) for all
  timing. The ring buffer push is behind a non-blocking mutex (`try_lock`).
- `memory_peak_bytes` is read from DataFusion's `MemoryPool::reserved()` — this is
  the DataFusion-tracked allocation, not process RSS. It complements (but does not
  replace) the watchdog's RSS monitoring (S-2.02).
- `per_source_latency_ms` keys MUST use the format `"{sensor_id}/{client_id}"` —
  consistent with the format used in `get_diagnostics(subsystem: "fanout")` so
  that consumers can correlate individual queries with aggregate sensor stats.
- `QueryMetrics` MUST be serialized via `serde::Serialize` — do NOT hand-craft JSON.
  This ensures the schema stays consistent as fields are added.
- Ring buffer lock contention: if `try_lock()` fails, skip the push and increment a
  `metrics_dropped_count` atomic counter. Expose this in `get_diagnostics` so
  operators know if metrics are being lost under high query concurrency.

## Dev Notes

- The 100-query ring buffer is sized to cover a typical 5-minute window at 1 query
  per 3 seconds (the high end for ad-hoc usage). Scheduled queries run less frequently
  and are also captured. This is not a persistent metrics store — it is a rolling
  debug window.
- `_query_cost_ms` (the hidden column from S-3.08) should be populated from
  `per_source_latency_ms` proportionally per-row: `per_source_latency_ms[source] /
  rows_from_source`. This requires S-3.08 and S-3.09 to be wired together; if S-3.08
  runs first, populate `_query_cost_ms` as NULL until S-3.09 is implemented.
- `cache_hits` and `cache_misses` are 0 until S-3.11 (in-query dedup caching)
  is implemented. Leave the fields in the struct from day one so that S-3.11 only
  needs to wire up the counters, not change the struct.
- The materialization pipeline build-out (Phase A) is the largest single-story scope in the project to date (~13 points). The implementer should commit per phase (A1-A2, A3, A4-A5, A6-A7) and run `just iter prism-query` between phases, then `just check` at end of Phase A and end of Phase B before declaring GREEN. Per-phase commits also enable bisect-friendly history if regressions emerge in adversary cascade.
- Reference: `tests/end_to_end_tests.rs` is the FUNCTIONAL test surface. The existing tests in `profiling_integration_tests.rs` (which manually construct QueryMetrics) MUST be either (a) deleted in favor of the new e2e tests, or (b) re-purposed as `metrics_tests.rs` unit tests for QueryMetrics struct serde shape only — not as AC verification surface. The S-3.09 LOCAL pass-1 finding F-PASS1-CRIT-6 will be re-checked in pass-2.
- DataFusion `MemoryPool` API: confirm `reserved()` vs `consumed()` against datafusion 53.1 docs at TDD time. The story spec referenced `reserved()` originally; if that name is wrong, document the deviation and use the correct name.
- BC-2.11.005 ephemerality test: use `Arc::strong_count` probe via test-only accessor on `QueryEngine`, OR verify via Drop-side-effect (e.g., mock SessionContext with a Drop guard that increments a counter).
