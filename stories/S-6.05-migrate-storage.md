# S-6.05 â€” prism-bin: prism migrate-storage CLI Command

## Narrative

As an MSSP operator upgrading Prism to a new major version, I want a `prism
migrate-storage` command that safely applies pending RocksDB schema migrations, detects
lock conflicts with a running server, and supports a `--dry-run` preview, so that I can
perform schema upgrades confidently without risking data corruption or running migrations
against a live database.

## Objective

Implement the `prism migrate-storage` subcommand in `prism-bin`. The command detects
whether another Prism process holds the RocksDB lock, applies pending schema migrations
in order, updates the schema version record, and exits non-zero on failure. A `--dry-run`
flag prints what would change without writing. A migration summary is always printed
(N migrations applied, from version X to version Y). Partial migration state on next
startup triggers detection and either resumes or refuses to start (per P5-HIGH-3).

## Tasks

1. Define migration infrastructure in `crates/prism-storage/src/migrations/`:
   - `mod.rs`: declare `Migration` trait and `MigrationRegistry`
   - `Migration` trait:
     ```rust
     pub trait Migration: Send + Sync {
         fn version_from(&self) -> u32;
         fn version_to(&self) -> u32;
         fn description(&self) -> &str;
         fn apply(&self, backend: &dyn StorageBackend) -> Result<(), PrismError>;
         fn dry_run_description(&self) -> String; // Human-readable summary of what apply() would do
     }
     ```
   - `MigrationRegistry`: `Vec<Box<dyn Migration>>` sorted by `version_from`, exposed
     as `MigrationRegistry::pending(current_version: u32) -> Vec<&dyn Migration>`
   - Create one stub migration `V1ToV2` in `migrations/v1_to_v2.rs` as a placeholder
     that demonstrates the pattern (empty `apply()` body with a descriptive comment)

2. Implement `add_migration_args` in the clap `Commands` enum â€” add `MigrateStorage`
   variant in `crates/prism-bin/src/main.rs` (per S-6.01 scope expansion, this variant
   was pre-declared as a placeholder; this story replaces it):
   ```rust
   MigrateStorage(MigrateStorageArgs),
   ```

3. Define `MigrateStorageArgs` in `crates/prism-bin/src/commands/migrate_storage.rs`:
   ```rust
   #[derive(Args)]
   pub struct MigrateStorageArgs {
       /// Preview migrations without applying them
       #[arg(long)]
       dry_run: bool,

       /// Override the state directory path
       #[arg(long, env = "PRISM_STATE_DIR")]
       state_dir: Option<PathBuf>,
   }
   ```

4. Implement the `run` function in `commands/migrate_storage.rs`:
   a. Resolve `state_dir`: from `--state-dir` flag, `PRISM_STATE_DIR` env var, or config
      default (`~/.prism/state`)
   b. Attempt to open RocksDB in read-only mode first to read the current schema version:
      - If `StorageLockHeld` error: print "Another Prism process is running. Stop it
        before running migrations." and exit 1
   c. Read current schema version from the `default` CF (`_schema_version` key)
   d. Call `MigrationRegistry::pending(current_version)` to get the list
   e. If no pending migrations: print "Schema is up to date (version N). Nothing to do."
      and exit 0
   f. If `--dry-run`:
      - Print a formatted table of pending migrations: from_version â†’ to_version |
        description | dry_run_description
      - Print "Dry run complete. No changes were made."
      - Exit 0
   g. If not `--dry-run`:
      - Open RocksDB in write mode (will fail with `StorageLockHeld` if another process
        is running â€” re-check here for the write-mode open)
      - Set dirty bit: `StorageBackend::put(DirtyBits, b"migration_in_progress", timestamp)`
        with `sync: true` â€” survives crash (BC-2.15.005)
      - For each pending migration in version order:
        - Print "Applying migration v{from} â†’ v{to}: {description}..."
        - Call `migration.apply(backend)`:
          - On success: print "  OK"
          - On error: print "  FAILED: {error}"; exit 1 immediately (partial migration
            state â€” dirty bit remains set for startup detection)
      - Write new schema version: `put(Default, b"_schema_version", new_version_bytes)`
        with `sync: true`
      - Clear dirty bit: `remove(DirtyBits, b"migration_in_progress")`
      - Print "Migrations complete. Schema updated from v{start} to v{end}."
      - Exit 0

5. Implement startup dirty bit detection in `prism-storage` (used by S-6.01 startup
   sequence):
   - `StorageBackend::check_migration_dirty_bit() -> bool`: checks for key
     `b"migration_in_progress"` in the `dirty_bits` CF
   - If found on startup: log `ERROR "Schema migration was interrupted. Run
     'prism migrate-storage' to resume or inspect the database before starting."` and
     return `Err(PrismError::SchemaMigrationInterrupted)`
   - S-6.01 startup sequence catches `SchemaMigrationInterrupted` and exits 3 (same as
     `SchemaMigrationRequired`)

6. Write unit tests in `crates/prism-bin/src/commands/migrate_storage_tests.rs`:
   - Test `--dry-run` with 1 pending migration: verify no writes, output contains
     "Dry run complete"
   - Test `--dry-run` with 0 pending migrations: verify output says "up to date"
   - Test successful migration: verify schema version updated, dirty bit cleared
   - Test migration failure (mock migration that returns Err): verify exit 1, dirty bit
     NOT cleared (simulating crash-interrupted migration)
   - Test `StorageLockHeld` on open: verify exit 1 with lock-held message

## Acceptance Criteria

**AC-1:** Given `prism migrate-storage --dry-run` with 2 pending migrations, When run,
Then a table of pending migrations is printed (from_version, to_version, description,
what-would-change) and exit code is 0. No writes occur to RocksDB. (BC-2.15.001)

**AC-2:** Given `prism migrate-storage` (no dry-run) with 2 pending migrations, When
run, Then both migrations apply in order, the schema version record is updated to the
final version, and exit code is 0. (BC-2.15.001, BC-2.15.002)

**AC-3:** Given another Prism process holds the RocksDB lock, When `prism
migrate-storage` runs, Then it exits 1 with the message "Another Prism process is
running. Stop it before running migrations." No migration is attempted.

**AC-4:** Given a migration that fails mid-run (returns Err from `apply()`), When
`prism migrate-storage` runs, Then exit code is 1, the dirty bit remains set in RocksDB,
and subsequent `prism serve` startup detects the dirty bit and exits 3 with instructions
to run `prism migrate-storage`. (BC-2.15.005)

**AC-5:** Given `prism migrate-storage` completes successfully, When the dirty bit check
runs (either in `prism serve` startup or directly), Then no migration dirty bit is found.
(BC-2.15.005)

**AC-6:** Given `prism migrate-storage --state-dir /custom/path`, When run, Then the
RocksDB database at `/custom/path` is used instead of the default location.

**AC-7:** Given 0 pending migrations (schema already at latest version), When `prism
migrate-storage` runs, Then it prints "Schema is up to date (version N). Nothing to do."
and exits 0.

## Architecture Compliance Rules

- The dirty bit write MUST use `sync: true` â€” it must survive an OOM kill to enable
  crash detection (BC-2.15.005). This is the same requirement as query dirty bits.
- RocksDB lock detection MUST occur before any write is attempted â€” check on write-mode
  open, not just read-mode open, because another process could start between the two opens
- Migrations MUST apply in strict `version_from` order â€” the `MigrationRegistry` sorts
  by version and `pending()` returns a sorted slice
- The schema version key (`_schema_version`) MUST be written with `sync: true` â€” schema
  version is a critical consistency marker
- `prism migrate-storage` MUST NOT be called internally by `prism serve` â€” it is an
  operator-invoked maintenance command. `prism serve` exits 3 with instructions; the
  operator runs `prism migrate-storage` manually.
- `prism-storage` migration infrastructure MUST NOT depend on `prism-bin` â€” dependency
  goes only upward (bin depends on storage)
