# S-3.2.01 — prism-dtu-claroty: Multi-tenant state segregation

## Narrative

As a Prism MSSP operator managing multiple client organizations, I want the Claroty DTU
clone to key all mutable state by `(OrgId, resource_id)` rather than bare `resource_id`,
so that two clients whose Claroty xDome instances both have a device with ID `"dev-001"`
do not bleed each other's tag state during concurrent or interleaved query sessions.

## Objective

Migrate `prism-dtu-claroty/src/state.rs` from `HashMap<String, V>` keying to
`HashMap<(OrgId, String), V>` per ADR-008 §2.1 Step 6a. Add `reset_for(org_id)` and
rename `reset()` → `reset_all()` per ADR-008 §2.4. Add per-org proptest coverage that
kills OrgId-flipping mutations (TD-DTU-MUTATE-COVERAGE-001). Fold in known-prefix
post-restart eviction multi-tenant awareness (TD-W2-FIX-H-002).

## Tasks

1. **Migrate `tag_store` in `ClarotyState`** (`src/state.rs`):
   ```rust
   // Before
   pub tag_store: Mutex<HashMap<String, HashSet<String>>>,
   // After
   pub tag_store: Mutex<HashMap<(OrgId, String), HashSet<String>>>,
   ```

2. **Update call sites** — `add_tag`, `remove_tag`, `get_tags`:
   - Signature change: `fn add_tag(&self, org_id: OrgId, device_id: &str, tag: &str)`
   - Key construction: `(org_id, device_id.to_owned())`
   - Update `src/routes/tags.rs` to thread `OrgId` from request state/extensions.

3. **Add `DEFAULT_ORG_ID` test constant** (`#[cfg(test)]` only):
   ```rust
   #[cfg(test)]
   pub const DEFAULT_ORG_ID: OrgId = OrgId(uuid::uuid!("00000000-0000-7000-8000-000000000001"));
   ```
   Update all existing single-org unit tests to pass `DEFAULT_ORG_ID`.

4. **Add `reset_for(org_id: OrgId)`** and rename `reset()` → `reset_all()`:
   ```rust
   pub fn reset_for(&self, org_id: OrgId) {
       let mut store = self.tag_store.lock().expect("tag_store poisoned");
       store.retain(|(id, _), _| *id != org_id);
   }
   pub fn reset_all(&self) { /* clears all */ }
   // Compatibility shim (one-wave):
   pub fn reset(&self) { self.reset_all(); }
   ```

5. **TD-W2-FIX-H-002 — known_prefixes multi-tenant awareness**:
   If `ClarotyState` caches known API path prefixes (used for post-restart eviction),
   key that cache by `OrgId` so eviction does not cross org boundaries. If no such
   cache exists, document as N/A in PR.

6. **TD-DTU-MUTATE-COVERAGE-001 — add proptest in `tests/multi_tenant.rs`**:
   ```rust
   proptest! {
       fn prop_cross_org_lookup_returns_default(
           org_a: OrgId, org_b: OrgId, device_id: String, tag: String
       ) {
           prop_assume!(org_a != org_b);
           let state = ClarotyState::default();
           state.add_tag(org_a, &device_id, &tag);
           let result = state.get_tags(org_b, &device_id);
           prop_assert!(result.is_empty());
       }
   }
   ```
   Also add: `prop_write_isolation`, `prop_reset_for_selectivity` — matching
   VP-077 through VP-080.

7. **Gate**: `cargo test -p prism-dtu-claroty` green with no dead-code warnings for
   `DEFAULT_ORG_ID` in production paths.

## Acceptance Criteria

**AC-001** — Cross-org lookup returns default (traces to BC-3.2.001 postcondition 1):
Given tag `{"malware"}` is stored for `(org_id_A, "dev-001")`,
When `state.get_tags(org_id_B, "dev-001")` is called where `org_id_B ≠ org_id_A`,
Then the result is an empty `HashSet` (not an error, not orgA's tags).

**AC-002** — Write isolation (traces to BC-3.2.001 postcondition 2):
Given tags for `(org_id_A, "dev-001")` and `(org_id_B, "dev-001")` are stored independently,
When `state.add_tag(org_id_A, "dev-001", "new-tag")` is called,
Then `state.get_tags(org_id_B, "dev-001")` does not contain `"new-tag"`.

**AC-003** — Independent per-org state (traces to BC-3.2.001 postcondition 3):
Given `{"tag-A"}` stored for `(org_id_A, "dev-001")` and `{"tag-B"}` stored for `(org_id_B, "dev-001")`,
When both orgs look up `"dev-001"`,
Then `get_tags(org_id_A, "dev-001") == {"tag-A"}` and `get_tags(org_id_B, "dev-001") == {"tag-B"}`.

**AC-004** — Missing org returns default (traces to BC-3.2.001 postcondition 4):
Given `org_id_C` has never had any tags stored,
When `state.get_tags(org_id_C, "dev-001")` is called,
Then the result is an empty `HashSet` — no error, no panic.

**AC-005** — `reset_for` is selective (traces to BC-3.2.001 invariant 1, edge case EC-004):
Given tags stored for both `org_id_A` and `org_id_B`,
When `state.reset_for(org_id_A)` is called,
Then `get_tags(org_id_A, "dev-001")` returns empty AND `get_tags(org_id_B, "dev-001")` returns orgB's original tags.

**AC-006** — DEFAULT_ORG_ID is test-only (traces to BC-3.2.001 invariant 3):
Given the `ClarotyState` implementation compiles with `--release` (no `#[cfg(test)]`),
When the binary is inspected,
Then `DEFAULT_ORG_ID` does not appear in any production code path (enforced by `#[cfg(test)]` gate — compile error if misused).

**AC-007** — Proptest kills OrgId-flipping mutation (traces to BC-3.2.001 VP-079):
Given a proptest with adversarial `(org_id_A, org_id_B, device_id)` where `org_id_A ≠ org_id_B`,
When state is written under `org_id_A` and read under `org_id_B`,
Then the result is always empty — a mutation that replaces `org_id_A` with `org_id_B` in the key is killed by this test.

## Architecture Compliance Rules

- **Forbidden dependencies:** `prism-dtu-claroty` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any
  of these causes a build failure via `cargo-deny` rule.
- `DEFAULT_ORG_ID` MUST be gated `#[cfg(test)]`. Any production call site referencing
  it is a compile error.
- The composite key `(OrgId, String)` is the EXCLUSIVE keying scheme for mutable state
  in `ClarotyState` post-migration. No bare-String mutable store may exist.
- `reset_all()` is the implementation; `reset()` is a one-wave compatibility shim
  calling `reset_all()`. The shim MUST be removed before Wave 3 closes (same PR or
  follow-on).
- `BehavioralClone::reset()` trait method continues to call `state.reset_all()` —
  do NOT break the trait impl.
