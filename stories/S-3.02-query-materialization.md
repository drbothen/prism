# S-3.02 — prism-query: Query Tool and Materialization

## Narrative

As a Prism MCP server, I want a query engine that takes a PrismQL string and a set of
client scopes, fans out to all required sensor adapters in parallel, normalizes the
results to OCSF, materializes them as DataFusion MemTables, and runs the SQL plan —
so that analysts receive unified, cross-sensor, cross-client query results in a single
tool call.

## Objective

Implement the core query execution pipeline: from MCP tool params through fan-out,
OCSF normalization, Arrow RecordBatch materialization, DataFusion MemTable registration,
SQL plan execution, and result return. Include memory budget enforcement (200MB per
query), sensor filter push-down, cross-client scoping, virtual field injection
(`_sensor`, `_client`, `_source_table`), and ephemeral SessionContext lifecycle.

## Tasks

1. Implement `QueryEngine` struct in `crates/prism-query/src/engine.rs`:
   - Fields: `adapter_registry: Arc<AdapterRegistry>`, `credential_store: Arc<dyn CredentialStore>`,
     `ocsf_normalizer: Arc<OcsfNormalizer>`, `config: QueryEngineConfig`
   - `async fn execute(query_str: &str, clients: Option<Vec<TenantId>>, options: QueryOptions)
     -> Result<QueryResult, PrismError>` (BC-2.11.001)
   - `async fn execute_scheduled(query_str: &str, clients: Option<Vec<TenantId>>)
     -> Result<(QueryResult, Arc<SessionContext>), PrismError>` — for detection engine
     reuse (S-4.03 depends on this); the SessionContext is returned so detection rules
     can run additional queries against already-materialized data without re-fetching

2. Implement ephemeral materialization pipeline in `crates/prism-query/src/materialization.rs`
   (BC-2.11.005):
   - Step 1: Parse PrismQL string via S-3.01 parser → get source refs
   - Step 2: Resolve source refs to `(SensorType, client_id, SensorSpec)` tuples
   - Step 3: Fan out to sensor adapters via S-2.06 `fan_out()` — all sources in parallel
   - Step 4: Normalize each `Vec<serde_json::Value>` via S-1.04/S-1.05 `OcsfNormalizer`
     → `Vec<RecordBatch>`
   - Step 5: Inject virtual field columns into each RecordBatch (see Task 6)
   - Step 6: Register each source as a DataFusion `MemTable` in the SessionContext with
     the source ref as the table name (e.g. `crowdstrike.detections`)
   - Step 7: Execute the SQL plan against the registered MemTables
   - Step 8: Collect `SendableRecordBatchStream` → `Vec<RecordBatch>` → `QueryResult`

3. Implement DataFusion GreedyMemoryPool integration in `crates/prism-query/src/memory.rs`
   (BC-2.11.006):
   - Create `SessionContext` with `RuntimeEnv` configured to use `GreedyMemoryPool` with
     200MB limit (`200 * 1024 * 1024` bytes)
   - 3-tier fallback: (1) query runs within 200MB budget; (2) if pool trips, DataFusion
     returns `ResourcesExhausted` — map to E-QUERY-008; (3) if OOM before pool trips
     (should not happen), catch panic at the task boundary
   - The 200MB limit applies per query, not globally — each `SessionContext` gets its own pool
   - **Query timeout** (BC-2.11.006): wrap entire `execute()` in `tokio::time::timeout(30s)`.
     On timeout, return E-QUERY-004 (transient, retryable — NOT E-QUERY-003 which is
     for syntactic validation failures). Per BC-2.11.006, BC-2.11.001, error taxonomy.
   - **Materialization record cap** (BC-2.11.006): during fan-out (Step 3), maintain a
     streaming record counter across all sources. If cumulative count exceeds 10,000,
     abort materialization with E-QUERY-005 ("Materialization limit exceeded: {count}
     records from {sources} sources exceeds 10K cap"). This prevents unbounded memory
     growth from large sensor responses. The cap applies to external sources only —
     internal table scans use PRISM_MAX_INTERNAL_TABLE_SCAN instead (S-2.03).
   - **Integer overflow prevention** (BC-2.11.006): DataFusion uses i128 intermediate
     representation for aggregate expressions. No additional code needed — document that
     DataFusion handles this natively.

4. Implement sensor filter push-down in `crates/prism-query/src/pushdown.rs` (BC-2.11.007):
   - `classify_predicates(where_clause: &Expr, source: &SensorSpec) -> PushDownPlan`
   - `PushDownPlan { push_down: Vec<Predicate>, post_filter: Vec<Predicate> }`
   - Classification rules:
     - REQUIRED columns (defined per sensor spec via S-1.11): always `push_down`
     - OPTIONAL columns with sensor-side filter support: `push_down` if the sensor adapter
       reports `supports_filter(column)` = true
     - All other predicates: `post_filter` (DataFusion evaluates after fetch)
   - Pushed-down predicates are passed to `fan_out()` as `QueryParams.filters`

5. Implement cross-client query scoping in `crates/prism-query/src/scoping.rs` (BC-2.11.011):
   - `resolve_clients(clients: Option<Vec<TenantId>>, registry: &ClientRegistry)
     -> Vec<TenantId>`
   - If `clients` is `Some(list)`: validate each TenantId exists; return list
   - If `clients` is `None`: return all configured client IDs from registry
   - Results from all clients are merged into a single set of MemTables — each client's
     data for a source is a separate RecordBatch appended to the same MemTable

6. Implement virtual field injection in `crates/prism-query/src/virtual_fields.rs`
   (BC-2.11.012):
   - `inject_virtual_fields(batch: RecordBatch, sensor: &SensorType, client_id: &TenantId,
     source_table: &str) -> RecordBatch`
   - Add three string columns to every RecordBatch:
     - `_sensor`: sensor type string (e.g. `"crowdstrike"`)
     - `_client`: tenant ID string
     - `_source_table`: source table name (e.g. `"crowdstrike.detections"`)
   - These columns are filterable in WHERE clauses and selectable in SELECT *
   - Virtual field column names are reserved — sensors cannot emit fields with these names
     (validated in S-1.11 spec loading)

7. Implement ephemeral SessionContext lifecycle in `crates/prism-query/src/session.rs`:
   - `SessionScope` — RAII wrapper using `scopeguard::defer!` to drop the `SessionContext`
     after results are collected, even on panic
   - SessionContext must not outlive the `execute()` call for non-scheduled queries
   - `execute_scheduled` returns an `Arc<SessionContext>` that is kept alive by the
     detection engine (caller manages lifetime)

8. Implement `RocksDbTableProvider` in `crates/prism-query/src/internal_tables.rs`:
   - Wraps `InternalTableDescriptor` from prism-storage (S-2.03)
   - Implements DataFusion `TableProvider` trait for internal tables
   - `scan()` calls `StorageBackend::scan(domain, prefix)` and materializes rows
     as Arrow RecordBatches using the descriptor's schema
   - Capability gate for `prism_audit`: check `audit.read` capability, return
     E-QUERY-011 if denied
   - Scan truncation at `PRISM_MAX_INTERNAL_TABLE_SCAN` entries with
     `_meta.scan_truncated` flag
   - Register tables in SessionContext: prism_schedules, prism_alerts, prism_cases,
     prism_diff_results, prism_audit (per BC-2.15.011 from S-2.03)

9. Write VP-031 proptest in `crates/prism-query/proofs/vp031_pushdown.rs`:
   - Property: for any REQUIRED column from any sensor spec, `classify_predicates()`
     always produces `PushDown` — never `PostFilter`
   - Use proptest's `prop_compose!` to generate arbitrary `Predicate` trees over the
     set of REQUIRED column names

## Acceptance Criteria

**AC-1:** Given `execute("SELECT * FROM crowdstrike.detections LIMIT 10", clients: ["acme"])`,
When the query runs, Then results contain OCSF-normalized CrowdStrike detections with
`_sensor = "crowdstrike"`, `_client = "acme"`, `_source_table = "crowdstrike.detections"`
virtual fields present in every row. (BC-2.11.001, BC-2.11.012)

**AC-2:** Given a query targeting `crowdstrike.detections` and `claroty.alerts`, When
materialization runs, Then fan-out fetches both sources in parallel (observable via
tracing spans), both are normalized and registered as separate MemTables, and the SQL
plan runs across both. (BC-2.11.005)

**AC-3:** Given a query that causes DataFusion's GreedyMemoryPool to exceed 200MB,
When the pool limit is hit, Then `E-QUERY-008` is returned and no partial results are
emitted. (BC-2.11.006)

**AC-4:** Given a WHERE clause with `severity_id >= 3` where `severity_id` is a REQUIRED
column on the CrowdStrike source, When push-down classification runs, Then `severity_id
>= 3` is in `PushDownPlan.push_down` and is passed as a filter to the sensor adapter
fetch call. (BC-2.11.007)

**AC-5:** Given `execute(query, clients: None)`, When the query runs, Then all configured
client IDs are included in the fan-out targets and results from all clients appear in
the response. (BC-2.11.011)

**AC-6:** Given a query targeting two clients ("acme" and "contoso") for the same source,
When materialization completes, Then the source MemTable contains rows from both clients,
distinguished by `_client` virtual field. (BC-2.11.011)

**AC-7:** Given the SessionContext is created for a non-scheduled query, When `execute()`
returns (including on error or panic), Then the SessionContext is dropped and its memory
is released. (BC-2.11.005)

**AC-8:** VP-031 proptest passes: REQUIRED columns are always classified as `PushDown`
for all generated predicate trees.

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Query materialization / DataFusion integration | `prism-query` (SS-11) | Mixed |
| Sensor fan-out | `prism-sensors` (SS-01) | Effectful |
| OCSF Arrow schema | `prism-ocsf` (SS-02) | Pure |

`prism-query` implements `QueryEngine::execute()` — it creates an ephemeral DataFusion
`SessionContext` (AD-002), registers sensor table providers, executes the PrismQL AST, and
materializes OCSF-normalized Arrow batches. `prism-sensors` performs HTTP fan-out during
scan. `prism-ocsf` provides the Arrow schema. Per `architecture/module-decomposition.md`,
`prism-query` orchestrates `SensorAdapter`, `OcsfNormalizer`, and `SpecEngine`.

## Architecture Compliance Rules

- The materialization pipeline is the ONLY path that touches sensor adapters from the
  query layer — no other module in `prism-query` may call `AdapterRegistry` directly
- `SessionContext` MUST be ephemeral for non-scheduled queries — it MUST NOT be stored
  in any long-lived struct (e.g. no `Arc<SessionContext>` fields on `QueryEngine`)
- The 200MB memory pool is per-query, not per-process — do not use a shared global pool
  (that would serialize all queries through a single budget)
- Virtual field column names (`_sensor`, `_client`, `_source_table`) are reserved at
  the engine level — the engine MUST overwrite any sensor-emitted columns with those names
  to prevent spoofing
- `execute_scheduled` exists solely for the detection engine's reuse pattern — do NOT
  use it for regular analyst queries; the caller owns the SessionContext lifetime

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | DataFusion GreedyMemoryPool exceeds 200MB during query | `E-QUERY-008` returned; no partial results emitted; SessionContext dropped via RAII (BC-2.11.006) |
| EC-002 | Query execution exceeds 30-second timeout | `E-QUERY-004` returned (transient, retryable); NOT `E-QUERY-003` (BC-2.11.006) |
| EC-003 | Cumulative record count across all sources exceeds 10,000 during fan-out | Materialization aborted with `E-QUERY-005`; message includes count and source list (BC-2.11.006) |
| EC-004 | `execute()` panics inside materialization pipeline | `SessionScope` RAII drop fires; SessionContext released even on panic (BC-2.11.005) |
| EC-005 | Sensor emits a field named `_sensor`, `_client`, or `_source_table` | Engine overwrites it with canonical virtual field value; sensor-emitted value discarded (BC-2.11.012 spoofing prevention) |

## Dev Notes

- DataFusion 53 uses `SessionContext::new_with_config_rt()` (or similar) to inject a custom
  `RuntimeEnv` with the memory pool — verify the exact constructor signature against the
  DataFusion 53 API docs before implementation (API has changed significantly since earlier versions)
- The MemTable registration uses the source ref as the table name. DataFusion table names
  containing dots (`crowdstrike.detections`) must be quoted with backticks in SQL:
  `` SELECT * FROM `crowdstrike.detections` ``. The SQL mode parser (S-3.01) should
  emit backtick-quoted table names when generating DataFusion SQL from its AST.
- Filter push-down is a best-effort optimization — incorrect push-down classification
  produces wrong results, but missing push-down only affects performance. When in doubt,
  classify as `PostFilter`.
- `fan_out()` from S-2.06 already handles partial failure (returns successes + errors).
  The materialization pipeline should log partial failures at WARN level and include
  them in `QueryResult.warnings`, not fail the entire query.
