# S-5.05 — prism-mcp: Config Loading and Validation

## Narrative

As a Prism operator setting up the server for an MSSP deployment, I want configuration
errors to be caught early with clear, actionable messages — all in a single validation
pass — so that I can correct my prism.toml before the server starts and never face
a partially-initialized server due to a config typo.

> **Scope boundary (Phase 3 patch):** This story implements single-file TOML configuration
> loading and validation. Multi-repo git subscription sync, `[[config_sources]]` priority
> merging, and the `prism config diff`/`show --effective`/`show --trace` CLI commands are
> scoped to **S-5.07 Multi-Repo Git Config Subscriptions**. This story's AC set is complete
> when a single file loads/validates correctly; S-5.07 builds on top.

## Objective

Implement TOML configuration loading, multi-error validation, environment variable
overrides, credential reference resolution, capability merging, the `--dry-run` flag,
and client ID validation. This is the startup config pipeline that gates all server
initialization.

## Tasks

1. Define `PrismConfig` struct in `crates/prism-mcp/src/config/mod.rs`:
   - `[server]`: `state_dir: PathBuf`, `log_level: String`
   - `[defaults.capabilities]`: `HashMap<String, CapabilityValue>` — global defaults
   - `[clients.{id}]`: per-client map, key validated as `TenantId`
   - `[clients.{id}.sensors.{type}]`: `{ credential_ref: String, sources: Vec<String>,
     enabled: bool }`
   - `[clients.{id}.capabilities]`: per-client capability overrides
   - Derive: `serde::Deserialize`, `Clone`, `Debug`
   (BC-2.06.001, BC-2.06.002)

2. Implement TOML loader in `crates/prism-mcp/src/config/loader.rs`:
   - Read file path from CLI arg or default (`$XDG_CONFIG_HOME/prism/prism.toml`,
     fallback `~/.config/prism/prism.toml`, fallback `./prism.toml`)
   - Deserialize via `toml::from_str::<PrismConfig>`
   - On parse error: wrap in `ConfigError::ParseError { path, line, message }`
   (BC-2.06.001)

3. Implement credential reference resolution in `crates/prism-mcp/src/config/loader.rs`:
   - Scan all `credential_ref` fields in sensor configs
   - For each ref: validate reachability using the 3-tier resolution chain from
     BC-2.06.003 and S-1.07's `resolve_secret()`:
     1. Check `{NAME}_FILE` env var exists (file path to credential)
     2. Check `{NAME}` env var exists (direct credential value)
     3. Check `CredentialStore::exists(tenant, sensor, name)` from S-1.06
   - If ALL three tiers fail: add to error list (do not abort — collect all missing refs)
   - A ref is valid if ANY tier resolves — do not require all three (BC-2.06.003)

4. Implement capability merging in `crates/prism-mcp/src/config/loader.rs`:
   - Produce `ResolvedClientConfig` by merging `defaults.capabilities` with
     `clients.{id}.capabilities` using `ClientCapabilities::merge()` from prism-core
   - More-specific wins: a client-level capability entry overrides the global default
     for the same key
   - Trace: each merged config includes `resolution_source` per capability (default vs
     client override) (BC-2.06.004)

5. Implement multi-error validation in `crates/prism-mcp/src/config/validation.rs`:
   - `validate(config: &PrismConfig) -> Result<(), Vec<ConfigError>>`
   - Run all checks in sequence, collecting errors into a `Vec`
   - Never short-circuit (do not use `?` inside the validation loop)
   - Checks: required fields present, client IDs valid, sensor types known, credential
     refs exist, sources non-empty for enabled sensors (BC-2.06.005)

6. Implement `--dry-run` mode in `crates/prism-mcp/src/config/redact.rs`:
   - `redact(config: &PrismConfig) -> RedactedConfig` — mask all `credential_ref`
     values as `"***"` in the printed output
   - Print redacted config as formatted TOML to stdout
   - Print validation result: "OK — N clients, M sensors" or list of errors
   - Exit 0 on success, exit 1 on validation errors (BC-2.06.006)

7. Implement actionable error messages in `crates/prism-mcp/src/config/validation.rs`:
   - `ConfigError::MissingField { path: String, suggestion: String }` — include the
     TOML path and a concrete example of what to add
   - E.g., for missing `credential_ref`: `"Missing required field
     'clients.acme.sensors.crowdstrike.credential_ref' — add: credential_ref = \"cs_api_key\""`
   (BC-2.06.007)

8. Implement environment variable overrides in `crates/prism-mcp/src/config/env_override.rs`:
   - After TOML load, scan for env vars matching `PRISM_{SECTION}_{KEY}` (uppercase,
     underscores)
   - Override corresponding config fields: e.g., `PRISM_STORAGE_STATE_DIR=/tmp/prism`
     sets `config.server.state_dir`
   - Log each override applied at `DEBUG` level to stderr (BC-2.06.008)

9. Implement client ID validation in `crates/prism-mcp/src/config/validation.rs`:
   - For each key in `[clients]`: call `TenantId::new(key)` and collect failures
   - Error: `ConfigError::InvalidClientId { id: String, reason: String }` with error
     code `E-CFG-003`
   (BC-2.06.010)

10. Implement DI-029 cross-validation in `crates/prism-mcp/src/config/validation.rs`
    (BC-2.06.005 postcondition — cross-field diagnostic):
    - After all field-level checks pass, iterate over all `correlation` and `sequence`
      detection rules referenced in the config
    - For each rule, compare its `window` duration against the `interval` of its feeding
      schedule (identified by matching `schedule_id` or sensor source)
    - If `window < interval`: emit a WARN-level diagnostic log:
      `"correlation window < schedule interval for rule '{rule_id}' (window={window}s, interval={interval}s)"`
    - This check is non-blocking: the config loads successfully regardless; the WARN is
      informational only and does not add to the `Vec<ConfigError>` error list

11. Write tests in `crates/prism-mcp/tests/config.rs`:
    - Test valid minimal prism.toml loads correctly
    - Test 3 validation errors are all reported in one pass
    - Test env var override takes precedence over TOML value
    - Test capability merge: client override wins over default
    - Test `--dry-run` output masks credential_ref values
    - Test client ID `"acme/../../etc"` produces `E-CFG-003`

## Acceptance Criteria

**AC-1:** Given a valid `prism.toml` with two clients and their sensor configs, When
loaded at startup, Then `PrismConfig` is fully populated with both clients' sensor
mappings and capability settings.
(traces to BC-2.06.001 postcondition 1)

**AC-2:** Given `[clients.acme.sensors.crowdstrike]` and `[clients.acme.sensors.claroty]`
in the config, When loaded, Then `config.clients["acme"].sensors` contains exactly two
entries, one for each sensor type.
(traces to BC-2.06.002 postcondition 1)

**AC-3:** Given `credential_ref = "cs_api_key"` in a sensor config, When validation
runs, Then `CredentialStore::exists(tenant_id, "crowdstrike", "cs_api_key")` is called and, if missing, a
`ConfigError` naming `"cs_api_key"` is added to the error list.
(traces to BC-2.06.003 postcondition 1)

**AC-4:** Given `defaults.capabilities = {"read" → Allow}` and `clients.acme.capabilities
= {"crowdstrike.write" → Allow}`, When capabilities are merged for "acme", Then the
result contains both `read → Allow` (from default) and `crowdstrike.write → Allow` (from
client override).
(traces to BC-2.06.004 postcondition 1)

**AC-5:** Given a config with 3 distinct validation errors (missing field, invalid client
ID, missing credential ref), When `validate()` is called, Then it returns a `Vec` with
all 3 errors — not just the first one.
(traces to BC-2.06.005 postcondition 1)

**AC-6:** Given `--dry-run` flag, When executed against a valid config, Then the tool
prints a redacted config summary (all `credential_ref` values replaced with `"***"`) to
stdout and exits with code 0.
(traces to BC-2.06.006 postcondition 1)

**AC-7:** Given `--dry-run` flag with a config missing `credential_ref`, When executed,
Then the error message includes the TOML path and a concrete example of the missing
field, and exits with code 1.
(traces to BC-2.06.007 postcondition 1)

**AC-8:** Given `PRISM_STORAGE_STATE_DIR=/tmp/prism` in the environment and
`state_dir = "/var/prism"` in prism.toml, When config is loaded, Then
`config.server.state_dir` is `/tmp/prism` (env var overrides TOML).
(traces to BC-2.06.008 postcondition 1)

**AC-9:** Given a client key `"acme/../../etc"` in the `[clients]` table, When
validated, Then `E-CFG-003` is returned with a message indicating the invalid characters.
(traces to BC-2.06.010 postcondition 1)

**AC-10:** Given the config file changes (new or modified sensor/capability/feature-flag
entries), When the config reload loop processes the change and validates successfully,
Then the config subsystem emits a `ConfigReloaded` event carrying the new capabilities,
and downstream consumers (MCP server per BC-2.10.005) trigger
`notifications/tools/list_changed` atomically after effective-set swap.
(traces to BC-2.06.009 postcondition 1)

**AC-11:** Given a correlation rule with `window = 30s` AND its feeding schedule has
`interval = 60s`, When config loads, Then a WARN-level diagnostic is emitted:
`"correlation window < schedule interval for rule '{id}' (window=30s, interval=60s)"`,
and the config still loads successfully (non-blocking — no entry added to error list).
(traces to BC-2.06.005 / DI-029 cross-validation postcondition)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Config loading MCP tools | `prism-mcp` (SS-06) | Effectful |
| TOML config parsing | `prism-spec-engine` (SS-16) | Mixed |

`prism-mcp` implements the startup config loading pipeline and exposes the `reload_config` MCP tool for runtime config re-reads (BC-2.16.005). `prism-spec-engine` performs TOML parsing and schema validation. Config validation logic is pure; filesystem reads and arc-swap publication are effectful. Per `architecture/module-decomposition.md`, SS-06 (Client Configuration) is co-owned by `prism-spec-engine` and `prism-mcp`.

## Architecture Compliance Rules

- Config loading MUST use `toml` crate for deserialization — not serde_json or a custom
  parser
- Environment variable override MUST use `std::env::var` only — do not pull in a
  dotenv crate
- `validate()` MUST collect all errors before returning — never short-circuit with `?`
  inside the validation loop
- Credential reference validation MUST use `CredentialStore::exists()` from prism-credentials
  (S-1.06), not filesystem checks
- `RedactedConfig` MUST mask ALL fields containing credential data — err on the side of
  masking more than less
- The config precedence order (defaults < TOML < env) MUST be enforced in this exact
  sequence — env vars override TOML, not the other way around

## Dev Notes

- The `validate()` function will grow over time as new config fields are added. Structure
  it as a list of independent validator functions called in sequence, each appending to
  the error vec — this keeps it maintainable.
- The default config file path search order (`XDG_CONFIG_HOME` → `~/.config` → `./`)
  matches standard Unix conventions. Log the resolved path at `INFO` level so operators
  can confirm which file was loaded.
- The `--dry-run` exit code convention (0 = valid, 1 = invalid) allows it to be used
  in CI/CD pipelines to validate config changes before deployment.
- prism.toml should never contain raw credential values — only `credential_ref` pointers
  to the credential store. The redact pass is a safety net, not the primary protection.
