# S-3.05 — prism-query: Pagination and Caching

## Narrative

As a Prism MCP server, I want ephemeral cursor-based pagination for large query result
sets and a TTL-based response cache for repeated identical queries, so that analysts and
LLM agents can page through results incrementally without re-executing expensive sensor
fan-outs, and so that repeated queries within a time window are served from memory rather
than hitting sensor APIs again.

## Objective

Implement ephemeral cursor-based pagination (in-memory result sets, forward-only cursor
tokens, 60-second expiry, 200-cursor cap) and a response cache (LRU with configurable
TTL, SHA-256 cache key derivation, write-operation invalidation, 50MB memory bound with
LRU eviction).

## Tasks

1. Implement ephemeral cursor in `crates/prism-query/src/cursor.rs` (BC-2.07.001):
   - `CursorRegistry` — singleton (via `OnceLock`) holding active cursors
   - `CursorEntry { result_rows: Vec<serde_json::Value>, offset: usize, created_at: Instant,
     query_str: String, client_id: TenantId }`
   - `CursorToken` — opaque `Uuid` v4 string returned to the caller
   - `CursorRegistry::create(rows: Vec<serde_json::Value>, page_size: usize)
     -> Result<(Vec<serde_json::Value>, Option<CursorToken>), PrismError>`:
     if `rows.len() <= page_size`, return all rows and `None` token (no next page);
     otherwise store entry and return first page + token
   - `CursorRegistry::next_page(token: CursorToken)
     -> Result<(Vec<serde_json::Value>, Option<CursorToken>), PrismError>`:
     advance offset by page_size; return next page + new token (or None if exhausted)
   - No state is persisted to disk — all cursors live in process memory (BC-2.07.001)

2. Implement cursor lifecycle management in `cursor.rs` (BC-2.07.002):
   - Cursor expiry: 60 seconds from creation — check `created_at.elapsed() > 60s` on
     every `next_page()` call; return E-QUERY-004 (cursor expired) if elapsed
   - Forward-only: cursors track a monotonically increasing `offset` — no seeking backward
   - 200-cursor cap: this prism-query `CursorRegistry` wraps `prism-core::CursorRegistry`
     (S-1.02 VP-029) for cap enforcement. On `create()`: call `core_registry.allocate()`
     first — if it returns Err(CursorCapExceeded), return E-QUERY-002 without creating
     the entry. On expiry/exhaustion: call `core_registry.release()`. The prism-core
     registry tracks the count; prism-query stores the result sets.
   - Background cleanup: spawn a `tokio::spawn` task that wakes every 30 seconds and
     removes all entries where `created_at.elapsed() > 60s` (prevents unbounded growth)

3. Implement response cache in `crates/prism-query/src/cache.rs` (BC-2.07.003):
   - Use `lru::LruCache<CacheKey, CacheEntry>` with capacity based on entry count (start
     at 500 entries; each entry ~10KB per NFR-015)
   - `CacheEntry { rows: Vec<serde_json::Value>, cached_at: Instant, ttl: Duration }`
   - Default TTL: 300 seconds (5 minutes), configurable via `[query] cache_ttl_secs` in
     `prism.toml`
   - `QueryCache::get(key: &CacheKey) -> Option<Vec<serde_json::Value>>`: return cached
     rows if present and not expired; remove entry if expired and return None
   - `QueryCache::put(key: CacheKey, rows: Vec<serde_json::Value>)`: insert entry; if at
     memory bound, evict LRU entry before inserting

4. Implement cache invalidation in `crates/prism-query/src/invalidation.rs` (BC-2.07.004):
   - `CacheInvalidator::invalidate_for_sensor(sensor_type: SensorType)`: scan all cache
     keys; evict entries whose `cache_key.sources` contain `sensor_type`
   - `CacheInvalidator::invalidate_for_client(client_id: &TenantId)`: evict entries whose
     `cache_key.clients` contain `client_id`
   - Trigger points: call `invalidate_for_sensor()` from `configure_credential_source()` handler;
     call `invalidate_for_client()` from any client management write operation
   - Invalidation is synchronous — write operations do not return until relevant cache
     entries are removed

5. Implement cache key derivation in `crates/prism-query/src/cache_key.rs` (BC-2.07.005):
   - `CacheKey::derive(query_str: &str, clients: &[TenantId], push_down_params: &PushDownPlan)
     -> CacheKey`
   - Canonical form: sort `clients` alphabetically before hashing to ensure
     `clients: ["acme", "contoso"]` and `clients: ["contoso", "acme"]` produce the same key
   - Hash input: `SHA-256(canonical_query_str || sorted_clients_json || push_down_params_json)`
   - `CacheKey` is a newtype over `[u8; 32]` — fixed-size, stack-allocated, no heap alloc

6. Implement cache memory bounds in `cache.rs` (BC-2.07.006):
   - Track total cache memory usage: `total_bytes: AtomicUsize` updated on each put/evict
   - Hard limit: 50MB (`50 * 1024 * 1024` bytes)
   - On `put()`: if `total_bytes + new_entry_size > 50MB`, evict LRU entries until
     there is sufficient space (loop, not single eviction)
   - `new_entry_size` approximation: `rows.len() * AVG_ROW_SIZE_BYTES` where
     `AVG_ROW_SIZE_BYTES = 512` (configurable constant)

7. Implement VP-025 Kani proof — Cache Key Derivation Deterministic
   - Harness file: `crates/prism-query/proofs/vp025_cache_key.rs`
   - Method: kani
   - Traces to: BC-2.07.005 postconditions (cache key is a deterministic function of (query, scope))
   - Proof sketch: for symbolic `query_str` and `clients` inputs, assert `cache_key(q, c) == cache_key(q, c)`
     and that permuted-but-equivalent `clients` orderings produce the same `CacheKey`

## Acceptance Criteria

**AC-1:** Given a query returning 1,000 rows with a page size of 100, When the first
`execute()` call returns, Then the response contains 100 rows and a non-null cursor token,
and the remaining 900 rows are held in memory. (BC-2.07.001)

**AC-2:** Given a cursor token returned from the first page, When `next_page(token)` is
called, Then the next 100 rows are returned and a new cursor token is provided (or None
if the last page). (BC-2.07.001)

**AC-3:** Given a cursor token that is 61 seconds old, When `next_page(token)` is called,
Then E-QUERY-004 is returned and the cursor entry is removed from the registry.
(BC-2.07.002)

**AC-4:** Given 200 active cursor tokens already exist, When a new query attempts to
create a 201st cursor, Then E-QUERY-002 is returned. (BC-2.07.002)

**AC-5:** Given an identical query executed twice within the cache TTL window (default
300s), When the second execution runs, Then cached rows are returned without calling any
sensor adapter. (BC-2.07.003)

**AC-6:** Given a cached query result for `crowdstrike.detections`, When `configure_credential_source`
is called for the CrowdStrike sensor, Then the cache entry for that query is evicted
before `configure_credential_source` returns. (BC-2.07.004)

**AC-7:** Given the same query string and client list passed in different orderings
(`["acme", "contoso"]` vs `["contoso", "acme"]`), When cache keys are derived, Then
both produce the identical `CacheKey`. (BC-2.07.005)

**AC-8:** Given the cache at 50MB capacity, When a new result set is inserted, Then at
least one LRU entry is evicted before the new entry is stored, keeping total memory at or
below 50MB. (BC-2.07.006)

**AC-9:** VP-025 passes (method: kani) — for all bounded symbolic inputs, `CacheKey::derive(q, clients, params)` returns the same `CacheKey` on repeated invocations, and permuted `clients` orderings produce the identical key.
(traces to BC-2.07.005)

## Architecture Compliance Rules

- Cursors are ephemeral and in-process ONLY — no cursor state is written to RocksDB or
  any other persistent store; process restart invalidates all cursors
- Cache invalidation MUST be synchronous with the write operation — do NOT use async
  background invalidation that could serve stale data after a write returns
- The 200-cursor cap is enforced via the `CursorRegistry` from `prism-core` (S-1.02
  VP-029) — do NOT implement a second independent cap in this crate
- `CacheKey` derivation MUST be deterministic and order-independent for client lists —
  sort before hashing; this is a correctness requirement, not just an optimization
- Cache entries store serialized JSON rows (`Vec<serde_json::Value>`), not Arrow
  RecordBatches — this simplifies memory accounting and avoids Arrow schema lifecycle
  complexity in the cache layer

## Previous Story Intelligence

This story depends on S-3.02 (`QueryEngine::execute()`) because:
- `CursorRegistry::create()` is called inside `execute()` after results are collected —
  when result count exceeds the page size, results are stored and a token returned
- `QueryCache::get()` is called at the start of `execute()` — cache hit short-circuits
  the entire fan-out pipeline
- `CacheInvalidator` hooks are called from MCP tool handlers implemented in S-5.01/S-5.02

The `CursorRegistry` 200-cursor cap was formally proved in S-1.02 (VP-029). This story's
`CursorRegistry` implementation MUST use the `prism-core` enforcement mechanism, not a
separate counter.

## Dev Notes

- BC-2.07.003 through BC-2.07.006 are marked P1 in the behavioral contract spec. They
  are included in this story because their implementation is tightly coupled to the cursor
  infrastructure and the overhead of a separate story would exceed the benefit. However,
  if wave scheduling requires dropping P1 items to meet a deadline, the cache subsystem
  (Tasks 3–6) can be deferred while the cursor subsystem (Tasks 1–2) ships as P0.
- The 50MB cache bound is informed by the 512MB process memory budget. With the 200MB
  per-query DataFusion pool and other process overhead, 50MB is a conservative allocation
  for the cache. It can be made configurable via `[query] cache_max_bytes` in `prism.toml`.
- Cache key derivation includes `push_down_params` to handle the case where the same
  query text results in different push-down classifications (e.g. after a sensor spec
  reload changes REQUIRED columns). Without this, stale cache entries could serve results
  derived from a different push-down plan.
- The background cursor cleanup task (30-second wake interval) is a tokio `spawn` with a
  `tokio::time::interval`. It should be started once during `QueryEngine` initialization
  and run for the process lifetime. Use `tokio::select!` with a shutdown signal so it
  terminates cleanly on process exit.
