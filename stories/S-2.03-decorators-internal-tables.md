# S-2.03 — prism-storage: Decorators and Internal Tables

## Narrative

As a Prism query engine consumer, I want query results automatically enriched with
context metadata and RocksDB domains exposed as queryable DataFusion tables, so that
analysts can join live sensor data against stored state (alerts, schedules, cases)
without bespoke integration code.

## Objective

Implement context decorator injection into query results, the three-phase decorator
lifecycle (config-time, query-time, periodic), and registration of RocksDB-backed
domains as DataFusion table providers. Include capability gating for the audit table
and scan truncation with metadata signaling.

## Tasks

1. Define `DecoratorKind` enum in `crates/prism-storage/src/decorators.rs`:
   - `ConfigTime { key: String, value: serde_json::Value }` — static from `prism.toml`,
     injected into every result's `_meta` envelope
   - `QueryTime { field: VirtualField }` — computed per query; values are filterable
   - `Periodic { key: String, source: PeriodicSource, ttl_seconds: u64 }` — cached,
     refreshed on TTL expiry, stored in `decorators` CF (BC-2.15.010)

2. Define `VirtualField` enum:
   - `Sensor` → column name `_sensor`, value = sensor type string, filterable
   - `ClientId` → column name `_client_id`, value = client ID string, filterable
   - `SourceTable` → column name `_source_table`, value = table name, filterable
   - Virtual fields are injected as top-level columns; `_meta` envelope fields are NOT
     filterable and must NOT be accepted in WHERE clauses (BC-2.15.009)

3. Implement `DecoratorInjector` in `crates/prism-storage/src/decorators.rs`:
   - `inject(batch: RecordBatch, context: &QueryContext) -> Result<RecordBatch, PrismError>`
   - Appends virtual field columns (`_sensor`, `_client_id`, `_source_table`) to the batch
   - Appends a `_meta` column containing a JSON object with config-time and periodic
     decorator values (BC-2.15.009)
   - Uses `arrow::compute` to append columns without re-allocating existing data

4. Implement periodic decorator cache in `crates/prism-storage/src/decorators.rs`:
   - Periodic values stored in `decorators` CF: key=decorator_key,
     value=`{value_json}:{last_refreshed_ts}`
   - `get_periodic(key: &str, ttl: u64) -> Result<serde_json::Value, PrismError>`:
     if `now - last_refreshed_ts > ttl`, refresh from source and update CF; otherwise
     return cached value (BC-2.15.010)
   - `PeriodicSource` enum: `Static(Value)` (test), `McpTool { tool: String }` (runtime)

5. Implement filterable vs. non-filterable distinction for WHERE clause pushdown:
   - `VirtualField` columns are registered in the DataFusion schema with standard
     Arrow types; the query planner includes them in pushdown predicates normally
   - `_meta` is registered as `DataType::Utf8` (JSON string); reject predicates
     referencing `_meta.*` sub-fields with `Err(PrismError::MetaFieldNotFilterable)`
     mapping to E-QUERY-xxx (BC-2.15.009)

6. Implement internal table registration helpers in `crates/prism-storage/src/internal_tables.rs`:
   - Implement `InternalTableDescriptor` struct: table_name, StorageDomain, Arrow schema, capability gate
   - Implement `InternalTableRegistry::descriptors()` returning all internal tables:
     - `prism_schedules` → `StorageDomain::Schedules`
     - `prism_alerts` → `StorageDomain::Alerts`
     - `prism_cases` → `StorageDomain::Cases`
     - `prism_diff_results` → `StorageDomain::DiffResults`
     - `prism_audit` → `StorageDomain::AuditBuffer` (capability-gated, see task 7)
     (BC-2.15.011)
   - NOTE: The actual `TableProvider` trait impl (DataFusion) lives in `prism-query`,
     not `prism-storage`, to respect dependency layering (prism-storage must not
     depend on DataFusion). prism-storage provides the scan data via `StorageBackend`;
     prism-query wraps it in `RocksDbTableProvider` implementing `TableProvider`.

7. Implement `audit.read` capability gate for `prism_audit`:
   - `RocksDbTableProvider::scan()` checks the `QueryContext` for `audit.read` capability
   - If absent, return `Err(PrismError::CapabilityRequired { capability: "audit.read" })`
     mapping to E-QUERY-011 before any data is read (BC-2.15.011)

8. Implement scan truncation in `RocksDbTableProvider::scan()`:
   - Read `PRISM_MAX_INTERNAL_TABLE_SCAN` from env (default: 10,000)
   - If scan would return more rows, truncate at the limit
   - Set `_meta.scan_truncated = true` in the injected decorator for the batch (BC-2.15.011)

## Acceptance Criteria

**AC-1:** Given a query result `RecordBatch`, When `DecoratorInjector::inject()` runs with
a `QueryContext` containing `sensor="crowdstrike"` and `client_id="acme"`, Then the output
batch has columns `_sensor`, `_client_id`, `_source_table`, and `_meta`. (BC-2.15.009)

**AC-2:** Given a WHERE clause `WHERE _sensor = 'crowdstrike'`, When the query plan is
built, Then the predicate pushes down correctly using the `_sensor` virtual field column.
Given a WHERE clause `WHERE _meta.client_id = 'acme'`, When the query plan is built, Then
`Err(PrismError::MetaFieldNotFilterable)` is returned. (BC-2.15.009)

**AC-3:** Given a periodic decorator with `ttl_seconds=60` last refreshed 61 seconds ago,
When `get_periodic()` is called, Then the value is refreshed from its source and the new
value is returned. (BC-2.15.010)

**AC-4:** Given `prism_alerts` is registered as an internal table and contains 3 alert
records, When `SELECT * FROM prism_alerts` is executed, Then 3 rows are returned with
the alert fields. (BC-2.15.011)

**AC-5:** Given a client context without the `audit.read` capability, When
`SELECT * FROM prism_audit` is executed, Then `Err(PrismError::CapabilityRequired)`
(E-QUERY-011) is returned before any data is read. (BC-2.15.011)

**AC-6:** Given a `prism_schedules` table with 15,000 entries and `PRISM_MAX_INTERNAL_TABLE_SCAN=10000`,
When a full scan is executed, Then 10,000 rows are returned and `_meta.scan_truncated = true`
is set on the result batch.

## Architecture Compliance Rules

- `RocksDbTableProvider` MUST implement the DataFusion `TableProvider` trait exactly —
  no wrapper or adapter shim; DataFusion must be able to register and scan it natively
- Virtual field columns (`_sensor`, `_client_id`, `_source_table`) MUST be appended to
  existing batches, not prepended — downstream code assumes data columns come first
- The `audit.read` capability check MUST happen before any RocksDB scan begins — fail
  fast, never leak partial data to unauthorized callers
- Periodic decorator refresh MUST write back to the `decorators` CF atomically — use
  `put_batch` with key and timestamp together
- `PRISM_MAX_INTERNAL_TABLE_SCAN` MUST be read from the environment at startup and cached;
  do not re-read on every query

## Notes

- The Arrow version (53) must align with the version used in the query engine
  crate (`prism-query`) — do not upgrade independently. DataFusion `TableProvider`
  implementation for internal tables lives in prism-query, not this crate.
- When appending columns to an existing `RecordBatch`, use
  `RecordBatch::try_new(schema, columns)` with the merged schema rather than Arrow's
  projection primitives — it is simpler and avoids schema evolution complexity.
- The `_meta` column should be a single `Utf8` (JSON string) column, not a struct
  column — JSON string avoids Arrow nested type complexity while remaining
  machine-parseable by downstream tools.
- `PRISM_MAX_INTERNAL_TABLE_SCAN` has no upper bound validation — if set to
  `usize::MAX` by an operator, the scan will run until OOM; document this in the
  operator guide as a footgun.
- Internal tables (prism_schedules, prism_alerts, prism_cases, prism_diff_results)
  will return empty results until the corresponding subsystem stories (S-4.01, S-4.05,
  S-4.06, S-4.02) populate their column families. The `InternalTableDescriptor`
  defines a fixed Arrow schema per table; the RocksDB CFs exist from S-2.01 (all 14
  CFs created at init) but contain no records until later waves write to them.
