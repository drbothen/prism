# S-2.03 — prism-storage: Decorators and Internal Tables

## Narrative

As a Prism platform implementer, I want the decorator data structures, three-phase
decoration model, and internal table descriptors defined in prism-storage and prism-core,
so that the query engine layer (prism-query, S-3.02) can inject `_meta` envelope fields
and `_sensor`/`_client`/`_source_table` virtual fields into every result set without
coupling the storage layer to DataFusion or Arrow.

## Objective

Implement the pure data structures and storage-layer logic for decorators and internal
table descriptors. This story deliberately stops at the boundary where DataFusion
appears: it defines `DecoratorContext`, `VirtualField`, `InternalTableDescriptor`, the
`DecorationStore`, and the RocksDB-backed periodic decorator cache. It does NOT
implement `TableProvider`, Arrow `RecordBatch` manipulation, or any DataFusion type —
those live in S-3.02.

The split is:
- **This story (S-2.03, prism-storage/prism-core):** types, trait, cache, registration
  metadata
- **S-3.02 (prism-query):** `RocksDbTableProvider` (implements DataFusion
  `TableProvider`), `RecordBatch` construction, `_meta` envelope injection

## Tasks

### Phase 1 — prism-core type definitions (BC-2.15.009, BC-2.15.010)

1. Add `VirtualField` enum to `crates/prism-core/src/virtual_fields.rs` (implements
   BC-2.15.009 virtual field category):
   ```rust
   #[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
   pub enum VirtualField {
       Sensor,        // maps to column name "_sensor"
       Client,        // maps to column name "_client"
       SourceTable,   // maps to column name "_source_table"
   }

   impl VirtualField {
       pub fn column_name(&self) -> &'static str {
           match self {
               VirtualField::Sensor => "_sensor",
               VirtualField::Client => "_client",
               VirtualField::SourceTable => "_source_table",
           }
       }
   }
   ```
   This enum lives in prism-core and has no Arrow or DataFusion dependencies.

2. Add `DecoratorContext` struct to `crates/prism-core/src/decorator_context.rs`
   (implements BC-2.15.009 decorator field category and BC-2.15.010 three phases):
   ```rust
   #[derive(Debug, Clone, Default, serde::Serialize, serde::Deserialize)]
   pub struct DecoratorContext {
       // Phase 1 (config-time) fields
       pub client_name: Option<String>,
       pub prism_version: Option<String>,

       // Phase 2 (query-time) fields
       pub analyst_id: Option<String>,
       pub query_source: Option<String>,    // "interactive" | "schedule:{name}" | "pack:{pack}.{query}"
       pub sensor_instance: Option<String>,

       // Phase 3 (periodic) fields
       pub sensor_health_status: Option<String>,
   }
   ```
   - All fields are `Option<String>`; missing values produce JSON `null` (BC-2.15.009
     error cases: decorator injection cannot fail).
   - `client_name` carries the human-readable name from TOML, not the TenantId.

3. Add `InternalTableDescriptor` struct to
   `crates/prism-core/src/internal_table_descriptor.rs` (implements BC-2.15.011
   registration metadata):
   ```rust
   #[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
   pub struct InternalTableDescriptor {
       /// PrismQL table name (underscore-delimited, e.g., "prism_alerts")
       pub table_name: &'static str,
       /// The RocksDB StorageDomain backing this table (None for non-RocksDB tables)
       pub domain: Option<StorageDomain>,
       /// Ordered list of columns and their types (prism-core ColumnType, not Arrow)
       pub columns: Vec<(String, ColumnType)>,
       /// Whether the table requires the `audit.read` capability
       pub requires_audit_read: bool,
       /// Whether the table is backed by RocksDB (true) or an alternative store (false).
       /// prism_aliases uses AliasStore (TOML-backed); all others use RocksDB.
       pub rocksdb_backed: bool,
   }
   ```
   `ColumnType` uses the existing prism-core type (no Arrow dependency).

4. Extend `ColumnType` in `crates/prism-core/src/column_type.rs` (or existing location)
   to ensure it covers all types needed by internal table schemas:
   - `Text`, `Int64`, `UInt64`, `Float64`, `Bool`, `Timestamp`, `Bytes` variants must
     exist (add any that are missing). This enum is the schema language shared across
     prism-core and prism-storage; Arrow conversion happens only in prism-query.

### Phase 2 — DecorationStore (BC-2.15.010)

5. Implement `DecorationStore` in `crates/prism-storage/src/decorators.rs`:
   ```rust
   pub struct DecorationStore {
       /// Phase 1: keyed by TenantId, updated on config load / reload
       config_time: Arc<RwLock<HashMap<TenantId, DecoratorContext>>>,
       /// Phase 3: periodic cache backed by RocksDB `decorators` CF
       backend: Arc<dyn StorageBackend>,
   }
   ```
   - `store_config_time(tenant: TenantId, ctx: DecoratorContext)`: write to the
     `config_time` map; called at startup and on config reload (BC-2.15.010 Phase 1).
   - `get_config_time(tenant: &TenantId) -> Option<DecoratorContext>`: read from the
     in-memory map (never panics; missing tenant returns `None`).
   - `store_periodic(tenant: &TenantId, ctx: &DecoratorContext) -> Result<(), PrismError>`:
     serialize with `bincode::encode_to_vec` and write to the `decorators` CF with key
     `periodic:{tenant_id}`. Persists across restarts (BC-2.15.010 Phase 3).
   - `load_periodic(tenant: &TenantId) -> Result<Option<DecoratorContext>, PrismError>`:
     prefix-scan the `decorators` CF for `periodic:{tenant_id}`, deserialize with
     `bincode::decode_from_slice`. Returns `None` if no cached value exists yet
     (EC-15-039: first query before first periodic refresh returns null periodic fields).
   - `merge(config_time: &DecoratorContext, query_time: &DecoratorContext,
     periodic: Option<&DecoratorContext>) -> DecoratorContext`: apply fields in
     config-time < query-time < periodic priority order (BC-2.15.010 invariant:
     last-write-wins, periodic > query-time > config-time).

6. Implement `E-DECOR-001` and `E-DECOR-002` handling in `decorators.rs`
   (BC-2.15.010 error cases):
   - `E-DECOR-001` (periodic refresh fails): `store_periodic` failure is logged as
     `tracing::warn!`; caller continues using the last successfully loaded periodic
     value (stale-value pattern).
   - `E-DECOR-002` (config-time decorator references missing config field): caller
     (config loader, not this module) sets the field to `None` and logs a warning;
     `DecoratorContext` accepts `None` without error.

### Phase 3 — Internal table registry (BC-2.15.011)

7. Implement the internal table registry in
   `crates/prism-storage/src/internal_tables.rs`:
   - Define `INTERNAL_TABLES: &[InternalTableDescriptor]` as a static slice with all
     seven entries from BC-2.15.011:
     - `prism_alerts` → `StorageDomain::Alerts`
     - `prism_cases` → `StorageDomain::Cases`
     - `prism_rules` → `StorageDomain::DetectionRules`
     - `prism_schedules` → `StorageDomain::Schedules`
     - `prism_diff_results` → `StorageDomain::DiffResults`
     - `prism_audit` → `StorageDomain::AuditBuffer`, `requires_audit_read: true`
     - `prism_aliases` → **AliasStore-backed** (NOT RocksDB). Per BC-2.11.008, aliases
       persist in `aliases.toml` (S-3.04), not in a RocksDB CF. The `prism_aliases`
       internal table descriptor uses a flag `rocksdb_backed: false` to signal that
       S-3.02's TableProvider should read from the in-memory AliasStore (loaded from
       TOML) rather than `StorageBackend::scan()`. The `aliases` StorageDomain variant
       and CF exist but are reserved for future use (not used by alias storage).
   - `get_descriptor(table_name: &str) -> Option<&'static InternalTableDescriptor>`:
     linear scan of `INTERNAL_TABLES` returning the matching descriptor.
   - `all_descriptors() -> &'static [InternalTableDescriptor]`: return the full slice
     (used by `explain_query` in S-3.02 to populate available sources).

8. Implement scan-limit enforcement in `internal_tables.rs`
   (BC-2.15.011 internal table query semantics):
   - `fn scan_limit() -> usize`: reads `PRISM_MAX_INTERNAL_TABLE_SCAN` from the
     environment; if absent or unparseable, defaults to `50_000`.
   - This is a pure function with no RocksDB access. The actual truncation is enforced
     in S-3.02 when iterating RocksDB results. This story only provides the helper
     so the limit value is in one place and testable.

9. Implement capability gate logic in `internal_tables.rs` (BC-2.15.011 E-QUERY-011):
   - `fn check_table_access(descriptor: &InternalTableDescriptor, capabilities: &ClientCapabilities) -> Result<(), PrismError>`:
     - If `descriptor.requires_audit_read` is true AND `capabilities.audit_read` is not
       `Allow`, return `Err(PrismError::AuditTableAccessDenied)` which maps to E-QUERY-011.
     - Otherwise return `Ok(())`.
   - `ClientCapabilities` is the existing type from prism-core (S-1.02).
   - Error message (set in prism-core's `PrismError` Display): "Audit table requires
     audit.read capability. Grant via prism.toml [clients.{id}.capabilities]."
   - Note: write-query rejection (E-QUERY-010, "Internal tables are read-only via
     PrismQL") is enforced in prism-query (S-3.02) at the SQL parse step, not here.

### Phase 4 — Column schemas for internal tables (BC-2.15.011)

10. Define column schemas for each internal table in `internal_tables.rs` as private
    helper functions, each returning `Vec<(String, ColumnType)>`:
    - `alerts_columns()` — fields sufficient to satisfy `SELECT * FROM prism_alerts`:
      `alert_id (Text)`, `severity_id (UInt64)`, `device_ip (Text)`,
      `device_hostname (Text)`, `client_id (Text)`, `created_at (Timestamp)`,
      `rule_id (Text)`
    - `cases_columns()` — `case_id (Text)`, `title (Text)`, `severity_id (UInt64)`,
      `client_id (Text)`, `created_at (Timestamp)`, `status (Text)`
    - `rules_columns()` — `rule_id (Text)`, `name (Text)`, `client_id (Text)`,
      `enabled (Bool)`, `created_at (Timestamp)`
    - `schedules_columns()` — `schedule_id (Text)`, `name (Text)`, `client_id (Text)`,
      `query (Text)`, `interval_secs (UInt64)`, `last_run_at (Timestamp)`
    - `diff_results_columns()` — metadata-only per BC-2.15.011: `query_hash (Text)`,
      `client_id (Text)`, `previous_results_hash (Text)`, `epoch (UInt64)`,
      `counter (UInt64)`, `last_diff_time (Timestamp)`
    - `audit_columns()` — `trace_id (Text)`, `timestamp_ns (UInt64)`,
      `operation (Text)`, `client_id (Text)`, `analyst_id (Text)`, `outcome (Text)`,
      `capability (Text)`
    - `aliases_columns()` — `alias_id (Text)`, `alias (Text)`, `expansion (Text)`,
      `client_id (Text)`, `created_at (Timestamp)`
    These schemas determine what S-3.02 materializes into Arrow RecordBatches; they must
    match the bincode-serialized structs already written by prism-storage domain operations.

## Acceptance Criteria

**AC-1:** Given a Prism process that has started with config loaded, When
`DecorationStore::get_config_time(tenant_id)` is called, Then it returns a
`DecoratorContext` with `client_name` and `prism_version` populated and all Phase 2 and
Phase 3 fields absent (`None`). (traces to BC-2.15.010 postcondition — Phase 1
config-time decorator populated at startup from TOML)

**AC-2:** Given a completed query execution with analyst session, When
`DecoratorContext::merge(config_time, query_time, None)` is called with `analyst_id =
Some("joshua")` and `query_source = Some("interactive")` in the query-time context,
Then the merged result has `analyst_id = Some("joshua")` and `query_source =
Some("interactive")` with Phase 1 fields carried through unchanged. (traces to
BC-2.15.010 postcondition — Phase 2 query-time fields not cached; computed per-query)

**AC-3:** Given a scheduled query execution with no analyst session, When
`DecoratorContext` is populated for Phase 2, Then `analyst_id = None` and
`query_source = Some("schedule:{schedule_name}")` per EC-15-034. (traces to BC-2.15.009
edge case EC-15-034 and BC-2.15.010 Phase 2 postcondition)

**AC-4:** Given a periodic decorator stored via `store_periodic(tenant, ctx)` and a
Prism restart, When `load_periodic(tenant)` is called after startup, Then the stored
`DecoratorContext` is returned with all fields intact (round-trip via bincode + RocksDB
`decorators` CF). (traces to BC-2.15.010 postcondition — Phase 3 periodic decorators
cached in RocksDB `decorators` CF for persistence across restarts)

**AC-5:** Given three-phase contexts where config-time has `client_name = Some("Acme")`
and periodic has `sensor_health_status = Some("healthy")` and query-time has no
overlapping keys, When `DecorationStore::merge` is called, Then the result contains all
three values. When config-time and periodic both set a hypothetical key, periodic value
wins per the invariant. (traces to BC-2.15.010 invariant — last-write-wins: periodic >
query-time > config-time)

**AC-6:** Given `store_periodic` returns an error (e.g., RocksDB CF write failure),
When the refresh task encounters the error, Then it logs a `tracing::warn!` event and
the last successfully cached periodic value is returned on subsequent `load_periodic`
calls (stale-value pattern). (traces to BC-2.15.010 error case E-DECOR-001)

**AC-7:** Given `get_descriptor("prism_alerts")` is called, When the descriptor is
returned, Then `table_name == "prism_alerts"`, `domain == StorageDomain::Alerts`,
`requires_audit_read == false`, and `columns` contains `("alert_id", ColumnType::Text)`
as the first entry. (traces to BC-2.15.011 postcondition — each queryable domain
registered with underscore-delimited name)

**AC-8:** Given `get_descriptor("prism_audit")` is called, When the descriptor is
returned, Then `requires_audit_read == true`. (traces to BC-2.15.011 error case
E-QUERY-011 — audit table requires audit.read capability)

**AC-9:** Given `check_table_access(audit_descriptor, capabilities)` where capabilities
does NOT have `audit.read = Allow`, When the function is called, Then it returns
`Err(PrismError::AuditTableAccessDenied)` and the error Display contains the string
"audit.read capability". (traces to BC-2.15.011 error case E-QUERY-011)

**AC-10:** Given `check_table_access(alerts_descriptor, capabilities)` for any
capability set, When the function is called, Then it returns `Ok(())` because
`prism_alerts` does not require `audit.read`. (traces to BC-2.15.011 postcondition —
internal tables are registered and accessible)

**AC-11:** Given `PRISM_MAX_INTERNAL_TABLE_SCAN` is not set in the environment, When
`scan_limit()` is called, Then it returns `50_000`. Given the env var is set to `"1000"`,
Then it returns `1000`. Given it is set to `"abc"` (invalid), Then it returns `50_000`.
(traces to BC-2.15.011 — internal table scans bounded by configurable limit with
`PRISM_MAX_INTERNAL_TABLE_SCAN`)

**AC-12:** Given `VirtualField::Sensor.column_name()`, When called, Then it returns
`"_sensor"`. Given `VirtualField::Client.column_name()`, Then `"_client"`. Given
`VirtualField::SourceTable.column_name()`, Then `"_source_table"`. (traces to
BC-2.15.009 postcondition — virtual fields are underscore-prefixed queryable columns
`_sensor`, `_client`, `_source_table`)

**AC-13:** Given `all_descriptors()` is called, When the slice is inspected, Then it
contains exactly 7 entries, one per BC-2.15.011 registered table name:
`prism_alerts`, `prism_cases`, `prism_rules`, `prism_schedules`, `prism_diff_results`,
`prism_audit`, `prism_aliases`. (traces to BC-2.15.011 postcondition — all seven
domains registered)

**AC-14:** Given `diff_results_columns()`, When the column list is inspected, Then it
contains exactly `query_hash`, `client_id`, `previous_results_hash`, `epoch`, `counter`,
`last_diff_time` and does NOT contain any column named `previous_results` (raw payload
excluded per BC-2.15.011). (traces to BC-2.15.011 postcondition — `prism_diff_results`
exposes DiffState metadata columns only; raw sensor data inside `previous_results` is
NOT exposed as queryable columns)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Decorator definitions and storage | `prism-core` + `prism-storage` (SS-15) | Mixed |
| Internal virtual tables | `prism-query` (SS-11) | Mixed |

`prism-core` defines decorator types; `prism-storage` persists them in the `decorators`
column family (AD-004). `prism-query` registers internal virtual tables (`prism_metadata`,
`prism_schedules`, etc.) as DataFusion table providers. Decorator logic is pure; RocksDB
reads/writes are effectful. Per `architecture/module-decomposition.md`, `prism-query`
depends on `prism-storage` via `StorageBackend` trait.

## Architecture Compliance Rules

- **prism-storage MUST NOT depend on DataFusion or Arrow.** This is the hard boundary
  established across 28 adversarial passes. Any Arrow or DataFusion import in
  `prism-storage/Cargo.toml` is an architectural violation and must be rejected in PR
  review.

- **VirtualField enum lives in prism-core, not prism-storage.** The field names
  `_sensor`, `_client`, `_source_table` must be accessible to both prism-storage and
  prism-query without creating a circular dependency. prism-core is the correct home.

- **InternalTableDescriptor uses prism-core ColumnType, not Arrow DataType.** The Arrow
  schema for DataFusion is derived from `InternalTableDescriptor.columns` inside
  prism-query (S-3.02). Column types are declared here using the shared prism-core enum.

- **DecoratorContext fields are all Option<String>.** Never panic on missing context;
  inject null per BC-2.15.009 error cases. Partial decoration is never acceptable
  (BC-2.15.009 invariant: all decorator fields present in `_meta`, never partial).

- **bincode 2.x only.** Use `bincode::encode_to_vec` / `bincode::decode_from_slice`
  (v2 API). Do not mix with bincode v1 API. Encoding format must be consistent with
  the rest of prism-storage (established in S-2.01).

- **DecorationStore uses StorageBackend trait, not RocksDbBackend directly.** All
  `decorators` CF operations go through the trait interface so tests can inject
  `InMemoryBackend`.

- **The `decorators` CF is one of the 15 CFs created by S-2.01.** Do not create any
  new column families. This story reads and writes the `decorators` CF only.

- **Capability gate belongs in prism-storage.** `check_table_access` returns a typed
  error. The MCP tool layer (prism-query) calls this before initiating the scan. The
  storage module enforces the rule; the MCP layer enforces the tool availability.

- **Write-query rejection (E-QUERY-010) is enforced in prism-query, not here.** This
  story defines the descriptors (which are read-only declarations); the parser-level
  rejection of INSERT/UPDATE/DELETE belongs in S-3.02.

- **`prism-storage` depends on `prism-core` and no other internal crates** (rule
  carried forward from S-2.01).

## Previous Story Intelligence

S-2.01 established `StorageBackend`, `RocksDbBackend`, `InMemoryBackend`, and all 15
column families including the `decorators` CF. Use `InMemoryBackend` for all unit tests
in this story (add `prism-storage = { ..., features = ["test-utils"] }` as a
dev-dependency in prism-core tests if needed for decorator round-trip tests; otherwise
test via InMemoryBackend directly in prism-storage's own test suite).

S-2.02 established the pattern of using the `StorageBackend` trait for all CF
operations; follow the same pattern here — `DecorationStore` holds an
`Arc<dyn StorageBackend>`, never a concrete `RocksDbBackend`.

S-1.02 defined `ClientCapabilities` and the capability model that `check_table_access`
consumes. Confirm the `audit_read` field exists in `ClientCapabilities` before
implementing the gate; if the field is not yet present, add it as part of this story's
prism-core changes.

## Dev Notes

- **Phase 2 context is never stored.** Query-time fields (`analyst_id`, `query_source`,
  `sensor_instance`) are always computed inline by the caller (prism-query, S-3.02) and
  passed as a `DecoratorContext` to `DecorationStore::merge`. Do not add a cache for
  query-time fields; it would defeat the "always fresh" invariant (BC-2.15.010).

- **`prism_diff_results` schema is metadata-only by design.** The `previous_results`
  blob (raw bincode-encoded sensor data) is never exposed as a queryable column. The
  BC explicitly states: "use `get_diff_results` MCP tool to inspect diff content." Do
  not add a `previous_results` column to `diff_results_columns()` under any
  circumstances.

- **`prism_audit` is always read-only at the storage level.** The `audit_buffer` CF is
  append-only (DI-004). `check_table_access` enforces the capability gate; the
  descriptor itself must not offer any write path.

- **EC-15-039 handling (first query before first periodic refresh):** `load_periodic`
  returns `None` if no entry exists in the `decorators` CF for the tenant. The caller
  (S-3.02) passes `None` to `merge`, which produces a `DecoratorContext` with only
  config-time and query-time fields. This is correct; the Phase 3 field
  `sensor_health_status` will be `None` in the merged result, serialized as JSON `null`
  in `_meta`. No error is returned.

- **EC-15-040 (50 clients, bounded concurrency for periodic refresh):** The bounded
  concurrency constraint (max 8 concurrent health checks) is enforced in the periodic
  refresh task, which lives in prism-query (S-3.02). This story provides only the
  `store_periodic`/`load_periodic` primitives that the refresh task will call.

- **scan_limit is a pure function.** It reads an env var and returns a `usize`. It has
  no side effects and is trivially testable without a RocksDB instance. Keep it in
  `internal_tables.rs` so the value is co-located with the scan-limit logic, but do
  not give it access to the backend.

- **`query_source` format strings** (for AC-3 and implementer clarity):
  - Interactive: `"interactive"`
  - Scheduled: `"schedule:{schedule_name}"` (e.g., `"schedule:check_alerts"`)
  - Pack: `"pack:{pack_name}.{query_name}"` (e.g., `"pack:incident-response.recent_detections"`)
  These are set by prism-query in S-3.02; `DecoratorContext` just holds the string.
