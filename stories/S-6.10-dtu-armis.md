# S-6.10 — prism-dtu-armis: DTU for Armis Centrix API — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Armis Centrix API
at fidelity level L2 (stateful), so that integration tests for `armis_devices`,
`armis_device_activity`, `armis_alerts`, AQL pass-through, timestamp fallback, and
device tag write paths can run deterministically without real Armis credentials.

## Objective

Implement `prism-dtu-armis` — an L2 stateful behavioral clone of the Armis Centrix API.
The clone implements all 6 in-scope endpoints (4 read, 2 write), accepts AQL strings
without parsing (logs them for assertion), includes a fixture with null primary
timestamp to exercise Prism's timestamp fallback logic, maintains a stateful device tag
store, and uses BearerStatic auth (the simplest auth model in the sensor set). See
`dtu-assessment.md §3.4` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-armis/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `ArmisClone` struct implementing `BehavioralClone`:
   - `start()`: binds ephemeral port, spawns axum router with `LatencyLayer` + `FailureLayer`
   - `reset()`: calls `state.reset()`

3. Create `src/state.rs` — `ArmisState`:
   - `tag_store: Mutex<HashMap<String, HashSet<String>>>` — `device_id → {tag_keys}`
   - `aql_log: Mutex<Vec<String>>` — captured AQL strings in order received
   - `aql_log()` test API: returns `Vec<String>` for integration test assertions
   - `reset()` clears both stores

4. Bearer auth middleware:
   - All routes except `POST /dtu/*` must validate `Authorization: Bearer {non-empty}` header
   - If absent: 403 `{"error": "invalid or missing bearer token", "code": 403}`
   - Note: Armis returns 403 (not 401) for missing token — per `dtu-assessment.md §3.4`

5. Create `src/routes/devices.rs`:
   - `GET /api/v1/devices` or `POST /api/v1/devices`: AQL-forwarded device query
     - Query/body param: `aql` (string) — accepted but NOT parsed; stored in `aql_log`
     - Pagination: page-based (`page`, `size` query params)
     - Response: `{"data": {"devices": [...], "total": N, "page": N}}`
     - Loads fixture from `fixtures/devices.json`
     - Merges tag state from `tag_store`
     - **Key fixture requirement:** at least one device must have `last_seen: null` and
       `first_seen: "2024-01-15T10:00:00Z"` to exercise Prism's timestamp fallback logic
   - `GET /api/v1/devices/{device_id}/activity`: device activity log
     - Response: `{"data": {"activities": [...], "total": N}}`
     - Loads fixture from `fixtures/device-activity.json`
   - `GET /api/v1/devices/{device_id}/risk`: device risk score
     - Response: `{"data": {"device_id": "...", "risk_score": N, "risk_factors": [...]}}`
     - Loads risk score from `fixtures/devices.json` (column on device record)

6. Create `src/routes/alerts.rs`:
   - `GET /api/v1/alerts`: alert / policy violation list
     - Pagination: page-based
     - Response: `{"data": {"alerts": [...], "total": N}}`
     - Loads fixture from `fixtures/alerts.json`

7. Create `src/routes/tags.rs` (same pattern as Claroty):
   - `POST /api/v1/devices/{device_id}/tags/`: add tag; insert into `tag_store`
     - Response: 201 `{"device_id": "...", "tag_key": "...", "status": "added"}`
   - `DELETE /api/v1/devices/{device_id}/tags/{tag_key}`: remove tag
     - Response: 200 `{"status": "removed"}` or 404 if tag not found

8. Create fixture files:
   - `fixtures/devices.json`: 25 device objects: `device_id`, `name`, `ip_address`,
     `mac_address`, `type`, `manufacturer`, `os_name`, `os_version`, `risk_score` (0-100),
     `risk_factors`, `last_seen`, `first_seen`, `network_id`, `site`, `tags`
     - **Mandatory:** device `"d-001"` has `last_seen: null` and `first_seen: "2024-01-15T10:00:00Z"`
     - **Mandatory:** device `"d-002"` has both `last_seen` and `first_seen` populated
       (to contrast with the fallback case)
   - `fixtures/device-activity.json`: 10 activity records: `activity_id`, `device_id`,
     `activity_type`, `timestamp`, `details`
   - `fixtures/alerts.json`: 12 alert objects: `alert_id`, `name`, `severity`,
     `status`, `policy_name`, `device_id`, `created_at`, `updated_at`

9. Create `POST /dtu/aql-log` test API endpoint:
   - `GET /dtu/aql-log`: returns `{"aql_strings": [...]}` — the list of AQL strings
     received since last reset
   - This is the mechanism integration tests use to assert AQL push-down correctness
     (per `dtu-assessment.md §3.4` R-DTU-002 mitigation)

10. Create `tests/fidelity.rs`:
    - `#[tokio::test] fn armis_dtu_fidelity()`: starts `ArmisClone`, runs
      `FidelityValidator` against all 6 endpoints
    - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `GET /api/v1/devices?aql=in:type%3Dswitch` with a valid Bearer token,
Then the response is HTTP 200 with a `data.devices` array AND the received AQL string
`"in:type=switch"` is logged in `GET /dtu/aql-log` (AQL capture works).

**AC-2:** Given `fixtures/devices.json` contains device `"d-001"` with `last_seen: null`
and `first_seen: "2024-01-15T10:00:00Z"`, When the device detail is returned by the DTU,
Then the response JSON contains `last_seen: null` and a non-null `first_seen` — exercising
Prism's timestamp fallback path in the TOML spec.

**AC-3:** Given `POST /api/v1/devices/{device_id}/tags/` with `{"tag_key": "ot-critical"}`,
Then the response is HTTP 201 AND subsequent `GET /api/v1/devices` returns that device
with `"ot-critical"` in its `tags` array (stateful tagging).

**AC-4:** Given `DELETE /api/v1/devices/{device_id}/tags/ot-critical` after the tag is
added, Then the response is HTTP 200 `{"status": "removed"}` AND subsequent device
query returns the device without that tag.

**AC-5:** Given a request without `Authorization` header, Then the response is HTTP 403
`{"error": "invalid or missing bearer token", "code": 403}` — note: Armis uses 403
not 401, per API spec.

**AC-6:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then
HTTP 429 is returned — maps to `E-SENSOR-003`.

**AC-7:** Given `reset()` is called, Then the tag store is cleared, the AQL log is
cleared, and subsequent device queries return devices with empty `tags` arrays.

## Architecture Compliance Rules

- DTU clone MUST match the Armis Centrix API contract exactly (same endpoints, same
  request/response shapes, same status codes) as documented in `dtu-assessment.md §3.4`.
- DTU clone MUST NOT introduce new behaviors not present in the real Armis Centrix API.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution — all state is in-process.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — never compiled
  into production binaries.
- The AQL log (`GET /dtu/aql-log`) MUST capture every AQL string received, in order,
  without parsing or validation — the real Armis API passes AQL verbatim to its query engine.
- Armis auth MUST return 403 (not 401) for missing bearer token — this is per the
  Armis Centrix API spec and is intentionally different from Bearer-standard behavior.
- See `.factory/specs/architecture/dtu-assessment.md` for cross-cutting DTU rules.
- **Forbidden dependencies:** `prism-dtu-armis` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.
