# S-6.14 ŌĆö prism-dtu-threatintel: DTU for Threat Intel Aggregator ŌĆö L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the threat intelligence
aggregator at fidelity level L2 (stateful), modeling GreyNoise as primary with
VirusTotal and AbuseIPDB fixtures as secondary providers, so that integration tests for
the `threat_intel` infusion plugin can validate IP/domain/hash lookups, aggregated score
shapes, rate-limit behavior, and cache-hit semantics without hitting live threat intel APIs.

## Objective

Implement `prism-dtu-threatintel` ŌĆö an L2 stateful behavioral clone of the threat intel
aggregator infusion plugin's API surface. The DTU exposes unified lookup endpoints
returning the aggregated output shape (as consumed by Prism's infusion cache), not raw
per-provider formats. Fixture registry maps test IPs/domains/hashes to canned responses
covering malicious, benign, and unknown scenarios. See `dtu-assessment.md ┬¦3.6.1` for
full scope matrix.

## Tasks

1. Create `crates/prism-dtu-threatintel/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` ŌĆö `ThreatIntelClone` struct implementing `BehavioralClone`:
   - `start()`, `reset()`, `configure(config)`, `bound_addr()`
   - `configure()` handles `{"rate_limit_after": N}` and adding IPŌåÆfixture mappings
     `{"ip": "1.2.3.4", "fixture": "malicious"}`

3. Create `src/state.rs` ŌĆö `ThreatIntelState`:
   - `fixture_registry: Mutex<HashMap<String, FixtureKey>>` ŌĆö
     maps lookup values (IP/domain/hash) to fixture names
     - Pre-loaded defaults: `"45.55.100.1" ŌåÆ "malicious"`, `"8.8.8.8" ŌåÆ "benign"`,
       `"0.0.0.0" ŌåÆ "unknown"`, `"evil.example.com" ŌåÆ "malicious"`, `"safe.example.com" ŌåÆ "benign"`
   - `request_counter: AtomicU32` ŌĆö for rate-limit threshold
   - `reset()`: resets counter; restores default fixture registry (custom entries removed)

4. API key auth:
   - Accept `key` query parameter OR `Authorization: Bearer {key}` header
   - Any non-empty value accepted as valid
   - If both absent: return 401 `{"error": "missing API key", "code": 401}`

5. Create `src/routes/lookup.rs`:
   - `GET /v3/ip/{ip}`: IP address lookup
     - Look up `ip` in `fixture_registry`; return matching fixture response
     - If not found: return benign defaults
       `{"ip": "{ip}", "threat_score": 0, "threat_is_known_malicious": false, "threat_sources": []}`
     - Malicious fixture: `{"threat_score": 85, "threat_is_known_malicious": true, "threat_sources": ["greynoise", "abuseipdb"]}`
     - Benign fixture: `{"threat_score": 5, "threat_is_known_malicious": false, "threat_sources": ["greynoise"]}`
   - `GET /v3/domain/{domain}`: domain lookup ŌĆö same fixture dispatch as IP
   - `GET /v3/hash/{hash}`: file hash lookup ŌĆö returns VirusTotal-style shape:
     `{"hash": "{hash}", "threat_score": N, "threat_is_known_malicious": bool, "threat_sources": ["virustotal"]}`
   - All three endpoints: increment `request_counter`; apply rate-limit check

6. Rate-limit behavior:
   - When counter exceeds configured threshold: return 429 with `Retry-After: 30`
   - Configurable via `POST /dtu/configure {"rate_limit_after": N}`

7. Create `POST /dtu/configure`:
   - `{"rate_limit_after": N}` ŌĆö sets rate-limit threshold
   - `{"ip": "x.x.x.x", "fixture": "malicious"|"benign"|"unknown"}` ŌĆö adds to registry
   - Returns 200 on success

8. Create fixture files:
   - `fixtures/malicious-ip.json`: `{"threat_score": 85, "threat_is_known_malicious": true, "threat_sources": ["greynoise", "abuseipdb"], "greynoise_classification": "malicious", "abuseipdb_confidence_score": 90, "virustotal_detections": 45}`
   - `fixtures/benign-ip.json`: `{"threat_score": 5, "threat_is_known_malicious": false, "threat_sources": ["greynoise"], "greynoise_classification": "benign"}`
   - `fixtures/malicious-hash.json`: `{"threat_score": 95, "threat_is_known_malicious": true, "threat_sources": ["virustotal"], "virustotal_detections": 62, "virustotal_first_seen": "2024-03-01"}`

9. Create `tests/fidelity.rs`:
   - Tests IP malicious lookup, IP benign lookup, domain lookup, hash lookup
   - Tests 401 on missing API key
   - Tests 429 after rate-limit threshold
   - Tests `POST /dtu/configure` to add a custom IPŌåÆfixture mapping
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `GET /v3/ip/45.55.100.1` with a valid API key, Then the response contains
`threat_score: 85`, `threat_is_known_malicious: true`, and `threat_sources` including
`"greynoise"`.

**AC-2:** Given `GET /v3/ip/8.8.8.8` with a valid API key, Then the response contains
`threat_is_known_malicious: false` and `threat_score` below 20.

**AC-3:** Given `GET /v3/ip/1.1.1.1` (not in registry), Then the response contains the
benign defaults: `threat_score: 0`, `threat_is_known_malicious: false`,
`threat_sources: []`.

**AC-4:** Given `GET /v3/hash/{sha256}` for a pre-registered malicious hash, Then the
response includes `threat_sources` containing `"virustotal"` and `threat_score` above 80.

**AC-5:** Given both `key` query param and `Authorization` header absent, Then the
response is HTTP 401 `{"error": "missing API key", "code": 401}` ŌĆö maps to
`E-INFUSION-AUTH-001`.

**AC-6:** Given `POST /dtu/configure {"rate_limit_after": 3}`, When the 4th lookup
request arrives, Then the response is HTTP 429 with `Retry-After: 30`.

**AC-7:** Given `POST /dtu/configure {"ip": "10.0.0.1", "fixture": "malicious"}`, Then
`GET /v3/ip/10.0.0.1` returns the malicious fixture response (dynamic registry addition).

## Architecture Compliance Rules

- DTU clone MUST match the threat intel aggregator API contract exactly (same endpoints,
  same request/response shapes, same status codes) as documented in `dtu-assessment.md ┬¦3.6.1`.
- DTU clone MUST NOT introduce new behaviors not present in the real threat intel aggregator API.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution ŌĆö all state is in-process.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` ŌĆö never compiled
  into production binaries.
- See `.factory/specs/architecture/dtu-assessment.md` (or `dtu-assessment.md ┬¦4`) for
  cross-cutting DTU rules including ephemeral port binding and `BehavioralClone` contract.
- **Forbidden dependencies:** `prism-dtu-threatintel` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.

## Dev Notes

L2 (stateful) fidelity: the DTU maintains a mutable fixture registry and per-request
rate-limit counter. It does NOT implement the full multi-provider aggregation logic of
the real GreyNoise/VirusTotal/AbuseIPDB aggregator ŌĆö it returns fixture responses keyed
by a lookup value. The aggregated output shape (not per-provider formats) is the contract.
This approach ensures integration tests validate Prism's infusion plugin behavior, not
third-party provider behavior.

Fidelity level L2 was chosen (vs L4) because the threat intel aggregator's primary
integration risk is Prism's infusion cache semantics and error handling, not the exact
aggregation algorithm. The fixture registry covers all required test scenarios
(malicious/benign/unknown/rate-limit) without L4 complexity.
