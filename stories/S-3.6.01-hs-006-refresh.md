## Acceptance Criteria

### AC-001 — HS-006 frontmatter `behavioral_contracts` field lists current BC IDs (traces to BC-3.6.002 postcondition 2)

The refreshed `.factory/holdout-scenarios/HS-006-state-recovery.md` frontmatter contains:

No retired Phase 1b persistent-cursor BC IDs remain in the frontmatter.
TD-HOLDOUT-W2-002 is listed in the `closes_td` field.

### AC-002 — Scenario shape updated for multi-tenant harness context (traces to BC-3.6.002 postcondition 3)

HS-006 sub-scenarios are reframed around `N`-customer harness restart (not single-tenant
file-cursor resume). Specifically: sub-scenarios describe a process restart with prior state
in RocksDB across N customers; per-org dirty bits tracked; per-org eviction; cross-tenant
integrity preserved after restart. Retired Phase 1b scenario language (per-tenant cursor
files, FileStore, SIGTERM/SIGKILL patterns) is removed or updated to reflect the RocksDB +
harness model.

### AC-003 — Crash detection during restart exercised in at least one sub-scenario (traces to BC-3.6.002 postcondition 1 and 2)

At least one HS-006 sub-scenario explicitly tests: harness process restart mid-operation;
`CloneCrashed` detection within 1s of task exit (BC-3.6.002 postcondition 1); crash does
not corrupt other orgs' state (BC-3.6.002 postcondition 3). The sub-scenario names the
specific `HarnessError::CloneCrashed` variant as the expected error.

### AC-004 — Per-org failure injection used as crash trigger (traces to BC-3.6.001 postcondition 1)

At least one sub-scenario uses `inject_failure(org_slug, dtu_type, FailureMode::InternalError)`
to trigger a clone crash, then verifies: only that org's clone is affected; other orgs
continue returning HTTP 200; harness reports `CloneCrashed` for the affected org only.
This exercises BC-3.6.001 per-org scoping guarantee.

### AC-005 — Cross-tenant integrity assertion defined (traces to BC-3.5.001 postcondition 2)

Each sub-scenario states an explicit cross-tenant integrity check: after N-customer restart,
devices from org A do not appear in org B's query results. The assertion cites
`devices(OrgA) ∩ devices(OrgB) = ∅` (BC-3.5.001 postcondition 2).
