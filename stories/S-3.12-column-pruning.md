# S-3.12 — prism-query: Column Pruning and Field Selection Push-Down

## Narrative

As a Prism query engine, I want to push the set of required columns to sensor APIs
that support field selection so that only the fields the query actually needs are
returned over the wire — reducing API response size, bandwidth, and normalization
overhead for large sensor result sets.

## Objective

Analyze the query AST to determine which columns are referenced across all clauses
(SELECT, WHERE, GROUP BY, ORDER BY, JOIN ON). For sensors that declare field selection
support in their spec, pass the used-column set as a fetch parameter. Handle the three
PrismQL modes distinctly: SQL analyzes all clauses, pipe uses the `fields` stage plus
upstream references, filter mode uses predicate columns (implicit SELECT * — no pruning
unless combined with pipe `fields`).

## Tasks

1. Extend `[[sensor.tables]]` in sensor spec schema:
   ```toml
   [[sensor.tables]]
   table_name = "detections"
   supports_field_selection = true        # new field; default false
   field_selection_param = "fields"       # API query param name; required if above is true
   ```
   Validate: `field_selection_param` is required when `supports_field_selection = true`.
   Validate: `field_selection_param` is a non-empty string.

2. Implement column usage analysis in `crates/prism-query/src/column_usage.rs`:
   - `extract_used_columns(plan: &QueryPlan) -> Option<HashSet<String>>`
   - Returns `None` when pruning is not possible (SELECT * or filter mode without
     explicit field list — callers treat None as "fetch all columns")
   - Returns `Some(columns)` when an explicit column set can be determined
   - For SQL mode:
     - Walk SELECT projection list (non-wildcard)
     - Walk WHERE predicates (column references)
     - Walk GROUP BY expressions (column references)
     - Walk ORDER BY expressions (column references)
     - Walk JOIN ON conditions (column references from each side)
     - Union all references across all clauses
   - For pipe mode:
     - If a `fields` stage exists: start from its explicit column list
     - Add columns referenced by upstream pipe stages (`where`, `stats`, `sort`,
       `join on`, `eval`) that appear before the `fields` stage
     - Union all
   - For filter mode: return `None` (always implicit SELECT *; no pruning)
   - Always include virtual fields in the used set:
     `_sensor`, `_client`, `_source_table`, `_source_type` (added automatically —
     not stripped by pruning even if not in the query)

3. Extend `SensorAdapter::fetch()` signature in `prism-sensors`:
   ```rust
   async fn fetch(
       spec: &SensorSpec,
       params: &QueryParams,
       auth: &dyn SensorAuth,
       fields: Option<Vec<String>>,   // new parameter
   ) -> Result<Vec<RecordBatch>, SensorError>
   ```
   - If `fields` is `Some(list)` and the table spec declares
     `supports_field_selection = true`: construct the API parameter
     (e.g., `?fields=severity,device_ip,hostname`) using `field_selection_param`
     from the spec
   - If `fields` is `None` or `supports_field_selection = false`: pass no field
     selection to the API (fetch all fields — existing behavior)

4. Wire column usage analysis into `fanout.rs`:
   - Before fan-out, call `extract_used_columns(&query_plan)` to get
     `used_columns: Option<HashSet<String>>`
   - Pass `used_columns.map(|s| s.into_iter().collect::<Vec<_>>())` as `fields`
     to each `SensorAdapter::fetch()` call
   - If the sensor table does not support field selection: `fields` is ignored by
     the adapter (it passes no field selection param) — no error, no special handling

5. Handle SELECT * correctly:
   - If the SQL query is `SELECT *` (or equivalent), `extract_used_columns` returns
     `None` and no field pruning is applied — all columns fetched
   - If `SELECT *` is combined with explicit predicates in WHERE: still returns
     `None` because the SELECT clause is a wildcard (we cannot prune columns that
     might appear in the output)
   - This is conservative: we never prune columns that could appear in output

6. Emit a DEBUG log entry when pruning is applied:
   ```
   [DEBUG query] field_selection push-down: crowdstrike/detections → 
     fields=[severity_id,device_ip,device_hostname] (was: all)
   ```

## Acceptance Criteria

**AC-1:** Given a sensor spec with `supports_field_selection = true` and
`field_selection_param = "fields"`, When a SQL query `SELECT severity, device_ip FROM
crowdstrike_detections WHERE severity_id >= 4` executes, Then the sensor API is called
with `?fields=severity,device_ip,severity_id` (union of SELECT + WHERE columns).

**AC-2:** Given `SELECT * FROM crowdstrike_detections`, When the query executes,
Then no `fields` parameter is added to the sensor API call — all columns fetched.

**AC-3:** Given a pipe query `FROM crowdstrike_detections | where severity_id >= 3 |
fields severity_id, device_ip`, When the query executes, Then the sensor API call
includes `?fields=severity_id,device_ip` (the `fields` stage columns, plus WHERE
column `severity_id` which is already in the set).

**AC-4:** Given a sensor spec with `supports_field_selection = false`, When a query
with an explicit column list executes against that sensor, Then no `fields` API
parameter is added — the optimization is silently skipped for that sensor.

**AC-5:** Given a JOIN query `SELECT a.cve_id, b.hostname FROM armis_vulnerabilities a
JOIN crowdstrike_hosts b ON a.device_ip = b.device_ip`, When the query executes,
Then armis gets `fields=[cve_id, device_ip]` and crowdstrike gets
`fields=[hostname, device_ip]` (each side gets only its relevant columns, plus the
JOIN key `device_ip`).

**AC-6:** Given a filter mode query `severity >= high AND device_ip = "10.0.1.50"`,
When the query executes, Then no field pruning is applied (filter mode → implicit
SELECT * → `extract_used_columns` returns None).

**AC-7:** Given the virtual fields `_sensor`, `_client`, `_source_table` are always
in the schema, When pruning is applied, Then these virtual fields are NOT listed in
the `fields` API parameter (they are injected by Prism, not fetched from the API).

## Architecture Compliance Rules

- Column pruning MUST be conservative: when in doubt (SELECT *, filter mode,
  ambiguous AST), return `None` and fetch all columns. It is better to over-fetch
  than to prune a column that a downstream DataFusion operation needs.
- Virtual fields (`_sensor`, `_client`, `_source_table`, `_source_type`) are injected
  by Prism AFTER the fetch — they MUST NOT appear in the `fields` API parameter sent
  to the sensor. Scrub them from the used_columns set before constructing the param.
- The `field_selection_param` value from the sensor spec is used verbatim as the
  API query parameter name — no normalization or escaping (sensor spec authors are
  responsible for correct values).
- The column list passed to the API MUST be deduplicated (use `HashSet` before
  converting to `Vec`) and consistently sorted (sort alphabetically) to enable
  reproducible behavior and cache-key stability.
- Adding `fields: Option<Vec<String>>` to `SensorAdapter::fetch()` changes the trait
  signature — all existing adapter call sites must be updated. Search for all
  `adapter.fetch(` call sites in `fanout.rs` and related tests.

## Notes

- The column pruning optimization is most impactful for wide-schema sensors (Claroty
  with 9 data sources, Armis with rich device metadata) where analysts typically only
  need 5–10 fields from a schema of 50+. Reducing the response payload by 80% can
  cut API latency significantly.
- Not all sensor APIs support field selection. The spec `supports_field_selection`
  flag makes this opt-in per table — sensor spec authors declare it only when the API
  actually supports it.
- The JOIN case (AC-5) is particularly important: both sides of a JOIN need the JOIN
  key column in their fetched field set, even if it's not in the final SELECT list.
  The AST walker MUST include JOIN ON column references in the used_columns set for
  both sides.
- Over time, as more sensor specs declare `supports_field_selection = true`,
  the optimization's coverage grows without code changes — pure spec-driven
  extensibility.
