# S-6.15 — prism-dtu-nvd: DTU for NVD/NIST CVSS API (L2 stateful fidelity)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the NVD API 2.0 at
fidelity level L2 (stateful), so that integration tests for the `vuln_context` infusion
plugin can validate single CVE fetch, bulk fetch with pagination, dual rate-limit buckets
(authenticated vs unauthenticated), cache-hit semantics (Prism should not call the DTU
twice for the same CVE), and 403/404 error paths without hitting the live NVD API.

## Objective

Implement `prism-dtu-nvd` — an L2 stateful behavioral clone of the NVD API 2.0. The
DTU implements the single `GET /rest/json/cves/2.0` endpoint for both single-CVE and
bulk fetches, maintains dual rate-limit buckets (5/30s unauthenticated, 50/30s
authenticated), pre-loads 10 fixture CVEs covering the full severity spectrum, tracks
per-CVE request counts for cache-hit assertion, and simulates the `apiKey` query parameter
auth model. See `dtu-assessment.md §3.6.2` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-nvd/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `tokio` with `time` feature (for 30-second rate-limit window)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `NvdClone` struct implementing `BehavioralClone`:
   - `start()`, `reset()`, `configure()`, `bound_addr()`
   - `request_count_for(&self, cve_id: &str) -> u32`: test API — returns request count
     for a specific CVE (used to assert cache-hit: count should be 1 after two queries
     if Prism caches correctly)

3. Create `src/state.rs` — `NvdState`:
   - `cve_registry: HashMap<String, CveRecord>` — pre-loaded from `fixtures/cves.json`
     at construction time (immutable after load)
   - `request_counters: Mutex<HashMap<String, u32>>` — `cve_id → request count`
   - `rate_limit_buckets: Mutex<HashMap<Option<String>, RateLimitBucket>>` —
     keyed by `apiKey` value (or `None` for unauthenticated)
   - `RateLimitBucket { count: u32, window_start: Instant, limit: u32 }`:
     - Unauthenticated (no `apiKey`): `limit: 5` per 30 seconds
     - Authenticated (`apiKey` present): `limit: 50` per 30 seconds
   - `reset()`: clears `request_counters`, resets all rate-limit buckets

4. Create `src/routes/cves.rs`:
   - `GET /rest/json/cves/2.0`: unified CVE endpoint
     - `apiKey` query param: optional; if present, use authenticated rate-limit bucket
     - If unauthenticated bucket exceeded: return 403
       `{"error": "Forbidden. Too many requests. (Unauthenticated rate limit exceeded.)"}`
     - If authenticated bucket exceeded: return 429
       `{"error": "Too many requests. Retry after current 30-second window expires."}`
     - If `apiKey` present but invalid (DTU configured with `auth_mode=reject`):
       return 403 `{"error": "Forbidden. apiKey not verified."}`
     
     **Single CVE fetch** (when `cveId` query param is present):
     - Look up `cve_id` in `cve_registry`
     - Increment `request_counters[cve_id]`
     - 404 if not found: `{"error": "CVE not found", "cveId": "{cveId}"}`
     - Response: `{"resultsPerPage": 1, "startIndex": 0, "totalResults": 1, "vulnerabilities": [{cve object}]}`
     
     **Bulk CVE fetch** (when `startIndex` + `resultsPerPage` params, no `cveId`):
     - Accepts `lastModStartDate` / `lastModEndDate` date range params (ignored — returns fixture subset)
     - Returns paginated results from `cve_registry`
     - `startIndex` and `resultsPerPage` control pagination
     - Response uses same format as single fetch but `totalResults` reflects full registry size

5. CVE fixture set (pre-loaded at construction from `fixtures/cves.json`):
   - 10 fixture CVEs covering the full CVSS spectrum:
     - `CVE-2024-0001`: CVSS 9.8 Critical, CISA KEV, CWE-89 (SQL Injection)
     - `CVE-2024-0002`: CVSS 7.5 High, no KEV
     - `CVE-2024-0003`: CVSS 5.3 Medium
     - `CVE-2024-0004`: CVSS 3.1 Low
     - `CVE-2024-0005`: CVSS 0.0 Informational
     - `CVE-2024-0006`: CVSS 9.8 Critical, CISA KEV, CWE-787 (OOB Write)
     - `CVE-2024-0007`: CVSS 8.8 High
     - `CVE-2024-0008`: CVSS 6.5 Medium
     - `CVE-2024-0009`: CVSS 4.3 Medium, patch available
     - `CVE-2024-0010`: CVSS 10.0 Critical, CISA KEV
   - CVE object shape (per NVD API 2.0):
     `{"cve": {"id": "CVE-2024-0001", "published": "...", "lastModified": "...", "vulnStatus": "Analyzed", "descriptions": [{"lang": "en", "value": "..."}], "metrics": {"cvssMetricV31": [{"cvssData": {"baseScore": 9.8, "baseSeverity": "CRITICAL"}, "exploitabilityScore": 3.9}]}, "weaknesses": [{"description": [{"lang": "en", "value": "CWE-89"}]}], "configurations": [], "references": [], "cisaKevVulnAdded": "2024-01-15"}}`

6. Create `GET /dtu/request-count/{cve_id}` test API:
   - Returns `{"cve_id": "...", "count": N}` — how many times that CVE was requested
   - Integration test asserts `count == 1` after two queries (Prism hit cache on second)

7. Create `tests/fidelity.rs`:
   - Tests single CVE fetch (CVSS 9.8, CISA KEV check)
   - Tests bulk CVE fetch with pagination
   - Tests 403 for unverified apiKey (auth_mode=reject)
   - Tests 404 for unknown CVE
   - Tests unauthenticated rate-limit (5 requests, 6th returns 403)
   - Tests cache-hit assertion via `GET /dtu/request-count`
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `GET /rest/json/cves/2.0?cveId=CVE-2024-0001` with a valid `apiKey`,
Then the response is HTTP 200 with `totalResults: 1` and `vulnerabilities[0].cve.id: "CVE-2024-0001"`
containing CVSS base score 9.8 and a CISA KEV date.

**AC-2:** Given the same CVE is requested twice, Then `GET /dtu/request-count/CVE-2024-0001`
returns `count: 2` (if Prism does not cache) or `count: 1` (if Prism hits cache after first
request). Integration test asserts `count == 1` to verify cache-hit behavior.

**AC-3:** Given `GET /rest/json/cves/2.0?cveId=CVE-9999-9999` (not in fixture), Then the
response is HTTP 404 `{"error": "CVE not found"}` — maps to empty CVSS in Prism's infusion.

**AC-4:** Given no `apiKey` parameter and 5 prior unauthenticated requests in a 30-second
window, When the 6th request arrives, Then the response is HTTP 403
`{"error": "Forbidden. Too many requests. (Unauthenticated rate limit exceeded.)"}`.

**AC-5:** Given `apiKey: "valid-key"` and 50 prior authenticated requests in a 30-second
window, When the 51st request arrives, Then the response is HTTP 429 — maps to
`E-INFUSION-RATE-001`.

**AC-6:** Given `auth_mode=reject` configured via `POST /dtu/configure`, When any request
with `apiKey` arrives, Then the response is HTTP 403
`{"error": "Forbidden. apiKey not verified."}` — maps to `E-INFUSION-AUTH-002`.

**AC-7:** Given `GET /rest/json/cves/2.0` with `startIndex=0&resultsPerPage=5`,
Then the response contains `resultsPerPage: 5`, `totalResults: 10`, and 5 CVE entries.

## Risk Mitigations

### R-DTU-009: NVD API rate limits

**Risk:** The NVD API enforces strict rate limits: 5 requests per 30 seconds without
an API key, 50 requests per 30 seconds with a key. Integration tests that hammer the
real NVD API would be throttled. The `prism-dtu-nvd` must accurately simulate 429
behavior including the `Retry-After` response header.

**Mitigation:** `prism-dtu-nvd` implements rate-limit state per virtual API key (keyed
by `apiKey` query parameter). Unauthenticated requests use a separate 5/30s bucket.
The `POST /dtu/configure` endpoint allows tests to pre-exhaust the rate limit bucket to
simulate throttling. No real NVD calls in CI. See `dtu-assessment.md §10 R-DTU-009`
for the full risk statement.
