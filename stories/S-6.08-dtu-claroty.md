# S-6.08 — prism-dtu-claroty: DTU for Claroty xDome API — L4 (adversarial)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Claroty xDome API
at fidelity level L4 (adversarial), so that integration tests for the `claroty_devices`,
`claroty_alerts`, `claroty_vulnerabilities`, and tag write-path can run deterministically
without live Claroty credentials or API access.

## Objective

Implement `prism-dtu-claroty` — a full L4 (adversarial) behavioral clone of the Claroty
xDome API. The clone implements all 7 in-scope endpoints (5 read via POST-body filtering,
2 write), supports `group_by` parameter behavior, maintains a stateful device tag store,
bootstraps fixture schemas from the reference implementation at
`.references/mcp-claroty-xdome/src/types/claroty.ts`, and can inject failure modes via
`FailureLayer`. See `dtu-assessment.md §3.2` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-claroty/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `ClarotyClone` struct implementing `BehavioralClone`:
   - Fields: `config: StubConfig`, `state: Arc<ClarotyState>`,
     `server_handle: Option<JoinHandle<()>>`, `bound_addr: Option<SocketAddr>`
   - `start()`: binds ephemeral port, spawns axum router with `LatencyLayer` + `FailureLayer`
   - `reset()`: calls `state.reset()`

3. Create `src/state.rs` — `ClarotyState`:
   - `tag_store: Mutex<HashMap<String, HashSet<String>>>` — `device_id → {tag_keys}`
   - `reset()` clears all tag state

4. Create `src/routes/devices.rs`:
   - `POST /api/v1/devices`: POST-body filter; accepts JSON body with optional
     `fields`, `filters`, `group_by`, `page`, `page_size`
     - Auth: validate `Authorization: Bearer {non-empty}` header; 401 if absent
     - When `group_by` is present: return only grouped field values (not full device objects)
     - Pagination: return `{"devices": [...], "total": N, "page": N}`
     - Merge tag state from `tag_store` into response device objects
     - Load fixture from `fixtures/devices.json`

5. Create `src/routes/alerts.rs`:
   - `POST /api/v1/alerts`: alert list with filter params
     - Response: `{"alerts": [...], "total": N, "page": N}`
     - Load fixture from `fixtures/alerts.json`
   - `POST /api/v1/alerts/{alert_id}/devices`: devices associated with an alert
     - Response: `{"devices": [...], "total": N}`
     - Load fixture from `fixtures/alerted-devices.json`

6. Create `src/routes/vulnerabilities.rs`:
   - `POST /api/v1/vulnerabilities`: vulnerability inventory
     - Response: `{"vulnerabilities": [...], "total": N, "page": N}`
     - Load fixture from `fixtures/vulnerabilities.json`
   - `POST /api/v1/vulnerabilities/{vuln_id}/devices`: devices affected by vulnerability
     - Response: `{"devices": [...], "total": N}`
     - Load fixture from `fixtures/vulnerability-devices.json`

7. Create `src/routes/tags.rs`:
   - `POST /api/v1/devices/{device_id}/tags/`: add tag
     - Body: `{"tag_key": "...", "tag_value": "..."}` (or simple string key)
     - Insert into `tag_store[device_id]`
     - Response: 201 `{"device_id": "...", "tag_key": "...", "status": "added"}`
   - `DELETE /api/v1/devices/{device_id}/tags/{tag_key}`: remove tag
     - Remove from `tag_store[device_id]`
     - Response: 200 `{"status": "removed"}` or 404 if tag not found

8. Create fixture files derived from `.references/mcp-claroty-xdome/src/types/claroty.ts`:
   - `fixtures/devices.json`: 20 device objects with fields from Claroty TypeScript types:
     `device_uid`, `name`, `ip_list`, `mac_list`, `type`, `category`, `subcategory`,
     `os_revision`, `firmware_version`, `risk_level`, `network_id`, `site_id`, `tags`
   - `fixtures/alerts.json`: 10 alert objects: `alert_id`, `name`, `severity`,
     `status`, `created_at`, `resolved_at`, `affected_devices_count`
   - `fixtures/alerted-devices.json`: 5 device objects (subset of devices fixture)
   - `fixtures/vulnerabilities.json`: 15 vulnerability objects: `vuln_id`, `cve_id`,
     `name`, `severity`, `cvss_score`, `affected_devices_count`, `patch_available`
   - `fixtures/vulnerability-devices.json`: 5 device objects (subset of devices fixture)

9. Create error simulation routes:
   - `POST /dtu/configure`: accepts `{"auth_mode": "reject"}` or `{"rate_limit_after": N}`;
     updates FailureLayer state
   - `POST /dtu/reset`: calls `state.reset()` and resets FailureLayer counters

10. Create `tests/fidelity.rs`:
    - `#[tokio::test] fn claroty_dtu_fidelity()`: starts `ClarotyClone`, runs
      `FidelityValidator` against all 7 endpoints, asserts `checks_failed == 0`
    - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /api/v1/devices` with a valid Bearer token and no body filters,
Then the response is HTTP 200 with a `devices` array of 20 objects, each containing
`device_uid`, `name`, `ip_list`, and `risk_level` fields.

**AC-2:** Given `POST /api/v1/devices` with `{"group_by": "type"}` in the body,
Then the response contains only the grouped field values, not full device objects
(matches xDome reference implementation behavior confirmed in xdome-api-client.ts).

**AC-3:** Given `POST /api/v1/devices/{device_id}/tags/` is called with
`{"tag_key": "quarantine", "tag_value": "true"}`, Then subsequent `POST /api/v1/devices`
returns that device with `tags` including the new tag (stateful tagging persists).

**AC-4:** Given `DELETE /api/v1/devices/{device_id}/tags/quarantine` is called after
the tag is added, Then subsequent `POST /api/v1/devices` returns the device with the
tag removed.

**AC-5:** Given any request without an `Authorization` header, Then the response is
HTTP 401 with a JSON body.

**AC-6:** Given `FailureMode::RateLimit { after_n_requests: 5, retry_after_secs: 30 }`
is configured via `POST /dtu/configure`, When the 6th request arrives, Then the
response is HTTP 429 with `Retry-After: 30`.

**AC-7:** Given `FailureMode::InternalError { at_request_n: 1 }` is configured (simulating
Claroty 500), When the first POST request arrives, Then the response is HTTP 500 —
exercising Prism's `E-SENSOR-002` path.

**AC-8:** Given `reset()` is called, Then the tag store is cleared; subsequent device
queries return devices with empty `tags` arrays (base fixture state).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `POST /api/v1/devices` with unrecognized filter field | Field ignored (Claroty API is permissive); response returned normally |
| EC-002 | `DELETE` tag that was never added | HTTP 404 `{"error": "tag not found"}` |
| EC-003 | `group_by` with value that is not a device field | DTU returns full objects (no recognized grouping); does not error |
| EC-004 | Pagination beyond last page | Returns empty `devices` array, `total` unchanged |
| EC-005 | 422 simulation (invalid filter syntax) | `FailureLayer` can be configured to return 422; maps to `E-SENSOR-004` |
