# S-3.7.00 — Schema derivation: Armis + CrowdStrike → Rust types

## Narrative

As a generator implementer, I want canonical Rust type definitions (or JSON Schema
files) for the Armis and CrowdStrike API response shapes derived from their Go SDK
sources, so that S-3.7.04 and S-3.7.05 can produce schema-conformant fixtures that
pass `BC-3.4.002` validation without reverse-engineering live APIs.

## Objective

Translate Go struct types from `armis-sdk-go/v2` (consumed by `.references/poller-coaster/internal/`)
and `gofalcon` (consumed by `.references/poller-cobra/internal/`) into:

1. Rust struct definitions (preferred) or JSON Schema files checked into
   `.references/schemas/armis/` and `.references/schemas/crowdstrike/`
2. A brief derivation note per sensor documenting field mapping decisions,
   nullable → `Option<T>` choices, and any polymorphic ID handling
3. The `default_page_size()` constant for each sensor (D-055), used by
   `PaginationEdgeCases` archetype baseline

This is a one-time research and translation work story. No production Rust code is
produced. Output is checked-in schema artifacts and a derivation note.

## Tasks

1. Read `.references/poller-coaster/internal/` — identify all Go struct types that
   map to Armis API response fields (asset, alert, AQL search response, pagination
   cursor). Document field names, Go types, and nullability.

2. Translate Armis Go structs to Rust struct definitions in
   `.references/schemas/armis/types.rs` (annotated with `#[allow(dead_code)]` and
   `#[derive(Debug, serde::Deserialize, schemars::JsonSchema)]` so that JSON Schema
   can be auto-generated). Key types: `ArmisAsset`, `ArmisAlert`, `AqlResponse`,
   `ArmisPage<T>`.

3. Read `.references/poller-cobra/internal/` and `crates/prism-dtu-crowdstrike/fixtures/`
   — identify all Go struct types and existing fixture JSON shapes for CrowdStrike
   (detection, device, containment response). Cross-reference with
   `crates/prism-dtu-crowdstrike/src/state.rs` (`containment_store`,
   `detection_status_store` field names).

4. Translate CrowdStrike Go structs to Rust struct definitions in
   `.references/schemas/crowdstrike/types.rs`. Key types: `FalconDevice`,
   `FalconDetection`, `ContainmentResponse`, `OAuth2TokenResponse`, `IdPage` (for
   2-step IDs→detail pattern used by S-3.7.05).

5. Determine `default_page_size()` for each sensor from the SDK or poller source:
   - Armis: inspect AQL pagination parameters in poller-coaster
   - CrowdStrike: inspect FQL pagination limit in poller-cobra
   Write constants into the schema files as doc-comments.

6. Write `.references/schemas/armis/DERIVATION.md` and
   `.references/schemas/crowdstrike/DERIVATION.md` documenting:
   - Source Go structs mapped to each Rust type
   - Nullable → `Option<T>` mapping decisions
   - Polymorphic field handling (e.g., IDs as string vs. number)
   - `default_page_size` rationale

  field names are authoritative for the stateful schema (ADR-009 §1.2).
- All output files must be committed to `.references/schemas/` and referenced in
  DERIVATION.md before this story is considered done.

## Acceptance Criteria

### AC-001 — Armis Rust types cover all asset + alert fields (traces to BC-3.4.002 precondition 2)

The file `.references/schemas/armis/types.rs` defines at minimum: `ArmisAsset`,
`ArmisAlert`, `AqlResponse<T>`, and a pagination wrapper type. Every field present
in the `armis-sdk-go/v2` consumer structs found in `.references/poller-coaster/internal/`
is represented. Nullable fields are typed as `Option<T>`. Polymorphic ID fields
(string vs. number) use `serde_json::Value` or a custom `ArmisId` newtype with
`deserialize_any`.

### AC-002 — CrowdStrike Rust types cover device + detection + containment + OAuth2 fields (traces to BC-3.4.002 precondition 2)

The file `.references/schemas/crowdstrike/types.rs` defines at minimum: `FalconDevice`,
`FalconDetection`, `ContainmentResponse`, `OAuth2TokenResponse`, `IdPage`. Field names
match `crates/prism-dtu-crowdstrike/src/state.rs` (`containment_store`,
`detection_status_store`) and existing fixture JSON in `crates/prism-dtu-crowdstrike/fixtures/`.

### AC-003 — `default_page_size` constants documented (traces to BC-3.4.003 postcondition for PaginationEdgeCases)

`DERIVATION.md` for each sensor documents the `default_page_size` value and its
source (Go SDK constant, API documentation, or empirical observation from fixture
data). S-3.7.04 and S-3.7.05 implementers can read this value without re-researching
the source.

### AC-004 — Derivation notes document mapping decisions (traces to BC-3.4.002 invariant 3)

Each `DERIVATION.md` documents: (a) the source Go struct for every Rust type, (b)
nullable → `Option<T>` rationale, (c) any polymorphic field handling decisions, (d)
any Go field omitted from the Rust translation and why. The notes must be sufficient
for the S-3.7.04/S-3.7.05 implementer to write schema-conformant fixture generators
without reading the Go SDK source.

### AC-005 — No generator code written in this story (traces to BC-3.4.002 invariant 3)

This story produces only schema reference files. No `generate()` function, no
`#[cfg(feature = "fixture-gen")]` module, and no test that calls a generator is
authored here. Generator implementation is deferred to S-3.7.04 and S-3.7.05.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Armis API returns asset IDs as either strings or integers | Documented in DERIVATION.md; `ArmisId` newtype handles both via `deserialize_any` |
| EC-002 | CrowdStrike 2-step pattern: first call returns `[id]`, second returns `[detail]` | `IdPage` type captures the first-step response shape; documented separately |
| EC-003 | Go field uses a type with no direct Rust analog (e.g., `interface{}`) | Map to `serde_json::Value`; documented in DERIVATION.md with rationale |
| EC-004 | Existing `crates/prism-dtu-crowdstrike/fixtures/` JSON has fields not in Go structs | Include them in Rust types as `Option<T>`; flag in DERIVATION.md as empirically observed |
