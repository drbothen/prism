# S-3.6.02 — HS-007 multi-tenant cross-repo failure holdout refresh

## Narrative

As a quality gate owner, I want the HS-007 holdout scenario file refreshed to describe
per-customer DTU failure isolation (customer A's sensor fails; customer B unaffected) using
current Wave 3 BC language (BC-3.6.001, BC-3.5.002), so that the holdout evaluation harness
can resolve all BC references and the scenario exercises the network-isolation path that
catches cross-process routing bugs.

## Acceptance Criteria

### AC-001 — HS-007 frontmatter lists current Wave 3 BC IDs (traces to BC-3.6.001 precondition 1)

The refreshed `.factory/holdout-scenarios/HS-007-cross-repo-failure.md` frontmatter contains:
`behavioral_contracts: [BC-3.6.001, BC-3.6.002, BC-3.5.002]`.
No Phase 1b cross-repo failure BC IDs (e.g., references to retired MemoryStore, poller-cobra
ordering bug BCs) remain. `phase` updated to `3.A`; `timestamp` to `2026-04-27`.

### AC-002 — Core scenario: customer A's DTU auth-reject; customer B unaffected (traces to BC-3.6.001 postcondition 2)

The primary HS-007 sub-scenario describes:
1. Two-customer harness in `IsolationMode::Network` (BC-3.5.002).
2. `inject_failure(org_slug_A, DtuType::PagerDuty, FailureMode::AuthReject)` applied.
3. Customer A's PagerDuty DTU queries return HTTP 401.
4. Customer B's queries to all their DTU clones return HTTP 200 with valid data.
5. No cross-contamination: customer B's `FailureLayerShared` state is unchanged.

The scenario names `FailureMode::AuthReject` and cites BC-3.6.001 postcondition 1.

### AC-003 — Graceful per-customer degradation described (traces to BC-3.6.001 postcondition 3)

After `clear_failure(org_slug_A, DtuType::PagerDuty)`, customer A's queries resume returning
HTTP 200. The scenario asserts: failure is cleared idempotently; subsequent requests observe
restored normal behavior (BC-3.6.001 postcondition 3).

### AC-004 — Audit trail records per-org failure event (traces to BC-3.6.001 postcondition 1)

The scenario describes an audit trail assertion: after `inject_failure` for org A, the
`prism-audit` store contains an event scoped to org A's `OrgId` recording the failure mode
and timestamp. No audit event is written for org B (per-org scoping of audit records).

### AC-005 — Network mode used for cross-process leakage detection (traces to BC-3.5.002 postcondition 2)

The scenario explicitly requires `IsolationMode::Network` (not Logical), citing BC-3.5.002
postcondition 2: a request with org A credentials routed to org B's endpoint returns HTTP 401,
making routing bugs observable. The sub-scenario includes a routing-bug detection test step:
send org A credentials to org B's `SocketAddr`; assert HTTP 401 response.
