# S-4.01 — prism-operations: Schedule CRUD and Execution Loop

## Narrative

As a Prism operations engine, I want a schedule management subsystem with CRUD tools and
a tick-based execution loop, so that analysts can define recurring queries that run
automatically against selected clients without manual intervention.

## Objective

Implement scheduled query management and tick-based execution loop. Schedules are
persisted to RocksDB, exposed via MCP tools (create_schedule, list_schedules,
delete_schedule), and executed by a 60-second tick loop that applies per-schedule splay
and skips in-flight executions without queuing.

## Tasks

1. Define `ScheduleEntry` and `ScheduleState` types in
   `crates/prism-operations/src/schedule/schedule.rs`:
   - `ScheduleEntry { id: ScheduleId, query: String, interval: Duration, clients: Vec<ClientId>,
     splay_offset: Duration, created_at: Timestamp }`
   - `ScheduleState { last_run_at: Option<Timestamp>, next_run_at: Timestamp, epoch: u64,
     in_flight: bool }`
   - `ScheduleId` newtype wrapping UUID v7

2. Implement `create_schedule` tool handler in
   `crates/prism-operations/src/schedule/crud.rs` (BC-2.12.001):
   - Accept: query string, interval (e.g. "5m", "1h"), clients scope (list of ClientIds or
     wildcard), optional display name
   - Validate: interval within bounds [1m, 24h], query parseable by PrismQL parser,
     clients list non-empty
   - Compute deterministic splay offset: `hash(schedule_id) % (interval / 4)` — capped at
     15 minutes
   - Persist `ScheduleEntry` + initial `ScheduleState` to RocksDB `schedules` CF
   - Enforce schedule count cap (configurable via `PRISM_MAX_SCHEDULES`, default 500);
     return E-SCHED-008 when at cap
   - Return created `ScheduleEntry` with `next_run_at` computed

3. Implement `list_schedules` tool handler (BC-2.12.002):
   - Iterate all entries in `schedules` CF
   - For each schedule return: id, query (truncated at 200 chars), interval, clients,
     next_run_at, last_run_at, epoch, in_flight status
   - Support optional filter by client_id

4. Implement `delete_schedule` tool handler (BC-2.12.003):
   - If schedule is active (has run at least once or is in-flight): require confirmation
     token via prism-security confirmation token subsystem (S-1.09)
   - On confirmed deletion: remove `ScheduleEntry` and `ScheduleState` from RocksDB,
     cancel in-flight execution if possible (best-effort via cancellation token)
   - If schedule has never run: delete without confirmation

5. Implement tick-based execution loop in
   `crates/prism-operations/src/schedule/executor.rs` (BC-2.12.004):
   - Tick interval: 60 seconds (configurable via `PRISM_SCHEDULER_TICK_SECS`)
   - On each tick: load all `ScheduleEntry` records, check `next_run_at <= now()`
   - Apply splay: effective fire time = `next_run_at + splay_offset`
   - Global schedule semaphore: `Arc<Semaphore>` with 16 permits (per CI-005 in
     concurrency-architecture.md). Shared between schedule execution AND action report
     queries (S-4.08). `try_acquire()` — if all 16 permits occupied, log skip and
     continue — do NOT queue
   - On successful acquire: spawn tokio task to execute the schedule's query via
     `QueryEngine`, update `ScheduleState` (epoch + 1, last_run_at = now(),
     next_run_at = now() + interval), release semaphore on completion or error
   - Graceful shutdown: stop accepting new ticks on SIGTERM, wait for in-flight tasks up
     to 30 seconds

6. Implement schedule state persistence helpers in
   `crates/prism-operations/src/schedule/persistence.rs` (BC-2.12.010):
   - `save_schedule_state(id, state)`: write `ScheduleState` to RocksDB `schedules` CF
     under key `sched:{id}:state`
   - `load_schedule_state(id)`: read back `ScheduleState`, return `None` if not found
   - `list_all_schedules()`: prefix scan over `sched:` keys to return all entries
   - On startup: recover all schedule states and rehydrate in-memory semaphores

7. Write VP-026 Kani proof in `crates/prism-operations/proofs/vp026_splay_deterministic.rs`:
   - Property: for all `ScheduleId` values, `compute_splay(id)` returns the same value on
     every call — pure function of the ID, no randomness

8. Write VP-030 Kani proof in `crates/prism-operations/proofs/vp030_schedule_cap.rs`:
   - Property: `create_schedule` with `count = MAX_SCHEDULES` always returns E-SCHED-008,
     never creates a schedule

## Acceptance Criteria

**AC-1:** Given `create_schedule(query: "SELECT * FROM crowdstrike.detections WHERE
severity_id >= 3", interval: "5m", clients: ["acme"])`, When created, Then the schedule
persists to RocksDB and appears in `list_schedules` output with a valid `next_run_at`.
(BC-2.12.001, BC-2.12.002)

**AC-2:** Given 500 existing schedules (at cap), When `create_schedule` is called, Then
E-SCHED-008 is returned and no schedule is created. (BC-2.12.001 / DI-028)

**AC-3:** Given an active schedule (has run at least once), When `delete_schedule` is
called without a confirmation token, Then the tool responds requesting a confirmation
token. (BC-2.12.003)

**AC-4:** Given a schedule with 5-minute interval and a computed 30-second splay offset,
When the tick fires at T+5m, Then execution starts at T+5m+30s and `ScheduleState` is
updated. (BC-2.12.004)

**AC-5:** Given a schedule whose execution is still in-flight, When the next tick fires
and `try_acquire` fails, Then the execution is skipped (not queued) and a skip event is
logged. (BC-2.12.004)

**AC-6:** Given a schedule runs at epoch 5, When state is persisted, Then
`epoch=5`, `last_run_at`, and `next_run_at` are stored in RocksDB and recoverable after
restart. (BC-2.12.010)

**AC-7:** VP-026 and VP-030 Kani proofs pass under `cargo kani`.

|-----------|--------|----------------|

| Schedule persistence | `prism-storage` (SS-15) | Effectful |
| MCP schedule tools | `prism-mcp` (SS-10) | Effectful |

`prism-operations` implements `Scheduler` — the tokio timer loop that fires scheduled
PrismQL queries. Schedule state machines and tick logic are pure; RocksDB persistence
(`schedules` CF), timer registration, and `QueryEngine::execute_scheduled()` calls are
effectful. `prism-mcp` exposes schedule CRUD tools. Per `architecture/module-decomposition.md`,
`prism-operations` depends on `prism-query` and `prism-storage`.

## Architecture Compliance Rules

- The tick loop MUST use `try_acquire` (non-blocking) on per-schedule semaphores — never
  `acquire` (blocking). Blocking acquire would cause head-of-line blocking across all
  schedules.
- Splay computation MUST be a pure function of `ScheduleId` — no random number
  generation, no wall clock. This ensures determinism across restarts and Kani
  provability.
- Schedule CRUD handlers MUST NOT execute queries inline — they only manage schedule
  metadata. Query execution is the executor's responsibility.
- `ScheduleState` writes to RocksDB MUST be synchronous (not fire-and-forget). State loss
  across restart is a correctness defect.
- The schedule count cap MUST be checked under a RocksDB transaction or equivalent atomic
  read-modify-write to prevent TOCTOU races at the cap boundary.

## Dev Notes

- The 60-second tick is a minimum check frequency; schedules with intervals shorter than
  60 seconds are not supported (enforce minimum interval of 1 minute on create).
- Splay is intentionally not user-configurable. It exists to prevent thundering herd when
  many schedules share the same interval. The formula `hash(id) % (interval / 4)` spreads
  load without requiring coordination.
- In-flight skip (not queue) is a deliberate design choice: if a query consistently takes
  longer than its interval, the operator should increase the interval rather than having
  a growing backlog of queued executions.
- The `clients: ["*"]` wildcard in `create_schedule` means the schedule runs against all
  clients the calling analyst has access to. The executor resolves this to concrete
  ClientIds at fire time, not at creation time.
