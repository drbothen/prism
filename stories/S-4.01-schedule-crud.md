# S-4.01 — prism-operations: Schedule CRUD and Execution Loop

## Narrative

As a Prism operations engine, I want a schedule management subsystem with CRUD tools and
a tick-based execution loop, so that analysts can define recurring queries that run
automatically against selected clients without manual intervention.

## Objective

Implement scheduled query management and tick-based execution loop. Schedules are
persisted to RocksDB, exposed via MCP tools (create_schedule, list_schedules,
delete_schedule), and executed by a 60-second tick loop that applies per-schedule splay
and skips in-flight executions without queuing.

   `crates/prism-operations/src/schedule/schedule.rs`:

     (per ADR-013 §2.6 and ADR-018 §2.3 `pack_origin`/`pack_version` fields;

2. Implement `create_schedule` tool handler in
   `crates/prism-operations/src/schedule/crud.rs` (BC-2.12.001):
   - Accept: query string, cron expression (croner format) or interval shorthand (e.g.
     "5m", "1h"), client scope (`ClientFilter` enum — wildcard or explicit list), optional
     display name. The `org_id` is set from the calling session's org context ONLY —
     NOT from a tool parameter. A request supplying a mismatched `org_id` parameter is
     rejected with `E-ORG-MISMATCH` (P1-S-4.01-A-M-001).
   - Validate: interval within bounds [1m, 24h], query parseable by PrismQL parser,
     client_id_filter non-empty (ClientFilter::All counts as non-empty)
   - Compute deterministic splay offset per ADR-013 §2.2:
     `splay_seconds = u64::from_le_bytes(blake3(schedule_id_bytes)[0..8]) % (interval_seconds / 4)`,
     capped at `min(result, 900)` (15 minutes)
   - Persist `ScheduleEntry` + initial `ScheduleState` to RocksDB `schedules` CF under
     key `{org_id_bytes}:{schedule_id_bytes}`
   - Enforce schedule count cap (configurable via `PRISM_MAX_SCHEDULES`, default 500);
     return E-SCHED-008 when at cap (cap enforced under atomic prefix-count scan per
     ADR-013 §2.6 to prevent TOCTOU races)
   - Emit `ScheduleChangeNotification::Created(org_id, schedule_id)` via the watch channel
     (ADR-013 §2.7) after successful persist
   - Return created `ScheduleEntry` with `next_run_at` computed

3. Implement `list_schedules` tool handler (BC-2.12.002):
   - Prefix-scan `schedules` CF restricted to `{org_id_bytes}:` prefix (ADR-013 §2.6)
   - For each schedule return: id, query (truncated at 200 chars), cron_expr,
     client_id_filter, next_run_at, last_run_at, epoch, in_flight status
   - Support optional filter by client_id
   - Only returns schedules belonging to the calling session's `org_id`

4. Implement `delete_schedule` tool handler (BC-2.12.003):
   - If schedule is active (has run at least once or is in-flight): require confirmation
     token via prism-security confirmation token subsystem (S-1.09)
   - On confirmed deletion: remove `ScheduleEntry` and `ScheduleState` from RocksDB,
     cancel in-flight execution if possible (best-effort via cancellation token)
   - Emit `ScheduleChangeNotification::Deleted(org_id, schedule_id)` via watch channel
   - If schedule has never run: delete without confirmation

5. Implement tick-based execution loop in
   `crates/prism-operations/src/schedule/executor.rs` (BC-2.12.004):
   - Tick interval: 60 seconds (configurable via `PRISM_SCHEDULER_TICK_SECS` [10, 3600])
   - Cron expression parsing: use `croner = "3"` (ADR-013 §2.8; do NOT use the `cron`
     crate — it lacks DST/timezone correctness required for multi-tenant MSSP scheduling)
   - On each tick: evaluate fire-eligible schedules from in-memory splay-target cache;
     check `splay_adjusted_fire_time <= now()`
   - Per-subsystem semaphore (D-209 LOCKED / ADR-013 §2.3): `Arc<Semaphore>` with
     **8 permits** owned exclusively by `executor.rs` — NOT shared with action delivery
     (S-4.08 owns its own independent 8-permit `action_delivery_semaphore`).
     `try_acquire()` — if all 8 permits occupied, emit `ScheduleFireMissed` audit event
     with `miss_reason: SemaphoreExhausted` (per ADR-013 §2.4) and continue — do NOT queue.
     (`ScheduleFireSkipped` is reserved per ADR-013 §2.5 for the distinct in-flight skip case
     where a previous tick's JoinHandle is still active.)
   - In-flight tracking: `DashMap<(OrgId, ScheduleId), JoinHandle<()>>` per ADR-013 §2.5.
     On each tick: check if `(org_id, schedule_id)` exists and `!is_finished()` → skip
     with `ScheduleFireSkipped { reason: InFlight }` audit event. If finished, remove stale
     entry and proceed with normal fire evaluation.
   - On successful acquire: spawn tokio task to execute the schedule's query via
     `QueryEngine`, update `ScheduleState` (epoch + 1, last_run_at = now(),
     next_run_at = now() + interval), release semaphore permit on completion or error
   - Subscribe to watch channel receiver (ADR-013 §2.7): on `ScheduleChangeNotification`
     receipt, invalidate the affected schedule's entry from the in-memory splay-target
     cache; reload from RocksDB for `Created`/`Updated`; for `Deleted`: remove from
     splay-target cache AND evict from in-flight `DashMap<(OrgId, ScheduleId), JoinHandle<()>>`
     (ADR-013 §2.7 explicit Deleted-handler requirement)
   - Graceful shutdown: stop accepting new ticks on SIGTERM, wait for in-flight tasks up
     to 30 seconds

6. Implement schedule state persistence helpers in
   `crates/prism-operations/src/schedule/persistence.rs` (BC-2.12.010):
   - `save_schedule_state(org_id, schedule_id, state)`: write `ScheduleState` to RocksDB
     `schedules` CF under key `{org_id}:{schedule_id}:state`
   - `load_schedule_state(org_id, schedule_id)`: read back `ScheduleState`, return `None`
     if not found
   - `list_all_schedules(org_id)`: prefix scan over `{org_id}:` keys to return all entries
     for the org; never returns entries from other orgs
   - On startup: recover all schedule states and rehydrate in-memory splay-target cache

7. Wire the `tokio::sync::watch` channel (ADR-013 §2.7, D-211):
   - Construct `(watch_tx, watch_rx)` in `prism-operations` module init
   - Inject `watch_tx` (sender) into CRUD handlers (Task 2, 3, 4 emit notifications)
   - Inject `watch_rx` (receiver) into executor (Task 5 subscribes)
   - Channel carries:
     ```rust
     enum ScheduleChangeNotification {
         Created(OrgId, ScheduleId),
         Updated(OrgId, ScheduleId),
         Deleted(OrgId, ScheduleId),
     }
     ```
   - Document Arc ownership: `lib.rs` constructs the channel; CRUD layer holds the sender;
     executor holds the receiver

8. Write VP-026 Kani proof in `crates/prism-operations/proofs/vp026_splay_deterministic.rs`:

9. Write VP-030 Kani proof in `crates/prism-operations/proofs/vp030_schedule_cap.rs`:

## Acceptance Criteria

**AC-1:** Given `create_schedule(query: "SELECT * FROM crowdstrike.detections WHERE
severity_id >= 3", interval: "5m", clients: ["acme"])`, When created, Then the schedule
persists to RocksDB and appears in `list_schedules` output with a valid `next_run_at`.
(BC-2.12.001, BC-2.12.002)

**AC-2:** Given 500 existing schedules (at cap), When `create_schedule` is called, Then
E-SCHED-008 is returned and no schedule is created. (BC-2.12.001 / DI-028)

**AC-3:** Given an active schedule (has run at least once), When `delete_schedule` is
called without a confirmation token, Then the tool responds requesting a confirmation
token. (BC-2.12.003)

**AC-4:** Given a schedule with 5-minute interval and a computed 30-second splay offset,
When the executor evaluates fire eligibility, Then the schedule fires within the window
`[next_run_at + splay_offset, next_run_at + splay_offset + tick_interval]`, and the
recorded `last_run_at` falls within that same window. The test asserts on the recorded
`last_run_at` field, not on wall-clock elapsed time. (BC-2.12.004 / QUAL-401-001)

**AC-5:** Given a schedule whose execution is still in-flight, When the next tick fires
and `try_acquire` fails, Then the execution is skipped (not queued) and a skip event is
logged. (BC-2.12.004)

**AC-6:** Given a schedule runs at epoch 5, When state is persisted, Then
`epoch=5`, `last_run_at`, and `next_run_at` are stored in RocksDB and recoverable after
restart. (BC-2.12.010)

**AC-7:** VP-026 and VP-030 Kani proofs pass under `cargo kani`.

**AC-8:** Given schedules belonging to org A (with `org_id = A`), When `list_schedules` is
called in org B's session context, Then org A's schedules are not returned — the response
contains only schedules scoped to org B. (BC-2.12.002 / ADR-006 §2.1 / DRIFT-401-003)

**AC-8b:** Given a `create_schedule` call whose tool parameters include an explicit
`org_id` value different from the calling session's org context, When the handler
validates the request, Then the call is rejected with `E-ORG-MISMATCH` and no schedule
is created. The `org_id` on the resulting `ScheduleEntry` is always sourced from the
session context, never from a tool parameter. (BC-2.12.001 / P1-S-4.01-A-M-001)

## Architecture Compliance Rules

- The tick loop MUST use `try_acquire` (non-blocking) on the schedule executor semaphore —
  never `acquire` (blocking). Blocking acquire would cause head-of-line blocking across all
  schedules. (ADR-013 §2.5)
- The schedule executor semaphore MUST have exactly 8 permits and MUST NOT be shared with
  the action delivery semaphore. D-209 (LOCKED) established independent per-subsystem
  semaphores; cross-module handle passing is prohibited. (ADR-013 §2.3)
- Splay computation MUST be a pure function of `ScheduleId` and `interval_seconds` — no
  random number generation, no wall clock. Formula:
  `u64::from_le_bytes(blake3(schedule_id_bytes)[0..8]) % (interval_seconds / 4)`,
  capped at 900 seconds (little-endian byte extraction, explicitly pinned per ADR-013 §2.2).
  (ADR-013 §2.2 / VP-026)
- Cron expression parsing MUST use `croner = "3"`. The `cron` crate (any version) is
  FORBIDDEN — it lacks DST/timezone awareness required for multi-tenant MSSP correctness.
  (ADR-013 §2.8 / R-2)
- Schedule CRUD handlers MUST emit `ScheduleChangeNotification` via the `watch` channel
  after every create/update/delete. The executor subscribes to this channel to invalidate
  its in-memory splay-target cache. (ADR-013 §2.7 / D-211)
- Schedule CRUD handlers MUST NOT execute queries inline — they only manage schedule
  metadata. Query execution is the executor's responsibility.
- `ScheduleState` writes to RocksDB MUST be synchronous (not fire-and-forget). State loss
  across restart is a correctness defect.
- The schedule count cap MUST be checked under a RocksDB transaction or equivalent atomic
  prefix-count scan to prevent TOCTOU races at the cap boundary. (ADR-013 §2.6)
- All RocksDB `schedules` CF keys MUST use the `{org_id_bytes}:{schedule_id_bytes}` format.
  No schedule key may omit the `org_id` prefix. (ADR-013 §2.6 / ADR-008 universal re-keying
  rule)

## Dev Notes

- The 60-second tick is a minimum check frequency; schedules with intervals shorter than
  60 seconds are not supported (enforce minimum interval of 1 minute on create).
- Splay is intentionally not user-configurable. It exists to prevent thundering herd when
  many schedules share the same interval. The formula per ADR-013 §2.2 is
  `u64::from_le_bytes(blake3(schedule_id_bytes)[0..8]) % (interval_seconds / 4)`,
  capped at 900 seconds. Little-endian byte extraction is explicitly pinned (ADR-013 §2.2).
  Only the first 8 bytes of the 32-byte blake3 output are used as the u64 modulus operand.
- In-flight skip (not queue) is a deliberate design choice: if a query consistently takes
  longer than its interval, the operator should increase the interval rather than having
  a growing backlog of queued executions.
- The `ClientFilter::All` variant in `create_schedule` (replacing the old `clients: ["*"]`
  wildcard string) means the schedule runs against all clients the calling analyst has
  access to. The executor resolves this to concrete ClientIds at fire time, not at
  creation time. `ClientFilter` is a shared enum in `prism-operations::types` (also used
  by action delivery in S-4.08 per ADR-016 §2.5).
