# S-3.10 — prism-query: Cost Estimation with API Latency Awareness

## Narrative

As a Prism query engine, I want the query planner to use observed API latency
histograms to optimize JOIN order and fan-out scheduling so that queries across
multiple sensors complete as fast as possible — putting the slowest sensor's
API call first (to maximize parallel wait overlap) and using the faster source
as the JOIN probe side.

## Objective

Maintain per-sensor, per-source latency histograms (p50/p95/p99) derived from
`QueryMetrics` (S-3.09). Use these estimates in JOIN ordering (faster source as
probe side) and fan-out scheduling (start slowest source first). Expose cost
estimates in `explain_query` and allow sensor specs to declare cold-start defaults.

## Tasks

1. Implement `LatencyHistogram` in `crates/prism-query/src/latency_histogram.rs`:
   - Sliding window: last 100 observations per `(sensor_id, client_id, table_name)`
   - Computes p50, p95, p99 from sorted observations (simple sort-based quantile
     for simplicity — not HDR histogram; 100 samples fits in L1 cache easily)
   - `observe(latency_ms: u64)` — add observation, evict oldest if at capacity
   - `p50() -> u64`, `p95() -> u64`, `p99() -> u64` — return 0 if no observations
   - Thread-safe: `Arc<RwLock<LatencyHistogram>>` per source key
   - `LatencyHistogramRegistry`: `HashMap<SourceKey, Arc<RwLock<LatencyHistogram>>>`
     stored in `QueryEngine`; keyed by `(sensor_id, client_id, table_name)`

2. Feed `QueryMetrics.per_source_latency_ms` into `LatencyHistogramRegistry` after
   each `execute()` completes:
   - In `engine.rs`, after metrics are finalized, iterate `per_source_latency_ms`
     and call `registry.observe(source_key, latency_ms)`
   - Non-blocking: skip if histogram write-lock is contended

3. Implement `CostModel` in `crates/prism-query/src/cost_model.rs`:
   - `estimate_latency(source: &SourceKey) -> u64` — returns p50 from histogram
     if observations exist; otherwise returns the spec-declared default (see task 4);
     ultimately falls back to `500ms` hardcoded constant if neither is available
   - `estimate_rows(source: &SourceKey) -> u64` — returns spec-declared hint or 1000
   - `CostEstimate { source: SourceKey, latency_ms: u64, row_estimate: u64 }`
   - `estimate_all(sources: &[SourceKey]) -> Vec<CostEstimate>` — called by the planner

4. Extend `[[sensor.tables]]` in sensor spec schema:
   ```toml
   [[sensor.tables]]
   table_name = "detections"
   estimated_latency_ms = 500       # cold-start default; overridden by observed data
   estimated_rows_per_call = 250    # hint for row count estimation
   ```
   Validate: `estimated_latency_ms` min 1, max 30000. `estimated_rows_per_call` min 1.
   Both are optional; omitting them uses the hardcoded fallback.

5. Use `CostModel` in fan-out scheduling in `fanout.rs`:
   - Before spawning tokio tasks, call `cost_model.estimate_all(targets)`
   - Sort targets by estimated latency descending (slowest first)
   - Spawn all tasks immediately (still parallel) but submit slowest tasks first
     to the tokio scheduler — maximizes overlap in the parallel wait period
   - This is a scheduling hint, not serialization: all tasks are spawned in the same
     `tokio::spawn` loop; order of spawn does matter for tokio's fair scheduling

6. Use `CostModel` in JOIN ordering in `planner.rs`:
   - For a two-source JOIN: estimate latency and row count for both sides
   - Place the source with lower estimated rows as the build side (hash join build)
     and higher estimated rows as the probe side — this minimizes hash table size
   - If row estimates are equal: place the source with lower latency as the probe side
     (slower source does more work in parallel)
   - This is a hint to DataFusion via `LogicalPlan` table ordering — DataFusion's
     optimizer may further reorder; the Prism planner sets the initial order

7. Expose cost estimates in `explain_query` output:
   - Add `"cost_estimates"` array to explain response:
     ```json
     {
       "cost_estimates": [
         {
           "source": "crowdstrike/acme/detections",
           "estimated_latency_ms": 234,
           "estimated_rows": 250,
           "latency_source": "observed_p50",
           "observation_count": 47
         },
         {
           "source": "armis/acme/devices",
           "estimated_latency_ms": 500,
           "estimated_rows": 1000,
           "latency_source": "spec_default",
           "observation_count": 0
         }
       ],
       "join_order": "crowdstrike_detections (probe) JOIN armis_devices (build)"
     }
     ```
   - `latency_source`: `"observed_p50"` | `"spec_default"` | `"hardcoded_fallback"`

## Acceptance Criteria

**AC-1:** Given 10 prior queries with `per_source_latency_ms["crowdstrike/acme/detections"]
= [200, 220, 210, ...]`, When `cost_model.estimate_latency("crowdstrike/acme/detections")`
is called, Then it returns the p50 of those 10 observations (approximately 210ms).

**AC-2:** Given a JOIN between `crowdstrike_detections` (est. 250 rows) and
`armis_devices` (est. 1000 rows), When the planner orders the JOIN, Then
`crowdstrike_detections` is the build side and `armis_devices` is the probe side.

**AC-3:** Given a fan-out across 3 sources with estimated latencies [800ms, 200ms,
500ms], When tasks are spawned, Then the 800ms source task is submitted to the tokio
scheduler first (spawn order: 800ms, 500ms, 200ms).

**AC-4:** Given a sensor spec with `estimated_latency_ms = 750` and no prior
observations, When `cost_model.estimate_latency()` is called for that source, Then
it returns 750 (spec default).

**AC-5:** Given no spec default and no observations, When `cost_model.estimate_latency()`
is called, Then it returns 500 (hardcoded fallback constant).

**AC-6:** Given `explain_query` for a two-source JOIN, When the response is returned,
Then `cost_estimates` contains one entry per source, each with `estimated_latency_ms`,
`estimated_rows`, and `latency_source` fields.

**AC-7:** Given all three PrismQL modes, When each executes a multi-source query,
Then cost estimation and fan-out ordering are applied identically — the optimization
is transparent to the query syntax.

## Architecture Compliance Rules

- `LatencyHistogram` MUST use a sliding window (last N observations) rather than
  lifetime accumulation — API latency changes over time (sensor upgrades, network
  shifts) and old data should not dominate estimates.
- **Event-stream tables (S-2.08) are excluded from latency histograms.** Their latency
  is always ~5ms (local RocksDB scan) and does not need optimization. Tag observations
  with `source_type: PointInTime | EventStream` and only feed PointInTime observations
  into the histogram. For EventStream tables, use a fixed 5ms estimate in cost model.
- Cost-based JOIN ordering is a hint to DataFusion — Prism MUST NOT disable
  DataFusion's own optimizer. Set the initial logical plan order based on cost
  estimates, then let DataFusion optimize further.
- Histogram updates MUST be non-blocking on the query execution path. Use
  `RwLock::try_write()` — if contended, skip the update. Diagnostics accuracy
  is best-effort; query correctness is not.
- Cold-start fallback chain MUST be: observed p50 > spec default >
  hardcoded 500ms. Never return 0ms (would cause divide-by-zero if used in
  further cost computations) and never return negative values.
- `LatencyHistogramRegistry` is a long-lived singleton in `QueryEngine` — it persists
  across queries and accumulates observations throughout the process lifetime.
  It is NOT reset on config reload.

## Dev Notes

- The "slowest first" fan-out heuristic is derived from osquery's synthesis
  recommendation and is well-established in distributed query systems: starting
  the slowest operation earliest minimizes total wall time when all operations
  run in parallel (reduces tail latency exposure).
- The p50 is used for JOIN ordering (steady-state behavior) rather than p99 (which
  would optimize for worst-case, causing build/probe flips on every outlier).
  For fan-out scheduling, p95 might be more appropriate — consider making the
  quantile selector configurable via `[query] cost_estimation_quantile = "p50"` in
  a follow-on.
- The hash table size heuristic (lower row count = build side) is correct for
  DataFusion's hash join implementation: the build side is the smaller table loaded
  into the hash table, and the probe side streams through it. If row estimates are
  wildly wrong (e.g., pagination caps API results at 250 but actual dataset is 10K),
  the estimate-based ordering may be suboptimal but is never incorrect.
