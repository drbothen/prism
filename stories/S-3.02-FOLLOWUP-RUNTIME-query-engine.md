# S-3.02-FOLLOWUP-RUNTIME â€” prism-query: QueryEngine Execution Pipeline

## Narrative

As the Prism query engine, I want all eight `todo!()` stub sites in `prism-query`
replaced with real implementations, so that `QueryEngine::execute` functions end-to-end:
sensor fan-out materializes records, DataFusion processes them, internal tables (RocksDB
domains) are accessible as SQL tables, and the MCP `query` tool returns real results to
the analyst.

No `todo!()` or `unimplemented!()` may remain in any of these nine sites before this

## Tasks

1. Implement `QueryEngine::execute` at `crates/prism-query/src/engine.rs:276`:
   ```rust
   pub async fn execute(&self, query_str: &str, options: QueryOptions)
       -> Result<QueryResult, PrismError>
   {
       // 1. Parse PrismQL â†’ LogicalPlan (reuse existing PrismQlParser)
       // 2. Apply security limits (BC-2.11.006): size cap, depth limit, timeout
       // 3. Call run_materialization_pipeline to fan-out + materialize
       // 4. Execute DataFusion plan against materialized MemTable
       // 5. Return QueryResult with virtual fields injected (_sensor, _client, _source_table)
   }
   ```
   The existing S-3.02 parsing and safety-check infrastructure MUST be reused, not
   re-implemented. Import from `prism_query::parser` and `prism_query::safety_check`.

2. Implement `QueryEngine::execute_scheduled` at `engine.rs:317`:
   - Identical pipeline to `execute` but returns `(QueryResult, Arc<SessionContext>)`.
   - The `SessionContext` is returned for differential result computation (S-4.01 consumer).

3. Implement `run_materialization_pipeline` at `materialization.rs:241`:
   - Call `resolve_source_refs` to get the list of `(org_id, sensor_name)` pairs.
   - For each pair: call `AdapterRegistry::get_adapter(org_id, sensor_name)`.
   - Fan-out parallel fetch via `SensorAdapter::fetch(query_context)` (existing trait method).
   - Apply OCSF normalization via `OcsfNormalizer`.
   - Collect into Arrow `RecordBatch` slices.
   - Register as `MemTable` in a new `SessionContext`.
   - Apply the 10K row materialization cap (existing constant) + `GreedyMemoryPool`.
   - Returns `Arc<SessionContext>` ready for DataFusion SQL execution.

4. Implement `resolve_source_refs` at `materialization.rs:263`:
   - Parse the `FROM` clause (or `SELECT ... FROM ...`) for table references.
   - For each reference: validate it maps to a known sensor (per `ConfigManager`) or internal table.
   - Return `Vec<SourceRef>` where `SourceRef` = `{ org_id, sensor_name }` for external sources
     or `{ internal_table_domain }` for internal tables.
   - Cross-client resolution: expand `ALL` client scope into individual `(org_id, sensor_name)` pairs
     per BC-2.11.011.

5. Implement `RocksDbTableProvider` at `internal_tables.rs`:

   **`schema()` â†’ `SchemaRef`:** Return a fixed Arrow schema for each internal table domain:
   - `prism_audit`: `{ timestamp: Utf8, event_type: Utf8, org_id: Utf8, payload: Utf8 }`
   - `prism_alerts`: `{ alert_id: Utf8, rule_id: Utf8, severity: Int32, timestamp: Utf8 }`
   - `prism_cases`: `{ case_id: Utf8, status: Utf8, severity: Int32, created_at: Utf8 }`
   - `prism_schedules`: `{ schedule_id: Utf8, query: Utf8, cron: Utf8, next_run: Utf8 }`
   - Schema is fixed at compile time; no dynamic schema discovery.

   **`table_type()` â†’ `TableType`:** Return `TableType::Base`.

   **`scan()` â†’ `SendableRecordBatchStream`:** Scan the RocksDB CF for the domain:
   - Call `StorageBackend::scan(domain)` to iterate all key-value pairs.
   - Deserialize values (bincode 2.x encoding per AD-012).
   - Convert to Arrow `RecordBatch` in chunks of 1000 rows.
   - Return as `SendableRecordBatchStream`.
   - Internal tables are read-only from the DataFusion perspective (writes go through
     `WriteExecutor`, not via DataFusion DML â€” per AD-022).

   **`supports_filters_pushdown()` â†’ `Vec<TableProviderFilterPushDown>`:** Return
   `vec![TableProviderFilterPushDown::Inexact]` for all filters (no server-side filter
   push-down into RocksDB scan for MVP; full scan with DataFusion-side filtering).

6. Implement `register_internal_tables` at `internal_tables.rs:168`:
   ```rust
   pub fn register_internal_tables(
       ctx: &SessionContext,
       storage: Arc<dyn RocksStorageBackend>,
   ) -> Result<(), PrismError> {
       for domain in [
           StorageDomain::AuditBuffer,
           StorageDomain::Alerts,
           StorageDomain::Cases,
           StorageDomain::Schedules,
           // Add all 17 domains per AD-004 as tables become relevant
       ] {
           let provider = Arc::new(RocksDbTableProvider::new(storage.clone(), domain));
           ctx.register_table(domain.table_name(), provider)?;
       }
       Ok(())
   }
   ```
   This function is called from `prism-bin` boot step 7. The `table_name()` method returns
   e.g., `"prism_audit"`, `"prism_alerts"`, etc. â€” the SQL-addressable table names.

7. Inject virtual fields (`_sensor`, `_client`, `_source_table`) into `QueryResult` per BC-2.11.012:
   - After DataFusion executes the plan, add three Utf8 columns to the result `RecordBatch`:
     `_sensor` (sensor_name), `_client` (org_slug), `_source_table` (original table ref).
   - For multi-sensor fan-out results, each `RecordBatch` slice carries its originating
     source annotation before union.

8. Write integration tests in `crates/prism-query/tests/execute_integration_tests.rs`:
   - Test: execute a simple PrismQL query against a DTU-backed sensor â†’ non-empty result.
   - Test: execute `SELECT * FROM prism_audit LIMIT 10` â†’ returns `prism_audit` rows.
   - Test: cross-client query with ALL scope â†’ result includes rows from all configured orgs.
   - Test: query hitting BC-2.11.006 size limit â†’ `E-QUERY-003` (materialization records limit).
   - Test: query hitting timeout â†’ `E-QUERY-004` (timeout).
   - Test: query that exceeds depth limit â†’ `E-QUERY-005` (parse error).
   - Test: `_sensor` virtual field is present and non-empty in all results.

## Acceptance Criteria

**AC-1:** Given `QueryEngine::execute("SELECT * FROM crowdstrike_detections LIMIT 5", options)`,
When the CrowdStrike DTU is registered and the pipeline runs, Then a non-empty `QueryResult`
is returned with `results.len() <= 5` and the `_sensor` virtual field equals `"crowdstrike"`.
(traces to BC-2.11.001 postcondition â€” query tool accepts scoping + PrismQL)

**AC-2:** Given `run_materialization_pipeline` with a valid source ref, When the pipeline
runs, Then records are fetched via `SensorAdapter::fetch`, OCSF-normalized, converted to
Arrow `RecordBatch`, and registered as a `MemTable` in a fresh `SessionContext`. The
resulting context must be usable for DataFusion SQL execution without re-fetching from
the sensor.
(traces to BC-2.11.005 postconditions â€” ephemeral materialization)

**AC-3:** Given a query that would materialize more than 10,000 rows, When `run_materialization_pipeline`
runs, Then `PrismError::QueryError(E-QUERY-003)` is returned and no DataFusion execution begins.
(traces to BC-2.11.006 postcondition â€” security limits enforced)

**AC-4:** Given a sensor with a pushable filter (e.g., `WHERE hostname = 'target'`), When
`resolve_source_refs` processes the query, Then the filter predicate is extracted and
passed to `SensorAdapter::fetch` as a push-down parameter (not executed in DataFusion
after full materialization).
(traces to BC-2.11.007 postcondition â€” sensor filter push-down)

**AC-5:** Given `register_internal_tables(ctx, storage)` is called at boot step 7, When
a subsequent query executes `SELECT * FROM prism_audit LIMIT 20`, Then the query returns
rows from the RocksDB `audit_buffer` CF without error.
(traces to BC-2.15.011 postcondition â€” RocksDB domains as DataFusion tables)

**AC-6:** Given a cross-client query with `ALL` scope, When `resolve_source_refs` expands
the scope, Then each registered `(org_id, sensor_name)` pair is fanned out independently
and results are merged with the `_client` virtual field populated per org.
(traces to BC-2.11.011 postcondition â€” cross-client query scoping)

**AC-7:** Given any executed query, When the `QueryResult` is returned, Then all three
virtual fields `_sensor`, `_client`, `_source_table` are present as Utf8 columns with
non-null values for every row.
(traces to BC-2.11.012 postcondition â€” virtual fields in queries)

**AC-8:** No `todo!()`, `unimplemented!()`, or `panic!("stub")` may remain in any of
the nine implementation sites listed in the Objective before this story merges. Any
remaining stub residue blocks merge per POL-12.

## Dev Notes

- DataFusion `SessionContext` is ephemeral per query â€” construct a new one for each
  `execute()` call. Do not share `SessionContext` across queries (this is the "ephemeral
  data lake" pattern per AD-002).
- `RocksDbTableProvider::scan` should stream records in batches of 1000 rows rather than
  loading all CF data into memory at once. Use `StorageBackend::scan_range` if available,
  or prefix scan by StorageDomain key prefix.
- The `supports_filters_pushdown` returning `Inexact` means DataFusion applies its own
  filter post-scan. For P1, this is acceptable (full CF scan). A future story can add
  server-side prefix filtering using RocksDB seek.
- Test fixtures: use the DTU clones (prism-dtu-crowdstrike etc.) for sensor fan-out
  integration tests rather than mocking `SensorAdapter`. The DTU clones are available
  as dev-dependencies per the established pattern.
