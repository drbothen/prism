# S-1.12 — prism-spec-engine: Hot Reload and Runtime Management

## Objective

Implement hot reload for sensor specs and config, plus runtime management tools.

## Behavioral Contracts

| BC ID | Title |
|-------|-------|
| BC-2.16.005 | reload_config MCP Tool — Re-Read All Config Files, Validate, Atomic Swap, Notify |
| BC-2.16.006 | Arc-Swap Config Access on Hot Path — Lock-Free Reads for Query-Time Config Access |
| BC-2.16.007 | Sensor Spec Hot Reload — Add/Remove/Update Sensor Tables Without Restart |
| BC-2.16.008 | add_sensor_spec MCP Tool — Upload a New Sensor Spec at Runtime |
| BC-2.16.010 | list_sensor_specs MCP Tool — List Loaded Sensor Specs with Table Schemas and Status |

## Tasks

1. Implement `reload_config` logic: re-read all TOML files, validate, create new `ConfigSnapshot`, atomic swap via `arc-swap`, in-flight queries unaffected (BC-2.16.005)
2. Implement `ArcSwap`-based config access: `ConfigManager` wraps `ArcSwap<ConfigSnapshot>`, lock-free reads on query hot path (BC-2.16.006)
3. Implement sensor spec hot reload: detect add/remove/update of `.sensor.toml` files; add/remove SensorTableDescriptor entries and notify prism-query to update DataFusion tables without restart (BC-2.16.007)
4. Implement `add_sensor_spec` logic: accept TOML content, validate, write to spec directory, trigger reload (BC-2.16.008) — on write/cleanup failure return `E-SPEC-002` with path
5. Implement `list_sensor_specs` logic: return all loaded specs with table schemas, source count, and validation status (BC-2.16.010)
6. Write VP-032 proptest: if validation fails during reload, old config is retained (failed validation → no change)

## Acceptance Criteria

- **AC-1:** Given a modified `prism.toml`, When `reload_config` is called, Then the new config takes effect for subsequent queries while in-flight queries use the old config (BC-2.16.005)
- **AC-2:** Given concurrent query execution, When config is read, Then no lock contention occurs (arc-swap lock-free) (BC-2.16.006)
- **AC-3:** Given a new `.sensor.toml` dropped into the spec directory, When hot reload detects it, Then new tables appear in the query namespace (BC-2.16.007)
- **AC-4:** Given a valid TOML spec uploaded via `add_sensor_spec`, When it is processed, Then the spec is written and tables are registered (BC-2.16.008)
- **AC-5:** Given loaded specs, When `list_sensor_specs` is called, Then each spec's tables, source count, and status are returned (BC-2.16.010)
- **AC-6:** VP-032 proptest passes (failed validation retains old config)

## Verification Properties

| VP ID | Description |
|-------|-------------|
| VP-032 | Proptest: failed validation during reload retains old config unchanged |

## Notes

- `ArcSwap<ConfigSnapshot>` provides epoch-based memory reclamation. In-flight queries hold an `Arc` guard for the duration of execution; the old snapshot is not freed until all guards are dropped. This is the correct model — do not use a `RwLock`.
- Hot reload file detection can use `notify` crate (cross-platform filesystem events) or poll on a configurable interval. The mechanism must be configurable; the default is filesystem events.
- `add_sensor_spec` must validate the TOML before writing to disk. If validation fails, no file is written and the error is returned immediately.
- `E-SPEC-002` must include the filesystem path where the write failed and the underlying OS error, to support operator debugging without requiring log access.
- `list_sensor_specs` response must include a `status` field per spec: `loaded`, `failed_validation`, `pending_reload`. This gives operators visibility into partial failure states.
- DataFusion table unregistration on spec removal must handle the case where queries are currently executing against the removed table — those queries should complete against the old registration; new queries should see the table as absent.
