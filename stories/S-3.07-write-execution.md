# S-3.07 â€” prism-query: Write Execution Pipeline

## Narrative

As a PrismQL query engine, I want a write execution pipeline that routes parsed write
AST nodes through feature flag gates, risk classification, dry-run/confirm gating, and
sensor API dispatch so that analysts can take action on query results with the identical
safety guarantees that dedicated MCP write tools provide.

## Objective

Implement the six-phase write execution pipeline defined in AD-022. Phases 1 (parse) and
3 (fetch) reuse the existing read pipeline. This story adds: Phase 2 (safety pre-check),
Phase 4 (dry-run/confirm gate), Phase 5 (write dispatch), and Phase 6 (result + audit).
Extend `SensorAdapter` with a `write()` method. Extend DataFusion `TableProvider` write
stubs (using WriteTableDescriptors exported by S-1.13; TableProvider creation and catalog registration happen here in S-3.07) with real dispatch routing. Per AD-022.

2. Implement `WriteExecutor` in `crates/prism-query/src/write_pipeline.rs` â€” the
   top-level orchestrator for all six phases:
   ```rust
   pub struct WriteExecutor {
       feature_flags: Arc<dyn FeatureFlagResolver>,
       confirmation_store: Arc<dyn ConfirmationTokenStore>,
       audit_writer: Arc<dyn AuditWriter>,
       sensor_registry: Arc<SensorRegistry>,
       write_semaphore: Arc<Semaphore>,   // capacity: 4
   }

   impl WriteExecutor {
       pub async fn execute(
           &self,
           plan: WritePlan,       // from parsed WriteNode or DmlNode
           context: QueryContext,
           dry_run: bool,
       ) -> Result<WriteOutcome, PrismError>
   }
   ```
   `WriteOutcome` is `Either<WritePreview, WriteResult>`.

3. Implement Phase 2 (Safety Pre-check) in `crates/prism-query/src/safety_check.rs`.
   All operations in this phase are pure â€” no I/O:

   a. **Write target validation:** Verify the write target is a single external sensor
      source (not a composite source like EVENTS, not an internal prism_* table). If
      composite source: return `E-QUERY-020`. If internal table: return `E-QUERY-010`.

   b. **Compile-time feature flag gate (Gate 1):** Check that the Cargo feature
      `{sensor}-write` is compiled in for the target sensor (e.g., `crowdstrike-write`,
      `cyberint-write`, `claroty-write`, `armis-write` per BC-2.04.001). If absent, return
      `E-FLAG-002: write capability not compiled for sensor 'NAME'`.

   b. **Runtime TOML capability gate (Gate 2):** Call
      `FeatureFlagResolver::evaluate(capability_path, client_id)`. If `Deny` or
      no-match, return `E-FLAG-001`. Collect the evaluation result as an audit
      intent (emitted at the Phase 2â†’3 boundary, not during pure Phase 2 â€”
      per BC-2.05.009). Hidden verb: if `Deny`, the write verb should already
      be excluded from auto-complete (BC-2.04.005) â€” enforcement is here at Gate 2.

   c. **Unbounded write pre-check:** Inspect the write plan for the presence of
      a LIMIT clause or bounding WHERE on the source fetch. If absent, return
      `E-QUERY-022` before fetching any records.

   d. **Batch limit pre-check (structural):** Resolve `min(endpoint.batch_limit,
      client_override, system_ceiling)`. If the write plan has an explicit LIMIT
      that already exceeds the resolved batch limit, return `E-QUERY-021` immediately.
      If the LIMIT is within bounds, proceed. If there is no explicit LIMIT
      (E-QUERY-022 should have fired but did not due to a code path error), fail-safe
      with `E-QUERY-021`.

   e. **Risk tier classification:** Resolve `WriteEndpointSpec.risk_tier` for the
      target endpoint. For `DELETE FROM` SQL DML with no spec override, classify as
      `Irreversible` unconditionally (AD-022). Attach the resolved tier to the
      write plan for Phase 4.

4. Implement Phase 3 (Fetch) â€” reuse existing read fan-out from S-3.02:
   - Call `QueryMaterializer::execute(source_query, context)` to materialize the
     records that write operations will target
   - Apply the same 10K materialization cap and `GreedyMemoryPool` as read queries
   - The fetched `RecordBatch` is the write payload â€” its `record_id_field` column
     provides the IDs passed to the sensor API

5. Implement Phase 4 (Dry-Run / Confirm Gate) in `crates/prism-query/src/dry_run.rs`
   (BC-2.04.008):

   a. If `dry_run = true` (default): materialize `WritePreview` from the fetched
      records. For `Irreversible` tier, generate a `ConfirmationToken` via
      `ConfirmationTokenStore::create()` (S-1.09) and embed it in the preview.
      For `Reversible` tier, no token is generated. Return `WriteOutcome::Preview`.

   b. If `dry_run = false`:
      - For `Reversible` tier: proceed directly to Phase 5.
      - For `Irreversible` tier: require a valid `ConfirmationToken` to be present
        in `QueryContext`. Call `ConfirmationTokenStore::consume(token_id)`. If absent
        or expired, return `E-FLAG-008 (token not found) or E-FLAG-003 (token expired)`. If valid, proceed to Phase 5.

   c. No audit INTENT record is written for dry-run executions â€” Phase 5 is never
      entered when `dry_run = true`.

6. Implement Phase 5 (Write Dispatch) in `crates/prism-query/src/write_dispatch.rs`:

   a. **Audit INTENT record (fail-closed):** Write audit intent to RocksDB via
      `AuditWriter::write_intent(write_plan, context)` (S-2.04 pattern). If this
      write fails, abort with an error â€” no sensor API is contacted until the intent
      is durable. Return `E-AUDIT-001` if intent persistence fails.

   b. **Write semaphore acquisition:** Acquire from `write_semaphore` (capacity 4).
      This bounds concurrent write fan-out to 4 per query (separate from the 10-permit
      read semaphore from S-3.02).

   c. **Write fan-out:** For each `(client_id, sensor_name)` pair in the query context,
      call `SensorAdapter::write(endpoint_spec, record_batch_slice, params)`. Fan-out
      is parallel within the semaphore bound. Collect per-record results into
      `Vec<RecordWriteResult>`.

   d. **Per-record result tracking:** A failed per-record write does NOT abort the
      batch unless all records fail. Partial success is normal and expected. Accumulate
      failed records into `WriteResult.sensor_errors`. Only return a top-level error
      if audit persistence fails or the semaphore cannot be acquired.

   e. **Audit OUTCOME record:** After fan-out completes (all records attempted), write
      the outcome record via `AuditWriter::write_outcome(intent_id, result)`. If
      outcome persistence fails, log the failure but do NOT unwind the write â€” the
      sensor API calls are already complete.

7. Implement Phase 6 (Return): construct `WriteResult` from the assembled per-record
   results, count aggregates, and audit intent ID. Return `WriteOutcome::Result(write_result)`.

8. Extend `SensorAdapter` trait in `crates/prism-sensors/src/adapter.rs` with:
   ```rust
   async fn write(
       &self,
       endpoint: &WriteEndpointSpec,
       records: &RecordBatch,
       params: &HashMap<String, String>,
       client_id: &TenantId,
   ) -> Result<Vec<RecordWriteResult>, SensorError>
   ```
   Implement the default method body using the `steps` array from `WriteEndpointSpec`:
   - For each step: interpolate `body_template` and `path_template` using the
     `record_id_field` column from `records` and `params`
   - HTTP dispatch using the sensor's existing authenticated HTTP client
   - Parse response per `response_path` JSONPath; classify per `success_status` list

9. Wire `WriteCapableTableProvider` (using WriteTableDescriptors exported by S-1.13; TableProvider creation and catalog registration happen here in S-3.07) to route `insert_into()`,
   `update()`, and `delete_from()` DataFusion calls through `WriteExecutor::execute()`:
   - Extract `WritePlan` from the DataFusion `LogicalPlan` or `ExecutionPlan`
   - Call `WriteExecutor` with the `QueryContext` attached to the DataFusion
     `SessionContext` state extension
   - Return results as a DataFusion `SendableRecordBatchStream` (WriteResult serialized
     to a single-row schema for compatibility)
   - If `failed_count > 0 && succeeded_count > 0`: return `E-QUERY-025` (partial failure)
     with the full `WriteResult` including per-record detail. This is not an abort â€”
     successful records are committed; the error signals that the caller should inspect
     `per_record_results` for failed items.

10. Implement batch limit post-fetch check in `write_pipeline.rs` between Phase 3 and
    Phase 4:
    - After materialization, check `record_batch.num_rows()` against
      `min(endpoint.batch_limit, client_override, system_ceiling)`
    - If exceeded: return `E-QUERY-021` with counts: "Batch limit exceeded:
      query would affect N records; limit for 'ENDPOINT' on client 'CLIENT' is M"
    - No sensor API is contacted; no audit entry is written

11. Write unit and integration tests in `crates/prism-query/tests/write_pipeline_tests.rs`:
    - Dry-run default: `contain` with `dry_run=true` (default) â†’ `WritePreview` with
      `confirmation_token` present (Irreversible endpoint)
    - Token confirmation: valid token consumed â†’ `WriteResult` returned, sensor adapter
      called once
    - Batch limit: records > `batch_limit` â†’ `E-QUERY-021` before API call
    - Feature flag deny: `E-FLAG-001` before any fetch
    - Audit fail-closed: `AuditWriter::write_intent()` fails â†’ abort, no sensor call
    - SQL mode: `UPDATE crowdstrike_detections SET status='acknowledged' WHERE severity_id <= 2`
      â†’ same safety pipeline as pipe mode

## Acceptance Criteria

**AC-1:** Given `FROM crowdstrike_hosts | where hostname = 'target' | contain`, When
executed with `dry_run = true` (the default), Then `WritePreview` is returned with
`would_affect_count = 1`, `risk_tier = Irreversible`, and `confirmation_token` present.
(BC-2.04.007, BC-2.04.008)

**AC-2:** Given a `WritePreview` with a valid `confirmation_token`, When `confirm_action`
is called with the token ID before expiry, Then the write executes and `WriteResult` is
returned with `succeeded_count >= 1` and `audit_intent_id` populated. (BC-2.04.008)

**AC-3:** Given a write query where the source SELECT returns more records than
`batch_limit`, When the post-fetch batch limit check runs, Then `E-QUERY-021` is returned
with the affected count and limit values â€” before any sensor API call is made.

**AC-4:** Given a write targeting an internal table (e.g. `prism_alerts`), When the
pre-check runs, Then `E-QUERY-010` is returned (already caught at parse time by S-3.06,
but defense-in-depth check here also rejects it before any API contact).

**AC-5:** Given a write operation, When `AuditWriter::write_intent()` fails, Then the
entire write operation is aborted with `E-AUDIT-001` and no sensor API call is made.
(BC-2.05.009 fail-closed behavior)

**AC-6:** Given `UPDATE crowdstrike_detections SET status = 'acknowledged' WHERE severity_id <= 2`,
When executed via SQL mode through `WriteCapableTableProvider`, Then the same six-phase
safety pipeline runs as a pipe-mode write â€” same feature flag gates, same risk tier
classification, same dry-run default, same audit entry. (BC-2.04.007)

**AC-7:** Given a write endpoint with `capability_path = "sensor.armis.device_write"`
and the client's TOML sets `"sensor.armis.device_write" = "Deny"`, When Phase 2
safety pre-check runs, Then `E-FLAG-001` is returned and the evaluation outcome is
audit-logged regardless of deny. (BC-2.04.001, BC-2.05.009)

**AC-8:** Given `FROM crowdstrike_hosts | contain` with no WHERE clause and no LIMIT,
When Phase 2 unbounded write check runs, Then `E-QUERY-022` is returned before any
fetch or sensor API contact.

**AC-9:** Given a write verb (e.g., `contain`, `isolate`) targets a sensor whose write
capability is disabled for the requesting client, When Phase 2 safety pre-check runs
(or the MCP tool is invoked), Then the operation is rejected with `E-FLAG-001`
(structured capability error â€” not "unknown tool"), and that write tool is absent from
the client's `tools/list` response only if it is disabled for ALL configured clients;
if it is enabled for at least one other client it remains visible in `tools/list` and
per-client denial is enforced at invocation time.
(traces to BC-2.04.005 postcondition â€” Hidden Tools Pattern: write tool visibility is
stateless; per-client denial via `E-FLAG-001` at invocation time; tool absent from
`tools/list` only when disabled for all clients)

## Architecture Compliance Rules

- Phase 2 (safety pre-check) MUST be entirely pure â€” no `async fn`, no I/O. It receives
  all required inputs synchronously and returns a `Result`. Async I/O begins at Phase 3.
- The audit INTENT record (Phase 5a) MUST be a synchronous `await` that resolves before
  any sensor HTTP call is initiated â€” no fire-and-forget pattern
- Write semaphore capacity MUST be 4 â€” do not share with the read semaphore (10) from
  S-3.02. Write operations must be throttled more aggressively.
- `SensorAdapter::write()` MUST route through the same authenticated HTTP client as
  read-side fetch steps â€” do NOT create a second HTTP client or bypass credential resolution
- `WriteCapableTableProvider::insert_into/update/delete_from` MUST all route through
  `WriteExecutor::execute()` â€” no shortcuts that bypass the safety pipeline for SQL DML
- Partial batch failure (some records succeed, others fail) is NOT an error return from
  `WriteExecutor` â€” it is represented in `WriteResult.per_record_results`. Only
  infrastructure failures (audit, semaphore) are `Err()` returns.
- The `dry_run` parameter comes from the MCP tool layer (QueryContext), NOT from the
  query string â€” there is no PrismQL syntax for controlling dry-run mode

## Previous Story Intelligence

**S-3.02 (Query Materialization):** The read fan-out (Phase 3) is a direct reuse of
`QueryMaterializer::execute()`. Do not re-implement fan-out logic. The 10K row cap,
`GreedyMemoryPool`, and read semaphore all apply unchanged to the source fetch step.
The write pipeline adds a separate semaphore for the write fan-out phase only.

**S-1.08 (Feature Flags):** `FeatureFlagResolver::evaluate(capability_path, client_id)`
is the interface for Gate 2. Ensure `Phase2SafetyCheck` depends on the trait, not a
concrete type, for testability. Mock the feature flag resolver in unit tests.

**S-1.09 (Confirmation Tokens):** `ConfirmationTokenStore::create()` and `consume()`
are the interfaces for Phase 4 irreversible gate. Tokens have a 300-second TTL (AD-022).
`consume()` is single-use â€” a consumed token cannot be replayed.

**S-2.04 (Audit Construction):** The intent log pattern (write intent before action,
write outcome after) is established by S-2.04. Follow the same `AuditEntry` schema.
Add new `EventType::WriteIntent` and `EventType::WriteOutcome` variants. The write
pipeline must use `AuditWriter::write_intent()` synchronously before any HTTP call.

**S-3.06 (Write Parser):** `WriteNode` and `DmlNode` from S-3.06 are the inputs to
`WriteExecutor`. These types are located in `prism_query::write_ast` â€” the canonical
module path established by S-3.06 Â§AST Module Path. Import from there; do NOT define
a second copy of these AST types in S-3.07 scope. Ensure `WritePlan` (internal to the
execution pipeline) is trivially constructible from both node types â€” no re-parsing of
query strings in the executor.

## Dev Notes

- The write semaphore (capacity 4) is per `WriteExecutor` instance, not per query. If
  multiple concurrent MCP write tool calls arrive, they share the semaphore. The
  `write_rate_limit_per_minute = 200` ceiling from `prism.toml` is a future-scope
  rate limiter; implement the semaphore now, defer rate limiting to a later story.
- `SensorAdapter::write()` should log each HTTP step at `TRACE` level and the batch
  outcome at `INFO` level. Use structured fields: `sensor`, `client_id`, `endpoint_id`,
  `affected_count`, `succeeded_count`, `failed_count`.
- The `WriteCapableTableProvider` DataFusion integration (Task 9) is the most complex
  integration point. **DataFusion 53.1 API surface for `TableProvider::insert_into`
  (with `InsertOp` enum) requires confirmation at TDD time.** The story is sized at
  5 pts assuming existing DataFusion-provided extension hooks are sufficient. If
  extension-planner work is needed beyond what `TableProvider` exposes, file a
  W3-FIX-* story rather than expanding S-3.07 scope. The implementer MUST read
  DataFusion 53.1 source (`TableProvider` trait in `datafusion/src/datasource/provider.rs`)
  before implementing Task 9 to confirm the exact `insert_into`, `update`, and
  `delete_from` signatures and whether `InsertOp` is a required parameter.
- UPDATE/DELETE trait surface in DataFusion 53.x is also unconfirmed â€” some versions
  expose these as separate trait methods, others via `insert_into` with `InsertOp`.
  Treat this as a TDD-time verification gate, not a blocker for story dispatch.
- For the initial implementation, `DELETE FROM` SQL DML is always classified as
