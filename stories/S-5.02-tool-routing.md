# S-5.02 — prism-mcp: Tool Routing, Errors, and Client Scoping

## Narrative

As an MSSP analyst using Prism, I want every tool call to be scoped to a specific client
(or set of clients) and to receive structured, actionable error responses when something
goes wrong, so that I can reason about failures and route queries correctly in a
multi-tenant environment.

## Objective

Implement client_id validation middleware, structured error responses with all required
fields, and the `list_capabilities` meta-tool. This story closes the gap between raw
rmcp dispatch and safe multi-tenant tool execution.

## Tasks

1. Implement `ClientIdGuard` middleware in `crates/prism-mcp/src/middleware/client_id.rs`:
   - Every tool input struct that targets a single client MUST include `client_id: String`
   - Every tool input struct that targets multiple clients MAY include `clients: Vec<String>`
   - On entry: validate each `client_id` via `TenantId::new()` from prism-core
   - If `client_id` is missing or empty: return `E-MCP-001` immediately, do not dispatch
   - If `client_id` is present but invalid (bad chars, length): return `E-MCP-001` with
     the specific validation failure reason (BC-2.10.004)

2. Implement `ToolError` → `rmcp::CallToolResult` conversion in
   `crates/prism-mcp/src/error_response.rs`:
   - Output struct: `{ error_code, category, message, retryable, retry_after_seconds,
     suggestion, original_params_valid }`
   - `error_code`: from `PrismError` enum (e.g., `"E-MCP-001"`)
   - `category`: one of `auth`, `sensor`, `query`, `config`, `safety`, `internal`
   - `message`: human-readable explanation; upstream sensor messages stripped and
     replaced with sanitized version (never pass raw sensor error text to AI context)
   - `retryable`: `bool` — true for transient errors (rate limit, timeout), false for
     permanent (invalid params, auth invalid)
   - `retry_after_seconds`: `Option<u64>` — populated from HTTP 429 headers when known
   - `suggestion`: short actionable string, e.g., `"Check credential_ref in prism.toml"`
   - `original_params_valid`: `bool` — true if the parameters were structurally valid
     (error was not caused by bad input shape) (BC-2.10.007)

3. Implement `list_capabilities` tool in `crates/prism-mcp/src/tools/meta.rs`:
   - Input: `{ client_id: String }`
   - Resolve effective `ClientCapabilities` for the given client by calling
     `prism-core`'s capability resolution with config-loaded defaults + per-client
     overrides
   - Output: `{ client_id, effective_capabilities: Map<String, CapabilityValue>,
     resolution_trace: Vec<ResolutionStep> }`
   - `resolution_trace` shows each layer that contributed (global default → client
     override → result) to make debugging intuitive (BC-2.10.011)

4. Write unit tests in `crates/prism-mcp/tests/routing.rs`:
   - Test missing `client_id` returns `E-MCP-001`
   - Test invalid `client_id` value returns `E-MCP-001` with validation message
   - Test `clients: ["acme", "acme/../../etc"]` rejects the invalid entry
   - Test error response struct has all required fields populated
   - Test `list_capabilities` returns resolution trace with at least one step

## Acceptance Criteria

**AC-1:** Given a tool call with no `client_id` field, When the tool is dispatched, Then
the response is an error with `error_code: "E-MCP-001"` and `original_params_valid: false`.
(traces to BC-2.10.004 postcondition 1)

**AC-2:** Given a tool call with `clients: ["acme", "acme/../../etc"]`, When validated,
Then the call is rejected with `E-MCP-001` identifying `"acme/../../etc"` as the invalid
entry.
(traces to BC-2.10.004 postcondition 2)

**AC-3:** Given any tool error response, When the response JSON is inspected, Then all
seven fields (`error_code`, `category`, `message`, `retryable`, `retry_after_seconds`,
`suggestion`, `original_params_valid`) are present with correct types.
(traces to BC-2.10.007 postcondition 1)

**AC-4:** Given a sensor rate-limit error (HTTP 429 with `Retry-After: 30`), When
converted to a tool error response, Then `retryable: true` and `retry_after_seconds: 30`.
(traces to BC-2.10.007 postcondition 2)

**AC-5:** Given `list_capabilities(client_id: "acme")` for a client with a default
read-only capability set and one write override, When called, Then the response includes
both the merged effective capabilities and a `resolution_trace` with at least two steps.
(traces to BC-2.10.011 postcondition 1)

## Architecture Compliance Rules

- Upstream sensor error messages MUST be sanitized before appearing in `message` — raw
  sensor API responses may contain internal system details or prompt-injection payloads
- `client_id` validation MUST use `TenantId::new()` from prism-core — do not duplicate
  the regex validation logic in prism-mcp
- The `list_capabilities` tool MUST call the same capability resolver used by the query
  engine (S-1.03), not a separate implementation
- `ToolError` serialization MUST produce a flat JSON object (not nested), as Claude Code
  reads tool error content directly
- `retry_after_seconds` MUST be `null` in JSON (not absent) when not applicable, for
  consistent client-side handling

## Notes

- The `resolution_trace` in `list_capabilities` is diagnostic output for analyst
  debugging (e.g., "why can't I run this query for client X?"). It should be concise —
  one line per resolution step, not a full config dump.
- Prompt injection defense (S-1.10) applies to sensor error messages surfaced through
  `message`. Run the sanitizer before populating that field.
- For the `clients` multi-target case (used by the `query` tool to fan out), reject the
  entire call if any client_id is invalid — do not partially succeed.
