# S-5.06 — prism-mcp: Action and Infusion MCP Tools

## Narrative

As a Claude Code analyst working within Prism, I want MCP tools to inspect, manage, and
manually trigger actions and infusions, so that I can diagnose enrichment data freshness,
reload stale infusion databases, monitor action delivery history, trigger one-off
notifications, and inspect WASM plugin health — all without leaving my Claude Code
session.

## Objective

Register MCP tools for the action delivery framework (S-4.08) and infusion enrichment
framework (S-1.14): CRUD and status inspection tools for actions, infusions, and WASM
plugins, following existing tool registration patterns (S-5.01, S-5.02) with client_id
scoping, structured errors, and audit logging. Implement `fire_action` for manual
trigger and `reload_infusion` for on-demand data refresh.

2. Implement `reload_infusion` tool in `crates/prism-mcp/src/tools/infusion/reload_infusion.rs`:
   - Parameters: `infusion_id: String`
   - Triggers an immediate data reload for the named infusion (re-reads the source file,
     regardless of `refresh_interval`)
   - For `type = "local_lookup"` infusions: re-read the source file, build new lookup
     structure, arc-swap. Returns `{ "status": "reloaded", "infusion_id": "geoip",
     "records_loaded": 18432156, "duration_ms": 420 }`
   - For `type = "plugin"` infusions: reload is handled by the plugin itself — return
     `{ "status": "not_applicable", "reason": "Plugin infusions do not have local data files. Use reload_plugin to reload the WASM plugin." }`
   - Capability gate: `infusion.write` (treat as a write operation — data reloads affect
     running queries)
   - On unknown `infusion_id`: return structured error `E-INFUSE-006: "Infusion '{id}' not found."`
   - Audit-log the reload event

7. Implement `fire_action` tool in `crates/prism-mcp/src/tools/action/fire_action.rs`:

   - Fire-and-forget per INV-ACTION-003: execute the delivery, return result immediately

   - On plugin action: `external_id` may be populated (e.g., Jira ticket key `"ACME-SEC-1234"`)
   - Inject `context.analyst_id` and `context.client_id` for scoping
   - Capability gate: `action.write`
   - Audit-log with `{ event: "action_manual_trigger", action_id, analyst_id, outcome }`

## Acceptance Criteria

**AC-1:** Given `list_infusions` is called, When the `InfusionRegistry` has `geoip` and
`asset_inventory` loaded, Then the response contains both entries with `status: "loaded"`,
`fields` arrays, and `cache_hit_rate_percent` values.
(traces to INV-TOOL-004)

**AC-2:** Given `reload_infusion` is called with `infusion_id = "geoip"`, When the
MaxMind MMDB file exists on disk, Then the file is re-read, the new reader is arc-swapped
in, and the response contains `records_loaded` and `duration_ms`.

**AC-3:** Given `fire_action` is called without `action.write` capability, When the
MCP server processes the request, Then the response is `E-FLAG-001` (capability denied)
and no delivery is attempted.
(traces to INV-TOOL-001)

**AC-4:** Given `fire_action` is called with `action.write` capability and
`action_id = "slack_soc_critical"`, When the action fires, Then the webhook is called
with a rendered payload containing the caller-provided context variables, an `action_manual_trigger`
audit event is logged, and the response contains `{ "status": "delivered", "duration_ms": N }`.
(traces to INV-TOOL-001, INV-TOOL-003)

**AC-5:** Given `action_status` is called for an action with 2 pending retries and 1
dead-letter entry, When the `action_state` CF is queried, Then the response JSON contains
`pending_retries` array with 2 entries and `dead_letters` array with 1 entry, each with
`alert_id`, `attempt`/`failed_at`, and error message.
(traces to INV-TOOL-004)

**AC-6:** Given `test_action` is called for a webhook action, When the synthetic payload
is delivered, Then rate limiting is bypassed (the action can fire even if `max_per_hour`
is 0), the delivery result is returned, and the test is audit-logged with
`{ "event": "action_test_delivery" }`.
(traces to INV-TOOL-002, INV-TOOL-003)

**AC-7:** Given `list_plugins` is called, When the plugin runtime has `threat_intel.prx`
and `jira.prx` loaded, Then both are returned with `plugin_type`, `status`, `version`,
and `last_loaded` fields.
(traces to INV-TOOL-004)

**AC-8:** Given `reload_plugin` is called with `action.write` (or `plugin.write`) capability
and a valid `plugin_id`, When the `.prx` file is recompiled and swapped, Then the response
contains `{ "status": "reloaded", "duration_ms": N }` and a `plugin_reload` audit event
is logged.
(traces to INV-TOOL-003, BC-2.17.005 — atomic swap; in-flight calls complete against old module)

**AC-9:** Given `fire_action` is called with `action.write` capability and `trigger = "manual"`,
When the action fires, Then the destination is called exactly once with no retry on failure,
and the response is returned immediately without blocking beyond the destination timeout.
(traces to BC-2.18.003 — manual triggers are fire-and-forget, result returned immediately)

**AC-10:** Given `reload_infusion` is called with a valid `infusion_id` for a local-lookup infusion,
When the MMDB or CSV file is re-read and arc-swapped, Then in-flight queries using the old
registry complete without error and new queries use the updated data.
(traces to BC-2.19.004 — infusion hot reload atomicity; failed validation retains previous registration)

**AC-11:** Given any tool in this story is invoked (success or failure), When the MCP
dispatch middleware processes the call, Then exactly one `AuditEntry` is written to the
audit buffer for that invocation.
(traces to BC-2.05.001 — audit entry per tool invocation, regardless of outcome)

SS-18 is owned exclusively by `prism-operations`; SS-19 is owned exclusively by
`prism-spec-engine`.

## Architecture Compliance Rules

- All tools MUST use the `client_id` scoping pattern from S-5.02 — never return
  data from one client's actions to a request scoped to another client.
- Capability gates (`action.write`, `plugin.write`) MUST be checked via
  `CapabilityResolver::resolve(capability, client_id, analyst_id)` from S-1.03 — do
  NOT implement ad-hoc capability checks in tool handlers.
- `fire_action` MUST fire the action synchronously and return the result in the same
  MCP tool response. Fire-and-forget means no background tasks that outlive the tool
  call — the response IS the delivery report.
- `test_action` MUST clearly mark its test payload as synthetic in the audit log and
  in the delivery payload itself. Add a `"prism_test": true` field to the test template
  context so downstream systems (Slack, PagerDuty) can identify and ignore test payloads.
- Tool response schemas MUST be defined in the tool's JSON schema declaration (MCP tool
  registration). AI agents depend on stable schemas for reasoning — do not return ad-hoc
  prose in success responses.
- `create_action` writes a new `.toml` file to disk. Use atomic write (write to temp
  file, rename) to avoid partial writes that corrupt the config directory. Follow the
  same atomic write pattern as sensor spec writes in S-1.13.

## Previous Story Intelligence

S-5.01 (Server Bootstrap and Tool Registration) established the `register_tool(name, description, schema, handler)` pattern and the `PrismContext` struct. Add `action_engine: Arc<ActionEngine>`, `infusion_registry: Arc<InfusionRegistry>`, and `plugin_runtime: Arc<PluginRuntime>` to `PrismContext` and populate them during server initialization.

S-5.02 (Tool Routing, Errors, and Client Scoping) established the `client_id` scoping
pattern and the `E-FLAG-001` structured error for capability denials. Reuse these error
codes and the capability check pattern verbatim — do not invent new patterns.

S-5.04 (Sensor Health Subsystem) provides a good example of how to expose operational
status data through MCP tools (the `check_sensor_health` tool pattern). Follow the same

## Dev Notes

- The 13 new tools registered in this story bring the total MCP tool count from 18
  (S-5.01 through S-5.05) to 31. Verify the tool registration loop in S-5.01 does not
  have a hardcoded tool count limit.
- `create_action` writes a file to disk. The config directory (`{config_dir}/actions/`)
  must be writable by the Prism process. If it is not (e.g., read-only mount), return
  `E-ACTION-011: "Cannot create action spec: config directory is not writable."` rather
  than panicking.
- `action_status` reads from the `action_state` CF using a prefix scan on `"{action_id}:"`.
  Be careful about the prefix scan not leaking into adjacent key prefixes — use an
  explicit upper bound in the RocksDB iterator.
- Tool response JSON should include a `"timestamp"` field at the top level so AI agents
  can reason about data freshness. Follow the pattern from `check_sensor_health` responses.
- The `plugin.write` capability may not yet be defined in the capability registry
  (established in S-1.03). If it is not defined, add it to the capability tree alongside
  `action.write`. Coordinate with S-1.03's capability list before adding new capability
  names.
