# S-6.12 — prism-dtu-pagerduty: DTU for PagerDuty Events API v2 — L3 (behavioral)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the PagerDuty Events
API v2 at fidelity level L3 (behavioral), so that integration tests for PagerDuty action
delivery can validate the full incident lifecycle (trigger → acknowledge → resolve),
deduplication key semantics, severity mapping, and 429/auth-fail error paths without
hitting the live PagerDuty API.

## Objective

Implement `prism-dtu-pagerduty` — an L3 behavioral clone of the PagerDuty Events API v2.
The clone implements the single `/v2/enqueue` endpoint with full stateful incident
lifecycle tracking (triggered → acknowledged → resolved), dedup key enforcement, severity
validation, and configurable error injection. See `dtu-assessment.md §3.5.2` for full
scope matrix.

## Tasks

1. Create `crates/prism-dtu-pagerduty/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `PagerDutyClone` struct implementing `BehavioralClone`:
   - `start()`: binds ephemeral port, spawns axum router
   - `reset()`: calls `state.reset()`
   - `incidents(&self) -> Vec<IncidentRecord>`: test API — returns current incident registry

3. Create `src/state.rs` — `PagerDutyState`:
   - `incident_registry: Mutex<HashMap<String, IncidentRecord>>` —
     `dedup_key → IncidentRecord`
   - `IncidentRecord { dedup_key: String, status: IncidentStatus, severity: String, summary: String }`
   - `IncidentStatus` enum: `Triggered`, `Acknowledged`, `Resolved`
   - `reset()` clears the registry

4. Create `src/routes/enqueue.rs`:
   - `POST /v2/enqueue`:
     - Required fields: `routing_key` (string), `event_action` (one of `trigger`,
       `acknowledge`, `resolve`), `payload` (object with `summary`, `severity`, `source`)
     - `routing_key` validation: if `auth_mode == "reject"`, return 403
       `{"status": "invalid key", "message": "Forbidden"}`
     - `event_action` validation: if not one of the three accepted values, return 400
       `{"status": "invalid event_action"}`
     - Missing `routing_key`: return 400 `{"status": "missing routing_key"}`
     - Severity validation: accepted values are `critical`, `error`, `warning`, `info`
       only; other values return 400 `{"status": "invalid severity"}`
     - `dedup_key`: read from body if present; if absent, generate a UUID

     **`trigger` action:**
     - If `dedup_key` already exists in registry with status `Triggered` or `Acknowledged`:
       return 202 `{"status": "success", "message": "Event processed", "dedup_key": "..."}`
       WITHOUT creating a new incident (idempotent per PagerDuty spec)
     - If `dedup_key` is new OR previous incident was `Resolved`: create new incident in
       registry with status `Triggered`; return 202 `{"status": "success", "dedup_key": "..."}`

     **`acknowledge` action:**
     - If `dedup_key` not in registry: return 400 `{"status": "invalid dedup_key"}`
     - If incident status is `Resolved`: return 400
       `{"status": "cannot acknowledge a resolved incident"}`
     - Otherwise: transition to `Acknowledged`; return 200 `{"status": "success"}`

     **`resolve` action:**
     - If `dedup_key` not in registry: return 400 `{"status": "invalid dedup_key"}`
     - Transition to `Resolved` regardless of current status (resolve is idempotent)
     - Return 200 `{"status": "success"}`

5. Rate-limit behavior:
   - When `FailureMode::RateLimit { after_n_requests }` configured:
     - Return 429 with `Retry-After: 60` — maps to Prism action retry

6. Create `GET /dtu/incidents` test API:
   - Returns `{"incidents": [{dedup_key, status, severity, summary}, ...]}` for all
     incidents currently in the registry

7. Create `POST /dtu/configure` and `POST /dtu/reset`.

8. Create fixture files:
   - `fixtures/trigger-response.json`: example 202 response for trigger action
   - `fixtures/acknowledge-response.json`: example 200 response for acknowledge action
   - `fixtures/resolve-response.json`: example 200 response for resolve action
   - (These serve as schema references for the fidelity validator)

9. Create `tests/fidelity.rs`:
   - Tests the full create→acknowledge→resolve lifecycle
   - Asserts dedup key idempotency (second trigger with same dedup_key does not create
     new incident)
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /v2/enqueue` with `event_action: "trigger"` and a new `dedup_key`,
Then the response is HTTP 202 `{"status": "success", "dedup_key": "..."}` AND
`GET /dtu/incidents` shows the incident with status `Triggered`.

**AC-2:** Given a subsequent `POST /v2/enqueue` with `event_action: "acknowledge"` and
the same `dedup_key`, Then the response is HTTP 200 `{"status": "success"}` AND
`GET /dtu/incidents` shows status `Acknowledged`.

**AC-3:** Given a subsequent `POST /v2/enqueue` with `event_action: "resolve"` and the
same `dedup_key`, Then the response is HTTP 200 AND status is `Resolved`.

**AC-4:** Given `event_action: "acknowledge"` on a `Resolved` incident, Then the
response is HTTP 400 `{"status": "cannot acknowledge a resolved incident"}`.

**AC-5:** Given `event_action: "trigger"` with the same `dedup_key` as an unresolved
incident, Then the response is HTTP 202 without creating a duplicate incident (idempotent).

**AC-6:** Given `severity: "fatal"` (not in the accepted set), Then the response is
HTTP 400 `{"status": "invalid severity"}`.

**AC-7:** Given `routing_key` is missing from the body, Then the response is HTTP 400
`{"status": "missing routing_key"}`.

**AC-8:** Given `auth_mode == "reject"` configured via `POST /dtu/configure`, Then any
request returns HTTP 403 `{"status": "invalid key"}` — maps to `E-ACTION-AUTH-001`.

**AC-9:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then
HTTP 429 with `Retry-After: 60`.

## Architecture Compliance Rules

- DTU clone MUST match the PagerDuty Events API v2 contract exactly (same `/v2/enqueue`
  endpoint, same request/response shapes, same status codes) as documented in
  `dtu-assessment.md §3.5.2`.
- DTU clone MUST NOT introduce new behaviors not present in the real PagerDuty Events API.
- DTU clone MUST be deterministic for a fixed seed; dedup key UUIDs MUST use
  ChaCha20Rng from `prism-dtu-common` when auto-generating.
- DTU clone MUST NOT require network access during test execution.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]`.
- Severity validation MUST be case-sensitive: `"CRITICAL"` MUST return 400 — lowercase
  only (`critical`, `error`, `warning`, `info`) per PagerDuty spec.
- Dedup key semantics MUST match PagerDuty: re-triggering an active incident is a no-op
  (idempotent 202); triggering a resolved incident starts a fresh incident.
- See `.factory/specs/architecture/dtu-strategy.md` for cross-cutting DTU rules.
- **Forbidden dependencies:** `prism-dtu-pagerduty` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.
