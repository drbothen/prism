# S-2.04 — prism-audit: Audit Entry Construction and Compliance

## Narrative

As a Prism security operator, I want every MCP tool invocation to produce exactly one
immutable, SOC 2 / ISO 27001-compliant audit entry with credentials redacted, so that
the platform's complete action history is preserved for compliance reporting and
forensic review.

## Objective

Implement the `prism-audit` crate as the compliance audit layer. This includes the
`AuditEmitter` middleware that wraps every MCP tool invocation, the `AuditEntry` struct
with all required compliance fields, credential redaction, write-operation detail
capture, and append-only storage into the RocksDB `audit_buffer` CF. The fail-closed
contract for write operations is a hard requirement.

## Tasks

1. Define `AuditEntry` struct in `crates/prism-audit/src/audit_entry.rs`:
   - `trace_id: Uuid` — V7 UUID correlating all events for a single invocation
   - `timestamp: DateTime<Utc>` — RFC 3339 nanosecond precision
   - `tool_name: String` — name of the MCP tool called
   - `client_id: String` — from the request context (ISO 27001: "who")
   - `parameters: serde_json::Value` — redacted copy of tool parameters (ISO 27001: "what")
   - `outcome: AuditOutcome` — `Success | Failure { error_code: String }`
   - `duration_ms: u64` — wall-clock time from tool entry to audit emission
   - `error_code: Option<String>` — structured error code if `Failure`
   - `data_classification: DataClassification` — `Public | Internal | Confidential | Restricted`
   - `system_id: String` — fixed string `"prism"` (ISO 27001: "where") (BC-2.05.002, BC-2.05.008)
   - Derive: `Serialize, Deserialize, Debug, Clone`

2. Implement `AuditEmitter` in `crates/prism-audit/src/audit_emitter.rs`:
   - Tower `Layer` + `Service` implementation wrapping each MCP tool handler
   - Before calling the inner handler: record `start_time`, generate `trace_id`
   - After calling the inner handler: construct `AuditEntry`, call `emit(entry)`
   - `emit(entry: AuditEntry) -> Result<(), PrismError>`:
     serialize to JSON, write to `audit_buffer` CF via `StorageBackend::put()`
     (BC-2.05.001)

3. Implement fail-closed contract for write operations in `AuditEmitter`:
   - Classify each tool as `ReadTool` or `WriteTool` via a `ToolClassification` registry
   - For `WriteTool`: if `emit()` returns `Err`, abort the write operation before it
     executes and return `Err(PrismError::AuditPersistenceFailed)` (E-AUDIT-001) to the
     caller — the write MUST NOT proceed without a persisted audit entry (BC-2.05.001)
   - For `ReadTool`: if `emit()` returns `Err`, log the failure but do NOT abort the
     read — reads are not fail-closed

4. Implement credential redaction in `crates/prism-audit/src/redaction.rs`:
   - `redact(params: serde_json::Value) -> serde_json::Value`:
     recursively walks the JSON value; replaces any string value whose key matches
     a credential pattern with `"***REDACTED***"` (BC-2.05.003)
   - Credential key patterns (case-insensitive): `password`, `secret`, `token`,
     `api_key`, `credential`, `private_key`, `passphrase`, `bearer`
   - Object keys matching patterns: replace the entire value regardless of type
   - Called on `parameters` before constructing `AuditEntry`

5. Implement write operation audit detail in `crates/prism-audit/src/write_audit.rs`:
   - `WriteAuditDetail` struct embedded in `AuditEntry.parameters` for write tools:
     - `capability_check: CapabilityCheckResult` — `Granted | Denied { reason: String }`
     - `risk_tier: RiskTier` — `Low | Medium | High | Critical`
     - `confirmation_token_used: Option<String>` — token ID if a confirmation token was
       consumed for this write
     - `execution_outcome: WriteOutcome` — `Committed | RolledBack | Aborted`
     (BC-2.05.004)

6. Implement append-only storage contract in `AuditEmitter::emit()`:
   - Keys in `audit_buffer` CF use the format `audit:{timestamp_ns}:{trace_id}` ensuring
     monotonic ordering
   - No `StorageBackend::remove()` or `put()` with an existing key is ever called on
     `audit_buffer` — only `put()` with new keys (BC-2.05.006)
   - The `StorageDomain::AuditBuffer` domain MUST NOT be passed to `remove()` anywhere
     in `prism-audit`; add a `#[deny(unused_must_use)]` lint and a doc comment warning

7. Ensure SOC 2 Type II and ISO 27001 field completeness in `AuditEntry`:
   - SOC 2 required: who (`client_id`), what (`tool_name` + `parameters`), when
     (`timestamp`), where (`system_id`), outcome (`outcome`)
   - ISO 27001 adds: `data_classification` field on every entry (BC-2.05.008)
   - Write a compile-time assertion (const check) that `AuditEntry` fields include all
     required names; use a `static_assertions` crate check

## Acceptance Criteria

**AC-1:** Given a query tool invocation that completes successfully, When `AuditEmitter`
processes the response, Then exactly one audit entry exists in the `audit_buffer` CF with
matching `trace_id`. (BC-2.05.001)

**AC-2:** Given a write tool invocation where `emit()` fails (storage unavailable), When
`AuditEmitter` processes the write, Then the write operation is aborted and
`E-AUDIT-001` is returned to the caller; no write side effect occurs. (BC-2.05.001)

**AC-3:** Given an audit entry for any tool invocation, When it is serialized to JSON,
Then it contains all fields: `trace_id`, `timestamp`, `tool_name`, `client_id`,
`parameters`, `outcome`, `duration_ms`, `data_classification`, `system_id`. (BC-2.05.002,
BC-2.05.008)

**AC-4:** Given a credential set operation with `parameters: { "secret": "abc123" }`, When
the audit entry is constructed, Then `entry.parameters["secret"] == "***REDACTED***"`.
(BC-2.05.003)

**AC-5:** Given a write operation audit entry, When the entry is examined, Then
`parameters.WriteAuditDetail` contains `capability_check`, `risk_tier`, and
`execution_outcome` fields. (BC-2.05.004)

**AC-6:** Given an audit entry has been written to the `audit_buffer` CF with key
`audit:{timestamp_ns}:{trace_id}`, When any code path in `prism-audit` is examined,
Then no `StorageBackend::remove()` call with `StorageDomain::AuditBuffer` exists and
no `put()` call reuses an existing audit key — only monotonically-increasing new keys
are written. The `StorageDomain::AuditBuffer` domain is never passed to `remove()`
anywhere in `prism-audit` (enforced via a clippy lint or integration test that scans
for the pattern). If a correction to a prior entry is needed, a new audit entry is
emitted referencing the original `trace_id`, not an overwrite of the existing entry.
(traces to BC-2.05.006 postcondition — audit entries are append-only and immutable)

## Architecture Compliance Rules

- `prism-audit` depends on `prism-core` and `prism-storage` (S-2.01, S-2.02) only
- `AuditEmitter` MUST be a Tower `Layer` — it integrates with the MCP transport layer
  via the standard Tower middleware stack, not via ad-hoc function wrapping
- `StorageDomain::AuditBuffer` MUST NEVER be passed to `StorageBackend::remove()` in
  this crate — enforce via a clippy lint or integration test that scans for the pattern
- Credential redaction MUST happen before serialization — the `parameters` field in
  `AuditEntry` must never contain a raw credential value, even transiently in memory
- The fail-closed check for write tools MUST occur before the inner handler is called —
  not after; the Tower middleware must await `emit()` before forwarding the request

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Write tool invocation where `emit()` fails before write executes | Write aborted; `E-AUDIT-001` returned to caller; no write side effect (fail-closed, BC-2.05.001) |
| EC-002 | Read tool invocation where `emit()` fails | Failure logged at ERROR; read proceeds normally (reads are not fail-closed) |
| EC-003 | Parameters contain nested object with key `"api_key"` at any depth | `redact()` replaces entire value with `"***REDACTED***"` regardless of nesting level (BC-2.05.003) |
| EC-004 | Two concurrent tool invocations produce entries with same millisecond timestamp | Keys use nanosecond timestamp + trace_id; uniqueness guaranteed even at sub-millisecond resolution (BC-2.05.006) |
| EC-005 | Audit entry emitted with `StorageDomain::AuditBuffer` passed to `remove()` | Compile-time clippy lint or integration test detects and blocks this pattern (BC-2.05.006 append-only invariant) |

## Dev Notes

- Tower `Layer` requires `Clone` on the service — ensure `AuditEmitter` derives or
  implements `Clone`. Use `Arc<StorageBackend>` for the shared backend handle.
- The `ToolClassification` registry (ReadTool vs WriteTool) must be populated at startup
  from the MCP tool manifest; it is a static `HashMap<&'static str, ToolClass>`.
- `data_classification` defaults to `Internal` for all tool invocations unless the tool
  manifest specifies otherwise (e.g., `credential_*` tools are `Confidential`).
- VP-033 (Audit buffer RocksDB-write-before-delivery ordering) is an integration test
  owned by S-6.07 (prism-dtu-crowdstrike). S-2.04 implements the audit buffer write
  ordering; S-6.07's DTU provides the test vehicle. The test uses `InMemoryBackend`
  from S-2.01; the ordering guarantee comes from the key format, not RocksDB internals.
