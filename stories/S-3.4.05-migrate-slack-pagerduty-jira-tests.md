# S-3.4.05 — Migrate prism-dtu-slack/pagerduty/jira tests to prism-dtu-harness (shared-mode)

## Narrative

As a Prism integration-test maintainer, I want the `prism-dtu-slack`,
`prism-dtu-pagerduty`, and `prism-dtu-jira` test suites rewritten to use
`prism-dtu-harness` with `mode = "shared"`, so that MSSP Coordination DTUs are
tested under the same harness framework as Security Telemetry DTUs while respecting
their shared-instance deployment model.

## Objective

Rewrite tests for Slack, PagerDuty, and Jira DTUs to use `HarnessBuilder` with shared
mode. These three are MSSP Coordination types — they run as single shared instances
across all customer orgs, distinguishing org payloads by embedding `OrgId` in the request
body (BC-3.2.004). The harness wraps the single shared instance; tests verify that
`OrgId` appears correctly tagged in outbound payloads. Combined story because these three
crates have structurally similar (smaller) test matrices.

## Tasks

1. For each of Slack, PagerDuty, and Jira:
   a. Read existing `crates/prism-dtu-{slack,pagerduty,jira}/tests/` to catalog test functions.
   b. Rewrite `tests/harness_tests.rs` using harness with `mode = "shared"` designation in the `CustomerSpec`.

2. Shared-mode harness pattern for MSSP Coordination DTUs:
   - `Harness::builder().isolation(IsolationMode::Logical).with_customer("mssp-shared").build().await`
   - The "mssp-shared" customer spec registers a single Slack/PagerDuty/Jira clone with `mode = "shared"`
   - Tests dispatch actions on behalf of specific `OrgId` values

3. For each dispatched action, assert that the captured outbound payload body contains the `OrgId` UUID (not the slug) as a structured field:
   - Slack: `OrgId` in Block Kit context block
   - PagerDuty: `OrgId` in `custom_details` field
   - Jira: `OrgId` in a designated issue field
   - `OrgId` MUST NOT appear in HTTP headers or URL path segments (BC-3.2.004)

4. Add `prism-dtu-harness` to dev-deps in all three `Cargo.toml` files.

5. Verify that `mode = "client"` for any Slack/PagerDuty/Jira DTU in a test `CustomerSpec` does not produce a startup error (MSSP Coordination types permit client mode override — BC-3.3.001-startup EC-003).

## Acceptance Criteria

**AC-001:** All existing Slack DTU tests pass against the harness-hosted clone in `IsolationMode::Logical` with `mode = "shared"`.
(traces to BC-3.5.001 postcondition 1)

**AC-002:** All existing PagerDuty DTU tests pass against the harness-hosted clone.
(traces to BC-3.5.001 postcondition 1)

**AC-003:** All existing Jira DTU tests pass against the harness-hosted clone.
(traces to BC-3.5.001 postcondition 1)

**AC-004:** For a dispatched Slack action on behalf of `OrgId(A)`, the captured outbound webhook payload body contains the `OrgId(A)` UUID as a structured field; the `OrgId` does not appear in any HTTP header or URL path segment.
(traces to BC-3.2.004 postcondition — OrgId in payload body not routing headers)

**AC-005:** For a dispatched PagerDuty alert on behalf of `OrgId(B)`, the captured payload `custom_details` field contains the `OrgId(B)` UUID.
(traces to BC-3.2.004 postcondition)

**AC-006:** For a created Jira ticket on behalf of `OrgId(C)`, the captured payload contains the `OrgId(C)` UUID in the designated issue field.
(traces to BC-3.2.004 postcondition)

**AC-007:** A `CustomerSpec` with `mode = "client"` for Slack/PagerDuty/Jira does NOT produce a startup error (MSSP Coordination types permit client override).
(traces to BC-3.5.001 precondition 2 — valid customer registered; BC-3.3.001-startup EC-003)

**AC-008:** No test file in any of the three crates directly instantiates the DTU clone outside the harness after migration.
(traces to BC-3.5.001 precondition 4)
