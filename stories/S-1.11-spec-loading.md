# S-1.11 — prism-spec-engine: Spec Loading and Pipeline Execution

## Objective

Implement config-driven sensor adapter engine — TOML spec parsing, multi-step fetch pipeline, column-to-OCSF mapping.

## Behavioral Contracts

| BC ID | Title |
|-------|-------|
| BC-2.16.001 | Sensor Spec File Loading — Parse TOML, Validate Schema, Register Tables |
| BC-2.16.002 | Multi-Step Fetch Pipeline Execution — Sequential Steps with Variable Interpolation |
| BC-2.16.003 | Column-to-OCSF Mapping at Query Time — Map Sensor Columns to OCSF Fields Per Spec |
| BC-2.16.004 | Rust Escape Hatch for Custom Adapters — Trait-Based Override When Config Is Insufficient |
| BC-2.16.009 | Spec File Validation — Schema Validation, Variable Reference Resolution, OCSF Field Validation |

## Tasks

1. Implement TOML spec file parser: load `.sensor.toml` files, parse into `SensorSpec` struct (`sources`, `auth`, `fetch` pipeline, `column_mappings`) (BC-2.16.001)
2. Implement DataFusion table registration from spec: each `source` in a spec becomes a queryable table name (BC-2.16.001)
3. Implement multi-step fetch pipeline executor: steps execute sequentially; each step can reference variables from previous steps via `${step1.response.total}`; variable interpolation applies JSON-escape and percent-encode safety (BC-2.16.002)
4. Implement column-to-OCSF field mapping at query time: spec defines `column → OCSF field path` mapping, applied during normalization (BC-2.16.003)
5. Implement `CustomAdapter` trait: Rust escape hatch when TOML spec is insufficient; trait-based override with full access to HTTP client and response parsing (BC-2.16.004)
6. Implement spec file validation: schema validation (required fields present), variable reference resolution (no dangling refs), OCSF field path validation (valid paths only) (BC-2.16.009)
7. Write VP-023 fuzz target: `SpecParser` never panics on arbitrary TOML input

## Acceptance Criteria

- **AC-1:** Given a valid `crowdstrike.sensor.toml`, When `SpecParser` loads it, Then all sources are registered as DataFusion tables (BC-2.16.001)
- **AC-2:** Given a two-step CrowdStrike fetch (OAuth token → API call), When the pipeline executes, Then step 2 uses the token from step 1 via `${step1.response.access_token}` (BC-2.16.002)
- **AC-3:** Given a spec with column mapping `"created_timestamp" → "time"`, When query results are normalized, Then the OCSF `time` field is populated (BC-2.16.003)
- **AC-4:** Given a sensor that needs custom parsing, When a `CustomAdapter` is registered, Then it overrides the TOML spec pipeline (BC-2.16.004)
- **AC-5:** Given a spec with dangling variable ref `${nonexistent.field}`, When validation runs, Then it reports the error with line number (BC-2.16.009)
- **AC-6:** VP-023 fuzz target runs 30 minutes without panics

## Notes

- Variable interpolation must apply safety escaping before substitution: JSON-escape for values used in JSON bodies, percent-encode for values used in URL path/query segments. The spec author declares which context each variable is interpolated into.
- `CustomAdapter` trait implementors are registered by crate name at startup. The TOML spec can declare `adapter = "crowdstrike-custom"` to route to the registered Rust impl instead of the generic pipeline. This is the intended extension point for sensors that require binary parsing, websocket streaming, or complex authentication flows.
- DataFusion table names are derived from `[source.name]` in the spec. Name conflicts across specs must be detected at load time, not query time.
- OCSF field path validation requires the OCSF schema to be embedded or bundled with the crate. Do not fetch it at runtime.
- Fuzz target should be placed in `fuzz/fuzz_targets/spec_parser.rs` and registered in `fuzz/Cargo.toml`.
