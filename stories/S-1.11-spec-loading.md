# S-1.11 â€” prism-spec-engine: Spec Loading and Pipeline Execution

## Narrative

As a Prism platform implementer, I want a config-driven sensor adapter engine in
prism-spec-engine that parses TOML sensor specs, executes multi-step fetch pipelines,
and maps sensor columns to OCSF fields, so that new sensors can be onboarded via a
TOML file without writing Rust â€” with a Rust escape hatch available for sensors that
require custom parsing beyond what declarative config supports.

## Objective

Implement config-driven sensor adapter engine â€” TOML spec parsing, multi-step fetch pipeline, column-to-OCSF mapping.

## Tasks

1. Implement TOML spec file parser: load `.sensor.toml` files, parse into `SensorSpec` struct (`sources`, `auth`, `fetch` pipeline, `column_mappings`) (BC-2.16.001)
2. Export `SensorTableDescriptor` from spec: each `source` in a spec produces a descriptor with table name, column schema (using prism-core ColumnType), and source metadata. DataFusion `TableProvider` registration lives in prism-query (S-3.02), not here â€” prism-spec-engine does not depend on DataFusion. (BC-2.16.001)
3. Implement multi-step fetch pipeline executor: steps execute sequentially; each step can reference variables from previous steps via `${step1.response.total}`; variable interpolation applies JSON-escape and percent-encode safety (BC-2.16.002)
4. Implement column-to-OCSF field mapping at query time: spec defines `column â†’ OCSF field path` mapping, applied during normalization (BC-2.16.003)
5. Implement `CustomAdapter` trait: Rust escape hatch when TOML spec is insufficient; trait-based override with full access to HTTP client and response parsing (BC-2.16.004)
6. Implement spec file validation: schema validation (required fields present), variable reference resolution (no dangling refs), OCSF field path validation (valid paths only) (BC-2.16.009)
7. Write VP-023 fuzz target: `SpecParser` never panics on arbitrary TOML input

8. Write VP-059 proptest in `crates/prism-spec-engine/src/proofs/spec_validator.rs`:
   - Generate SensorSpecs with N distinct injected validation errors (N in 1..=20)
   - Assert `validate_sensor_spec(spec).unwrap_err().len() == N` (all errors collected)
   - Assert no early return: each error type appears independently in output
   - Harness file: `crates/prism-spec-engine/src/proofs/spec_validator.rs`
   - Method: proptest
   - Traces to: BC-2.16.009 postconditions

## Acceptance Criteria

- **AC-1:** Given a valid `crowdstrike.sensor.toml`, When `SpecParser` loads it, Then all sources produce SensorTableDescriptor entries for downstream DataFusion registration by prism-query (BC-2.16.001)
- **AC-2:** Given a two-step CrowdStrike fetch (OAuth token â†’ API call), When the pipeline executes, Then step 2 uses the token from step 1 via `${step1.response.access_token}` (BC-2.16.002)
- **AC-3:** Given a spec with column mapping `"created_timestamp" â†’ "time"`, When query results are normalized, Then the OCSF `time` field is populated (BC-2.16.003)
- **AC-4:** Given a sensor that needs custom parsing, When a `CustomAdapter` is registered, Then it overrides the TOML spec pipeline (BC-2.16.004)
- **AC-5:** Given a spec with dangling variable ref `${nonexistent.field}`, When validation runs, Then it reports the error with line number (BC-2.16.009)
- **AC-6:** VP-023 fuzz target runs 30 minutes without panics

- **AC-7:** VP-059 proptest passes: for any SensorSpec with N distinct validation errors, `validate_sensor_spec` returns Err with exactly N items (no fail-fast).

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| TOML spec parser and validator | `prism-spec-engine` (SS-16) | Mixed |

`prism-spec-engine` (Layer 2: Business Logic) implements `SpecParser` â€” the TOML-based
loader for sensor specs, infusion specs, and action specs. TOML parsing and schema
validation are pure; filesystem reads and arc-swap config publication are effectful.
Per `architecture/module-decomposition.md`, `prism-sensors`, `prism-operations`, and
`prism-query` all consume spec-engine output via `ConfigManager` (arc-swap).

## Dev Notes

- Variable interpolation must apply safety escaping before substitution: JSON-escape for values used in JSON bodies, percent-encode for values used in URL path/query segments. The spec author declares which context each variable is interpolated into.
- `CustomAdapter` trait implementors are registered by crate name at startup. The TOML spec can declare `adapter = "crowdstrike-custom"` to route to the registered Rust impl instead of the generic pipeline. This is the intended extension point for sensors that require binary parsing, websocket streaming, or complex authentication flows.
- DataFusion table names are derived from `[source.name]` in the spec. Name conflicts across specs must be detected at load time, not query time.
- OCSF field path validation requires the OCSF schema to be embedded or bundled with the crate. Do not fetch it at runtime.
- Fuzz target should be placed in `fuzz/fuzz_targets/spec_parser.rs` and registered in `fuzz/Cargo.toml`.
