# S-4.07 — prism-operations: Case Metrics and Acknowledge Alert

## Narrative

As a Prism operations platform, I want auto-computed MTTD/MTTR metrics and an alert
acknowledgment capability, so that MSSP analysts and clients can measure detection and
response performance and track which alerts have been reviewed.

## Objective

Implement MTTD (time from earliest alert event_time to case creation) and MTTR (time
from case creation to Resolved transition) auto-computation, the `case_metrics` MCP tool
for aggregated metric reporting, and the `acknowledge_alert` MCP tool per BC-2.14.012.

BC-2.14.012 (`acknowledge_alert` MCP Tool) is fully specified and committed. Implementers
may proceed with `acknowledge_alert` implementation directly against BC-2.14.012.

## Tasks

1. Read BC-2.14.012 at `.factory/specs/behavioral-contracts/BC-2.14.012-acknowledge-alert.md`
   before beginning `acknowledge_alert` implementation. BC-2.14.012 is fully specified
   and committed; it has passed adversarial review as part of the Burst 1 product-owner delivery.

2. Implement per-case TTD (Time to Detect) computation (BC-2.14.008):
   - Per-alert TTD = `alert.triggered_at - alert.event_time` (detection latency:
     time between earliest triggering event and alert generation)
   - Per-case TTD = `min(alert.ttd for alert in case.alert_ids)` (fastest detection)
   - Load alert records from `alerts` CF; if event_time is null, that alert's TTD is null
   - If all alerts have null TTD, per-case TTD is null
   - Floor at 0 (never negative)

3. Implement per-case TTI (Time to Investigate) computation (BC-2.14.008):
   - TTI = `case.created_at - min(alert.created_at for alert in case.alert_ids)`
     (triage latency: earliest alert creation → case creation)
   - This measures how long between the first alert firing and someone creating a case
   - If no alerts linked, TTI is null
   - If earliest alert.created_at > case.created_at (alert linked retroactively), TTI = 0

4. Implement per-case TTR (Time to Resolve) computation (BC-2.14.008):
   - TTR = `case.resolved_at - case.created_at` (investigation duration: case
     creation → first resolution)
   - If `case.resolved_at` is null, TTR is null
   - For cases with reopen cycles, `resolved_at` preserves first resolution time per
     BC-2.14.002 — TTR reflects time to first resolution
   - All per-case metrics (TTD, TTI, TTR) recomputed on each call — not stored

5. Implement `case_metrics` MCP tool handler in
   `crates/prism-operations/src/cases/case_metrics.rs` (BC-2.14.010):
   - Parameters: optional `client_id`, optional `date_range: (Timestamp, Timestamp)`,
     optional `include_open: bool` (default false — open cases have no MTTR)
   - Aggregate across matching cases:
     - `avg_mttd_minutes`: average MTTD (Mean Time to Detect) in minutes
     - `avg_mtti_minutes`: average MTTI (Mean Time to Investigate) in minutes
     - `avg_mttr_hours`: average MTTR (Mean Time to Resolve) in hours
     - `p50_mttd_minutes`, `p95_mttd_minutes`: percentile TTD
     - `p50_mtti_minutes`, `p95_mtti_minutes`: percentile TTI
     - `p50_mttr_hours`, `p95_mttr_hours`: percentile TTR
     - `case_counts`: `{ new, acknowledged, investigating, resolved, closed }`
     - `severity_distribution`: `{ critical, high, medium, low, informational }`
     - `by_client: Vec<ClientMetricSummary>` (if no client_id filter provided)
   - Cap: if date range spans more than 90 days (`range_days > 90`), return E-METRICS-001
     (range too large). Exactly 90 days succeeds. Add boundary test for the exclusive edge.
   - Percentile computation: use `hdrhistogram = "7.5"` (`Histogram::<u64>::new_with_bounds`)
     for latency-like metrics (TTD, TTI, TTR). For any metric with unbounded or skew-heavy
     distribution, use `tdigest = "0.2"` as escape hatch. Document choice per metric in impl.
   - For large client counts (>10,000 cases), compute aggregations via streaming hdrhistogram
     — iterate cases CF exactly once, do NOT load all values into memory for sorting.

6. Implement `acknowledge_alert` MCP tool handler in
   `crates/prism-operations/src/cases/acknowledge_alert.rs` (BC-2.14.012):
   - Parameters: `alert_id: AlertId`
   - **Org-context check (per ADR-006, DRIFT-407-003):** before any alert access, verify
     that the alert's `org_id` matches the calling session's `org_id`. A mismatch MUST
     return `E-ALERT-ORG-MISMATCH` and MUST NOT leak the alert's existence.
   - Check capability: verify caller has `CAPABILITY_ACKNOWLEDGE_ALERT` (feature flag
     from S-1.08); return E-FLAG-001 if not enabled
   - Load alert from `alerts` CF; return E-ALERT-001 if not found
   - If `alert.acknowledged_at` is already set: return E-ALERT-002 (already acknowledged)
   - Set `alert.acknowledged_at = now()`, `alert.acknowledged_by = caller_analyst_id`
   - Persist updated alert to `alerts` CF using read-modify-write under a per-`AlertId`
     `Mutex<()>` to prevent races. Consider `dashmap::DashMap<AlertId, Mutex<()>>` over
     `HashMap<AlertId, Mutex<()>>` for concurrent access — document choice in implementation.
   - If the alert is linked to any case: append `Note` timeline entry to each
     linked case with content `"Alert {alert_id} acknowledged by {analyst_id}"`
     (Note, not StatusChange — acknowledging an alert is not a case state transition)
   - Return updated alert record

7. Verify `CAPABILITY_ACKNOWLEDGE_ALERT` flag registration (DRIFT-407-003):
   - Confirm `CAPABILITY_ACKNOWLEDGE_ALERT` is registered in the S-1.08 feature flag
     registry (`crates/prism-security/src/capabilities.rs` or equivalent).
   - If the flag is not yet defined, add it as part of this story (cross-crate fix-PR
     targeting `prism-security`). Flag must be present before this story can pass its
     integration tests.

## Acceptance Criteria

**AC-1:** Given an alert with `event_time = T` and `triggered_at = T+2min`, When per-alert
TTD is computed, Then `ttd = 2 minutes` (triggered_at - event_time). Given a case with
that alert, When per-case MTTD is computed, Then `mttd = 2 minutes`. (BC-2.14.008)

**AC-2:** Given a case where the earliest alert's record has been evicted, When MTTD is
computed, Then `triggered_at` is used as fallback and `mttd_approx = true` is set.
(BC-2.14.008)

**AC-3:** Given a case created at T and resolved at T+2h, When MTTR is computed, Then
`mttr = 2 hours`. (BC-2.14.008)

**AC-4:** Given a case that is not yet resolved, When MTTR is computed, Then `mttr` is
`None`. (BC-2.14.008)

**AC-5:** Given 10 cases across 2 clients with varying statuses and resolutions, When
`case_metrics` is called without filters, Then the response includes `avg_mttd_minutes`,
`avg_mttr_hours`, percentile values, `case_counts`, `severity_distribution`, and
`by_client` breakdown. (BC-2.14.010)

**AC-6:** Given `case_metrics` with a date range exceeding 90 days, When called, Then
E-METRICS-001 is returned. (BC-2.14.010)

**AC-7:** Given an unacknowledged alert belonging to the caller's org, When
`acknowledge_alert` is called by a caller with `CAPABILITY_ACKNOWLEDGE_ALERT`, Then
`acknowledged_at` is set, `acknowledged_by` is set, and a `Note` timeline entry is
appended to any linked case. Given the same alert is accessed by a caller from a
different org, Then E-ALERT-ORG-MISMATCH is returned and no alert field is modified.
(BC-2.14.012, ADR-006 §2.1)

**AC-8:** Given an already-acknowledged alert, When `acknowledge_alert` is called again,
Then E-ALERT-002 is returned. (BC-2.14.012)

**AC-9:** Given a caller without `CAPABILITY_ACKNOWLEDGE_ALERT`, When `acknowledge_alert`
is called, Then E-FLAG-001 is returned. (BC-2.14.012)

## Architecture Compliance Rules

- MTTD and MTTR are computed on demand, not stored. This avoids cache invalidation
  complexity when alerts are added to a case after creation or when timeline entries are
  amended.
- `case_metrics` aggregation MUST iterate the `cases` CF exactly once per call.
  Do not make per-case RocksDB lookups inside the aggregation loop — load all required
  data in a single prefix scan.
- `acknowledge_alert` MUST be gated behind the `CAPABILITY_ACKNOWLEDGE_ALERT` feature
  flag from S-1.08. This tool modifies alert records, which are audit-sensitive. The
  feature flag ensures it is explicitly enabled in the operator configuration.
- Alert updates (acknowledgment) MUST be written to RocksDB as a full alert struct
  replacement, not a partial update. RocksDB does not support field-level updates;
  read-modify-write under an in-memory `Mutex` per `AlertId` prevents races.
- BC-2.14.012 is fully specified and committed (Phase 3 Burst 1). The implementing agent
  reads and implements against the committed BC file.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | All alerts linked to a case have null `event_time` | Per-case TTD is null; `mttd_approx` not set (no fallback available) |
| EC-002 | Alert record evicted from `alerts` CF before MTTD computed | `triggered_at` used as TTD fallback; `mttd_approx = true` set on result |
| EC-003 | Case reopened after initial resolution (Resolved → Investigating) | `resolved_at` preserved (first resolution); TTR reflects time to first resolution; `closed_at` cleared |
| EC-004 | `case_metrics` with date range spanning exactly 90 days | Succeeds — boundary is exclusive: `range_days > 90` triggers E-METRICS-001; `range_days == 90` is within limit. Add a boundary test asserting exactly-90-days succeeds and exactly-91-days fails. |
| EC-005 | `case_metrics` with date range spanning 91 days | Returns E-METRICS-001 (range too large — `range_days > 90`) |
| EC-006 | `case_metrics` over 10,000+ cases | Streaming `hdrhistogram` used for percentile computation; not loaded fully into memory. `tdigest = "0.2"` as escape hatch for unbounded distributions. |
| EC-007 | `acknowledge_alert` called by caller without `CAPABILITY_ACKNOWLEDGE_ALERT` | Returns E-FLAG-001; alert not modified |
| EC-008 | `acknowledge_alert` called on already-acknowledged alert | Returns E-ALERT-002 (idempotent — does not re-set `acknowledged_at`) |
| EC-009 | Alert linked to a case that has since been deleted from RocksDB | Timeline append skipped for missing case; primary alert acknowledgment still succeeds |

## Dev Notes

- BC-2.14.012 is fully specified and committed (Phase 3 Burst 1). The implementing agent
  implements against the committed BC. No further prerequisite gate applies.
- MTTD percentile computation (p50, p95) over large case sets may be memory-intensive.
  For sets larger than 10,000 cases, use a streaming approximate percentile algorithm
  (t-digest or GK summary) rather than loading all values into memory for sorting.
- The `by_client` breakdown in `case_metrics` is only included when no `client_id`
  filter is applied. When a specific client is requested, the response is flat (no
  per-client nested structure).
- `acknowledge_alert` is a lightweight operation by design. It sets two fields and
  writes one timeline entry. If future requirements call for acknowledgment workflows
  (multi-step acknowledgment, acknowledgment with notes), that is a separate story.
- Percentile library choice (R-10): `hdrhistogram = "7.5"` is the default for all
  latency-like metrics (TTD in minutes, TTI in minutes, TTR in hours). Its bounded-range,
  constant-relative-precision model is well-suited to case resolution timelines. Reserve
  `tdigest = "0.2"` for any future metric where the value distribution is unbounded or
  skew-heavy (e.g., alert count per case, alert volume per tenant).
