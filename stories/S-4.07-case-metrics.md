# S-4.07 — prism-operations: Case Metrics and Acknowledge Alert

## Narrative

As a Prism operations platform, I want auto-computed MTTD/MTTR metrics and an alert
acknowledgment capability, so that MSSP analysts and clients can measure detection and
response performance and track which alerts have been reviewed.

## Objective

Implement MTTD (time from earliest alert event_time to case creation) and MTTR (time
from case creation to Resolved transition) auto-computation, the `case_metrics` MCP tool
for aggregated metric reporting, and the `acknowledge_alert` MCP tool. BC-2.14.012 for
`acknowledge_alert` is currently a STUB. PREREQUISITE: BC-2.14.012 must be fully
specified as a separate Phase 3 task and pass adversarial review BEFORE this story's
implementation begins. The implementing agent must NOT author the BC — it must already
exist and be reviewed.

## Tasks

1. **GATE: BC-2.14.012 must be fully specified and reviewed before this story begins.**
   The BC is currently a STUB. It must be authored as a separate Phase 3 task by the
   product-owner agent, pass adversarial review, and be committed to the BC file before
   any implementation work in this story starts. The implementing agent must NOT author
   the BC — they implement against the reviewed spec.

2. Implement MTTD computation in
   `crates/prism-operations/src/cases/metrics.rs` (BC-2.14.008):
   - MTTD = `case.created_at - min(alert.event_time for alert in case.alert_ids)`
   - `min(alert.event_time)` requires loading alert records from the `alerts` CF to
     read their `event_time` field
   - If any linked alert cannot be found (was evicted), use `alert.triggered_at` as
     fallback with a `mttd_approx: true` flag on the metric
   - MTTD is recomputed on each `get_case` and `case_metrics` call — it is not stored
     as a field on the `Case` struct

3. Implement MTTR computation (BC-2.14.008):
   - MTTR = timestamp of the `Resolved` `StatusChange` timeline entry minus
     `case.created_at`
   - If the case has not been resolved, MTTR is `None`
   - Scan `case.timeline` for the most recent `StatusChange` entry with content matching
     `"* → Resolved *"` to find the resolution timestamp
   - MTTR is recomputed on each call — not stored

4. Implement `case_metrics` MCP tool handler in
   `crates/prism-operations/src/cases/case_metrics.rs` (BC-2.14.010):
   - Parameters: optional `client_id`, optional `date_range: (Timestamp, Timestamp)`,
     optional `include_open: bool` (default false — open cases have no MTTR)
   - Aggregate across matching cases:
     - `avg_mttd_minutes`: average MTTD in minutes (only cases where MTTD is computable)
     - `avg_mttr_hours`: average MTTR in hours (only resolved/closed cases)
     - `p50_mttd_minutes`, `p95_mttd_minutes`: percentile MTTD
     - `p50_mttr_hours`, `p95_mttr_hours`: percentile MTTR
     - `case_counts`: `{ open, in_progress, pending_vendor, resolved, closed }`
     - `severity_distribution`: `{ critical, high, medium, low, informational }`
     - `by_client: Vec<ClientMetricSummary>` (if no client_id filter provided)
   - Cap: if date range spans more than 90 days, return E-METRICS-001 (range too large)
   - For large client counts, compute aggregations lazily (iterate cases CF once)

5. Implement `acknowledge_alert` MCP tool handler in
   `crates/prism-operations/src/cases/acknowledge_alert.rs` (BC-2.14.012):
   - Parameters: `alert_id: AlertId`
   - Check capability: verify caller has `CAPABILITY_ACKNOWLEDGE_ALERT` (feature flag
     from S-1.08); return E-FEAT-002 if not enabled
   - Load alert from `alerts` CF; return E-ALERT-001 if not found
   - If `alert.acknowledged_at` is already set: return E-ALERT-002 (already acknowledged)
   - Set `alert.acknowledged_at = now()`, `alert.acknowledged_by = caller_analyst_id`
   - Persist updated alert to `alerts` CF
   - If the alert is linked to any case: append `StatusChange` timeline entry to each
     linked case with content `"Alert {alert_id} acknowledged by {analyst_id}"`
   - Return updated alert record

## Acceptance Criteria

**AC-1:** Given a case created at T+30min with an alert whose `event_time` is T, When
MTTD is computed, Then `mttd = 30 minutes`. (BC-2.14.008)

**AC-2:** Given a case where the earliest alert's record has been evicted, When MTTD is
computed, Then `triggered_at` is used as fallback and `mttd_approx = true` is set.
(BC-2.14.008)

**AC-3:** Given a case created at T and resolved at T+2h, When MTTR is computed, Then
`mttr = 2 hours`. (BC-2.14.008)

**AC-4:** Given a case that is not yet resolved, When MTTR is computed, Then `mttr` is
`None`. (BC-2.14.008)

**AC-5:** Given 10 cases across 2 clients with varying statuses and resolutions, When
`case_metrics` is called without filters, Then the response includes `avg_mttd_minutes`,
`avg_mttr_hours`, percentile values, `case_counts`, `severity_distribution`, and
`by_client` breakdown. (BC-2.14.010)

**AC-6:** Given `case_metrics` with a date range exceeding 90 days, When called, Then
E-METRICS-001 is returned. (BC-2.14.010)

**AC-7:** Given an unacknowledged alert, When `acknowledge_alert` is called by a caller
with `CAPABILITY_ACKNOWLEDGE_ALERT`, Then `acknowledged_at` is set, `acknowledged_by`
is set, and a timeline entry is appended to any linked case. (BC-2.14.012)

**AC-8:** Given an already-acknowledged alert, When `acknowledge_alert` is called again,
Then E-ALERT-002 is returned. (BC-2.14.012)

**AC-9:** Given a caller without `CAPABILITY_ACKNOWLEDGE_ALERT`, When `acknowledge_alert`
is called, Then E-FEAT-002 is returned. (BC-2.14.012)

## Architecture Compliance Rules

- MTTD and MTTR are computed on demand, not stored. This avoids cache invalidation
  complexity when alerts are added to a case after creation or when timeline entries are
  amended.
- `case_metrics` aggregation MUST iterate the `cases` CF exactly once per call.
  Do not make per-case RocksDB lookups inside the aggregation loop — load all required
  data in a single prefix scan.
- `acknowledge_alert` MUST be gated behind the `CAPABILITY_ACKNOWLEDGE_ALERT` feature
  flag from S-1.08. This tool modifies alert records, which are audit-sensitive. The
  feature flag ensures it is explicitly enabled in the operator configuration.
- Alert updates (acknowledgment) MUST be written to RocksDB as a full alert struct
  replacement, not a partial update. RocksDB does not support field-level updates;
  read-modify-write under an in-memory `Mutex` per `AlertId` prevents races.
- The BC-2.14.012 spec completion task (Task 1) is a mandatory prerequisite. The
  implementer MUST write and commit the completed BC before writing any implementation
  code for `acknowledge_alert`. Do not implement against a STUB BC.

## Previous Story Intelligence

S-4.06 established the `Case`, `CaseStatus`, `Timeline`, `Alert`, and `DispositionCode`
types, the `cases` and `alerts` RocksDB CFs, and the `get_case` / `update_case` / `list_cases`

The timeline append pattern from S-4.06 (`StatusChange` auto-generation on transitions)

the same timeline append helper from `crates/prism-operations/src/cases/timeline.rs`.

The feature flag pattern for capability gating follows S-1.08. The specific flag
`CAPABILITY_ACKNOWLEDGE_ALERT` must be defined in the feature flag registry (S-1.08)
before this story is implemented. If it is not yet defined, add it as part of Task 1
alongside the BC completion.

## Notes

- BC-2.14.012 is a STUB. This is the only story in Phase 3 with an incomplete BC as a
  prerequisite task. The implementer must author the full BC (preconditions,
  postconditions, error cases, capability gating) before writing code. The completed BC
  must be reviewed for consistency with the existing BC-2.14.001–BC-2.14.010 set.
- MTTD percentile computation (p50, p95) over large case sets may be memory-intensive.
  For sets larger than 10,000 cases, use a streaming approximate percentile algorithm
  (t-digest or GK summary) rather than loading all values into memory for sorting.
- The `by_client` breakdown in `case_metrics` is only included when no `client_id`
  filter is applied. When a specific client is requested, the response is flat (no
  per-client nested structure).
- `acknowledge_alert` is a lightweight operation by design. It sets two fields and
  writes one timeline entry. If future requirements call for acknowledgment workflows
  (multi-step acknowledgment, acknowledgment with notes), that is a separate story.
