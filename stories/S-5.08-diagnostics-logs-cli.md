# S-5.08 — prism-mcp: Diagnostics — prism logs CLI + get_diagnostics + Trace IDs

## Split Notice

This story covers the full diagnostics surface. Token budget is estimated at ~19,700 tokens
(see table below), within the 30% boundary for a 90k-context agent. If the implementing agent determines
at implementation time that the context would be exceeded, it MUST split into:
- **S-5.08a**: Rolling log file writer + `prism logs` CLI + trace ID generation/propagation
- **S-5.08b**: `get_diagnostics` MCP tool + diagnostic resource templates

The split point is the `get_diagnostics` handler and its resource templates (S-5.08b depends
on S-5.08a for the trace ID infrastructure). Document the split at the top of both files if
the split occurs.

---

## Narrative

As a Prism operator and MSSP analyst, I want a persistent rolling log system with trace ID
propagation and the `get_diagnostics` MCP tool, so that I can correlate specific requests
end-to-end across subsystems, query live and historical logs by subsystem or trace ID from
the CLI, and ask the AI assistant for structured diagnostic summaries that are injection-
scanned before delivery.

## Objective

Implement: rolling log file writer (50MB rotation, 10-file retention) in `{state_dir}/logs/`;
UUID v7 trace ID generation at request entry, propagation through `tracing::Span`s, and
injection into MCP response `_meta.trace_id`; `prism logs` CLI with `--subsystem`, `--since`,
`--trace-id`, `--follow`, `--grep`, and `--format json` flags; `get_diagnostics` MCP tool
with 10 subsystem query modes and injection scanning on results; and `prism://diagnostics/`
MCP resource templates.

## Tasks

1. Implement rolling log file writer in `crates/prism-mcp/src/logging/writer.rs`:
   - Create a `tracing_subscriber::Layer` that writes structured JSON log lines to
     `{state_dir}/logs/prism.log`
   - Rotation trigger: when the current file reaches 50MB, rename to `prism.log.1`,
     shifting existing `.1`–`.9` files up; delete `.10` if it exists
   - Retention: keep at most 10 rotated files (`prism.log.1` through `prism.log.10`)
   - Log line format (JSON, one object per line):
     ```json
     {"timestamp": "2026-04-16T12:34:56.789Z", "level": "INFO",
      "subsystem": "query", "trace_id": "019...", "message": "...",
      "fields": {...}}
     ```
   - `subsystem` field: derived from the `tracing::Span`'s `subsystem` field (set at
     request entry). Possible values: `query`, `audit`, `credentials`, `detection`,
     `actions`, `plugins`, `storage`, `mcp`, `config`, `infusions`
   - File writes are synchronous (inside `spawn_blocking`) — log output must not drop
     entries under load

2. Implement trace ID generation in `crates/prism-mcp/src/logging/trace_id.rs`:
   - `fn new_trace_id() -> TraceId`: generate UUID v7 (time-sortable)
   - `TraceId` is a newtype wrapping `uuid::Uuid`
   - Inject `trace_id` into the root `tracing::Span` at MCP request entry:
     ```rust
     let span = tracing::info_span!("mcp_request", trace_id = %trace_id);
     ```
   - All child spans within the request inherit the trace ID via span context
   - Every log line emitted within a span includes `trace_id` in its JSON fields
     (the `tracing_subscriber` layer extracts it from the span context)

3. Implement `_meta.trace_id` injection in MCP responses:
   - In S-5.02's tool response construction, add `"trace_id": "{trace_id}"` to the
     `_meta` block of every MCP tool response
   - The `trace_id` comes from the active `tracing::Span`'s `trace_id` field
   - `_meta.trace_id` is present on ALL tool responses (success and error) — it is the
     operator's primary handle for log correlation
   - This is a modification to S-5.02's response builder (`tool_response.rs` or equivalent)

4. Implement `prism logs` CLI subcommand in `crates/prism-mcp/src/cli/logs.rs`:
   - `prism logs [flags]` — reads from `{state_dir}/logs/prism.log` and rotated files
   - Flags:
     - `--subsystem <name>`: filter to one of the 10 subsystems (exact match on `subsystem` JSON field)
     - `--since <duration>`: show only entries newer than this (e.g., `--since 1h`, `--since 30m`)
     - `--trace-id <uuid>`: show only entries with the specified `trace_id` (exact match)
     - `--follow`: tail mode — wait for new entries after reaching end of file (like `tail -f`)
     - `--grep <pattern>`: filter by regex on the `message` field (uses `regex` crate)
     - `--format json`: output raw JSON lines (default: human-readable text with colors)
   - Multi-file support: when `--since` spans multiple log files, read from rotated files
     in chronological order (oldest first)
   - `--follow` opens the current `prism.log` file and polls for new bytes on a 100ms
     interval using `tokio::fs::File`; detects rotation (file replaced) by inode check

5. Implement `get_diagnostics` MCP tool in `crates/prism-mcp/src/tools/get_diagnostics.rs`:
   - Tool registration: registered as `"get_diagnostics"` alongside other tools at startup
     (S-5.01's tool list)
   - Parameters:
     - `subsystem: Option<String>` — one of 10 values: `query`, `audit`, `credentials`,
       `detection`, `actions`, `plugins`, `storage`, `mcp`, `config`, `infusions`. If absent,
       return summary across all 10.
     - `trace_id: Option<String>` — filter to a specific trace
     - `since: Option<String>` — time window (e.g., `"1h"`, `"30m"`)
     - `limit: Option<u32>` — max entries (default 100, max 500)
   - Response structure:
     ```json
     {
       "subsystem": "query" | "all",
       "entries": [ { "timestamp": "...", "level": "...", "message": "...", "trace_id": "..." } ],
       "summary": { "error_count": N, "warn_count": N, "subsystem_counts": {...} },
       "_meta": { "trace_id": "...", "safety_flags": [...], "trust_level": "untrusted_external" }
     }
     ```
   - Injection scanning: ALL `message` and `fields` values from log entries are scanned
     by `InjectionScanner` (S-1.10) before inclusion in the response. Detected patterns
     are accumulated in `_meta.safety_flags`. Values are INCLUDED in the response (not
     stripped) per BC-2.09.004 semantics — flag, don't block.
   - `trust_level: "untrusted_external"` is set on ALL `get_diagnostics` responses because
     log content originates from external sensor APIs and user inputs.

6. Implement MCP diagnostic resource templates in `crates/prism-mcp/src/resources/diagnostics_resources.rs`:
   - `prism://diagnostics/summary`: returns `get_diagnostics` result with no subsystem filter
     (all 10 subsystems, last 1 hour, top 100 entries per subsystem)
   - `prism://diagnostics/{subsystem}`: parameterized template — `subsystem` must be one
     of the 10 valid values; returns that subsystem's recent entries
   - `prism://diagnostics/trace/{trace_id}`: returns all log entries for the given trace ID
     across all subsystems and all log files
   - Resource descriptions registered via S-5.03's resource registration mechanism
   - All resource responses include `_meta.safety_flags` and `trust_level: "untrusted_external"`

7. Write tests in `crates/prism-mcp/tests/diagnostics_tests.rs`:
   - Test log writer: write 60MB of entries → verify rotation occurs, 10 files retained
   - Test `prism logs --trace-id abc123` → returns exactly entries with that trace_id
   - Test `prism logs --subsystem query --since 1h` → filters correctly
   - Test `get_diagnostics` with no filter → returns summary across all 10 subsystems
   - Test `get_diagnostics` injection scan: inject a prompt injection pattern into a log
     message → verify `_meta.safety_flags` is populated, entry is still returned
   - Test `_meta.trace_id` present on tool response
   - Test `prism://diagnostics/trace/{trace_id}` resource → returns correct cross-subsystem entries

## Acceptance Criteria

**AC-1:** Given Prism has processed 1,000 MCP requests, When `prism logs --trace-id <id>`
is called with a specific trace ID, Then exactly the log entries emitted during that
request are returned — no entries from other requests.
(traces to BC-2.10.003 postcondition 1 — `_meta` trace propagation)

**AC-2:** Given the log file has grown to 50MB, When the next log entry is written,
Then `prism.log` is rotated to `prism.log.1`, a new `prism.log` is opened, and the
count of rotated files does not exceed 10.

**AC-3:** Given `prism logs --follow --subsystem query`, When new query log entries
are appended by a running server, Then they are printed in real time without restarting
the command.

**AC-4:** Given `get_diagnostics` is called with no `subsystem` parameter, When executed,
Then the response includes summary counts (`error_count`, `warn_count`) and entries from
all 10 subsystems, along with `_meta.trace_id` and `trust_level: "untrusted_external"`.
(traces to BC-2.10.002 postcondition 1)

**AC-5:** Given a log entry with `message` containing a prompt injection pattern
(e.g., `"Ignore all previous instructions"`), When `get_diagnostics` returns it,
Then `_meta.safety_flags` lists the detected pattern and `trust_level: "untrusted_external"`
is set. The entry is still present in the `entries` array.
(traces to BC-2.10.006 postcondition 1)

**AC-6:** Given an MCP tool call that results in an error, When the error response is
returned, Then `_meta.trace_id` is present in the error response (trace IDs are on ALL
responses, not just successes).
(traces to BC-2.10.003 postcondition 1)

**AC-7:** Given `prism://diagnostics/trace/{trace_id}` is requested as an MCP resource,
When the trace_id matches logged entries across multiple subsystems, Then the resource
response includes entries from all matching subsystems in chronological order.
(traces to BC-2.10.008/BC-2.10.009 — resource template handling)

## Architecture Compliance Rules

- Log file writes MUST use `tokio::task::spawn_blocking` — file I/O must not block
  the async runtime's executor threads.
- The rolling writer MUST be added to the global `tracing` subscriber at startup
  (S-5.01's `TracingInit`). It MUST NOT be initialized lazily — log entries before
  lazy init are lost.
- `_meta.trace_id` MUST be populated from the active `tracing::Span` context, not
  from a thread-local or global static. Trace IDs must correctly propagate across
  `tokio::task::spawn` boundaries using `tracing`'s span propagation.
- `get_diagnostics` MUST run `InjectionScanner` on log content before including it
  in the response. Log entries may contain attacker-controlled strings from external
  APIs. This is a security requirement, not optional.
- `trust_level: "untrusted_external"` MUST be set on ALL `get_diagnostics` and
  diagnostic resource responses — these are the ONLY MCP surfaces that return content
  from external log sources.
- Log rotation MUST be atomic: rename existing file, open new file. A crash mid-rotation
  (between rename and open) must not cause log loss beyond the in-flight write.
- `prism logs --follow` MUST detect log rotation (file replaced by rotation). Strategy:
  compare inode of the opened file with the inode of `{state_dir}/logs/prism.log` on
  each poll interval. If different, the file was rotated — close old fd, open new file.

## Notes

- UUID v7 is time-sortable: the first 48 bits encode the Unix timestamp in milliseconds.
  This means `prism logs --since 1h` can binary-search for the start offset in log files
  using trace_id timestamps — far more efficient than scanning all entries. This
  optimization is NOT required for the initial implementation (linear scan is acceptable)
  but the timestamp-sortable property enables it as a future optimization.
- The 10 subsystem names (`query`, `audit`, `credentials`, `detection`, `actions`,
  `plugins`, `storage`, `mcp`, `config`, `infusions`) must be treated as an enum with
  exhaustive matching. Adding a new subsystem later requires adding it to this enum and
  all switch statements.
- For the `--follow` tail implementation: `notify` crate is already a dependency (S-1.12).
  An alternative to polling is to use `notify::Watcher` to watch the log file for changes.
  Either approach is acceptable; the polling approach (100ms interval) is simpler for
  the initial implementation.
- `prism://diagnostics/trace/{trace_id}` must read across all rotated log files (`prism.log`
  through `prism.log.10`). This means a single trace may span multiple files if the trace
  occurred during rotation. The resource implementation must search all files.
