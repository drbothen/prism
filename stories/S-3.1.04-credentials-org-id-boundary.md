# S-3.1.04 — prism-credentials: migrate credential namespace key to OrgId

## Narrative
- **As a** credential lookup layer
- **I want** credential namespace keys to be keyed by `OrgId` (UUID string) rather than
  `OrgSlug` (display string)
- **So that** credential access remains stable across org renames and cross-tenant
  credential isolation is structurally enforced at the type level

## Description

ADR-006 §4 Step 3 migrates the credential namespace key in `prism-credentials/src/namespace.rs`
from `&TenantId` (now `&OrgSlug`) to `&OrgId`. The new namespace key format is
`"{org_id_uuid}/{sensor}/{name}"` where `org_id_uuid` is the hyphenated lowercase UUID v7
string. This makes credential access rename-stable (the UUID never changes when the slug
changes) and satisfies BC-3.2.002's AI-opaque credential principle (UUID is opaque to
human-readable analysis). All credential store trait methods (`get`, `set`, `delete`,
`rotate`) gain an `&OrgId` parameter in place of `&OrgSlug`/`&TenantId`.

## Acceptance Criteria

**AC-1 (traces to BC-3.2.002 precondition 1):** `namespace_key(org_id, sensor, name)`
in `prism-credentials/src/namespace.rs` accepts `&OrgId` (not `&OrgSlug` or `&str`)
as the first argument and produces the string `"{org_id.to_string()}/{sensor}/{name}"`.
No slug-keyed fallback path exists after this migration.

**AC-2 (traces to BC-3.2.002 postcondition 1):** `CredentialStore::get(org_id_A, sensor, name)`
returns `Ok(credential)` if and only if a credential was stored under
`"{org_id_A_uuid}/{sensor}/{name}"`. A proptest with adversarial (org_id, sensor_name)
pairs verifies lookup isolation.

**AC-3 (traces to BC-3.2.002 postcondition 2):** `CredentialStore::get(org_id_A, sensor, name)`
returns `Err(CredentialError::NotFound)` when no credential is stored under that namespace,
even if `org_id_B` has a credential for the same sensor and name.

**AC-4 (traces to BC-3.2.002 postcondition 3):** An org credential stored before a
hypothetical rename (same `OrgId`, different slug) remains accessible under the same
`OrgId` after the rename — validated by a test that stores under `org_id_A`, changes the
slug mapping in `OrgRegistry`, and verifies `get(org_id_A, ...)` still returns the
credential.

**AC-5 (traces to BC-3.2.002 invariant 1):** `grep -rn "OrgSlug\|TenantId" crates/prism-credentials/src/namespace.rs` returns zero hits after migration — the namespace key no longer references slug types.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `get(org_id_A, "claroty", "api_key")` — `org_id_A` has credentials | `Ok(credential_for_A)` |
| EC-002 | `get(org_id_B, "claroty", "api_key")` — `org_id_B` has no claroty credentials | `Err(NotFound)` |
| EC-003 | Org renames slug-A → slug-A2; `get(org_id_A, ...)` after rename | `Ok(credential)` — UUID-stable, rename has no effect |
| EC-004 | Credential cache (if present) keyed by `(OrgId, sensor_name, credential_name)` | Same isolation guarantee; cache key is `(OrgId, ...)` not `(OrgSlug, ...)` |
| EC-005 | Credential value appears in an error message | `CredentialError` variants MUST NOT contain the credential value — AI-opaque credentials principle (BC-3.2.002 postcondition 4) |
