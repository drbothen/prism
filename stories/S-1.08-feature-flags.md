# S-1.08 — prism-security: Feature Flags (P0 Core)

## Objective

Implement the two-tier feature flag system — compile-time Cargo features plus runtime per-client TOML capabilities.

## Behavioral Contracts

| BC ID | Title |
|-------|-------|
| BC-2.04.001 | Compile-Time Cargo Features Gate Write Code Families |
| BC-2.04.002 | Runtime Per-Client TOML Feature Flag Configuration |
| BC-2.04.003 | Hierarchical Capability Resolution (BTreeMap, Most-Specific-Path Wins, Deny Support) |
| BC-2.04.004 | Two-Tier Gate — Both Compile-Time and Runtime Must Permit Operation |
| BC-2.04.005 | Hidden Tools Pattern — Disabled Write Tools Omitted from tools/list |
| BC-2.04.006 | list_capabilities Meta-Tool for Capability Discovery |
| BC-2.04.013 | Feature Flag Evaluation Audit Logging for Write Operations |
| BC-2.04.015 | Structured Error When Write Capability Is Denied |

## Tasks

1. Define Cargo feature flags: `crowdstrike-write`, `cyberint-write`, `claroty-write`, `armis-write` — write-operation code behind `#[cfg(feature = "...")]` (BC-2.04.001)
2. Implement `FeatureFlagEvaluator` that loads per-client capabilities from TOML config (BC-2.04.002)
3. Integrate with `prism-core`'s `ClientCapabilities` for hierarchical resolution via `BTreeMap`; most-specific path wins; explicit `Deny` entries block inheritance (BC-2.04.003)
4. Implement two-tier gate: `check_permission(client_id, capability_path)` → first check compile-time feature, then runtime capability; both must be `Allow` (BC-2.04.004)
5. Implement hidden tools registry: tools gated by denied capabilities are not listed in `tools/list` response (BC-2.04.005)
6. Implement `list_capabilities` logic: returns effective capabilities for a given client (BC-2.04.006)
7. Implement audit event emission on every feature flag evaluation for write operations (BC-2.04.013)
8. Implement structured error `E-FLAG-001` when write capability is denied: includes `path`, `client_id`, and `resolution_trace` (BC-2.04.015)
9. Write VP-020 Kani proof: if compile-time feature is disabled, `check_permission` always returns `Deny` regardless of runtime config

## Acceptance Criteria

- **AC-1:** Given a binary compiled without `crowdstrike-write` feature, When `check_permission("acme", "crowdstrike.hosts.write")` is called, Then it returns `Deny` (BC-2.04.001, BC-2.04.004)
- **AC-2:** Given runtime config `{"crowdstrike.hosts.read" → Allow}`, When `check_permission("acme", "crowdstrike.hosts.read")` is called, Then it returns `Allow` (BC-2.04.002)
- **AC-3:** Given a denied write capability, When tool listing is requested, Then the write tool is absent from the list (BC-2.04.005)
- **AC-4:** Given client `"acme"` with specific capabilities, When `list_capabilities("acme")` is called, Then it returns the full effective capability set (BC-2.04.006)
- **AC-5:** Given a write operation check, When the evaluation occurs, Then an audit event is emitted (BC-2.04.013)
- **AC-6:** Given a denied write attempt, When the error is returned, Then it matches E-FLAG-001 format with `resolution_trace` (BC-2.04.015)
- **AC-7:** VP-020 Kani proof passes

## Verification Properties

| VP ID | Description |
|-------|-------------|
| VP-020 | Kani proof: compile-time feature disabled → check_permission always returns Deny |

## Notes

- The two-tier gate is intentionally asymmetric: compile-time `Deny` cannot be overridden by runtime config; runtime `Deny` cannot be overridden by client request. Both must independently grant `Allow`.
- `BTreeMap` is used for capability storage to guarantee deterministic iteration order, which is required for the resolution trace in E-FLAG-001.
- Hidden tools pattern means the tools are not compiled out — they exist in the binary but are conditionally excluded from the `tools/list` response based on runtime evaluation. This is intentional: compile-time removal is done via Cargo features; runtime hiding is a UX layer.
- Audit events for write operations include both `Allow` and `Deny` outcomes.
