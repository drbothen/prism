# S-6.16 — prism-dtu-datadog: DTU for Datadog Logs API — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Datadog Logs API
at fidelity level L2 (stateful), so that integration tests for the Datadog log-forwarding
destination can validate batched log ingestion, `DD-API-KEY` header auth, 413 (payload
too large), 429 (rate limit), and required log field validation without hitting the live
Datadog API.

## Objective

Implement `prism-dtu-datadog` — an L2 stateful behavioral clone of the Datadog Logs
ingestion API. The DTU implements the single `POST /api/v2/logs` endpoint, validates
the `DD-API-KEY` header, validates batch structure and required fields, captures all
received log batches for test assertion, and simulates 413 and 429 error responses.
See `dtu-assessment.md §3.7.1` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-datadog/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `bytes` (1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `DatadogClone` struct implementing `BehavioralClone`:
   - `received_logs(&self) -> Vec<serde_json::Value>`: test API — returns all captured
     log batches (flattened to individual log objects)

3. Create `src/state.rs` — `DatadogState`:
   - `received_logs: Mutex<Vec<Value>>` — individual log objects from all batches
   - `request_count: AtomicU32` — for rate-limit
   - `reset()` clears both

4. Create `src/routes/logs.rs`:
   - `POST /api/v2/logs`:
     - Validate `DD-API-KEY` header: if absent or empty, return 403
       `{"status": "error", "errors": ["Forbidden"]}`
     - Check `Content-Length` or body size: if > 5MB (5,242,880 bytes), return 413
       `{"status": "error", "errors": ["Request size limit exceeded (5MB)"]}`
     - Parse body as JSON array: if not an array, return 400
       `{"status": "error", "errors": ["Invalid JSON payload"]}`
     - Validate required fields in each log object: `ddsource`, `service`, `message`
       - If any log object is missing a required field, return 400
         `{"status": "error", "errors": ["Missing required field: {field}"]}`
     - On success: append each log object to `received_logs`; return 202
       `{"status": "ok"}`
     - Rate-limit check: if `request_count` exceeds threshold, return 429 with
       `Retry-After: 60`

5. Create `GET /dtu/received-logs` test API:
   - Returns `{"logs": [...], "count": N}` for assertion in integration tests

6. Create fixture files:
   - `fixtures/valid-batch.json`: example valid log batch (array of 3 log objects)
   - `fixtures/413-response.json`: example 413 response body

7. Create `tests/fidelity.rs`:
   - Tests valid batch ingestion, API key auth failure, 413 simulation, 429 rate-limit
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /api/v2/logs` with a valid `DD-API-KEY` header and a JSON array
of 3 log objects each containing `ddsource`, `service`, and `message`, Then the
response is HTTP 202 `{"status": "ok"}` AND `GET /dtu/received-logs` returns 3 objects.

**AC-2:** Given `POST /api/v2/logs` without a `DD-API-KEY` header, Then the response
is HTTP 403 `{"status": "error", "errors": ["Forbidden"]}`.

**AC-3:** Given `POST /api/v2/logs` with a body exceeding 5MB, Then the response is
HTTP 413 `{"status": "error", "errors": ["Request size limit exceeded (5MB)"]}`.

**AC-4:** Given `POST /api/v2/logs` with a JSON array containing a log object missing
the `service` field, Then the response is HTTP 400 with an error listing the missing field.

**AC-5:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then
HTTP 429 with `Retry-After: 60`.

**AC-6:** Given `reset()` is called, Then `received_logs` is empty and rate-limit
counter is reset.

## Architecture Compliance Rules

- DTU clone MUST match the Datadog Logs API v2 contract exactly (same endpoint,
  same request/response shapes, same status codes) as documented in `dtu-assessment.md §3.7.1`.
- DTU clone MUST NOT introduce new behaviors not present in the real Datadog Logs API.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution — all state is in-process.
- The 5MB payload limit check MUST be enforced before JSON parsing to match Datadog's
  behavior (413 before 400).
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — never compiled
  into production binaries.
- See `.factory/specs/architecture/dtu-assessment.md` (or `dtu-assessment.md §4`) for
  cross-cutting DTU rules including ephemeral port binding and `BehavioralClone` contract.
- **Forbidden dependencies:** `prism-dtu-datadog` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Empty JSON array `[]` POSTed | HTTP 202 success; no logs captured (empty batch) |
| EC-002 | `DD-API-KEY` present but empty string | HTTP 403 (empty API key treated as missing) |
| EC-003 | Content-Type header absent | DTU attempts JSON parse anyway; fails with 400 if not valid JSON |
| EC-004 | Log batch with one valid and one invalid object | 400 on first invalid object; entire batch rejected (Datadog behavior: all-or-nothing) |
| EC-005 | Network timeout via FailureLayer::Latency | Prism log forwarder must handle timeout; DTU does not enforce client-side timeout |
| EC-006 | Malformed JSON body (not an array, not even valid JSON) | HTTP 400 `{"status": "error", "errors": ["Invalid JSON payload"]}` |
| EC-007 | `reset()` called between batches in a multi-batch test | `received_logs` clears; subsequent `GET /dtu/received-logs` shows only post-reset logs |

## Dev Notes

L2 (stateful) fidelity: the DTU captures all received log objects for test assertion via
`received_logs()` / `GET /dtu/received-logs`. The all-or-nothing batch rejection behavior
matches the real Datadog Logs API — a single invalid log object in a batch causes the
entire batch to be rejected with 400. The 5MB payload limit check is enforced before JSON
parsing, matching Datadog's documented behavior.

Fidelity L2 is appropriate for the Datadog destination because the primary integration
risk is Prism correctly formatting log batches (required fields, correct auth header, batch
size limits). The captured log test API enables assertion on the exact log shapes Prism
emits without requiring a live Datadog account.
