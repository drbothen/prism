# S-6.19 — prism-dtu-otlp: DTU for OTLP/HTTP Log Ingestion (L2 stateful fidelity)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the OTLP/HTTP log
ingestion endpoint at fidelity level L2 (stateful), so that integration tests for the
OTLP log-forwarding destination can validate protobuf payload decoding, JSON fallback
acceptance, schema version pinning, 400/429/503 error paths, and received request capture
without running a live OpenTelemetry Collector.

## Objective

Implement `prism-dtu-otlp` — an L2 stateful behavioral clone of the OTLP/HTTP `/v1/logs`
endpoint. The DTU accepts `ExportLogsServiceRequest` in both binary protobuf
(`Content-Type: application/x-protobuf`) and JSON (`Content-Type: application/json`)
formats, decodes the protobuf using `prost`-generated types from bundled `.proto` files,
captures all received requests for test assertion, and simulates 400/429/503 error
responses. gRPC (`/opentelemetry.proto.collector.logs.v1.LogsService/Export`) is
explicitly deferred. See `dtu-assessment.md §3.7.4` for full scope matrix.

**Proto schema version:** Coordinate with the log-forwarder implementer to pin to the
same `opentelemetry-proto` revision. The proto files must be the SAME version as
used by `prism-mcp` (the log forwarder — see S-5.09). A schema mismatch causes
protobuf parse errors that are silent and hard to diagnose. Pin by git commit hash
in the `build.rs` comment.

## Tasks

1. Create `crates/prism-dtu-otlp/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `prost` (0.12.x), `bytes` (1.x)
   - Build dependencies: `prost-build` (0.12.x)
   - `[features] dtu = []`
   - **CRITICAL:** `prost` version must match the version used by `prism-mcp`
     (the log forwarder — see S-5.09). Coordinate with implementer of the OTLP log
     forwarder. If `prism-mcp` uses `prost 0.12.x`, this crate must also use `0.12.x`.

2. Create `build.rs`:
   - `prost_build::compile_protos(&["proto/opentelemetry/proto/logs/v1/logs.proto",
     "proto/opentelemetry/proto/common/v1/common.proto",
     "proto/opentelemetry/proto/resource/v1/resource.proto",
     "proto/opentelemetry/proto/collector/logs/v1/logs_service.proto"],
     &["proto/"])?`
   - Comment: `// Proto version: opentelemetry-proto commit {HASH} (coordinate with log-forwarder implementer)`
   - Generated types available as `crate::proto::opentelemetry::proto::collector::logs::v1::ExportLogsServiceRequest`

3. Bundle proto files in `proto/` directory:
   - `proto/opentelemetry/proto/logs/v1/logs.proto`
   - `proto/opentelemetry/proto/common/v1/common.proto`
   - `proto/opentelemetry/proto/resource/v1/resource.proto`
   - `proto/opentelemetry/proto/collector/logs/v1/logs_service.proto`
   - Source: OTLP proto files from the `opentelemetry-proto` repository at the agreed commit hash
   - These are compile-time artifacts; the commit hash must be documented in `build.rs`

4. Create `src/lib.rs` — `OtlpClone` struct implementing `BehavioralClone`:
   - `received_requests(&self) -> Vec<DecodedOtlpRequest>`: test API — returns decoded
     request payloads for assertion
   - `DecodedOtlpRequest { content_type: String, log_count: usize, raw_json: Value }`

5. Create `src/state.rs` — `OtlpState`:
   - `received_requests: Mutex<Vec<DecodedOtlpRequest>>`
   - `request_count: AtomicU32`
   - `reset()` clears both

6. Create `src/routes/logs.rs`:
   - `POST /v1/logs`:
     - Check `Content-Type` header:
       - `application/x-protobuf`: parse body as `ExportLogsServiceRequest` using prost
         - Parse failure: return 400
           `{"error": "Invalid protobuf: {prost error message}"}`
         - On success: decode to JSON for storage; capture in `received_requests`
       - `application/json`: parse body as JSON `ExportLogsServiceRequest`
         - Parse failure: return 400 `{"error": "Invalid JSON: ..."}`
         - On success: capture in `received_requests`
       - Missing or unrecognized `Content-Type`: return 400
         `{"error": "Unsupported Content-Type"}`
     - Rate-limit: if threshold exceeded, return 429 with `Retry-After: 30`
     - `collector_unavailable` mode (configurable): return 503
       `{"error": "Collector unavailable"}` — tests Prism's OTLP retry path
     - On success: return 200 with empty body (OTLP spec: empty `ExportLogsServiceResponse`)
     
     Note on gRPC deferral: requests with `Content-Type: application/grpc` return 501
     `{"error": "gRPC not supported by DTU; use OTLP/HTTP (application/x-protobuf or application/json)"}`.
     This makes the deferral explicit and actionable for the implementer.

7. Create `GET /dtu/received-requests` test API:
   - Returns `{"requests": [{content_type, log_count, raw_json}, ...], "count": N}`

8. Create `POST /dtu/configure`:
   - `{"rate_limit_after": N}` — rate-limit threshold
   - `{"collector_unavailable": true|false}` — enables/disables 503 mode

9. Create fixture files:
   - `fixtures/export-logs-response.json`: empty JSON object `{}` (OTLP success response)
   - `fixtures/valid-export-request.json`: minimal valid `ExportLogsServiceRequest` in JSON
     format (for fidelity test reference; not used by the DTU itself)

10. Create `tests/fidelity.rs`:
    - Tests JSON format ingestion, protobuf format ingestion, 400 on malformed protobuf,
      400 on missing Content-Type, 429 rate-limit, 503 collector unavailable, gRPC 501
    - `received_requests()` assertion: decoded log count matches input
    - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /v1/logs` with `Content-Type: application/json` and a valid
`ExportLogsServiceRequest` JSON body, Then the response is HTTP 200 with empty body AND
`GET /dtu/received-requests` returns a captured request with `content_type: "application/json"`.

**AC-2:** Given `POST /v1/logs` with `Content-Type: application/x-protobuf` and a valid
protobuf-encoded `ExportLogsServiceRequest`, Then the response is HTTP 200 AND
`GET /dtu/received-requests` returns the decoded request with correct `log_count`.

**AC-3:** Given `POST /v1/logs` with `Content-Type: application/x-protobuf` and a
malformed/truncated protobuf body, Then the response is HTTP 400
`{"error": "Invalid protobuf: ..."}`.

**AC-4:** Given `POST /v1/logs` with `Content-Type: application/grpc`, Then the
response is HTTP 501 `{"error": "gRPC not supported by DTU..."}` (explicit deferral message).

**AC-5:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then
HTTP 429 with `Retry-After: 30`.

**AC-6:** Given `{"collector_unavailable": true}` configured via `POST /dtu/configure`,
Then any `POST /v1/logs` returns HTTP 503 `{"error": "Collector unavailable"}` — maps to
Prism's OTLP retry path.

**AC-7:** Given `reset()` is called, Then `received_requests` is empty and rate-limit
counter is reset.

## Risk Mitigations

### R-DTU-011: OTLP protobuf schema versioning

**Risk:** OTLP defines its wire format via protobuf schemas that evolve across
OpenTelemetry specification versions (e.g., 0.9 → 1.0 → 1.3). The `prism-dtu-otlp`
must match the protobuf version that Prism's `otlp` log forwarder in `prism-mcp`
uses. If `prost` serializes using a newer schema than the DTU expects, deserialization
in the DTU will fail silently.

**Mitigation:** Pin the OTLP schema version in `prism-dtu-otlp` to match the
`opentelemetry-proto` crate version used by Prism's forwarder (`prism-mcp`, S-5.09).
Include the protobuf schema version in the crate's `CHANGELOG.md`. DTU update is
required when the forwarder bumps its proto dependency. See `dtu-assessment.md §10
R-DTU-011` for the full risk statement.
