# S-4.05 — prism-operations: Alert Generation

## Narrative

As a Prism detection engine, I want to generate rich alerts from detection rule matches
with interpolated templates, inline event snapshots, and per-rule rate limiting, so that
case managers and downstream consumers receive actionable, self-contained alert records
without needing to re-query sensor data.

## Objective

Implement `TemplateInterpolator::render` with 4-level variable resolution, `EventSnapshot`
inlining, alert persistence to RocksDB with UUID v7 alert IDs, alert broadcast to the
action engine, per-rule rate limiting (100/hour default) under `Mutex`, and injection
safety scanning for template variables. Deliver VP-028 fuzz target verifying the
interpolator never panics.

## Tasks

1. Implement `TemplateInterpolator` in
   `crates/prism-operations/src/detection/template.rs` (BC-2.13.005):
   - Template syntax: `${variable_path}` — dot-notation paths to nested values
   - 4-level variable resolution order (first match wins):
     1. `rule.{field}` — rule metadata (name, id, severity)
     2. `event.{field_path}` — fields from the matched event record
     3. `_meta.{field}` — execution metadata (timestamp, schedule_id, epoch)
     4. Default value: if the variable is not found in levels 1-3, render as
        `"[undefined:{variable_path}]"` — never error, never panic
   - `render(template: &str, vars: &TemplateVars) -> String`
   - Template strings are scanned by `InjectionScanner` (from S-1.10) before rendering;
     if injection patterns are detected, `_safety_flags` are set on the resulting alert
     and the flagged variables are rendered as `"[REDACTED]"`

2. Implement `EventSnapshot` in
   `crates/prism-operations/src/detection/snapshot.rs` (BC-2.13.005):
   - `EventSnapshot { fields: HashMap<String, JsonValue> }`: a copy of all fields from
     the matched event record at the time of alert generation
   - Snapshots are inlined into the `Alert` record — no reference to the original source
     record is stored. Consumers do not need to re-query to get event context.
   - Cap snapshot size at 64KB per event. If the event record exceeds 64KB, truncate
     low-priority fields (raw bytes, base64 blobs) and set `snapshot.truncated = true`.

3. Implement alert construction and persistence in
   `crates/prism-operations/src/detection/alert.rs` (BC-2.13.005):
   - `Alert { id: AlertId, rule_id: RuleId, rule_name: String, title: String,
     description: String, severity_id: SeverityId, event_snapshot: EventSnapshot,
     client_id: ClientId, event_time: Timestamp, triggered_at: Timestamp,
     created_at: Timestamp, acknowledged_at: Option<Timestamp>,
     acknowledged_by: Option<AnalystId>, resolved_at: Option<Timestamp>,
     _safety_flags: Vec<String> }`
   - `event_time`: timestamp from the source event (used for TTD: triggered_at - event_time)
   - `triggered_at`: when the detection rule fired (may differ from event_time)
   - `created_at`: when the alert record was persisted to RocksDB (used for TTI: case.created_at - min(alert.created_at))
   - `acknowledged_by`/`resolved_at`: set by downstream stories (S-4.07, S-4.06)
   - `AlertId`: UUID v7 (time-sortable)
   - Persist to RocksDB `alerts` CF under key `alert:{id}` (lexicographic = time order)
   - After persistence, emit alert to broadcast channel (step 5)

4. Implement per-rule rate limiting in
   `crates/prism-operations/src/detection/rate_limit.rs` (BC-2.13.005):
   - Per-rule counter: `rule_id -> (count, window_start: Timestamp)`
   - Rate limit window: 1-hour sliding window anchored to `window_start`
   - Default limit: 100 alerts/hour (overridable per rule via `rate_limit.max_per_hour`)
   - Enforcement: before generating an alert, check counter; if at limit, suppress and
     log suppression event; if under limit, increment and proceed
   - Counter state: in-memory `HashMap<RuleId, RateLimitBucket>` guarded by `Mutex`.
     Persist to RocksDB `detection_state` CF under `\x01{rule_id}` key after each write
     (CI-004: RocksDB write under Mutex)
   - On restart: load rate limit counters from `detection_state` CF `\x01` prefix scan

5. Implement alert broadcast in
   `crates/prism-operations/src/detection/broadcast.rs` (BC-2.13.005):
   - Use a `tokio::sync::broadcast` channel with capacity 1,000
   - Subscribers: action engine (S-4.06+ scope), MCP streaming endpoint (future)
   - On channel full (lagged): log warning but do NOT block alert persistence. Alerts
     are always persisted to RocksDB regardless of broadcast delivery.

6. Write VP-028 fuzz target in `crates/prism-operations/fuzz/vp028_template_fuzz.rs`:
   - `fuzz_target!(|data: &[u8]| { ... })`: deserialize `data` as an arbitrary template
     string and arbitrary variable map, call `TemplateInterpolator::render()`, verify
     it returns a `String` without panicking
   - Register in `fuzz/Cargo.toml` as a libfuzzer-sys target

## Acceptance Criteria

**AC-1:** Given a rule with template `"Alert: ${rule.name} on ${event.device.hostname}"`
and a matched event with `device.hostname = "server-01"`, When an alert is generated,
Then the alert title is `"Alert: High Severity Detection on server-01"`. (BC-2.13.005)

**AC-2:** Given a template referencing `${event.nonexistent_field}`, When rendered,
Then the output contains `"[undefined:event.nonexistent_field]"` — no panic, no error.
(BC-2.13.005)

**AC-3:** Given an alert, When persisted to RocksDB, Then the alert record contains an
`event_snapshot` with all fields from the matched event record. (BC-2.13.005)

**AC-4:** Given a rule that has already fired 100 alerts this hour, When another match
occurs, Then the alert is suppressed with a logged suppression event and the rate limit
counter is NOT incremented. (BC-2.13.005)

**AC-5:** Given a template containing `${event.malicious_field}` where the field value
contains an injection pattern detected by `InjectionScanner`, When rendered, Then the
alert has `_safety_flags` set and the field value is rendered as `"[REDACTED]"`.
(BC-2.13.005)

**AC-6:** Given an alert is generated, When the broadcast channel has a subscriber,
Then the subscriber receives the alert. When the channel is full, Then the alert is still
persisted to RocksDB without blocking. (BC-2.13.005)

**AC-7:** VP-028 fuzz target runs 30 minutes without any panics.

## Architecture Compliance Rules

- `TemplateInterpolator::render` MUST never panic. Any input (malformed template
  syntax, missing variables, unicode edge cases) must be handled gracefully. The fuzz
  target (VP-028) validates this.
- Rate limit counter writes to RocksDB MUST happen under the same `Mutex` that guards
  the in-memory counter (CI-004). A separate Mutex for the RocksDB write would create
  a TOCTOU window.
- Alert persistence to RocksDB MUST complete before the broadcast channel send. If
  persistence fails, the alert must not be broadcast (no phantom alerts).
- `EventSnapshot` is a copy, not a reference. The original query result records may be
  freed after evaluation. Do not store pointers or indices into the original record
  batch.
- `InjectionScanner` integration follows the pattern established in S-1.10. The scanner
  runs against each `${event.*}` variable value before interpolation. Rule-defined
  variables (`${rule.*}` and `${_meta.*}`) are trusted and not scanned.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Template references `${event.nonexistent_field}` | Rendered as `"[undefined:event.nonexistent_field]"`; no panic, no error (BC-2.13.005) |
| EC-002 | Event record exceeds 64KB snapshot cap | Low-priority fields (raw bytes, base64 blobs) truncated first; `snapshot.truncated = true` set |
| EC-003 | Rule already at 100 alerts/hour rate limit | Alert suppressed; suppression event logged; rate limit counter NOT incremented |
| EC-004 | `TemplateInterpolator::render` receives malformed template syntax (e.g., unclosed `${`) | Returns string with literal `${...}` treated as plain text; VP-028 fuzz validates no panic |
| EC-005 | Broadcast channel at capacity (1,000 msgs) when alert generated | Alert persisted to RocksDB first; broadcast attempted; if lagged, WARN logged; persistence never blocked |
| EC-006 | `InjectionScanner` detects prompt injection in `${event.*}` variable | `_safety_flags` set on alert; field rendered as `"[REDACTED]"`; alert still persisted |
| EC-007 | Rate limit counter load from `detection_state` CF fails on restart | In-memory counter initialised to 0 (conservative: fresh window); WARN logged; server continues |
| EC-008 | `alert.event_time` is null (sensor did not populate timestamp) | `triggered_at` used as fallback; `ttd_approx = true` flag set on alert record |

## Dev Notes

- UUID v7 for `AlertId` provides time-sortable identifiers. When stored in RocksDB under
  key `alert:{id}`, prefix scans by `alert:` automatically return alerts in creation
  order — no separate index is needed for time-ordered listing.
- The 64KB snapshot cap is a balance between completeness and storage cost. Security
  events often contain large base64-encoded payloads (certificates, PE headers). These
  are the low-priority fields to truncate first; human-readable fields (`hostname`,
  `username`, `process_name`) should always be preserved.
- The broadcast channel capacity of 1,000 is intentionally small. The action engine
  (case management in S-4.06) should process alerts quickly. If the channel consistently
  fills, it indicates the action engine is overloaded, not that the capacity needs
  increasing.
- This story covers only alert generation. Alert acknowledgment (`acknowledge_alert`)
  is implemented in S-4.07 because its BC (BC-2.14.012) is currently a STUB that needs
  to be completed before implementation.
