# S-4.05 — prism-operations: Alert Generation

## Narrative

As a Prism detection engine, I want to generate rich alerts from detection rule matches
with interpolated templates, inline event snapshots, and per-rule rate limiting, so that
case managers and downstream consumers receive actionable, self-contained alert records
without needing to re-query sensor data.

## Objective

Implement `TemplateInterpolator::render` with 4-level variable resolution, `EventSnapshot`
inlining, alert persistence to RocksDB with UUID v7 alert IDs, alert broadcast to the
action engine, per-rule rate limiting (100/hour default) under `Mutex`, and injection
safety scanning for template variables. Deliver VP-028 fuzz target verifying the
interpolator never panics.

## Tasks

1. Implement `TemplateInterpolator` in
   `crates/prism-operations/src/detection/template.rs` (BC-2.13.005):
   - Template syntax: `{{variable_path}}` — handlebars-style, dot-notation paths to nested
     values. This syntax is authoritative for S-4.05 and aligns with S-4.08 + ADR-016 §2.9
     (P1-S-4.05-A-M-002). The prior `${var}` syntax is rejected; any code referencing
     `${...}` interpolation in this story is a syntax inconsistency defect.
   - 4-level variable resolution order (first match wins):
     1. `rule.{field}` — rule metadata (name, id, severity)
     2. `event.{field_path}` — fields from the matched event record
     3. `_meta.{field}` — execution metadata (timestamp, schedule_id, epoch)
     4. Default value: if the variable is not found in levels 1-3, render as
        `"[undefined:{variable_path}]"` — never error, never panic
   - `render(template: &str, vars: &TemplateVars) -> String`
   - Template strings are scanned by `InjectionScanner` (from S-1.10) before rendering.
     On injection detection (P1-S-4.05-A-M-003 — reconciled with BC-2.09.004 "flag, don't
     strip"): `_safety_flags` are set on the resulting Alert metadata, AND the field value
     IS rendered in the output (operator sees the value plus the flag). The value is NOT
     redacted. This allows operators to inspect the injected content while being alerted to
     the risk. Only `{{event.*}}` variable values are scanned; `{{rule.*}}` and `{{_meta.*}}`
     are trusted and not scanned.

2. Implement `EventSnapshot` in
   `crates/prism-operations/src/detection/snapshot.rs` (BC-2.13.005):
   - `EventSnapshot { fields: HashMap<String, JsonValue> }`: a copy of all fields from
     the matched event record at the time of alert generation
   - Snapshots are inlined into the `Alert` record — no reference to the original source
     record is stored. Consumers do not need to re-query to get event context.
   - Cap snapshot size at 64KB per event. If the event record exceeds 64KB, truncate
     low-priority fields (raw bytes, base64 blobs) and set `snapshot.truncated = true`.

3. Implement alert construction and persistence in
   `crates/prism-operations/src/detection/alert.rs` (BC-2.13.005):
   - `Alert { id: AlertId, org_id: OrgId, rule_id: RuleId, rule_name: String, title: String,
     description: String, severity_id: SeverityId, event_snapshot: EventSnapshot,
     client_id: ClientId, event_time: Timestamp, triggered_at: Timestamp,
     created_at: Timestamp, acknowledged_at: Option<Timestamp>,
     acknowledged_by: Option<AnalystId>, resolved_at: Option<Timestamp>,
     ttd_approx: bool, _safety_flags: Vec<String> }`
     (`ttd_approx: bool` — set to `true` when `event_time` is null and `triggered_at` is
     used as fallback for TTD calculation; P1-S-4.05-A-L-004)
   - `org_id: OrgId` is MANDATORY — added per ADR-006 §2.1 and DRIFT-405-002. Every alert
     is scoped to exactly one org; `list_alerts` MUST filter by the caller's org context.
   - `event_time`: timestamp from the source event (used for TTD: triggered_at - event_time).
     If null, `triggered_at` is used as fallback and `ttd_approx` is set to `true`.
   - `triggered_at`: when the detection rule fired (may differ from event_time)
   - `created_at`: when the alert record was persisted to RocksDB (used for TTI: case.created_at - min(alert.created_at))
   - `acknowledged_at`/`acknowledged_by`: set by downstream stories (S-4.07 acknowledge_alert, S-4.06 case management does not mutate alerts)
   - `AlertId`: UUID v7 (time-sortable)
   - Persist to RocksDB `alerts` CF under key `{org_id}:alert:{id}` (org_id FIRST per
     ADR-008 universal re-keying rule + ADR-016 §2.5; DRIFT-405-002 remediation). Lexicographic
     order within an org is preserved since `{id}` is UUID v7.
   - After persistence, emit alert to broadcast channel (step 5)

4. Implement per-rule rate limiting in
   `crates/prism-operations/src/detection/rate_limit.rs` (BC-2.13.005):
   - Per-rule counter: `rule_id -> (count, window_start: Timestamp)`
   - Rate limit window: 1-hour sliding window anchored to `window_start`
   - Default limit: 100 alerts/hour (overridable per rule via `rate_limit.max_per_hour`)
   - Enforcement: before generating an alert, check counter; if at limit, suppress and
     log suppression event; if under limit, increment and proceed
   - Counter state: in-memory `HashMap<RuleId, RateLimitBucket>` guarded by `Mutex`.
     **CRITICAL (P1-S-4.05-A-H-001 — discriminator collision fix):** Persist to RocksDB
     `action_state` CF (NOT `detection_state` CF) under key
     `{org_id}:\x00:{action_id}:{hour_bucket}` (per ADR-016 §2.5 `action_state` CF
     discriminator table: `\x00` = rate limit counter). The `detection_state` CF does NOT
     contain rate-limit entries — that CF is owned by ADR-015 (correlation/sequence/dedup).
     (CI-004: RocksDB write under Mutex)
   - On restart: load rate limit counters from `action_state` CF `{org_id}:\x00:` prefix scan

5. Implement alert broadcast in
   `crates/prism-operations/src/detection/broadcast.rs` (BC-2.13.005):
   - Use a `tokio::sync::broadcast` channel with capacity 1,000. Note: tokio internally
     rounds this to the next power of two — effective ring size is 1,024 slots (R-12).
     Specify 1,024 explicitly in code comments so the actual capacity is not surprising.
   - Subscribers: action engine (S-4.06+ scope), MCP streaming endpoint (future)
   - On `RecvError::Lagged(n)`: MUST increment `metrics::counter!("broadcast.lagged.alert", n)`
     before continuing — dropped messages must never be silent (R-12 mandatory observability).
     The receiver is NOT disconnected after Lagged; continue consuming.
   - On channel full (lagged): log warning but do NOT block alert persistence. Alerts
     are always persisted to RocksDB regardless of broadcast delivery.
   - Broadcast loop pattern:
     ```rust
     loop {
         match rx.recv().await {
             Ok(alert) => process(alert),
             Err(RecvError::Lagged(n)) => {
                 metrics::counter!("broadcast.lagged.alert", n);
                 // continue — receiver not disconnected
             }
             Err(RecvError::Closed) => break,
         }
     }
     ```

6. Write VP-028 fuzz target in `crates/prism-operations/fuzz/vp028_template_fuzz.rs`
   using the canonical cargo-fuzz + libfuzzer-sys + arbitrary stack (R-5):
   - Create `fuzz/` subcrate via `cargo fuzz init` (or manually); add `libfuzzer-sys = "0.4"`
     and `arbitrary = { version = "1", features = ["derive"] }` to `fuzz/Cargo.toml`.
   - Define structured input: `#[derive(Arbitrary)] struct FuzzInput { template: String, vars: HashMap<String, String> }`
   - `fuzz_target!(|input: FuzzInput| { let _ = TemplateInterpolator::render(&input.template, &input.vars); })`
   - CI cadence: `cargo +nightly fuzz run vp028_template_fuzz -- -max_total_time=300` on
     PR CI (30-min short run). Longer soak (≥1h) on nightly schedule. Document in TD
     register if full 24h run is deferred.
   - Do NOT roll a custom fuzzing harness — libfuzzer-sys 0.4 + cargo-fuzz is the only
     supported canonical path (R-5).

## Acceptance Criteria

**AC-1:** Given a rule with `name = <rule_name>` and template
`"Alert: {{rule.name}} on {{event.device.hostname}}"`, and a matched event with
`device.hostname = <hostname>`, When an alert is generated, Then the alert title equals
`"Alert: <rule_name> on <hostname>"` where `<rule_name>` and `<hostname>` are the exact
values from the test fixture — the AC is parameterized on fixture values, not hardcoded
strings. Template syntax is `{{var}}` handlebars-style (P1-S-4.05-A-M-002). (BC-2.13.005 —
QUAL-405-002 remediation; traces to BC-2.13.005 postcondition: template variables resolved
in declared order)

**AC-2:** Given a template referencing `{{event.nonexistent_field}}`, When rendered,
Then the output contains `"[undefined:event.nonexistent_field]"` — no panic, no error.
(BC-2.13.005)

**AC-3:** Given an alert, When persisted to RocksDB, Then the alert record contains an
`event_snapshot` with all fields from the matched event record. (BC-2.13.005)

**AC-4:** Given a rule that has already fired 100 alerts this hour, When another match
occurs, Then the alert is suppressed with a logged suppression event and the rate limit
counter is NOT incremented. (BC-2.13.005)

**AC-5:** Given a template containing `{{event.malicious_field}}` where the field value
contains an injection pattern detected by `InjectionScanner`, When rendered, Then the
alert has `_safety_flags` set in Alert metadata AND the field value IS rendered in the
output (not redacted) — the operator sees the injected value alongside the safety flag.
(BC-2.13.005 — traces to BC-2.09.004 "flag, don't strip" invariant; P1-S-4.05-A-M-003)

**AC-6:** Given an alert is generated, When the broadcast channel has a subscriber,
Then the subscriber receives the alert. When the channel is full (Lagged), Then the alert
is still persisted to RocksDB without blocking AND `broadcast.lagged.alert` metric counter
is incremented by the number of skipped messages. (BC-2.13.005; R-12 observability rule)

**AC-6b:** Given alerts belonging to org A stored under keys `{org_a_id}:alert:{id}`,
When `list_alerts` is called in org B's context, Then org A's alerts are NOT returned.
(traces to ADR-006 §2.1 org-isolation invariant; ADR-008 universal re-keying; DRIFT-405-002
remediation; key format updated from `alert:{org_id}:{id}` to `{org_id}:alert:{id}` per
ADR-016 §2.5)

**AC-7:** VP-028 fuzz target (`cargo +nightly fuzz run vp028_template_fuzz -- -max_total_time=300`)
runs at least 30 minutes on a CI runner without any panics. If the full 24h soak run is
deferred, it is documented in the TD register with a scheduled follow-up. (BC-2.13.005)

## Architecture Compliance Rules

- `TemplateInterpolator::render` MUST never panic. Any input (malformed template
  syntax, missing variables, unicode edge cases) must be handled gracefully. The fuzz
  target (VP-028) validates this.
- Rate limit counter writes to RocksDB MUST happen under the same `Mutex` that guards
  the in-memory counter (CI-004). A separate Mutex for the RocksDB write would create
  a TOCTOU window.
- Alert persistence to RocksDB MUST complete before the broadcast channel send. If
  persistence fails, the alert must not be broadcast (no phantom alerts).
- `EventSnapshot` is a copy, not a reference. The original query result records may be
  freed after evaluation. Do not store pointers or indices into the original record
  batch.
- `InjectionScanner` integration follows the pattern established in S-1.10 (P1-S-4.05-A-M-003,
  reconciled with BC-2.09.004 "flag, don't strip"). The scanner runs against each
  `{{event.*}}` variable value before interpolation. On detection: `_safety_flags` is set
  on the Alert metadata AND the value IS rendered in the output — it is NOT redacted.
  Operators see the value plus the flag. `{{rule.*}}` and `{{_meta.*}}` variables are
  trusted and not scanned. Any code that redacts (replaces with `[REDACTED]`) injected
  values contradicts BC-2.09.004 and this story's semantics.
- `Alert.org_id` MUST be populated from the evaluation context before persistence.
  Alerts without `org_id` MUST NOT be persisted — they are a multi-tenant isolation
  defect (ADR-006 §2.1; DRIFT-405-002 remediation).
- RocksDB `alerts` CF key MUST be `{org_id}:alert:{id}` with org_id FIRST (NOT
  `alert:{org_id}:{id}` and NOT bare `alert:{id}`). The org_id-first format satisfies
  ADR-008 universal re-keying rule + ADR-016 §2.5. Bare keys or wrong-order keys allow
  `list_alerts` to leak cross-org results — a critical isolation failure.
- Rate-limit counter writes MUST go to `action_state` CF (ADR-016 §2.5). Writing
  rate-limit state to `detection_state` CF is a CF-ownership violation (P1-S-4.05-A-H-001
  discriminator collision fix; in lockstep with P1-S-4.04-A-H-001).
- On `RecvError::Lagged(n)` from the broadcast channel, `metrics::counter!("broadcast.lagged.alert", n)`
  MUST be incremented before continuing. Silently discarding `Lagged` errors means
  dropped messages are undetectable in production (R-12 mandatory observability rule).

## Dev Notes

- UUID v7 for `AlertId` provides time-sortable identifiers. When stored in RocksDB under
  key `{org_id}:alert:{id}` (org_id first per ADR-008 + ADR-016 §2.5), prefix scans by
  `{org_id}:alert:` automatically return that org's alerts in creation order — no separate
  index is needed for time-ordered listing within an org.
- The 64KB snapshot cap is a balance between completeness and storage cost. Security
  events often contain large base64-encoded payloads (certificates, PE headers). These
  are the low-priority fields to truncate first; human-readable fields (`hostname`,
  `username`, `process_name`) should always be preserved.
- The broadcast channel capacity of 1,000 is intentionally small. The action engine
  (case management in S-4.06) should process alerts quickly. If the channel consistently
  fills, it indicates the action engine is overloaded, not that the capacity needs
  increasing.
- This story covers only alert generation. Alert acknowledgment (`acknowledge_alert`)
  is implemented in S-4.07 because its BC (BC-2.14.012) is currently a STUB that needs
  to be completed before implementation.
