# S-5.01-FOLLOWUP-MCP-BOOT — prism-mcp: PrismServer, rmcp 1.4, Tool Router, Injection Defense

## Narrative

As the Prism MCP server, I want the `prism-mcp` crate fully implemented with a `PrismServer`
struct, rmcp 1.4 `ServerHandler` implementation, `#[tool_router]` macro for all tools in
the BC-2.13.* catalog, per-tool prompt injection defense via the existing
`prism-security::InjectionScanner`, `ResponseEnvelope` wrapping on every tool response, and
per-tool input validation with structured MCP error codes, so that Claude Code (or any MCP
client) can connect to `prism-bin` over stdio and invoke all Prism capabilities.

## Objective

Create `crates/prism-mcp/src/server.rs` with the `PrismServer` struct per ADR-022 §F.
Add `rmcp = "1.4"` to `prism-mcp/Cargo.toml` and `[workspace.dependencies]`. Implement
the `#[tool_router]` macro for all tools declared in `api-surface.md` BC-2.13.*. Wire
injection defense (BC-2.09.001..008 from S-1.10) at EVERY tool entry boundary — no exemptions.
Implement per-tool input validation and MCP error code mapping per ADR-022 §F.

Per ADR-022 §F: `prism-mcp` currently has no rmcp dependency; `lib.rs` is 10 lines
(`pub mod safety_envelope` + `pub mod tool_registry` only). This story builds the full
MCP server from those stubs.

No `todo!()` or `unimplemented!()` may remain in any tool handler or the server struct
before merge. Per POL-12.

2. Create `crates/prism-mcp/src/server.rs`:

   ```rust
   use rmcp::{ServerHandler, tool_router, McpServer};
   use rmcp::transport::stdio_transport;

   pub struct PrismServer {
       engine: Arc<QueryEngine>,
       write_executor: Arc<WriteExecutor>,
       audit: Arc<AuditEmitter>,
       security: Arc<SecurityConfig>,
       injection_scanner: Arc<InjectionScanner>,
       tool_router: ToolRouter<PrismServer>,
   }

   impl PrismServer {
       pub fn new(
           engine: Arc<QueryEngine>,
           write_executor: Arc<WriteExecutor>,
           audit: Arc<AuditEmitter>,
           security: Arc<SecurityConfig>,
       ) -> Self {
           let injection_scanner = Arc::new(InjectionScanner::new());
           // Build tool router via #[tool_router] macro
           let tool_router = PrismServer::tool_router();
           Self { engine, write_executor, audit, security, injection_scanner, tool_router }
       }

       pub async fn serve_stdio(self) -> Result<(), McpError> {
           let service = McpServer::new(self);
           let (stdin, stdout) = (tokio::io::stdin(), tokio::io::stdout());
           service.serve(stdio_transport(stdin, stdout)).await
       }
   }
   ```

3. Read `api-surface.md` (v1.6) for the canonical tool inventory BEFORE implementing
   the tool router. The tool router MUST implement all tools in the BC-2.13.* catalog.

   Tool categories (minimum viable set per BC-2.13.*):
   - **Query tools:** `query`, `explain_query`, `create_alias`, `list_aliases`, `delete_alias`, `explain_alias`
   - **Sensor health tools:** `check_sensor_health`, `get_diagnostics`
   - **Config management tools:** `reload_config`, `add_sensor_spec`, `list_sensor_specs`, `validate_config`, `list_capabilities`
   - **Write tools:** `confirm_action` (all write operations go through `prism query`; direct write tool surface is confirm_action only per S-3.07)
   - **Schedule tools:** `create_schedule`, `list_schedules`, `delete_schedule`, `get_diff_results`
   - **Detection tools:** `create_rule`, `list_rules`, `delete_rule`
   - **Case management tools:** `create_case`, `list_cases`, `get_case`, `update_case`, `case_metrics`

   Note: prism-operations (SS-12/13/14) stories are not yet merged. Tool handlers for
   schedule, detection, and case management tools that depend on prism-operations MUST
   return a structured `McpError::NotImplemented` response (not panic) until prism-operations
   is implemented. Use the `[NEW-BC-NEEDED]` pattern — document in Open Questions.

4. Implement `#[tool_router]` macro block for `PrismServer`:

   ```rust
   #[tool_router]
   impl PrismServer {
       /// Query Prism data sources using PrismQL.
       /// DATA SOURCE: Sensor APIs. DATA TRUST LEVEL: External/untrusted...
       /// [full security warning per BC-2.09.006]
       #[tool(name = "query", description = "...")]
       async fn tool_query(&self, params: QueryToolParams) -> Result<CallToolResult, McpError> {
           // Step 1: Injection scan on ALL raw string inputs
           let scan = self.injection_scanner.scan_all(&params.raw_inputs())?;
           if scan.has_violations() {
               return Err(McpError::custom(-32002, "Input rejected: prompt injection detected"));
           }
           // Step 2: Validate required fields
           let query_str = params.query.ok_or_else(|| McpError::invalid_params("query is required"))?;
           // Step 3: Execute
           let result = self.engine.execute(&query_str, params.into()).await
               .map_err(map_prism_error)?;
           // Step 4: Wrap in ResponseEnvelope
           Ok(ResponseEnvelope::new(result).into())
       }
       // ... all other tools follow same pattern
   }
   ```

5. Implement injection defense wiring at EVERY tool entry boundary (NON-NEGOTIABLE per ADR-022 §F):

   ```rust
   // Template for every tool handler — copy verbatim; do not skip for any tool
   let scan_result = self.injection_scanner.scan_all(&params.raw_inputs())?;
   if scan_result.has_violations() {
       return Err(McpError::custom(-32002, "Input rejected: prompt injection detected"));
   }
   ```
   BC-2.09.001..008 are ALREADY IMPLEMENTED in `prism-security::InjectionScanner` (S-1.10,
   status: merged). This story WIRES the scanner at every tool boundary. There are NO exempt
   tools. Skipping even one tool violates BC-2.09.001 invariant.

6. Implement per-tool input validation:
   - Deserialize MCP tool parameters from JSON.
   - Validate required fields (non-null, type correct, within bounds).
   - On validation failure: return `McpError` with code `-32602` (Invalid params) and a
     message naming the invalid field. NEVER `panic!()` or `unwrap()`.

7. Implement MCP error code mapping at `crates/prism-mcp/src/error_mapping.rs`:

   ```rust
   pub fn map_prism_error(err: PrismError) -> McpError {
       match &err {
           PrismError::ParseError { .. } =>
               McpError::invalid_params(format!("PrismQL parse error: {err}")),
           PrismError::WriteNotImplemented { sensor } =>
               McpError::custom(-32003, format!("Write not supported for sensor: {sensor}")),
           PrismError::PermissionDenied { .. } | PrismError::FeatureFlagDenied { .. } =>
               McpError::custom(-32002, format!("Feature flag denied: {err}")),
           PrismError::QueryTimeout { .. } =>
               McpError::custom(-32001, "Query timeout exceeded".into()),
           _ =>
               McpError::internal_error(format!("Internal error; see audit log")),
       }
   }
   ```
   Mapping per ADR-022 §F table. Implement all variants listed in `error-taxonomy.md`.

8. Wrap EVERY tool response in `ResponseEnvelope::new(result)`:
   - `ResponseEnvelope` is already in `prism-mcp/src/safety_envelope.rs` (from S-1.10).
   - Every tool return value MUST pass through `ResponseEnvelope::new(...)` before being
     returned as `CallToolResult`. This provides the `{data, _meta: {trust_level, safety_flags,
     source_sensor, query_id}}` wrapper per BC-2.09.008.

9. Implement tool description security warnings per BC-2.09.006:
   - Every sensor query tool description MUST include `DATA SOURCE:`, `DATA TRUST LEVEL: External/untrusted`,
     `SECURITY NOTE:` sections.
   - Use the `provenance_framing` helper from S-1.10 (`prism-mcp/src/tool_registry.rs`)
     to append warnings. Do NOT embed warnings by hand in each tool's description string.

10. Wire `OutputSchema` per BC-2.09.007:
    - Each tool registered via `#[tool_router]` must include an `outputSchema` (JSON Schema).
    - The schema must declare `_meta` envelope fields and `results` array item types.
    - Reuse the `OutputSchema` types from `prism-security/src/output_schema.rs` (S-1.10).

11. Write integration tests in `crates/prism-mcp/tests/tool_dispatch_tests.rs`:
    - Test: `tool_query` with injection payload → rejected with `-32002`.
    - Test: `tool_query` with valid PrismQL → `ResponseEnvelope` returned with `trust_level`.
    - Test: `tool_query` with missing required field → `-32602` (Invalid params).
    - Test: `PrismError::ParseError` maps to `-32602`.
    - Test: `PrismError::WriteNotImplemented` maps to `-32003`.
    - Test: All 15 behavioral contract IDs are represented by at least one test case.

## Acceptance Criteria

**AC-1:** Given an MCP client connects via stdio and sends `tools/list`, When `PrismServer`
receives the request, Then it returns the full tool catalog from BC-2.13.* with all tools
listed, each with a `description` containing `DATA TRUST LEVEL: External/untrusted` and a
`SECURITY NOTE:` section for sensor-data-returning tools.
(traces to BC-2.10.002 postcondition — tool registration; BC-2.09.006 — security warnings)

**AC-2:** Given any tool is invoked with string parameters containing `"ignore previous instructions"`,
When the tool handler runs, Then `InjectionScanner.scan_all()` fires before any domain logic,
the input is rejected with `McpError::custom(-32002, "Input rejected: prompt injection detected")`,
and the `QueryEngine` is NEVER called.
(traces to BC-2.09.003 postcondition — scanner at every boundary; BC-2.09.001 structural separation)

**AC-3:** Given a valid `tool_query` call with `query = "FROM crowdstrike_detections LIMIT 5"`,
When the query executes and returns results, Then the MCP response is wrapped in
`ResponseEnvelope::new(result)` with `_meta.trust_level = "untrusted_external"` and
`_meta.safety_flags` present (empty array if no patterns detected).
(traces to BC-2.09.008 postcondition — response envelope; BC-2.09.005 — trust metadata)

**AC-4:** Given `tool_query` called with `query` field null/missing, When the tool handler
runs, Then `McpError` with code `-32602` is returned naming the missing field. No panic,
no `unwrap()`.
(traces to BC-2.10.007 postcondition — structured error responses)

**AC-5:** Given `PrismError::ParseError` from `QueryEngine::execute`, When `map_prism_error`
runs, Then `McpError` with code `-32602` is returned with `"PrismQL parse error: {detail}"`.
(traces to BC-2.10.007 postcondition — error code mapping)

**AC-6:** Given `PrismServer::serve_stdio(self)` is called (from boot step 9), When a SIGTERM
arrives, Then `serve_stdio` exits cleanly after processing any in-flight tool call, without
dropping the audit buffer or corrupting RocksDB state.
(traces to BC-2.10.010 postcondition — graceful shutdown)

**AC-7:** Given a tool targeting a write-disabled sensor (feature flag = Deny), When the
tool handler calls `WriteExecutor` which returns `FeatureFlagDenied`, Then the tool returns
`McpError::custom(-32002, "Feature flag denied: ...")`. The tool is NOT absent from `tools/list`
if it is enabled for at least one other client (stateless visibility model per BC-2.10.003).
(traces to BC-2.10.003 postcondition — conditional tool registration)

**AC-8:** Given `tool_query` with `client = "acme"` and `client = "globex"` in separate calls,
When both execute on the same `PrismServer` instance (stateless model), Then results are
scoped to the respective client and no cross-client data leakage occurs.
(traces to BC-2.10.004 postcondition — client scoping on every tool)

**AC-9:** Given every registered tool's `tools/list` entry is inspected, When the tool
definition is read, Then each tool includes an `outputSchema` declaring `_meta` envelope
fields, `results` array item types, and `_meta.safety_flags` as `type: "array"` with
structured items. No per-field parallel `safety_flag` fields appear.
(traces to BC-2.09.007 postcondition — OutputSchema for type-safe LLM reasoning)

**AC-10:** No `todo!()` or `unimplemented!()` may remain in any tool handler, `PrismServer`,
or `map_prism_error` before merge. Per POL-12. Tools that depend on prism-operations (not
yet merged) return `McpError::NotImplemented` (not panic).

## Dev Notes

- The rmcp 1.4 crate API (specifically `#[tool_router]` macro, `McpServer::new`, and
  `stdio_transport`) must be verified at TDD time. The API may differ from what ADR-022
  §F describes if rmcp 1.4 is not yet published under that name.
  Check `https://crates.io/crates/rmcp` for current stable version and API.
- prism-operations tools (schedule, detection, case management) returning structured
  `NotImplemented` responses is the correct partial-merge behavior. Document these in
  the PR description. They will be filled by Wave 4/5 story delivery.
- `PrismServer` must be `Send + Sync` because `McpServer::new(self)` requires it.
  Ensure all fields (`Arc<...>`) satisfy this constraint.
- Tool parameter structs should derive `serde::Deserialize`. Use `#[serde(deny_unknown_fields)]`
  on parameter structs to prevent parameter injection via unrecognized fields.
- The `raw_inputs()` method on tool params should return all string field values as a
  `Vec<&str>` for the injection scanner. Implement it as a simple method collecting
  all `String` fields from the params struct.
