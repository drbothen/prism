# S-3.3.01 — prism-customer-config: TOML schema, parser, and startup validator

## Narrative

As a Prism operator, I want each customer organization described in a
`customers/{org_slug}.toml` file that is validated at startup, so that
misconfigured customer entries are caught before the server binds any MCP
connections and before any AI context window receives config data.

## Objective

Create the `prism-customer-config` crate (or extend `prism-spec-engine`, per
architect's final call — the module boundary is declared here; the crate name is
the implementation target). The crate owns the TOML schema struct, serde
deserialization with `deny_unknown_fields`, and the multi-error validation pass
that produces structured `ConfigError` values from all 12 R-CUST-NNN rules plus
the credential heuristic pass (BC-3.3.002) and the schema-version gate
(BC-3.3.003).

## Tasks

1. Create `crates/prism-customer-config/Cargo.toml`:
   - Workspace path dep; features: `["dtu"]` gated on `#[cfg(any(test, feature = "dtu"))]`
   - Dependencies: `serde` (1.x derive), `serde_json` (1.x), `toml` (0.8.x),
     `uuid` (1.x, features = ["v7"]), `thiserror` (1.x), `tracing` (0.1.x)

2. Create `src/schema.rs` — serde structs for the TOML schema (ADR-010 §2.2 + §2.3):
   - `CustomerConfig { schema_version: u64, org_id: Uuid, org_slug: String, display_name: String, dtu: Vec<DtuBlock>, shared_infra: Option<SharedInfra> }`
   - `DtuBlock { r#type: String, mode: String, credential_ref: String, spec: Option<String>, data: Option<DtuData> }`
   - `DtuData { archetype: Option<String>, scale: Option<f64>, seed: Option<u64> }`
   - `SharedInfra` struct (ADR-010 §2.4 fields)
   - All structs derive `Deserialize` with `#[serde(deny_unknown_fields)]`

3. Create `src/error.rs` — `ConfigError` enum:
   - Variants map 1:1 to error codes:
     `MissingField { file, field }` → `E-CFG-001`
     `SlugMismatch { file, slug, stem }` → `E-CFG-002`
     `InvalidOrgIdVersion { file, value }` → `E-CFG-003`
     `UnknownDtuType { file, dtu_type }` → `E-CFG-004`
     `InvalidCredentialRef { file, field }` → `E-CFG-005`
     `UnknownArchetype { file, value }` → `E-CFG-006`
     `InvalidSeed { file, value }` → `E-CFG-007`
     `InvalidScale { file, value }` → `E-CFG-008`
     `InvalidMode { file, index, value }` → `E-CFG-009`
     `UnknownField { file, field }` → `E-CFG-010`
     `DuplicateOrgId { file1, file2, org_id }` → `E-CFG-011`
     `DuplicateOrgSlug { file1, file2, slug }` → `E-CFG-012`
     `MissingClientSpec { file, dtu_index }` → `E-CFG-013`
     `SuspectedCredentialValue { file, field_name }` → `E-CFG-020`
     `MissingSchemaVersion { file }` → `E-CFG-030`
     `UnsupportedSchemaVersion { file, found, migration_hint: bool }` → `E-CFG-031`
     `TomlParseError { file, inner }` → `E-CFG-000`
   - Implement `Display` producing error lines per BC-3.3.004 §Postconditions
   - Error message for `E-CFG-020` MUST NOT include the field value (BC-3.3.002 Invariant 3)

4. Create `src/credential_check.rs` — heuristic credential scanner:
   - Recursively walks the parsed TOML `Value` tree
   - Matches field names ending in `_token`, `_secret`, `_key`, `_password`, `_pass`,
     or exactly `password` (BC-3.3.002 R-CRED-001 through R-CRED-006)
   - A matching field name with a non-scheme-prefixed value produces `SuspectedCredentialValue`
   - Allowed scheme prefixes: `vault://`, `env://`, `file://`, `keyring://`
   - `credential_ref` field validated separately as R-CRED-007 (produces `E-CFG-005`)

5. Create `src/validator.rs` — `validate_all(dir: &Path) -> Vec<ConfigError>`:
   - Reads all `*.toml` files from `dir` in lexicographic filename order (BC-3.3.004 Invariant 4)
   - Non-`.toml` files are silently skipped (EC-3.3.001-07)
   - For each file:
     a. TOML parse → `E-CFG-000` on parse error
     b. `schema_version` check first (BC-3.3.003 Invariant 4): absent → `E-CFG-030`; value ≠ 1 → `E-CFG-031` with migration hint if > 1
     c. Credential heuristic pass (BC-3.3.002) on the raw TOML value tree
     d. Structural validation (R-CUST-001 through R-CUST-013)
     e. UUID version nibble check: `org_id` version byte must be 7 (R-CUST-003)
     f. `org_slug` must equal file stem, case-sensitive (R-CUST-002)
     g. `demo-server` absent from `DTU_DEFAULT_MODE`; any `[[dtu]]` with `type = "demo-server"` → `E-CFG-004`
     h. Security Telemetry type + `mode = "shared"` → startup error (BC-3.3.001-startup).
        **Wave 3 unconditional guard:** the ST guard is unconditional — `allow_shared_override`
        is NOT IMPLEMENTED (ADR-007 §7 OQ-1 DEFERRED to Wave 4). Any `allow_shared_override`
        field in a `[[dtu]]` block triggers `E-CFG-010` (unknown field via `deny_unknown_fields`)
        before the ST guard is even reached.
   - Cross-file: collect all `(org_id, file)` and `(org_slug, file)` pairs; detect duplicates → `E-CFG-011`/`E-CFG-012`
   - Returns ALL errors collected across ALL files (multi-error, not fail-fast)
   - Returns empty vec on success

6. Create `src/lib.rs` — public API:
   - `pub fn load_and_validate(dir: &Path) -> Result<Vec<CustomerConfig>, Vec<ConfigError>>`
   - Returns `Err(errors)` if any errors; `Ok(configs)` if all pass
   - Exposes `CustomerConfig`, `DtuBlock`, `ConfigError` as public types

7. Create `tests/validation_tests.rs` — unit tests covering all 13 canonical test vectors
   from BC-3.3.004 (TV-3.3.004-01 through TV-3.3.004-13), all 8 from BC-3.3.002
   (TV-3.3.002-01 through TV-3.3.002-08), and all 7 from BC-3.3.003
   (TV-3.3.003-01 through TV-3.3.003-07).

## Acceptance Criteria

**AC-001:** Given a `customers/` directory with zero `.toml` files, `load_and_validate` returns `Ok([])` and the process proceeds normally.
(traces to BC-3.3.004 precondition 5; EC-3.3.004-01)

**AC-002:** Given a `customers/acme.toml` with `org_id` field absent, `load_and_validate` returns `Err([ConfigError::MissingField { field: "org_id", .. }])` containing error code `E-CFG-001` in the Display output, with exit code 1, and `OrgRegistry` receives zero registrations.
(traces to BC-3.3.004 postcondition "On any validation failure" clause 1, 2, 3, 4)

**AC-003:** Given a `customers/acme-corp.toml` with `org_slug = "acme-new"`, `load_and_validate` returns `Err` containing `E-CFG-002` naming both `"acme-new"` and `"acme-corp"` in the error message.
(traces to BC-3.3.004 postcondition "Specific rejection rules" R-CUST-002)

**AC-004:** Given a `customers/acme.toml` with a valid UUID v4 value for `org_id`, `load_and_validate` returns `E-CFG-003` stating "UUID v4; must be UUID v7".
(traces to BC-3.3.004 postcondition R-CUST-003; EC-3.3.004-06)

**AC-005:** Given a `customers/acme.toml` with `[[dtu]] type = "demo-server"`, `load_and_validate` returns `E-CFG-004` naming `"demo-server"`.
(traces to BC-3.3.004 postcondition R-CUST-004; BC-3.3.004 invariant 3)

**AC-006:** Given a `customers/acme.toml` with `credential_ref = "bearer-token-abc123"` (no scheme prefix), `load_and_validate` returns `E-CFG-005` listing the four allowed schemes.
(traces to BC-3.3.004 postcondition R-CUST-005)

**AC-007:** Given a `customers/acme.toml` with `data.scale = 0.0`, `load_and_validate` returns `E-CFG-008` naming `0.0`. Given `data.scale = NaN`, returns `E-CFG-008` stating "NaN is not a positive finite float".
(traces to BC-3.3.004 postcondition R-CUST-008; EC-3.3.004-04)

**AC-008:** Given two `customers/*.toml` files with the same `org_id` UUID, `load_and_validate` returns `E-CFG-011` naming both filenames.
(traces to BC-3.3.004 postcondition R-CUST-011)

**AC-009:** Given a valid `customers/acme.toml` (all required fields correct, UUID v7, slug matches stem, `schema_version = 1`, opaque credential refs), `load_and_validate` returns `Ok([config])` with `OrgRegistry` receiving exactly one entry.
(traces to BC-3.3.004 postcondition "On successful validation" clause 1)

**AC-010:** Given a `customers/acme.toml` with a single file containing three distinct violations (missing `org_id`, unknown field, bad seed), `load_and_validate` returns all three error codes (`E-CFG-001`, `E-CFG-010`, `E-CFG-007`) in the error vec — multi-error, not fail-fast.
(traces to BC-3.3.004 EC-3.3.004-02)

**AC-011:** Given `[[dtu]] bearer_token = "abc123"` in a customer TOML, `load_and_validate` returns `E-CFG-020` naming the field `'bearer_token'`; the error message MUST NOT contain the string `"abc123"`.
(traces to BC-3.3.002 postcondition "On credential value detected" clauses 2, 3)

**AC-012:** Given `credential_ref = "vault://sensors/acme/cs/secret"`, `credential_ref = "env://ACME_KEY"`, `credential_ref = "file:///etc/prism/key"`, and `credential_ref = "keyring://prism/acme/sensor"` — all four scheme prefixes pass credential validation without error.
(traces to BC-3.3.002 postcondition "On valid config" clause 1; BC-3.3.002 invariant 4)

**AC-013:** Given `schema_version` field absent, `load_and_validate` returns `E-CFG-030` (distinct from `E-CFG-031`).
(traces to BC-3.3.003 postcondition "On schema_version absent" clause 2; invariant 2)

**AC-014:** Given `schema_version = 2` (future version), `load_and_validate` returns `E-CFG-031` containing the migration hint string `"prism config migrate"`.
(traces to BC-3.3.003 postcondition "On schema_version present but not equal to 1" clause 3; invariant 3)

**AC-015:** Given a `customers/` directory containing both a `.toml` file and a `README.md`, only the `.toml` file is validated; `README.md` is silently skipped.
(traces to BC-3.3.004 EC-3.3.004-07)

**AC-016:** Given a `customers/acme.toml` with `[[dtu]] type = "claroty" mode = "shared"`, `load_and_validate` returns a startup error naming `"claroty"` and the config file path, with a suggestion to set `mode = "client"`. The error message does NOT mention `allow_shared_override` because the escape hatch is not implemented in Wave 3.
(traces to BC-3.3.001-startup postcondition clause 1, 2; ADR-007 §2.4 rule 3 + §7 OQ-1 DEFERRED)

**AC-017:** Given a `customers/acme.toml` with `[[dtu]] allow_shared_override = true`, `load_and_validate` returns `E-CFG-010` (unknown field) because `deny_unknown_fields` is set and `allow_shared_override` is not a recognized field in Wave 3. The process does not start.
(traces to BC-3.3.004 postcondition R-CUST structural validation via `deny_unknown_fields`; ADR-007 §7 OQ-1 DEFERRED to Wave 4)

- `prism-dtu-harness` (test infra; no production dependency on harness)

  `DTU_DEFAULT_MODE` registry

## Architecture Compliance Rules

Per `architecture/decisions/ADR-010-customer-config-schema.md`:
1. Validation MUST complete for all files before `OrgRegistry::register` is called for any file (§2.5 step order) — enforced by returning `Vec<ConfigError>` from `validate_all` before the registry phase begins.
2. `deny_unknown_fields` is mandatory on all serde structs — no optional unknown field passthrough.
3. Schema version check runs FIRST within the per-file validation pass (§2.6), before UUID, slug, or DTU type checks.
4. `demo-server` is excluded from `DTU_DEFAULT_MODE` by absence — there is no explicit denylist; the exclusion is structural (D-051 lock).
5. **`allow_shared_override` is NOT IMPLEMENTED in Wave 3** (ADR-007 §7 OQ-1, DEFERRED to Wave 4). The field is not present in any serde struct. Any `allow_shared_override` field in a customer TOML is rejected by `deny_unknown_fields` as `E-CFG-010` before the Security Telemetry guard runs. The error message for ST+shared mode violations (AC-016) MUST NOT mention `allow_shared_override`. This keeps Wave 3 clean — no half-implemented escape hatch.
