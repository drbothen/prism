# S-3.11 — prism-query: In-Query Dedup Caching

## Narrative

As a Prism query engine, I want to detect when the same sensor source is requested
multiple times within a single query execution (e.g., a self-JOIN or multiple pipe
stages referencing the same table) and reuse the already-fetched `RecordBatch` so
that we never make duplicate API calls within one `execute()` invocation.

## Objective

Implement a per-query, ephemeral source cache scoped to a single `QueryEngine::
execute()` call. Before each sensor API fan-out, check if the same
`(sensor_id, client_id, table_name, push_down_params_hash)` was already fetched
in this query and reuse the `Arc<RecordBatch>` if so. The cache is dropped when
`execute()` returns. Wire cache hit/miss counters into `QueryMetrics` (S-3.09).

## Tasks

1. Define `CacheKey` in `crates/prism-query/src/cache_key.rs`:
   ```rust
   #[derive(Hash, Eq, PartialEq, Clone, Debug)]
   pub struct CacheKey {
       pub sensor_id: String,
       pub client_id: String,
       pub table_name: String,
       pub push_down_params_hash: u64,   // FxHash of serialized push-down params
   }
   ```
   - `push_down_params_hash`: hash of the serialized `PushDownParams` struct using
     `rustc-hash::FxHasher` (fast, deterministic within a process)
   - Same push-down params (same predicates, same time range, same required columns)
     → same hash → cache hit

2. Implement `InQueryCache` in `crates/prism-query/src/query_cache.rs`:
   ```rust
   pub struct InQueryCache {
       store: HashMap<CacheKey, Arc<Vec<RecordBatch>>>,
       pub hits: u32,
       pub misses: u32,
   }
   impl InQueryCache {
       pub fn lookup(&self, key: &CacheKey) -> Option<Arc<Vec<RecordBatch>>>;
       pub fn insert(&mut self, key: CacheKey, batches: Vec<RecordBatch>);
   }
   ```
   - `lookup()`: returns `Some(Arc<...>)` on hit (increments `hits`), `None` on miss
     (increments `misses`)
   - `insert()`: wraps `batches` in `Arc`, stores under `key`
   - No eviction needed — the cache lives for one `execute()` call only

3. Create `InQueryCache` at the top of `QueryEngine::execute()` in `engine.rs`:
   ```rust
   let mut query_cache = InQueryCache::new();
   ```
   Pass `&mut query_cache` into the fan-out orchestrator.

4. Modify fan-out in `fanout.rs` to check the cache before each API fetch:
   ```rust
   let cache_key = CacheKey::from(target, &push_down_params);
   if let Some(cached) = query_cache.lookup(&cache_key) {
       // Arc clone — zero copy, no API call
       results.push(FanOutResult::Cached { batches: cached, source: target });
   } else {
       let batches = adapter.fetch(...).await?;
       query_cache.insert(cache_key, batches.clone());
       results.push(FanOutResult::Fetched { batches, source: target });
   }
   ```
   Note: when the cache is hit, the `FanOutResult` must still carry the source
   metadata (`sensor_id`, `client_id`, `table_name`) for virtual field injection
   (`_sensor`, `_client`, `_source_table`) — the cache stores RecordBatches
   WITHOUT virtual fields already injected; injection happens after the fan-out
   phase (existing behavior, unchanged).

5. Wire `query_cache.hits` and `query_cache.misses` into `QueryMetrics` at the
   end of `execute()` before the metrics are finalized (S-3.09 adds these fields).
   If S-3.09 is not yet implemented, skip this step — the cache still works
   correctly without metrics.

6. Drop `InQueryCache` at end of `execute()` — it is a local variable on the stack,
   so this happens automatically when `execute()` returns or errors.

## Acceptance Criteria

**AC-1:** Given a SQL self-JOIN `SELECT a.device_ip, b.hostname FROM crowdstrike_hosts
a JOIN crowdstrike_hosts b ON a.device_ip = b.device_ip`, When the query executes,
Then `crowdstrike_hosts` is fetched from the sensor API exactly once (not twice), and
`cache_hits == 1` in `QueryMetrics`.

**AC-2:** Given a pipe query `FROM crowdstrike_hosts | join crowdstrike_hosts on
device_ip`, When the query executes, Then the sensor is called once and the second
reference is served from the cache.

**AC-3:** Given a query referencing the same table with DIFFERENT push-down predicates
(e.g., subquery with different WHERE clause), When the query executes, Then each
distinct set of push-down params results in a separate API fetch (cache miss) — the
cache does NOT return stale filtered results for a broader predicate.

**AC-4:** Given two different clients (`acme` and `globex`) both querying
`crowdstrike_hosts` in the same cross-client query, When the fan-out runs, Then each
client's fetch is treated as a separate cache entry — `(crowdstrike, acme,
crowdstrike_hosts, hash)` and `(crowdstrike, globex, crowdstrike_hosts, hash)` are
distinct keys.

**AC-5:** Given `execute()` completes (successfully or with error), When the function
returns, Then the `InQueryCache` is dropped and all `Arc<Vec<RecordBatch>>` reference
counts held by the cache are decremented — the memory is freed unless other
`Arc` clones still exist in the result set.

**AC-6:** Given a filter-mode query targeting a single table, When the query executes,
Then `cache_hits == 0` and `cache_misses == 1` (only one source, always a miss on
first fetch — expected behavior).

`prism-query` implements a within-query deduplication cache for repeated sub-expression
results (e.g., multiple joins against the same sensor table). Cache key computation is
pure; the `Arc<Mutex<HashMap>>` shared state is effectful. This is distinct from the
adapter response cache in S-3.05. Per `architecture/module-decomposition.md`, this
optimization lives inside `QueryEngine::execute()`.

## Architecture Compliance Rules

- The cache is EPHEMERAL — it MUST NOT persist beyond a single `execute()` call.
  Do NOT store `InQueryCache` in `QueryEngine` fields or any other long-lived struct.
- The cache stores `Arc<Vec<RecordBatch>>` WITHOUT virtual fields already applied.
  Virtual field injection (`_sensor`, `_client`, `_source_table`, `_source_type`)
  happens AFTER the fan-out phase — the cache must not pre-inject virtual fields,
  or they will be double-injected on the second reference.
- `CacheKey` hash MUST include `client_id` — cross-client queries must not share
  cached data between clients (data isolation requirement).
- `push_down_params_hash` MUST be deterministic within a process. Using
  `FxHasher` with stable serialization (not `HashMap` iteration order) is required.
- **Event-stream tables (S-2.08) bypass the in-query cache entirely.** They read from
  local RocksDB (~5ms), not remote APIs (~500ms), so caching provides no benefit.
  Check `table_type == TableType::PointInTime` before cache lookup; EventStream
  tables go directly to the buffer scan path.
  Use `serde_json::to_string()` of the `PushDownParams` struct (which sorts keys)
  as the hash input.
- Do NOT use `std::collections::HashMap`'s iteration order as a hash input source —
  it is randomized per process in Rust's default hasher.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-11-100 | Self-JOIN where both sides have identical push-down params | Sensor API called exactly once; second reference served from `InQueryCache` |
| EC-11-101 | Same table with different push-down predicates in same query | Two separate cache entries; two API calls (cache miss per distinct params) |
| EC-11-102 | Cross-client query: same table, different clients | `(sensor, clientA, table, hash)` and `(sensor, clientB, table, hash)` are distinct keys; two API calls |
| EC-11-103 | EventStream table referenced multiple times in one query | Cache bypassed entirely; RocksDB scan path used for each reference |
| EC-11-104 | `execute()` returns error mid-flight | `InQueryCache` dropped; all held `Arc<RecordBatch>` reference counts decremented |

## Dev Notes

- This optimization is most valuable for SQL JOINs where both sides of a self-JOIN
  target the same sensor table. In practice, security analysts often write queries
  like "find devices that appear in both CrowdStrike and Armis" which can be expressed
  as a JOIN on the same composite source.
- Pipe mode rarely benefits because pipe stages typically operate on a single source
  passed through the pipeline — a second `FROM` reference in a pipe `| join` stage
  would benefit.
- Filter mode never benefits (single source, single fetch by definition) but the
  cache overhead for a miss is minimal: one `HashMap::get()` and one `HashMap::insert()`.
- The `Arc<Vec<RecordBatch>>` clone on a cache hit is O(1) — it increments a reference
  count, not copying data. Memory usage is the same as a single fetch because the
  batches are shared, not duplicated.
