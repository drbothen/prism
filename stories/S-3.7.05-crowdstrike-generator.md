# S-3.7.05 — CrowdStrike fixture generator

## Narrative

As a test harness consumer, I want a `generate(org_id, SensorType::CrowdStrike, archetype, opts)`
implementation using the Rust types derived by S-3.7.00 from `gofalcon`, so that integration
tests exercise the 2-step IDs→detail pagination pattern, OAuth2 token responses, and all
8 archetypes without live API calls.

## Acceptance Criteria

### AC-001 — All 8 archetypes produce correct baseline counts for CrowdStrike at scale=1.0 (traces to BC-3.4.003 postcondition table)

`generate(org, CrowdStrike, archetype, GenOpts::default())` returns:
- `HealthyOtEnvironment`: 50 device records, 5 detection records; no containment
- `CompromisedEndpoint`: 50 device records, 20 detection records; ≥3 `severity_id >= 4`; ≥1 device with `containment_status = "contained"`
- `AuthOutage`: 20 device records; first OAuth2 response record has HTTP 401; recovery after 1 call (default)
- `LargeScale`: 10,000 device records, 500 detection records
- `PaginationEdgeCases`: `default_page_size(CrowdStrike) × 3` device records; 2-step pattern: 3 `IdPage` records + 3 detail pages
- `SchemaDrift`: 30 device records; `records[0]` violates CrowdStrike device schema; `provenance.schema_valid = false`
- `HighChurn`: 200 device records; ≥20 tombstones
- `DormantTenant`: 0 records; 0 cursors

(traces to BC-3.4.003 postcondition per archetype row)

### AC-002 — 2-step IDs→detail pagination pattern preserved (traces to BC-3.4.002 postcondition 1 — CrowdStrike row)

`PaginationEdgeCases` and `LargeScale` archetypes produce `FixtureSet` with two record
categories: `IdPage` records (list of device IDs, matching `IdPage` shape from
`.references/schemas/crowdstrike/types.rs`) and detail records (`FalconDevice`).
The ordering mirrors the real API: client fetches IDs first, then uses IDs to fetch details.
`FixtureSet::cursors` contains FQL offset cursors for the ID-list step.

### AC-003 — OAuth2 token response generated for AuthOutage archetype (traces to BC-3.4.002 postcondition 1 — CrowdStrike row)

`AuthOutage` archetype produces an `OAuth2TokenResponse` record (matching
`.references/schemas/crowdstrike/types.rs:OAuth2TokenResponse`) with `status_code = 401`
as the first record, followed by a valid `OAuth2TokenResponse` with a deterministic
`access_token` string on subsequent records. Token string format: `"tok-{org_slug}-{seed}-{call_n}"`.

### AC-004 — Generated records align with existing state.rs field names (traces to BC-3.4.002 postcondition 1 — CrowdStrike row)

Device `device_id` field and detection `detection_id` field match the key names used in
`crates/prism-dtu-crowdstrike/src/state.rs` (`containment_store`, `detection_status_store`).
No field name drift between generator output and DTU state machine.

### AC-005 — Org-tagged IDs for devices and detections (traces to BC-3.4.004 postcondition — CrowdStrike row)

Device records: `device_id` starts with `"dev-{org_slug}-{seed}-"`.
Detection records: `detection_id` starts with `"alert-{org_slug}-{seed}-"`.
Tombstone records: `"dev-{org_slug}-{seed}-tomb-{n}"`.

### AC-006 — Two identical calls are byte-identical (traces to BC-3.4.001 postcondition 1)

No `thread_rng` or `SystemTime::now` in call stack. CI static check + unit test confirm.

### AC-007 — OrgA and OrgB ID sets are disjoint (traces to BC-3.4.004 postcondition 3)

Proptest: `generate(orgA).ids ∩ generate(orgB).ids = ∅` for distinct slugs.

- Generator impl in `crates/prism-dtu-crowdstrike/src/generator.rs` only.

- `#[cfg(feature = "fixture-gen")]` gate mandatory on all generator code.
- Forbidden dependency: `prism-dtu-crowdstrike` generator must NOT depend on `prism-dtu-armis`, `prism-dtu-claroty`, or `prism-dtu-cyberint`.

## Risks / Open Questions

- The 2-step `IdPage` + detail pattern must be represented within a single `FixtureSet::records`
  slice — implementer must define a convention (e.g., tagged JSON objects with `"_record_type"`
  field) and document it in `generator.rs` comments. This convention must be consistent with
  how `prism-dtu-crowdstrike`'s DTU handler reads fixture data.
- `state.rs` field names are locked — if they diverge from `DERIVATION.md`, AC-004 test will
  catch it; the generator field names follow `state.rs`, not the Go SDK, as the authoritative source.
