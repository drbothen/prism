# S-1.06 — prism-credentials: Credential Store Trait and Backends

## Narrative

As a Prism platform implementer, I want a credential store abstraction with keyring
and encrypted-file backends in prism-credentials, so that sensor API credentials are
stored securely using the best available OS facility — with namespace isolation between
clients, path traversal rejection, and formally verified encryption correctness.

## Objective

Implement the `CredentialStore` trait and its two backends (keyring via keyring-rs 3.x,
encrypted file via AES-256-GCM). Implement backend selection logic, startup probe, and
credential namespace isolation. Two proptests (VP-034 encryption round-trip, VP-035 key
derivation determinism) formally verify the cryptographic properties. Credential CRUD
operations are deferred to S-1.07.

## Tasks

1. Create `crates/prism-credentials/Cargo.toml`:
   - Dependencies: `prism-core`, `keyring` (keyring-rs 3.x), `aes-gcm`, `argon2`
     (for key derivation), `rand`, `serde`, `serde_json`, `tokio`, `tracing`, `async-trait`
   - Dev dependencies: `proptest`, `tempfile`

2. Define `CredentialStore` trait in `crates/prism-credentials/src/trait_.rs`
   (BC-2.03.001):
   ```rust
   #[async_trait]
   pub trait CredentialStore: Send + Sync {
       async fn get(
           &self,
           tenant: &TenantId,
           sensor: &str,
           name: &CredentialName,
       ) -> Result<Option<SecretString>, PrismError>;

       async fn set(
           &self,
           tenant: &TenantId,
           sensor: &str,
           name: &CredentialName,
           value: SecretString,
       ) -> Result<(), PrismError>;

       async fn delete(
           &self,
           tenant: &TenantId,
           sensor: &str,
           name: &CredentialName,
       ) -> Result<bool, PrismError>; // Returns true if deleted, false if not found

       async fn list(
           &self,
           tenant: &TenantId,
       ) -> Result<Vec<(String, CredentialName)>, PrismError>; // Returns (sensor, name) pairs

       async fn exists(
           &self,
           tenant: &TenantId,
           sensor: &str,
           name: &CredentialName,
       ) -> Result<bool, PrismError>; // Used by S-5.05 config validation
   }
   ```
   - Use `secrecy::SecretString` for values — never pass credentials as plain `String`

3. Implement namespace key construction in `crates/prism-credentials/src/namespace.rs`
   (BC-2.03.004):
   - `fn namespace_key(tenant: &TenantId, sensor: &str, name: &CredentialName) -> String`
   - Format: `"{tenant}/{sensor}/{name}"` — e.g., `"acme/crowdstrike/api_key"`
   - This key is used as the keyring service name and the encrypted file entry key
   - All three components are validated before key construction (tenant via TenantId
     newtype, name via CredentialName newtype, sensor is validated non-empty ASCII)

4. Implement `KeyringBackend` in `crates/prism-credentials/src/keyring.rs`
   (BC-2.03.002):
   - Wraps keyring-rs 3.x `keyring::Entry`
   - `KeyringBackend::new(app_name: &str) -> Self` — `app_name` is `"prism"` always
   - For `get()`: create `Entry::new(app_name, &namespace_key(...))`, call `get_password()`
   - For `set()`: create entry, call `set_password(value.expose_secret())`
   - For `delete()`: create entry, call `delete_credential()`, map `NoEntry` to `Ok(false)`
   - For `list()`: not directly supported by keyring-rs — maintain a separate index
     in a `CredentialIndex` file (plaintext JSON list of namespace keys, no values)
   - Map keyring-rs errors to `PrismError::CredentialStoreError`

5. Implement `EncryptedFileBackend` in `crates/prism-credentials/src/file.rs`
   (BC-2.03.003):
   - Storage format: a directory per tenant, one file per credential entry
   - File contents: JSON envelope `{ "nonce": "<base64>", "ciphertext": "<base64>",
     "salt": "<base64>" }`
   - Encryption: AES-256-GCM with per-credential random nonce (12 bytes, via `rand`)
   - Key derivation: Argon2id from master passphrase + per-credential salt (16 bytes)
     - Argon2 params: m=65536 (64MB), t=3 iterations, p=1 lane
   - `EncryptedFileBackend::new(base_dir: PathBuf, passphrase: SecretString) -> Self`
   - For `get()`: read file, derive key from passphrase + file salt, decrypt AES-GCM
   - For `set()`: generate fresh nonce + salt, derive key, encrypt, write atomically
     (write to `{name}.tmp`, rename to `{name}`)
   - For `delete()`: remove the file
   - For `list()`: scan the tenant directory for files (VP-034, VP-035)

6. Implement `CredentialIndex` helper in `crates/prism-credentials/src/index.rs`:
   - Used by `KeyringBackend` for `list()` because keyring-rs has no enumeration API
   - Stores a `Vec<NamespaceKey>` in a plaintext JSON file at a fixed path
   - File contains no credential values — only keys (safe to store unencrypted)
   - Operations: `add(key)`, `remove(key)`, `list() -> Vec<NamespaceKey>`
   - Atomic writes (tmp + rename pattern)

7. Implement startup probe in `crates/prism-credentials/src/probe.rs`
   (BC-2.03.011):
   - `async fn probe_keyring(app_name: &str) -> KeyringStatus`:
     - Write a test value: `Entry::new(app_name, "prism_probe")`, `set_password("ok")`
     - Read it back: verify value matches
     - Delete it: cleanup
     - Return `KeyringStatus::Available` or `KeyringStatus::Unavailable(reason)`
   - Log at `WARN` level if unavailable, `INFO` if available
   - Never panic — unavailability is handled gracefully

8. Implement `BackendSelector` in `crates/prism-credentials/src/selector.rs`
   (BC-2.03.012):
   - `async fn select_backend(config: &CredentialConfig) -> Box<dyn CredentialStore>`:
     - If `config.backend == "keyring"` (or `"auto"`): run probe; use keyring if available
     - If keyring unavailable and `backend == "auto"`: log warning, use encrypted file
     - If `config.backend == "file"`: use encrypted file directly, skip probe
     - If `config.backend == "keyring"` but probe fails: return `Err(CredentialStoreError)`
       (explicit keyring selection with unavailable keyring is a hard error)
   - `CredentialConfig` fields: `backend: String`, `file_path: Option<PathBuf>`,
     `passphrase_env: Option<String>` (env var name for passphrase, never stored inline)

9. Write VP-034 proptest in `crates/prism-credentials/src/tests/proptest_crypto.rs`:
   - Strategy: generate arbitrary byte sequences as plaintext (up to 4096 bytes)
   - For each: encrypt with `EncryptedFileBackend`'s internal encrypt function, then
     decrypt with the same key
   - Assert: decrypted output equals original plaintext (BC-2.03.003 round-trip)
   - Property name: `prop_encrypt_decrypt_round_trip`

10. Write VP-035 proptest in `crates/prism-credentials/src/tests/proptest_crypto.rs`:
    - Strategy: generate arbitrary passphrases (non-empty UTF-8) and fixed salts
    - For each: derive key twice with the same passphrase + salt
    - Assert: both derived keys are identical (key derivation is deterministic)
    - Property name: `prop_key_derivation_deterministic`

11. Write unit tests in `crates/prism-credentials/src/tests/`:
    - Test `namespace_key()` produces the expected `"{tenant}/{sensor}/{name}"` format
    - Test `EncryptedFileBackend` set → get round-trip (using tempdir)
    - Test `KeyringBackend` when keyring unavailable falls through (mock keyring-rs)
    - Test `BackendSelector` with `backend = "auto"` and unavailable keyring → file backend
    - Test `CredentialIndex` add, remove, list operations

## Acceptance Criteria

**AC-1:** Given a `CredentialStore` backed by `KeyringBackend`, When
`set("acme", "crowdstrike", "api_key", "secret123")` is called followed by
`get("acme", "crowdstrike", "api_key")`, Then it returns `Some("secret123")`.
(BC-2.03.001, BC-2.03.002)

**AC-2:** Given keyring is unavailable (mocked), When `BackendSelector` selects with
`backend = "auto"`, Then it returns an `EncryptedFileBackend` instance and logs a
`WARN` message. (BC-2.03.012)

**AC-3:** Given an `EncryptedFileBackend`, When a credential is stored and retrieved,
Then the value matches the original (AES-256-GCM round-trip). (BC-2.03.003)

**AC-4:** Given tenant `"acme"` stores `"crowdstrike"/"api_key"`, When
`get("beta", "crowdstrike", "api_key")` is called, Then it returns `None` — namespace
isolation prevents cross-client access. (BC-2.03.004)

**AC-5:** Given a `CredentialName` input of `"../../etc/passwd"`, When `set()` is
called, Then it returns `Err(PrismError::InvalidCredentialName)` before any storage
operation is attempted. (BC-2.03.008)

**AC-6:** Given a fresh system, When the startup probe runs, Then it correctly
detects keyring availability and logs the result. (BC-2.03.011)

**AC-7:** Given `BackendSelector` with `backend = "keyring"` and the keyring probe
fails, When `select_backend()` is called, Then it returns `Err` (explicit keyring
selection is a hard requirement, not silently downgraded). (BC-2.03.012)

**AC-8:** VP-034 proptest passes: AES-256-GCM encrypt → decrypt returns original value
for arbitrary byte sequences.

**AC-9:** VP-035 proptest passes: same passphrase + salt always produces the same
derived key across 10,000+ generated cases.

**AC-10:** Given `list("acme")` is called after storing three credentials for
`"acme"`, Then it returns exactly 3 `(sensor, name)` pairs.

## Architecture Compliance Rules

- Credential values MUST use `secrecy::SecretString` — never `String` or `&str` for
  plaintext credentials in memory
- `passphrase` for `EncryptedFileBackend` MUST come from env var or OS keyring — never
  from `prism.toml` config file (config files may be committed to VCS)
- Keyring operations MUST be wrapped in `tokio::task::spawn_blocking` — keyring-rs
  is synchronous and must not block the async runtime
- Atomic writes (tmp + rename) are REQUIRED for `EncryptedFileBackend` — partial writes
  must not corrupt existing credentials
- `CredentialStore` trait MUST be `Send + Sync` (used from async tokio tasks)

## Notes

- `keyring-rs` 3.x changed its API significantly from 2.x. Use `keyring::Entry::new(service, user)`
  where `service = app_name` and `user = namespace_key`. The service/user split in
  keyring-rs maps to the platform's credential namespace.
- `spawn_blocking` wrapping keyring calls is non-negotiable. The macOS keychain
  can prompt for password confirmation, which blocks the thread. Wrapping in
  `spawn_blocking` isolates this to a dedicated thread pool.
- The `CredentialIndex` file (used by KeyringBackend for `list()`) is NOT a security
  boundary — it contains only namespace keys, not values. It can be stored alongside
  the credential store configuration directory.
- Argon2 parameters (m=65536, t=3, p=1) are intentionally conservative for a CLI
  tool where derivation happens infrequently (not on every API call). Key caching in
  memory is acceptable for the session lifetime but is deferred to S-1.07.
- The `passphrase_env` field in `CredentialConfig` stores the NAME of an env var,
  not the passphrase itself. `BackendSelector` calls `std::env::var(passphrase_env)`
  at runtime to retrieve the actual value.
