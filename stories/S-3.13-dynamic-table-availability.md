# S-3.13 — prism-query: Dynamic Table Availability

## Narrative

As a Prism analyst, I want the query engine to only expose tables for sensors that
are currently configured and loaded so that my query workspace reflects reality —
unloaded sensors produce helpful errors with suggestions, not cryptic DataFusion
schema failures, and adding or removing a sensor spec file immediately changes
which tables are available.

## Objective

Implement dynamic DataFusion table registration driven by the live `ConfigSnapshot`.
When a sensor spec is loaded, its tables are registered in the DataFusion schema.
When a spec is unloaded or removed, its tables are deregistered. Queries against
unregistered tables return `E-QUERY-001` with a `did you mean` suggestion listing
configured sensors. Hot reload adds and removes tables atomically within the
config swap. Per osquery's dynamic table availability pattern.

## Tasks

1. Implement `TableRegistry` in `crates/prism-query/src/table_registry.rs`:
   ```rust
   pub struct TableRegistry {
       registered: Arc<RwLock<HashSet<String>>>,  // table names currently available
       schema_by_table: Arc<RwLock<HashMap<String, SchemaRef>>>,
   }
   impl TableRegistry {
       pub fn register_sensor(&self, spec: &SensorSpec) -> Result<()>;
       pub fn deregister_sensor(&self, sensor_id: &str) -> Result<()>;
       pub fn is_registered(&self, table_name: &str) -> bool;
       pub fn registered_tables(&self) -> Vec<String>;
       pub fn sensor_for_table(&self, table_name: &str) -> Option<String>;
   }
   ```
   - `register_sensor()`: for each `[[sensor.tables]]` entry in the spec, add
     `{sensor_id}_{table_name}` to `registered` and store its Arrow schema
   - `deregister_sensor()`: remove all tables whose names start with `{sensor_id}_`
   - Thread-safe: `Arc<RwLock>` on both collections; reads are non-blocking in normal
     operation

2. Initialize `TableRegistry` on `QueryEngine` construction:
   - Load initial set from the `ConfigSnapshot` at startup
   - For each sensor spec in the snapshot: call `register_sensor()`
   - Store `TableRegistry` as `Arc<TableRegistry>` in `QueryEngine`

3. Integrate `TableRegistry` with the hot-reload path (S-1.12):
   - After a successful config swap, compute the delta:
     - New sensors in `new_config` not in `old_config`: call `register_sensor()`
     - Sensors in `old_config` not in `new_config`: call `deregister_sensor()`
     - Sensors in both: check if spec changed; if so, deregister + re-register
   - Send MCP `notifications/tools/list_changed` if the registered table set changed
   - In-flight queries that started before deregistration: they hold an
     `Arc<ConfigSnapshot>` reference (CI-007); their `SensorAdapter` calls proceed
     against the old spec. Only new queries see the updated table set.

4. Validate table references against `TableRegistry` during query planning in
   `planner.rs`:
   - After parsing, for each `source_ref` in the `QueryPlan`: check
     `table_registry.is_registered(source_ref)`
   - If not registered: return `Err(PrismError::QueryError(E-QUERY-001 { ... }))`
     BEFORE the fan-out phase (fail fast)
   - Error message format:
     ```
     Table 'armis_alerts' is not available. Sensor 'armis' is not configured.
     Available sensors: [crowdstrike, claroty, cyberint]
     Available tables: [crowdstrike_alerts, crowdstrike_hosts, claroty_devices, ...]
     ```
   - Use edit-distance comparison (Levenshtein ≤ 3) between the requested table name
     and all registered tables to produce a "Did you mean: crowdstrike_alerts?" hint

5. Update `explain_query` to list only registered tables:
   - `"available_tables"` in the explain response MUST reflect
     `table_registry.registered_tables()` — not a hardcoded list
   - Call `registered_tables()` at explain time (not cached in the plan)

6. Update `prism://config/clients` MCP resource to reflect only loaded sensors:
   - The resource handler reads `table_registry.registered_tables()` grouped by
     sensor_id to build the sensor list
   - On hot-reload that changes the table set, the `notifications/resources/list_changed`
     MCP notification causes the host to re-read the resource

7. Update `prism://diagnostics/summary` to include unavailable sensor warnings:
   - If a query was attempted against an unregistered table in the last hour:
     include a summary entry `"unregistered_table_queries": [{"table": "armis_alerts",
     "count": 3, "last_attempted": "10:05:00"}]`

## Acceptance Criteria

**AC-1:** Given `armis.sensor.toml` is loaded and `crowdstrike.sensor.toml` is NOT
loaded, When `table_registry.is_registered("crowdstrike_alerts")` is called, Then
it returns `false`. When `table_registry.is_registered("armis_alerts")` is called,
Then it returns `true`.

**AC-2:** Given a query `SELECT * FROM crowdstrike_alerts` when CrowdStrike is not
configured, When the query is planned, Then it returns `E-QUERY-001` with the message
"Sensor 'crowdstrike' is not configured. Available sensors: [armis, claroty]" —
before any fan-out occurs.

**AC-3:** Given a user types `SELECT * FROM crowdstrike_alert` (missing 's'), When
the error is returned, Then it includes `"did_you_mean": "crowdstrike_alerts"`.

**AC-4:** Given `armis.sensor.toml` is added via hot-reload (file drop or
`add_sensor_spec` MCP call), When the config swap completes, Then
`table_registry.is_registered("armis_alerts")` returns `true` and a
`notifications/resources/list_changed` MCP notification is sent.

**AC-5:** Given `claroty.sensor.toml` is removed via hot-reload, When the config
swap completes, Then `table_registry.is_registered("claroty_devices")` returns
`false`. In-flight queries that started before the removal complete normally.

**AC-6:** Given `explain_query` is called, When the response is returned, Then
`"available_tables"` lists only the currently registered tables — not a static list.

**AC-7:** Given `prism://config/clients` MCP resource is read, When only CrowdStrike and
Claroty are configured, Then the resource lists exactly those two sensors and their
tables — not Armis or Cyberint.

**AC-8:** Given a SQL query in SQL mode, filter mode, and pipe mode each reference
an unregistered table, When each is planned, Then all three return `E-QUERY-001` —
the validation is mode-agnostic, applied at the plan level.

## Architecture Compliance Rules

- Table availability check MUST happen at plan time (before fan-out), not at parse
  time. The parser produces a syntactically valid AST regardless of whether tables
  exist; the planner validates semantic correctness against the live registry.
- `TableRegistry` reads during query planning use `RwLock::read()` (non-exclusive) —
  they MUST NOT block query execution. `RwLock::write()` is only held during
  registration/deregistration which happen on the config-reload path, not the
  query path.
- The edit-distance suggestion (Levenshtein ≤ 3) MUST be computed at error time
  from `table_registry.registered_tables()` — not a static list. This ensures
  suggestions are always accurate even as sensors are added/removed.
- In-flight query isolation (CI-007): a query that started before a deregistration
  holds a reference to the old `ConfigSnapshot` and the old `AdapterRegistry`. It
  MUST NOT be terminated mid-execution. Only new queries see the updated table set.
- `notifications/resources/list_changed` is sent only when the table set actually
  changes — not on every hot-reload. Use set comparison before/after the swap.

## Dev Notes

- The osquery pattern: osquery only registers tables for extensions that are currently
  running. If an extension crashes, its tables disappear from the schema. Prism's
  equivalent is sensor specs — if a spec is removed, its tables disappear from the
  DataFusion namespace.
- The error message with sensor suggestions is particularly important for AI-generated
  queries. When Claude generates a query against a sensor the analyst hasn't configured,
  the error message tells Claude exactly which sensors ARE available — enabling it to
  self-correct without human intervention.
- Edit-distance suggestions: compute Levenshtein distance between the requested name
  and all registered table names; return the top 1 match if distance ≤ 3. The
  `edit-distance` crate (pure Rust, no unsafe) is suitable for this. Avoid bringing
  in a full fuzzy-search library for this simple use case.
- Hot-reload table delta: the simplest correct implementation is deregister-all-for-sensor
  then re-register. This handles schema changes within a spec (e.g., new table added
  to an existing sensor) without needing fine-grained diff logic.
