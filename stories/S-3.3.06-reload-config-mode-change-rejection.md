# S-3.3.06 — prism-spec-engine: reload_config detects and warns on DTU mode changes without applying them

## Narrative

As a Prism MSSP operator, I want the `reload_config` MCP tool to detect when a customer
config file has changed the `mode` field of a `[[dtu]]` block and emit a structured
warning explaining that mode changes require a process restart, so that I receive
actionable feedback without accidentally promoting an untested mode change into the
running process.

## Objective

Extend the `reload_config` handler in `prism-spec-engine` to compare the `mode` field
of each `[[dtu]]` block in the new config against the currently-active `DtuMode` stored
in the `OrgRegistry`/`SensorSpec`. When a mode discrepancy is detected:
1. Emit a `WARN`-level structured tracing event naming the affected org, DTU type, old
   mode, and new mode
2. Emit an audit warning entry for the mode-change detection event
3. Return `ReloadStatus::Ok` (or `PartialReload`) but with the old mode preserved — the
   new mode is silently dropped from the applied config

This implements BC-3.2.005 invariant 4 and EC-006: "reload_config detects the mode
change, emits a warning that mode changes require restart, but does not apply the change;
running mode is preserved."

## Tasks

1. **Locate mode fields in new config diff** (`src/reload_config.rs`):
   - After `diff_snapshots` produces `modified: Vec<ModifiedSpec>`, inspect each modified
     spec for a change in the `mode` field.
   - The existing `ModifiedSpec` struct (or the `diff_snapshots` utility) may need a new
     field or helper: `mode_changed: Option<ModeChange>` where
     `ModeChange { old: DtuMode, new: DtuMode, org_slug: String, dtu_type: String }`.
   - If `ModifiedSpec` cannot be extended without breaking the existing API, add a
     parallel `mode_change_warnings: Vec<ModeChange>` vec alongside it.

2. **Emit structured tracing warning** for each mode change detected:
   ```rust
   tracing::warn!(
       org_slug = %change.org_slug,
       dtu_type = %change.dtu_type,
       old_mode = ?change.old,
       new_mode = ?change.new,
       "DTU mode change detected in reload — restart required to apply; \
        running mode preserved"
   );
   ```

3. **Emit audit warning** via the audit trail (use whatever audit emission mechanism
   exists in `prism-spec-engine` post-S-3.1.07 / S-3.3.02; if not available yet, emit a
   `tracing::warn!` event with `audit = true` field and document the TODO to wire the
   audit entry in a follow-on story):
   - Event fields: `event_type: "dtu_mode_change_rejected"`, `org_slug`, `dtu_type`,
     `old_mode`, `new_mode`, `config_file`, `timestamp`

4. **No-apply guarantee**: the `reload_config` function MUST NOT update the `DtuMode`
   for any sensor spec registration even when mode-change detection triggers. The
   `ArcSwap` atomic store of the new `ConfigSnapshot` proceeds normally, but the applied
   snapshot's `DtuMode` values are taken from the OLD snapshot for any org/DTU pair that
   had a mode change. Concretely:
   - When applying a new snapshot with mode changes, preserve the old mode in the merged
     config (write the new config with old modes patched in, or rebuild the sensor spec
     registrations from the old snapshot for changed entries).
   - The simplest implementation: after detecting mode changes, reconstruct the
     `candidate` snapshot so that mode-changed entries carry the old `DtuMode` value.

5. **Wire VP-094 integration test** at
   `crates/prism-spec-engine/tests/hot_reload_tests.rs`:
   - Test name: `test_reload_config_mode_change_warns_without_applying`
   - Setup: write a valid customer TOML with `mode = "client"` for a claroty DTU;
     call `reload_config`; confirm mode is `DtuMode::Client`.
   - Edit: change `mode = "shared"` in the TOML file.
   - Act: call `reload_config` again.
   - Assert: `ReloadResult.status` is `ReloadStatus::Ok` (not an error);
     `reload_result.mode_change_warnings` (or equivalent) is non-empty with the
     correct org_slug and DTU type; the active `DtuMode` for the claroty DTU remains
     `DtuMode::Client`.
   - Use `tracing_test` or subscriber capture to assert the `WARN` event was emitted.

6. **Unit test**: `test_reload_config_no_mode_change_no_warning` — reload with identical
   mode values produces no mode_change_warnings.

7. **Gate**: `cargo test -p prism-spec-engine` green.

## Acceptance Criteria

**AC-001** — Mode-change detection (traces to BC-3.2.005 invariant 4):
Given a `reload_config` call where the new customer TOML has changed the `mode` field
of a `[[dtu]]` block from `"client"` to `"shared"` (or vice versa),
When `reload_config` returns,
Then `ReloadResult` contains a non-empty mode-change warning collection naming the
affected org slug, DTU type, old mode, and new mode.

**AC-002** — No-apply guarantee (traces to BC-3.2.005 invariant 4 + EC-006):
Given a `reload_config` call where the new config has a mode change,
When `reload_config` completes successfully,
Then the active `DtuMode` for the affected sensor registration is unchanged from the
pre-reload value; the process continues with the original mode.

**AC-003** — Warning emission (traces to BC-3.2.005 invariant 4 + EC-006):
Given a mode change is detected during `reload_config`,
When the handler processes the change,
Then a `WARN`-level structured tracing event is emitted with fields
`org_slug`, `dtu_type`, `old_mode`, `new_mode`, and the message
"DTU mode change detected in reload — restart required to apply".

**AC-004** — Audit trail entry (traces to BC-3.2.005 invariant 4):
Given a mode change is detected during `reload_config`,
When the handler processes the change,
Then an audit warning event of type `"dtu_mode_change_rejected"` is emitted (or a
tracing event with `audit = true` if the audit plumbing is not yet wired in this crate)
containing the config file path, org slug, DTU type, old mode, and new mode.

**AC-005** — Normal reload proceeds (traces to BC-3.2.005 postcondition 5):
Given a `reload_config` call that detects mode changes,
When the call completes,
Then non-mode-changed fields (e.g., `data.scale`, `display_name`) from the new config
ARE applied normally; only the `mode` field change is suppressed.

**AC-006** — No mode-change warning on unchanged mode (traces to BC-3.2.005 invariant 4
negative path):
Given a `reload_config` call where the new config has the same `mode` values as the
currently active config,
When the call completes,
Then no mode-change warnings are produced.

### Subsystem Anchor Justification

SS-06 (Client Configuration) owns this story because `reload_config` is part of the
config-management subsystem in `prism-spec-engine`. The mode-immutability enforcement is
a config-lifecycle concern, not a sensor-adapter concern.

### Dependency Anchor Justification

`depends_on: [S-3.3.02]` — OrgRegistry boot (S-3.3.02) establishes the `DtuMode` per-org
storage model at startup. This story extends the reload path to detect deviations from that
startup-established mode. Without S-3.3.02, there is no canonical mode to compare against.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Mode change from `"shared"` to `"client"` (MSSP Coordination type) | Warning emitted; old mode preserved; no startup error (not a startup path) |
| EC-002 | Mode change on a Security Telemetry type from `"client"` to `"shared"` | Warning emitted (double protection: startup guard also blocks this); old `DtuMode::Client` preserved |
| EC-003 | Multiple `[[dtu]]` blocks in one org config all have mode changes | One warning per changed block; all old modes preserved |
| EC-004 | `reload_config` called with `dry_run = true` and mode changes | Mode-change warnings are included in the dry-run result but not emitted as tracing events (no side effects in dry-run) |
| EC-005 | Config file deleted and recreated with mode change during reload | If `diff_snapshots` sees it as a `removed + added` rather than `modified`, mode-change detection must handle both paths |
