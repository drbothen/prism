# S-3.7.02 — Claroty fixture generator

## Narrative

As a test harness consumer, I want a `generate(org_id, SensorType::Claroty, archetype, opts)`
implementation backed by the poller-bear OpenAPI spec, so that integration tests can run
realistic Claroty fixture data for all 8 archetypes without hitting a live API.

## Acceptance Criteria

### AC-001 — All 8 archetypes produce correct baseline counts at scale=1.0 (traces to BC-3.4.003 postcondition table)

`generate(org, Claroty, archetype, GenOpts::default())` returns:
- `HealthyOtEnvironment`: 50 device records + 5 alert records
- `CompromisedEndpoint`: 50 device records + 20 alert records, ≥3 with `severity_id >= 4`
- `AuthOutage`: 20 device records; first simulated call record has `status_code = 401`
- `LargeScale`: 10,000 device records + 500 alert records across ≥100 distinct subnets
- `PaginationEdgeCases`: `default_page_size(Claroty) × 3` device records; 3 cursor values
- `SchemaDrift`: 30 device records; `records[0]` fails Claroty OpenAPI schema; `provenance.schema_valid = false`
- `HighChurn`: 200 device records; ≥20 with `status = "tombstone"`
- `DormantTenant`: 0 records; 0 cursors

(traces to BC-3.4.003 postconditions per archetype row)

### AC-002 — Non-SchemaDrift records pass `.references/poller-bear/docs/specs.json` validation (traces to BC-3.4.002 postcondition 1)

In `#[cfg(test)]`, every record in the `FixtureSet` for all archetypes except `SchemaDrift`
validates against the Claroty OpenAPI spec at `.references/poller-bear/docs/specs.json`.
A schema violation causes a test panic identifying sensor type, archetype, record index,
and schema error (BC-3.4.002 postcondition 2).

### AC-003 — SchemaDrift archetype produces schema_valid=false (traces to BC-3.4.002 postcondition for SchemaDrift)

`generate(org, Claroty, SchemaDrift, opts).provenance.schema_valid == false`.
`records[0]` intentionally violates the Claroty spec (missing required field, wrong
type, or extra unknown field). Remaining 29 records are schema-valid.

### AC-004 — All record IDs carry org-slug prefix (traces to BC-3.4.004 postcondition 1)

Every device record has `device_id` starting with `"dev-{org_slug}-{seed}-"`.
Every alert record has `alert_id` starting with `"alert-{org_slug}-{seed}-"`.
Tombstone records follow `"dev-{org_slug}-{seed}-tomb-{n}"` pattern (BC-3.4.004 postconditions 1–3).

### AC-005 — Two calls with identical inputs produce byte-identical records (traces to BC-3.4.001 postcondition 1)

`generate(orgA, Claroty, HealthyOtEnvironment, GenOpts::default())` called twice in the
same process returns `records` that are byte-identical (JSON serialization match).
No `thread_rng` or `SystemTime::now` call exists in the generator call stack (BC-3.4.001 invariant 1).

### AC-006 — OrgA and OrgB ID sets are disjoint (traces to BC-3.4.004 postcondition 3)

`generate(orgA, ...).records.ids ∩ generate(orgB, ...).records.ids = ∅`
when `orgA.slug ≠ orgB.slug` (verified by proptest in test suite).

### AC-007 — Schema validation absent from release build (traces to BC-3.4.002 invariant 4)

`#[cfg(test)]` gate confirmed: `cargo build --release` produces no schema-validation
symbols. CI grep check added.

## Tasks

1. Read `.references/poller-bear/docs/specs.json` — identify device, alert, and tag
   response schema shapes. Map JSON Schema paths to `device_id`, `alert_id` field names.
2. Implement `generate(org_id, archetype, opts: GenOpts) -> FixtureSet` in
   `crates/prism-dtu-claroty/src/generator.rs` behind `#[cfg(feature = "fixture-gen")]`.
3. Use `seeded_rng(opts.seed, org_id)` from S-3.7.01 for all randomness.
4. Implement each of the 8 archetype branches using `GenOpts::scale` for record counts.
5. Apply org-tagged ID format per BC-3.4.004: `device_id = format!("dev-{slug}-{seed}-{i}")`.
6. Implement `SchemaDrift` — mutate `records[0]` to remove a required field.
7. Set `provenance.schema_valid = false` for `SchemaDrift`; `true` for all others.
8. In `#[cfg(test)]`, add schema validation: load spec from `.references/poller-bear/docs/specs.json`,
   validate all non-SchemaDrift records; assert `schema_valid == false` for SchemaDrift.
9. Write parameterised integration test over all 8 archetypes verifying count, ID format, and schema.

## Previous Story Intelligence

S-3.7.01 defines `GenOpts`, `seeded_rng`, `FixtureSet`, `Provenance`, and `default_page_size`.
Read `crates/prism-dtu-common/src/generator/` before implementing to avoid re-inventing types.
Specifically: `seeded_rng` signature, `apply_overrides` for `AuthOutage` recovery override.

## Architecture Compliance Rules

- Generator impl lives in `crates/prism-dtu-claroty/src/generator.rs` — NOT in `prism-dtu-common`.
- Schema source is `.references/poller-bear/docs/specs.json` — treated as read-only (BC-3.4.002 invariant 3).
- `#[cfg(feature = "fixture-gen")]` gate is mandatory on all generator code.
- Forbidden dependency: `prism-dtu-claroty` must NOT depend on `prism-dtu-armis`, `prism-dtu-crowdstrike`, or `prism-dtu-cyberint`.

## Library & Framework Requirements

- `jsonschema` crate (dev-dependency, `#[cfg(test)]`) — version from workspace pin
- `serde_json` (workspace pin) — for record construction and schema validation
- `rand_chacha` via `prism-dtu-common::generator::seeded_rng` — do not re-import directly
