# S-3.03 — prism-query: Explain and Query Diagnostics

## Narrative

As an analyst or LLM agent, I want an `explain_query` tool that shows me the execution
plan for a PrismQL query — what sources will be fetched, which predicates will be pushed
down to sensor APIs, the DataFusion logical plan, and the memory budget — without
actually executing the query, so that I can diagnose performance issues and validate
query correctness before running expensive fan-out operations.

## Objective

Implement the `explain_query` MCP tool: parse the PrismQL string, resolve sources,
classify push-down predicates, generate the DataFusion logical plan, estimate memory
requirements, and return a structured JSON `ExplainResult` — all without issuing any
sensor API calls.

## Tasks

1. Implement `explain(query_str: &str, options: ExplainOptions) -> Result<ExplainResult, PrismError>`
   in `crates/prism-query/src/explain.rs` (BC-2.11.010):
   - `ExplainOptions` carries `clients: Option<Vec<OrgSlug>>`, `sensors: Option<Vec<SensorType>>`,
     `sources: Option<Vec<String>>`, `alias_registry: HashMap<String, String>`,
     `client_registry: Option<Arc<ClientRegistry>>`, and `audit_sink`
   - Parse `query_str` via `PrismQlParser::parse()` — return parse errors immediately without
     attempting further analysis
   - Resolve source refs to sensor specs (same logic as S-3.02 materialization, but no
     fan-out)
   - Resolve client scope via `resolve_clients()` from `scoping.rs` (no fan-out);
     populate `ExecutionPlan.clients_to_query` with the resolved list (AC-5)
   - Classify predicates via `classify_predicates()` from `pushdown.rs` for each source
   - DataFusion plan elision (I-LOCAL-001): use AST-derived `post_fetch_operations` list
     in place of a DataFusion `LogicalPlan`. Full DataFusion plan visibility deferred to
     TD-S303-DATAFUSION-PLAN-001. The BC postcondition is satisfied by the AST-derived list.
   - Collect results into `ExplainResult`

2. Define `ExplainResult` struct in `explain.rs`:
   ```
   ExplainResult {
     parsed_mode: String,            // "filter", "sql", or "pipe"
     original_query: String,         // raw query string as provided
     alias_expansion: HashMap<String, String>,
     expanded_query: String,
     field_resolution: HashMap<String, FieldResolution>,
     execution_plan: ExecutionPlan,
     estimated_cost: CostEstimate,
   }

   ExecutionPlan {
     sensors_to_query: Vec<ExplainSource>,
     post_fetch_operations: Vec<String>,   // AST-derived (DataFusion plan deferred: TD-S303-DATAFUSION-PLAN-001)
     clients_to_query: Vec<OrgSlug>,       // AC-5: all clients in scope (pure registry lookup)
   }

   ExplainSource {
     source_ref: String,             // e.g. "crowdstrike.detections"
     sensor_type: SensorType,
     api_filters_pushed: Vec<String>,      // predicates sent to sensor API
     post_filter_predicates: Vec<String>, // predicates applied after fetch
     estimated_row_count: Option<u64>,    // None if sensor doesn't support row count hints
   }
   ```

3. Serialize `ExplainResult` to JSON for MCP tool response. Use `serde_json::to_string_pretty`
   so that LLM consumers receive readable output.

4. Write unit tests in `crates/prism-query/src/tests/explain_tests.rs`:
   - Test: valid query → correct sources and push-down decisions
   - Test: query with push-downable predicates → api_filters_pushed populated
   - Test: invalid query → parse errors returned, no ExplainResult
   - Test: multi-client query → all clients listed in execution_plan.clients_to_query

## Acceptance Criteria

**AC-1:** Given a valid PrismQL query, When `explain_query` is called, Then an
`ExplainResult` is returned without issuing any sensor API calls (verified by asserting
zero calls to `AdapterRegistry`). (BC-2.11.010)

**AC-2:** Given a query with `WHERE severity_id >= 3` where `severity_id` is a REQUIRED
push-downable column, When explained, Then `ExplainSource.api_filters_pushed` contains
`"severity_id >= 3"` for the relevant source. (BC-2.11.010)

**AC-3:** Given a query targeting two sources (`crowdstrike.detections` and
`claroty.alerts`), When explained, Then `execution_plan.sensors_to_query` contains two
`ExplainSource` entries, one per source. (BC-2.11.010)

**AC-4:** Given a syntactically invalid PrismQL query, When `explain_query` is called,
Then a structured parse error is returned (not a panic), and `ExplainResult` is not
produced. (BC-2.11.010)

**AC-5:** Given a query with `clients: None`, When explained, Then `execution_plan.clients_to_query`
lists all configured client IDs without any sensor fan-out occurring. (BC-2.11.010)

## Architecture Compliance Rules

- `explain_query` MUST NOT call `fan_out()` or any sensor adapter `fetch()` method —
  it is a read-only plan analysis operation; calling sensors would defeat its purpose
  and incur unnecessary API costs
- `ExplainResult` MUST be serializable to JSON without any custom serializer —
  all fields use standard types (`String`, `Vec<String>`, `u64`, `Option`)
- The DataFusion logical plan is obtained against empty (schema-only) MemTables —
  do not register actual data; this keeps explain cheap and deterministic
- `explain_query` reuses `classify_predicates()` from S-3.02 pushdown.rs directly —
  do NOT duplicate the push-down classification logic

## Dev Notes

- `estimated_memory_mb` is a heuristic: assume ~1MB per 1,000 rows × number of sources ×
  number of clients. This is rough guidance for analysts, not a guarantee.
- If a sensor adapter exposes a `count_hint()` method (not required in S-2.06), use it
  to populate `ExplainSource.estimated_row_count`. Otherwise leave it `None`.
- The `datafusion_logical_plan` string comes from DataFusion's `LogicalPlan::display_indent()`
  — it is already human-readable and needs no further formatting.
- This story intentionally has no Kani proofs or fuzz targets: the explain path is
  nearly identical to the execute path (which has VP-031) and the parser is already
  covered by VP-014, VP-015, VP-021.
