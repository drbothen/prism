# S-3.01 â€” prism-query: PrismQL Parser (Filter + SQL + Pipe)

## Narrative

As a Prism query engine, I want a complete PrismQL parser that handles all three query
modes (filter, SQL, and pipe), so that analysts and automation can express queries in
the syntax most natural to their use case without requiring separate parsing paths.

## Objective

Extend the existing `prism-query` crate (which already contains S-2.08 virtual-field
injection code and S-3.2.08 CrowdStrike session-ID scoping code) by adding PrismQL
parser modules using Chumsky 0.12. The new parser supports all three query modes:
filter mode (`source | predicate`), SQL mode (full SELECT/JOIN/subquery), and pipe mode
(`source | stage | stage`). Include security limits for query size and nesting depth,
error recovery for partial inputs, and Kani/fuzz verification properties. Do NOT
overwrite existing modules (`types.rs`, `materialization.rs`, `org_scoped_session_id.rs`)
or the existing `Cargo.toml` dep list â€” merge new deps alongside existing ones.

4. Define PrismQL AST types in `crates/prism-query/src/ast.rs`:
   - `FilterExpr { source: SourceRef, predicate: Predicate }`
   - `SqlQuery { select: SelectClause, from: FromClause, joins: Vec<Join>, where_: Option<Expr>,
     group_by: Vec<Expr>, having: Option<Expr>, order_by: Vec<OrderExpr>, limit: Option<u64> }`
   - `PipeQuery { source: SourceRef, stages: Vec<PipeStage> }`
   - `PipeStage`: `Where(Expr)`, `Sort(Vec<SortExpr>)`, `Limit(u64)`, `Join(JoinStage)`,

   - AST types are org-agnostic: no tenant/org_id fields; executor injects org scope at
     planning time (ADR-006 compliance)

5. Implement filter mode parser in `crates/prism-query/src/filter_parser.rs` (BC-2.11.002):
   - Grammar: `source_name '|' predicate`
   - `source_name`: dotted identifier e.g. `crowdstrike.detections`
   - `predicate`: boolean expression tree
   - Boolean combinators: `AND`, `OR`, `NOT` (case-insensitive keywords)
   - Comparison operators: `=`, `!=`, `>`, `<`, `>=`, `<=`, `LIKE`, `IN (...)`
   - Field paths: dot-notation traversal e.g. `device.hostname`
   - Literals: strings (single/double quoted), integers, floats, booleans, NULL
   - Mode detection: filter mode is detected when a `|` immediately follows the source
     ref with no `SELECT` or `FROM` keyword â€” grammar-level disambiguation

6. Implement SQL mode parser in `crates/prism-query/src/sql_parser.rs` (BC-2.11.003):
   - `SELECT` (with `*`, `table.*`, `expr AS alias`, `DISTINCT`)
   - `FROM source_ref [AS alias]`
   - `JOIN` â€” all types: `INNER JOIN`, `LEFT JOIN`, `RIGHT JOIN`, `FULL OUTER JOIN`
   - `WHERE expr` (supports subqueries: `WHERE field IN (SELECT ...)`)
   - `GROUP BY expr_list`
   - `HAVING expr`
   - `ORDER BY expr [ASC|DESC] [, ...]`
   - `LIMIT n`
   - Subqueries in WHERE clause `(SELECT ... FROM ... WHERE ...)`
   - Source refs in FROM/JOIN use dot-notation matching sensor.table convention
   - Mode detection: SQL mode detected when input starts with `SELECT` keyword

7. Implement pipe mode parser in `crates/prism-query/src/pipe_parser.rs` (BC-2.11.004):
   - Grammar: `source_ref ('|' pipe_stage)*`
   - Mode detection: pipe mode detected when input starts with `FROM` keyword
   - BC-2.11.004 stages (7 required):
     - `where expr` â€” filter stage (same grammar as filter mode)
     - `sort field [asc|desc] [, field [asc|desc]]*` â€” sort stage
     - `head N` â€” take first N rows (equivalent to LIMIT N)
     - `tail N` â€” take last N rows (reverse sort + limit + reverse)
     - `stats agg_func [by field]` â€” aggregation (count, sum, avg, min, max) + GROUP BY
     - `dedup field [, field]*` â€” deduplicate by specified fields
     - `fields [+|-] field [, field]*` â€” include/exclude specific fields
   - Additional stages (from other BCs/architecture):
     - `join source_ref on field_path` â€” join stage (query-engine.md JOIN support)
     - `enrich infusion(field_path)` â€” enrichment stage (AD-020, S-1.14)
   - `limit N` accepted as alias for `head N`
   - All keywords case-insensitive

8. Implement query security limits in `crates/prism-query/src/security.rs` (DI-019):
   - `check_query_size(raw: &str) -> Result<(), PrismError>`: reject if `raw.len() >
     PRISM_MAX_QUERY_SIZE` (env var, default 65_536 bytes (64KB per BC-2.11.006)); return E-QUERY-003
   - `check_nesting_depth(ast: &Expr, depth: u32) -> Result<(), PrismError>`: recursive
     descent counting; reject if depth > `PRISM_MAX_NESTING_DEPTH` (default **64** per
     BC-2.11.006, DI-019, EC-002 â€” canonical limit is 64, not 32); return E-QUERY-003
   - `check_pipe_stage_count(stages: &[PipeStage]) -> Result<(), PrismError>`: reject if
     stages.len() > 32 (per BC-2.11.006 pipe stage count limit); return E-QUERY-003
   - `check_regex_pattern_length(pattern: &str) -> Result<(), PrismError>`: reject if
     pattern.len() > 1024 bytes (per BC-2.11.006 regex limit in `matches` predicates);
     return E-QUERY-003
   - All checks run before AST is returned to caller

9. Implement error recovery in parser modules using Chumsky's built-in recovery strategies:
   - `skip_then_retry_until` for recovering past unknown tokens in filter/pipe stages
   - `nested_delimiters` for recovering inside parenthesized subexpressions in SQL
   - Recovery errors are accumulated; parser returns both partial AST and all errors
   - Chumsky 0.12 error type: use `Rich<'a, char>` (or `Rich<'a, Token>` for token-level)
     via `extra::Err<Rich<...>>` â€” NOT `Simple<Token>` (removed in 0.10+)

## Acceptance Criteria

**AC-1:** Given `crowdstrike.detections | severity_id >= 3`, When parsed in filter mode,
Then the AST contains `source = "crowdstrike.detections"` and `predicate = severity_id >= 3`.
(BC-2.11.002)

**AC-2:** Given `crowdstrike.detections | severity_id >= 3 AND category = 'malware'`, When
parsed in filter mode, Then the predicate AST contains an AND node with two comparison
children. (BC-2.11.002)

**AC-3:** Given `SELECT * FROM crowdstrike.detections WHERE severity_id >= 3 ORDER BY time
DESC LIMIT 100`, When parsed in SQL mode, Then the AST contains correct SELECT / FROM /
WHERE / ORDER BY / LIMIT nodes. (BC-2.11.003)

**AC-4:** Given `SELECT a.*, b.* FROM crowdstrike.detections a JOIN claroty.alerts b ON
a.device_id = b.device_id`, When parsed in SQL mode, Then the AST contains an INNER JOIN
node with join condition `a.device_id = b.device_id`. (BC-2.11.003)

**AC-5:** Given `SELECT * FROM crowdstrike.detections WHERE device_id IN (SELECT device_id
FROM claroty.alerts)`, When parsed in SQL mode, Then the AST contains a subquery node
inside the WHERE IN clause. (BC-2.11.003)

**AC-6:** Given `crowdstrike.detections | where severity_id >= 3 | sort time desc | limit
100`, When parsed in pipe mode, Then the AST contains source `crowdstrike.detections` and
three pipeline stages: Where, Sort, Limit. (BC-2.11.004)

**AC-7:** Given `crowdstrike.detections | where severity_id >= 3 | join claroty.alerts on
device_id | enrich infusion(hostname)`, When parsed in pipe mode, Then the AST contains
Join and Enrich stages with correct parameters. (BC-2.11.004)

**AC-8:** Given a query string exceeding 65,536 bytes (64KB), When `parse()` is called, Then
`E-QUERY-003` is returned and no AST is produced. (traces to BC-2.11.006 postcondition 1)

**AC-9:** Given a syntactically malformed query with a recoverable error, When parsed,
Then partial AST nodes that are valid are returned alongside structured error messages
identifying the recovery point.

**AC-10:** VP-014 and VP-015 Kani proofs pass under `cargo kani`. VP-021 fuzz target
runs 30 minutes without any panics.

## Architecture Compliance Rules

- Parser modules MUST NOT import from `prism-sensors`, `prism-storage`, or any I/O crate
  â€” parsing is a pure function: `&str -> Result<Ast, Vec<ParseError>>`
- The three parser modes (filter, SQL, pipe) MUST share AST expression types â€” do NOT
  duplicate `Expr`, `Literal`, or `CompareOp` between modules
- Security checks (size, depth) MUST run before any AST is returned to callers â€” they
  are not optional post-processing
- Chumsky parsers MUST use `recover_with` strategies â€” bare `unwrap()` on parse results
  is forbidden; parser panics are a security issue (injection vector)
- `SourceRef` dot-notation parsing MUST reject path traversal characters (`/`, `\`, `..`)
  at the parser level; this is not a semantic check

## Dev Notes

- Chumsky 0.12 uses a different API from 0.9 â€” verify the `Parser` trait bounds and
  `.recover_with()` signatures against the 0.12 changelog before starting
- The three query modes are syntactically unambiguous: filter mode requires a `|`
  immediately after the source ref with no SELECT keyword; SQL mode starts with SELECT;
  pipe mode's stages use lowercase keywords after the initial `|`
- The parser intentionally does NOT resolve sensor names or validate that sources exist â€”
  that is the query engine's job in S-3.02. The parser is source-agnostic.
- Error messages from ariadne should be serialized to JSON strings for MCP tool
  responses â€” LLM consumers need structured errors, not ANSI-colored terminal output
