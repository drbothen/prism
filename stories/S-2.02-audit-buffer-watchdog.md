# S-2.02 — prism-storage: Audit Buffer and Watchdog

## Narrative

As a Prism platform operator, I want audit entries buffered durably in RocksDB with
overflow protection, and a resource watchdog that terminates runaway queries and
denylists repeat offenders, so that the system remains stable under load while
preserving the complete audit trail.

## Objective

Implement audit buffer persistence with exponential backoff retry and 100K-entry overflow
protection. Implement the resource watchdog with graduated memory limit levels, query
termination on limit violation, and a persistent query denylist with manual override
support.

## Tasks

1. Implement audit buffer write path in `crates/prism-storage/src/audit_buffer.rs`:
   - `append_audit_entry(entry: &AuditEntry) -> Result<(), PrismError>`
   - Key: `{timestamp_ns}:{trace_id}` (lexicographically ordered; enables ordered scan)
   - Value: `bincode::encode(entry)`
   - Write to the `audit_buffer` CF via the shared `StorageBackend` (BC-2.15.003)

2. Implement exponential backoff retry for failed audit forwarding in
   `crates/prism-storage/src/audit_buffer.rs`:
   - Forwarding attempts: base delay 1s, multiplier 2x, max delay 60s, max attempts 10
   - On final failure after all retries, emit a `tracing::error!` event with structured
     fields; entry remains in the buffer for the next forwarding cycle (BC-2.15.003)

3. Implement audit buffer overflow protection in
   `crates/prism-storage/src/audit_buffer.rs`:
   - `check_and_purge_overflow() -> Result<usize, PrismError>`:
     scan count of entries in `audit_buffer` CF
   - If count exceeds 100,000, delete the oldest entries (lowest timestamp keys) until
     count reaches 90,000 (10% headroom) (BC-2.15.004)
   - Return number of entries purged
   - Called by the background maintenance task every 60 seconds

4. Implement `ResourceWatchdog` in `crates/prism-storage/src/watchdog.rs`:
   - Struct fields: `warn_pct: f64` (0.70), `throttle_pct: f64` (0.85),
     `kill_pct: f64` (0.95), `budget_bytes: usize` (512 * 1024 * 1024)
   - `WatchdogLevel` enum: `Normal`, `Warn`, `Throttle`, `Kill`
   - `current_level() -> WatchdogLevel`: reads current RSS from `/proc/self/status`
     (Linux) or `sysinfo` crate (cross-platform), returns level based on thresholds
     (BC-2.15.006)

5. Implement watchdog query termination in `ResourceWatchdog`:
   - `check_query(cancel_token: CancellationToken) -> Result<(), PrismError>`:
     if `current_level() == Kill`, call `cancel_token.cancel()` and return
     `Err(PrismError::WatchdogKilled)` mapping to E-WATCH-001 (BC-2.15.007)
   - Watchdog runs a background tokio task that polls every 500ms and cancels all
     registered query cancel tokens when `Kill` level is reached

6. Implement query denylist in `crates/prism-storage/src/denylist.rs`:
   - Storage: `watchdog` CF in RocksDB, key=`denylist:{fingerprint}`,
     value=`{failure_count}:{last_failure_ts}:{expiry_ts}`
   - `record_failure(fingerprint: &str) -> Result<DenylistStatus, PrismError>`:
     increment failure count; if count >= `DENYLIST_THRESHOLD` (default: 3),
     set expiry to `now + 1h` and mark as denylisted (BC-2.15.008)
   - `is_denylisted(fingerprint: &str) -> Result<bool, PrismError>`:
     lazy expiry check — if expiry_ts < now, remove from denylist and return false;
     otherwise return true (BC-2.15.008)
   - `clear_denylist(fingerprint: Option<&str>) -> Result<usize, PrismError>`:
     if `Some(fp)`, remove that entry; if `None`, remove all denylist entries;
     return count of entries removed (BC-2.15.008)

7. Implement `watchdog_status` tool data source in
   `crates/prism-storage/src/watchdog.rs`:
   - `WatchdogStatus { level: WatchdogLevel, budget_bytes: usize, current_bytes: usize,
     denylist: Vec<DenylistEntry> }`
   - `get_watchdog_status() -> Result<WatchdogStatus, PrismError>`
   - Caller (MCP tool layer) gates this behind the `runtime` capability

## Acceptance Criteria

**AC-1:** Given audit entries written to the buffer, When Prism restarts, Then entries
are readable from the `audit_buffer` CF in timestamp order. (BC-2.15.003)

**AC-2:** Given 100,001 entries in the `audit_buffer` CF, When `check_and_purge_overflow()`
runs, Then entries are purged until count is at or below 90,000, and the return value
equals the number of entries deleted. (BC-2.15.004)

**AC-3:** Given current RSS at 86% of the 512MB budget, When `current_level()` is called,
Then it returns `WatchdogLevel::Throttle`. (BC-2.15.006)

**AC-4:** Given a running query with a registered cancel token and RSS at 96% of budget,
When the watchdog background task fires, Then the cancel token is cancelled and the query
returns `Err(PrismError::WatchdogKilled)` (E-WATCH-001). (BC-2.15.007)

**AC-5:** Given a query fingerprint that has failed 3 consecutive times, When the same
fingerprint is submitted and `is_denylisted()` is checked, Then it returns `true` and the
caller returns `Err(PrismError::QueryDenylisted)` (E-WATCH-002). (BC-2.15.008)

**AC-6:** Given a denylisted fingerprint, When `clear_denylist(Some(fingerprint))` is
called with the `runtime` capability, Then `is_denylisted()` returns `false` for that
fingerprint.

## Architecture Compliance Rules

- `ResourceWatchdog` and `AuditBuffer` MUST use the `StorageBackend` trait, not
  `RocksDbBackend` directly — enables test injection of `InMemoryBackend`
- Memory measurement MUST use a cross-platform approach via the `sysinfo` crate;
  do not use platform-specific syscalls directly
- The denylist expiry check MUST be lazy (at query start), not eager — a background
  reaper is unnecessary and adds complexity
- Watchdog kill level MUST cancel ALL registered tokens, not just the newest —
  use a `DashMap<QueryId, CancellationToken>` registry

## Notes

- The 100K overflow threshold is absolute, not configurable at runtime; it is defined
  as `const AUDIT_BUFFER_MAX_ENTRIES: usize = 100_000` and the purge target is
  `const AUDIT_BUFFER_PURGE_TARGET: usize = 90_000`.
- The denylist threshold `DENYLIST_THRESHOLD` should be configurable via `prism.toml`
  under `[watchdog] denylist_threshold = 3`; read it from `ConfigSnapshot` once at
  startup rather than on every check.
- Watchdog polling interval (500ms) is intentionally aggressive to catch memory spikes
  quickly; this is acceptable given the poll is a single RSS read, not a full scan.
- The `clear_denylist` capability check is enforced by the MCP tool layer, not by
  `prism-storage` — the storage module provides the operation unconditionally.
