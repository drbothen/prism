# S-3.3.05 — prism-dtu-harness: builder ergonomics, per-test overrides, and documentation

## Narrative

As a Prism test author migrating from single-tenant DTU tests to the multi-tenant
harness, I want a fluent builder API with per-test overrides for archetype, scale,
seed, and failure injection, so that I can write readable test setup code without
hand-constructing raw `CustomerSpec` structs or duplicating fixture config per test.

## Objective

Polish the `HarnessBuilder` API introduced in S-3.3.03/04 to expose
`with_customer_overrides`, chainable override closures, and documented usage
examples. No new behavioral contracts — this story refines the ergonomics of the
BC-3.5.* and BC-3.6.* implementation without changing observable behavior.

## Tasks

1. Extend `CustomerSpec` (in `src/types.rs`) with override fields:
   - `archetype: Option<Archetype>` — overrides the default archetype from `customers/*.toml`
   - `scale: Option<f64>` — overrides `data.scale`
   - `seed: Option<u64>` — overrides `data.seed`
   - `initial_failure: Option<FailureMode>` — injects a failure mode at harness build time (applied before first test request)

2. Implement `HarnessBuilder::with_customer_overrides(slug: &str, f: impl FnOnce(&mut CustomerSpec)) -> Self`:
   - Looks up the `CustomerSpec` for `slug` (inserted by prior `with_customer` call)
   - Calls `f(&mut spec)` to apply overrides
   - Replaces the spec in the builder's internal map
   - Returns `self` for chaining

3. Implement `HarnessBuilder::with_failure(slug: &str, dtu_type: DtuType, mode: FailureMode) -> Self`:
   - Shorthand for setting `initial_failure` via `with_customer_overrides`
   - Applied during `build()` before the harness returns to the caller

4. Ensure `HarnessBuilder` is fully chainable (all methods return `Self`); example:
   ```rust
   let harness = Harness::builder()
       .isolation(IsolationMode::Logical)
       .with_customer("acme-corp")
       .with_customer_overrides("acme-corp", |c| {
           c.dtu("claroty").set_archetype(Archetype::AuthOutage);
       })
       .with_customer("globex")
       .build()
       .await?;
   ```

5. Add rustdoc to all public items in `src/builder.rs` and `src/harness.rs`:
   - Document each method with a `# Examples` block
   - Include at minimum: single-customer logical example, multi-customer network example, override example, failure injection example

6. Create `crates/prism-dtu-harness/README.md`:
   - Section: "Quick start" — minimal single-customer test
   - Section: "Multi-tenant scenarios" — 3-org pattern with network mode
   - Section: "Failure injection" — `inject_failure` + `clear_failure` usage
   - Section: "Per-test overrides" — `with_customer_overrides` usage

7. Add compile-fail test confirming that harness items are not accessible without `feature = "dtu"` or `cfg(test)`.

## Acceptance Criteria

**AC-001:** `Harness::builder().isolation(IsolationMode::Logical).with_customer("acme-corp").with_customer_overrides("acme-corp", |c| c.dtu("claroty").set_archetype(Archetype::AuthOutage)).build().await` compiles and builds a harness where the acme-corp Claroty clone uses the `AuthOutage` archetype.
(traces to BC-3.5.001 precondition 2; builder ergonomics over BC-3.5.001)

**AC-002:** All `HarnessBuilder` methods return `Self`, enabling a single fluent chain from `Harness::builder()` to `.build().await` without intermediate `let` bindings.
(traces to BC-3.5.001 precondition 2; ergonomics)

**AC-003:** `with_customer_overrides` called for an unknown slug returns a builder that produces `Err(HarnessError::UnknownOrg)` at `build()` time.
(traces to BC-3.6.001 EC-001)

**AC-004:** `with_failure(slug, dtu_type, mode)` injects the failure mode at build time; the first request to that clone after `build()` returns observes the injected mode without requiring a separate `inject_failure` call.
(traces to BC-3.6.001 postcondition clause 1)

**AC-005:** All public items in `prism-dtu-harness` are inaccessible from a crate that does not enable `feature = "dtu"` and is not a test binary; the compile-fail test asserts this.
(traces to BC-3.5.001 precondition 6; BC-3.5.002 precondition 7)

**AC-006:** The harness crate README contains working code examples for single-customer logical mode, multi-customer network mode, per-test overrides, and failure injection; examples must be `cargo test --doc`-passing.
(traces to BC-3.5.001 usage documentation; BC-3.6.001 postcondition clarity)

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `with_customer_overrides` called before `with_customer` for the same slug | Builder defers resolution to `build()`; `Err(UnknownOrg)` at build time |
| EC-002 | `with_failure` called with `FailureMode::None` | No-op (clears any previously set `initial_failure`); consistent with `clear_failure` semantics |
| EC-003 | Multiple `with_customer_overrides` calls for the same slug | Each closure applied in call order; last write wins for overlapping fields |
