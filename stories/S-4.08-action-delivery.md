# S-4.08 â€” prism-operations: Action Delivery Framework

## Narrative

As a Prism operations platform, I want alert, case, schedule, and manual triggers to
deliver notifications and reports to external systems â€” Slack, PagerDuty, Jira, email,
syslog, and custom webhooks â€” via declarative `.action.toml` specs and optional WASM
plugins, so that MSSP analysts never need to write code to wire up client notification
workflows.

## Objective

Implement the action delivery engine per AD-021: parse `.action.toml` specs into
`ActionSpec` structs, implement `ActionEngine` with trigger evaluation for alert/case/
schedule/manual modes, implement built-in destination types (webhook, email, syslog),
implement WASM plugin action delegation via `PluginRuntime` (S-1.15), implement at-least-
once delivery with exponential backoff retry, track delivery state in RocksDB
`action_state` CF, enforce per-action rate limiting and deduplication windows, execute
scheduled report queries through the existing `QueryEngine` semaphore, and participate
in hot reload (AD-018).

## Tasks

1. Define `ActionSpec` and related types in `crates/prism-operations/src/action/mod.rs`:
   - `ActionSpec { action_id, name, trigger: TriggerMode, filter: Option<String>, clients: Vec<ClientId>, case_events: Vec<CaseEvent>, schedule: Option<String>, destination: DestinationConfig, credentials: HashMap<String, CredentialRef>, plugin_config: HashMap<String, String>, rate_limit: RateLimitConfig }`
   - `TriggerMode`: `Alert | Case | Schedule | Manual`
   - `CaseEvent`: `Created | StatusChanged | DispositionSet | AlertLinked | AnnotationAdded | AssigneeChanged`
   - `DestinationConfig`: `Webhook(WebhookConfig) | Email(EmailConfig) | Syslog(SyslogConfig) | Plugin(PluginDestConfig)`
   - `RateLimitConfig { max_per_hour: u32, cooldown_seconds: u32, deduplicate_window_seconds: u32 }` â€” defaults: 100/hr, 30s, 300s
   - `ActionRegistry`: `HashMap<String, ActionSpec>` behind `arc_swap::ArcSwap`
   - `ActionEngine`: holds `ActionRegistry`, `PluginRuntime` reference, `QueryEngine` reference, `RocksDB` reference for `action_state` CF

2. Implement TOML spec loader in `crates/prism-operations/src/action/loader.rs`:
   - Load `*.action.toml` files from `{config_dir}/actions/` directory
   - Validate required fields: `action_id`, `trigger`, `[action.destination]` with valid `type`
   - Validate credential references use `{ source = "env", key = "..." }` form â€” reject inline string values with `E-ACTION-001: "Credential '{field}' in action '{id}' must use a reference (source = 'env', key = 'KEY_NAME'), not an inline value."`
   - Validate `trigger = "case"` requires at least one entry in `case_events`
   - Validate `trigger = "schedule"` requires a valid cron expression (parse with `cron` crate)
   - Validate `clients` entries are non-empty strings if present
   - On validation error, reject the spec â€” do not partially register

3. Implement trigger evaluation in `crates/prism-operations/src/action/trigger.rs`:
   - `AlertTrigger`: subscribes to the alert broadcast channel (from S-4.05). For each alert:
     - Find all `ActionSpec` with `trigger = Alert`
     - For specs with a `clients` list: skip if `alert.client_id` not in the list
     - Evaluate `filter` expression against alert fields using a simple predicate evaluator
       (same DataFusion expression evaluator already used in detection rules â€” reuse `prism-operations::detection::filter::evaluate_predicate`)
     - If filter matches: check rate limit â†’ render template â†’ deliver
   - `CaseTrigger`: subscribes to the case event channel (new tokio broadcast channel,
     separate from the alert channel). For each case event (created, status_changed, etc.):
     - Find all `ActionSpec` with `trigger = Case` and matching `case_events` entry
     - Apply `clients` filter and `filter` expression (evaluated against case fields)
     - If matches: check rate limit â†’ render template â†’ deliver
   - `ScheduleTrigger`: runs a 1-second tick loop (tokio interval). For each tick:
     - Evaluate all `trigger = Schedule` action specs' cron expressions against current time
     - Fire matching specs via `ActionEngine::fire_schedule`
   - `ManualTrigger`: called directly by the `fire_action` MCP tool (S-5.06)

4. Implement built-in destination types in `crates/prism-operations/src/action/destinations/`:
   - **webhook.rs**: `WebhookDestination::deliver(rendered_body: String, config: WebhookConfig) -> Result<(), DeliveryError>`
     - Resolve URL via `CredentialResolver` (same pattern as sensor auth in S-2.06)
     - POST/PUT the rendered body with configured `Content-Type` header
     - HTTP 2xx â†’ success; HTTP 4xx (except 429) â†’ non-retryable `DeliveryError::Rejected`; HTTP 429 / 5xx / network error â†’ retryable `DeliveryError::Transient`
   - **email.rs**: `EmailDestination::deliver(rendered_body: String, config: EmailConfig) -> Result<(), DeliveryError>`
     - Resolve SMTP credentials via `CredentialResolver`
     - Connect to SMTP using `lettre` crate with STARTTLS
     - Build MIME message with subject, from, to, HTML or text body
     - For report actions: attach rendered HTML report as body
   - **syslog.rs**: `SyslogDestination::deliver(alert: &Alert, config: SyslogConfig) -> Result<(), DeliveryError>`
     - Connect to syslog endpoint (UDP / TCP / TCP+TLS per config)
     - Format message: `cef` (ArcSight CEF), `leef` (IBM QRadar LEEF), or `json` (plain JSON)
     - CEF: `CEF:0|1898andCo|Prism|{version}|{rule_id}|{rule_name}|{severity}|...`
     - LEEF: `LEEF:2.0|1898andCo|Prism|{version}|{rule_id}|...`
   - **plugin.rs**: `PluginDestination::deliver(ctx: ActionContext, config: PluginDestConfig) -> Result<(), DeliveryError>`
     - Call `PluginRuntime::fire_alert(plugin_id, ctx, config)` or `fire_case` / `fire_report`
     - Map `PluginError::Trapped` / `PluginError::Timeout` â†’ retryable `DeliveryError::Transient`
     - Map `PluginError::NotLoaded` â†’ non-retryable `DeliveryError::Rejected`

5. Implement template rendering in `crates/prism-operations/src/action/template.rs`:
   - Reuse `TemplateInterpolator` from S-4.05 (same `${variable_path}` syntax)
   - Alert context variables: populate from `AlertContext` struct with all fields from the
     template variable table in AD-021 (alert.alert_id, alert.rule_id, alert.severity, etc.)
   - Case context variables: populate from `CaseContext` with all case fields
   - **UUID v7 validation for `${case.alert_ids_quoted}`**: before interpolating this variable,
     validate each alert ID in the list matches UUID v7 format (regex or uuid crate parse).
     Drop any non-conforming values with `WARN` log: `"Dropping non-UUID v7 value from alert_ids_quoted in action '{action_id}': '{value}'"`.
     If all values are dropped, produce an empty string (query returns empty result set, not error).
   - Run `InjectionScanner` (S-1.10) on all `${alert.*}` and `${event.*}` variable values
     before interpolation. Flag detected patterns as `_safety_flags` on the audit log entry.

6. Implement rate limiting in `crates/prism-operations/src/action/rate_limit.rs`:
   - Three controls per action:
     - `max_per_hour`: counter per `action_id` keyed in `action_state` CF under
       `"{action_id}:fire_count:{hour_bucket}"`. Hour bucket = Unix timestamp / 3600.
     - `cooldown_seconds`: minimum elapsed time between fires for the same `(action_id, rule_id, client_id)` triple. Enforced via `"{action_id}:last_fired:{rule_id}:{client_id}"` key in `action_state` CF.
     - `deduplicate_window_seconds`: suppress if a delivery with the same content hash
       (`SHA-256` of `alert.rule_id + alert.client_id`) was already delivered within the window.
       Key: `"{action_id}:dedup:{content_hash}"` â†’ delivery timestamp in `action_state` CF.
   - When suppressed: audit-log the suppression with reason (rate limit / cooldown / dedup)
   - All `action_state` CF reads/writes use `StorageDomain::ActionState`

7. Implement at-least-once delivery with retry in `crates/prism-operations/src/action/retry.rs`:
   - On delivery failure (retryable `DeliveryError::Transient`):
     - Exponential backoff: attempt 1 = 2s, 2 = 4s, 3 = 8s, 4 = 30s, 5 = 60s (max 5 attempts)
     - Write retry state to `action_state` CF: `"{action_id}:retry:{alert_id}"` â†’ JSON with
       `{ attempt: u32, next_retry_at: Timestamp, last_error: String }`
     - Use `tokio::time::sleep` between attempts (do NOT block the trigger evaluation loop)
   - After max retries: write failure record to `action_state` CF dead-letter key:
     `"{action_id}:dead_letter:{alert_id}"` â†’ JSON with final error and context.
     Audit-log as `action_delivery_failed` event. The alert is NOT lost (still in RocksDB
     `alerts` CF from S-4.05).
   - On success: delete the `retry` key from `action_state` CF.
   - Schedule triggers: no retry state â€” just log failure and wait for next cron tick.
   - Manual triggers: return error immediately, no retry.

8. Implement scheduled report execution in `crates/prism-operations/src/action/report.rs`:
   - For `trigger = Schedule` + `[action.destination.report]`: on cron fire:
     1. Acquire schedule semaphore via `try_acquire()` â€” if unavailable, log skip and return
     2. For each query in `report.queries`:
        - Set dirty bit in `dirty_query_set` (same pattern as S-4.01 scheduled queries)
        - Execute via `QueryEngine::execute(query, client_id, 200MB_budget, 30s_timeout)`
        - On success: render section with `title` and query results as HTML table / markdown
        - On failure (timeout, OOM, error): include error section: `"[Section '{title}' failed: {error}]"`
        - Clear dirty bit
     3. Assemble full report from all sections (partial on error per INV-ACTION-005)
     4. Deliver assembled report via the destination (email or webhook)

9. Implement cron tick loop in `crates/prism-operations/src/action/cron.rs`:
   - `CronScheduler::run(registry: Arc<ActionRegistry>, engine: Arc<ActionEngine>)`:
     - `tokio::time::interval(Duration::from_secs(1))` tick
     - For each tick: iterate over `Schedule`-trigger actions, evaluate cron expression
       against current UTC time using `cron` crate
     - Fire matching actions via `ActionEngine::fire_schedule(spec, client_id_list)`

10. Participate in hot reload (AD-018):
    - Watch `{config_dir}/actions/*.action.toml`
    - On change: re-validate, arc-swap `ActionRegistry`
    - In-flight deliveries continue using the old spec; next trigger fires use new spec
    - Plugin hot reload is handled by `PluginRuntime` (S-1.15) independently

11. Emit case event channel:
    - `ActionEngine` provides a `case_event_sender: tokio::sync::broadcast::Sender<CaseEvent>`
    - S-4.06 (Case Management) must call `action_engine.emit_case_event(event)` after every
      case mutation (create, update, annotation). Wire the sender into `CaseManager::update_case`.
    - This requires a minor modification to S-4.06's `CaseManager` to accept an optional
      `ActionEngine` reference. If not provided, case events are not emitted (graceful degradation).

12. Write tests in `crates/prism-operations/tests/action_tests.rs`:
    - Load `slack_soc.action.toml` fixture â†’ verify `ActionSpec` parsed and registered
    - Emit a mock alert matching the filter â†’ verify webhook POST called with rendered body
    - Emit a mock alert NOT matching the filter â†’ verify no delivery
    - Exceed `max_per_hour` â†’ verify delivery suppressed with audit log
    - Verify cooldown: fire same rule twice within cooldown window â†’ second delivery suppressed
    - Simulate webhook returning 500 â†’ verify retry with backoff â†’ verify dead-letter after 5 failures
    - Load action with inline credential value â†’ verify `E-ACTION-001` rejection
    - Verify `${case.alert_ids_quoted}` with a non-UUID value â†’ verify dropped with WARN log
    - Verify case trigger: `case_events = ["created"]` fires on case creation, not on status_changed

## Acceptance Criteria

**AC-1:** Given a `slack_soc.action.toml` with `trigger = "alert"` and `filter = 'severity_id >= 4'`,
When an alert with `severity_id = 5` is broadcast, Then a webhook POST is made to the
Slack URL with the rendered body template within 2 seconds.
(traces to BC-2.18.001 / INV-ACTION-001)

**AC-2:** Given an action with `max_per_hour = 10` and 10 alerts already delivered this hour,
When an 11th alert fires the trigger, Then the delivery is suppressed and a
`action_suppressed` event is audit-logged with `reason: "rate_limit"`.
(traces to BC-2.18.001 / INV-ACTION-001, BC-2.18.008 / INV-ACTION-008)

**AC-3:** Given a webhook destination returns HTTP 503, When the action fires, Then the
delivery is retried with exponential backoff (2s, 4s, 8s, 30s, 60s). After 5 failed
attempts, a dead-letter record is written to the `action_state` CF and an
`action_delivery_failed` event is audit-logged.
(traces to BC-2.18.001 / INV-ACTION-001, BC-2.18.008 / INV-ACTION-008)

**AC-4:** Given a case-triggered action with `case_events = ["created", "status_changed"]`,
When a case is created, Then the action fires. When the case's annotation is updated
(not a status change), Then the action does NOT fire.
(traces to BC-2.18.001 / INV-ACTION-001)

**AC-5:** Given a scheduled action with `schedule = "0 8 * * *"` and report queries defined,
When the cron fires at 08:00 UTC, Then the report queries execute via `QueryEngine`,
each section is rendered, and the assembled report is delivered via the configured
destination.
(traces to BC-2.18.002 / INV-ACTION-002, BC-2.18.004 / INV-ACTION-004)

**AC-6:** Given a scheduled report action fires while all 16 schedule semaphore permits are
held, When the cron tick occurs, Then the report execution is skipped for this tick with
a log entry `"action report '{action_id}' skipped: schedule semaphore unavailable"`.
(traces to BC-2.18.004 / INV-ACTION-004)

**AC-7:** Given a scheduled report with 3 queries where query 2 times out, When the
report is assembled, Then sections 1 and 3 contain results and section 2 contains an
error note. The report is delivered with partial content.
(traces to BC-2.18.005 / INV-ACTION-005)

**AC-8:** Given an action template with `${alert.description}` where the description
contains a prompt injection pattern, When the template is rendered, Then
`InjectionScanner` detects the pattern, `_safety_flags` is set on the audit log entry,
but the original value is STILL interpolated into the template (flag, don't strip â€” per
BC-2.09.004 and BC-2.18.006). The safety flags warn downstream consumers; the data is
preserved intact.
(traces to BC-2.18.006, BC-2.09.004)

**AC-9:** Given an `.action.toml` with `routing_key = "my-secret-value"` (inline value),
When the loader validates the spec, Then it returns `E-ACTION-001` and the action is
NOT registered.
(traces to BC-2.18.007 / INV-ACTION-007)

**AC-10:** Given `${case.alert_ids_quoted}` contains the list `["01905a7b-...", "not-a-uuid"]`,
When the template is rendered, Then only the valid UUID v7 is interpolated, `"not-a-uuid"`
is dropped with a WARN log, and the output is the single valid UUID.
(traces to BC-2.18.009 / INV-ACTION-009)

**AC-11:** Given a `trigger = "manual"` action registered in the `ActionRegistry`, When
`fire_action` is called by the AI agent with a valid `action_id`, Then the destination
is called exactly once (no retry on failure), the `fire_action` MCP tool call returns
immediately to the caller with `{ "status": "delivered" | "failed", "error": ...,
"delivery_latency_ms": ... }` â€” it does NOT block beyond the destination timeout (10s
for webhooks). On failure, `status: "failed"` is returned with the error message; no
retry state is written to the `action_state` CF; no dead-letter entry is created. An
`action_manual_fired` audit event is emitted with `action_id`, `status`, and `latency_ms`.
(traces to BC-2.18.003 / INV-ACTION-003 â€” manual triggers are fire-and-forget, result returned immediately)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Action delivery engine | `prism-operations` (SS-12, SS-13, SS-14, SS-18) | Mixed |
| Action spec loading | `prism-spec-engine` (SS-16) | Mixed |
| WASM action plugins | `prism-spec-engine` (SS-17) | Mixed |
| Action state persistence | `prism-storage` (SS-15) | Effectful |

`prism-operations` implements `ActionEngine` (AD-021) â€” evaluates action specs against
alert/case/schedule triggers, renders Jinja-style templates, and delivers via built-in
types (Slack, PagerDuty, Jira, email, syslog, webhook) or WASM plugins. Template
rendering and trigger evaluation are pure; HTTP delivery, retry logic, and
`action_state` CF writes are effectful. Per `architecture/module-decomposition.md`,
SS-18 (Action Delivery Engine) is owned by `prism-operations`.

## Architecture Compliance Rules

- Alert delivery MUST complete persistence to RocksDB `action_state` CF before returning
  success. A delivery that claimed success but did not persist retry state would violate
  at-least-once guarantees on restart.
- `ActionEngine::fire_schedule` MUST use `try_acquire()` on the 16-permit schedule
  semaphore, NOT `acquire()`. Using `acquire()` would block the cron tick loop if all
  permits are taken, stalling ALL scheduled actions including detection schedules.
- Case event emission from S-4.06 MUST happen after the case mutation is persisted to
  RocksDB. Emitting the event before persistence creates a race where the action engine
  reads stale case data.
- Template rendering MUST run `InjectionScanner` on all `${alert.*}`, `${event.*}`, and
  `${case.*}` variables before interpolation. Variables from internal sources
  (`${prism_version}`, `${date}`, `${client_id}`) are trusted and not scanned.
- `${case.alert_ids_quoted}` UUID v7 validation MUST occur before interpolation, not after.
  The validation protects the generated PrismQL query string from injection.
- The cron tick loop (1-second interval) MUST be a separate tokio task from the alert
  trigger subscriber. They must not block each other.
- WASM plugin actions use `PluginRuntime::fire_alert` / `fire_case` / `fire_report`.
  These calls MUST NOT be made on the trigger evaluation loop's task â€” spawn a new
  `tokio::task` for each plugin invocation to isolate plugin CPU time limits from the
  trigger loop.

## Dev Notes

- Syslog CEF format: the severity field maps OCSF `severity_id` to CEF severity 0-10:
  `0=Unknown, 1=Informational(1-3), 4=Low, 6=Medium, 8=High, 10=Critical`.
- The lettre SMTP client supports both STARTTLS and implicit TLS. Default to STARTTLS on
  port 587. Support `smtp_tls = "implicit"` config for port 465.
- For Jira and PagerDuty action types, the architecture spec designates these as
  `type = "plugin"` destinations using `.prx` WASM plugins. The built-in `webhook` type
  covers Slack, Teams, Discord, and generic HTTP endpoints. Do NOT implement Jira or
  PagerDuty as built-in Rust types â€” they belong in plugins.
- Action IDs must be unique within the registry. Detect duplicate `action_id` across
  multiple `.action.toml` files at load time and reject with `E-ACTION-002:
  "Duplicate action_id '{id}' found in '{path2}' â€” already registered from '{path1}'."`
- The `clients = []` (empty) convention means all clients, not no clients. Validate this
  interpretation is consistent with how `filter` evaluation handles client scoping.
