# S-4.08 — prism-operations: Action Delivery Framework

## Narrative

As a Prism operations platform, I want alert, case, schedule, and manual triggers to
deliver notifications and reports to external systems — Slack, PagerDuty, Jira, email,
syslog, and custom webhooks — via declarative `.action.toml` specs and optional WASM
plugins, so that MSSP analysts never need to write code to wire up client notification
workflows.

## Objective

Implement the action delivery engine per ADR-016 (Action Delivery Framework) and
`.factory/specs/architecture/actions.md` (AD-021): parse `.action.toml` specs into
`ActionSpec` structs (with `org_id: OrgId` and `client_id_filter: ClientFilter` per D-208),
implement `ActionDeliveryEngine` with trigger evaluation for alert/case/schedule/manual
modes, implement built-in destination types (webhook, email, syslog), implement WASM plugin
action delegation via `PluginRuntime` (S-1.15), implement at-least-once delivery with
exponential backoff retry (2s/4s/8s/16s/32s + ±10% jitter per ADR-016 §2.6), track
delivery state in RocksDB `action_state` CF (org-prefixed per ADR-008), enforce per-action
rate limiting and deduplication windows, execute scheduled report queries through the
action delivery's own 8-permit semaphore (D-209 / ADR-016 §2.11), use `prism-siem-formats`
crate for CEF/LEEF syslog encoding (ADR-019 / D-212), and participate in hot reload per
ADR-018. This story depends on S-4.01 Task 0 for `prism-operations` crate workspace
registration (DRIFT-408-001).

## Tasks

1. Define `ActionSpec` and related types in `crates/prism-operations/src/action/mod.rs`:
   - `ActionSpec { org_id: OrgId, action_id, name, trigger: TriggerMode, filter: Option<String>, client_id_filter: ClientFilter, case_events: Vec<CaseEvent>, schedule: Option<String>, destination: DestinationConfig, credentials: HashMap<String, CredentialRef>, plugin_config: HashMap<String, String>, rate_limit: RateLimitConfig }` — per D-208 / ADR-016 §2.5
   - `ClientFilter` enum: `All` (from TOML `clients = ["*"]`) | `Specific(Vec<ClientId>)` (non-empty vec; enforced at validation per D-210)
   - `TriggerMode`: `Alert | Case | Schedule | Manual`
   - `CaseEvent`: `Created | StatusChanged | DispositionSet | AlertLinked | AnnotationAdded | AssigneeChanged`
   - `DestinationConfig`: `Webhook(WebhookConfig) | Email(EmailConfig) | Syslog(SyslogConfig) | Plugin(PluginDestConfig)`
   - `RateLimitConfig { max_per_hour: u32, cooldown_seconds: u32, deduplicate_window_seconds: u32 }` — defaults: 100/hr, 30s, 300s
   - `ActionRegistry`: `HashMap<String, ActionSpec>` behind `arc_swap::ArcSwap`
   - `ActionDeliveryEngine`: holds `ActionRegistry`, `PluginRuntime` reference, `QueryEngine` reference, `RocksDB` reference for `action_state` CF, `Arc<Semaphore>` (8 permits, action-delivery-private — NOT shared with schedule executor per D-209 / ADR-016 §2.11)

2. Implement TOML spec loader in `crates/prism-operations/src/action/loader.rs`:
   - Load `*.action.toml` files from `{config_dir}/actions/` directory
   - Validate required fields: `action_id`, `trigger`, `[action.destination]` with valid `type`
   - Validate credential references use `{ source = "env", key = "..." }` form — reject inline string values with `E-ACTION-001: "Credential '{field}' in action '{id}' must use a reference (source = 'env', key = 'KEY_NAME'), not an inline value."`
   - Validate `trigger = "case"` requires at least one entry in `case_events`
   - Validate `trigger = "schedule"` requires a valid 5-field cron expression (parse with `croner` crate: `croner::Cron::new(expr).without_seconds().parse()`). 6-field expressions (seconds precision) are rejected at load time with `E-ACTION-CRON-SECONDS-NOT-SUPPORTED`.
   - Validate `clients` entries: empty list `[]` is REJECTED with `E-ACTION-CLIENTS-EMPTY` (D-210); `["*"]` → `ClientFilter::All`; non-empty list → `ClientFilter::Specific(vec)`
   - Validate `org_id` is a valid UUID v7 (required per D-208)
   - On validation error, reject the spec — do not partially register

3. Implement trigger evaluation in `crates/prism-operations/src/action/trigger.rs`:
   - `AlertTrigger`: subscribes to the alert broadcast channel (from S-4.05, capacity 1000 / effective 1024 per R-12). For each alert:
     - Handle `RecvError::Lagged(n)` by incrementing `metrics::counter!("action_delivery.broadcast.lagged", n)` and continuing (per ADR-016 §2.12 / R-12) — do NOT break
     - Find all `ActionSpec` with `trigger = Alert` whose `org_id` matches `alert.org_id`
     - Apply `client_id_filter`: `ClientFilter::All` passes all; `ClientFilter::Specific(ids)` skips if `alert.client_id` not in list
     - Evaluate `filter` expression against alert fields using the DataFusion predicate evaluator already used in detection rules — reuse `prism-operations::detection::filter::evaluate_predicate`
     - If filter matches: check rate limit → render template → deliver
   - `CaseTrigger`: subscribes to the case event channel (new tokio broadcast channel,
     separate from the alert channel). For each case event (created, status_changed, etc.):
     - Handle `RecvError::Lagged(n)` by incrementing `metrics::counter!("action_delivery.broadcast.lagged", n)` and continuing
     - Find all `ActionSpec` with `trigger = Case` and matching `case_events` entry; apply org_id guard
     - Apply `client_id_filter` and `filter` expression (evaluated against case fields)
     - If matches: check rate limit → render template → deliver
   - `ScheduleTrigger`: runs a 60-second default tick loop (configurable via PRISM_SCHEDULER_TICK_SECS, range 10-3600s) (tokio interval). For each tick:
     - Evaluate all `trigger = Schedule` action specs' cron expressions against current UTC time using `croner::Cron::new(expr).is_time_matching(&now)?` (per R-2 / ADR-016 §2.2 / ADR-013 §2.8)
     - Tick-loop dedup: since `tokio::time::interval` jitter can cause double-fires at minute boundaries, check that the spec's `last_cron_fire_minute` differs from current minute before firing
     - Fire matching specs via `ActionDeliveryEngine::fire_schedule(spec, org_id, client_id_filter)`
   - `ManualTrigger`: called directly by the `fire_action` MCP tool (S-5.06)

4. Implement built-in destination types in `crates/prism-operations/src/action/destinations/`:
   - **webhook.rs**: `WebhookDestination::deliver(rendered_body: String, config: WebhookConfig) -> Result<(), DeliveryError>`
     - Resolve URL via `CredentialResolver` (opaque reference schemes per ADR-010 §2.3.1)
     - Use `reqwest = "0.12"` with features `["json", "rustls-tls", "cookies"]` (NOT default native-tls) per ADR-016 §2.9
     - Body templating uses `{{var}}` interpolation (handlebars-style per ADR-016 §2.9); all interpolated values pass through `prism-security::InjectionScanner::scan()` BEFORE template render (BC-2.09.004 / ADR-016 §2.9); inline credential values rejected (VP-046, `E-ACTION-INLINE-CRED`)
     - POST/PUT the rendered body with configured `Content-Type` header
     - HTTP 2xx → success; HTTP 4xx (except 429) → non-retryable `DeliveryError::Rejected`; HTTP 429 / 5xx / network error → retryable `DeliveryError::Transient`
   - **email.rs**: `EmailDestination::deliver(rendered_body: String, config: EmailConfig) -> Result<(), DeliveryError>`
     - Resolve SMTP credentials via `CredentialResolver` (opaque reference schemes per ADR-010 §2.3.1 / ADR-016 §2.3)
     - Connect to SMTP using `lettre = "0.11"` with features `["tokio1", "smtp-transport", "rustls", "rustls-aws-lc-rs", "rustls-platform-verifier", "builder", "tracing"]` — the deprecated `tokio1-rustls-tls` feature MUST NOT be used (per R-3 / ADR-016 §2.8)
     - Default TLS: `Tls::Required` (STARTTLS on port 587); opt-in: `Tls::Wrapper` (implicit TLS on port 465)
     - Auth mechanisms list: `[Mechanism::Xoauth2, Mechanism::Plain]` default — XOAUTH2 is attempted first; Plain is attempted only as a fallback. `Mechanism::Login` available as opt-in via `smtp_auth_fallback = true` for legacy Microsoft tenants (per R-3 / ADR-016 §2.8)
     - **XOAUTH2 deployment note**: Microsoft Exchange Online deprecated basic SMTP auth on 2026-04-30. Operators targeting Exchange Online MUST configure XOAUTH2 with an Azure AD application having `SMTP.Send` permission; `tenant_id`, `client_id`, and `client_secret` stored as opaque `credential_ref` values
     - Build MIME message with subject, from, to, HTML or text body; for report actions: assembled report as body
   - **syslog.rs**: `SyslogDestination::deliver(alert: &Alert, config: SyslogConfig) -> Result<(), DeliveryError>`
     - Connect to syslog endpoint (UDP port 514 default / TCP port 514 / TLS port 6514 per config, per ADR-016 §2.10)
     - **DO NOT implement CEF/LEEF encoding inline.** Delegate to `prism-siem-formats` crate (ADR-019 / D-212):
       - `prism_siem_formats::cef::v0::Encoder::new().encode(&cef_event)?` → `Vec<u8>`
       - `prism_siem_formats::leef::v2::Encoder::new().encode(&leef_event)?` → `Vec<u8>`
     - Map OCSF `severity_id` to CEF severity 0–10 (0=Unknown, 1–3=1 Informational, 4=Low, 6=Medium, 8=High, 10=Critical) before constructing `CefEvent`
     - Transport boundary: ADR-019 owns the `Vec<u8>` format bytes; this file owns all socket I/O, framing, and TLS
   - **plugin.rs**: `PluginDestination::deliver(ctx: ActionContext, config: PluginDestConfig) -> Result<(), DeliveryError>`
     - Use `wasmtime = "44"` with `features = ["component-model"]`; use `wasmtime::component::bindgen!` macro reading WIT under `crates/prism-operations/wit/action-delivery.wit` (per R-13 / ADR-016 §2.7)
     - Per-fire instantiation: `Component::new(&engine, wasm_bytes)?` compiled at load; `bindgen!`-generated `instantiate(&mut store, &component, &linker)?` per delivery call for isolation
     - Async host functions via `Linker::func_wrap_async` when `fire_alert`/`fire_case`/`fire_report` need to await Tokio
     - Call `PluginRuntime::fire_alert(plugin_id, ctx, config)` or `fire_case` / `fire_report`
     - Map `PluginError::Trapped` / `PluginError::Timeout` → retryable `DeliveryError::Transient`
     - Map `PluginError::NotLoaded` → non-retryable `DeliveryError::Rejected`

5. Implement template rendering in `crates/prism-operations/src/action/template.rs`:
   - Template syntax: handlebars-style `{{variable_path}}` interpolation (ADR-016 §2.9). Note: BC invariant names retain `${}` notation for historical compatibility; the implementation uses `{{}}`.
   - Alert context variables: populate from `AlertContext` struct with all fields from the
     template variable table in `.factory/specs/architecture/actions.md` (AD-021): `alert.alert_id`, `alert.rule_id`, `alert.severity`, etc.
   - Case context variables: populate from `CaseContext` with all case fields
   - **UUID v7 validation for `{{case.alert_ids_quoted}}`** (BC-2.18.009 invariant): before interpolating this variable,
     validate each alert ID in the list matches UUID v7 format (regex or uuid crate parse).
     Drop any non-conforming values with `WARN` log: `"Dropping non-UUID v7 value from alert_ids_quoted in action '{action_id}': '{value}'"`.
     If all values are dropped, produce an empty string (query returns empty result set, not error).
   - Run `InjectionScanner` (S-1.10) on all `{{alert.*}}`, `{{event.*}}`, and `{{case.*}}` variable values
     before interpolation. Flag detected patterns as `_safety_flags` on the audit log entry. Internal-source variables (`{{prism_version}}`, `{{date}}`, `{{client_id}}`) are trusted and not scanned.

6. Implement rate limiting in `crates/prism-operations/src/action/rate_limit.rs`:
   - All `action_state` CF keys MUST be prefixed with `{org_id}:` per ADR-008 universal re-keying rule (DRIFT-408-002 / ADR-016 §2.5):
     - `max_per_hour`: counter key `"{org_id}:\x00:{action_id}:{hour_bucket}"`. Hour bucket = Unix timestamp / 3600.
     - `cooldown_seconds`: last-fire key `"{org_id}:\x01:{action_id}"` per ADR-016 §2.5 key table.
     - `deduplicate_window_seconds`: dedup key `"{org_id}:\x02:{action_id}:{idempotency_key}"` per ADR-016 §2.5.
   - **Dedup hash**: use SHA-256 with explicit null-byte delimiter to avoid collision: `sha2::Sha256::digest(format!("{}\0{}", rule_id, client_id))` — do NOT use plain string concatenation `rule_id + client_id` (collision-prone when either field contains the other as prefix)
   - When suppressed: audit-log the suppression with reason (rate limit / cooldown / dedup)
   - All `action_state` CF reads/writes use `StorageDomain::ActionState`

7. Implement at-least-once delivery with retry in `crates/prism-operations/src/action/retry.rs`:
   - On delivery failure (retryable `DeliveryError::Transient`):
     - Exponential backoff per ADR-016 §2.6: base 2s, multiplier 2x, cap 32s, max 5 attempts → nominal 2s, 4s, 8s, 16s, 32s; apply ±10% jitter per attempt
     - Write retry state to `action_state` CF (org-prefixed): `"{org_id}:\x04:{action_id}:{alert_id}"` → bincode-encoded `RetryState { attempt: u8, next_attempt_at: Timestamp, last_error: Option<String> }` (TTL 24h) per BC-2.18.001 v1.5 + ADR-016 §2.5 retry-state row
     - Use `tokio::time::sleep` between attempts (do NOT block the trigger evaluation loop)
   - After max retries: write failure record to `action_state` CF dead-letter key (org-prefixed per ADR-016 §2.5):
     `"{org_id}:\x03:{action_id}:{idempotency_key}"` → bincode-encoded `DeadLetterRecord` with final error and context (where `{idempotency_key}` resolves to `alert_id` for alert triggers per ADR-016 §2.4; per ADR-016 §2.5 v0.6 dead-letter row).
     Audit-log as `ActionDeliveryFailed` event with fields: `action_id`, `org_id`, `trigger_kind`, `destination_kind`, `attempts`, `last_error` (per ADR-016 §2.6). The alert is NOT lost (still in RocksDB `alerts` CF from S-4.05).
   - On success: delete the `retry` key from `action_state` CF.
   - Schedule triggers: no retry state — log failure and wait for next cron tick.
   - Manual triggers: return error immediately, no retry, no dead-letter.

8. Implement scheduled report execution in `crates/prism-operations/src/action/report.rs`:
   - For `trigger = Schedule` + `[action.destination.report]`: on cron fire:
     1. Acquire the action delivery semaphore (8 permits, action-delivery-private) via `try_acquire()` — if unavailable, log skip and emit `ActionDeliverySkipped { reason: SemaphoreExhausted }` audit event; return without retry (per D-209 / ADR-016 §2.11). This semaphore is NOT the schedule executor semaphore from S-4.01.
     2. For each query in `report.queries`:
        - Set dirty bit in `dirty_query_set` (same pattern as S-4.01 scheduled queries)
        - Execute via `QueryEngine::execute(query, client_id, 200MB_budget, 30s_timeout)`
        - On success: render section with `title` and query results as HTML table / markdown
        - On failure (timeout, OOM, error): include error section: `"[Section '{title}' failed: {error}]"`
        - Clear dirty bit
     3. Assemble full report from all sections (partial on error per INV-ACTION-005)
     4. Deliver assembled report via the destination (email or webhook)

9. Implement cron tick loop in `crates/prism-operations/src/action/cron.rs`:
   - `CronScheduler::run(registry: Arc<ActionRegistry>, engine: Arc<ActionDeliveryEngine>)`:
     - `tokio::time::interval(Duration::from_secs(prism_scheduler_tick_secs))` tick — where `prism_scheduler_tick_secs` is read from env `PRISM_SCHEDULER_TICK_SECS` (default 60, range [10, 3600])
     - For each tick: iterate over `Schedule`-trigger actions, evaluate cron expression against current UTC time using `croner::Cron::new(expr).without_seconds().parse()?.is_time_matching(&now)?` (per R-2 / ADR-016 §2.2 / ADR-013 §2.8 canonical decision). Strict 5-field cron only — any expression with 6 fields (seconds precision) is rejected at spec-load time with `E-ACTION-CRON-SECONDS-NOT-SUPPORTED`.
     - Tick-loop double-fire dedup: maintain `last_fired_minute: HashMap<action_id, u64>` in-memory; skip fire if current minute == last_fired_minute for this action
     - Fire matching actions via `ActionDeliveryEngine::fire_schedule(spec, org_id, client_id_filter)`

10. Participate in hot reload (AD-018):
    - Watch `{config_dir}/actions/*.action.toml`
    - On change: re-validate, arc-swap `ActionRegistry`
    - In-flight deliveries continue using the old spec; next trigger fires use new spec
    - Plugin hot reload is handled by `PluginRuntime` (S-1.15) independently

11. Emit case event channel:
    - `ActionDeliveryEngine` provides a `case_event_sender: tokio::sync::broadcast::Sender<CaseEvent>`
    - S-4.06 (Case Management) must call `action_delivery_engine.emit_case_event(event)` after every

    - This requires a minor modification to S-4.06's `CaseManager` to accept an optional

13. Write VP-044 Kani proof in `crates/prism-operations/src/proofs/retry_state_machine.rs`:

14. Write VP-045 proptest in `crates/prism-operations/src/proofs/schedule_semaphore.rs`:

    - Assert error message contains field name but NOT field value

**AC-1:** Given a `slack_soc.action.toml` with `trigger = "alert"` and `filter = 'severity_id >= 4'`,

## Architecture Compliance Rules

Per ADR-016 (Action Delivery Framework) and ADR-019 (SIEM Output Formats):

- Alert delivery MUST complete persistence to RocksDB `action_state` CF before returning
  success. A delivery that claimed success but did not persist retry state would violate
  at-least-once guarantees on restart.
- `ActionDeliveryEngine::fire_schedule` MUST use `try_acquire()` on the action-delivery-private
  8-permit `Arc<Semaphore>` (constructed in `action/delivery.rs`, NOT exported). Using
  `acquire()` would block the cron tick loop stalling ALL action delivery. This semaphore
  is completely separate from the schedule executor semaphore in `schedule/executor.rs`
  (D-209 LOCKED / ADR-016 §2.11). If `prism-operations` gains a dependency on the
  schedule executor semaphore, the build MUST fail (module-private enforcement).
- All `action_state` CF keys MUST be prefixed with `{org_id}:` per ADR-008. Keys without
  this prefix are a multi-tenant isolation violation. Pattern: `{org_id}:\xNN:{action_id}:...`.
- Action specs MUST carry `org_id: OrgId`. Trigger evaluation MUST match `alert.org_id ==
  spec.org_id` before delivering. Cross-org delivery is a security violation (D-208 / ADR-016 §2.5).
- `clients = []` MUST be rejected at load time with `E-ACTION-CLIENTS-EMPTY` (D-210 / ADR-016 §2.5).
  `clients = ["*"]` → `ClientFilter::All`; any non-empty list → `ClientFilter::Specific`.
- Case event emission from S-4.06 MUST happen after the case mutation is persisted to
  RocksDB. Emitting the event before persistence creates a race where the action engine
  reads stale case data.
- Template rendering MUST run `InjectionScanner` on all `{{alert.*}}`, `{{event.*}}`, and
  `{{case.*}}` variables before interpolation. Variables from internal sources
  (`{{prism_version}}`, `{{date}}`, `{{client_id}}`) are trusted and not scanned.
- `{{case.alert_ids_quoted}}` UUID v7 validation MUST occur before interpolation, not after.
  The validation protects the generated PrismQL query string from injection.
- The cron tick loop (1-second interval) MUST be a separate tokio task from the alert
  trigger subscriber. They must not block each other.
- WASM plugin actions use `PluginRuntime::fire_alert` / `fire_case` / `fire_report`.
  These calls MUST NOT be made on the trigger evaluation loop's task — spawn a new
  `tokio::task` for each plugin invocation to isolate plugin CPU time limits from the
  trigger loop.
- Syslog CEF/LEEF encoding MUST use `prism-siem-formats` crate (ADR-019 / D-212). DO NOT
  implement CEF or LEEF format logic inline in `syslog.rs`. The boundary is: ADR-019 owns
  `Vec<u8>` format bytes; `syslog.rs` owns socket I/O and framing only.
- SMTP delivery MUST NOT use the deprecated `tokio1-rustls-tls` lettre feature flag
  (R-3 / ADR-016 §2.8). Use `rustls` + `rustls-aws-lc-rs` + `rustls-platform-verifier`.
- `keyring://` credential scheme is REJECTED AT LOAD TIME with `E-ACTION-KEYRING-DEFERRED`
  until S-1.07 wires up the OS keyring backend (TD-S-1.07-01 P1 blocks Wave 5 close). Any
  `.action.toml` whose `credential_ref` uses the `keyring://` scheme must fail validation
  immediately — do NOT accept the spec and defer resolution; accepting it would allow an
  action to be registered that can never deliver (per ADR-016 §2.3 v0.2).
- Forbidden dependency: `prism-siem-formats` MUST NOT depend on `prism-core`. If this
  dependency appears in `crates/prism-siem-formats/Cargo.toml`, the build MUST fail.
- **WIT type synchronization (ADR-016 §2.7.1):** `cargo check` in `prism-operations` MUST
  fail if WIT record definitions in `crates/prism-operations/wit/action-delivery.wit` drift
  from the canonical Rust struct shapes in `crates/prism-operations/src/types/`. Enforcement
  is via `cargo-component` or a `bindgen!`-generated output diff'd against canonical Rust
  types. CI runs this check on every PR touching the WIT file or the types module. Drift is
  a compile error, not a runtime failure.
- **LEEF Custom Delimiter forbidden set (ADR-019 §5):** `Delim::Custom(c)` construction MUST
  reject the following characters with `FormatError::InvalidAttributeKey(reason: "forbidden delimiter")`:
  `\t` (tab), `|` (pipe), `^` (caret), `\n` (newline), `\r` (carriage return), `=` (equals),
  `"` (double-quote), `\0` (NUL). Any other ASCII printable or single-codepoint Unicode
  character is allowed. This forbidden set is exhaustive and must not be silently extended.
- **ARCH-INDEX SS-18 update (ADR-019 §9):** In the same burst that introduces `prism-siem-formats`,
  state-manager MUST update ARCH-INDEX.md SS-18 (Action Delivery Engine) subsystem entry to
  include `prism-siem-formats` in the crate list. Story-writer cross-references this obligation;
  the actual ARCH-INDEX edit is state-manager's responsibility.
- **Subsystems list confirmed (ADR-016 v0.3):** `subsystems: [SS-12, SS-13, SS-14, SS-18]`
  matches ADR-016 v0.3 `subsystems_affected: [SS-18, SS-12, SS-13, SS-14]` (order-independent).
  Verified per architect Pass 2 fix (P2-S-4.08-A-H-001).

## Previous Story Intelligence

**S-4.01 (Schedule CRUD and Execution Loop) — CRITICAL PREREQUISITE (DRIFT-408-001):**
S-4.01 Task 0 creates the `prism-operations` crate and registers it in the workspace
`Cargo.toml`. This story cannot be dispatched until S-4.01 Task 0 is complete. S-4.01
also owns the *schedule executor* semaphore (`Arc<Semaphore>` with 8 permits per D-209).
This story does NOT share that semaphore — `ActionDeliveryEngine` constructs its own
independent 8-permit semaphore at init time inside `action/delivery.rs` (ADR-016 §2.11 /
D-209 LOCKED). The two semaphores are module-private and structurally disjoint (VP-137 +
VP-143 verify this).

**S-4.05 (Alert Generation):** established the `tokio::sync::broadcast` alert channel
with capacity 1,000 (effective 1,024 power-of-two ring per R-12). `AlertTrigger` subscribes
using `broadcast::Receiver`. Handle `RecvError::Lagged(n)` by incrementing
`metrics::counter!("action_delivery.broadcast.lagged", n)` and continuing — do NOT break.
`RecvError::Closed` → break and shut down.

**S-4.06 (Case Management):** owns case mutations. This story adds a `case_event_sender`
parameter to `CaseManager`. If S-4.06 was implemented without this hook, a targeted
modification to `CaseManager::update_case` and `CaseManager::create_case` is needed to
emit events after persistence. The case event channel is a separate `tokio::sync::broadcast`
channel — handle `RecvError::Lagged` the same way as the alert channel.

**S-4.03 (Detection Rule Loading):** owns the DataFusion filter expression evaluator. The
same `evaluate_predicate(expr: &str, context: &HashMap<String, Value>) -> Result<bool>`
function used for detection rule filters can be reused for action filter evaluation.
Import from `prism-operations::detection::filter`.

**S-1.15 (WASM Plugin Runtime):** provides `PluginRuntime`. Construct it as part of

## Error Taxonomy

The following error codes are introduced or confirmed by this story. Each must be added to
`prd-supplements/error-taxonomy.md` (or the equivalent project error registry) as part of
the S-4.08 implementation burst. **Task:** implementer must open a sub-task or add a
checklist item to update the error taxonomy file with these entries.

| Error Code | Trigger | Retryable | Description |
|------------|---------|-----------|-------------|
| `E-ACTION-001` / `E-ACTION-INLINE-CRED` | Spec load | No | Credential field contains a literal value instead of an opaque reference scheme (`vault://`, `env://`, `file://`). Field name included in message; value NEVER included. |
| `E-ACTION-002` | Spec load | No | Duplicate `action_id` detected across two `.action.toml` files. Both file paths included in message. |
| `E-ACTION-CLIENTS-EMPTY` | Spec load | No | `clients = []` empty list — ambiguous intent; must use `["*"]` for all clients (D-210 LOCKED). |
| `E-ACTION-TOML-PARSE` | Spec load | No | Syntactically invalid TOML in `.action.toml`. File path included; other specs continue loading. |
| `E-ACTION-TEMPLATE-INJECTION` | Delivery | No (abort) | `InjectionScanner` detected injection pattern in a template variable value. Delivery aborted; `_safety_flags` set on audit entry. |
| `E-ACTION-TEMPLATE-INJECTION-UNTYPED` | Spec load | No | `{{var}}` expression in webhook body or email subject is neither a UUID v7 reference nor a recognized explicitly typed scalar binding. |
| `E-ACTION-CRED-RESOLVE-FAILED` | Delivery | No | Credential reference resolved to a missing or unreadable value at delivery time (e.g., `env://UNSET_VAR` with unset variable, `file://` with missing file). Classified as `DeliveryError::Rejected` — the credential will not appear between retries. **NEW — add to error taxonomy (per ADR-016 §2.3 v0.2, finding P1-S-4.08-A-M-003).** |
| `E-ACTION-KEYRING-DEFERRED` | Spec load | No | `credential_ref` uses `keyring://` scheme, which is unimplemented until S-1.07 completes (TD-S-1.07-01 P1). Spec rejected at load time to prevent registering an action that can never deliver. **NEW — add to error taxonomy (per ADR-016 §2.3 v0.2, finding P1-S-4.08-A-M-003).** |
| `E-ACTION-CRON-SECONDS-NOT-SUPPORTED` | Spec load | No | Cron expression has 6 fields (seconds precision). Only 5-field cron is supported (per P1-S-4.08-A-L-004 fix). |

## Dev Notes

- Syslog CEF format: the severity field maps OCSF `severity_id` to CEF severity 0–10:
  `0=Unknown, 1–3=1 Informational, 4=Low, 6=Medium, 8=High, 10=Critical`. Encoding is
  handled by `prism_siem_formats::cef::v0::CefEvent::new(...)` — do NOT inline CEF format strings.
- The lettre SMTP client supports both STARTTLS (port 587, `Tls::Required`) and implicit
  TLS (port 465, `Tls::Wrapper`). Do NOT use the deprecated `tokio1-rustls-tls` feature.
  Auth mechanism default: `[Xoauth2, Plain]`. Microsoft Exchange Online has deprecated
  basic SMTP auth (effective 2026-04-30); XOAUTH2 is mandatory for Exchange Online operators.
- For Jira and PagerDuty action types, the architecture spec designates these as
  `type = "plugin"` destinations using `.prx` WASM plugins. The built-in `webhook` type
  covers Slack, Teams, Discord, and generic HTTP endpoints. Do NOT implement Jira or
  PagerDuty as built-in Rust types — they belong in plugins.
- Action IDs must be unique within the registry. Detect duplicate `action_id` across
  multiple `.action.toml` files at load time and reject with `E-ACTION-002:
  "Duplicate action_id '{id}' found in '{path2}' — already registered from '{path1}'."`
- `clients = []` (empty list) is REJECTED with `E-ACTION-CLIENTS-EMPTY` (D-210 LOCKED).
  `clients = ["*"]` is the explicit sentinel for "all clients in org" → `ClientFilter::All`.
  This is NOT a convention change from the original story — it is a locked architectural
  decision (D-210 / ADR-016 §2.5 / ADR-016 §4.2).
- Dedup hash: use `sha2::Sha256::digest(format!("{}\0{}", rule_id, client_id))` with a
  null-byte delimiter. Plain string concatenation `rule_id + client_id` is collision-prone
  when one field is a prefix of the other.
- `keyring://` credential scheme: REJECTED AT LOAD TIME with `E-ACTION-KEYRING-DEFERRED`
  until S-1.07 wires up the OS keyring (TD-S-1.07-01 P1). Do not accept specs using this
  scheme — they will never deliver until the keyring backend is implemented (ADR-016 §2.3).
- Cron scheduling uses `croner = "3"`, not `cron = "0.12.x"`. Use strict 5-field parsing:
  `croner::Cron::new(expr).without_seconds().parse()?.is_time_matching(&now)?`. 6-field
  (seconds-precision) expressions are rejected at spec-load time with `E-ACTION-CRON-SECONDS-NOT-SUPPORTED`.
- The `prism-siem-formats` crate will be created as part of this story per ADR-019.
  Add it to the workspace `Cargo.toml` members list (alongside `prism-operations` creation
  in S-4.01 Task 0). `prism-operations` depends on `prism-siem-formats = { path = "../prism-siem-formats" }`.
