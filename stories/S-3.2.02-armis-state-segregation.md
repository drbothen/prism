# S-3.2.02 — prism-dtu-armis: Multi-tenant state segregation

## Narrative

As a Prism MSSP operator managing multiple client organizations, I want the Armis DTU
clone to key its tag store by `(OrgId, device_id)` rather than bare `device_id`, so
that two clients whose Armis instances share asset IDs cannot read or overwrite each
other's tag state during concurrent query sessions.

## Objective

Migrate `prism-dtu-armis/src/state.rs` from `HashMap<String, V>` tag store keying to
`HashMap<(OrgId, String), V>` per ADR-008 §2.1 Step 6b. The immutable fixture
registries (`device_registry`, `devices_ordered`, `activity_fixture`, `alert_fixture`)
are NOT re-keyed — they are pre-populated read-only data. Add `reset_for(org_id)`,
rename `reset()` → `reset_all()`, add per-org proptest coverage (TD-DTU-MUTATE-COVERAGE-001).

## Tasks

1. **Migrate `tag_store` in `ArmisState`** (`src/state.rs` line 72):
   ```rust
   // Before
   pub tag_store: Mutex<HashMap<String, HashSet<String>>>,
   // After
   pub tag_store: Mutex<HashMap<(OrgId, String), HashSet<String>>>,
   ```

2. **Update call sites** — `add_tag`, `remove_tag`, `tags_for`:
   - Signature: `fn add_tag(&self, org_id: OrgId, device_id: &str, tag: &str)`
   - Key: `(org_id, device_id.to_owned())`
   - Thread `OrgId` through route handler state/extensions.

3. **Leave read-only fixture registries unchanged**:
   `device_registry`, `devices_ordered`, `activity_fixture`, `alert_fixture` are
   immutable fixture data loaded at construction time. They are NOT re-keyed per
   ADR-008 §2.1 Step 6b rationale.

4. **Add `DEFAULT_ORG_ID` test constant** (`#[cfg(test)]` only):
   ```rust
   #[cfg(test)]
   pub const DEFAULT_ORG_ID: OrgId = OrgId(uuid::uuid!("00000000-0000-7000-8000-000000000001"));
   ```
   Update all existing single-org unit tests.

5. **Add `reset_for(org_id: OrgId)`** and rename `reset()` → `reset_all()` with shim.

6. **Add `tests/multi_tenant.rs`** with proptest:
   - `prop_cross_org_tag_isolation` — store tag under org_A, assert org_B lookup is empty.
   - `prop_write_does_not_affect_other_org` — add tag for org_A, verify org_B unchanged.
   - `prop_reset_for_selectivity` — reset org_A, verify org_B intact.
   Kills OrgId-flipping mutations (TD-DTU-MUTATE-COVERAGE-001).

7. **Gate**: `cargo test -p prism-dtu-armis` green.

## Acceptance Criteria

**AC-001** — Cross-org tag lookup returns empty (traces to BC-3.2.001 postcondition 1):
Given `{"tag-A"}` is stored for `(org_id_A, "device-42")`,
When `state.tags_for(org_id_B, "device-42")` is called where `org_id_B ≠ org_id_A`,
Then the result is an empty `HashSet`.

**AC-002** — Write isolation (traces to BC-3.2.001 postcondition 2):
Given org_A writes tag `"compromised"` for `"device-42"`,
When org_B reads `"device-42"` tags,
Then org_B's result does not contain `"compromised"`.

**AC-003** — Independent state per org (traces to BC-3.2.001 postcondition 3):
Given `{"tag-A"}` stored for `(org_id_A, "device-42")` and `{"tag-B"}` for `(org_id_B, "device-42")`,
Then lookups for each org return their respective tag sets independently.

**AC-004** — Fixture registries unaffected (traces to BC-3.2.001 invariant 1):
Given `device_registry` and `devices_ordered` are read-only,
When the migration is applied,
Then those fields remain `HashMap<String, _>` (bare String key) and all existing
device-list route tests continue to pass without modification.

**AC-005** — `reset_for` is selective (traces to BC-3.2.001 edge case EC-004):
Given tags for both org_A and org_B,
When `state.reset_for(org_id_A)` is called,
Then org_A's tag store is empty and org_B's tags are unaffected.

**AC-006** — OrgId-flipping mutation killed (traces to BC-3.2.001 VP-079):
Given a proptest with distinct org_A and org_B and any shared device_id,
When state is written under org_A and read under org_B,
Then the result is always empty regardless of device_id value.

## Architecture Compliance Rules

- **Forbidden dependencies:** `prism-dtu-armis` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`.
- `DEFAULT_ORG_ID` MUST be `#[cfg(test)]` only. Any production reference is a
  compile error.
- Read-only fixture registries (`device_registry`, `devices_ordered`, `activity_fixture`,
  `alert_fixture`) MUST remain bare-String keyed — re-keying them is explicitly out
  of scope per ADR-008 §2.1 Step 6b.
- The composite key `(OrgId, String)` is the exclusive keying scheme for `tag_store`
  post-migration. No other mutable HashMap may use a bare String key.
- `BehavioralClone::reset()` must continue to delegate to `reset_all()`.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Two orgs share the same Armis device ID (common in large deployments) | Stored under separate `(OrgId, id)` keys; no collision |
| EC-002 | Lookup for org with no tag history | Returns `HashSet::default()` — no panic |
| EC-003 | Fixture registries are read before org context is resolved | Fixture data is global/shared; lookup does not require OrgId for read-only fixture fields |
| EC-004 | `reset_for(org_A)` while org_B concurrent write in flight | Mutex guards each operation atomically; no data corruption (one-writer-at-a-time) |
