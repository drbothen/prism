# S-3.4.02 — Migrate prism-dtu-armis tests to prism-dtu-harness

## Narrative

As a Prism integration-test maintainer, I want the `prism-dtu-armis` test suite
rewritten to use `prism-dtu-harness` instead of the single-tenant `ArmisClone::start()`
pattern, so that multi-tenant isolation is verified alongside Armis clone fidelity.

## Objective

Rewrite all existing `prism-dtu-armis` tests — 9 ACs, `reset_state_invariants`,
`fidelity_validator`, and 2 TD tests — to use `HarnessBuilder`. Single-customer tests
use `Harness::builder().with_customer("test-tenant").build()`. Multi-customer scenarios
exercise both isolation modes. All 9 original ACs must continue to pass.

## Tasks

1. Read existing `crates/prism-dtu-armis/tests/` to catalog all test functions (9 ACs, `reset_state_invariants`, fidelity test, 2 TD tests).

2. Rewrite `tests/harness_tests.rs`:
   - Replace `ArmisClone::start()` with `Harness::builder().isolation(IsolationMode::Logical).with_customer("test-tenant").build().await`
   - Use `harness.endpoints()` for clone address in test HTTP client
   - All 9 original ACs pass against harness-hosted clone

3. Add multi-customer logical isolation test:
   - Build 2-org harness: `"test-tenant"` + `"other-tenant"` (both Armis)
   - Query each org; assert pairwise-disjoint device sets

4. Add network isolation test:
   - `IsolationMode::Network`; cross-org credential mismatch → HTTP 401

5. Migrate `reset_state_invariants` test to use harness reset pattern (call `harness.reset_customer` or re-build harness between state-sensitive assertions).

6. Add `prism-dtu-harness` as `[dev-dependencies]` in `prism-dtu-armis/Cargo.toml`.

## Acceptance Criteria

**AC-001:** All 9 original `prism-dtu-armis` ACs pass when run against the harness-hosted clone in `IsolationMode::Logical`.
(traces to BC-3.5.001 postcondition 1)

**AC-002:** The `fidelity_validator` reports `checks_failed == 0` for all Armis endpoints under the harness.
(traces to BC-3.5.001 precondition 3)

**AC-003:** A 2-org logical harness returns pairwise-disjoint device sets for `"test-tenant"` and `"other-tenant"`.
(traces to BC-3.5.001 postcondition 2; TV-2)

(traces to BC-3.5.002 postcondition 2; TV-3)

**AC-005:** `reset_state_invariants` passes: after a state-mutating operation followed by reset, the clone's state is clean and all invariants hold.
(traces to BC-3.5.001 precondition 5 — initial state populated from multi-tenant generator)

**AC-006:** No test file in `prism-dtu-armis/tests/` directly instantiates `ArmisClone` outside the harness after migration.
