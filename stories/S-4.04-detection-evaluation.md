# S-4.04 — prism-operations: Detection Evaluation (Single/Correlation/Sequence)

## Narrative

As a Prism detection engine, I want to evaluate detection rules against incoming query
results using single-event, correlation, and sequence detection modes, so that analyst-
defined rules automatically fire alerts when suspicious patterns emerge across live sensor
data without requiring manual review of every query result.

## Objective

Implement the three detection evaluation modes (single-event predicate per record,
threshold-over-sliding-window correlation with reset-after-fire, and ordered multi-event
sequence matching with timeout), detection state persistence in RocksDB, alert
deduplication per match mode, and proper resource management ensuring `SessionContext` is
dropped on all error and panic paths.

## Tasks

1. Implement `DetectionEvaluator` in
   `crates/prism-operations/src/detection/evaluator.rs`:
   - Entry point: `evaluate(added_records: &[Row], rules: &[CompiledRule]) -> Vec<Alert>`
   - Input: `added_records` from S-4.02 `DiffResult.added` — removed records are NOT
     evaluated by the detection engine
   - For each rule: register `added_records` as the `events` DataFusion MemTable, execute
     the compiled SQL via the shared `SessionContext`, collect matches
   - `SessionContext` lifecycle: wrap in `scopeguard::defer!` to guarantee drop on all
     exit paths including panics

2. Implement single-event evaluation in
   `crates/prism-operations/src/detection/single.rs` (BC-2.13.002):
   - For batch predicate evaluation: plan the logical `Expr` to `PhysicalExpr` ONCE via
     `create_physical_expr(&expr, &df_schema)` at rule-compile time; reuse
     `PhysicalExpr::evaluate(&RecordBatch) -> ColumnarValue` across batches within the
     same evaluation context. There is no `Expr::evaluate` direct method in DataFusion
     53.x for full RecordBatch evaluation — do not reference it (ADR-015 §2.4, R-1).
   - Register `added_records` as events MemTable via `MemTable::try_new(schema, vec![vec![batch]])`
     + `Arc::new` + `ctx.register_table("events", Arc::new(table))` (canonical 53.x pattern;
     no `register_batch` shortcut exists in 53.x).
   - On match: call dedup check, then call alert generator (S-4.05)

3. Implement correlation evaluation in
   `crates/prism-operations/src/detection/correlation.rs` (BC-2.13.003):
   - Sliding window: window boundaries are anchored to `event_time` field of the incoming
     records, NOT to `now()`. This ensures replay and backtest correctness.
   - `group_by` key support: records are grouped by the specified field before threshold
     counting. Each group key tracks its own counter independently.
   - Per-rule group key cap: default 10,000 group keys per rule. When the cap is reached,
     new group keys are silently dropped (not queued). Log a warning with rule ID.
   - Reset-after-fire: once a correlation rule fires for a group key, that key's counter
     resets to 0. The window does not reset — only the count.
   - State persistence: correlation window state stored in RocksDB `detection_state` CF
     under keys `{org_id}:\x00:{rule_id}:{group_key}` (ADR-015 §2.8, ADR-008 universal
     re-keying rule; DRIFT-404-002 remediation)

4. Implement sequence evaluation in
   `crates/prism-operations/src/detection/sequence.rs` (BC-2.13.004):
   - Ordered multi-event pattern: `[event_A, event_B, event_C]` where each event is a
     predicate that must match in order
   - Partial match tracking: maintain a `SequenceTracker` per (rule, entity_id) pair that
     records which steps have been satisfied
   - Timeout: partial matches older than `sequence.time_window` from the first event in the
     sequence are evicted from state. Eviction runs on every scheduler tick — not only when
     new events arrive for a given entity (to bound state size).
   - Sequence fires when all steps are satisfied for the same entity within the time window
   - State persistence: sequence trackers stored in RocksDB `detection_state` CF under
     keys `{org_id}:\x01:{rule_id}` (ADR-015 §2.8; sequence state uses \x01 discriminator).
     Entries self-evict on the next compaction tick once their age from first event exceeds
     the rule's `time_window` (P1-S-4.04-A-M-003; reconciled with Task 5 eviction policy).

5. Create the `detection_state` RocksDB column family in
   `crates/prism-operations/src/detection/state.rs` (BC-2.13.012):
   - CF name: `"detection_state"`
   - Key encoding — OrgId prefix FIRST, then single-byte type discriminator, per
     ADR-015 §2.8 and ADR-008 universal re-keying rule (DRIFT-404-002):
     - `{org_id}:\x00:{rule_id}:{group_key}` — correlation window state
     - `{org_id}:\x01:{rule_id}` — sequence tracker state (auto-evict on next compaction
       tick if older than rule's `time_window` from first event; P1-S-4.04-A-M-003)
     - `{org_id}:\x02:{rule_id}:{dedup_key}` — alert deduplication entries
     - `\x03` — reserved for future use
   - **CRITICAL — discriminator collision resolved (P1-S-4.04-A-H-001):** Rate-limit
     counters are NOT in `detection_state` CF. They live in `action_state` CF (per ADR-016
     v0.2 §2.5). S-4.05 writes rate-limit counters to `action_state:{org_id}:\x00:{action_id}:{hour_bucket}`.
     Any code writing rate-limit state to `detection_state` CF is a CF-ownership violation.
   - The `{org_id}:` prefix enables `reset_for(org_id)` prefix-scan delete to cleanly
     remove all org-A state without touching org-B (ADR-008 compliance).
   - Eviction policy:
     - `\x00` (correlation group state): evict stale entries on spec reload (7-day default)
     - `\x01` (sequence tracker): auto-evict on next compaction tick if entry age from
       first event exceeds the rule's `time_window` (P1-S-4.04-A-M-003)
     - `\x02` (dedup): NOT evicted by age — they persist for the dedup window duration
     - `\x03` (reserved): not written

6. Implement alert deduplication in
   `crates/prism-operations/src/detection/dedup.rs` (BC-2.13.013):
   - `Single` mode dedup key: `blake3(rule_id + record_hash)` — a given record that
     already fired a rule won't fire again within the dedup window
   - `Correlation` mode dedup key: `blake3(rule_id + group_key + window_start_epoch)`
     — same group + window combination fires at most once
   - `Sequence` mode dedup key: `blake3(rule_id + entity_id + sequence_completion_epoch)`
     — same entity completing the same pattern fires at most once per completion
   - Dedup check: look up `{org_id}:\x02:{rule_id}:{dedup_key}` in `detection_state` CF;
     if present and not expired, suppress alert; if absent, write entry with TTL and proceed
   - Dedup window: use `effective_dedup_window` from `DetectionRuleCache` (resolved at
     rule load time per ADR-015 §2.7, D-211 LOCKED). Never resolve dedup window during
     per-evaluation hot path — only read from cache.
     **Eventual-consistency window (ADR-015 §2.7):** Cache invalidation is delivered
     asynchronously via the ADR-013 §2.7 `watch` channel; convergence ≤60s. In-flight
     evaluations may use a stale dedup window for at most one tick cycle — documented as
     acceptable bounded staleness per ADR-015 §2.7.

7. Write VP-027 proptest in `crates/prism-operations/props/vp027_dedup_keys.rs`:
   - Property: for any two distinct (rule, record) pairs, their single-mode dedup keys
     are different. For equal pairs, keys are identical.
   - Also verify: single, correlation, and sequence modes produce structurally distinct
     keys (different key prefixes or formats) so cross-mode collisions are impossible.

## Acceptance Criteria

**AC-1:** Given a single-event rule `severity_id >= 4`, When a record with
`severity_id = 5` is in `added_records`, Then an alert fires for that record.
(BC-2.13.002)

**AC-2:** Given a single-event rule `severity_id >= 4`, When a record with
`severity_id = 3` is in `added_records`, Then no alert fires. (BC-2.13.002)

**AC-3:** Given a correlation rule `count >= 5 within 10m group_by src_ip`, When 5
records with the same `src_ip` arrive within 8 minutes, Then an alert fires and the
counter for that `src_ip` resets to 0. (BC-2.13.003)

**AC-4:** Given a correlation rule with a group key cap of 10,000, When the 10,001st
distinct group key arrives, Then it is silently dropped (no alert, no error) and a
warning is logged. (BC-2.13.003)

**AC-5:** Given a sequence rule `[login_failure, login_failure, login_success]` with a
2-minute timeout, When three matching events arrive in order within 90 seconds for the
same `user_id`, Then the sequence fires. (BC-2.13.004)

**AC-6:** Given a sequence rule with a 2-minute timeout, When the third event arrives
3 minutes after the first, Then the sequence does NOT fire (timeout exceeded) and the
partial match tracker is evicted. (BC-2.13.004)

**AC-7:** Given detection state at epoch N persisted to RocksDB, When Prism restarts,
Then correlation windows and sequence trackers are recovered and evaluation continues
without resetting counters. (BC-2.13.012)

**AC-8:** Given a single-event alert that already fired for record R, When R appears
again in `added_records` within the dedup window, Then no duplicate alert fires.
(BC-2.13.013)

**AC-9:** VP-027 proptest passes — dedup keys are distinct for distinct inputs and
identical for equal inputs; cross-mode collisions do not occur.

**AC-10:** Given detection state persisted for org A (correlation windows, dedup entries),
When org B's rules are evaluated, Then org B's prefix-scan on `{org_b_id}:\x00:...` returns
no entries from org A. Org isolation in `detection_state` CF is enforced by the `{org_id}:`
key prefix (traces to ADR-015 §2.8 invariant / ADR-008 universal re-keying rule;
DRIFT-404-002 remediation).

## Architecture Compliance Rules

- Detection evaluation receives `added_records` ONLY from the diff engine. Removed
  records are never evaluated — a disappearing row is not a detection event.
- `SessionContext` MUST be wrapped in `scopeguard::defer!` before any fallible or async
  operation. This is the CI guarantee against context leaks (VP-036 validates this).
- Detection evaluation reuses the per-schedule `Arc<SessionContext>` returned by
  `QueryEngine::execute_scheduled()` (S-3.02) — it does NOT create a separate pool
  AND does NOT use a global shared pool. Each scheduled query gets its own ephemeral
  200MB `GreedyMemoryPool`; detection operates within that budget.
- VP-036 (SessionContext drop on error) is an integration test owned by S-6.07
  (prism-dtu-crowdstrike). S-4.04 implements detection evaluation with proper session
  cleanup; S-6.07's DTU provides the test vehicle via its FailureLayer fault injection.
- Correlation window timestamps MUST be anchored to `event_time` (record field), not
  `Instant::now()`. Using wall clock time breaks replay correctness and makes tests
  non-deterministic.
- Rate limit counter writes MUST be under `Mutex` (CI-004). Do not use atomic integers
  for rate limit state — the RocksDB write + in-memory counter combination requires
  mutual exclusion.
- All `detection_state` CF keys MUST be prefixed with `{org_id}:` BEFORE the type
  discriminator byte, per ADR-008 universal re-keying rule and ADR-015 §2.8. Key format:
  `{org_id}:\x00:{rule_id}:{group_key}` (correlation), `{org_id}:\x01:{rule_id}` (sequence),
  `{org_id}:\x02:{rule_id}:{dedup_key}` (dedup). `\x03` is reserved. Any key without the
  org prefix is a multi-tenant isolation defect (DRIFT-404-002 remediation).
- **Rate-limit counters MUST NOT be written to `detection_state` CF (P1-S-4.04-A-H-001 /
  ADR-016 v0.2 §2.5).** Rate-limit state lives exclusively in `action_state` CF, owned by
  the action delivery engine (S-4.05/S-4.08). The `detection_state` CF discriminator `\x01`
  is sequence tracker, NOT rate-limit. Writing rate-limit state to `detection_state` CF is
  a CF-ownership violation and a discriminator collision.
- **Workspace panic profile audit (P1-S-4.04-A-M-002):** The workspace `Cargo.toml` has no
  `[profile.release]` section and no `panic = "abort"` setting. Default is `panic = "unwind"`.
  `scopeguard::defer!` therefore runs on all panic paths in both debug and release builds.
  If a future PR adds `panic = "abort"` to the workspace profile, this compliance rule MUST
  be re-evaluated and explicit-drop fallback added before merging.
- Wave 4 detection evaluation explicitly excludes `removed_records`. The `DetectionEvaluator`
  MUST only process `added_records` from `DiffResult`. Detection rules triggering on record
  removal (e.g., "host went offline") are deferred to Wave 5+ and require a separate ADR
  before implementation (P1-S-4.04-A-M-004).
- Predicate evaluation in `single.rs` MUST use `create_physical_expr` + `PhysicalExpr::evaluate`
  (planned once, reused across batches). There is no `Expr::evaluate` method in DataFusion
  53.x — code referencing it will not compile (ADR-015 §2.4, R-1).
- The Mutex protecting `detection_state` writes is per-rule (`Mutex<()>` keyed by RuleId),
  NOT a single global Mutex (per QUAL-404-003 recommendation). Per-CF lock would serialize
  all detection evaluation across all rules.

## Dev Notes

- Correlation sliding windows use `event_time` anchoring. This means the window boundary
  for a record is `[record.event_time - window_duration, record.event_time]`. Records
  older than the window are excluded from the count even if they arrived recently. This
  enables correct replay but requires that `event_time` is populated in all records.
- The 10,000 group key cap per rule is the default. Rules with high-cardinality group
  keys (e.g., grouping by user_id in a large enterprise) may need this cap raised.
  Expose it as a per-rule configuration field in the `CorrelationSpec`.
- Sequence matching is inherently stateful and grows with the number of entities being
  tracked. The timeout eviction is critical to bounding state size. Ensure the eviction
  runs on every tick, not just when new events arrive for a given entity.
- Rate-limit counters are NOT in `detection_state` CF. They live in `action_state` CF
  (per ADR-016 v0.2 §2.5; P1-S-4.04-A-H-001 / P1-S-4.05-A-H-001 lockstep fix). This story
  creates `detection_state` CF and defines its key encoding (`\x00` correlation, `\x01`
  sequence, `\x02` dedup, `\x03` reserved). S-4.05 writes rate-limit counters to
  `action_state` CF under `{org_id}:\x00:{action_id}:{hour_bucket}`.
