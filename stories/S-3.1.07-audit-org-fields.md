# S-3.1.07 — prism-audit: add org_id + org_slug to AuditEntry; SHA-256 aql_hash

## Narrative
- **As a** compliance operator querying the audit log
- **I want** every audit entry to carry both `org_id: OrgId` (stable UUID) and
  `org_slug: OrgSlug` (slug at write time), and `aql_hash` to be SHA-256 (not
  DefaultHasher)
- **So that** forensic queries by UUID return all records for an org regardless of
  slug rename history, and AQL query hashes are cryptographically sound

## Description

ADR-006 §4 Step 5 adds `org_id: OrgId` and `org_slug: OrgSlug` as non-optional
required fields to `AuditEntry`. Both fields are persisted: the UUID is the stable
forensic key; the slug is denormalized for human readability at the time of write
(pre-rename records retain the old slug, providing a complete rename history per
BC-3.1.002 postcondition 5). All `emit_*` functions in `prism-audit` are updated to
accept both fields (typically via a context struct or paired params).

Additionally, tech debt item TD-ADR005-002 is resolved in this story: the `aql_hash`
field in audit emission is migrated from `std::collections::DefaultHasher` (non-deterministic
across processes) to `sha2::Sha256` (deterministic, cryptographically sound, suitable
for audit deduplication and forensic correlation).

## Acceptance Criteria

**AC-1 (traces to BC-3.1.002 postcondition 1):** `AuditEntry` struct has `org_id: OrgId`
as a required (non-`Option`) field. The field is present in all serialized audit records.
A unit test asserts that `serde_json::to_value(&entry)["org_id"]` is a non-null UUID
string.

**AC-2 (traces to BC-3.1.002 postcondition 2):** `AuditEntry` struct has
`org_slug: OrgSlug` as a required (non-`Option`) field. The field is the slug current
at write time. A unit test asserts `serde_json::to_value(&entry)["org_slug"]` equals the
slug passed to the `emit_*` function.

**AC-3 (traces to BC-3.1.002 postcondition 3):** Neither `org_id` nor `org_slug` is
null, empty, or omitted in any serialized audit record. A proptest generates `AuditEntry`
values and asserts both fields are present and non-empty.

**AC-4 (traces to BC-3.1.002 invariant 3):** `AuditEntry` construction fails at
compile time if either field is absent — both fields are required struct members with no
`Option` wrapper. Verified by attempting to construct `AuditEntry` without `org_id` in a
test that must NOT compile (use `#[compile_fail]` or document as a compile error test).

**AC-5 (traces to BC-3.1.002 postcondition 5):** A test simulating an org rename:
emit audit entry with `(org_id_A, slug="acme-corp")`; simulate rename to `slug="acme-na"`;
emit second audit entry with `(org_id_A, slug="acme-na")`. First entry retains
`org_slug="acme-corp"`; second entry has `org_slug="acme-na"`; both have `org_id=org_id_A`.
No retroactive update of the first entry.

**AC-6 (traces to BC-3.1.002 invariant 1 / TD-ADR005-002):** `aql_hash` in `AuditEntry`
is computed with `sha2::Sha256` (producing a 64-hex-char lowercase string). The field
value is deterministic across process restarts for the same input AQL string. A unit test
asserts `aql_hash("SELECT * FROM crowdstrike.devices")` equals the expected SHA-256
hex digest.

| EC-006 | `aql_hash` computed for the same AQL string in two separate processes | Identical SHA-256 digest — deterministic (TD-ADR005-002 fix) |

## Architecture Compliance Rules

| Rule | Source | Enforcement |
|------|--------|-------------|
| `org_id` and `org_slug` are non-`Option` required struct fields — no `Option<OrgId>` | BC-3.1.002 invariant 3 | Rust type system; code review |
