# S-3.4.01 — Migrate prism-dtu-claroty tests to prism-dtu-harness

## Narrative

As a Prism integration-test maintainer, I want the `prism-dtu-claroty` test suite
rewritten to use `prism-dtu-harness` instead of the single-tenant `ClarotyClone::start()`
pattern, so that multi-tenant isolation and failure injection scenarios are covered by the
same test file that validates Claroty clone fidelity.

## Objective

Rewrite all existing `prism-dtu-claroty` tests — 11 ACs, edge cases, `fidelity_validator`,
and 2 TD tests — to use `HarnessBuilder`. Single-customer tests use
`Harness::builder().with_customer("test-tenant").build()`. Multi-customer scenarios
exercise both `IsolationMode::Logical` (fast, in-process) and `IsolationMode::Network`
(real HTTP, catches routing bugs). All 11 original ACs must continue to pass. Add
harness-specific isolation ACs.

## Tasks

1. Read existing `crates/prism-dtu-claroty/tests/` to catalog all existing test functions (11 ACs, edge cases, fidelity test, 2 TD tests).

2. Rewrite `tests/fidelity.rs` (or rename to `tests/harness_tests.rs`):
   - Replace `ClarotyClone::start()` with `Harness::builder().isolation(IsolationMode::Logical).with_customer("test-tenant").build().await`
   - Use `harness.endpoints()` to get the clone address for the test HTTP client
   - All 11 original ACs must pass against the harness-hosted clone

3. Add multi-customer logical isolation test:
   - Build a 2-org harness: `"test-tenant"` (Claroty) + `"other-tenant"` (Claroty)
   - Query each org; assert device ID sets are pairwise disjoint
   - Uses `IsolationMode::Logical`

4. Add network isolation test:
   - Build a 2-org harness with `IsolationMode::Network`
   - Send `"test-tenant"` credentials to `"other-tenant"`'s endpoint
   - Assert HTTP 401 response (cross-org credential mismatch)

5. Keep `fidelity_validator` as-is (it validates response schema); update its `base_url` parameter to come from `harness.endpoints()`.

6. Remove direct `ClarotyClone::start()` calls from all test files; all clone access goes through harness.

7. Add `prism-dtu-harness` as a `[dev-dependencies]` entry in `prism-dtu-claroty/Cargo.toml` (with `features = ["dtu"]`).

## Acceptance Criteria

**AC-001:** All 11 original `prism-dtu-claroty` ACs (from S-6.08) continue to pass when run against the harness-hosted clone in `IsolationMode::Logical`.
(traces to BC-3.5.001 postcondition 1 — query scoped to registered org returns only that org's records)

**AC-002:** The `fidelity_validator` runs against the harness-hosted clone and reports `checks_failed == 0` for all Claroty endpoints.
(traces to BC-3.5.001 precondition 3 — all clone tasks running and bound to assigned ports)

**AC-003:** A 2-org logical harness (`"test-tenant"` + `"other-tenant"`) returns pairwise-disjoint device ID sets when each org is queried independently.
(traces to BC-3.5.001 postcondition 2; TV-2)

**AC-004:** A 2-org network harness with `IsolationMode::Network` — when `"other-tenant"` credentials are sent to `"test-tenant"`'s endpoint — returns HTTP 401, not HTTP 200.
(traces to BC-3.5.002 postcondition 2; TV-3)

**AC-005:** All existing edge case tests (malformed request, auth rejection, pagination) continue to pass after migration; no test is dropped.
(traces to BC-3.5.001 precondition 3 — harness provides same behavioral contract as direct clone)

**AC-006:** No test file in `prism-dtu-claroty/tests/` directly instantiates `ClarotyClone` outside the harness after this migration.
(traces to BC-3.5.001 precondition 4 — test client configured via `Harness::endpoints` table)
