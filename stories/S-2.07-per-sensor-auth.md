# S-2.07 — prism-sensors: Per-Sensor Auth and Pagination

## Narrative

As a Prism sensor adapter, I want complete per-sensor authentication and pagination
implementations for all four sensor platforms (CrowdStrike, Cyberint, Claroty, Armis),
so that the query engine can retrieve full result sets from each sensor API using the
correct auth flow and pagination strategy without manual integration.

## Objective

Implement the four platform-specific `SensorAdapter` concrete types with their full
authentication flows and pagination strategies: CrowdStrike OAuth2 with two-step fetch,
Cyberint cookie-based auth with multi-format timestamp parsing, Claroty bearer token with
polymorphic ID handling and offset-based hybrid pagination, and Armis bearer token with
AQL query forwarding and timestamp fallback chain.

## Tasks

1. Implement `CrowdStrikeAdapter` in `crates/prism-sensors/src/auth/crowdstrike.rs`:
   - OAuth2 token acquisition: `POST /oauth2/token` with `client_id` + `client_secret`
     form body → parse `access_token` + `expires_in` from JSON response (BC-2.01.005)
   - Token cache: `Arc<RwLock<Option<CachedToken>>>` where `CachedToken { token: String,
     expires_at: Instant }`; refresh if expired or on 401 response (BC-2.01.005)
   - Two-step fetch implementation for resource IDs → full details:
     - Step 1: `GET /queries/{resource_type}` → returns `resources: Vec<String>` (IDs)
     - Step 2: `POST /entities/{resource_type}/GET` with body `{ "ids": [...] }` → full records
     - Batch step-2 requests at 100 IDs per batch to avoid request size limits
   - Implements `SensorAdapter` trait

2. Implement `CyberintAdapter` in `crates/prism-sensors/src/auth/cyberint.rs`:
   - Cookie-based auth: `POST /login` with credentials → extract `Set-Cookie` session
     cookie → include in all subsequent requests (BC-2.01.006)
   - Cookie refresh: detect 401 response → re-authenticate → retry original request once
   - Multi-format timestamp parsing in `crates/prism-sensors/src/timestamp.rs`:
     - `parse_timestamp(s: &str) -> Result<DateTime<Utc>, SensorError>`
     - Try in order: RFC 3339 (`chrono::DateTime::parse_from_rfc3339`), Unix epoch
       (parse as `i64`, convert via `DateTime::from_timestamp`), custom format
       `"%Y-%m-%dT%H:%M:%S"` (BC-2.01.006)
     - Return `Err(SensorError::UnparseableTimestamp { raw: s.to_string() })` if all
       formats fail
   - Implements `SensorAdapter` trait

3. Implement `ClarotyAdapter` in `crates/prism-sensors/src/auth/claroty.rs`:
   - Bearer token auth: read static bearer token from credential store, include as
     `Authorization: Bearer {token}` header on all requests (BC-2.01.007)
   - Polymorphic ID handling: Claroty IDs may be integers (`42`) or UUIDs
     (`"550e8400-..."`); deserialize via `ClarotyId` enum:
     ```
     enum ClarotyId { Int(i64), Uuid(Uuid) }
     impl<'de> Deserialize<'de> for ClarotyId { ... }
     ```
     Serialize back to string for use in downstream fields (BC-2.01.007)
   - Offset-based hybrid pagination for audit logs in `crates/prism-sensors/src/pagination.rs`:
     - `OffsetCursor { offset: usize, total_count: usize, page_size: usize }`
     - `paginate_claroty(endpoint: &str, page_size: usize, client: &reqwest::Client)
       -> impl Stream<Item = Result<Vec<serde_json::Value>, SensorError>>`
     - First request: `GET {endpoint}?offset=0&limit={page_size}` → parse `total_count`
       from response
     - Subsequent requests: increment offset by page_size until `offset >= total_count`
       (BC-2.01.004)
   - Implements `SensorAdapter` trait

4. Implement `ArmisAdapter` in `crates/prism-sensors/src/auth/armis.rs`:
   - Bearer token auth: same pattern as Claroty (read from credential store, static
     bearer header) (BC-2.01.008)
   - AQL query forwarding: `SensorSpec` may contain an `aql_query: Option<String>` field;
     if present, pass directly to the Armis API as the `aql` parameter; if absent,
     construct a default AQL from the table name and filter pushdowns (BC-2.01.008)
   - Timestamp fallback chain for asset timestamps (BC-2.01.008):
     - Try `firstSeen` field → parse as DateTime
     - If absent or null, try `lastSeen` field
     - If absent or null, use `DateTime::now()` as a fallback
     - Log a `tracing::warn!` when the `now()` fallback is used (unexpected data gap)
   - Implements `SensorAdapter` trait

5. Register all four adapters in `AdapterRegistry` during startup:
   - Call `registry.register(Arc::new(CrowdStrikeAdapter::new(...)))` etc.
   - Startup registration happens in `prism-sensors/src/lib.rs::init_registry(cfg)`

6. Wire auth implementations to the sealed `SensorAuth` trait:
   - `CrowdStrikeAuth`, `CyberintAuth`, `ClarotyAuth`, `ArmisAuth` each implement
     `SensorAuth` (sealed) — concrete type is passed to `SensorAdapter::fetch()`
   - The `fan_out()` in S-2.06 resolves the correct `SensorAuth` subtype from the
     credential store based on `SensorType`; no runtime downcasting is needed

7. Write integration tests using `wiremock` to mock each sensor's API:
   - CrowdStrike: mock OAuth2 token endpoint + two-step entity fetch
   - Cyberint: mock login + data endpoint with Unix epoch timestamp
   - Claroty: mock paginated audit log endpoint (3 pages of 100 entries)
   - Armis: mock AQL query endpoint with `firstSeen` absent (tests fallback chain)

## Acceptance Criteria

**AC-1:** Given valid CrowdStrike credentials, When `CrowdStrikeAdapter::fetch()` runs,
Then it calls the OAuth2 token endpoint once, caches the token, and uses it for the
two-step entity fetch. (BC-2.01.005)

**AC-2:** Given a CrowdStrike token returning 401 on first use, When `fetch()` retries
authentication, Then the token endpoint is called again and the retry uses the new token.

**AC-3:** Given a Cyberint API response with timestamp field `"1710500000"` (Unix epoch),
When `parse_timestamp("1710500000")` is called, Then it returns the correct
`DateTime<Utc>` for Unix epoch 1710500000. (BC-2.01.006)

**AC-4:** Given a Claroty asset response with `"id": 12345` (integer), When deserialized
as `ClarotyId`, Then it is handled as `ClarotyId::Int(12345)` without error. (BC-2.01.007)

**AC-5:** Given a Claroty audit log endpoint with `total_count=500` and `page_size=100`,
When `paginate_claroty()` streams results, Then exactly 5 HTTP requests are made (offsets
0, 100, 200, 300, 400) and all 500 entries are returned. (BC-2.01.004)

**AC-6:** Given an Armis asset response with `firstSeen: null` and `lastSeen: "2024-03-15T10:00:00Z"`,
When the timestamp fallback chain runs, Then `lastSeen` is used as the timestamp.
Given both `firstSeen` and `lastSeen` are null, When the fallback chain runs, Then
`DateTime::now()` is used and a `tracing::warn!` is emitted. (BC-2.01.008)

## Architecture Compliance Rules

- Each sensor adapter MUST live in its own module under `auth/` — no shared auth
  code between adapters except through the sealed `SensorAuth` trait
- Token caches MUST use `Arc<RwLock<...>>` for interior mutability — `Mutex` is
  acceptable but `RwLock` is preferred for read-heavy token reuse
- The `ClarotyId` polymorphic deserializer MUST handle both integer and UUID inputs
  from the same JSON field without panicking — test with both types in unit tests
- AQL forwarding MUST pass the `aql_query` string verbatim — no modification,
  sanitization, or injection prevention at this layer (the query engine validates PrismQL
  before it reaches AQL conversion)
- `paginate_claroty()` MUST return a `Stream`, not collect into a `Vec` — this allows
  the caller to apply backpressure and avoids materializing the entire result set

## Dev Notes

- CrowdStrike's two-step fetch requires batching step-2 requests at 100 IDs — exceeding
  this limit causes API errors. The batch size is a `const CROWDSTRIKE_BATCH_SIZE: usize = 100`
  in the adapter module.
- Cyberint's `Set-Cookie` header must be preserved across redirects if `reqwest` follows
  them. Set `reqwest::ClientBuilder::cookie_store(true)` to enable automatic cookie
  management rather than manual header extraction.
- The `parse_timestamp` function should be in a shared `timestamp.rs` module (not inside
  `cyberint.rs`) because Armis also has timestamp handling — the fallback chain logic
  is distinct but the parsing step is reusable.
- `wiremock` integration tests require a running tokio runtime — use `#[tokio::test]`
  and ensure tests do not share mock server state (each test creates its own
  `MockServer::start().await` instance).
- The AQL forwarding spec (BC-2.01.008) does not require PrismQL-to-AQL translation
  in this story — that is handled by the query engine (prism-query). This story
  only forwards the pre-converted AQL string that arrives in `SensorSpec.aql_query`.
