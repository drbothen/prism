# S-3.3.02 — OrgRegistry boot from customers/*.toml at startup

## Narrative

As the Prism startup sequence, I want to load all `customers/*.toml` files,
validate them via `prism-customer-config`, and populate `OrgRegistry` with one
entry per file, so that org resolution is available before any MCP connection
is accepted and duplicate org identities are caught before the server starts.

## Objective

Wire `prism-customer-config::load_and_validate` into the Prism startup path
(before MCP transport bind). On success, call `OrgRegistry::register` for each
validated `CustomerConfig` in lexicographic file order. On any validation error,
write all errors to stderr and exit with code 1. Enforce bijectivity and duplicate
rejection via `BC-3.1.003` and `BC-3.1.004`.

## Tasks

1. Identify the startup entry point where MCP transport is bound (likely `prism-mcp/src/main.rs` or `prism-core/src/startup.rs`).
2. Add `prism-customer-config` as a dependency to the relevant crate.
3. Before MCP bind, call `prism_customer_config::load_and_validate(customers_dir)`:
   - `customers_dir` resolved from workspace root or `PRISM_CUSTOMERS_DIR` env var.
4. On `Err(errors)`: write each `error.to_string()` to stderr (one per line) in lexicographic file order; call `std::process::exit(1)`.
5. On `Ok(configs)`: iterate configs in order; call `OrgRegistry::register(slug, org_id)` for each.
6. If `register` returns `Err(RegistrationError::SlugConflict | IdConflict)`: write the error to stderr; `std::process::exit(1)`. (Duplicates should already be caught by `load_and_validate` via `E-CFG-011`/`E-CFG-012`; this is a defense-in-depth guard.)
7. After all registrations succeed: continue to MCP transport bind.
8. Add integration test `tests/startup_boot_test.rs` covering the scenarios in AC-001 through AC-005.

## Acceptance Criteria

**AC-001:** Given a `customers/` directory with two valid, distinct TOML files (different `org_id` and `org_slug`), the startup sequence registers both orgs in `OrgRegistry` and proceeds to bind the MCP transport without error.
(traces to BC-3.3.004 postcondition "On successful validation" clause 1; BC-3.1.003 postcondition clause 1)

**AC-002:** Given a `customers/` directory where two files share the same `org_id` UUID, the startup sequence writes `E-CFG-011` to stderr (naming both filenames), exits with code 1, and `OrgRegistry` contains zero entries.
(traces to BC-3.1.004 postcondition clause 2; BC-3.3.004 postcondition "On any validation failure" clause 1, 4)

**AC-003:** Given a `customers/` directory where two files share the same `org_slug`, the startup sequence writes `E-CFG-012` to stderr, exits with code 1, and `OrgRegistry` contains zero entries.
(traces to BC-3.1.004 postcondition clause 3; BC-3.3.004 postcondition clause 4)

**AC-004:** After successful boot with N valid customer files, `OrgRegistry` contains exactly N entries; for each entry, `resolve(slug) == Some(org_id)` and `slug_for(org_id) == Some(slug)` hold simultaneously (bijectivity invariant).
(traces to BC-3.1.003 postcondition clause 1; BC-3.1.003 invariant 1)

**AC-005:** Given a `customers/` directory with zero `.toml` files, the startup sequence registers nothing and proceeds normally — empty `OrgRegistry` is valid.
(traces to BC-3.3.004 precondition 5; EC-3.3.004-01)

**AC-006:** Validation errors from multiple files are all written to stderr before exit — the startup sequence does not stop at the first file error (multi-error pass).
(traces to BC-3.3.004 postcondition "On any validation failure" clause 2; EC-3.3.004-03)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|---------------|
| Startup orchestrator | `prism-mcp/src/main.rs` (or `prism-core/src/startup.rs`) | Effectful |
| `OrgRegistry::register` | `prism-core/src/org_registry.rs` | Pure (in-memory) |
| `load_and_validate` | `prism-customer-config/src/lib.rs` | Effectful |

Architecture compliance: ADR-010 §2.5 — validation MUST complete for ALL files before `OrgRegistry::register` is called for ANY file. The startup sequence calls `load_and_validate` (which validates all files) and only proceeds to registration if the result is `Ok`.

---

## Forbidden Dependencies

The startup orchestrator MUST NOT call `OrgRegistry::register` before `load_and_validate` returns `Ok`. Partial registration (registering some orgs before all are validated) violates ADR-010 §2.5 and BC-3.3.004 Invariant 1.

| Rule | Source | Enforcement |

| MCP transport not bound on error | BC-3.3.004 postcondition clause 1 | `load_and_validate` called before transport bind in startup sequence |
