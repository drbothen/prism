# S-5.04 — prism-mcp: Sensor Health Subsystem

## Narrative

As an MSSP analyst managing multiple clients and sensors, I want accurate, detailed
health information for each sensor — connectivity status, authentication validity, rate
limit state, and last successful query time — so that I can quickly identify and remediate
data gaps before they affect detection coverage.

## Objective

Implement the detailed sensor health monitoring subsystem: per-sensor connectivity
checks, authentication validity detection, rate limit state derived from HTTP headers,
last successful query timestamp tracking, and partial health reporting when only some
sensors are available.

## Tasks

1. Create `crates/prism-mcp/src/health/` module directory with:
   - `mod.rs`: `SensorHealthChecker` struct, `HealthCheckResult` type
   - `connectivity.rs`: per-sensor connectivity probe logic
   - `auth.rs`: authentication validity detection
   - `rate_limit.rs`: rate limit state detection from HTTP headers
   - `timestamp.rs`: last successful query timestamp tracking

2. Implement connectivity check in `health/connectivity.rs`:
   - For each sensor type, define a lightweight "probe" operation (e.g., CrowdStrike:
     `GET /sensors/queries/devices/v1?limit=1`; Claroty: version endpoint)
   - Probe MUST exercise the actual API endpoint — not a TCP ping or DNS check
   - Map probe results: 2xx → `up`, 5xx → `degraded`, connection error → `down`
   - Capture `latency_ms` using `tokio::time::Instant` around the request
   (BC-2.08.001)

3. Implement auth validity check in `health/auth.rs`:
   - Issue the same probe as connectivity check
   - HTTP 401 or 403 → `auth_invalid` (distinct from `down`)
   - Connection error (no HTTP response) → `{ connectivity: down, auth: unknown }`
   - Never conflate "unreachable" with "auth invalid" — these require different
     remediation actions (BC-2.08.002)

4. Implement rate limit detection in `health/rate_limit.rs`:
   - Track HTTP 429 responses per sensor per client
   - Extract `Retry-After` header (seconds or HTTP-date format, parse both)
   - Expose `RateLimitState { is_rate_limited: bool, retry_after: Option<DateTime<Utc>>,
     observed_at: DateTime<Utc> }`
   - Rate limit state clears automatically after `retry_after` time passes
   (BC-2.08.003)

5. Implement last successful query timestamp in `health/timestamp.rs`:
   - Maintain `HashMap<(TenantId, SensorType), DateTime<Utc>>` in `PrismContext`
   - Update on every successful sensor data fetch (not just health checks)
   - Expose via `SensorHealthChecker::last_successful_query(client_id, sensor_type)`
   - Persist timestamps in RocksDB `StorageDomain::Default` under a known key prefix
     so they survive server restarts (BC-2.08.004)

6. Implement partial health aggregation in `health/mod.rs`:
   - `HealthCheckResult::aggregate(results: Vec<SensorHealthResult>) -> OverallStatus`
   - `OverallStatus`: `healthy` (all up), `partial` (some up, some down/degraded),
     `unhealthy` (all down)
   - `partial` is a SUCCESS state, not an error — do not return an error response for
     partial health (BC-2.08.007)

7. Write tests in `crates/prism-mcp/tests/health.rs`:
   - Mock sensor returning 200 → `up` with latency
   - Mock sensor returning 401 → `auth_invalid`
   - Mock sensor returning 429 with `Retry-After: 60` → `is_rate_limited: true`,
     `retry_after` is 60s from now
   - Connection refused mock → `down`
   - 3 of 4 sensors up → `OverallStatus::Partial`
   - Last successful query timestamp persists across `SensorHealthChecker` restarts

## Acceptance Criteria

**AC-1:** Given a CrowdStrike sensor endpoint returning HTTP 200, When connectivity is
checked, Then `SensorHealthResult.status` is `up` and `latency_ms` is a non-zero value.
(traces to BC-2.08.001 postcondition 1)

**AC-2:** Given a sensor endpoint returning HTTP 401, When the health check runs, Then
`SensorHealthResult.status` is `auth_invalid` (not `down`), indicating that the sensor
is reachable but credentials are invalid.
(traces to BC-2.08.002 postcondition 1)

**AC-3:** Given a sensor returning HTTP 429 with header `Retry-After: 30`, When rate
limit state is checked, Then `is_rate_limited: true` and `retry_after` is approximately
30 seconds in the future.
(traces to BC-2.08.003 postcondition 1)

**AC-4:** Given a successful CrowdStrike query at 10:30 UTC, When
`last_successful_query("acme", CrowdStrike)` is called at 10:35, Then the returned
timestamp is 10:30 UTC.
(traces to BC-2.08.004 postcondition 1)

**AC-5:** Given a Prism restart, When `last_successful_query` is called after restart,
Then the persisted timestamp from before the restart is returned (not `None`).
(traces to BC-2.08.004 postcondition 2)

**AC-6:** Given a client with 3 sensors up and 1 sensor down, When
`HealthCheckResult::aggregate` is called, Then `OverallStatus` is `Partial` and the
tool response is a success (not an error).
(traces to BC-2.08.007 postcondition 1)

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Sensor health probes | `prism-mcp` (SS-08) + `prism-sensors` (SS-01) | Effectful |

`prism-mcp` exposes the `get_sensor_health` MCP tool; `prism-sensors` implements the
health probe logic (lightweight HEAD/ping requests per sensor, auth token validation,
latency measurement). Health probe dispatch and result aggregation are effectful (HTTP
I/O). Per `architecture/module-decomposition.md`, SS-08 (Sensor Health) is co-owned by
`prism-mcp` and `prism-sensors`.

## Architecture Compliance Rules

- Connectivity probes MUST use the same HTTP client and credential resolver used by the
  data source layer (S-2.07) — do not create a separate HTTP client in the health module
- `auth_invalid` and `down` MUST be distinct enum variants — tools that report
  remediation suggestions need to distinguish them
- Rate limit state MUST be derived from observed HTTP headers only — do not infer it
  from latency or other signals
- Timestamps MUST be stored in UTC — no local time
- The health aggregation MUST NOT return `PrismError` for partial health — partial is
  a valid operational state, not a failure

## Notes

- The choice of "probe" endpoint per sensor is important: it should be a real data
  endpoint (not `/healthz` or `/ping`) so that the probe validates end-to-end auth,
  not just network reachability. A `limit=1` query on the device or vulnerability
  endpoint works well for CrowdStrike.
- Rate limit clearing is time-based, not event-based. When `retry_after` elapses,
  the next health check will see a cleared rate limit state automatically — no explicit
  "reset" needed.
- For sensors that return 429 without a `Retry-After` header, default to `retry_after:
  60 seconds` as a conservative fallback.
