# S-3.7.04 — Armis fixture generator

## Narrative

As a test harness consumer, I want a `generate(org_id, SensorType::Armis, archetype, opts)`
implementation using the Rust types derived by S-3.7.00 from `armis-sdk-go/v2`, so that
integration tests can exercise all 8 archetypes including AQL query response shapes and
polymorphic asset IDs without live API calls.

## Acceptance Criteria

### AC-001 — All 8 archetypes produce correct baseline counts for Armis at scale=1.0 (traces to BC-3.4.003 postcondition table)

`generate(org, Armis, archetype, GenOpts::default())` returns `FixtureSet` with:
- `HealthyOtEnvironment`: 50 asset records, 5 alert records; all `status = "online"` or `"active"`
- `CompromisedEndpoint`: 50 asset records, 20 alert records; ≥3 with `severity_id >= 4`; ≥1 with anomalous lateral-movement indicators in status
- `AuthOutage`: 20 asset records; first call record has `status_code = 401`
- `LargeScale`: 10,000 asset records, 500 alert records
- `PaginationEdgeCases`: `default_page_size(Armis) × 3` asset records; exactly 3 AQL cursor values
- `SchemaDrift`: 30 asset records; `records[0]` violates Armis asset schema; `provenance.schema_valid = false`
- `HighChurn`: 200 asset records; ≥20 with `status = "tombstone"` or `deleted_at` present
- `DormantTenant`: 0 records; 0 cursors

(traces to BC-3.4.003 postcondition per archetype row)

### AC-002 — Generated records conform to `.references/schemas/armis/types.rs` shapes (traces to BC-3.4.002 postcondition 1 — Armis row)

In `#[cfg(test)]`, all non-SchemaDrift records serialise to JSON matching the Armis type
shapes derived in S-3.7.00: `ArmisAsset`, `ArmisAlert`, `AqlResponse<T>` structure.
Nullable fields use `null` values (not absent keys). Test panics on any structural violation.

### AC-003 — AQL query response shape is preserved (traces to BC-3.4.002 postcondition 1 — Armis row)

`PaginationEdgeCases` archetype records are wrapped in `AqlResponse<ArmisAsset>` envelope
matching the shape documented in `.references/schemas/armis/DERIVATION.md`. The pagination
cursor field is at the documented path in the response envelope.

### AC-004 — Polymorphic asset IDs handled via `ArmisId` (traces to BC-3.4.002 postcondition 1 — EC-001)

Asset records use the `ArmisId` type from `.references/schemas/armis/types.rs` which
handles both string and integer representations. Org-tagged ID format: `asset_id` contains
`"{org_slug}-{seed}-{i}"` as a substring (may be wrapped in `ArmisId` newtype serialised
as string). Integer-form IDs use a deterministic numeric hash of the slug+seed+index.

### AC-005 — Org-tagged IDs for assets and alerts (traces to BC-3.4.004 postcondition — Armis row)

Asset records: primary `asset_id` field contains `org_slug` as substring.
Alert records: primary `alert_id` starts with `"alert-{org_slug}-{seed}-"`.
Tombstone records: ID pattern `"dev-{org_slug}-{seed}-tomb-{n}"` (BC-3.4.004 postcondition — tombstone row).

### AC-006 — Determinism: two identical calls are byte-identical (traces to BC-3.4.001 postcondition 1)

No `thread_rng` or `SystemTime::now` in the call stack. Verified by CI static check and
unit test (two sequential calls, byte-level JSON comparison).

### AC-007 — OrgA and OrgB ID sets are disjoint (traces to BC-3.4.004 postcondition 3)

Proptest over distinct slug pairs: `generate(orgA).ids ∩ generate(orgB).ids = ∅`.

## Tasks

1. Read `.references/schemas/armis/types.rs` and `DERIVATION.md` — understand field
   shapes, nullable conventions, `ArmisId` serialisation, `default_page_size` value.
2. Implement `generate(org_id, archetype, opts) -> FixtureSet` in
   `crates/prism-dtu-armis/src/generator.rs` behind `#[cfg(feature = "fixture-gen")]`.
3. Build records as `serde_json::Value` matching `ArmisAsset` / `ArmisAlert` shapes from
   the derived types (do not depend on the `.references/` path at runtime; embed field
   structure in the generator).
4. Implement AQL envelope for `PaginationEdgeCases` using `AqlResponse<T>` shape.
5. Handle `ArmisId` duality: generate string-form IDs by default; generate integer-form
   IDs for every 5th record (to exercise polymorphic parsing downstream).
6. Implement `SchemaDrift`: remove a required field from `records[0]`.
7. In `#[cfg(test)]`, validate against derived JSON Schema (auto-generate from types.rs
   via `schemars` or use hand-authored schema from `DERIVATION.md`).
8. Write parameterised tests over all 8 archetypes.

## Architecture Compliance Rules

- Generator impl in `crates/prism-dtu-armis/src/generator.rs` only.
- `.references/schemas/armis/types.rs` is a read-only reference — do NOT import it as a Rust
  module in `prism-dtu-armis`. Embed field structure directly in the generator.
- Forbidden dependency: `prism-dtu-armis` must NOT depend on `prism-dtu-claroty`, `prism-dtu-crowdstrike`, or `prism-dtu-cyberint`.
- `#[cfg(feature = "fixture-gen")]` gate mandatory.

## Risks / Open Questions

  string prefixes, the org-tagging invariant for integer-form IDs requires a defined
  numeric encoding. Implementer must document the chosen encoding in `generator.rs` comments.
- S-3.7.00 must be complete before this story begins — `depends_on: [S-3.7.00]`.
