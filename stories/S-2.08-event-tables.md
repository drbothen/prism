# S-2.08 — prism-sensors: Event Table Abstraction and Local Buffering

## Narrative

As a Prism detection engine, I want sensor tables classified as either point-in-time
(live API fetch per query) or event-stream (locally buffered, queried from buffer) so
that scheduled detection queries against high-frequency event sources execute as fast
local RocksDB reads rather than live API round-trips — matching the osquery event
publisher/subscriber pattern.

## Objective

Distinguish the two table types in the sensor spec schema, implement a local RocksDB
`event_buffer` column family for event-stream tables, run a background polling task
per event-stream table that ingests and TTL-expires events, and route query execution
through the correct path (live API or buffer scan) transparently to all three PrismQL
modes.

## Tasks

1. Add `TableType` enum to `prism-sensors` (or `prism-core` if shared):
   ```rust
   pub enum TableType {
       PointInTime,    // fetch on every query
       EventStream,    // buffer locally, query from buffer
   }
   ```

2. Extend `[[sensor.tables]]` in sensor spec schema:
   ```toml
   [[sensor.tables]]
   table_name = "process_events"
   type = "event_stream"           # new field; default = "point_in_time"
   poll_interval = "60s"           # how often to call sensor API to ingest events
   retention = "24h"               # TTL for buffered events; default 24h
   ```
   Validate in `SpecParser`: `poll_interval` and `retention` are only valid when
   `type = "event_stream"`. `poll_interval` minimum: 10s. `retention` maximum: 7d.

3. Use the existing `EventBuffer` variant from `StorageDomain` (defined in S-1.01, initialized in S-2.01) in `prism-storage`:
   - CF name: `"event_buffer"` — this becomes the 16th column family
   - Key format: `{sensor_id}/{table_name}/{client_id}/{timestamp_micros_be}/{ulid}`
   - Timestamp in big-endian bytes enables range scans in time order
   - Value: OCSF-normalized JSON blob per event record

4. Implement `EventBufferStore` in `crates/prism-sensors/src/event_buffer.rs`:
   - `write_events(sensor_id, table_name, client_id, records: Vec<NormalizedRecord>)
     -> Result<usize>`
   - `scan_events(sensor_id, table_name, client_id, since: SystemTime, until: SystemTime)
     -> Result<Vec<NormalizedRecord>>`
   - `evict_expired(sensor_id, table_name, retention: Duration) -> Result<u64>`
     — lazy eviction: called at read time before returning results; also called
     on schedule from the background poller after each ingest cycle

5. Implement background poller in `crates/prism-sensors/src/poller.rs`:
   - `EventPoller` struct: one instance per `(sensor_id, table_name, client_id)`
   - Spawned as a `tokio::task` by `AdapterRegistry::start_pollers()` at startup
   - Loop: sleep `poll_interval` → call sensor API (via existing `SensorAdapter::fetch`)
     → write to `EventBufferStore` → evict expired records → repeat
   - On API error: log WARN with sensor/table/client context, continue loop (do not crash)
   - On shutdown signal (`CancellationToken`): drain and exit cleanly
   - Configurable via `[query] event_poller_concurrency` (default 4 — max concurrent
     background pollers, shared semaphore)

6. Implement query routing in `crates/prism-sensors/src/fanout.rs`:
   - Before fan-out: check `TableSpec.table_type`
   - `PointInTime` → existing live API fetch path (unchanged)
   - `EventStream` → `EventBufferStore::scan_events()` for the query time range
   - Both paths return `Vec<RecordBatch>` to the query engine; routing is transparent
   - If `EventStream` table has no buffered data (cold start, poller not yet run):
     fall back to live API fetch once, log INFO "event_buffer cold-start fallback:
     {sensor}/{table}/{client}"

7. Inject `_source_type` virtual field in `materialization.rs` (or equivalent):
   - `_source_type: Utf8` — value `"live"` (PointInTime or cold-start fallback)
     or `"buffered"` (EventStream scan from RocksDB)
   - Added alongside existing `_sensor`, `_client`, `_source_table` virtual fields

8. Update `get_diagnostics(subsystem: "fanout")` to report per-table event_buffer stats:
   - `event_buffer_size_bytes` per `(sensor, table, client)`
   - `last_poll_time` and `last_poll_record_count`
   - `poller_status`: running | error | cold_start

## Acceptance Criteria

**AC-1:** Given a sensor spec with `type = "event_stream"` and `poll_interval = "60s"`,
When `AdapterRegistry::start_pollers()` is called at startup, Then a background
`EventPoller` task is spawned for that table and calls the sensor API approximately
every 60 seconds.

**AC-2:** Given buffered events in `event_buffer` CF for `(crowdstrike, process_events,
acme)`, When a PrismQL query targets `crowdstrike_process_events`, Then results are
served from RocksDB (no live API call) and `_source_type = "buffered"` in every row.

**AC-3:** Given a PointInTime table, When queried, Then the live API fetch path is taken
(unchanged behavior) and `_source_type = "live"`.

**AC-4:** Given events older than the table's `retention` period in the buffer, When
`evict_expired()` runs after an ingest cycle, Then those records are deleted from
RocksDB and do not appear in subsequent scans.

**AC-5:** Given `type = "event_stream"` with no prior poll (cold start), When a query
arrives, Then the engine falls back to a single live API fetch, writes results to the
buffer, and logs the cold-start fallback at INFO level.

**AC-6:** Given an event_stream poller that encounters an HTTP 429 from the sensor API,
When the error occurs, Then the poller logs WARN and continues the loop after the
configured retry backoff — it does NOT crash or stop the poller task.

**AC-7:** Given a sensor spec with `poll_interval = "5s"` (below 10s minimum), When
`SpecParser` validates the spec, Then validation fails with a descriptive error message
before the spec is loaded.

**AC-8:** Given a PrismQL query in any of the three modes (filter, SQL, pipe) against
an event_stream table, When the query executes, Then the routing is transparent —
the query result schema is identical regardless of source_type, with only
`_source_type` differing.

## Architecture Compliance Rules

- `event_buffer` is included in the 16-CF `StorageDomain` enum (S-1.01) and
  initialized by S-2.01. The existing CFs must not be renumbered.
- Timestamp keys in `event_buffer` MUST use big-endian byte encoding to enable
  lexicographic range scans in chronological order. Do NOT use little-endian.
- Background pollers share the same `reqwest::Client` and HTTP semaphore as query-time
  fetches — they are NOT exempt from the 200-connection global cap.
- The `_source_type` virtual field MUST be injected via the same virtual field
  injection path as `_sensor`, `_client`, and `_source_table` — not as a
  DataFusion UDF or a post-query transform.
- All three PrismQL modes MUST route through the same table-type dispatch logic.
  The parser and planner are unaffected; only the fetch phase differs.
- Event data in `event_buffer` is NOT audit-logged as a query result — it is
  infrastructure state. The audit log records the query execution, not the buffer
  internals.

## Dev Notes

- **Hot-reload poller lifecycle:** When a sensor spec is hot-reloaded (S-1.12/S-3.13),
  event pollers must be started/stopped accordingly:
  - Spec added with `type = "event_stream"`: spawn new EventPoller task
  - Spec removed: cancel the corresponding EventPoller via CancellationToken
  - Spec modified (poll_interval changed): cancel old poller, spawn new one
  This hooks into the same config-swap notification used by S-3.13 (dynamic table
  availability). Both table registration and poller lifecycle must update atomically.
- The osquery event publisher/subscriber pattern is the direct inspiration here:
  osquery's event publishers run continuously and write to a local SQLite backing store;
  subscribers (tables) expose that buffer as queryable tables. Prism's equivalent uses
  RocksDB instead of SQLite, which gives us better write throughput and TTL semantics.
- Detection engine benefits are significant: a 5-minute schedule checking process events
  goes from ~500ms live API round-trip to ~5ms local RocksDB range scan. This reduces
  detection latency and sensor API pressure simultaneously.
- `poll_interval` is a best-effort scheduling target — actual intervals may be slightly
  longer under load. This is acceptable for detection workloads (5-minute schedules
  tolerate ±5s jitter).
- Cold-start behavior (fall-back to live API on first query) means there is no hard
  dependency on the poller having run before queries can succeed. This is especially
  important for the first query after a restart.
