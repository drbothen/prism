# S-1.05 — prism-ocsf: Field Mapping and Normalization

## Narrative

As a Prism platform implementer, I want per-sensor field mapping implementations
in prism-ocsf, so that CrowdStrike, Cyberint, Claroty, and Armis raw records are
transformed into correctly populated OCSF DynamicMessage instances — with no silent
data loss, structured error reporting, and formally verified unmapped field preservation.

## Objective

Implement the four sensor-specific field mappers that plug into the `OcsfNormalizer`
infrastructure from S-1.04. Each mapper translates vendor-specific JSON fields to
OCSF paths, preserves unmapped fields in `raw_extensions`, and returns structured
errors on failure. VP-017 proptest verifies that no fields are silently dropped.

## Tasks

1. Define the `SensorMapper` trait in `crates/prism-ocsf/src/mappers/mod.rs`:
   ```rust
   pub trait SensorMapper: Send + Sync {
       fn sensor_id(&self) -> &'static str;
       fn record_types(&self) -> &'static [&'static str];
       fn map(
           &self,
           record_type: &str,
           raw: &serde_json::Value,
           msg: &mut DynamicMessage,
           extensions: &mut serde_json::Map<String, serde_json::Value>,
       ) -> Result<String, PrismError>; // Returns source record ID
   }
   ```
   The `extensions` map accumulates unmapped fields. The mapper moves fields from
   `raw` into either `msg` (known fields) or `extensions` (unknown fields). After
   the mapper returns, `OcsfNormalizer` writes `extensions` into `msg.raw_extensions`.

2. Implement `CrowdStrikeMapper` in `crates/prism-ocsf/src/mappers/crowdstrike.rs`
   (BC-2.02.003):
   - Record types: `detection`, `incident`, `event`
   - Detection field mapping:
     - `detection_id` → `finding_info.uid`
     - `severity` (string enum) → `severity_id` (via enum lookup: Critical=5, High=4, etc.)
     - `created_timestamp` → `time` (parse RFC3339)
     - `device.hostname` → `device.name`
     - `device.device_id` → `device.uid`
     - `device.local_ip` → `device.ip`
     - `device.os_version` → `device.os.version`
     - `behaviors[*].tactic` → `attacks[*].tactic.name`
     - `behaviors[*].technique` → `attacks[*].technique.name`
     - `ioc_type` + `ioc_value` → `evidences[0].data.type` + `evidences[0].data.value`
   - All unmapped top-level fields → `extensions`
   - Extract and return `detection_id` as the source record ID

3. Implement `CyberintMapper` in `crates/prism-ocsf/src/mappers/cyberint.rs`
   (BC-2.02.004):
   - Record types: `alert`, `ioc`
   - Alert field mapping:
     - `ref_id` → `finding_info.uid` (source record ID)
     - `title` → `finding_info.title`
     - `severity` → `severity_id`
     - `status` → `status_id`
     - `created_date` — multi-format timestamp parsing (try in order):
       1. RFC3339: `"2024-03-15T10:30:00Z"`
       2. ISO 8601 without timezone: `"2024-03-15T10:30:00"` (assume UTC)
       3. Unix timestamp integer: `1710498600`
       4. If all fail: emit structured error with field name and raw value
     - `threat_type` → `category_name`
     - `tags[*]` → `labels[*]`
   - All unmapped fields → `extensions`

4. Implement `ClarotyMapper` in `crates/prism-ocsf/src/mappers/claroty.rs`
   (BC-2.02.005):
   - 9 record types: `asset`, `alert`, `vulnerability`, `network_baseline`,
     `network_event`, `process`, `connection`, `insight`, `policy_violation`
   - Polymorphic ID handling: Claroty IDs may be integer OR string; always convert
     to string for OCSF uid fields: `uid = format!("claroty:{}", id_value)`
   - Asset mapping: `id` → `device.uid`, `name` → `device.name`,
     `ip_v4` → `device.ip`, `mac_address` → `device.mac_addr`,
     `firmware_version` → `device.hw_info.version`, `site_name` → `device.location.name`
   - Alert mapping: `id` → `finding_info.uid`, `alert_type` → `category_name`,
     `severity` → `severity_id`, `timestamp` → `time`
   - Vulnerability mapping: `id` → `finding_info.uid`, `cve_id` → `cve.uid`,
     `cvss_score` → `cve.cvss.base_score`
   - For record types not in the above list, return `Err(OcsfUnknownRecordType)`
   - All unmapped fields → `extensions`

5. Implement `ArmisMapper` in `crates/prism-ocsf/src/mappers/armis.rs`
   (BC-2.02.006):
   - 7 record types: `device`, `alert`, `activity`, `vulnerability`, `policy_violation`,
     `user`, `network_segment`
   - AQL query forwarding: Armis results may embed AQL metadata; forward as-is into
     `extensions["armis_aql_meta"]`
   - Timestamp fallback chain for all record types (try in order):
     1. `last_seen`
     2. `created_at`
     3. `timestamp`
     4. If none present: use current time and log a warning (never fail on missing time)
   - Device mapping: `id` → `device.uid` (convert int to string),
     `name` → `device.name`, `ipAddress` → `device.ip`,
     `macAddress` → `device.mac_addr`, `type` → `device.type`,
     `manufacturer` → `device.hw_info.manufacturer`
   - Alert mapping: `alertId` → `finding_info.uid`, `type` → `category_name`,
     `severity` → `severity_id`
   - All unmapped fields → `extensions`

6. Implement four-tier field alias resolution in `crates/prism-ocsf/src/alias.rs`
   (BC-2.02.008):
   - Tier 1 (Prism-specific metadata): direct lookup for `source_sensor`,
     `source_record_type`, `client_id` — these fields are set directly on the
     `OcsfEvent` wrapper, not in the `DynamicMessage`
   - Tier 2 (Proto descriptor fields): recursive descent into `DynamicMessage`
     with dot notation (e.g., `device.hostname`, `severity_id`); array indexing
     supported for paths like `attacks[0].technique.name`
   - Tier 3 (raw_extensions JSON): unmapped fields retained as JSON blob in
     `raw_extensions`; OCSF proto fields (tier 2) take precedence over
     `raw_extensions` fields with the same name
   - Tier 4 (None): field absent — not an error; resolution terminates
   - The first tier that produces a value wins; later tiers are not consulted
   - `AliasResolver::resolve(field: &str, event: &OcsfEvent) -> AliasResult`:
     - `AliasResult::PrismMetadata(value)` — tier 1 match
     - `AliasResult::ProtoField(value)` — tier 2 match
     - `AliasResult::RawExtension(value)` — tier 3 match
     - `AliasResult::Absent` — tier 4 (field not found in any tier)
   - Resolution order is deterministic; same input always produces same output

7. Integrate alias resolution into all four mappers: for each vendor field not in
   the explicit field mapping, call `AliasResolver::resolve()` before deciding it
   goes to extensions. This enables new vendor fields to be covered by alias tables
   without changing mapper code.

8. Write VP-017 proptest in `crates/prism-ocsf/src/tests/proptest_extensions.rs`:
   - Strategy: generate arbitrary JSON objects representing raw sensor records with
     a mix of known and unknown fields
   - Call the mapper with the generated record
   - If the call returns `Ok`, collect all field keys from the input JSON
   - Assert: every input key appears in EITHER the DynamicMessage (as a set field)
     OR the `extensions` map — no key is silently dropped (BC-2.02.007)
   - Property name: `prop_no_fields_silently_dropped`

9. Write unit tests in `crates/prism-ocsf/src/tests/`:
   - CrowdStrike: test full detection JSON → correct OCSF fields
   - Cyberint: test RFC3339 timestamp, Unix timestamp integer, malformed timestamp
   - Claroty: test integer ID → string uid conversion, unknown record type error
   - Armis: test timestamp fallback chain (no timestamps → uses current time + warning)
   - Alias: test tier 1, tier 2, and tier 3 paths explicitly

## Acceptance Criteria

**AC-1:** Given a CrowdStrike detection JSON with `"severity": "High"` and
`"detection_id": "ldt:abc123"`, When normalized, Then the DynamicMessage has
`severity_id = 4` and `finding_info.uid = "ldt:abc123"`. (BC-2.02.003)

**AC-2:** Given a CrowdStrike detection with `"behaviors": [{"tactic": "Discovery"}]`,
When normalized, Then `attacks[0].tactic.name == "Discovery"`. (BC-2.02.003)

**AC-3:** Given a Cyberint alert with `"created_date": 1710498600` (Unix timestamp),
When normalized, Then `time` is the correct RFC3339 representation of that timestamp.
(BC-2.02.004)

**AC-4:** Given a Cyberint alert with `"created_date": "not-a-date"`, When normalized,
Then the result is `Err(PrismError::OcsfNormalizationFailed)` containing the field name
and raw value. (BC-2.02.004, BC-2.02.011)

**AC-5:** Given a Claroty asset with `"id": 42` (integer), When normalized, Then
`device.uid == "claroty:42"`. (BC-2.02.005)

**AC-6:** Given an Armis device with no timestamp fields, When normalized, Then the
DynamicMessage has a valid `time` field (current time fallback) and a warning is logged.
(BC-2.02.006)

**AC-7:** Given any sensor record with a custom vendor field `"custom_vendor_field": "xyz"`,
When normalized, Then `"custom_vendor_field"` appears in `extensions` and is not lost.
(BC-2.02.007)

**AC-8:** Given an `OcsfEvent` where the query field `"source_sensor"` is accessed, When
`AliasResolver::resolve` is called, Then it returns `AliasResult::PrismMetadata` (Tier 1)
without descending into the `DynamicMessage` proto fields. Given a field `"device.hostname"`
is accessed, Then it returns `AliasResult::ProtoField` (Tier 2) via recursive descent.
Given a vendor-specific field `"custom_vendor_field"` not present in the proto descriptor
is accessed, Then it returns `AliasResult::RawExtension` (Tier 3) from `raw_extensions`,
not `AliasResult::ProtoField`. Given a field absent from all three tiers is accessed, Then
it returns `AliasResult::Absent` (Tier 4).
(traces to BC-2.02.008 postcondition — four-tier field alias resolution)

**AC-9:** Given a normalization failure (e.g., required field `"detection_id"` missing
from CrowdStrike detection), When the error is returned, Then it contains the source
record context and specific field name. (BC-2.02.011)

**AC-10:** VP-017 proptest passes: no input fields are silently dropped across 10,000+
generated test cases.

## Architecture Compliance Rules

- All four mappers MUST implement `SensorMapper` — the normalizer dispatches via this
  trait, not via `match sensor {}`
- Mappers MUST NOT call `unwrap()` or `expect()` on JSON field access — always use
  `get()` and handle `None` explicitly
- Alias tables MUST be static (`phf_map!` recommended) — no runtime HashMap building
  in the hot path
- The `extensions` map MUST be populated before the mapper returns — callers rely on
  complete extensions before writing `raw_extensions` to the DynamicMessage

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| CrowdStrike detection missing `detection_id` | Returns `Err(PrismError::OcsfNormalizationFailed)` containing field name and source context (AC-9, BC-2.02.011) |
| Cyberint `created_date` with unrecognised format (e.g., `"not-a-date"`) | Returns `Err(PrismError::OcsfTimestampParseError)` containing field name and raw value (AC-4, BC-2.02.004) |
| Cyberint `created_date` as Unix timestamp integer | Parsed correctly as RFC3339 UTC (AC-3, BC-2.02.004) |
| Claroty `id` field as integer (not string) | Converted to string via `format!("claroty:{}", n)` — polymorphic ID handling (AC-5, BC-2.02.005) |
| Claroty unknown record type (not in the 9 supported) | Returns `Err(PrismError::OcsfUnknownRecordType)` |
| Armis device with no timestamp fields (`last_seen`, `created_at`, `timestamp` all absent) | Falls back to current time + emits tracing warning — never fails on missing time (AC-6, BC-2.02.006) |
| Vendor record with custom field not in explicit mapping | Field is preserved in `extensions` map — VP-017 proptest enforces no silent drops (AC-7, BC-2.02.007) |
| Alias resolution: field absent from all 4 tiers | Returns `AliasResult::Absent` — not an error; resolution terminates gracefully (AC-8, BC-2.02.008) |
| Alias tier 2 and tier 3 both have a field with same name | Tier 2 (proto descriptor field) wins — resolution is ordered; later tiers not consulted after first match |
| Mapper file exceeds 300 lines (expected for Claroty with 9 record types) | Split into `mappers/claroty/mod.rs` + `mappers/claroty/record_types.rs` per Dev Notes |

## Dev Notes

- The `SensorMapper::map()` signature passes `extensions` as a mutable reference rather
  than having the mapper return a separate extensions map. This avoids an allocation
  per-field and allows mappers to write directly into the accumulator.
- Claroty's polymorphic ID (int or string) is a common OT/IoT pattern. Use
  `serde_json::Value::as_i64().map(|n| n.to_string()).or_else(|| value.as_str().map(str::to_owned))`
  rather than a match — this handles both without cloning unnecessarily.
- Cyberint's multi-format timestamp parsing should try formats in order and stop at
  the first success. `chrono::DateTime::parse_from_rfc3339` first, then
  `NaiveDateTime::parse_from_str` with explicit format, then `as_i64()` for Unix.
- If a mapper file exceeds 300 lines (likely for Claroty with 9 record types), split
  into `mappers/claroty/mod.rs` and `mappers/claroty/record_types.rs`.
