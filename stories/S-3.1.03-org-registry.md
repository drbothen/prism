# S-3.1.03 â€” prism-core: implement OrgRegistry

## Narrative
- **As a** startup sequence and MCP dispatch layer
- **I want** an `OrgRegistry` that provides bijective O(1) resolution between `OrgSlug`
  and `OrgId`, with conflict-detecting registration
- **So that** all downstream components (credentials, sensors, audit) can resolve
  org identity without filesystem I/O on the hot path

## Description

D-047 mandates that `OrgRegistry` lives in `prism-core` (not a new `prism-orgs` crate).
This story implements the full registry module at
`crates/prism-core/src/org_registry.rs`. The registry uses a `BiMap<OrgSlug, OrgId>`
(from the `bimap` crate or equivalent) to provide O(1) lookup in both directions.
Registration is performed once at startup from `customers/*.toml` files; the registry
is read-only for the process lifetime after initialization. Idempotent re-registration
of the same (slug, uuid) pair is `Ok` per D-050; conflicting tuples return a structured
`RegistrationError` and abort startup.

## Acceptance Criteria

**AC-1 (traces to BC-3.1.001 postcondition 1):** If `register(slug, id)` succeeds and
`resolve(slug)` returns `Some(id)`, then `slug_for(id)` returns `Some(slug)` â€” round-trip
consistency holds.

**AC-2 (traces to BC-3.1.001 postcondition 2):** `resolve(unknown_slug)` returns `None`
without modifying registry state (verified by checking registry size before/after).

**AC-3 (traces to BC-3.1.001 invariant 4):** `resolve` and `slug_for` perform no
filesystem I/O and no blocking operations â€” validated by running inside a
`tokio::runtime::Builder::new_current_thread()` with no blocking allowed and asserting
they complete without yielding.

**AC-4 (traces to BC-3.1.003 postcondition 1):** After a successful `register(slug, id)`,
both `resolve(slug) == Some(id)` and `slug_for(id) == Some(slug)` hold.

**AC-5 (traces to BC-3.1.003 invariant 1):** The forward-map entry count always equals
the reverse-map entry count â€” verified by a proptest that applies a random sequence of
valid registrations and asserts `forward_len == reverse_len` after each.

**AC-6 (traces to BC-3.1.004 postcondition 2):** `register(slug, id)` where `slug` is
already bound to a different `OrgId` returns
`Err(RegistrationError::SlugConflict { slug, existing_id, attempted_id })` and leaves
the registry unchanged (verified by `resolve(slug) == Some(original_id)` after the
failed call).

**AC-7 (traces to BC-3.1.004 postcondition 3):** `register(slug, id)` where `id` is
already bound to a different `OrgSlug` returns
`Err(RegistrationError::IdConflict { id, existing_slug, attempted_slug })` and leaves
the registry unchanged.

**AC-8 (traces to BC-3.1.004 postcondition 4 / invariant 1):** `register(slug, id)` with
the exact same `(slug, id)` pair already registered returns `Ok(())` (idempotent per
D-050) â€” not an error.

> AC count: 8. This is above the 5-AC split threshold. The story is NOT split because
> all 8 ACs map to a single, cohesive module (`org_registry.rs`) with strongly coupled
> invariants; splitting would require cross-story BC coverage with no implementability
> gain. Flagged for PO review.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `resolve` called before any `register` | Returns `None`; no panic |
| EC-002 | `slug_for` called with an OrgId that was never registered | Returns `None`; no log emitted (BC-3.1.001 postcondition 4) |
| EC-003 | All customer configs conflict at startup | `OrgRegistry::load_from_configs()` collects all `RegistrationError`s in one pass and returns a multi-error; process aborts before MCP bind (BC-3.1.004 postcondition 5) |
| EC-004 | Slug at max length (64 chars of `a-zA-Z0-9_-`) | Registered and resolved correctly |
| EC-005 | Concurrent reads from multiple async tasks | `OrgRegistry` is `Arc<RwLock<BiMap<...>>>` or equivalent; concurrent reads are safe; verified by a multi-threaded proptest |
