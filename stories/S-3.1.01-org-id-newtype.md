# S-3.1.01 — prism-core: declare OrgId(Uuid v7) newtype

## Narrative
- **As a** downstream crate author (prism-credentials, prism-sensors, prism-audit)
- **I want** a stable `OrgId` newtype backed by UUID v7 declared in `prism-core`
- **So that** I can key credential stores, event buffers, and audit entries by a
  rename-stable canonical org identity rather than a mutable display string

## Description

ADR-006 §2.1 (D-041) separates the stable canonical org identity (`OrgId`, UUID v7)
from the analyst-facing display name (`OrgSlug`, formerly `TenantId`). This story uses
the existing `uuid_v7_newtype!` macro at `crates/prism-core/src/ids.rs` to declare
`OrgId` in the same location where other ID newtypes (e.g., `SensorId`, `SessionId`)
live. This is a purely additive change — no existing code is modified; only a new type
is declared. All downstream migration stories (S-3.1.03 through S-3.1.07) depend on
this type existing first.

## Acceptance Criteria

**AC-1 (traces to BC-3.1.001 precondition 3):** `OrgId` is declared using the
`uuid_v7_newtype!` macro in `crates/prism-core/src/ids.rs`, producing a newtype over
`uuid::Uuid`. The macro enforces UUID v7 at construction: `OrgId::new()` generates a
fresh v7 UUID; constructing from an arbitrary `Uuid` requires an explicit
`OrgId::from_uuid_v7(u)` that panics if the version is not 7 (matching the macro's
existing contract for other ID types).

**AC-2 (traces to BC-3.1.001 precondition 3):** `OrgId` is re-exported from
`prism-core/src/lib.rs` so that `use prism_core::OrgId` compiles from any downstream
crate.

**AC-3 (traces to BC-3.1.001 invariant 1):** `OrgId` derives `Debug`, `Clone`, `Copy`,
`PartialEq`, `Eq`, `Hash`, `serde::Serialize`, `serde::Deserialize`. These derives are
present (or generated by the macro); verified by a compile-time test that stores `OrgId`
in a `HashMap<OrgId, String>`.

**AC-4 (traces to BC-3.1.001 invariant 3):** `OrgId` displays as its UUID string
representation (hyphenated lowercase) via `Display` — verified by a unit test asserting
`OrgId::from_uuid_v7(known_uuid).to_string() == known_uuid_str`.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | `OrgId::from_uuid_v7(uuid_v4)` called with a v4 UUID | Panics with a clear message identifying the version mismatch (existing macro contract) |
| EC-002 | Two `OrgId::new()` calls in the same millisecond | Both produce valid v7 UUIDs; monotonic ordering is a best-effort property of v7, not a guarantee — this is acceptable |
| EC-003 | `OrgId` stored as HashMap key | Compiles and works; `Hash + Eq` derived from the underlying `Uuid` |
