# S-6.01 — prism-bin: CLI, Startup, and Initialization

## Narrative

As an MSSP operator installing Prism, I want a single binary with clear CLI commands
(`serve`, `init`, `health`, `--dry-run`) and a robust startup sequence that validates
configuration and initializes all subsystems before accepting MCP traffic, so that
misconfiguration or database issues are caught immediately with clear error messages
rather than manifesting as runtime failures mid-query.

## Objective

Implement the `prism` binary crate with clap-based CLI argument parsing, the full
ordered startup sequence, signal handling for graceful shutdown, and tracing subscriber
configuration. This is the integration point that wires all prism-* crates into a
working binary.

## Tasks

1. Create `crates/prism-bin/Cargo.toml`:
   - Declare binary target `prism`
   - Dependencies: `clap` (4.x, derive feature), `tokio` (full), `tracing`,
     `tracing-subscriber`, `prism-mcp`, `prism-core`, `prism-storage`

2. Implement clap CLI in `crates/prism-bin/src/main.rs`:
   - Top-level: `prism [OPTIONS] <COMMAND>`
   - Global options: `--config <PATH>` (override config file path), `--dry-run`
   - Commands: `serve`, `init`, `health`, `register`, `install-secops-factory`
   - Each command dispatches to a handler in `src/commands/`

3. Implement `prism serve` in `crates/prism-bin/src/commands/serve.rs`:
   - Call `startup::run(config_path, dry_run)` which executes the ordered startup
     sequence
   - On dry-run: print redacted config summary and exit 0/1 per validation result
   - On success: hand off to MCP server stdio loop (blocks until shutdown)

4. Implement ordered startup sequence in `crates/prism-bin/src/startup.rs`:
   a. Parse CLI args (already done by clap before this function is called)
   b. Load config: `config::loader::load(config_path)` → `PrismConfig`
   c. Validate config: `config::validation::validate(&config)` → collect errors, exit 1
      if any
   d. Initialize tracing subscriber: stderr writer, configured log level per subsystem
   e. Open RocksDB: `StorageEngine::open(state_dir)` — if schema version mismatch, print
      migration instructions and exit 3
   f. Probe keyring: verify the OS keychain is accessible, log warning if not
   g. Load sensor specs: `SpecEngine::load_all(&config)` — populate spec registry
   h. Load detection rules: `RuleEngine::load_all(&storage)` — warm rule cache
   i. Initialize scheduler: `Scheduler::start(&config, &storage)` — restore persisted
      schedules and resume any that fired while server was down
   j. Build `PrismContext`: assemble all initialized subsystems into shared context
   k. Register tools: `PrismServer::new(context)` — applies feature-flag gating
   l. Start stdio transport: `rmcp::stdio(server).await` — blocks here

5. Implement `prism init` in `crates/prism-bin/src/commands/init.rs`:
   - Check for existing `prism.toml` in current directory; abort with message if present
   - Write a template `prism.toml` with commented-out examples for common sensor types
   - Print "Created prism.toml — edit to add your clients and credentials"

6. Implement `prism health` in `crates/prism-bin/src/commands/health.rs`:
   - Load config (same as serve, but skip storage/scheduler init)
   - Instantiate a minimal `PrismContext` (no RocksDB, no scheduler)
   - Call `SensorHealthChecker::check_all(&config)` for all configured sensors
   - Print per-sensor status table to stdout: sensor | client | status | latency_ms
   - Exit 0 if all sensors healthy or partial, exit 1 if all sensors down

7. Configure tracing subscriber in `crates/prism-bin/src/startup.rs`:
   - Default: INFO level, text format
   - `--log-json` flag: switch to JSON format (for log aggregators)
   - Per-subsystem overrides via env: `RUST_LOG=prism_query_engine=debug,prism_mcp=info`
   - All output to stderr (BC-2.10.006 stdio reservation)

8. Implement signal handling in `crates/prism-bin/src/commands/serve.rs`:
   - Install handlers using `tokio::signal` before starting the stdio transport
   - On SIGTERM or Ctrl-C: call `PrismContext::shutdown()` which flushes audit buffer
     and closes RocksDB
   - Hard timeout: if shutdown takes > 5 seconds, force exit with code 1

## Acceptance Criteria

**AC-1:** Given `prism serve` with a valid `prism.toml`, When started, Then the binary
initializes all subsystems in order and the MCP server responds to `initialize` on stdio.

**AC-2:** Given `prism init` run in a directory without a `prism.toml`, When executed,
Then a `prism.toml` is created containing commented examples for CrowdStrike, Claroty,
Cyberint, and Armis sensors.

**AC-3:** Given `prism health` with configured sensors, When executed, Then a per-sensor
status table is printed to stdout with columns: sensor, client, status, latency_ms.

**AC-4:** Given a RocksDB database with a schema version older than the current binary's
expected version, When `prism serve` starts, Then the startup sequence prints a migration
instruction and exits with code 3.

**AC-5:** Given SIGTERM delivered to a running `prism serve`, When received, Then the
server completes graceful shutdown within 5 seconds and the RocksDB LOCK file is released.

**AC-6:** Given `prism serve --dry-run` with a config containing 2 validation errors,
When executed, Then both errors are printed with their TOML paths and suggestions, and
the process exits with code 1.

## Architecture Compliance Rules

- `prism-bin` MUST be a thin integration crate — no business logic belongs here; all
  logic lives in `prism-mcp`, `prism-storage`, etc.
- The startup sequence order (config → validate → storage → specs → rules → scheduler →
  context → tools → transport) is MANDATORY — deviating from this order risks using
  uninitialized subsystems
- `prism health` MUST NOT open RocksDB — it is a network connectivity check only and
  should not require a state directory to exist
- Tracing output MUST go to stderr — even `prism init` success messages should be
  written to stderr or stdout-as-user-facing-text, never mixed with MCP protocol data
- On exit code 3 (schema mismatch), the binary MUST print the expected and actual
  schema versions to help operators identify which migration to run

## Notes

- The template `prism.toml` created by `prism init` is the primary onboarding surface
  for new operators. Make it readable: include comment blocks explaining each section,
  example values with realistic-looking (but clearly fake) credentials, and a link to
  documentation.
- The startup sequence is intentionally fail-fast up through config validation, then
  transitions to structured error reporting for storage and subsystem issues. Operators
  need config problems caught before any resource allocation happens.
- `prism health` is designed to be runnable without a state directory, making it useful
  in CI/CD pipelines to validate credential and network connectivity before deploying.
