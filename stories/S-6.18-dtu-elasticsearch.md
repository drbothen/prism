# S-6.18 — prism-dtu-elasticsearch: DTU for Elasticsearch Bulk API — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Elasticsearch
Bulk API at fidelity level L2 (stateful), so that integration tests for the Elasticsearch
log-forwarding destination can validate NDJSON bulk indexing, index auto-creation, partial
failure handling (`errors: true` on HTTP 200), auth (Basic and API key), and Prism's
partial-failure error path without hitting live Elasticsearch.

## Objective

Implement `prism-dtu-elasticsearch` — an L2 stateful behavioral clone of the Elasticsearch
Bulk API. The DTU implements `POST /{index}/_bulk` with NDJSON action-metadata + document
pair parsing, maintains an index registry (auto-create on first use), supports configurable
partial failure injection (fail every Nth item), returns the Elasticsearch bulk response
format with per-item results, and captures all received documents for assertion.
See `dtu-assessment.md §3.7.3` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-elasticsearch/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `base64` (0.21.x for Basic auth decoding)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `ElasticsearchClone` struct implementing `BehavioralClone`:
   - `received_documents(&self, index: &str) -> Vec<serde_json::Value>`: test API
   - `configure_partial_failure(&self, fail_every_nth: u32)`: enables partial failure mode

3. Create `src/state.rs` — `ElasticsearchState`:
   - `index_registry: Mutex<HashMap<String, IndexState>>` — `index_name → IndexState`
   - `IndexState { created: bool, document_count: u32, documents: Vec<Value> }`
   - `fail_every_nth: Option<u32>` — if Some(N), every Nth document in a bulk request fails
   - `global_request_count: AtomicU32` — for rate-limit
   - `reset()` clears index registry

4. Auth middleware:
   - Accept `Authorization: Basic {base64}` (any valid base64 user:pass accepted) OR
     `Authorization: ApiKey {key}` (any non-empty key accepted)
   - If absent: return 401 `{"error": {"reason": "missing authentication credentials", "type": "security_exception"}}`
   - If `auth_mode == "reject"`: return 403 `{"error": {"reason": "action forbidden", "type": "security_exception"}}`

5. Create `src/routes/bulk.rs`:
   - `POST /{index}/_bulk`:
     - Parse NDJSON body as action-metadata + document pairs:
       - Odd lines: action metadata `{"index": {"_index": "...", "_id": "..."}}` or
         `{"create": {...}}`
       - Even lines: document body
       - If body is not valid NDJSON pairs: return 400
         `{"error": {"reason": "Invalid bulk request", "type": "parse_exception"}}`
     - Index auto-create: if `{index}` not in registry, create it with `document_count: 0`
     - Process each document:
       - If `fail_every_nth` is configured and this document number is a multiple of N:
         mark it as a failure in the response item
       - Otherwise: add document to `index_registry[index].documents`; mark as success
     - Return 200 with Elasticsearch bulk response:
       ```json
       {"took": 5, "errors": false|true, "items": [
         {"index": {"_index": "...", "_id": "...", "_version": 1, "result": "created", "status": 201}},
         {"index": {"_index": "...", "_id": "...", "error": {"type": "mapper_parse_exception", "reason": "..."}, "status": 400}}
       ]}
       ```
       - `"errors": true` if ANY item failed (even on HTTP 200 — this is the key Elasticsearch behavior)
   - `HEAD /{index}`: returns 200 if index exists in registry, 404 otherwise
   - `PUT /{index}`: explicit index creation; returns 200 `{"acknowledged": true}`

6. Create `GET /dtu/documents/{index}` test API:
   - Returns `{"index": "...", "documents": [...], "count": N}` for assertion

7. Create `POST /dtu/configure`:
   - `{"fail_every_nth": N}` — enables partial failure injection
   - `{"fail_every_nth": 0}` — disables partial failure injection

8. Create fixture files:
   - `fixtures/bulk-success-response.json`: example bulk response with `errors: false`
   - `fixtures/bulk-partial-failure-response.json`: example with some failed items, `errors: true`
   - `fixtures/index-created-response.json`: `{"acknowledged": true}`
   - `fixtures/auth-error-response.json`: 401 error response

9. Create `tests/fidelity.rs`:
   - Tests bulk insert (all success), partial failure (errors:true on 200), auth failure,
     index auto-create, `received_documents()` assertion
   - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /prism-logs/_bulk` with valid auth and 5 NDJSON pairs (action + doc),
Then the response is HTTP 200 with `errors: false` and 5 items each having `result: "created"`.

**AC-2:** Given `fail_every_nth: 3` configured, When `POST /prism-logs/_bulk` with 6 documents,
Then the response is HTTP 200 with `errors: true`, items 3 and 6 have
`"result"` set to an error, and items 1, 2, 4, 5 have `"result": "created"`.

**AC-3:** Given `POST /prism-logs/_bulk` with the index `"prism-logs"` not yet existing,
Then the bulk write succeeds AND the index is auto-created (subsequent `HEAD /prism-logs`
returns HTTP 200).

**AC-4:** Given `POST /prism-logs/_bulk` without an `Authorization` header, Then the
response is HTTP 401 with `{"error": {"type": "security_exception"}}`.

**AC-5:** Given 5 successful bulk writes, When `GET /dtu/documents/prism-logs` is called,
Then the response contains all 5 documents in the `documents` array.

**AC-6:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then HTTP 429.

**AC-7:** Given a NDJSON body where a document line contains a field conflicting with the
index's implied mapping, Then that document's response item has `"result"` as an error
with `"type": "mapper_parse_exception"` and `errors: true` in the response envelope.

## Risk Mitigations

### R-DTU-010: Elasticsearch version incompatibility

**Risk:** The Elasticsearch Bulk API response shape differs between Elasticsearch 7.x,
8.x, and OpenSearch 1.x/2.x. The `prism-dtu-elasticsearch` may target one version
while Prism's production deployment uses another, causing integration tests to pass
against the wrong schema.

**Mitigation:** Design the DTU against Elasticsearch 8.x Bulk API as the primary
target. Provide separate fixture sets for ES 7.x and OpenSearch variants in the
`fixtures/` directory. The `POST /dtu/configure` API accepts a `version_profile`
parameter to switch fixture sets. See `dtu-assessment.md §10 R-DTU-010` for the
full risk statement.
