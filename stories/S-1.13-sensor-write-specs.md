# S-1.13 — prism-spec-engine: Sensor Spec Write Endpoints

## Narrative

As a sensor spec consumer, I want sensor TOML files to declare write endpoints
alongside read tables so that PrismQL pipe verbs and SQL DML targets are available
at runtime without hardcoding sensor-specific write logic into the query engine.

## Objective

Extend the sensor TOML spec format to support `[[sensor.write_endpoints]]` sections.
Parse, validate, and register write endpoints in the DataFusion catalog and a
`WriteEndpointRegistry` for runtime dispatch. Implement write endpoint specs for the
four initial sensors (CrowdStrike, Cyberint, Claroty, Armis). Per AD-022.

## Tasks

1. Define `WriteEndpointSpec` struct in `crates/prism-spec-engine/src/write_endpoint.rs`:
   ```rust
   pub struct WriteEndpointSpec {
       pub endpoint_id: String,
       pub display_name: String,
       pub pipe_verb: String,
       pub sql_table: String,
       pub risk_tier: RiskTier,           // enum: Reversible | Irreversible
       pub reversibility_note: String,
       pub capability_path: String,
       pub batch_limit: u32,              // 0 = unlimited
       pub batch_limit_note: Option<String>,
       pub steps: Vec<WriteStep>,
   }

   pub struct WriteStep {
       pub name: String,
       pub method: HttpMethod,
       pub path_template: String,
       pub body_template: String,
       pub record_id_field: String,
       pub response_path: String,
       pub success_status: Vec<u16>,
   }

   pub enum RiskTier { Reversible, Irreversible }
   ```

2. Implement `WriteEndpointRegistry` in `crates/prism-spec-engine/src/write_endpoint.rs`:
   - Internal map: `HashMap<(SensorName, Verb), WriteEndpointSpec>`
   - `fn register(sensor: &str, spec: WriteEndpointSpec)`
   - `fn lookup(sensor: &str, verb: &str) -> Option<&WriteEndpointSpec>`
   - `fn verbs_for_sensor(sensor: &str) -> Vec<&str>` — for parser verb registration
   - `fn sql_tables() -> Vec<(TableName, &WriteEndpointSpec)>` — for DataFusion catalog

3. Extend `SpecParser` in `crates/prism-spec-engine/src/spec_parser.rs` to parse
   `[[sensor.write_endpoints]]` TOML arrays (BC-2.16.001):
   - Parse each array entry into `WriteEndpointSpec`
   - Call `WriteEndpointRegistry::register()` for each parsed endpoint
   - Parsing failure on any endpoint is fatal for the sensor spec load

4. Implement write endpoint validation in `crates/prism-spec-engine/src/validation.rs`
   (BC-2.16.009):
   - `pipe_verb` uniqueness: no two write endpoints in the same sensor spec may share a verb
   - Reserved keyword collision: reject if `pipe_verb` matches any read-side stage keyword:
     `where`, `fields`, `stats`, `sort`, `head`, `join`, `enrich` — return E-SPEC-003
   - `sql_table` uniqueness: no collision with read-side table names from `[[sensor.tables]]`
   - `capability_path` format: dot-separated alphanumeric segments, `[a-z0-9._]+`
   - `risk_tier` enum: only `"reversible"` or `"irreversible"` accepted; others rejected
   - `batch_limit = 0` + `risk_tier = "irreversible"`: emit structured warning (not error),
     include `batch_limit_note` if present, message: "Unlimited irreversible batch write:
     ensure analyst confirmation is enforced at query time"
   - `steps` array must be non-empty
   - `record_id_field` in each step must be a valid column name identifier (`[a-z0-9_]+`)

5. Implement DataFusion registration of write-capable tables in
   `crates/prism-spec-engine/src/datafusion_registration.rs`:
   - For each `WriteEndpointSpec`, register `sql_table` as a DataFusion `TableProvider`
     using a `WriteCapableTableProvider` wrapper that implements `insert_into()`,
     `update()`, and `delete_from()` as stubs that route to the write execution pipeline
   - `WriteCapableTableProvider` must NOT implement read methods — writes are write-only
     virtual tables in the catalog, not queryable as sources

6. Implement variable interpolation for write templates in
   `crates/prism-spec-engine/src/interpolation.rs`:
   - `${record_ids}`: resolves to JSON array of all values from `record_id_field` column
     in the source `RecordBatch` — e.g. `["dev_01", "dev_02"]`
   - `${params.KEY}`: resolves to the value of write_arg `key=value` passed in the
     write stage — e.g. `params.status` for `update_status status="acknowledged"`
   - `${params.KEY|default:VALUE}`: uses VALUE if the param was not provided
   - Apply same safety rules as read-side: JSON-escape for body templates,
     percent-encode for path templates

7. Add write endpoint specs to all four initial sensor TOML files:
   - `crowdstrike.sensor.toml`: `contain` (irreversible), `uncontain` (reversible),
     `acknowledge` (reversible), `update_status` (reversible), `assign` (reversible,
     targets `crowdstrike_detection_assignment`) — per AD-022 risk classification table
   - `cyberint.sensor.toml`: `acknowledge` (reversible), `close_alert` (irreversible,
     targets `cyberint_closed_alerts`) — per AD-022 risk classification table
   - `claroty.sensor.toml`: `acknowledge` (reversible), `tag` (reversible), `remove_tag` (reversible)
   - `armis.sensor.toml`: `tag` (reversible), `remove_tag` (reversible), `update_status` (reversible)
   - Total: 10 write endpoints across 4 sensors, matching AD-022 risk classification table

8. Write unit tests in `crates/prism-spec-engine/tests/write_endpoint_tests.rs`:
   - Valid spec load: all write verbs registered, DataFusion tables present
   - Reserved keyword rejection: pipe_verb = "where" → E-SPEC-003
   - Risk tier rejection: risk_tier = "read" → parse error
   - Batch limit + irreversible warning: emitted without failing load
   - Cross-sensor verb registry: `verbs_for_sensor("crowdstrike")` returns expected verbs

## Acceptance Criteria

**AC-1:** Given a sensor spec with `[[sensor.write_endpoints]]` containing `pipe_verb =
"contain"`, When `SpecParser` loads the file, Then the "contain" verb is registered in
`WriteEndpointRegistry` for that sensor. (BC-2.16.001)

**AC-2:** Given a write endpoint with `pipe_verb = "where"` (reserved keyword), When
validation runs, Then `E-SPEC-003` is returned with the message "pipe_verb 'where' is
reserved for read-side pipe stage" and the spec is not loaded. (BC-2.16.009)

**AC-3:** Given a write endpoint with `batch_limit = 0` and `risk_tier = "irreversible"`,
When validation runs, Then a structured warning is logged (spec load succeeds, no error
returned). (BC-2.16.009)

**AC-4:** Given a write endpoint with `sql_table = "crowdstrike_contained_hosts"`, When the
spec is loaded, Then the DataFusion catalog contains `crowdstrike_contained_hosts` as a
write-capable `TableProvider` (implements `insert_into`). (BC-2.16.001)

**AC-5:** Given all four sensor specs (CrowdStrike, Cyberint, Claroty, Armis) are loaded,
When `WriteEndpointRegistry` is queried, Then at least 4 distinct write verbs are available
across sensors.

**AC-6:** Given a write endpoint with `risk_tier = "read"`, When parsed, Then a parse
error is returned — `"read"` is not a valid risk tier for write endpoints.

**AC-7:** Given a write step with `body_template` containing `${params.status|default:"open"}`,
When the template is interpolated with no `status` param provided, Then the body contains
`"open"` at that position.

## Architecture Compliance Rules

- `WriteEndpointSpec` and `WriteEndpointRegistry` MUST live in `prism-spec-engine` —
  do NOT define them in `prism-query` or any other crate
- `WriteCapableTableProvider` MUST NOT implement `scan()` / read methods — write
  virtual tables are write-only catalog entries; reads route through read-side sources
- Variable interpolation in write templates MUST apply the same escaping rules as
  read-side step templates — no raw string substitution without context-appropriate escaping
- `RiskTier` enum MUST NOT have a `Read` variant — write endpoints are always at minimum
  `Reversible`; read-only endpoints live in `[[sensor.tables]]`
- Validation MUST run at spec load time, not at query time — invalid write endpoint specs
  prevent the sensor from loading entirely

## Previous Story Intelligence

S-1.11 implemented the read-side `SpecParser`, `SensorSpec` struct, and DataFusion table
registration. This story extends those without replacing them.

- Reuse `SpecParser`'s TOML loading infrastructure — do NOT duplicate file I/O
- Reuse `validation.rs` patterns for error construction (field path, line number, E-SPEC-NNN)
- Reuse `interpolation.rs` safety escaping functions for write templates
- Read-side table names from `[[sensor.tables]]` must be available at validation time to
  check `sql_table` uniqueness — ensure `SpecParser` passes both collections to the
  write endpoint validator

S-1.12 (hot-reload) will reload write endpoints when sensor specs change. Write endpoint
registration must be idempotent: re-registering the same verb replaces the previous entry
in `WriteEndpointRegistry`.

## Notes

- The DataFusion `TableProvider` write methods (`insert_into`, `update`, `delete_from`)
  on `WriteCapableTableProvider` should return `NotImplemented` stubs at this story's
  scope — the actual dispatch logic is wired in S-3.07. The catalog registration is what
  matters here so that S-3.06 parser tests can resolve write table names.
- Sensor TOML write endpoint specs should be co-located with the read-side spec in the
  same file (`crowdstrike.sensor.toml`) — do not create separate write spec files
- The `pipe_verb` registered for each sensor is what S-3.06 (parser) uses to build the
  dynamic Chumsky grammar production for write stages — coordinate the registry API
  contract between S-1.13 and S-3.06 before implementation begins
- `batch_limit` from the spec is the endpoint's declared limit. The system ceiling
  (`prism.toml [defaults.write_limits].max_batch_size`) is validated in S-3.07, not here.
  Spec-level validation only checks that `batch_limit` is a non-negative integer.
