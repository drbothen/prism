# S-1.13 — prism-spec-engine: Sensor Spec Write Endpoints

## Objective

Extend the sensor TOML spec format to support `[write_endpoints]` sections. Parse, validate,
and export write endpoint descriptors via a `WriteEndpointRegistry` for downstream DataFusion
registration by prism-query (S-3.07). Implement write endpoint specs for the four initial
sensors (CrowdStrike, Cyberint, Claroty, Armis). Per AD-022.

NOTE: prism-spec-engine does NOT depend on DataFusion — it exports data structures only.
DataFusion `TableProvider` registration is done by prism-query (S-3.07).

## Tasks

1. Extend `SensorSpec` TOML schema with `[write_endpoints.{verb}]` section:
   ```toml
   [write_endpoints.contain]
   pipe_verb = "contain"
   sql_table = "crowdstrike_contained_hosts"
   risk_tier = "irreversible"
   capability_path = "crowdstrike.hosts.write"
   batch_limit = 10
   batch_mode = "serial"
   record_id_field = "device_id"
   [[write_endpoints.contain.steps]]
   method = "POST"
   url = "/devices/entities/host-actions/v2"
   body_template = '{"action_name": "contain", "ids": ${record_ids}}'
   ```

2. Implement `WriteEndpointSpec` struct in `crates/prism-spec-engine/src/write_endpoint.rs`:
   - Fields: pipe_verb, sql_table, risk_tier (Reversible|Irreversible), capability_path,
     batch_limit, batch_mode (Serial|Parallel), record_id_field, steps (Vec<WriteStep>)
   - `WriteStep`: method, url, body_template, response_path (optional JSONPath)

3. Implement `WriteEndpointRegistry` in `crates/prism-spec-engine/src/write_endpoint.rs`:
   - Maps (sensor_type, verb) -> WriteEndpointSpec
   - `verbs_for_sensor(sensor: &str) -> Vec<&str>` for parser dynamic grammar
   - `get(sensor: &str, verb: &str) -> Option<&WriteEndpointSpec>` for execution dispatch
   - `WriteTableDescriptor` export: sql_table name, write_only flag, sensor, verb

4. Implement spec validation for write endpoints (BC-2.16.009):
   - `pipe_verb` must not collide with read-side reserved keywords (where, sort, limit,
     join, enrich, head) — return E-SPEC-003
   - `pipe_verb` must be unique across all sensors (no two sensors can claim "contain")
   - `risk_tier` must be "reversible" or "irreversible" only
   - `batch_limit = 0` + `risk_tier = "irreversible"`: emit structured warning (not error)
   - `steps` array must be non-empty
   - `record_id_field` must be a valid column name identifier (`[a-z0-9_]+`)

5. Export `WriteTableDescriptor` for each write endpoint:
   - Contains sql_table name, write_only flag, and write method routing metadata
   - DataFusion `TableProvider` registration (insert_into, update, delete_from) lives
     in prism-query (S-3.07), NOT prism-spec-engine

6. Implement variable interpolation for write templates in
   `crates/prism-spec-engine/src/interpolation.rs`:
   - `${record_ids}`: JSON array of record_id_field values from matched records
   - `${params.KEY}`: resolves to write_arg key=value from pipe stage
   - `${params.KEY|default:VALUE}`: uses VALUE if param not provided
   - Apply same safety rules as read-side: JSON-escape for body, percent-encode for URL

7. Add write endpoint specs to all four initial sensor TOML files:
   - `crowdstrike.sensor.toml`: contain (irreversible), uncontain (reversible),
     update_status (reversible), assign (reversible) — 4 endpoints
   - `cyberint.sensor.toml`: acknowledge (reversible), close_alert (irreversible) — 2 endpoints
   - `claroty.sensor.toml`: tag (reversible), remove_tag (reversible) — 2 endpoints
   - `armis.sensor.toml`: tag (reversible), remove_tag (reversible) — 2 endpoints
   - Total: 10 write endpoints across 4 sensors, matching AD-022 risk classification table

8. Write unit tests in `crates/prism-spec-engine/tests/write_endpoint_tests.rs`:
   - Valid spec load: all write verbs registered, WriteTableDescriptors present in registry
   - Reserved keyword rejection: pipe_verb = "where" -> E-SPEC-003
   - Risk tier rejection: risk_tier = "read" -> parse error
   - Batch limit + irreversible warning: emitted without failing load
   - Cross-sensor verb registry: `verbs_for_sensor("crowdstrike")` returns expected verbs

## Acceptance Criteria

**AC-1:** Given a valid `crowdstrike.sensor.toml` with `[write_endpoints.contain]`, When the
spec is loaded, Then `WriteEndpointRegistry.get("crowdstrike", "contain")` returns the
`WriteEndpointSpec` with risk_tier=Irreversible and batch_limit=10.

**AC-2:** Given a write endpoint with `pipe_verb = "where"` (reserved keyword), When
validation runs, Then E-SPEC-003 is returned.

**AC-3:** Given a write endpoint with `batch_limit = 0` and `risk_tier = "irreversible"`,
When validation runs, Then a structured warning is logged (spec load succeeds, no error
returned). (BC-2.16.009)

**AC-4:** Given a write endpoint with `sql_table = "crowdstrike_contained_hosts"`, When the
spec is loaded, Then `WriteEndpointRegistry` contains a `WriteTableDescriptor` with
`sql_table = "crowdstrike_contained_hosts"` and `write_only = true`. DataFusion catalog
registration is verified in S-3.07, not here. (BC-2.16.001)

**AC-5:** Given all four sensor specs loaded, When `WriteEndpointRegistry` is queried, Then
at least 10 write verbs are available across the 4 sensors.

---

## Architecture Compliance Rules

- prism-spec-engine MUST NOT depend on DataFusion or Arrow — export descriptors only
- DataFusion `TableProvider` registration lives in prism-query (S-3.07), NOT here
- `WriteEndpointRegistry` is the single source of truth for registered write verbs
- Write verb uniqueness is enforced globally (no two sensors can register the same verb)

## Notes

- This story exports `WriteTableDescriptor` and `WriteEndpointRegistry` data structures
  only — NO DataFusion code lives here. The actual `TableProvider` implementation
  (`WriteCapableTableProvider`) with `insert_into`/`update`/`delete_from` methods is
  created in prism-query by S-3.07 using these descriptors. prism-spec-engine does NOT
  depend on DataFusion.
- Sensor TOML write endpoint specs are co-located with read-side specs in the same file
- The `pipe_verb` registered for each sensor is what S-3.06 (parser) uses to build the
  dynamic Chumsky grammar production for write stages
