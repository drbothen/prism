# S-4.03 — prism-operations: Detection Rule Loading and Compilation

## Narrative

As a Prism detection engine, I want to load, validate, and compile detection rules from
`.detect` files and MCP tool calls, so that security analysts can define detection logic
once and have it automatically applied against incoming query results across the
appropriate client scopes.

## Objective

Implement `.detect` rule file loading and validation, MCP CRUD tools for rule management
(create_rule, list_rules, delete_rule), rule-to-SQL compilation targeting DataFusion WHERE
clauses, registration of security UDFs (subnet_contains, ioc_match, time_window), and
three-scope rule resolution (global + client + analyst merge).

## Tasks

1. Define detection rule types in `crates/prism-operations/src/detection/rule.rs`:
   - `DetectionRule { id: RuleId, name: String, condition: RuleCondition, scope: RuleScope,
     alert_template: AlertTemplate, rate_limit: RateLimit, created_at: Timestamp }`
   - `RuleCondition`: enum of `Single { where_clause: String }`,
     `Correlation { group_by, min_count, time_window, where_clause }`,
     `Sequence { events: Vec<SequenceStep>, time_window }` (ADR-015 §2.3 — three variants only;
     `Sequence` requires `events.len() >= 2`; load-time validation rejects shorter sequences
     with `E-RULE-SEQ-TOO-SHORT` per ADR-015 §2.3)
   - `RuleScope`: enum of `Global`, `Client { org_id: OrgId, client_id: ClientId }`,
     `Analyst { org_id: OrgId, analyst_id: AnalystId }` (D-208 LOCKED per ADR-015 §2.2)
   - `AlertTemplate { title: String, description: String, severity: SeverityId,
     variables: HashMap<String, String> }`
   - `RateLimit { max_per_hour: u32 }` — default 100/hour
   - `RuleId` newtype wrapping UUID v7
   - Global rule storage key: `__global__:{rule_id}` in `detection_rules` CF (sentinel prefix
     `__global__:`, NOT org-prefixed — Global rules apply across all orgs; deliberate exception
     to ADR-008 per ADR-015 §2.2.1). Detection state for Global rules is still per-org-keyed
     in `detection_state` CF: `{org_id}:\x00:__global__:{rule_id}:{group_key}` etc.

2. Implement `.detect` TOML file parser and validator in
   `crates/prism-operations/src/detection/loader.rs` (BC-2.13.001):
   - TOML schema: `[rule]` header with `name`, `scope`, `condition` (type + expression),
     `[alert]` section with `title`, `description`, `severity`, optional `rate_limit`
   - Validate: all required fields present, condition type is one of
     `{single, correlation, sequence}`, severity_id in range [0, 6], rate_limit.max_per_hour
     in [1, 10_000]
   - Validate: `sources` referenced in condition expressions exist in registered sensor
     specs (checked against S-1.11 spec registry at load time)
   - Reject invalid rules with structured `RuleValidationError` listing all failures
   - Load all `*.detect` files from configured rule directories on startup and hot reload

3. Implement `create_rule` MCP tool handler in
   `crates/prism-operations/src/detection/crud.rs` (BC-2.13.006):
   - Accept rule definition inline (same fields as .detect TOML format)
   - Run same validation pipeline as file loader
   - Persist to RocksDB `detection_rules` CF under key `rule:{id}`
   - Return created `DetectionRule` with assigned `RuleId`
   - Scope determines visibility: `Global` rules apply to all clients; `Client{org_id,client_id}`
     rules apply only to that `(OrgId, ClientId)` pair (ADR-015 §2.2, ADR-006); `Analyst{org_id,analyst_id}`
     rules apply only in that analyst's session within their org

4. Implement `list_rules` MCP tool handler (BC-2.13.007):
   - Parameters: optional `scope` filter (`global`, `client:{id}`, `analyst:{id}`)
   - Return all matching rules with id, name, condition type, scope, rate_limit status
   - Analyst-scoped rules are only visible to the owning analyst

5. Implement `delete_rule` MCP tool handler (BC-2.13.008):
   - For `Global`-scoped rules: require confirmation token (same pattern as S-1.09)
   - For `Client` or `Analyst`-scoped rules: no confirmation required
   - Remove from RocksDB `detection_rules` CF
   - Remove any associated state in `detection_state` CF (correlation windows, dedup keys)

6. Implement rule-to-SQL compiler in
   `crates/prism-operations/src/detection/compiler.rs` (BC-2.13.009):
   - `compile_rule(rule: &DetectionRule) -> CompiledRule`
   - `CompiledRule { sql: String, bind_params: Vec<SqlValue> }`
   - For `Single` conditions: produce `SELECT * FROM events WHERE {expr}` where `events`
     is a DataFusion MemTable populated by the schedule executor
   - For `Correlation` conditions: produce windowed aggregate query with `GROUP BY` and
     `HAVING count(*) >= threshold`
   - For `Sequence` conditions: produce a series of correlated subqueries or a window
     function query encoding the ordered pattern
   - Compilation is deterministic: same rule always produces the same SQL string

7. Register security UDFs in DataFusion session context in
   `crates/prism-operations/src/detection/udfs.rs` (BC-2.13.010):
   - All three UDFs MUST be implemented via the `ScalarUDFImpl` trait (NOT `create_udf`
     convenience), per ADR-015 §2.4 and R-1. Required trait methods:
     `as_any`, `name`, `signature`, `return_type`, `invoke_with_args`.
   - Volatility assignments per ADR-015 §2.5 (P1-S-4.03-A-M-003):
     - `subnet_contains`: `Volatility::Immutable` — purely computes CIDR membership from two
       static strings; DataFusion may constant-fold across queries.
     - `ioc_match`: `Volatility::Stable` — deterministic within one query, but the underlying
       `PatternStore` may be hot-reloaded between queries; DataFusion will NOT constant-fold
       across queries.
     - `time_window`: `Volatility::Stable` — deterministic within one query (evaluates against
       a fixed `Utc::now()` snapshot at `SessionContext` construction), but varies across
       queries; DataFusion will NOT constant-fold across queries.
   - `subnet_contains(ip: Utf8, cidr: Utf8) -> Boolean`: parses IP and CIDR via
     `ipnet = "2"` crate; returns true if IP is within the subnet; returns false (not error)
     on malformed inputs
   - `ioc_match(value: Utf8, list_name: Utf8) -> Boolean`: delegates to the `PatternStore`
     (§8a); `list_name` is the `.ioc` file basename; acquires `Arc<PatternStore>` via
     `ArcSwap::load()` at query start. If `list_name` does not exist in the store: returns
     `false` and emits a WARN log — no error propagated to query (ADR-015 §2.5,
     P1-S-4.03-A-M-004). `time_window` `window` parameter is parsed via
     `humantime::parse_duration` (ADR-015 §2.5 pin).
   - `time_window(timestamp: Timestamp, window_spec: Utf8) -> Boolean`: evaluates whether
     the timestamp falls within a window spec (e.g. "last 10m", "business_hours"); `window_spec`
     is parsed via `humantime::parse_duration` (ADR-015 §2.5)
   - All UDFs are registered once at `SessionContext` construction via `ctx.register_udf(ScalarUDF::new_from_impl(...))`
   - UDFs must never panic on malformed input — return `false` or `NULL` instead
   - UDF registration: `MemTable::try_new(schema, vec![vec![batch]])` + `Arc::new` +
     `ctx.register_table("events", ...)` is the canonical 53.x pattern for events table;
     there is no `SessionContext::register_batch` shortcut in DataFusion 53.x (ADR-015 §2.4)

8. Implement three-scope rule resolver in
   `crates/prism-operations/src/detection/scope_resolver.rs` (BC-2.13.011):
   - `resolve_rules(org_id: OrgId, client_id: ClientId, analyst_id: AnalystId) -> Vec<DetectionRule>`:
     merge global + client + analyst rules into a single ordered list
   - Scope resolution — UNION/MERGE model (ADR-015 §2.2, P1-S-4.03-A-H-001): all three scope
     tiers are UNIONED at evaluation time. Each rule fires independently regardless of name
     collision across scopes. Silent scope-override by name shadowing is explicitly PROHIBITED.
     There is NO implicit Analyst > Client > Global precedence.
   - Suppression model: a Global rule may declare `disabled_for: [<org_id>/<client_id>]` or
     `disabled_for: [<org_id>/<analyst_id>]`. Suppression of a Global rule for a specific
     context is ONLY achievable via this explicit `disabled_for` list — no other mechanism
     exists (ADR-015 §2.2).
   - Org isolation: `Client`-scoped rules are filtered to the exact `(OrgId, ClientId)` pair;
     `Analyst`-scoped rules are filtered to the exact `(OrgId, AnalystId)` pair.
     A rule scoped to org A's client "acme" MUST NOT be returned in org B's context.
   - Merge semantics: all three sets are unioned within org context — global rules do not block
     client rules and vice versa. Duplicate rule names across scopes are NOT deduplicated; each fires
     independently (this is by design: every active rule is visible and independently auditable).
   - Evaluation order: lexicographic by rule name within each scope tier, global first,
     then client, then analyst
   - Global rules are fetched from `detection_rules` CF using sentinel prefix `__global__:` (not
     org-prefixed); detection state for Global rules is org-keyed in `detection_state` CF
   - Result is deterministic: same inputs always produce the same ordered list

8a. Implement IOC file loading and `ioc_match` UDF
    registration (previously `ioc_match` was listed as a UDF stub; this expansion makes
    IOC file loading a concrete deliverable of this story):
   - **IOC file parser:** Read `*.ioc` files from `{config_dir}/ioc/` directory.
     File format: one indicator per line (plain text), lines starting with `#` are comments.
     Supported indicator types (auto-detected by format using the following ordered precedence,
     P1-S-4.03-A-M-002):
     1. Contains `:` followed by hex chars and colons → **IPv6** (exact match)
     2. Matches IPv4 regex `^(\d{1,3}\.){3}\d{1,3}$` → **IPv4** (exact match)
     3. Exactly 64 hex characters → **SHA-256** (case-insensitive exact match)
     4. Exactly 32 hex characters → **MD5** (case-insensitive exact match)
     5. Contains `/` → **file path** (case-sensitive exact match; case-insensitive on Windows builds)
     6. Contains `.` and matches hostname regex `^[a-zA-Z0-9]([a-zA-Z0-9\-]{0,61}[a-zA-Z0-9])?(\.[a-zA-Z0-9]([a-zA-Z0-9\-]{0,61}[a-zA-Z0-9])?)*$` → **hostname** (case-insensitive exact match)
     7. Else → **exact-string literal** (case-sensitive exact match via aho-corasick Layer 1)
     The filename (without `.ioc` extension) becomes the `list_name` for `ioc_match`.
   - **In-memory pattern store — LAYERED ARCHITECTURE (ADR-015 §2.6, R-7):**
     `PatternStore` uses a two-layer design. `RegexSet`-only at 100k+ patterns is REJECTED
     (ADR-015 §4.1; regex Issue #1059 documents a 600+MB memory regression for moderate sets,
     which would exceed the 512MB process budget).
     - **Layer 1 — Literal IOCs** (`aho-corasick = "1.1"`): SHA-256, MD5, IPv4, IPv6,
       exact hostnames, exact URL strings. O(n+z) linear search; scales to 100k+ patterns.
     - **Layer 2 — Regex IOCs** (`regex = "1.10"` with `RegexSet`): pattern-based IOCs
       (e.g., suspicious domain templates). Capped at ~1,000 patterns; `dfa_size_limit`
       configured explicitly at 10 MB. Reject new regex patterns with `Err(RegexCapExceeded)`
       at load time if cap is reached.
     - The store maps list name → `PatternStore { literal_ac: AhoCorasick, regex_set: RegexSet }`.
     - Guarded by `arc_swap::ArcSwap<Arc<PatternStore>>` for atomic hot reload.
   - **`ioc_match(field, list_name)` UDF registration:** Register as a DataFusion scalar UDF
     via `ScalarUDFImpl` trait (see Task 7) alongside `subnet_contains` and `time_window`.
     Signature: `ioc_match(value: Utf8, list_name: Utf8) -> Boolean`.
     - Acquire `Arc<PatternStore>` clone at query start via `ArcSwap::load()` (no lock)
     - Check Layer 1 (aho-corasick) then Layer 2 (RegexSet); return `true` if either matches
     - Return `false` (not error) if `list_name` does not exist in the store (+WARN log)
     - NULL input field values return NULL (DataFusion null propagation)
   - **Hot reload participation:** Watch `{config_dir}/ioc/*.ioc` via `notify = "7"` +
     `notify-debouncer-full` crate (separate crate; per ADR-015 §2.6 + DRIFT-403-003).
     On file change: recompile the changed `.ioc` file into a new `PatternStore` entry,
     arc-swap the store. In-flight UDF calls holding the old `Arc` complete without error.
   - **Dedup-window resolution:** `effective_dedup_window` is resolved at rule load time
     (NOT per-evaluation). Resolution priority: (1) explicit `dedup_window` field in rule,
     (2) linked `ScheduleEntry.interval` if `schedule_id` set, (3) default 1 hour.
     Invalidated on schedule change via ADR-013 §2.7 watch channel (ADR-015 §2.7, D-211 LOCKED).
   - **IOC file size limits and validation (BC-2.13.014):**
     - Maximum 100,000 patterns per file (E-IOC-003 on excess)
     - Maximum file size 10 MB (E-IOC-002 on excess)
     - Maximum 50 IOC files total (E-IOC-004 on excess)
     - Lines with invalid regex patterns cause file rejection with E-IOC-001 (not line-skip)
     - Empty files are valid (RegexSet with 0 patterns; `ioc_match` always returns `false`)
   - **BC anchor (Burst 2.75 — resolved):** BC-2.13.014 `IOC File Loading and Pattern Store`
     is now committed. It formally specifies the IOC file format, indicator types, `IocStore`
     structure, hot reload semantics, and size/count limits. Anchored in frontmatter and BC
     traceability table above.

9. Enforce DI-028 rule capacity in `crates/prism-operations/src/detection/crud.rs`:
   - Before inserting a new rule, count existing rules in the `detection_rules` CF using a
     prefix scan over `rule:` keys
   - If count >= `max_rules` (default 1000, configurable via `[detection] max_rules` in
     prism.toml), reject with structured error `E-RULE-011` and message
     "Maximum rule count exceeded"
   - The count check MUST be performed under a RocksDB transaction or equivalent
     atomic read-modify-write to prevent TOCTOU races at the cap boundary
   - The VP-030 Kani proof for the schedule+rule count cap is anchored in S-4.01 (VP-INDEX
     source of truth). This story enforces the E-RULE-011 cap at runtime; the formal proof
     lives in S-4.01's `crates/prism-operations/proofs/vp030_schedule_rule_caps.rs`.

10. Write VP-018 proptest in `crates/prism-operations/props/vp018_compilation_deterministic.rs`:
   - Property: `compile_rule(rule)` produces the same SQL string on every call for the
     same rule — no randomness, no timestamp injection, no session state dependency
   - Also verify: invalid rules (missing required fields, unknown sources, bad severity)
     are always rejected with `Err(RuleValidationError)`, never `Ok(_)`

## Acceptance Criteria

**AC-1:** Given a valid `.detect` file with `condition.type = "single"`, When loaded,
Then the rule is parsed, validated, and registered in the detection engine with a unique
`RuleId`. (BC-2.13.001)

**AC-2:** Given a `.detect` file with `condition.type = "correlation"` but missing the
`threshold` field, When loaded, Then it is rejected with a `RuleValidationError`
specifying the missing field. (BC-2.13.001)

**AC-3:** Given `create_rule` with `scope: "client:acme"` in org A's context, When
created, Then the rule is persisted and `list_rules(scope: "client:acme")` in org A
includes it, but `list_rules(scope: "client:acme")` in org B's context does not.
(BC-2.13.006, BC-2.13.007 — traces to BC-2.13.011 invariant: org isolation enforced via
`RuleScope::Client { org_id, client_id }` dual key; ADR-015 §2.2; ADR-006 §2.1)

**AC-3b:** Given a rule scoped to org A's client "acme" (`RuleScope::Client { org_id: A, client_id: "acme" }`),
When `list_rules` is called in org B's context, Then the rule is NOT returned.
(traces to ADR-015 §2.2 org-isolation invariant; DRIFT-403-002 remediation)

**AC-4:** Given a `Global`-scoped rule, When `delete_rule` is called without a
confirmation token, Then the tool responds requesting confirmation. (BC-2.13.008)

**AC-5:** Given a `Single` detection rule with condition `severity_id >= 4`, When
compiled, Then the output SQL contains `WHERE severity_id >= 4` targeting the `events`
MemTable. (BC-2.13.009)

**AC-6:** Given the DataFusion session context, When `subnet_contains('192.168.1.5',
'192.168.1.0/24')` is evaluated, Then it returns `true`. When evaluated with
`'10.0.0.1'`, it returns `false`. (BC-2.13.010)

**AC-7:** Given a global rule G1 and a client rule C1 for client "acme" (where C1 has the
same name as G1), When `resolve_rules(client_id: "acme", analyst_id: "u1")` is called,
Then the result contains BOTH G1 and C1 — unioned, firing independently. C1 does NOT
suppress or override G1; there is no implicit precedence. Suppression of G1 is only
possible if G1's `disabled_for` list explicitly names `<org_id>/acme`. (BC-2.13.011 —
traces to ADR-015 §2.2 UNION model invariant; P1-S-4.03-A-H-001)

**AC-8:** VP-018 proptest passes — compilation is deterministic, invalid rules are always
rejected.

**AC-10:** Given 1000 existing rules (at max_rules cap), When `create_rule` is called,
Then structured error E-RULE-011 is returned with message "Maximum rule count exceeded"
and no rule is created.
(traces to BC-2.13.006 / DI-028 — rule capacity invariant)

**AC-9:** Given IOC files exist in `{config_dir}/ioc/` at startup, When the detection
engine loads, Then each `*.ioc` file is parsed into a layered `PatternStore` entry (literal
indicators → aho-corasick layer; regex patterns → `RegexSet` layer capped at ~1,000) keyed
by the file's basename (without `.ioc` extension); the `ioc_match` UDF is registered against
the pattern store; and an INFO log
`"IOC file loaded: {list_name} ({pattern_count} patterns, {elapsed_ms}ms)"` is emitted
for each successfully loaded file. A file containing invalid regex patterns is rejected
with `E-IOC-001`; a file exceeding 10 MB is rejected with `E-IOC-002`; a file exceeding
100,000 patterns is rejected with `E-IOC-003`; a 51st file when 50 are already loaded is
rejected with `E-IOC-004`. In all rejection cases the prior pattern store state for that
file is retained and the server does not crash.
(traces to BC-2.13.014 postconditions "Successful load" and INV-IOC-003 "Memory Cap Enforced" and INV-IOC-004 "No Crash on Malformed File"; ADR-015 §2.6 layered architecture)

## Architecture Compliance Rules

- Rule compilation MUST be a pure function of the `DetectionRule` struct. Compilers that
  read from global state, clocks, or I/O at compile time will fail the VP-018
  determinism property.
- Security UDFs MUST return `false` or `NULL` (never panic) on malformed inputs. A
  panicking UDF would crash the DataFusion query executor.
- UDFs are registered once per `SessionContext` at startup. They MUST NOT be re-registered
  on every query execution (registration is idempotent but wasteful).
- Rule scope resolution MUST implement the UNION/MERGE model (ADR-015 §2.2, P1-S-4.03-A-H-001).
  Global rule "alert-A" and client rule "alert-A" are different rules that BOTH fire
  independently — no implicit precedence. Any code implementing Analyst > Client > Global
  shadowing is a correctness defect. Suppression is ONLY via `disabled_for` list on Global rules.
- UDF volatility MUST match ADR-015 §2.5 (P1-S-4.03-A-M-003): `subnet_contains` is
  `Volatility::Immutable`; `ioc_match` and `time_window` are `Volatility::Stable`. Marking
  `ioc_match` or `time_window` as `Immutable` would allow DataFusion to constant-fold them
  across queries, breaking IOC hot-reload and time-window correctness.
- The `detection_rules` CF stores rule definitions. The `detection_state` CF (created in
  S-4.04) stores runtime evaluation state. These are separate CFs with different
  retention and eviction policies.
- Feature flag `FEATURE_DETECTION_ENGINE` (from S-1.08) MUST gate all detection rule
  operations. Return E-FLAG-001 if the flag is not enabled.
- `RuleScope::Client` MUST carry both `org_id: OrgId` and `client_id: ClientId` (D-208
  LOCKED; ADR-015 §2.2). The prior `Client(ClientId)` single-field form is invalid;
  any code producing it is a correctness defect.
- UDFs MUST be registered via `ScalarUDFImpl` trait (ADR-015 §2.4). `create_udf`
  convenience API may only be used for trivial one-liner UDFs; the three security UDFs
  are not trivial.
- `PatternStore` literal-IOC matching MUST use `aho-corasick = "1.1"` (Layer 1).
  `RegexSet` is ONLY for regex-pattern IOCs and MUST be capped at ~1,000 patterns with
  explicit `dfa_size_limit = 10_000_000` (ADR-015 §2.6). Any code that routes literal
  IOCs (IPs, hashes, exact hostnames) through `RegexSet` is a performance and memory defect.
- Dedup window MUST be resolved at rule load time and stored in `DetectionRuleCache`.
  Any code path that resolves `effective_dedup_window` during per-evaluation hot path is
  a correctness defect (VP-140 validates this; ADR-015 §2.7, D-211 LOCKED).

## Dev Notes

- `.detect` files are the "infrastructure as code" path for detection rules. MCP tool
  `create_rule` is the "interactive" path. Both paths must run the same validation
  pipeline — do not duplicate validation logic between `loader.rs` and `crud.rs`.
- The `events` MemTable referenced in compiled SQL is a DataFusion in-memory table
  populated per-execution by the schedule executor (S-4.04). The compiler produces SQL
  that targets this table by name. The actual table registration is done in S-4.04, not
  here.
- Correlation and sequence condition SQL compilation may be complex. For the initial
  implementation, correlation can use a simple `SELECT count(*) FROM events WHERE {expr}
  HAVING count(*) >= {threshold}` pattern. Sequence detection SQL is deferred to S-4.04
  which owns the stateful evaluation logic.
- `time_window` UDF should support at minimum: `"last Nm"` (last N minutes), `"last Nh"`
  (last N hours), and `"business_hours"` (Mon-Fri 08:00-18:00 UTC). Additional window
  specs can be added in later iterations.
