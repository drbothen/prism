# S-4.03 — prism-operations: Detection Rule Loading and Compilation

## Narrative

As a Prism detection engine, I want to load, validate, and compile detection rules from
`.detect` files and MCP tool calls, so that security analysts can define detection logic
once and have it automatically applied against incoming query results across the
appropriate client scopes.

## Objective

Implement `.detect` rule file loading and validation, MCP CRUD tools for rule management
(create_rule, list_rules, delete_rule), rule-to-SQL compilation targeting DataFusion WHERE
clauses, registration of security UDFs (subnet_contains, ioc_match, time_window), and
three-scope rule resolution (global + client + analyst merge).

## Tasks

1. Define detection rule types in `crates/prism-operations/src/detection/rule.rs`:
   - `DetectionRule { id: RuleId, name: String, condition: RuleCondition, scope: RuleScope,
     alert_template: AlertTemplate, rate_limit: RateLimit, created_at: Timestamp }`
   - `RuleCondition`: enum of `Single(Expr)`, `Correlation(CorrelationSpec)`,
     `Sequence(SequenceSpec)`
   - `RuleScope`: enum of `Global`, `Client(ClientId)`, `Analyst(AnalystId)`
   - `AlertTemplate { title: String, description: String, severity: SeverityId,
     variables: HashMap<String, String> }`
   - `RateLimit { max_per_hour: u32 }` — default 100/hour
   - `RuleId` newtype wrapping UUID v7

2. Implement `.detect` TOML file parser and validator in
   `crates/prism-operations/src/detection/loader.rs` (BC-2.13.001):
   - TOML schema: `[rule]` header with `name`, `scope`, `condition` (type + expression),
     `[alert]` section with `title`, `description`, `severity`, optional `rate_limit`
   - Validate: all required fields present, condition type is one of
     `{single, correlation, sequence}`, severity_id in range [0, 6], rate_limit.max_per_hour
     in [1, 10_000]
   - Validate: `sources` referenced in condition expressions exist in registered sensor
     specs (checked against S-1.11 spec registry at load time)
   - Reject invalid rules with structured `RuleValidationError` listing all failures
   - Load all `*.detect` files from configured rule directories on startup and hot reload

3. Implement `create_rule` MCP tool handler in
   `crates/prism-operations/src/detection/crud.rs` (BC-2.13.006):
   - Accept rule definition inline (same fields as .detect TOML format)
   - Run same validation pipeline as file loader
   - Persist to RocksDB `detection_rules` CF under key `rule:{id}`
   - Return created `DetectionRule` with assigned `RuleId`
   - Scope determines visibility: `Global` rules apply to all clients; `Client(id)` rules
     apply only to that client; `Analyst(id)` rules apply only in that analyst's session

4. Implement `list_rules` MCP tool handler (BC-2.13.007):
   - Parameters: optional `scope` filter (`global`, `client:{id}`, `analyst:{id}`)
   - Return all matching rules with id, name, condition type, scope, rate_limit status
   - Analyst-scoped rules are only visible to the owning analyst

5. Implement `delete_rule` MCP tool handler (BC-2.13.008):
   - For `Global`-scoped rules: require confirmation token (same pattern as S-1.09)
   - For `Client` or `Analyst`-scoped rules: no confirmation required
   - Remove from RocksDB `detection_rules` CF
   - Remove any associated state in `detection_state` CF (correlation windows, dedup keys)

6. Implement rule-to-SQL compiler in
   `crates/prism-operations/src/detection/compiler.rs` (BC-2.13.009):
   - `compile_rule(rule: &DetectionRule) -> CompiledRule`
   - `CompiledRule { sql: String, bind_params: Vec<SqlValue> }`
   - For `Single` conditions: produce `SELECT * FROM events WHERE {expr}` where `events`
     is a DataFusion MemTable populated by the schedule executor
   - For `Correlation` conditions: produce windowed aggregate query with `GROUP BY` and
     `HAVING count(*) >= threshold`
   - For `Sequence` conditions: produce a series of correlated subqueries or a window
     function query encoding the ordered pattern
   - Compilation is deterministic: same rule always produces the same SQL string

7. Register security UDFs in DataFusion session context in
   `crates/prism-operations/src/detection/udfs.rs` (BC-2.13.010):
   - `subnet_contains(ip: Utf8, cidr: Utf8) -> Boolean`: parses IP and CIDR, returns true
     if IP is within the subnet; returns false (not error) on malformed inputs
   - `ioc_match(value: Utf8, ioc_list: List<Utf8>) -> Boolean`: checks if value appears
     in the provided IOC list; case-insensitive for hostnames, exact-match for IPs/hashes
   - `time_window(timestamp: Timestamp, window_spec: Utf8) -> Boolean`: evaluates whether
     the timestamp falls within a window spec (e.g. "last 10m", "business_hours")
   - All UDFs are registered on the shared `SessionContext` used by the query engine
   - UDFs must never panic on malformed input — return `false` or `NULL` instead

8. Implement three-scope rule resolver in
   `crates/prism-operations/src/detection/scope_resolver.rs` (BC-2.13.011):
   - `resolve_rules(client_id, analyst_id) -> Vec<DetectionRule>`: merge global + client +
     analyst rules into a single ordered list
   - Merge semantics: all three sets are unioned — global rules do not block client rules
     and vice versa. Duplicate rule names across scopes are NOT deduplicated; each fires
     independently.
   - Evaluation order: lexicographic by rule name within each scope tier, global first,
     then client, then analyst
   - Result is deterministic: same inputs always produce the same ordered list

8a. [SCOPE EXPANSION — Phase 3 patch] Implement IOC file loading and `ioc_match` UDF
    registration (previously `ioc_match` was listed as a UDF stub; this expansion makes
    IOC file loading a concrete deliverable of this story):
   - **IOC file parser:** Read `*.ioc` files from `{config_dir}/ioc/` directory.
     File format: one indicator per line (plain text), lines starting with `#` are comments.
     Supported indicator types (auto-detected by format):
     - IP addresses (IPv4 and IPv6): exact match
     - Hostnames: case-insensitive exact match
     - SHA256 hashes (64 hex chars): case-insensitive exact match
     - MD5 hashes (32 hex chars): case-insensitive exact match
     - File paths: case-sensitive exact match (or case-insensitive on Windows builds)
     The filename (without `.ioc` extension) becomes the `list_name` for `ioc_match`.
   - **In-memory pattern store:** `IocStore`: `HashMap<list_name: String, IocList>` where
     `IocList` contains separate `HashSet`s per indicator type for O(1) lookup.
     Guarded by `arc_swap::ArcSwap<IocStore>` for hot reload.
   - **`ioc_match(field, list_name)` UDF registration:** Register as a DataFusion scalar UDF
     alongside `subnet_contains` and `time_window`. Signature:
     `ioc_match(value: Utf8, list_name: Utf8) -> Boolean`.
     - Lookup the `list_name` in the global `IocStore` (thread-safe via ArcSwap load)
     - Detect indicator type from `value` format, search the appropriate HashSet
     - Return `false` (not error) if `list_name` does not exist
     - Return `false` (not error) on malformed `value` — never panic
   - **Hot reload participation:** Watch `{config_dir}/ioc/*.ioc` via S-1.12's file watcher.
     On file change: reload the changed `.ioc` file into a new `IocList`, arc-swap the store.
     In-flight UDF calls using the old store complete without error (Arc keeps it alive).
   - **IOC file size limits and validation:**
     - Maximum 1,000,000 indicators per file (E-IOC-001 on excess)
     - Maximum file size 50MB (E-IOC-002 on excess)
     - Lines with invalid indicator format are skipped with a WARN log (not a hard error)
     - Empty files are valid (empty list always returns `false`)
   - **BC anchor (Burst 2.75 — resolved):** BC-2.13.014 `IOC File Loading and Pattern Store`
     is now committed. It formally specifies the IOC file format, indicator types, `IocStore`
     structure, hot reload semantics, and size/count limits. Anchored in frontmatter and BC
     traceability table above.

9. Write VP-018 proptest in `crates/prism-operations/props/vp018_compilation_deterministic.rs`:
   - Property: `compile_rule(rule)` produces the same SQL string on every call for the
     same rule — no randomness, no timestamp injection, no session state dependency
   - Also verify: invalid rules (missing required fields, unknown sources, bad severity)
     are always rejected with `Err(RuleValidationError)`, never `Ok(_)`

## Acceptance Criteria

**AC-1:** Given a valid `.detect` file with `condition.type = "single"`, When loaded,
Then the rule is parsed, validated, and registered in the detection engine with a unique
`RuleId`. (BC-2.13.001)

**AC-2:** Given a `.detect` file with `condition.type = "correlation"` but missing the
`threshold` field, When loaded, Then it is rejected with a `RuleValidationError`
specifying the missing field. (BC-2.13.001)

**AC-3:** Given `create_rule` with `scope: "client:acme"`, When created, Then the rule
is persisted and `list_rules(scope: "client:acme")` includes it, but
`list_rules(scope: "client:other")` does not. (BC-2.13.006, BC-2.13.007)

**AC-4:** Given a `Global`-scoped rule, When `delete_rule` is called without a
confirmation token, Then the tool responds requesting confirmation. (BC-2.13.008)

**AC-5:** Given a `Single` detection rule with condition `severity_id >= 4`, When
compiled, Then the output SQL contains `WHERE severity_id >= 4` targeting the `events`
MemTable. (BC-2.13.009)

**AC-6:** Given the DataFusion session context, When `subnet_contains('192.168.1.5',
'192.168.1.0/24')` is evaluated, Then it returns `true`. When evaluated with
`'10.0.0.1'`, it returns `false`. (BC-2.13.010)

**AC-7:** Given a global rule G1 and a client rule C1 for client "acme", When
`resolve_rules(client_id: "acme", analyst_id: "u1")` is called, Then the result contains
both G1 and C1 — merged, not overridden. (BC-2.13.011)

**AC-8:** VP-018 proptest passes — compilation is deterministic, invalid rules are always
rejected.

**AC-9:** Given IOC files exist in `{config_dir}/ioc/` at startup, When the detection
engine loads, Then each `*.ioc` file is compiled into a `regex::RegexSet` and stored in
the in-memory pattern store keyed by the file's basename (without `.ioc` extension); the
`ioc_match` UDF is registered against the pattern store; and an INFO log is emitted for
each successfully loaded file. A file exceeding 100,000 patterns or 10 MB is rejected
with `E-IOC-003` or `E-IOC-002` respectively, the prior pattern store state for that file
is retained, and the server does not crash.
(traces to BC-2.13.014 postconditions "Successful load" and INV-IOC-003 "Memory Cap Enforced" and INV-IOC-004 "No Crash on Malformed File")

## Architecture Compliance Rules

- Rule compilation MUST be a pure function of the `DetectionRule` struct. Compilers that
  read from global state, clocks, or I/O at compile time will fail the VP-018
  determinism property.
- Security UDFs MUST return `false` or `NULL` (never panic) on malformed inputs. A
  panicking UDF would crash the DataFusion query executor.
- UDFs are registered once per `SessionContext` at startup. They MUST NOT be re-registered
  on every query execution (registration is idempotent but wasteful).
- Rule scope resolution MUST NOT apply deduplication across scopes. Global rule "alert-A"
  and client rule "alert-A" are different rules that both fire — this is by design, not
  a conflict.
- The `detection_rules` CF stores rule definitions. The `detection_state` CF (created in
  S-4.04) stores runtime evaluation state. These are separate CFs with different
  retention and eviction policies.
- Feature flag `FEATURE_DETECTION_ENGINE` (from S-1.08) MUST gate all detection rule
  operations. Return E-FLAG-001 if the flag is not enabled.

## Notes

- `.detect` files are the "infrastructure as code" path for detection rules. MCP tool
  `create_rule` is the "interactive" path. Both paths must run the same validation
  pipeline — do not duplicate validation logic between `loader.rs` and `crud.rs`.
- The `events` MemTable referenced in compiled SQL is a DataFusion in-memory table
  populated per-execution by the schedule executor (S-4.04). The compiler produces SQL
  that targets this table by name. The actual table registration is done in S-4.04, not
  here.
- Correlation and sequence condition SQL compilation may be complex. For the initial
  implementation, correlation can use a simple `SELECT count(*) FROM events WHERE {expr}
  HAVING count(*) >= {threshold}` pattern. Sequence detection SQL is deferred to S-4.04
  which owns the stateful evaluation logic.
- `time_window` UDF should support at minimum: `"last Nm"` (last N minutes), `"last Nh"`
  (last N hours), and `"business_hours"` (Mon-Fri 08:00-18:00 UTC). Additional window
  specs can be added in later iterations.
