# S-3.3.03 â€” prism-dtu-harness: logical isolation mode + crash detection + failure injection

## Narrative

As a Prism integration-test author, I want a `prism-dtu-harness` crate that
spins multiple customer DTU clone instances in a single process with per-org
state segregation, crash detection, and per-`(OrgId, DtuType)` failure injection,
so that multi-tenant isolation scenarios can be verified deterministically without
spawning external processes.

## Objective

Create `crates/prism-dtu-harness`. Implement `IsolationMode::Logical` (in-process,
`(OrgId, DtuType)`-keyed Tokio task routing), failure injection via
`Harness::inject_failure` / `clear_failure`, and crash detection via `JoinHandle`
monitoring. Parallel startup via `tokio::join!` with 200ms latency budget
(D-058 lock). Use `dtu` feature flag throughout.

## Tasks

1. Create `crates/prism-dtu-harness/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path), `tokio` (1.x full features),
     `axum` (0.7.x), `reqwest` (0.12.x, features = ["json"]), `serde` (1.x),
     `serde_json` (1.x), `thiserror` (1.x), `tracing` (0.1.x), `uuid` (1.x)
   - `[features] dtu = ["prism-dtu-common/dtu"]`
   - All public items gated: `#[cfg(any(test, feature = "dtu"))]`

2. Create `src/error.rs` â€” `HarnessError` enum:
   - `UnknownOrg { slug: String }`
   - `UnknownDtuType { slug: String, dtu_type: String }`
   - `PortConflict { org: OrgId, dtu: DtuType }`
   - `StartupTimeout`
   - `PortExhausted`
   - `CloneCrashed { org_id: OrgId, dtu_type: DtuType, cause: String }`

3. Create `src/types.rs` â€” `IsolationMode`, `DtuType`, `OrgKey = (OrgId, DtuType)`, `CustomerSpec`.

4. Create `src/builder.rs` â€” `HarnessBuilder`:
   - `isolation(mode: IsolationMode) -> Self`
   - `with_customer(slug: &str) -> Self` â€” registers customer using `OrgRegistry` lookup
   - `with_customer_overrides(slug: &str, f: impl FnOnce(&mut CustomerSpec)) -> Self`
   - `build(self) -> Result<Harness, HarnessError>` (async):
     - Allocate a `TcpListener` per `(OrgId, DtuType)` â€” bind all simultaneously before any `start_on` call (D-058)
     - Start all clone tasks in parallel via `tokio::join!`
     - Full 12-clone build MUST complete within 200ms (BC-3.5.001 postcondition 5)
     - On timeout: abort all partially-started tasks; return `Err(StartupTimeout)`
     - Populate immutable `endpoints: HashMap<OrgKey, SocketAddr>`

5. Create `src/harness.rs` â€” `Harness` struct:
   - Fields: `endpoints: HashMap<OrgKey, SocketAddr>`, `crash_channels: HashMap<OrgKey, watch::Receiver<Option<String>>>`, `shutdown_senders: HashMap<OrgKey, oneshot::Sender<()>>`, `task_handles: HashMap<OrgKey, JoinHandle<()>>`
   - `endpoints(&self) -> &HashMap<OrgKey, SocketAddr>` â€” immutable accessor
   - Crash check helper: before each operation, call `try_recv` on crash channel for the target clone; if `Some(cause)` â†’ return `Err(CloneCrashed { ... })`
   - `inject_failure(&self, slug: &str, dtu_type: DtuType, mode: FailureMode) -> Result<(), HarnessError>`:
     - Look up `(OrgId, DtuType)` in `endpoints`; check crash state first
     - POST to `http://{addr}/dtu/configure` with `admin_token` auth
     - Returns `Ok(())` on HTTP 200; maps HTTP errors to `HarnessError`
   - `clear_failure(&self, slug: &str, dtu_type: DtuType) -> Result<(), HarnessError>`:
     - Sends `FailureMode::None` â€” equivalent to clearing any injected mode
   - `impl Drop for Harness`: consume all `shutdown_senders`; await task handles with 5s grace; hard-abort via `handle.abort()` on timeout

6. Create `src/crash_monitor.rs`:
   - Each clone task wraps its main loop in `catch_unwind`
   - On panic: send panic message string to `watch::Sender<Option<String>>` before unwinding
   - On `Err` return: send error `Display` string
   - On premature `Ok`: send `"task exited Ok before test completion"`
   - Crash state persists permanently for harness lifetime (BC-3.6.002 Invariant 2)

7. Create `tests/logical_isolation_test.rs` using the BC-3.5.001 canonical test vectors (TV-1 through TV-7) and BC-3.6.001 TV-1 through TV-6, BC-3.6.002 TV-1 through TV-6.

## Acceptance Criteria

**AC-001:** A `Harness::builder().isolation(IsolationMode::Logical).with_customer("acme-corp").build().await` succeeds; `harness.endpoints()` contains exactly one entry; a query to the acme-corp clone returns HTTP 200.
(traces to BC-3.5.001 precondition 3; postcondition 1)

**AC-002:** In a 3-org harness (acme-corp with Claroty, globex with Armis+CrowdStrike, initech with all 4), device IDs returned for each org contain only that org's `OrgId` namespace prefix; the pairwise intersection of any two orgs' device ID sets is empty.
(traces to BC-3.5.001 postcondition 1, 2; TV-2 through TV-4)

**AC-003:** A concurrent read by OrgA during a write by OrgB returns a response consistent with OrgA's own state; OrgB's write does not appear in OrgA's response.
(traces to BC-3.5.001 postcondition 3; TV-5)

**AC-004:** After `drop(harness)`, `TcpStream::connect` to every previously-recorded clone address returns `ConnectionRefused` within 1 second.
(traces to BC-3.5.001 postcondition 4; invariant 4; TV-6)

**AC-005:** A 12-clone harness (3 orgs Ă— 4 sensor types) completes `build().await` in under 200ms on a standard CI runner (D-058 locked decision; BC-3.5.001 postcondition 5 tightened).
(traces to BC-3.5.001 postcondition 5)

**AC-006:** No two clones in the same harness share a port; `endpoints` map contains exactly `|orgs| Ă— |dtu_types_per_org|` pairwise-distinct `SocketAddr` entries.
(traces to BC-3.5.001 invariant 1, 2)

**AC-007:** `inject_failure("acme-corp", DtuType::Claroty, FailureMode::AuthReject)` causes all subsequent requests to acme-corp's Claroty clone to return HTTP 401; globex's Claroty clone simultaneously returns HTTP 200 with valid data.
(traces to BC-3.6.001 postcondition clause 1 `AuthReject`; postcondition clause 2; TV-1)

**AC-008:** All five `FailureMode` variants (`AuthReject`, `InternalError { after_n }`, `RateLimit { after_n }`, `Timeout { delay_ms }`, `MalformedResponse`) produce the documented HTTP status code or behavior when injected on a specific `(org_slug, dtu_type)`.
(traces to BC-3.6.001 postcondition clause 1)

**AC-009:** `clear_failure` on a previously injected clone causes the clone to resume returning normal (HTTP 200) responses; `inject_failure` and `clear_failure` are idempotent.
(traces to BC-3.6.001 postcondition clause 3, 4; TV-4)

**AC-010:** `inject_failure` with an unknown `org_slug` returns `Err(HarnessError::UnknownOrg)`; no HTTP call is made; no side effects on any clone.
(traces to BC-3.6.001 EC-001; postcondition clause 2)

**AC-011:** When a clone task panics with a string message, the harness detects the crash within 1 second and the next operation targeting that clone returns `Err(HarnessError::CloneCrashed { cause })` where `cause` equals the panic message verbatim.
(traces to BC-3.6.002 postcondition clause 1, 2; TV-1, TV-2)

**AC-012:** A clone crash does not affect other `(OrgId, DtuType)` pairs; operations targeting non-crashed clones continue to return HTTP 200.
(traces to BC-3.6.002 postcondition clause 3; TV-3)

**AC-013:** `drop(harness)` after one or more clone crashes completes without hanging; no zombie Tokio tasks remain.
(traces to BC-3.6.002 postcondition clause 4; TV-4)

**AC-014:** `inject_failure` called on a crashed clone returns `Err(HarnessError::CloneCrashed)`; no HTTP POST is attempted.
(traces to BC-3.6.002 EC-006; BC-3.6.001 EC-004; TV-6)

**AC-015:** A clone that returns `Ok(())` prematurely (before the test ends) is treated as crashed; next operation returns `CloneCrashed { cause: "task exited Ok before test completion" }`.
(traces to BC-3.6.002 EC-003; postcondition clause 2; TV-5)

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Caller queries an org slug not registered | `HarnessError::UnknownOrg`; no HTTP request sent |
| EC-002 | Clone fails to bind during `build()` | `Err(HarnessError::PortConflict { org, dtu })`; no partial `Harness` returned |
| EC-003 | `drop(harness)` while a clone is mid-request | Graceful shutdown; clone completes in-flight request; exits within 5s; else hard abort |
| EC-004 | 12-clone startup exceeds 200ms budget | `Err(HarnessError::StartupTimeout)`; all partially-started tasks aborted |
| EC-005 | Clone panics with non-string payload (`panic!(42u32)`) | `cause = "(non-string panic payload)"`; `CloneCrashed` still returned |
| EC-006 | Two clones crash simultaneously | Both marked crashed independently; other clones unaffected |
| EC-007 | `Timeout` injection with `delay_ms = 0` | Treated as `FailureMode::None`; no latency injected; returns `Ok(())` |
