# S-3.06 — prism-query: PrismQL Write Parser Extensions

## Narrative

As the PrismQL query engine, I want the parser to recognize terminal write stages in
pipe mode, DML statements in SQL mode, and hard-reject writes in filter mode, so that
analysts can compose write operations with query results while the parser enforces
architectural read/write boundaries before any execution begins.

## Objective

Extend the PrismQL Chumsky 0.12 parser to support write operations. Pipe mode gains
terminal action stages (bare verbs from `WriteEndpointRegistry`). SQL mode gains DML
statement recognition (`INSERT INTO`, `UPDATE`, `DELETE`). Filter mode gains a
hard parse-time rejection of any write verb. All write AST nodes carry enough
information for the write execution pipeline (S-3.07) to dispatch without re-parsing.
Per AD-022.

## Tasks

1. Extend PrismQL AST in `crates/prism-query/src/ast.rs` with write node types:
   ```rust
   pub struct WriteNode {
       pub verb: String,                    // e.g. "contain", "acknowledge", "tag"
       pub args: Vec<WriteArg>,             // key=value pairs from pipe stage
       pub source_sensor: Option<String>,   // resolved at parse time from source_stage
   }

   pub struct WriteArg {
       pub key: String,
       pub value: Literal,
   }

   pub enum DmlOperation { InsertInto, Update, Delete }

   pub struct DmlNode {
       pub operation: DmlOperation,
       pub target_table: String,            // e.g. "crowdstrike_contained_hosts"
       pub assignments: Vec<Assignment>,    // for UPDATE: SET col = val pairs
       pub filter: Option<Expr>,            // for UPDATE/DELETE: WHERE clause
       pub source_select: Option<SqlQuery>, // for INSERT INTO ... SELECT ...
   }

   pub struct Assignment {
       pub column: String,
       pub value: Expr,
   }
   ```

   Extend `PipeQuery` to carry an optional terminal write node:
   ```rust
   pub struct PipeQuery {
       pub source: SourceRef,
       pub stages: Vec<PipeStage>,
       pub write: Option<WriteNode>,   // None for read-only pipelines
   }
   ```

   Extend `SqlStatement` to include DML variants alongside `SqlQuery`:
   ```rust
   pub enum SqlStatement {
       Select(SqlQuery),
       Dml(DmlNode),
   }
   ```

2. Implement `WriteVerbRegistry` in `crates/prism-query/src/write_verb_registry.rs`:
   - At parser initialization, load the set of registered write verbs from
     `WriteEndpointRegistry` (passed in as a `&dyn WriteVerbSource` trait object)
   - Exposes `fn is_write_verb(&self, s: &str) -> bool` and
     `fn verbs_for_sensor(&self, sensor: &str) -> Vec<&str>`
   - Registry is immutable after initialization — no hot-reload during a parse call
   - If `WriteEndpointRegistry` is unavailable (e.g. in unit tests), accept an explicit
     set of verbs for test injection

3. Extend pipe mode parser in `crates/prism-query/src/pipe_parser.rs` (BC-2.11.004):
   - Grammar extension:
     ```
     pipe_pipeline = source_stage ("|" pipe_stage)* ("|" write_stage)?
     write_stage   = write_verb (write_arg)*
     write_verb    = <any verb registered in WriteVerbRegistry>
     write_arg     = identifier "=" literal
     ```
   - `write_stage` is recognized only at the terminal position of a pipeline.
     If a `write_stage` appears in a non-terminal position (followed by another `|`
     stage), return `E-QUERY-024` with message: "Write stage must be the final stage
     in a pipeline; stages after 'VERB' are not permitted"
   - Validate that exactly one write stage appears per pipeline (zero is fine for
     read-only pipelines)
   - At parse time, propagate the source sensor name from `source_stage` into
     `WriteNode.source_sensor` (extract sensor prefix from `SourceRef`, e.g.
     `crowdstrike` from `crowdstrike_hosts`)
   - Unknown verbs: if a token in terminal position looks like an identifier but is
     not a registered `pipe_stage` keyword or `write_verb`, return `E-QUERY-001`
     with a suggestion listing available verbs for the source sensor from
     `WriteVerbRegistry::verbs_for_sensor()`

4. Extend SQL mode parser in `crates/prism-query/src/sql_parser.rs` to recognize DML:
   - `INSERT INTO table_name (col_list) SELECT ...` — parse into `DmlNode::InsertInto`
     with `source_select = Some(SqlQuery { ... })`
   - `UPDATE table_name SET col = val [, col = val]* WHERE expr` — parse into
     `DmlNode::Update` with `assignments` and `filter`
   - `DELETE FROM table_name WHERE expr` — parse into `DmlNode::Delete` with `filter`
   - Validate target table against internal table prefix list at parse time:
     internal tables begin with `prism_` — if target matches, return `E-QUERY-010`
     with message: "Internal Prism table 'prism_TABLE' is write-protected; use the
     dedicated MCP tool for this operation"
   - DML without WHERE clause: return `E-QUERY-022 (unbounded write) — DELETE and
     UPDATE must always include a WHERE clause; INSERT INTO ... SELECT must include
     a LIMIT or WHERE on the source SELECT
   - Leverage DataFusion's existing DML token recognition where possible; do not
     re-implement SQL tokenization from scratch

5. Extend filter mode parser in `crates/prism-query/src/filter_parser.rs`:
   - At parse time, if the predicate expression contains any token that matches a
     registered write verb (from `WriteVerbRegistry`), reject immediately with a
     parse error: "Write verbs are not permitted in filter mode; filter mode is
     permanently read-only"
   - This is a grammar-level hard rejection, not a semantic check — it fires
     regardless of whether the verb would resolve to a valid write endpoint

6. Extend `crates/prism-query/src/error.rs` with write-specific parse error variants:
   - `E-QUERY-024`: write stage not terminal — includes pipeline position and verb name
   - `E-QUERY-010`: internal table write attempt — includes table name (re-used from
     read-side; ensure same error code emitted from both parse and execution layers)
   - `E-QUERY-022 (unbounded write — emitted when DML lacks WHERE/LIMIT; includes
     suggestion to add `| head N` or `WHERE` clause

7. Write unit tests in `crates/prism-query/tests/write_parser_tests.rs`:
   - Parse: `FROM crowdstrike_hosts | where last_seen < 7d | contain` → `PipeQuery`
     with `write = Some(WriteNode { verb: "contain", args: [], ... })`
   - Parse: `FROM crowdstrike_hosts | where x = 1 | tag key="review" value="pending"` →
     `WriteNode` with two `WriteArg` entries
   - Reject: `FROM crowdstrike_hosts | contain | where severity >= 3` → `E-QUERY-024`
   - Reject: `UPDATE prism_alerts SET status = 'resolved'` → `E-QUERY-010`
   - Reject: `severity_id >= 4 | contain` (filter mode) → parse error (writes not allowed)
   - Reject: `FROM crowdstrike_hosts | where x = 1 | nonexistent_verb` → `E-QUERY-001`
     with suggestion listing available verbs
   - Parse: `INSERT INTO crowdstrike_contained_hosts (device_id) SELECT device_id FROM
     crowdstrike_hosts WHERE last_seen < 7d LIMIT 100` → `DmlNode::InsertInto`
   - Parse: `DELETE FROM armis_device_tags WHERE device_id = '123'` → `DmlNode::Delete`
   - Reject: `DELETE FROM armis_device_tags` (no WHERE) → `E-QUERY-022`

## Acceptance Criteria

**AC-1:** Given `FROM crowdstrike_hosts | where last_seen < 7d | contain`, When parsed
in pipe mode, Then the AST is a `PipeQuery` with one `Where` stage and
`write = Some(WriteNode { verb: "contain", args: [] })`. (BC-2.11.004)

**AC-2:** Given `FROM crowdstrike_hosts | contain | where severity >= 3`, When parsed,
Then `E-QUERY-024` is returned with message indicating the write stage is not terminal
and no AST is produced.

**AC-3:** Given `UPDATE prism_alerts SET status = 'resolved'`, When parsed, Then
`E-QUERY-010` is returned with message "Internal Prism table 'prism_alerts' is
write-protected" and no DML node is produced.

**AC-4:** Given `severity_id >= 4 | contain`, When parsed in filter mode (expression
begins with a predicate, no FROM keyword), Then a parse error is returned indicating
writes are not permitted in filter mode.

**AC-5:** Given `FROM crowdstrike_hosts | where x = 1 | nonexistent_verb`, When parsed,
Then `E-QUERY-023 (unknown write verb) with suggestion listing available write verbs for
the `crowdstrike` source.

**AC-6:** Given `FROM crowdstrike_hosts | where zone = "OT" | tag key="review" value="pending"`,
When parsed, Then `WriteNode.args` contains `[WriteArg { key: "key", value: "review" },
WriteArg { key: "value", value: "pending" }]`.

**AC-7:** Given `DELETE FROM armis_device_tags` with no WHERE clause, When parsed,
Then `E-QUERY-022` is returned suggesting the analyst add a WHERE clause.

**AC-8:** Given `INSERT INTO crowdstrike_contained_hosts (device_id) SELECT device_id FROM
crowdstrike_hosts WHERE last_seen < 7d LIMIT 10`, When parsed in SQL mode, Then
`DmlNode::InsertInto` is produced with `target_table = "crowdstrike_contained_hosts"`.

## Architecture Compliance Rules

- Parser extensions MUST remain pure: `WriteVerbRegistry` is initialized once before
  parse calls and is immutable during parsing — no `WriteEndpointRegistry` I/O during
  a parse call
- Write verbs in the Chumsky grammar MUST be dynamically registered from
  `WriteVerbRegistry`, NOT hardcoded — the grammar must expand as sensors are added
- `E-QUERY-010` (internal table write-protected) MUST be emitted at parse time for
  SQL DML targeting `prism_*` tables — do not defer this check to execution time
- `WriteNode` and `DmlNode` MUST carry sufficient context for S-3.07 (write execution)
  to dispatch without re-parsing the original query string
- Filter mode write rejection MUST be at the grammar level — `write_verb` is not a
  valid production in the filter mode rule set, not a post-parse semantic check
- Do NOT implement write execution logic in this story — `WriteCapableTableProvider`
  dispatch is S-3.07's scope; this story only produces AST nodes

## Previous Story Intelligence

S-3.01 established the three-mode parser architecture and AST foundation. Key patterns
to carry forward:

- Follow the same module split: each mode has its own `*_parser.rs` file; shared AST
  types live in `ast.rs`. Do NOT add write-specific AST types to individual parser modules.
- The `SourceRef` dot-notation convention from S-3.01 is what this story uses to extract
  the sensor name for `WriteNode.source_sensor`. The extraction is: split on `_` to get
  the sensor prefix (e.g. `crowdstrike` from `crowdstrike_hosts`). Use the same helper
  that S-3.02 uses for source resolution.
- S-3.01's `security.rs` query size check (`PRISM_MAX_QUERY_SIZE`) applies to write
  queries identically — no exemptions.
- Chumsky 0.12 `recover_with` strategies from S-3.01 apply to write stage parsing. A
  malformed write arg (e.g. `tag key`) should attempt recovery rather than halting all
  error collection.

## Notes

- The `WriteVerbSource` trait abstraction allows write parser tests to inject a static
  verb set without requiring `WriteEndpointRegistry` to be initialized with actual sensor
  specs. Define the trait in `write_verb_registry.rs` and implement it for both
  `WriteEndpointRegistry` (production) and `HashSet<String>` (tests).
- Chumsky's dynamic dispatch for write verbs (the verb set is a runtime value, not a
  compile-time literal) requires using `choice()` over a collected iterator of verb
  parsers, not a static `one_of![]` macro — verify this pattern against Chumsky 0.12 API.
- The parser does NOT validate that a write verb is available for the specific source
  table (e.g. `FROM armis_devices | contain` where `contain` is CrowdStrike-only). This
  cross-sensor write verb validation is semantic and belongs in S-3.07's Phase 2 safety
  pre-check, not the parser. The parser accepts any registered verb from any sensor.
- SQL DML parsing can leverage DataFusion's `DmlStatement` AST from `datafusion-sql` as
  the substrate for tokenization, then project into Prism's `DmlNode`. This avoids
  duplicating SQL keyword parsing while keeping Prism's AST decoupled from DataFusion
  internals.
