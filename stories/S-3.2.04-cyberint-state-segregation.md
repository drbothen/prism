# S-3.2.04 — prism-dtu-cyberint: Multi-tenant state segregation

## Narrative

As a Prism MSSP operator, I want the Cyberint DTU clone to key both its alert status
store and its session token store by `OrgId` so that Cyberint session tokens issued
for Client A cannot be accepted as valid in Client B's query context, even if both
clients' tokens happen to share the same UUID v4 string.

## Objective

Migrate `prism-dtu-cyberint/src/state.rs` per ADR-008 §2.1 Step 6d:
- `alert_store` (line 52): `HashMap<String, AlertStatus>` → `HashMap<(OrgId, String), AlertStatus>`
- `session_store` (line 56): `HashSet<String>` → `HashSet<(OrgId, String)>`
- Update `build_alert_store` (line 109) to accept `OrgId` parameter.
- Update `register_session` (line 214) and `is_valid_session` to take `(OrgId, token)`.
- Add `reset_for(org_id)`, rename `reset()` → `reset_all()`.
- Add per-org proptest coverage (TD-DTU-MUTATE-COVERAGE-001, VP-084 through VP-086).

This story has the largest blast radius of the four client-mode DTU migrations because
`build_alert_store` is a construction-time API change.

## Tasks

1. **Migrate `alert_store`** (`src/state.rs` line 52):
   ```rust
   // Before
   pub alert_store: Mutex<HashMap<String, AlertStatus>>,
   // After
   pub alert_store: Mutex<HashMap<(OrgId, String), AlertStatus>>,
   ```

2. **Migrate `session_store`** (`src/state.rs` line 56):
   ```rust
   // Before
   pub session_store: Mutex<HashSet<String>>,
   // After
   pub session_store: Mutex<HashSet<(OrgId, String)>>,
   ```

3. **Update `build_alert_store`** (`src/state.rs` line 109):
   - Add `org_id: OrgId` parameter.
   - Change key construction from `alert.alert_id.clone()` to
     `(org_id, alert.alert_id.clone())`.
   - If `CyberintState` is constructed before `OrgId` is known (two-phase construction
     risk per ADR-008 §8 Q2), use `DEFAULT_ORG_ID` in `#[cfg(test)]` contexts only;
     production callers must pass the real `OrgId` at construction time.

4. **Update `register_session`** (`src/state.rs` line 214):
   ```rust
   pub fn register_session(&self, org_id: OrgId, token: String) {
       let mut store = self.session_store.lock().expect("session_store poisoned");
       store.insert((org_id, token));
   }
   ```

5. **Update `is_valid_session`**:
   ```rust
   pub fn is_valid_session(&self, org_id: OrgId, token: &str) -> bool {
       let store = self.session_store.lock().expect("session_store poisoned");
       store.contains(&(org_id, token.to_owned()))
   }
   ```

6. **Add `DEFAULT_ORG_ID` test constant** (`#[cfg(test)]` only) and update existing tests.

7. **Add `reset_for(org_id: OrgId)`** and rename `reset()` → `reset_all()` with shim.
   `reset_for` clears both `alert_store` and `session_store` for the given org.

8. **Add `tests/multi_tenant.rs`**:
   - `prop_alert_cross_org_isolation` — write alert status for org_A, assert org_B lookup is None.
   - `prop_session_cross_org_isolation` — register session for org_A, assert `is_valid_session(org_B, ...)` is false.
   - `prop_session_identical_token_independent` — same UUID v4 token registered for both orgs; each is valid only in its own context.
   - `prop_reset_for_selectivity` — reset org_A; org_B alert + session entries intact.
   - `prop_refresh_preserves_org_binding` — token refresh stores new token under same OrgId.

9. **Gate**: `cargo test -p prism-dtu-cyberint` green.

## Acceptance Criteria

**AC-001** — Alert store cross-org isolation (traces to BC-3.2.001 postcondition 1):
Given alert `"alert-007"` has `AlertStatus::Open` stored for `org_id_A`,
When `state.alert_store` is queried with key `(org_id_B, "alert-007")`,
Then the result is `None`.

**AC-002** — Session cross-org isolation (traces to BC-3.2.003 postcondition 2):
Given session token `"tok-abc"` is registered for `org_id_A`,
When `state.is_valid_session(org_id_B, "tok-abc")` is called,
Then the result is `false`.

**AC-003** — Same token string, independent contexts (traces to BC-3.2.003 edge case EC-001):
Given the same UUID v4 string `"tok-shared"` is registered independently for both org_A and org_B,
When each org calls `is_valid_session` with their own OrgId and `"tok-shared"`,
Then both return `true` — each has an independent entry.

**AC-004** — Token refresh preserves OrgId binding (traces to BC-3.2.003 postcondition 3):
Given org_A has token `"old-tok"`,
When org_A refreshes to `"new-tok"` (removing old, inserting new under same OrgId),
Then `is_valid_session(org_B, "old-tok")` is still `false` and org_B's token is unaffected.

**AC-005** — `build_alert_store` accepts OrgId parameter (traces to BC-3.2.001 invariant 1):
Given `CyberintState` is constructed with `org_id: OrgId` and fixture data,
When `build_alert_store(org_id, fixture_data)` is called,
Then the resulting map has keys of the form `(org_id, alert_id)` for all fixture alerts.

**AC-006** — `reset_for` clears both stores for one org (traces to BC-3.2.001 edge case EC-004):
Given alerts and session tokens for both org_A and org_B,
When `state.reset_for(org_id_A)` is called,
Then org_A's alert_store and session_store are empty; org_B's entries are intact.

**AC-007** — OrgId-flipping proptest kills mutation (traces to BC-3.2.001 VP-079):
Given adversarial org_A ≠ org_B and any shared alert_id or token,
When state is written under org_A and read under org_B,
Then the result is always `None` / `false`.

## Architecture Compliance Rules

- **Forbidden dependencies:** `prism-dtu-cyberint` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`.
- `DEFAULT_ORG_ID` MUST be `#[cfg(test)]` only. Production `build_alert_store` callers
  MUST supply a real `OrgId` — no sentinel fallback in production paths.
- BOTH `alert_store` AND `session_store` MUST be migrated atomically in the same PR.
  Partial migration violates the type invariant.
- `is_valid_session` signature MUST take `(org_id: OrgId, token: &str)` — never
  `(token: &str)` alone. This is the structural isolation guarantee for BC-3.2.003.
- `reset_for` MUST clear both stores for the given OrgId in a single logical operation
  (may acquire and release each Mutex separately, sequentially).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Two orgs both acquire the same UUID v4 token string | Stored as `(org_A, token)` and `(org_B, token)` independently; each valid only in its own context |
| EC-002 | org_A token expires; refresh produces new token; org_B unaffected | `(org_A, new_token)` replaces `(org_A, old_token)`; `(org_B, tok)` unchanged |
| EC-003 | `build_alert_store` called before OrgId known (ADR-008 §8 Q2) | Use `DEFAULT_ORG_ID` only in `#[cfg(test)]`; production callers must have OrgId at construction time |
| EC-004 | `is_valid_session(org_B, tok_A)` where tok_A registered only for org_A | Returns `false`; no error propagation |
