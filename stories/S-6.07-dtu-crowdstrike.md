# S-6.07 — prism-dtu-crowdstrike: DTU for CrowdStrike Falcon API — L4 (adversarial)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the CrowdStrike Falcon
API at fidelity level L4 (adversarial), so that integration tests for sensor read paths,
write paths, VP-033 (audit buffer ordering), and VP-036 (SessionContext drop) can run
deterministically without real CrowdStrike credentials or API access.

## Objective

Implement `prism-dtu-crowdstrike` — a full L4 (adversarial) behavioral clone of the
CrowdStrike Falcon API. The clone implements all 8 in-scope endpoints (4 read, 4 write),
maintains a stateful write store (device containment status) and a session-scoped ID
registry for two-step fetch, supports OAuth2 token endpoint simulation, and can inject
configurable failure modes via the shared `FailureLayer` from `prism-dtu-common`. This
crate wires VP-033 and VP-036 integration tests. See `dtu-assessment.md §3.1` for full
scope matrix.

## Tasks

1. Create `crates/prism-dtu-crowdstrike/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `uuid` (1.x with v4 feature), `lru` (0.12.x)
   - `[features] dtu = []`
   - This crate is a dev-dependency of `prism-sensors` and `prism-audit`

2. Create `src/lib.rs` — `CrowdstrikeClone` struct implementing `BehavioralClone`:
   - Fields: `config: StubConfig`, `state: Arc<CrowdstrikeState>`,
     `server_handle: Option<JoinHandle<()>>`, `bound_addr: Option<SocketAddr>`
   - `start()`: binds `TcpListener::bind("127.0.0.1:0")`, spawns axum router, stores
     `bound_addr`
   - `reset()`: calls `state.reset()` — clears containment store and session registry
   - `configure(config: Value)`: parses `{ "auth_mode": "reject" }` or failure config,
     updates `FailureLayer` state

3. Create `src/state.rs` — `CrowdstrikeState`:
   - `containment_store: Mutex<HashMap<String, ContainmentStatus>>` —
     `device_id → ContainmentStatus { status: "normal"|"contained", updated_at }`
   - `detection_status_store: Mutex<HashMap<String, String>>` —
     `detection_id → status string`
   - `session_registry: Mutex<LruCache<String, SessionData>>` — max 1,000 entries;
     keyed by `X-DTU-Session-Id` header value
   - `SessionData { detection_ids: Vec<String>, host_ids: Vec<String> }`
   - `reset()` clears all three stores

4. Create `src/routes/oauth.rs`:
   - `POST /oauth2/token`: accepts `client_credentials` grant; returns
     `{"access_token": "dtu-fake-cs-token", "token_type": "bearer", "expires_in": 3600}`
   - If `auth_mode == "reject"`: return 401 with
     `{"errors": [{"code": 401, "message": "invalid_client"}]}`

5. Create `src/routes/detections.rs`:
   - `GET /detects/queries/detects/v1`: paginated detection ID list
     - Query params: `filter` (FQL string, accepted but not parsed), `limit` (default 100),
       `offset` (integer)
     - Auth: validate `Authorization: Bearer {any-non-empty}` header; 401 if absent
     - Registers returned IDs in session registry keyed by `X-DTU-Session-Id`
     - Response: `{"resources": ["det-001", "det-002", ...], "meta": {"pagination": {"offset": N, "limit": N, "total": N}}, "next_token": "..." or null}`
     - Loads fixture from `fixtures/detections-ids.json`
   - `POST /detects/entities/summaries/GET/v1`: batch detail fetch
     - Body: `{"ids": ["det-001", ...]}`; max batch 100; 400 if empty
     - Looks up IDs in session registry; returns matching fixture records
     - Response: `{"resources": [{detection object}, ...]}`
     - Loads fixture from `fixtures/detections-detail.json`

6. Create `src/routes/hosts.rs`:
   - `GET /devices/queries/devices/v1`: paginated host ID list
     - Same pagination pattern as detection IDs
     - Registers returned IDs in session registry
     - Loads fixture from `fixtures/hosts-ids.json`
   - `GET /devices/entities/devices/v2`: batch host detail fetch
     - Query param: `ids` (repeated, e.g., `?ids=h-001&ids=h-002`)
     - Merges containment status from `containment_store` into each device record
     - Loads base records from `fixtures/hosts-detail.json`

7. Create `src/routes/writes.rs`:
   - `POST /devices/entities/devices-actions/v2`:
     - `action_name=contain`: body `{"ids": ["device_id"]}` — 400 if ids empty;
       transitions device to `containment_status: "contained"` in store;
       returns 202 `{"resources": [{"device_id": "...", "containment_status": "contained"}]}`
     - `action_name=lift_containment`: transitions to `containment_status: "normal"`;
       returns 202 `{"resources": [{"device_id": "...", "containment_status": "normal"}]}`
     - Error: `contain` on already-contained device → 400
       `{"errors": [{"code": 400, "message": "device already contained"}]}`
   - `PATCH /detects/entities/detects/v2`:
     - Dispatches by presence of `assigned_to_uid` in body:
       - `update_status` path: body `{"ids": [...], "status": "..."}` — updates detection_status_store
       - `assign` path: body `{"ids": [...], "assigned_to_uid": "..."}` — records assignment
     - Returns 200 `{}` on success

8. Create fixture files:
   - `fixtures/detections-ids.json`: 50 fake detection IDs (`["det-001", ..., "det-050"]`)
   - `fixtures/detections-detail.json`: 50 detection objects with fields: `detection_id`,
     `status`, `severity`, `created_timestamp`, `updated_timestamp`, `device.device_id`,
     `device.hostname`, `behaviors[].tactic`, `behaviors[].technique`
   - `fixtures/hosts-ids.json`: 30 fake host IDs
   - `fixtures/hosts-detail.json`: 30 host objects with fields: `device_id`, `hostname`,
     `platform_name`, `os_version`, `status`, `containment_status`, `last_seen`,
     `external_ip`, `local_ip`, `agent_version`

9. Wire VP-033 integration test in `tests/integration_vp033.rs`:
   - Start `CrowdstrikeClone` with `StubConfig::default()` (seed 42, no failure)
   - Configure `prism-audit` backend (InMemoryBackend)
   - Configure `SensorAdapter` with `base_url = clone.base_url()`
   - Issue contain write; capture DTU request arrival timestamp via in-memory log
   - Assert: `WRITE_INTENT` audit record in backend BEFORE DTU request arrival timestamp
   - Assert: `WRITE_OUTCOME` audit record in backend AFTER DTU returns 202
   - Test must pass as part of `just integration-test`

10. Wire VP-036 integration test in `tests/integration_vp036.rs`:
    - Start `CrowdstrikeClone` with `FailureMode::InternalError { at_request_n: 2 }` (Step 2 fails)
    - Execute a full `crowdstrike_hosts` query (Step 1 succeeds, Step 2 returns 500)
    - Assert: `SessionContext` is dropped (via `Arc::weak_count` check) BEFORE `E-SENSOR-002`
      is returned to the caller
    - Test must pass as part of `just integration-test`

11. Create `tests/fidelity.rs`:
    - `#[tokio::test] fn crowdstrike_dtu_fidelity()`:
      - Starts `CrowdstrikeClone`
      - Runs `FidelityValidator` against all 8 endpoints
      - Asserts `FidelityReport.checks_failed == 0`
    - Must run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `CrowdstrikeClone::start()` is called, Then `bound_addr()` returns a
valid socket address and `GET /detects/queries/detects/v1` with a valid Bearer token
returns HTTP 200 with a `resources` array.

**AC-2:** Given `GET /devices/queries/devices/v1` is called (Step 1), Then the response
`resources` array contains host IDs AND those IDs are registered in the session registry
under the `X-DTU-Session-Id` header value; subsequent `GET /devices/entities/devices/v2`
with those IDs returns matching host detail records (two-step pipeline works).

**AC-3:** Given `POST /devices/entities/devices-actions/v2?action_name=contain` is
called with `{"ids": ["h-001"]}`, Then the response is HTTP 202 with
`containment_status: "contained"`, AND subsequent `GET /devices/entities/devices/v2?ids=h-001`
returns `containment_status: "contained"` in the host record (stateful write persists).

**AC-4:** Given `FailureMode::RateLimit { after_n_requests: 3, retry_after_secs: 60 }`
is configured, When the 4th request to any endpoint arrives, Then the response is HTTP 429
with `Retry-After: 60` header.

**AC-5:** Given `POST /oauth2/token` is called with any `client_credentials` body, Then
the response is HTTP 200 with `access_token: "dtu-fake-cs-token"`.

**AC-6:** Given `CrowdstrikeClone` is started with `seed: 42`, When
`GET /detects/queries/detects/v1` is called twice with the same query params, Then both
responses are identical (deterministic seeding).

**AC-7:** Given a request to any auth-required endpoint without an `Authorization` header,
Then the response is HTTP 401 with `{"errors": [{"code": 401, "message": "..."}]}`.

**AC-8:** Given `reset()` is called, Then the containment store, detection status store,
and session registry are all cleared; a subsequent `GET /devices/entities/devices/v2?ids=h-001`
returns the device with `containment_status: "normal"` (base fixture state).

**AC-9:** Given VP-033 integration test runs against this DTU, Then the test passes:
`WRITE_INTENT` audit record is durable before the DTU receives the HTTP call.

**AC-10:** Given VP-036 integration test runs against this DTU with Step 2 configured to
fail, Then the test passes: `SessionContext` drops before `E-SENSOR-002` propagates.

## Architecture Compliance Rules

- DTU clone MUST match the CrowdStrike Falcon API contract exactly (same endpoints,
  same request/response shapes, same status codes) as documented in `dtu-assessment.md §3.1`.
- DTU clone MUST NOT introduce new behaviors not present in the real CrowdStrike API.
- DTU clone MUST be deterministic for a fixed seed (ChaCha20Rng from `prism-dtu-common`);
  use of `rand::thread_rng()` is forbidden.
- DTU clone MUST NOT require network access during test execution — all state is in-process.
- This crate MUST be gated behind `#[cfg(any(test, feature = "dtu"))]` — never compiled
  into production binaries.
- See `.factory/specs/architecture/dtu-strategy.md` (or `dtu-assessment.md §4`) for
  cross-cutting DTU rules including ephemeral port binding and `BehavioralClone` contract.
- **Forbidden dependencies:** `prism-dtu-crowdstrike` MUST NOT depend on `prism-sensors`,
  `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`. Adding any of
  these to `Cargo.toml` MUST fail the build via deny rule.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `contain` called with empty `ids` array | HTTP 400 `{"errors": [{"code": 400, "message": "ids array must not be empty"}]}` |
| EC-002 | `contain` on a device already in `contained` state | HTTP 400 `{"errors": [{"code": 400, "message": "device already contained"}]}` — partial write failure path |
| EC-003 | Step 2 called with IDs not in session registry | Returns 200 with empty `resources` array (IDs not found is not an error in the real API) |
| EC-004 | Session registry at LRU capacity (1,000 entries) | Oldest session evicted; new session registered; no error |
| EC-005 | Mid-pagination 503 on Step 2 batch 2 of 3 | `FailureMode::InternalError { at_request_n: 3 }` triggers; maps to E-SENSOR-005 partial results |
| EC-006 | `reset()` called during active query | State cleared; in-flight Step 2 calls return empty resources; no panic |
