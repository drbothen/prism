# S-4.06 â€” prism-operations: Case Management

## Narrative

As a Prism operations platform, I want a full case management subsystem with create,
transition, annotate, list, and retrieve operations, so that security analysts can
organize related alerts into cases, track investigation progress through a defined state
machine, and build a structured audit trail of all case activity.

## Objective

Implement case management: `create_case` from alert IDs, a 5-state case state machine
with 12 valid transitions, `update_case` for transitions/dispositions/annotations,
`list_cases` with filtering, `get_case` for full detail, disposition enforcement on
Resolved transitions, append-only timeline with 5 annotation types, and RocksDB
persistence for full restart recovery.

## Tasks

1. Define case types in `crates/prism-operations/src/cases/case.rs`:
   - `Case { id: CaseId, title: String, status: CaseStatus, severity: SeverityId,
     client_id: ClientId, alert_ids: Vec<AlertId>, disposition: Option<DispositionCode>,
     timeline: Vec<TimelineEntry>, created_at: Timestamp, updated_at: Timestamp,
     created_by: AnalystId }`
   - `CaseId`: UUID v7 (time-sortable)
   - `CaseStatus`: uses prism-core's 5-state enum â€” `New`, `Acknowledged`, `Investigating`,
     `Resolved`, `Closed` (per BC-2.14.002)
   - 12 valid transitions from prism-core's `CaseStatus::can_transition_to()` (S-1.02):
     Forward: Newâ†’Acknowledged, Acknowledgedâ†’Investigating, Investigatingâ†’Resolved, Resolvedâ†’Closed
     Skip-ahead: Newâ†’Investigating, Newâ†’Resolved, Newâ†’Closed, Acknowledgedâ†’Resolved, Acknowledgedâ†’Closed, Investigatingâ†’Closed
     Reopen: Resolvedâ†’Investigating, Closedâ†’Investigating
   - All other transitions return E-CASE-004 (invalid transition)
   - `DispositionCode`: uses prism-core's enum â€” `TruePositive`, `FalsePositive`, `Benign`,
     `Inconclusive`, `Duplicate`, `TestAlert` (per S-1.02)
   - `TimelineEntry { id: Uuid, entry_type: TimelineEntryType, content: String,
     created_by: AnalystId, created_at: Timestamp }`
   - `TimelineEntryType`: enum â€” `Note`, `StatusChange`, `AlertLink`, `EvidenceLink`,
     `OtImpact`

2. Implement `create_case` MCP tool handler in
   `crates/prism-operations/src/cases/create_case.rs` (BC-2.14.001):
   - Parameters: `title`, `alert_ids: Vec<AlertId>` (1..=100), `severity: SeverityId`,
     `client_id: ClientId`
   - Validate: all `alert_ids` exist in the `alerts` CF; all alerts belong to the
     specified `client_id`
   - Create `Case` with status `New`, resolved_at: None, closed_at: None, no disposition, empty timeline
   - Auto-generate first timeline entry: `StatusChange` â†’ `"Case created with {n} alerts"`
   - Persist to RocksDB `cases` CF
   - Return created `Case`

3. Implement `update_case` MCP tool handler in
   `crates/prism-operations/src/cases/update_case.rs` (BC-2.14.003):
   - Parameters: `case_id`, optional `status: CaseStatus`, optional
     `disposition: DispositionCode`, optional `annotation: AnnotationInput`
   - All parameters are optional; at least one must be provided (return E-CASE-005 if
     all are absent)
   - Transition: if `status` is provided, call `CaseStatus::can_transition_to()`;
     return E-CASE-004 on invalid transition, E-CASE-005 on self-transition (per BC-2.14.002)
   - Timestamp management (per BC-2.14.002):
     - On Resolved: set `resolved_at` to now (only if null â€” preserve first resolution time)
     - On Closed: set `closed_at` to now
     - On reopen (Resolved/Closed â†’ Investigating): clear `closed_at`, preserve `resolved_at`
   - Alert cascade (per BC-2.14.002):
     - On Resolved: set `resolved_at` on all linked alerts via `source_alert_ids`
     - On reopen: clear linked alerts' `resolved_at`
   - Disposition: if `status == Resolved` and no disposition provided (current or new),
     return E-CASE-006 (disposition required, per BC-2.14.002)
   - Auto-generate `StatusChange` timeline entry when status changes
   - Persist updated `Case` to RocksDB in a single write

4. Implement `list_cases` MCP tool handler in
   `crates/prism-operations/src/cases/list_cases.rs` (BC-2.14.004):
   - Parameters: optional `status`, optional `client_id`, optional `severity`,
     optional `created_after: Timestamp`, optional `created_before: Timestamp`,
     optional `page_cursor`, `page_size` (default 20, max 100)
   - Scan `cases` CF with prefix `case:`, apply filters, return paginated results
   - Each result item: `id`, `title`, `status`, `severity`, `client_id`, `alert_count`,
     `created_at`, `updated_at`
   - Cursor-based pagination using `CaseId` (UUID v7) as cursor

5. Implement `get_case` MCP tool handler in
   `crates/prism-operations/src/cases/get_case.rs` (BC-2.14.005):
   - Parameters: `case_id`
   - Return full `Case` including: all fields, complete `timeline`, list of linked
     `Alert` summaries (id, title, severity, triggered_at), `disposition`
   - Return E-CASE-001 if case not found

6. Enforce disposition requirement in `update_case` (BC-2.14.006):
   - A transition to `Resolved` MUST have a `DispositionCode` set â€” either from the
     current case state or provided in the same `update_case` call
   - Return E-CASE-006 if the `Resolved` transition is attempted without a disposition

7. Implement timeline management in
   `crates/prism-operations/src/cases/timeline.rs` (BC-2.14.007):
   - Timeline is append-only. Entries are never deleted or modified.
   - `Note`: free-text annotation from analyst; max 10,000 characters
   - `StatusChange`: auto-generated on every status transition; content is
     `"{prev_status} â†’ {new_status} by {analyst_id}"`
   - `AlertLink`: link to an additional alert not in the original create_case call;
     content is the alert ID
   - `EvidenceLink`: external evidence URL or file reference; content is the URL/path
   - `OtImpact`: OT/ICS-specific impact annotation; content describes the OT asset or
     process impacted
   - All entries: UUID v4 id, `AnalystId` from the calling session, `Timestamp` of
     creation

8. Implement case persistence helpers in
   `crates/prism-operations/src/cases/persistence.rs` (BC-2.14.009):
   - `save_case(case: &Case)`: serialize to bincode, write to RocksDB `cases` CF under
     key `case:{id}`
   - `load_case(id: CaseId) -> Option<Case>`: read and deserialize
   - `list_cases_paginated(filter, cursor, page_size) -> (Vec<CaseSummary>, Option<Cursor>)`:
     prefix scan with filter application
   - On startup: scan `cases` CF to warm in-memory index (status â†’ Vec<CaseId>) for
     O(1) status-based filtering

## Acceptance Criteria

**AC-1:** Given 3 valid alert IDs for client "acme", When `create_case` is called, Then
a case is created with status `New`, 3 linked alerts, and an initial `StatusChange`
timeline entry. (BC-2.14.001)

**AC-2:** Given an alert ID that does not exist, When `create_case` is called with that
ID, Then E-CASE-008 (alert not found) is returned and no case is created. (BC-2.14.001)

**AC-3:** Given a case in `New` status, When `update_case(status: Acknowledged)` is
called, Then the transition succeeds, a `StatusChange` timeline entry is appended, and
the case status is `Acknowledged`. (BC-2.14.002, BC-2.14.003)

**AC-4:** Given a case in `Resolved` status, When `update_case(status: Acknowledged)` is
called, Then E-CASE-004 (invalid transition) is returned â€” backward transitions to
Acknowledged are not permitted, only reopen to Investigating. (BC-2.14.002)

**AC-5:** Given a case in `Investigating` status, When `update_case(status: Resolved)` is
called without a `disposition`, Then E-CASE-006 (disposition required) is returned.
(BC-2.14.006)

**AC-6:** Given a case in `Investigating` status, When `update_case(status: Resolved,
disposition: TruePositive)` is called, Then the transition succeeds, disposition is
set, and a `StatusChange` timeline entry is appended. (BC-2.14.006)

**AC-7:** Given 20 cases, When `list_cases(status: New, client_id: "acme")` is called,
Then only New cases for client "acme" are returned. (BC-2.14.004)

**AC-8:** Given a case with 3 timeline entries, When `get_case` is called, Then the full
case including all 3 timeline entries and linked alert summaries is returned. (BC-2.14.005)

**AC-9:** Given `update_case` with an `annotation` of type `"ot_impact"` and content
`"PLC unit 3 impacted"`, When processed, Then the timeline entry is appended with type
`OtImpact`. (BC-2.14.007)

**AC-10:** Given a case persisted to RocksDB, When Prism restarts, Then `get_case`
returns the full case including all timeline entries â€” complete recovery. (BC-2.14.009)

## Architecture Compliance Rules

- `CaseStatus::can_transition_to()` MUST be implemented as a pure function on the
  `CaseStatus` enum in `prism-core` (S-1.02 established the entity types). The 12
  valid transitions are a closed set â€” do not use a dynamic rule table.
- Timeline is append-only. There is no `update_timeline_entry` or `delete_timeline_entry`
  operation. This is an audit trail, not a mutable notes field.
- Case persistence uses bincode serialization for consistency with other RocksDB domains (cases can have large
  timelines). Do not use JSON for case persistence in the `cases` CF.
- The in-memory status index (status â†’ Vec<CaseId>) is a read optimization. It must be
  rebuilt from the `cases` CF on startup and updated on every `update_case` write.
  It is not the source of truth â€” RocksDB is.
- `update_case` MUST write the entire updated `Case` struct in a single RocksDB put.
  Do not use separate puts for status, disposition, and timeline â€” partial writes during
  a crash would leave the case in an inconsistent state.

## Notes

- The `ot_impact` annotation type is specific to the OT/ICS context of Prism's MSSP
  deployment (1898 & Co serves industrial clients). It allows analysts to record which
  OT assets (PLCs, RTUs, HMIs) were impacted by a security incident, which is critical
  for incident response in OT environments.
- If MSSP workflows need a "pending vendor" state (e.g., blocked on CrowdStrike support),
  model this as a timeline annotation of type `Note` on a case in `Investigating` status.
  The 5-state machine (New/Acknowledged/Investigating/Resolved/Closed) is intentionally
  minimal per BC-2.14.002.
- Cursor-based pagination using UUID v7 CaseId is correct because UUID v7 is
  time-sortable. A cursor value of `case_id_X` means "return cases with ID > X in
  lexicographic order," which equals "return cases created after case_id_X."
- The 12 valid transitions include two "reopen" paths: `Resolved â†’ Investigating` and
  `Closed â†’ Investigating` (per BC-2.14.002). Reopening always returns to Investigating,
  never to New or Acknowledged. On reopen, `closed_at` is cleared but `resolved_at` is
  preserved (first resolution time for accurate MTTR).
