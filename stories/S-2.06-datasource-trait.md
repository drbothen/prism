# S-2.06 ŌĆö prism-sensors: DataSource Trait and Auth Patterns

## Narrative

As a Prism query engine, I want a unified `SensorAdapter` trait and cross-client fan-out
orchestrator so that queries targeting multiple clients and sensors execute in parallel
with consistent partial failure handling and retry semantics, without per-sensor
boilerplate in the query layer.

## Objective

Implement the `prism-sensors` crate's adapter framework: the `SensorAdapter` trait,
`SensorAuth` sealed trait, `AdapterRegistry`, cross-client fan-out with up to 10
concurrent fetches per query, partial failure collection, exponential backoff with jitter
for transient errors, and a global HTTP connection semaphore capped at 200.

## Tasks

1. Define `SensorAdapter` trait in `crates/prism-sensors/src/adapter.rs`:
   - `async fn fetch(spec: &SensorSpec, params: &QueryParams, auth: &dyn SensorAuth)
     -> Result<Vec<RecordBatch>, SensorError>`
   - `fn sensor_type(&self) -> SensorType` ŌĆö identifies which sensor platform this
     adapter handles
   - Trait must be `Send + Sync + 'static` and object-safe (BC-2.01.013)
   - `SensorSpec` contains: `source_table: String`, `client_id: String`,
     `sensor_config: serde_json::Value`

2. Define `SensorAuth` sealed trait in `crates/prism-sensors/src/auth/mod.rs`:
   - Sealed via a private marker trait (`private::Sealed`): prevents external
     implementations and cross-sensor auth composition at compile time
   - Subtypes: `CrowdStrikeAuth`, `CyberintAuth`, `ClarotyAuth`, `ArmisAuth`
     ŌĆö each defined in its own module (`auth/crowdstrike.rs`, etc.)
   - Each auth type holds only its own credentials (no overlap in field types)

3. Implement `AdapterRegistry` in `crates/prism-sensors/src/registry.rs`:
   - `HashMap<SensorType, Arc<dyn SensorAdapter>>`
   - `register(adapter: Arc<dyn SensorAdapter>)` ŌĆö registers an adapter by sensor type
   - `get(sensor_type: SensorType) -> Option<Arc<dyn SensorAdapter>>`
   - Initialized at startup with all 4 sensor adapters (BC-2.01.013)

4. Implement cross-client fan-out orchestrator in `crates/prism-sensors/src/fanout.rs`:
   - `fan_out(targets: Vec<FanOutTarget>, registry: &AdapterRegistry,
     credentials: &CredentialStore) -> FanOutResult`
   - `FanOutTarget { client_id: String, sensor_type: SensorType, spec: SensorSpec }`
   - Spawn one tokio task per target using `tokio::spawn`; collect with
     `futures::future::join_all`
   - Limit concurrency to 10 per query via a `tokio::sync::Semaphore` with 10 permits
     acquired per task (BC-2.01.002)
   - Return `FanOutResult { successes: Vec<RecordBatch>, errors: Vec<FanOutError> }`

5. Implement partial failure handling in `fanout.rs`:
   - `FanOutError { client_id: String, sensor_type: SensorType, error: SensorError,
     retry_metadata: RetryMetadata }`
   - `RetryMetadata { attempts: u32, last_error_code: String, is_transient: bool }`
   - If at least one target succeeds, return partial results with all errors in
     `FanOutResult.errors` ŌĆö do NOT fail the whole query (BC-2.01.010)
   - If ALL targets fail, return `Err(SensorError::AllTargetsFailed { errors: Vec<FanOutError> })`

6. Implement exponential backoff with jitter in `crates/prism-sensors/src/retry.rs`:
   - `retry_with_backoff<F, T>(op: F, config: RetryConfig) -> Result<T, SensorError>`
   - `RetryConfig { base_delay_ms: u64 = 2000, multiplier: f64 = 2.0, max_delay_ms: u64 = 30_000,
     max_attempts: u32 = 3, transient_codes: &'static [u16] }`
   - Transient HTTP status codes: 429, 503, 504, 500ŌĆō599
   - Jitter: `delay = rand::random::<f64>() * computed_delay` (full jitter) (BC-2.01.014)
   - Non-transient errors (401, 403, 400, 404) are NOT retried; return immediately

7. Implement global HTTP connection semaphore in `crates/prism-sensors/src/http.rs`:
   - `static HTTP_SEMAPHORE: OnceLock<Semaphore>` initialized with 200 permits at startup
   - All sensor HTTP clients MUST acquire a permit before sending a request and release
     on response or error
   - Permit acquisition is timeout-bounded: 30s timeout; if exceeded, return
     `Err(SensorError::ConnectionPoolExhausted)`

## Acceptance Criteria

**AC-1:** Given a fan-out targeting 3 clients ├Ś 2 sensors (6 targets), When `fan_out()`
is called, Then 6 tokio tasks are spawned and all run in parallel (observable via
tracing spans with distinct `client_id` and `sensor_type` fields). (BC-2.01.002)

**AC-2:** Given 5 of 6 targets succeed and 1 returns HTTP 503, When `fan_out()` completes,
Then `FanOutResult.successes.len() == 5` and `FanOutResult.errors.len() == 1` with
`retry_metadata.is_transient == true`. (BC-2.01.010)

**AC-3:** Given a `CrowdStrikeAdapter` registered in `AdapterRegistry`, When
`registry.get(SensorType::CrowdStrike)` is called, Then the same adapter instance is
returned. (BC-2.01.013)

**AC-4:** Given a transient 429 response on the first attempt, When `retry_with_backoff()`
runs, Then the second attempt waits at least `base_delay_ms` milliseconds with jitter
applied. (BC-2.01.014)

**AC-5:** Given 200 active HTTP connections holding semaphore permits, When a 201st
`fan_out()` task attempts to acquire a permit, Then it blocks until a permit is released
(not rejected immediately).

## Architecture Compliance Rules

- `SensorAdapter` MUST be object-safe ŌĆö no generic methods; use `&dyn SensorAuth`
  in the trait signature, not `impl SensorAuth`
- Cross-sensor auth composition MUST be prevented at compile time via the sealed trait ŌĆö
  it is not sufficient to rely on runtime checks
- The fan-out concurrency limit (10 per query) is distinct from the global HTTP semaphore
  (200 total) ŌĆö both MUST be enforced simultaneously; a single task holds one fan-out
  permit AND one HTTP semaphore permit
- `retry_with_backoff` MUST NOT retry on 4xx errors (except 429) ŌĆö retrying on 400/404
  is wasteful and masks bugs

## Dev Notes

- The fan-out concurrency semaphore of 10 is intentionally conservative for the initial
  implementation. It can be made configurable via `[query] max_concurrent_sensor_fetches`
  in `prism.toml` in a follow-on story.
- `join_all` does not short-circuit on failure ŌĆö all tasks run to completion regardless
  of individual failures. This is the correct behavior for partial failure collection.
- The 30-second timeout on HTTP semaphore acquisition should produce a structured error
  log with the current permit count; this helps operators diagnose connection pool
  exhaustion.
- `SensorAdapter` implementations (CrowdStrikeAdapter, etc.) are skeletal stubs in this
  story ŌĆö S-2.07 provides the full per-sensor auth and pagination implementations.
