# S-5.09 — prism-mcp: External Log Forwarding Subsystem

## Narrative

As a Prism MSSP operator, I want diagnostic log entries forwarded to external SIEM and
observability destinations — Datadog, Splunk HEC, Elasticsearch, OpenTelemetry,
syslog, and custom webhooks — with a WASM plugin extension point for proprietary
destinations, so that Prism's operational telemetry integrates into existing MSSP
monitoring workflows without requiring log aggregation agents.

## Objective

Implement the external log forwarding subsystem per `observability.md` §External Log
Forwarding: 6 built-in forwarder types (`datadog`, `splunk_hec`, `elasticsearch`,
`otlp`, `syslog`, `webhook`), a 7th `plugin` type via `.prx` WIT interface, batched
delivery with an in-memory queue cap, no-recursive-forwarding guarantee, per-destination
`min_level` filtering, credential references via AD-017 opaque model, and best-effort
delivery (in-memory queue only, no persistent queue in v1).

**BC note:** No new BCs are created for this story. The forwarder behaviors are
implementation details of the observability subsystem, covered by integration tests
rather than formal behavioral contracts. This approach follows the same precedent as
the sensor adapter implementation stories (S-2.06, S-2.07) which implement
integration-level behavior without dedicated per-adapter BCs.

## Tasks

1. Define forwarder config schema in `crates/prism-mcp/src/forwarding/mod.rs`:
   - `ForwarderConfig { name: String, destination_type: DestinationType, min_level: Level,
     batch_size: u32, flush_interval_secs: u64, credentials: HashMap<String, CredentialRef>,
     endpoint: Option<String>, plugin_id: Option<String> }`
   - `DestinationType`: `Datadog | SplunkHec | Elasticsearch | Otlp | Syslog | Webhook | Plugin`
   - `min_level`: one of `trace`, `debug`, `info`, `warn`, `error` (default `info`)
   - `batch_size`: max entries per delivery call (default 100, max 1000)
   - `flush_interval_secs`: max time to hold before flushing even if batch not full (default 5s)
   - Credentials: `CredentialRef { source: "env" | "store", key: String }` — same pattern
     as action delivery (S-4.08). Inline credential values are rejected with `E-FWD-001`.
   - `ForwarderRegistry`: `Vec<Arc<dyn LogForwarder>>` built from config at startup

2. Implement shared queue and dispatch in `crates/prism-mcp/src/forwarding/queue.rs`:
   - `ForwarderQueue`: in-memory `VecDeque<LogEntry>` with configurable capacity
     (default 10,000 entries per forwarder destination)
   - Queue is per-destination (each forwarder has its own queue — slow destinations do
     not starve fast ones)
   - Overflow policy: drop OLDEST entry, increment `queue_overflow_count` counter,
     emit exactly ONE `WARN` log per drop batch (not one per dropped entry — this
     prevents recursive log generation)
   - Anti-recursion: overflow warning is emitted to the INTERNAL `tracing` sink only
     (NOT forwarded to external destinations). See Task 8.
   - Batching: flush when `batch_size` entries accumulated OR `flush_interval_secs`
     elapsed (whichever comes first). Use `tokio::time::interval` for the time trigger.

3. Implement `datadog` forwarder in `crates/prism-mcp/src/forwarding/destinations/datadog.rs`:
   - `DatadogForwarder::deliver(batch: Vec<LogEntry>) -> Result<(), ForwardError>`:
     - Resolve API key from `CredentialRef` at deliver time (not stored in struct)
     - POST to `https://http-intake.logs.datadoghq.com/api/v2/logs` (EU endpoint
       configurable via `endpoint` field)
     - Body: JSON array of Datadog log objects (ddsource, ddtags, message, service fields)
     - HTTP 202 → success; HTTP 4xx → non-retryable `ForwardError::Rejected`; 5xx →
       retryable `ForwardError::Transient` (caller handles retry)
     - DD-API-KEY header sourced from credential ref

4. Implement `splunk_hec` forwarder in `crates/prism-mcp/src/forwarding/destinations/splunk_hec.rs`:
   - POST to `{endpoint}/services/collector/event` (Splunk HEC endpoint)
   - Body: Splunk HEC JSON format (`{"time": ..., "event": {...}, "source": "prism", "sourcetype": "_json"}`)
   - `Authorization: Splunk {token}` header (token from credential ref)
   - Support `ssl_verify: bool` config (some Splunk deployments use self-signed certs)

5. Implement `elasticsearch` forwarder in `crates/prism-mcp/src/forwarding/destinations/elasticsearch.rs`:
   - Bulk index via `POST {endpoint}/_bulk` with `{"index": {"_index": "prism-logs"}}` header line
   - Authentication: Basic Auth (username + password from credential refs) or API key
   - Index name configurable via `index_name` field (default `"prism-logs"`)

6. Implement `otlp` forwarder in `crates/prism-mcp/src/forwarding/destinations/otlp.rs`:
   - Export log entries via OTLP/HTTP (OpenTelemetry Protocol over HTTP, protobuf or JSON)
   - Endpoint: `{endpoint}/v1/logs`
   - Use `opentelemetry-proto` log schema for serialization
   - Resource attributes: `service.name = "prism"`, `service.version = "{prism_version}"`
   - Optional: `Authorization: Bearer {token}` header (credential ref)

7. Implement `syslog` forwarder in `crates/prism-mcp/src/forwarding/destinations/syslog.rs`:
   - Formats: `rfc5424` (default), `rfc3164`, `json` (newline-delimited JSON)
   - Transport: `udp`, `tcp`, `tcp_tls` (per config)
   - Facility: `LOCAL0` (default) — configurable
   - Priority mapping: prism log level → syslog severity (ERROR=3, WARN=4, INFO=6, DEBUG=7)
   - For `tcp_tls`: verify server certificate unless `ssl_verify: false`

8. Implement `webhook` forwarder in `crates/prism-mcp/src/forwarding/destinations/webhook.rs`:
   - POST batch as JSON array to the configured URL
   - Optional `Authorization` header (credential ref)
   - Optional custom headers (static values only — no template variables to avoid injection)
   - HTTP 2xx → success; non-2xx → forward error

9. Implement `plugin` forwarder in `crates/prism-mcp/src/forwarding/destinations/plugin.rs`:
   - Delegates to `PluginRuntime::fire_alert` (S-1.15) using a WASM plugin identified
     by `plugin_id` from config
   - The WASM plugin implements `prism-log-forwarder-plugin.wit` (a new WIT interface
     alongside the existing sensor/infusion/action WIT files)
   - WIT interface: `forward-batch(entries: list<log-entry>) -> result<(), forward-error>`
   - `PluginError::NotLoaded` → `ForwardError::Rejected` (do not retry — plugin not available)
   - `PluginError::Trapped` → `ForwardError::Transient` (retry after backoff)

10. Implement no-recursive-forwarding guarantee in `crates/prism-mcp/src/forwarding/anti_recursion.rs`:
    - `ForwarderQueue` writes are gated by a thread-local `IS_FORWARDING` flag
    - When the forwarder task calls the delivery function: set `IS_FORWARDING = true`
    - Any log entry emitted while `IS_FORWARDING = true`: written ONLY to the internal
      `tracing` sink (local file), NOT enqueued to any forwarder queue
    - `IS_FORWARDING = false` after delivery completes (success or failure)
    - This breaks the recursive loop: forwarder error → log entry → forwarder → loop

11. Implement `prism logs --destination <name>` flag in `crates/prism-mcp/src/cli/logs.rs`
    (extends S-5.08's `logs.rs`):
    - Show entries that WOULD be forwarded to the named destination (apply the destination's
      `min_level` filter, show matching entries)
    - Does NOT trigger actual delivery — informational only

12. Write tests in `crates/prism-mcp/tests/forwarding_tests.rs`:
    - Test 2 destinations: emit 10 log entries → verify both receive all 10 (above min_level)
    - Test `min_level`: destination with `min_level = warn` does not receive INFO entries
    - Test destination failure: one destination returns 500 → does not affect the other destination
    - Test queue overflow: fill beyond capacity → verify oldest entries dropped, one WARN emitted
    - Test anti-recursion: delivery failure generates log, but that log is NOT re-enqueued
    - Test inline credential rejection (E-FWD-001)
    - Test `plugin` forwarder: mock PluginRuntime, verify batch is forwarded

## Acceptance Criteria

**AC-1:** Given 2 configured destinations with both above their `min_level`, When 5 log
entries are emitted at `INFO` level, Then each destination receives all 5 entries within
`flush_interval_secs` seconds.

**AC-2:** Given a destination with `min_level = "warn"`, When `INFO` and `WARN` log entries
are emitted, Then only the `WARN` entries are forwarded to that destination.

**AC-3:** Given destination A is slow (500ms per response) and destination B is fast,
When 100 entries are emitted rapidly, Then destination B continues to receive entries on
its own flush cadence — destination A's slowness does not block destination B.

**AC-4:** Given a queue at capacity, When the next entry arrives, Then the oldest entry
is dropped, a single `WARN` is emitted to the internal log (not forwarded externally), and
the counter `queue_overflow_count` is incremented.

**AC-5:** Given a delivery failure causes a `WARN` log entry, When that log entry is
processed by the forwarding subsystem, Then it is NOT enqueued to any external destination
(anti-recursion guarantee holds).

**AC-6:** Given a forwarder config with `api_key = "raw-value"` (inline credential),
When the config is loaded, Then `E-FWD-001` is returned and the forwarder is NOT registered.

**AC-7:** Given `prism logs --destination datadog`, When run, Then only entries matching
the datadog destination's `min_level` are displayed, with no actual forwarding attempted.

## Architecture Compliance Rules

- Each destination MUST have its own independent in-memory queue. Queues MUST NOT be
  shared between destinations. A full queue for one destination drops that destination's
  entries without affecting other destinations.
- Credential resolution MUST occur at delivery time, not at forwarder construction time.
  Credentials rotate; resolving at construction would serve stale values after rotation.
- The anti-recursion guard MUST use a thread-local flag (or `tokio::task_local!` equivalent),
  NOT a global atomic — the forwarding context is per-task, and a global would incorrectly
  suppress logging from unrelated tasks.
- `otlp` forwarder MUST NOT pull in the full `opentelemetry` SDK — use only
  `opentelemetry-proto` (protobuf schemas) and `reqwest` for HTTP delivery. The full
  OTel SDK introduces unwanted runtime overhead and global state.
- WASM plugin forwarder calls MUST be made via `tokio::task::spawn` (isolate CPU time
  limits from the forwarding dispatch loop), consistent with the S-4.08 constraint on
  plugin actions.
- DO NOT add the forwarding subsystem's own log entries to the forwarding queue during
  startup initialization. Log at startup before the queue is active.

## Previous Story Intelligence

S-5.08 (Diagnostics) established the rolling log writer and the `LogEntry` type. The
forwarding subsystem taps into the same `tracing_subscriber` layer pipeline — add a
`ForwarderLayer` that intercepts log events AFTER they are written to the local file.
The local file write happens first (guaranteed persistence), then forwarding (best-effort).

S-1.15 (WASM Plugin Runtime) provides `PluginRuntime`. The `plugin` forwarder requires
`PluginRuntime` to be available. Confirm the WIT interface for log forwarder plugins
(`prism-log-forwarder-plugin.wit`) is aligned with the existing WIT file structure before
implementing the plugin forwarder.

S-4.08 (Action Delivery) established the at-least-once delivery pattern with exponential
backoff. The forwarding subsystem uses BEST-EFFORT (no persistent retry) for v1 — this
is intentional and documented. If at-least-once forwarding is needed in v2, the audit
forwarding story (S-5.10) provides the persistent pattern to copy.

## Notes

- Best-effort delivery means: if Prism restarts while entries are in the in-memory queue,
  those entries are lost. This is acceptable for diagnostic logs (not audit-grade). The
  audit trail uses a separate persistent mechanism (S-5.10).
- For the Elasticsearch destination, the `_bulk` endpoint is preferred over individual
  `_doc` PUT calls — bulk indexing is 10-50x more efficient for high log volumes.
- syslog UDP does not provide delivery confirmation. Do not implement retry for UDP
  syslog — just fire and forget. TCP syslog can detect connection failure and retry at
  the queue level.
- The `prism-log-forwarder-plugin.wit` WIT interface is new and must be added to
  `crates/prism-spec-engine/wit/`. The `wit-bindgen` build step (from S-1.15) will
  need to be updated to include this new interface file.
