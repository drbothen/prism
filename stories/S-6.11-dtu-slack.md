# S-6.11 — prism-dtu-slack: DTU for Slack Webhook API — L2 (stateful)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Slack Incoming
Webhook API at fidelity level L2 (stateful), so that integration tests for Slack action
delivery can validate Block Kit payload shape, `message_ts` deduplication tracking,
and 429 rate-limit retry behavior without hitting the live Slack API.

## Objective

Implement `prism-dtu-slack` — an L2 stateful behavioral clone of the Slack webhook
endpoint. The clone validates received Block Kit payload structure, returns a `message_ts`
in successful responses, captures all received payloads for test assertion, and can inject
rate-limit (429) and error (400, 500) responses. No SMTP or syslog testing is included
here — those use `mailpit` and `SyslogReceiver` from `prism-dtu-common` respectively.
See `dtu-assessment.md §3.5.1` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-slack/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `SlackClone` struct implementing `BehavioralClone`:
   - `start()`: binds `127.0.0.1:0`, spawns axum router
   - `reset()`: clears payload capture store and rate-limit counter
   - `received_payloads(&self) -> Vec<serde_json::Value>`: test API for assertions

3. Create `src/state.rs` — `SlackState`:
   - `received_payloads: Mutex<Vec<Value>>` — captures each Block Kit payload received
   - `request_count: AtomicU32` — for rate-limit threshold tracking

4. Create `src/routes/webhook.rs`:
   - `POST /services/{token}` (path param `token` is ignored — any path segment accepted):
     - Validates payload is a JSON object with either `blocks` or `text` key present
     - If missing both: return 400 with body `"invalid_payload"` (matches real Slack behavior)
     - If valid: capture payload in `received_payloads`; return 200 JSON:
       `{"ok": true, "message_ts": "1234567890.123456"}` — `message_ts` is a fake stable
       string (not random) so integration tests can assert on it
     - Unknown top-level fields: rejected per `dtu-assessment.md §3.5.1`
       (return 400 `"unknown_field"` for non-spec top-level keys)

5. Rate-limit behavior:
   - Track `request_count` with `AtomicU32`
   - When `FailureMode::RateLimit { after_n_requests }` is configured:
     - After threshold: return 429 with body `"ratelimited"` and header `Retry-After: 30`
   - Reset counter on `POST /dtu/reset`

6. Create `POST /dtu/configure` and `POST /dtu/reset`:
   - `configure`: accepts `{"rate_limit_after": N}` or `{"fail_with": 500}`
   - `reset`: clears `received_payloads` and `request_count`; returns 200

7. Create `GET /dtu/received-payloads` test API:
   - Returns `{"payloads": [...]}` — list of all Block Kit payloads received since last reset
   - Used by integration tests to assert payload shape

8. Create fixture files:
   - `fixtures/valid-block-kit.json`: a sample valid Block Kit payload with `blocks` array
     containing a `section` block with `text` field — used as the reference shape for
     fidelity validation
   - `fixtures/block-kit-schema.json`: minimal Block Kit schema: allowed top-level fields
     (`blocks`, `text`, `username`, `icon_emoji`, `icon_url`, `attachments`) — used by
     the DTU's payload validator

9. Create `tests/fidelity.rs`:
   - `#[tokio::test] fn slack_dtu_fidelity()`: starts `SlackClone`, POSTs the valid
     Block Kit fixture, asserts HTTP 200 with `ok: true` and `message_ts` present

## Acceptance Criteria

**AC-1:** Given `POST /services/{token}` with a valid Block Kit payload containing
`blocks`, Then the response is HTTP 200 `{"ok": true, "message_ts": "1234567890.123456"}`.

**AC-2:** Given `POST /services/{token}` with a payload missing both `blocks` and `text`,
Then the response is HTTP 400 with body `"invalid_payload"` — exercising Prism's
`E-ACTION-004` error path.

**AC-3:** Given `POST /services/{token}` with a payload containing an unknown top-level
field `"unknown_key": "value"`, Then the response is HTTP 400 with body `"unknown_field"`.

**AC-4:** Given `FailureMode::RateLimit { after_n_requests: 3, retry_after_secs: 30 }`,
When the 4th POST arrives, Then the response is HTTP 429 with `Retry-After: 30` and body
`"ratelimited"` — exercising Prism's action retry logic.

**AC-5:** Given 3 successful webhook deliveries, When `GET /dtu/received-payloads` is
called, Then the response contains all 3 captured payloads in order.

**AC-6:** Given `reset()` is called, Then `received_payloads` is empty and the rate-limit
counter is reset to 0.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Empty JSON object `{}` POSTed as payload | HTTP 400 `"invalid_payload"` (missing both `blocks` and `text`) |
| EC-002 | 500 failure mode configured | Next POST returns HTTP 500 — tests retry exhaustion path |
| EC-003 | Very large Block Kit payload (>100KB) | DTU accepts it; no size limit enforced (Prism side enforces size) |
| EC-004 | `message_ts` used twice in deduplication | Integration test can assert that two deliveries produce the same `message_ts` value (stable fake value, not random) |
