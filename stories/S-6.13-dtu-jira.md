# S-6.13 — prism-dtu-jira: DTU for Jira REST API v3 — L3 (behavioral)

## Narrative

As a Prism integration-test author, I want a behavioral clone of the Jira Cloud REST
API v3 at fidelity level L3 (behavioral), so that integration tests for Jira action
delivery can validate issue creation, comment addition, status machine transitions
(Open → In Progress → Done), field validation, and Basic auth without hitting live Jira.

## Objective

Implement `prism-dtu-jira` — an L3 behavioral clone of the Jira Cloud REST API v3.
The clone implements all 5 in-scope endpoints (create issue, add comment, list
transitions, execute transition, get issue), maintains a stateful issue registry with
status machine enforcement (invalid transitions return 400), validates required fields on
creation, and supports Basic auth (`Authorization: Basic {base64}`) as primary.
See `dtu-assessment.md §3.5.3` for full scope matrix.

## Tasks

1. Create `crates/prism-dtu-jira/Cargo.toml`:
   - Dependencies: `prism-dtu-common` (path dep), `axum` (0.7.x), `tokio` (1.x full),
     `tower` (0.4.x), `serde` (1.x derive), `serde_json` (1.x), `http` (1.x),
     `tracing` (0.1.x), `base64` (0.21.x for auth decoding)
   - `[features] dtu = []`

2. Create `src/lib.rs` — `JiraClone` struct implementing `BehavioralClone`:
   - `start()`: binds ephemeral port, spawns axum router
   - `reset()`: calls `state.reset()`
   - `issues(&self) -> Vec<IssueRecord>`: test API — returns all current issues

3. Create `src/state.rs` — `JiraState`:
   - `issue_registry: Mutex<HashMap<String, IssueRecord>>` — `issueKey → IssueRecord`
   - `next_issue_num: AtomicU32` — monotonically increasing issue number (starts at 1000)
   - `IssueRecord { id: String, key: String, summary: String, issue_type: String,
     project_key: String, status: IssueStatus, comment_count: u32, fields: Value }`
   - `IssueStatus` enum: `Open`, `InProgress`, `Done`
   - Status machine (valid transitions only):
     - `Open → InProgress` (transition id `"11"`)
     - `InProgress → Done` (transition id `"21"`)
     - `Open → Done` (transition id `"31"`) — allowed by Jira but not common
     - All others: 400 invalid transition
   - `reset()` clears registry and resets `next_issue_num` to 1000

4. Basic auth middleware:
   - All routes except `POST /dtu/*` must validate `Authorization: Basic {base64}` header
   - Any valid Base64-encoded `user:token` string is accepted (DTU does not validate credentials)
   - If `Authorization` header absent or not `Basic`: return 401
     `{"errorMessages": ["Basic authentication required"], "errors": {}}`

5. Create `src/routes/issues.rs`:
   - `POST /rest/api/3/issue`: create issue
     - Required fields in body: `fields.project.key` (string), `fields.issuetype.name`
       (one of `"Task"`, `"Bug"`, `"Story"`, `"Epic"`, `"Incident"`), `fields.summary` (string)
     - Missing `project.key`: 400 `{"errorMessages": [], "errors": {"project": "required"}}`
     - Unknown `issuetype.name`: 400 `{"errorMessages": [], "errors": {"issuetype": "unknown"}}`
     - On success: assign `issueKey = "{project_key}-{next_issue_num}"`, increment counter
     - Store in registry with `status: Open, comment_count: 0`
     - Response: 201 `{"id": "10001", "key": "ACME-SEC-1000", "self": "http://..."}`
   - `GET /rest/api/3/issue/{issueKey}`: get issue
     - 404 if not in registry: `{"errorMessages": ["Issue does not exist"], "errors": {}}`
     - Response: full issue object with `id`, `key`, `fields.status.name`, `fields.summary`,
       `fields.issuetype.name`, `fields.comment.total` (= comment_count)

6. Create `src/routes/comments.rs`:
   - `POST /rest/api/3/issue/{issueKey}/comment`: add comment
     - 404 if issue not found
     - Body: `{"body": {...}}` — body content accepted as-is (not validated)
     - Increments `comment_count` on the issue record
     - Returns 201 `{"id": "10001", "self": "http://...", "created": "2026-04-16T..."}`

7. Create `src/routes/transitions.rs`:
   - `GET /rest/api/3/issue/{issueKey}/transitions`: list available transitions
     - Returns transitions valid for current issue status:
       - If `Open`: `[{"id": "11", "name": "Start Progress"}, {"id": "31", "name": "Close"}]`
       - If `InProgress`: `[{"id": "21", "name": "Done"}]`
       - If `Done`: `[]` (no transitions available)
     - 404 if issue not found
   - `POST /rest/api/3/issue/{issueKey}/transitions`: execute transition
     - Body: `{"transition": {"id": "11"}}`
     - Validates `transition.id` is in the available set for current status
     - Invalid transition id: 400 `{"errorMessages": ["Invalid transition id"], "errors": {}}`
     - On success: update issue status; return 204 No Content

8. Create fixture files:
   - `fixtures/issue-create-response.json`: example 201 response for reference
   - `fixtures/issue-get-response.json`: example GET issue response with all fields
   - `fixtures/transitions-open.json`: transitions available from Open status
   - `fixtures/transitions-inprogress.json`: transitions available from InProgress status

9. Create `GET /dtu/issues` test API endpoint:
   - Returns `{"issues": [{key, status, summary, comment_count}, ...]}` for assertions

10. Create `tests/fidelity.rs`:
    - Tests full create → add comment → transition to InProgress → transition to Done lifecycle
    - Asserts invalid transition (e.g., Done → InProgress) returns 400
    - Run as part of `just dtu-validate`

## Acceptance Criteria

**AC-1:** Given `POST /rest/api/3/issue` with valid `project.key`, `issuetype.name: "Task"`,
and `summary`, Then the response is HTTP 201 with `key` matching `"{project_key}-NNNN"` AND
`GET /dtu/issues` shows the issue with `status: "Open"`.

**AC-2:** Given `POST /rest/api/3/issue/ACME-SEC-1000/comment` with a body object, Then
the response is HTTP 201 AND `GET /rest/api/3/issue/ACME-SEC-1000` shows
`fields.comment.total: 1`.

**AC-3:** Given `GET /rest/api/3/issue/ACME-SEC-1000/transitions` when status is `Open`,
Then the response includes transition `"11"` (Start Progress) and `"31"` (Close).

**AC-4:** Given `POST /rest/api/3/issue/ACME-SEC-1000/transitions` with `{"transition": {"id": "11"}}`,
Then the response is HTTP 204 AND `GET /rest/api/3/issue/ACME-SEC-1000` shows
`fields.status.name: "In Progress"`.

**AC-5:** Given `POST /rest/api/3/issue/ACME-SEC-1000/transitions` with an invalid
transition id (e.g., attempting to go from `Done` back to `Open`), Then the response is
HTTP 400 `{"errorMessages": ["Invalid transition id"], "errors": {}}`.

**AC-6:** Given `POST /rest/api/3/issue` with missing `fields.project.key`, Then the
response is HTTP 400 `{"errorMessages": [], "errors": {"project": "required"}}`.

**AC-7:** Given `POST /rest/api/3/issue` with unknown `issuetype.name: "Feature"`, Then
the response is HTTP 400 `{"errorMessages": [], "errors": {"issuetype": "unknown"}}`.

**AC-8:** Given a request without `Authorization: Basic ...` header, Then the response is
HTTP 401 `{"errorMessages": ["Basic authentication required"], "errors": {}}`.

**AC-9:** Given `GET /rest/api/3/issue/UNKNOWN-999`, Then the response is HTTP 404
`{"errorMessages": ["Issue does not exist"], "errors": {}}`.

**AC-10:** Given `FailureMode::RateLimit` configured, When threshold exceeded, Then
HTTP 429 — maps to Prism's action retry path.

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | `POST /rest/api/3/issue` with both required fields but extra unknown field in `fields` | Unknown fields are ignored (Jira API is permissive about extra fields) |
| EC-002 | `GET /rest/api/3/issue/{issueKey}/transitions` when status is `Done` | Returns empty transitions list `{"transitions": []}` |
| EC-003 | Two `POST /rest/api/3/issue` calls with same project | Second call gets incremented `issueKey` (ACME-SEC-1001); no conflict |
| EC-004 | `POST /rest/api/3/issue/{issueKey}/comment` on a `Done` issue | Allowed (Jira permits comments on closed issues); returns 201 |
| EC-005 | `reset()` during active integration test | All issues cleared; subsequent GET returns 404 |

## Risk Mitigations

### R-DTU-008: Jira Cloud API tier restrictions

**Risk:** Jira Cloud restricts some REST API v3 endpoints to paid plans. The free tier
may not support OAuth 2.0 (three-legged) or certain issue transition APIs. If
`prism-dtu-jira` is designed against a paid-tier Jira and the MSSP uses a free-tier
Jira, fixture schemas may diverge.

**Mitigation:** Design `prism-dtu-jira` against the published Jira Cloud REST API v3
documentation (not captured from a live instance). Use Basic auth (email + API token)
as the primary auth model — it is available on all Jira Cloud plans. OAuth path is an
optional fixture; flag in the DTU README that OAuth 2.0 requires a paid Jira plan.
See `dtu-assessment.md §10 R-DTU-008` for the full risk statement.
