# S-3.2.08 — prism-query: scope CrowdStrike pagination session IDs per OrgId (D-048)

## Narrative

As a Prism query engine, I want CrowdStrike pagination session IDs to be generated with
the requesting `OrgId` embedded in the UUID v7 time field so that a session ID produced
in Org A's context can never accidentally match Org B's session data in the shared
`prism-dtu-crowdstrike` `session_registry`, satisfying the D-048 resolution to
BC-3.2.003.

## Objective

Implement D-048 at the query-engine layer: when `prism-query` generates a CrowdStrike
pagination session ID (a UUID used as the `X-DTU-Session-Id` header value), embed the
calling `OrgId` in the UUID v7 node/random bytes so that org-temporal uniqueness is
structural, not probabilistic. The `prism-dtu-crowdstrike` `session_registry` (keyed by
bare `String` session ID) requires no change — the enforcement happens at generation time
in the query engine.

Per BC-3.2.003 precondition 4 and ADR-008 §2.1 D-048: "CrowdStrike session IDs are
scoped per `org_id` at the query-engine layer. The query engine generates session IDs
with `OrgId` embedded (org-temporal uniqueness via UUID v7 time field), so a session ID
from Org A's context cannot match Org B's session data."

## Tasks

1. **Locate session ID generation in `prism-query`**:
   Search `crates/prism-query/src/` for UUID generation or `X-DTU-Session-Id` header
   construction. The session ID is a UUID that the query engine passes to
   `prism-dtu-crowdstrike` to maintain two-step pagination state.

2. **Design org-scoped UUID v7 generation**:
   UUID v7 is time-ordered with 74 bits of randomness. Embed the `OrgId` in the random
   bits using a deterministic XOR with the first 16 bytes of the OrgId UUID:
   ```rust
   fn generate_crowdstrike_session_id(org_id: OrgId) -> Uuid {
       let base = Uuid::now_v7();
       let org_bytes: [u8; 16] = org_id.as_uuid().into_bytes();
       let mut session_bytes: [u8; 16] = *base.as_bytes();
       // XOR the random portion (bytes 8-15) with the OrgId bytes (8-15)
       // Preserves UUID v7 version/variant bits (bytes 6-7 are version/variant)
       for i in 8..16 {
           session_bytes[i] ^= org_bytes[i];
       }
       Uuid::from_bytes(session_bytes)
   }
   ```
   **Note:** This is a determinism-free approach — each call still generates a new
   `Uuid::now_v7()` base, so collisions remain negligible even without the XOR. The XOR
   serves as a structural org-namespace marker. If the UUID v7 spec or workspace Uuid
   crate version constrains byte layout, adjust the XOR range to avoid version/variant
   bits. Document the byte layout in a code comment.

3. **Thread `OrgId` to session ID generation**:
   - Locate the call site in `prism-query` where the CrowdStrike session ID is created
     or the `X-DTU-Session-Id` header is set.
   - Update that call site to accept `org_id: OrgId` and invoke
     `generate_crowdstrike_session_id(org_id)`.
   - If session IDs are managed in a `CrowdStrikeQuerySession` struct or similar,
     add `org_id: OrgId` to that struct and wire it through.

4. **Add a `#[doc]` comment in `prism-dtu-crowdstrike/src/state.rs`** on the
   `session_registry` field:
   ```rust
   /// `session_registry` is keyed by session ID string (NOT org-scoped).
   /// Org isolation is enforced at generation time by the query engine (D-048):
   /// session IDs embed the calling OrgId in the random bytes, making cross-org
   /// collision structurally impossible. See ADR-008 §2.1 / D-048.
   pub session_registry: Mutex<LruCache<String, SessionData>>,
   ```

5. **Write cross-org isolation test** at
   `crates/prism-query/tests/crowdstrike_session_isolation.rs` (create if absent):
   - `test_crowdstrike_session_ids_differ_per_org` — call
     `generate_crowdstrike_session_id(org_id_A)` and
     `generate_crowdstrike_session_id(org_id_B)` with distinct OrgIds at the same
     millisecond (mock `Uuid::now_v7()` if needed, or generate many and compare).
     Assert that the resulting session IDs have different bytes in positions 8-15,
     demonstrating org-namespace separation.
   - `test_crowdstrike_session_id_org_a_not_found_in_org_b_registry` — simulate the
     full path: generate session_id_A for org_A; store `(session_id_A, SessionData)`
     in a bare `LruCache<String, _>` (mirroring the DTU's `session_registry`); look up
     `session_id_B` (generated for org_B at same time); assert not found.
   - Validate VP-084 property: token registered under org_id_A is invalid in
     org_id_B context.

6. **Gate**: `cargo test -p prism-query` green.

## Acceptance Criteria

**AC-001** — Session IDs are org-namespaced at generation (traces to BC-3.2.003
precondition 4 + D-048):
Given org_A and org_B query CrowdStrike simultaneously via `prism-query`,
When each query's CrowdStrike session ID is generated,
Then `session_id_A` and `session_id_B` differ in bytes 8-15 (the org-embedded random
bytes), demonstrating structural org-namespace separation.

**AC-002** — Cross-org session ID lookup fails closed (traces to BC-3.2.003
postcondition 2 + D-048):
Given `session_id_A` generated for org_A is stored in the bare `session_registry` keyed
by string,
When a lookup is performed for `session_id_B` (generated for org_B at the same instant),
Then the lookup returns `None` — org_B cannot accidentally resolve org_A's session data.

**AC-003** — `OrgId` threads through session generation (traces to BC-3.2.003
invariant 4):
Given `prism-query`'s CrowdStrike query path accepts a query plan with `org_id: OrgId`,
When a session ID is generated for the CrowdStrike pagination header,
Then the generating function takes `org_id` as input and embeds it in the session UUID —
there is no code path that generates a session ID without an OrgId.

**AC-004** — `session_registry` in `prism-dtu-crowdstrike` has D-048 doc comment (traces
to BC-3.2.003 invariant 4):
Given a code reader inspects `crates/prism-dtu-crowdstrike/src/state.rs`,
When they see the `session_registry` field,
Then a `///` doc comment explains the D-048 rationale (org-scoped at query-engine layer,
not clone state layer) and references ADR-008 §2.1.

**AC-005** — No regression on existing CrowdStrike session tests (traces to BC-3.2.003
precondition 4 + ADR-008 §2.1):
Given all existing tests in `prism-dtu-crowdstrike` pass before this story,
When the D-048 doc comment is added and session ID generation is updated in `prism-query`,
Then `cargo test -p prism-dtu-crowdstrike` continues to pass with no changes to clone
source code (only the comment and no logic changes in the clone itself).

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Two orgs generate session IDs at the exact same nanosecond | UUID v7 time field is the same; org XOR in bytes 8-15 still differs → different session IDs |
| EC-002 | OrgId is the nil UUID (all zeros) — test utility only | XOR with all-zeros is identity; the base UUID is unchanged; this is acceptable for test orgs only |
| EC-003 | UUID v7 timestamp overflows (far future) | Standard UUID v7 behavior; no org-scoping impact |
| EC-004 | `session_registry` LRU evicts org_A's session | Normal LRU behavior; org_A's next query generates a new session ID; no cross-org risk |
| EC-005 | CrowdStrike pagination is not in use (no session header) | No session ID generated; this code path is not invoked; no impact |
