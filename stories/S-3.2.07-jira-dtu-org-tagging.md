# S-3.2.07 — prism-dtu-jira: Shared-mode OrgId ingress tagging

## Narrative

As a Prism MSSP operator, I want the Jira DTU clone to embed the `OrgId` UUID string
in Jira issue fields (e.g., a label `"org_id:<uuid>"` or a custom field) so that tickets
created for different client organizations carry opaque forensic attribution that is
visible to system-level users but does not expose org slug to Jira project users.

## Objective

Apply the shared-mode OrgId tagging pattern (same as S-3.2.05, S-3.2.06) to
`prism-dtu-jira`. The `issue_registry` state store remains keyed by Jira issue key
(`"PROJ-NNN"` — MSSP-scoped) per ADR-008 §1.2. Add `org_id` to `IssueRecord` metadata.
Add `DtuMode::Shared` metadata. Validate mode immutability per BC-3.2.005.

## Tasks

1. **Add `org_id` to `IssueRecord`** (`src/state.rs`):
   ```rust
   pub struct IssueRecord {
       pub issue_key: String,       // e.g., "PROJ-1000" — MSSP-scoped, NOT org-scoped
       pub org_id: String,          // OrgId UUID form (AI-opaque)
       pub payload: Value,
   }
   ```

2. **Update `capture_issue`** to accept `org_id: OrgId`:
   ```rust
   pub fn capture_issue(&self, org_id: OrgId, issue_key: String, payload: Value) {
       let record = IssueRecord {
           issue_key: issue_key.clone(),
           org_id: org_id.to_string(),
           payload,
       };
       let mut registry = self.issue_registry.lock().expect("poisoned");
       registry.insert(issue_key, record);
   }
   ```

3. **Verify issue key is NOT org-scoped**:
   Jira issue keys (`"PROJ-NNN"`) are MSSP-scoped project keys. The `org_id` appears
   only in the `IssueRecord.org_id` field. Add a unit test asserting the issue_key
   contains no OrgId UUID.

4. **Thread OrgId through route handlers** — same approach as S-3.2.05/06.

5. **Add DtuMode::Shared metadata** — pick up from `prism-dtu-common` (added in S-3.2.05).

6. **Add `tests/org_tagging.rs`**:
   - `test_org_id_in_issue_record` — create issue for org_A; assert `IssueRecord.org_id == uuid-A`.
   - `test_issue_key_not_org_scoped` — assert issue_key contains no OrgId UUID.
   - `test_concurrent_issues_distinguished` — org_A and org_B create issues; each record contains its org_id.
   - `test_jira_user_sees_opaque_uuid` — verify `org_id` field is UUID form (no slug).
   - `test_mode_immutable_after_startup` — DtuMode::Shared cannot be changed at runtime.

7. **Gate**: `cargo test -p prism-dtu-jira` green.

## Acceptance Criteria

**AC-001** — OrgId in Jira issue record (traces to BC-3.2.004 postcondition 1):
Given org_A creates a Jira ticket via the shared Jira DTU,
When the captured `IssueRecord` is inspected,
Then `IssueRecord.org_id == "<uuid-A>"` (UUID form, not slug).

**AC-002** — OrgId absent from issue key and HTTP routing (traces to BC-3.2.004 postcondition 2):
Given any Jira issue creation payload,
When the `issue_key`, URL path, and headers are inspected,
Then none contain the OrgId UUID or org slug.

**AC-003** — Concurrent issues from different orgs are distinguished (traces to BC-3.2.004 postcondition 4):
Given org_A and org_B both create issues concurrently,
When both `IssueRecord`s are inspected,
Then each contains its sender's OrgId UUID independently.

**AC-004** — Mode metadata absent from query results (traces to BC-3.2.004 postcondition 5):
Given an analyst queries issue data from this shared DTU,
When result rows are inspected,
Then no `"mode"`, `"shared"`, or `"dtu_mode"` fields appear.

**AC-005** — OrgId UUID is opaque (traces to BC-3.2.004 invariant 1):
Given a Jira project user views an issue created by the Jira DTU,
When the `org_id` label or field value is inspected,
Then the value is a UUID string (e.g., `"org_id:00000000-0000-7000-8000-000000000001"`) — not a human-readable slug.

**AC-006** — DtuMode::Shared set at startup and immutable (traces to BC-3.2.005 postconditions 1 and 4):
Given config TOML has `mode = "shared"` for the Jira DTU,
When the process starts and any in-process API is called,
Then `DtuMode::Shared` remains the mode for the process lifetime.

## Architecture Compliance Rules

- `OrgId` UUID string MUST appear in `IssueRecord.org_id` field — NOT in `issue_key`, URL path, or headers.
- `issue_registry` (`HashMap<String, IssueRecord>`) is NOT re-keyed by OrgId — remains bare-String keyed by issue_key per ADR-008 §1.2.
- `org_id` MUST be stored as UUID string (not slug) for AI-opacity.
- **Forbidden dependencies:** `prism-dtu-jira` MUST NOT depend on `prism-sensors`, `prism-query`, `prism-operations`, `prism-mcp`, or `prism-spec-engine`.
