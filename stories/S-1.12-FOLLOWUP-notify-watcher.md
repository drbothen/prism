# S-1.12-FOLLOWUP — prism-spec-engine: HotReloadWatcher — notify v7, Debounce, Validation Gate, SIGHUP

## Narrative

As the Prism configuration management system, I want `HotReloadWatcher::start` and `stop`
implemented with notify v7 filesystem watching, 500ms debounce, a dry-run validation gate
before any arc-swap, audit emission on reload success/fail, and SIGHUP integration, so
that configuration changes to sensor TOML specs are automatically detected, validated, and
applied without a process restart — with rollback safety when validation fails.

## Objective

Implement `HotReloadWatcher::start` (currently `unimplemented!()` at `hot_reload.rs:66`)
and `HotReloadWatcher::stop` (currently `unimplemented!()` at `hot_reload.rs:72`) per
the specification in ADR-022 §E.

No `unimplemented!()` or `todo!()` may remain in either method before merge. The inverted-polarity
test `test_BC_2_16_007_hot_reload_watcher_start_is_stub` (which is green because the
function panics) MUST be replaced with a real behavior test. Per POL-12 and ADR-020 F-AUD-D3-01.

## Tasks

1. Implement `HotReloadWatcher::start` at `hot_reload.rs:66`:

   ```rust
   pub fn start(
       &self,
       manager: Arc<ArcSwap<ConfigManager>>,
       spec_dir: PathBuf,
       debounce_ms: u64,
   ) -> Result<(), SpecEngineError> {
       let (tx, rx) = std::sync::mpsc::channel::<notify::Result<notify::Event>>();
       let mut watcher = notify::RecommendedWatcher::new(tx, notify::Config::default())?;
       // Use NonRecursive to avoid following symlinks (security boundary per ADR-022 §E)
       watcher.watch(&spec_dir, notify::RecursiveMode::NonRecursive)?;
       // Store watcher in self (so it lives as long as HotReloadWatcher)
       *self.watcher.lock().unwrap() = Some(watcher);
       // Spawn background debounce task
       let manager = manager.clone();
       let spec_dir = spec_dir.clone();
       tokio::spawn(async move {
           Self::debounce_loop(rx, manager, spec_dir, debounce_ms).await;
       });
       Ok(())
   }
   ```

2. Implement debounce loop and `try_reload` at `hot_reload.rs`:

   ```rust
   async fn debounce_loop(
       rx: std::sync::mpsc::Receiver<notify::Result<notify::Event>>,
       manager: Arc<ArcSwap<ConfigManager>>,
       spec_dir: PathBuf,
       debounce_ms: u64,
   ) {
       // Accumulate events: HashMap<PathBuf, EventKind>
       // After debounce_ms of inactivity: flush → call try_reload()
       // Use tokio::time::sleep for debounce window
       // Only react to EventKind::Modify and EventKind::Create; ignore CLOSE_WRITE if no content change
       // Do NOT follow symlinks
   }

   pub async fn try_reload(
       manager: &Arc<ArcSwap<ConfigManager>>,
       spec_dir: &PathBuf,
       audit: &Arc<AuditEmitter>,
   ) -> Result<(), SpecEngineError> {
       // 1. Run parse_spec_directory(spec_dir) in dry-run (just validate, don't commit)
       match parse_spec_directory(spec_dir) {
           Ok(new_snapshot) => {
               // 2. Arc-swap: manager.store(Arc::new(ConfigManager::new(new_snapshot)))
               let n_sensors = new_snapshot.sensors().len();
               manager.store(Arc::new(ConfigManager::new(new_snapshot)));
               // 3. Emit audit ReloadSucceeded entry
               audit.emit(AuditEntry::reload_succeeded(spec_dir, n_sensors)).await?;
               tracing::info!(
                   spec_dir = %spec_dir.display(),
                   n_sensors,
                   "hot-reload: config swapped"
               );
               Ok(())
           }
           Err(err) => {
               // 4. Emit audit ReloadFailed entry (retain existing snapshot — no swap)
               audit.emit(AuditEntry::reload_failed(spec_dir, err.code())).await;
               tracing::warn!(
                   spec_dir = %spec_dir.display(),
                   error = %err,
                   "hot-reload: validation failed; retaining current config"
               );
               // Return Ok — failed reload is NOT a fatal error for the watcher task
               Ok(())
           }
       }
   }
   ```

3. Implement `HotReloadWatcher::stop` at `hot_reload.rs:72`:
   ```rust
   pub fn stop(&self) -> Result<(), SpecEngineError> {
       // Drop the watcher (stops filesystem event delivery)
       *self.watcher.lock().unwrap() = None;
       // The background tokio task will exit when rx is closed
       Ok(())
   }
   ```

4. Handle cross-platform quirks per ADR-022 §E table:

   | Quirk | Handling |
   |-------|----------|
   | macOS FSEvents batched callbacks | Debounce window collapses batches — already handled by accumulation |
   | inotify double MODIFY (atomic-write editors) | Filter: only `EventKind::Modify` + `EventKind::Create`; skip duplicate paths within same debounce window |
   | File deletion | Treat as validation failure: `parse_spec_directory` returns Err for missing files; `try_reload` emits `ReloadFailed` audit; does NOT remove sensor from running config |
   | Symlinks in spec_dir | `RecursiveMode::NonRecursive` on spec_dir; do NOT follow symlinks |
   | Watcher task panic | `tokio::task::spawn` join handle — if the task exits unexpectedly, log `tracing::error!` + emit degraded-mode alert; boot continues without hot-reload |

5. Wire SIGHUP integration: the `signals.rs` SIGHUP handler (from S-WAVE5-PREP-01) sends
   on `reload_tx: tokio::sync::mpsc::Sender<()>`. Boot step 10 spawns a task that listens
   on `reload_rx` and calls `HotReloadWatcher::try_reload()` directly — same code path as
   filesystem watcher. This is idempotent (concurrent SIGHUP + filesystem event → two
   sequential `try_reload` calls, both safe).

   Implement `HotReloadWatcher::listen_sighup`:
   ```rust
   pub fn listen_sighup(
       manager: Arc<ArcSwap<ConfigManager>>,
       spec_dir: PathBuf,
       audit: Arc<AuditEmitter>,
       mut reload_rx: tokio::sync::mpsc::Receiver<()>,
   ) {
       tokio::spawn(async move {
           while reload_rx.recv().await.is_some() {
               let _ = Self::try_reload(&manager, &spec_dir, &audit).await;
           }
       });
   }
   ```

6. Replace the inverted-polarity test `test_BC_2_16_007_hot_reload_watcher_start_is_stub`

## Acceptance Criteria

**AC-1:** Given `HotReloadWatcher::start(manager, spec_dir, 500)` is called with a valid
spec directory, When a TOML sensor spec file is modified in `spec_dir`, Then within
600ms (500ms debounce + processing time) the `ConfigManager` ArcSwap is updated with
the new `ConfigSnapshot` and `tracing::info!("hot-reload: config swapped")` is emitted.
(traces to BC-2.16.007 postcondition — spec hot reload without restart)

**AC-2:** Given a TOML file in spec_dir is modified to contain invalid TOML, When the
debounce window closes and `try_reload` runs, Then `parse_spec_directory` returns `Err`,
the `ArcSwap` is NOT updated (previous snapshot retained), a `ReloadFailed` audit entry
is emitted, and `tracing::warn!` is logged with the validation error.
(traces to BC-2.16.007 postcondition — failed reload retains current config; BC-2.16.006 arc-swap integrity)

**AC-3:** Given the SIGHUP handler fires via `listen_sighup`, When the handler calls
`try_reload`, Then the same validation gate + arc-swap + audit sequence runs as the
filesystem watcher path.
(traces to BC-2.16.007 postcondition — SIGHUP triggers same reload path)

**AC-4:** Given a file deletion in spec_dir (accidental TOML removal), When the debounce
fires, Then `parse_spec_directory` returns `Err` (missing file), `ReloadFailed` audit is
emitted, and the running config is NOT changed (deleted sensor remains active).
(traces to BC-2.16.007 postcondition — deletion is treated as validation failure for ops safety)

**AC-5:** Given `HotReloadWatcher::start` is called on a spec_dir containing a symlink,
When the watcher is active, Then the watcher does NOT follow the symlink (uses
`RecursiveMode::NonRecursive`). Modifications inside the symlink target do NOT trigger
a reload event.
(traces to BC-2.16.007 — security boundary: no symlink following)

**AC-6:** Given `HotReloadWatcher::stop` is called, When stop completes, Then the
filesystem watcher is dropped and no further reload events are processed. The background
debounce task exits cleanly.
(traces to BC-2.16.007 postcondition — stop is clean)

**AC-7:** The inverted-polarity test `test_BC_2_16_007_hot_reload_watcher_start_is_stub`
MUST NOT exist in the test suite when this story merges. It must be replaced by
`test_hot_reload_watcher_start_succeeds` (AC-7 from task 6). Per ADR-020 F-AUD-D3-01.
No `unimplemented!()` remains in `HotReloadWatcher::start` or `::stop`. Per POL-12.

## Architecture Compliance Rules

| Rule | Source | Enforcement |
|------|--------|-------------|
| Arc-swap MUST be used for ConfigManager hot swap — no `RwLock` | BC-2.16.006 invariant + S-1.14 precedent | Code review; no `RwLock<ConfigManager>` in hot_reload.rs |
| `try_reload` MUST run `parse_spec_directory` BEFORE swapping — validation gate required | ADR-022 §E + BC-2.16.007 postcondition | AC-2 integration test (invalid TOML → no swap) |
| Watcher MUST NOT follow symlinks | ADR-022 §E security note | AC-5 test; `RecursiveMode::NonRecursive` |
| Debounce MUST NOT busy-poll — use `tokio::time::sleep` or interval | ADR-022 §E spec | Code review |
| Watcher task panic MUST be caught and logged — NOT crash the host process | ADR-022 §E quirk table | `tokio::spawn` join handle monitoring |
| File deletion MUST NOT remove a sensor from running config | ADR-022 §E file-deletion quirk | AC-4 integration test |

## Edge Cases

| ID | Description | Expected Behavior |
|----|-------------|-------------------|
| EC-001 | Two TOML files modified within debounce window | Debounce collapses to one `try_reload` call covering both files |
| EC-002 | SIGHUP arrives while filesystem-triggered reload is in progress | Second `try_reload` call queues after first completes; both are safe (idempotent) |
| EC-003 | Watcher task crashes (notify returns Err event) | Log `tracing::error!`; hot-reload degrades (boot-time config remains active); no process crash |
| EC-004 | `parse_spec_directory` takes > 5s (large spec directory) | `try_reload` awaits completion; debounce prevents re-triggering during validation |
| EC-005 | `HotReloadWatcher::start` called while another start is already active | New watcher replaces old; old drop cleans up the filesystem watch |

## Dev Notes

- `notify::RecommendedWatcher` returns `std::sync::mpsc::Sender<notify::Result<Event>>`.
  The blocking `rx.recv()` must be run in a `spawn_blocking` wrapper to avoid blocking
  the tokio thread pool, OR use `tokio::sync::mpsc` with `notify`'s crossbeam integration.
  Check notify v7 docs for the preferred tokio async channel integration pattern.
- The `HotReloadWatcher` struct will need a `Mutex<Option<RecommendedWatcher>>` field to
  store the live watcher handle between `start` and `stop`. The watcher must be dropped
  to stop event delivery.
- For debounce: accumulate events in a `HashMap<PathBuf, notify::EventKind>`. Flush when
  the channel is empty (after `recv_timeout(500ms)` returns `RecvTimeoutError::Timeout`).
- The `debounce_ms` parameter defaults to 500 at boot step 10 per ADR-022 §B step 10.
  Integration tests should use a much smaller value (50ms) to keep test duration short.
