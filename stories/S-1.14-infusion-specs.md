# S-1.14 — prism-spec-engine: Infusion Spec Loading and UDF Registration

## Narrative

As a Prism query engine, I want infusion enrichment sources declared in `.infusion.toml`
files to be automatically loaded and exported as InfusionUdfDescriptors for downstream
DataFusion UDF registration by prism-query, and available as `enrich` pipe stages in
PrismQL, so that analysts can enrich query results with GeoIP, threat intelligence,
asset inventory, and vulnerability context without writing any Rust code.

## Objective

Implement the infusion enrichment framework per AD-020: TOML spec loading for infusion
definitions, `InfusionRegistry` mapping names to specs and loaded data, export of
InfusionUdfDescriptors for each declared field (DataFusion registration in prism-query
S-3.02, NOT here), `enrich` pipe stage descriptor in PrismQL, built-in source
type loaders (MaxMind MMDB, CSV, JSON lookup), three-tier caching (per-query dedup,
in-memory LRU, persistent RocksDB `infusion_cache` CF), and plugin infusion delegation
to the WASM runtime (S-1.15).

## Tasks

1. Define `InfusionSpec` and `InfusionRegistry` in `crates/prism-spec-engine/src/infusion/mod.rs`:
   - `InfusionSpec { infusion_id, name, infusion_type (LocalLookup | Plugin), source, fields: Vec<InfusionField>, pipe_stage: Option<PipeStageConfig>, plugin_config, credentials }`
   - `InfusionField { name, input_field, input_type, output_type, description }`
   - `InfusionRegistry`: `HashMap<String, (InfusionSpec, Arc<dyn InfusionSource>)>` wrapped in `arc_swap::ArcSwap` for hot reload
   - `InfusionSource` trait: `enrich_single(input: &str, input_type: &str) -> Option<serde_json::Value>` and `enrich_batch(inputs: &[String], input_type: &str) -> Vec<Option<serde_json::Value>>`

2. Implement TOML spec parser in `crates/prism-spec-engine/src/infusion/loader.rs`:
   - Load `*.infusion.toml` files from `{config_dir}/infusions/` directory
   - Validate required fields: `infusion_id`, at least one `[[infusion.fields]]` entry
   - Validate source type is one of: `maxmind_mmdb`, `csv`, `json_lookup`, `plugin`
   - Validate credential references use reference-based model (no inline values)
   - Validate `pipe_stage.adds_columns` matches the `[[infusion.fields]]` names declared
   - On validation error, return `Err` — do not partially register

3. Implement built-in source loaders:
   - `sources/mmdb.rs`: MaxMind MMDB reader using the `maxminddb` crate. Load `.mmdb` file into memory via `maxminddb::Reader`. Implement `InfusionSource`: input is IP address string, output is `serde_json::Value` with all declared fields (country ISO code, city, ASN, is_tor flag). Support `refresh_interval` — schedule re-read via tokio interval.
   - `sources/csv.rs`: Load CSV file with designated `key_column` into `HashMap<String, HashMap<String, String>>`. Implement `InfusionSource`: lookup by key, return declared `csv_column` fields. Support `refresh_interval` — reload into a new HashMap, then arc-swap.
   - `sources/json_lookup.rs`: Load JSON file as `serde_json::Map<String, Value>`. Key is the lookup key; value object provides declared output fields. Support `refresh_interval`.

4. Export `InfusionUdfDescriptor` for DataFusion UDF registration (done by prism-query S-3.02):
   - For each `InfusionField` in each loaded `InfusionSpec`, produce an `InfusionUdfDescriptor`
     containing: name (e.g., `geoip_country`), input type, output type, lookup function reference
   - prism-query (S-3.02) creates the actual `datafusion::logical_expr::ScalarUDF` and registers
     it in the SessionContext — prism-spec-engine does NOT depend on DataFusion
   - `InfusionRegistry::udf_descriptors()` → `Vec<InfusionUdfDescriptor>` (consumed by S-3.02)
   - On hot reload: notify prism-query to deregister old UDFs and register new ones

5. Export `EnrichStageDescriptor` for `enrich` pipe stage (execution in prism-query S-3.02):
   - Descriptor contains: infusion_name, input_field, output_columns (from pipe_stage.adds_columns)
   - Grammar extension (`ENRICH <infusion_name> ON <field_ref>`) parsed by S-3.01/S-3.06
   - Actual RecordBatch manipulation (extracting values, calling InfusionSource::enrich_batch,
     joining columns) lives in prism-query — prism-spec-engine does NOT depend on Arrow
   - `InfusionRegistry::enrich_descriptor(name: &str) -> Option<EnrichStageDescriptor>`
   - Missing infusion name → `E-INFUSE-001: "Unknown infusion '{name}'. Run list_infusions to see available enrichments."`

6. Implement three-tier caching in `crates/prism-spec-engine/src/infusion/cache.rs`:
   - **Tier 1 — Per-query dedup:** `HashMap<String, Option<serde_json::Value>>` scoped to a single query execution. Keyed by `"{infusion_id}:{input_value}"`. Allocated at query start, dropped at query end. Ensures 10K events with 200 unique IPs = 200 infusion calls, not 10K.
   - **Tier 2 — In-memory LRU:** `lru::LruCache<String, (serde_json::Value, Instant)>` with configurable capacity (default 10,000 entries) and TTL (configurable per infusion, default 1 hour). Shared across queries. Guarded by `tokio::sync::Mutex`.
   - **Tier 3 — Persistent cache:** Uses a `CacheBackend` trait (defined in prism-core, implemented by prism-storage's RocksDB backend). The trait is injected at startup by prism-bin, not imported directly — prism-spec-engine does NOT depend on prism-storage. Key: `"{infusion_id}:{sha256(input_value)}"`. Value: `(serde_json::Value, expiry_timestamp)` encoded as bincode 2.x. TTL eviction: check expiry on read, skip expired entries (lazy eviction). Uses `infusion_cache` CF in production.
   - Lookup order: Tier 1 → Tier 2 → Tier 3 → call InfusionSource → populate all tiers.

7. Implement plugin bridge in `crates/prism-spec-engine/src/infusion/plugin_bridge.rs`:
   - `type = "plugin"` infusions delegate to the WASM runtime (S-1.15)
   - `PluginInfusionSource`: implements `InfusionSource` by calling `PluginRuntime::enrich_single` / `enrich_batch` on the named `.prx` file
   - If S-1.15 is not yet built, stub with `unimplemented!()` and a clear error message — local lookup types must work independently of plugin runtime

8. Expose `InfusionRegistry::is_api_backed(udf_name: &str) -> bool` method:
   - Returns true if the named UDF comes from a `type = "plugin"` infusion
   - This method is consumed by S-4.03 (detection rule loader) to enforce E-RULE-012
     rejection of API-backed infusion UDFs in detection rule filters
   - NOTE: The actual E-RULE-012 enforcement lives in S-4.03, not here. This story
     provides the query interface; S-4.03 calls it during rule validation.

9. Participate in hot reload (AD-018):
   - Watch `{config_dir}/infusions/*.infusion.toml` and `{config_dir}/data/*`
   - On spec change: re-validate, re-register UDFs, arc-swap `InfusionRegistry`
   - On data file change: reload source data into new reader, arc-swap (old reader lives until in-flight queries complete)
   - Failed validation retains old registry (CI-002)

9a. Implement `infusion_cache` RocksDB CF initialization
    and per-query LRU cache structure (previously these were only mentioned in the Tier 3
    cache description; this task makes them explicit deliverables):
   - **`infusion_cache` CF initialization:** Register `"infusion_cache"` as a named CF in
     `crates/prism-storage/src/lib.rs` under `StorageDomain::InfusionCache`. CF must be
     opened with a TTL compaction filter configured (RocksDB's built-in TTL via
     `Options::set_merge_operator` or `rocksdb::DB::open_cf_with_ttl`). Key format:
     `"{infusion_id}:{sha256(input_value)}"`. Value format: bincode 2.x encoding of
     `(serde_json::Value, expiry_unix_secs: u64)`.
   - **Per-query LRU cache structure:** `QueryScopedInfusionCache` — a `HashMap<String, Option<serde_json::Value>>`
     allocated at query start, dropped at query end. Key format: `"{infusion_id}:{input_value}"`.
     Allocated fresh for each `QueryEngine::execute()` call; passed into the UDF execution
     context so all UDF calls within the query share the same dedup map.
   - **TTL-based eviction policy:** Each cache entry includes an `expiry_unix_secs` timestamp.
     On RocksDB read: check if `expiry_unix_secs < now()` → discard expired value and
     treat as cache miss. Write TTL per infusion spec's `cache_ttl_secs` field (default 3600s).
   - **Hot path read/write integration:** Tier 1 (query-scoped HashMap) check MUST occur
     before Tier 2 LRU check, which MUST occur before Tier 3 RocksDB check. On Tier 3
     hit: populate Tier 2 LRU. On live source call: populate all three tiers.

10. Write tests in `crates/prism-spec-engine/tests/infusion_tests.rs`:
    - Load `geoip.infusion.toml` with a test MMDB fixture → verify `geoip_country` UDF registered
    - Execute `SELECT geoip_country('203.0.113.1') FROM events` → verify correct country code returned
    - Execute `FROM events | enrich geoip ON device_ip` → verify adds `geoip_country`, `geoip_city` columns
    - Verify per-query dedup: 3 rows with same IP → `InfusionSource::enrich_single` called once
    - Verify `type = "plugin"` infusion UDF rejected in detection rule filter with `E-RULE-012`
    - Verify hot reload: modify spec, trigger watch event, verify new UDF registration, old queries still complete
    - Verify CSV source: load asset_inventory CSV, `asset_owner('192.168.1.1')` returns correct department

## Acceptance Criteria

**AC-1:** Given a `geoip.infusion.toml` with `type = "local_lookup"` and `source.type = "maxmind_mmdb"`,
When the spec engine loads it, Then the UDFs `geoip_country`, `geoip_city`, `geoip_asn`, and
`geoip_is_tor` are exported as InfusionUdfDescriptors by InfusionRegistry (DataFusion registration verified in S-3.02).
(traces to BC-2.19.001 / INV-INFUSE-001)

**AC-2:** Given a PrismQL query `SELECT geoip_country(device_ip), count(*) FROM EVENTS GROUP BY geoip_country(device_ip)`,
When the query executes against a result set with 500 events having 30 unique IPs,
Then the `geoip_country` source is called exactly 30 times (per-query dedup active).
(traces to BC-2.19.002 / INV-INFUSE-002)

**AC-3:** Given a pipe stage `| enrich geoip ON device_ip` in a PrismQL query,
When the query executes, Then the output schema includes the columns `geoip_country`,
`geoip_city`, `geoip_asn`, and `geoip_is_tor` added to the upstream result schema.
(traces to BC-2.19.001 / INV-INFUSE-001)

**AC-4:** Given an infusion named `threat_intel` with `type = "plugin"`, When a detection
rule references `threat_score(device_ip)` in its `filter` expression, Then the detection
rule loader rejects the rule with error `E-RULE-012` naming `threat_intel`.
(traces to BC-2.19.003 / INV-INFUSE-003)

**AC-5:** Given a valid infusion spec is loaded, When the `.infusion.toml` file is modified
and the file watcher fires, Then the `InfusionRegistry` is updated atomically and in-flight
queries using the old registry complete without error.
(traces to BC-2.19.004 / INV-INFUSE-004)

**AC-6:** Given an infusion spec with `[infusion.credentials]` entries, When the spec is
logged or an error is returned from the loader, Then no credential values appear in any
log output or error message.
(traces to BC-2.19.005 / INV-INFUSE-005)

**AC-7:** Given a CSV-backed `asset_inventory` infusion with `key_column = "ip_address"`,
When `asset_owner('192.168.1.10')` is called and that IP is in the CSV, Then the returned
value matches the `department` column value from the CSV file.

**AC-8:** Given an infusion result is fetched and stored in the RocksDB `infusion_cache` CF,
When the infusion is called again in a new query (after Tier 1 and Tier 2 LRU have expired),
Then the result is read from RocksDB without calling the `InfusionSource` again.

## Architecture Mapping

| Component | Module | Pure/Effectful |
|-----------|--------|----------------|
| Infusion spec parser and registry | `prism-spec-engine` (SS-16, SS-19) | Mixed |

`prism-spec-engine` extends its TOML parser with infusion spec support (`InfusionRegistry`,
`InfusionPluginExecutor`) per AD-020. Infusion UDFs are registered into `prism-query`'s
DataFusion `SessionContext` via `InfusionRegistry`. Spec parsing/validation is pure;
HTTP calls to external enrichment APIs and arc-swap publication are effectful.
Per `architecture/module-decomposition.md`, SS-19 (Infusion Enrichment Framework) is
owned by `prism-spec-engine`.

## Architecture Compliance Rules

- `InfusionRegistry` MUST use `arc_swap::ArcSwap` for hot reload — same pattern as
  `SensorRegistry` in S-1.11. Never use `RwLock` for the registry swap.
- Per-query dedup cache MUST be allocated per-query and dropped after the query completes.
  It must NOT be a shared in-memory structure — it is only safe within a single query
  execution context.
- API-backed infusion UDFs (plugin type) MUST NOT be registered for use inside
  DataFusion's synchronous UDF executor path for detection rules. They may make external
  HTTP calls that block. The enforcement point is spec validation at rule load time.
- Credential values from `[infusion.credentials]` MUST NOT appear in log output at any
  level, including TRACE. Use the same AI-opaque model as sensor credentials (AD-017).
- `infusion_cache` RocksDB CF key format MUST use SHA-256 hash of the input value, not
  the raw value, to normalize variable-length keys and avoid key size explosions.
- UDF names are global within a DataFusion `SessionContext`. Detect and reject duplicate
  UDF names across multiple infusion specs at load time, not at query time.

## Dev Notes

- The MaxMind GeoLite2 `.mmdb` format requires the `maxminddb` crate. The test fixture
  can be generated with the `maxminddb-writer` or `mmdbwriter` crate — add it as a
  `[dev-dependencies]` only.
- Per-query dedup scope: the dedup `HashMap` must be passed into the UDF execution
  context as part of `SessionState` or via a `QueryScopedCache` structure injected at
  query start. DataFusion's `ScalarUDF` receives a `ColumnarValue` argument — the dedup
  needs to operate at the Arrow array level, collecting unique input values before
  dispatching to the source.
- The `enrich` pipe stage implementation in `pipe_stage.rs` can be a post-execution
  transformation: run the upstream query, get the `RecordBatch`, then call
  `InfusionSource::enrich_batch` on unique values of the ON-field, then use
  `arrow::compute in prism-query (S-3.02)` to add new columns. This avoids modifying the DataFusion
  logical plan for the initial implementation.
- RocksDB Tier 3 uses lazy eviction (check TTL on read, discard if expired). Background
  compaction via RocksDB's TTL compaction filter can be added later. For now, lazy
  eviction is sufficient.
