# S-1.03 — prism-core: Capability Resolution Engine

## Narrative

As a Prism platform implementer, I want a hierarchical capability resolution engine in
prism-core, so that the feature flag system (prism-security, S-1.08) and the MCP tool
router (prism-mcp, S-5.02) can determine whether a given client is permitted to invoke
a specific capability — with formally proven deny-by-default and most-specific-wins
semantics.

## Objective

Implement `ClientCapabilities`, `CapabilityPath`, `CapabilityEffect`, and
`CapabilityExplanation` in prism-core. These types are used by prism-security (feature
flags) and prism-mcp (tool routing) to answer: "is client X allowed to do action Y?"
Three Kani proofs (VP-002, VP-003, VP-004) verify the resolution semantics are correct
under all inputs.

## Tasks

1. Implement `CapabilityPath` newtype in `crates/prism-core/src/capability.rs`:
   - Inner type: `Arc<str>`
   - Represents a dot-separated path: e.g., `"crowdstrike.hosts.write"`,
     `"audit.read"`, `"detections.acknowledge"`
   - Constructor: `CapabilityPath::new(s: &str) -> Result<Self, PrismError>`
   - Validation: non-empty, each segment is `[a-zA-Z0-9_]+`, max 8 segments,
     max total length 256 characters
   - Method: `parent(&self) -> Option<CapabilityPath>` strips the last segment
   - Method: `is_prefix_of(&self, other: &CapabilityPath) -> bool`
   - Derive: `Clone, Debug, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize`

2. Implement `CapabilityEffect` enum in `crates/prism-core/src/capability.rs`:
   - Variants: `Allow`, `Deny`
   - Derive: `Clone, Copy, Debug, PartialEq, Eq, serde::Serialize, serde::Deserialize`

3. Implement `CapabilityExplanation` struct in `crates/prism-core/src/capability.rs`:
   - Fields:
     - `allowed: bool` — the final decision
     - `matched_path: Option<CapabilityPath>` — the path that matched (None if default)
     - `effect: CapabilityEffect` — what the matched path specified
     - `reason: &'static str` — human-readable: "deny-by-default", "explicit-allow",
       "explicit-deny", "parent-allow", "parent-deny"
   - Purpose: used by audit logging and MCP error responses

4. Implement `ClientCapabilities` struct in `crates/prism-core/src/capability.rs`:
   - Inner: `BTreeMap<CapabilityPath, CapabilityEffect>` (sorted for deterministic
     iteration)
   - Constructor: `ClientCapabilities::new() -> Self` (empty = deny everything)
   - Constructor: `ClientCapabilities::from_iter(iter)` for building from config
   - Method: `grant(&mut self, path: CapabilityPath, effect: CapabilityEffect)`
   - Method: `is_allowed(&self, path: &CapabilityPath) -> (bool, CapabilityExplanation)`:
     - **Deny-by-default:** if no entry matches (directly or via parent prefix), return
       `(false, explanation citing "deny-by-default")`
     - **Most-specific wins:** walk from the exact path upward to its parents; the first
       match (longest prefix) determines the outcome
     - **Deny overrides Allow at same specificity:** if the exact path has both Allow
       and Deny (impossible via the API, but Kani must prove this), Deny wins
   - Method: `capabilities_for_display(&self) -> Vec<(&CapabilityPath, &CapabilityEffect)>`
     returns sorted list for MCP `list_capabilities` tool

5. Write VP-002 Kani proof in `crates/prism-core/src/proofs/capability.rs`:
   - Create an empty `ClientCapabilities`
   - For any symbolic `CapabilityPath`, assert `is_allowed()` returns `false`
   - Proof name: `proof_deny_by_default`

6. Write VP-003 Kani proof in `crates/prism-core/src/proofs/capability.rs`:
   - Set `"a.b"` = Deny, `"a.b.c"` = Allow
   - Assert `is_allowed("a.b.c")` returns `true` (most-specific "a.b.c" = Allow wins)
   - Set `"a.b"` = Allow, `"a.b.c"` = Deny
   - Assert `is_allowed("a.b.c")` returns `false` (most-specific "a.b.c" = Deny wins)
   - Proof name: `proof_most_specific_wins`

7. Write VP-004 Kani proof in `crates/prism-core/src/proofs/capability.rs`:
   - Set exactly `"a.b"` = Allow
   - Assert `is_allowed("a.b")` returns `true`
   - Verify the explanation's `matched_path` equals `"a.b"` and `reason` is
     `"explicit-allow"`
   - Proof name: `proof_exact_match_explanation`
   - Note: VP-004's stated constraint ("Deny overrides Allow at same specificity") is
     enforced by the API making it impossible to set both Allow and Deny for the same
     path — the `grant()` method is a simple insert, last write wins. The proof verifies
     that the `BTreeMap` semantics (single entry per key) hold under Kani.

8. Write unit tests in `crates/prism-core/src/tests/capability_tests.rs`:
   - Test empty capabilities → deny for arbitrary paths
   - Test `{"crowdstrike" → Allow}` → `is_allowed("crowdstrike.hosts.write")` = true
   - Test `{"crowdstrike" → Allow, "crowdstrike.hosts.write" → Deny}` →
     `is_allowed("crowdstrike.hosts.write")` = false
   - Test `CapabilityPath::parent()` for single-segment path returns None
   - Test `CapabilityPath::is_prefix_of()` correctly identifies prefix relationships
   - Test `CapabilityPath::new()` rejects paths with empty segments (e.g., `"a..b"`)

## Acceptance Criteria

**AC-1:** Given an empty `ClientCapabilities`, When `is_allowed("any.path")` is called,
Then it returns `(false, explanation)` where `explanation.reason == "deny-by-default"`.

**AC-2:** Given `ClientCapabilities` with `{"crowdstrike" → Allow}`, When
`is_allowed("crowdstrike.hosts.write")` is called, Then it returns `true` (inherits
Allow from parent path `"crowdstrike"`).

**AC-3:** Given `ClientCapabilities` with `{"crowdstrike" → Allow, "crowdstrike.hosts.write" → Deny}`,
When `is_allowed("crowdstrike.hosts.write")` is called, Then it returns `false`
(most-specific path `"crowdstrike.hosts.write"` = Deny wins over parent Allow).

**AC-4:** Given `ClientCapabilities` with `{"audit.read" → Allow}`, When
`is_allowed("audit.read")` is called, Then it returns `true` and `explanation.matched_path`
equals `"audit.read"`.

**AC-5:** Given `CapabilityPath::new("a..b")`, When validation runs, Then it returns
`Err` (empty segment between dots is invalid).

**AC-6:** Given `CapabilityPath("a.b.c")`, When `parent()` is called twice, Then the
result is `Some(CapabilityPath("a"))`.

**AC-7:** Given `ClientCapabilities` with `{"crowdstrike.hosts" → Allow}`, When
`is_allowed("crowdstrike.hosts.read")` is called, Then it returns `true` (parent
`"crowdstrike.hosts"` = Allow covers child paths).

**AC-8:** VP-002 Kani proof passes: deny-by-default for empty capabilities.

**AC-9:** VP-003 Kani proof passes: most-specific path wins.

**AC-10:** VP-004 Kani proof passes: exact match with explanation correctness.

## Architecture Compliance Rules

- `ClientCapabilities` MUST be immutable after construction for thread safety
  (expose `grant()` only during config loading, not during request handling)
- Path resolution MUST be O(depth) per lookup, not O(n) over all stored capabilities
  (walk parent chain, not full map scan)
- `CapabilityExplanation` MUST be returned with every `is_allowed()` call — callers
  must not need to re-invoke for audit logging
- The capability engine MUST NOT have any I/O dependencies — it is pure computation

## Notes

- The resolution algorithm walks from the exact path upward to the root. For a path
  `"a.b.c.d"`, the lookup order is: `"a.b.c.d"`, `"a.b.c"`, `"a.b"`, `"a"`.
  The first key present in the `BTreeMap` wins. If none match, deny-by-default applies.
- `BTreeMap` (not `HashMap`) is intentional: sorted order enables deterministic
  `capabilities_for_display()` without a sort step, and is required for reproducible
  Kani proofs.
- The `CapabilityExplanation.reason` field uses `&'static str` to avoid allocation in
  the hot path. The 4 reasons are known statically at compile time.
- prism-security's feature flag system (S-1.08) uses `ClientCapabilities` directly.
  prism-mcp's tool router (S-5.02) also uses it. Both receive a pre-built
  `ClientCapabilities` from configuration; neither calls `grant()` at runtime.
