# S-3.1.06 — prism-sensors: migrate adapter constructors and fan-out to OrgId

## Narrative
- **As a** sensor dispatch layer
- **I want** all adapter constructors and fan-out dispatch to operate on `OrgId` (not
  `OrgSlug` or a bare `String`)
- **So that** per-org sensor data isolation is structurally enforced at the type level
  and cross-tenant dispatch is a compile-time impossibility

## Description

ADR-006 §4 Step 4 and ADR-008 §2.1 migrate all mutable state stores in client-mode
Security Telemetry DTU crates from bare-`String` keys to `(OrgId, String)` composite keys.
This story handles the `prism-sensors` crate: adapter constructors gain an explicit
`OrgId` parameter (replacing the implicit/global `client_id: String` field in `SensorSpec`
at `prism-sensors/src/adapter.rs:38`); the fan-out logic in `init_registry` dispatches
per `OrgId`; and the event buffer key prefix at `event_buffer.rs:46` migrates from
`TenantId::as_str()` to `OrgId::to_string()`.

The client-mode DTU state store migrations (ClarotyState, ArmisState, CrowdStrikeState,
CyberintState) are specified in BC-3.2.001 Architecture Anchors and are implemented
in the respective DTU crate stories (separate from this story) — but `prism-sensors`
must provide the correct `OrgId` at dispatch time for those migrations to be correct.

## Acceptance Criteria

**AC-1 (traces to BC-3.2.001 precondition 2):** All mutable state `HashMap` fields in
`prism-sensors/src/adapter.rs` that were keyed by bare `String` (or `OrgSlug`) are
migrated to `(OrgId, String)` composite keys. A grep for `HashMap<String,` in
`adapter.rs` returns zero hits after migration.

**AC-2 (traces to BC-3.2.001 precondition 3):** The query plan struct carries
`org_id: OrgId` as a non-nullable field. The field type is `OrgId` (not `Option<OrgId>`);
absence of an org constraint is a compile error.

**AC-3 (traces to BC-3.2.001 postcondition 1):** A proptest that stores device state
under `(org_id_A, "dev-1")` and looks up `(org_id_B, "dev-1")` returns the default
value (empty set / `None`), not org_A's data.

**AC-4 (traces to BC-3.2.001 postcondition 2):** A write under `org_id_A` does not
modify any entry keyed under `org_id_B` — verified by proptest with adversarial
(org_id_A, org_id_B, resource_id) triples.

**AC-5 (traces to BC-3.2.001 invariant 1):** The event buffer key prefix at
`event_buffer.rs:46` is `OrgId::to_string()` (UUID string), not `OrgSlug::as_str()` or
a raw string. Verified by reading the constructed key and asserting it matches the UUID
format `[0-9a-f-]{36}`.

## Edge Cases

| ID | Scenario | Expected Behavior |
|----|----------|-------------------|
| EC-001 | Two orgs have devices with identical vendor-assigned IDs | Stored under `(org_id_A, "dev-1")` and `(org_id_B, "dev-1")` respectively; each lookup returns its org's data only (BC-3.2.001 TV-3.2.001-03) |
| EC-002 | `reset_for(org_id_A)` on a multi-org store | Removes all `(org_id_A, *)` entries; `(org_id_B, *)` entries unaffected (BC-3.2.001 EC-004) |
| EC-003 | Dispatch layer passes wrong OrgId to adapter | Dispatch layer must verify OrgId match before invoking DTU method (ADR-007 §2.2); mismatch is a fatal dispatch error |
| EC-004 | Shared-mode DTU (Slack, PagerDuty) query with OrgId in query plan | Shared-mode DTUs embed OrgId in the payload (BC-3.2.004, out of scope for this story) — NOT in the state store key |
| EC-005 | `DEFAULT_ORG_ID` test constant used in production code path | Build fails: `DEFAULT_ORG_ID` is `#[cfg(test)]` only (BC-3.2.001 invariant 3) |
