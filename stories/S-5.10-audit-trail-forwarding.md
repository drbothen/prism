# S-5.10 — prism-audit: Audit Trail External Forwarding

## Narrative

As a Prism MSSP operator, I want the audit trail forwarded to external compliance and
SIEM destinations with at-least-once delivery guarantees and durable on-disk buffering,
so that no audit entries are lost during destination outages and compliance obligations
(SOC2, NIST) are met even when external systems are temporarily unavailable.

## Objective

Implement audit trail external forwarding per `config-schema.md` §Audit Settings
`[audit.forward]`: at-least-once forwarding from the RocksDB `audit_buffer` CF to
external destinations (`vector`, `syslog`, `webhook`); exponential backoff retry
(2s base, 60s max); forwarding state tracked in RocksDB (delivered entry watermark);
on-disk durability — entries remain in `audit_buffer` until acknowledged by destination;
24-hour outage tolerance with FIFO eviction at configured cap; `prism audit status`
diagnostic command.

This story is separate from diagnostic log forwarding (S-5.09):
- S-5.09: best-effort, in-memory queue, diagnostic logs
- S-5.10: at-least-once, RocksDB-backed, audit-grade entries

## Tasks

1. Define `[audit.forward]` config schema in `crates/prism-audit/src/forward/mod.rs`:
   - `AuditForwardConfig { enabled: bool, destinations: Vec<AuditDestination>,
     buffer_cap_mb: u32, retry_max_attempts: u32, backoff_base_secs: u64,
     backoff_max_secs: u64 }`
   - Defaults: `buffer_cap_mb = 512`, `retry_max_attempts = 10`,
     `backoff_base_secs = 2`, `backoff_max_secs = 60`
   - `AuditDestination { name: String, destination_type: AuditDestType,
     endpoint: String, credentials: HashMap<String, CredentialRef>,
     min_severity: Option<SeverityId> }`
   - `AuditDestType`: `Vector | Syslog | Webhook`
   - Credentials: opaque `CredentialRef` per AD-017 — resolved at delivery time

2. Implement `AuditForwarder` trait in `crates/prism-audit/src/forward/mod.rs`:
   ```rust
   #[async_trait]
   pub trait AuditForwarder: Send + Sync {
       async fn deliver(&self, entries: &[AuditEntry]) -> Result<(), ForwardError>;
       fn name(&self) -> &str;
   }
   ```
   - `ForwardError`: `Transient(String)` (retry) | `Permanent(String)` (do not retry)
   - Permanent errors: invalid credentials (4xx auth errors), malformed entries
   - Transient errors: network errors, 5xx server errors, connection refused

3. Implement forwarding state tracker in `crates/prism-audit/src/forward/watermark.rs`:
   - Tracks which audit entries have been delivered to each destination
   - Uses a per-destination watermark key in the `audit_buffer` CF:
     `"forward_watermark:{destination_name}"` → last acknowledged entry ID (UUID v7)
   - `Watermark::advance(dest_name: &str, entry_id: EntryId) -> Result<(), StorageError>`:
     - Write the new watermark atomically ONLY after the destination has acknowledged
       receipt (2xx response or syslog TCP ACK)
   - `Watermark::last_delivered(dest_name: &str) -> Option<EntryId>`: read current watermark
   - On restart: read watermarks from RocksDB, resume forwarding from last-delivered+1
   - This provides at-least-once semantics: entries between crash and watermark advance
     are re-delivered after restart (destination must be idempotent or accept duplicates)

4. Implement `vector` forwarder in `crates/prism-audit/src/forward/vector.rs`:
   - Vector accepts log events via HTTP source (`POST {endpoint}/_bulk`)
   - Body: NDJSON (newline-delimited JSON), one audit entry per line
   - Authentication: `Authorization: Bearer {token}` (from credential ref)
   - HTTP 2xx → success; 4xx (except 429) → `ForwardError::Permanent`; others → `ForwardError::Transient`
   - Batch size: configurable, default 50 entries per POST

5. Implement `syslog` forwarder in `crates/prism-audit/src/forward/syslog.rs`:
   - RFC 5424 format with CEF/LEEF structured data extension for compliance systems
   - TCP transport with TLS (required — audit data must not transit in cleartext)
   - Acknowledgment: TCP ACK (connection-level; syslog RFC 5425 does not provide
     application-level ACK). Watermark advances after TCP send returns without error.
   - `ssl_ca_cert: Option<PathBuf>`: path to CA cert for server verification

6. Implement `webhook` forwarder in `crates/prism-audit/src/forward/webhook.rs`:
   - POST entries as JSON array to configured URL
   - `Authorization` header from credential ref
   - HTTP 2xx → advance watermark; other → ForwardError per HTTP status

7. Implement retry scheduler in `crates/prism-audit/src/forward/scheduler.rs`:
   - `AuditForwardScheduler::run(config: AuditForwardConfig, storage: Arc<RocksDB>)`:
     - Spawn one task per destination (independent — one destination's outage does not
       block others)
     - Each task: poll `audit_buffer` CF for entries since `last_delivered`, batch them,
       call `forwarder.deliver(batch)`
     - On success: advance watermark
     - On `Transient` error: exponential backoff
       - Attempt 1: wait 2s, attempt 2: 4s, ..., up to `backoff_max_secs` (60s)
       - After `retry_max_attempts`: log `ERROR`, continue retrying (audit delivery is
         critical — no dead-letter, no drop — keep trying until destination recovers)
     - On `Permanent` error: log `ERROR`, skip the offending entry (move watermark past
       it), continue with next entries. A permanently-undeliverable entry blocks nothing.
   - Poll interval when caught up (no pending entries): 5 seconds

8. Implement buffer overflow / FIFO eviction in `crates/prism-audit/src/forward/scheduler.rs`:
   - Monitor `audit_buffer` CF size against `buffer_cap_mb`
   - On exceeding cap: evict oldest entries (lowest UUID v7 keys) until below 90% of cap
   - Eviction skips entries with an unadvanced watermark for any destination (i.e., do
     not evict entries that have not yet been delivered to all destinations)
   - If the oldest entries have been delivered to all destinations, they are safe to evict
   - Log `WARN` on eviction: `"audit_buffer cap exceeded: evicting {n} oldest entries"`
   - This provides 24-hour tolerance at typical audit volumes (assumes `buffer_cap_mb`
     is sized appropriately by the operator)

9. Implement `prism audit status` diagnostic in `crates/prism-audit/src/cli/audit_status.rs`:
   - Output per destination:
     - `name`: destination name
     - `last_delivered_id`: most recently delivered entry ID (UUID v7)
     - `pending_count`: entries in `audit_buffer` since `last_delivered_id`
     - `last_error`: last retry error (if any)
     - `retry_count`: current retry attempt number (0 if healthy)
   - JSON output with `--format json`
   - This command reads directly from RocksDB (does not require the server to be running)

10. Write tests in `crates/prism-audit/tests/audit_forward_tests.rs`:
    - Test at-least-once: write 10 entries, simulate crash mid-delivery, restart →
      verify entries are re-delivered from watermark
    - Test destination outage: destination returns 503 → verify retry with backoff
    - Test FIFO eviction: fill to cap → verify oldest-delivered entries evicted first
    - Test permanent error: destination returns 400 on specific entry → entry skipped,
      watermark advances, subsequent entries delivered normally
    - Test 2 destinations: verify each destination maintains independent watermarks

## Acceptance Criteria

**AC-1:** Given Prism crashes after writing an audit entry to `audit_buffer` but before
the watermark advances, When Prism restarts, Then the entry is re-delivered to the
destination (at-least-once guarantee holds).
(traces to BC-2.05.001, BC-2.05.002 postcondition 1)

**AC-2:** Given a destination returns 503 for 3 consecutive delivery attempts, When the
4th attempt succeeds, Then all pending entries are delivered and the watermark advances.
The retry backoff sequence is 2s, 4s, 8s for the 3 failed attempts.
(traces to BC-2.05.003 postcondition 1)

**AC-3:** Given the `audit_buffer` CF exceeds `buffer_cap_mb`, When the overflow check
runs, Then the oldest entries that have been delivered to ALL destinations are evicted
(FIFO), and a WARN log is emitted. Entries NOT yet delivered to any destination are
NOT evicted.
(traces to BC-2.05.004 postcondition 1)

**AC-4:** Given destination A and destination B are both configured, When destination
A is in retry backoff (503), Then destination B continues delivering entries on its
normal schedule — the two destinations are independent.
(traces to BC-2.05.003 postcondition 1 — independent per-destination retry)

**AC-5:** Given `prism audit status`, When destination "vector-prod" has 150 pending
entries and is in retry attempt 3, Then the output shows `pending_count: 150`,
`retry_count: 3`, and the last error message.

**AC-6:** Given an audit entry with a permanent delivery error (destination returns 400),
When the scheduler processes it, Then the entry is skipped (watermark advances past it),
an ERROR is logged, and subsequent entries continue delivering normally.
(traces to BC-2.05.002 — best-effort when truly permanent; entry still exists in CF)

## Architecture Compliance Rules

- Watermark advance MUST happen AFTER the destination acknowledges receipt. Advancing
  the watermark before acknowledgment would create data loss on crash between advance
  and delivery.
- The `audit_buffer` CF MUST be the single source of truth for pending audit entries.
  Forwarder tasks read from it, advance watermarks against it, and evict from it. There
  is NO secondary queue or in-memory buffer that can become the source of truth.
- `AuditForwardScheduler` tasks MUST be independent per destination. A `tokio::task`
  per destination with no shared state between tasks (except the RocksDB handle,
  which is `Arc<RocksDB>`).
- Syslog forwarding MUST use TLS. Audit data contains security-sensitive events; unencrypted
  transit is prohibited. Enforce TLS in the syslog forwarder; do not accept `ssl_verify: false`
  unless explicitly documented as an insecure configuration.
- The `prism audit status` command MUST be able to run without the Prism server process
  running. It reads directly from the RocksDB file, not via an IPC socket. This is a
  safety property: operators need audit status even when the server is down.
- Credentials for destination authentication MUST be resolved at delivery time from the
  `CredentialStore`. Never store credential values in the `AuditForwarder` struct itself
  (they may rotate).

## Notes

- The `audit_buffer` CF uses UUID v7 keys (time-sortable). Iterating "since last_delivered"
  is efficient: use RocksDB's `seek_for_prev(watermark_id)` then iterate forward. This
  avoids full-table scans even with millions of audit entries.
- At-least-once semantics mean a destination that crashed mid-delivery may receive
  duplicate entries on recovery. This is acceptable and expected. Compliance logging
  systems (Vector, Splunk, Elasticsearch) are designed to handle duplicates via dedup
  on a unique audit entry ID.
- The 24-hour tolerance target requires `buffer_cap_mb` to be sized for the expected
  audit volume. A typical Prism deployment generating 10,000 audit entries/hour at ~1KB
  each needs: 10,000 * 24 * 1KB = 240MB. A 512MB default provides headroom.

## Product-Owner Notes (Flagged for Review)

**Proposed new BC: BC-2.05.009 Audit Forwarding At-Least-Once**

This story implements a non-trivial delivery contract (at-least-once with watermark,
retry, FIFO eviction). The behavior should be formally specified as a BC to enable:
- Adversarial review of the watermark semantics (advance after ACK vs. advance before)
- Formal verification via a Kani proof (watermark monotonicity invariant)
- Clear implementer contract for the retry-forever behavior vs. drop behavior on permanent errors

Proposed BC contents:
- Precondition: audit entry written to `audit_buffer` CF
- Postcondition 1: entry delivered to all configured destinations or watermark not advanced (entry retried on restart)
- Postcondition 2: permanent delivery failure skips that entry with ERROR log; watermark advances past it
- Invariant: watermark for destination D is monotonically increasing; entries below watermark have been delivered or permanently failed

Flag as BC-2.05.009 for product-owner to draft and adversarially review before Wave 5
implementation begins if possible.
